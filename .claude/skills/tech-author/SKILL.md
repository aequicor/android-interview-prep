# Tech Author — точность фактов + живой текст

Назначение: техническая книга/методичка, которая (1) **фактически точна** — ничего не
выдумано, и (2) **не пахнет ИИ** — её писал живой человек, ведущий читателя по разбору, а не генератор. Два провала, которые
этот скил предотвращает: уверенно неверный факт и текст-штамп, который рецензент за
секунду опознаёт как машинный.

Почему отдельный скил, а не «просто напиши»: и галлюцинации, и ИИ-маркеры лезут в
предсказуемых местах. Их дешевле ловить чек-листом, чем замечать постфактум.

## Режим A — пишешь контент

1. **Сначала факты, потом проза.** Собери и проверь факты по первоисточникам
   (developer.android.com, kotlinlang.org reference, release notes). Сигнатуры, версии,
   числа — дословно из источника, не по памяти. Протокол и горячие зоны —
   `references/fact-discipline.md`.
2. **Пиши на глубину Middle+/Senior.** Не пересказ API, а механизм + edge cases +
   trade-offs + failure modes + реальные follow-up-вопросы. Критерии и быстрый чек —
   `references/completeness.md`.
3. **Пиши в целевом регистре сразу.** Регистр методички — живой технический разбор (как
   сильный тех-блог), не лекция с кафедры и не канцелярит: спецификация и примеры —
   `references/voice.md`. При этом избегай ИИ-штампов (`references/ai-markers.md`) и не
   сваливайся в перебор — сленг, мемы, восклицания через слово.
4. **Прогон перед сдачей.** Самопроверка по чек-листу внизу + чек полноты из
   completeness.md. Для высоких ставок — отдельный ревьюер-субагент на факты.

## Режим B — чистишь готовый текст (de-slop)

Когда дают черновик (свой, чужой или сгенерированный другой моделью) и просят «сделать
человечнее / убрать ИИ-маркеры / проверить»:

1. **Маркеры: найди и перепиши.** Пройди текст против `references/ai-markers.md`.
   Каждый `✗`-оборот заменяй фактом из `✓`-колонки — не косметикой, а конкретикой
   (механизм, число, имя). Удаление штампа без добавления сути — полдела.
2. **Факты: каждое утверждение — проверяемо?** Версии, сигнатуры, числа, «официальные
   рекомендации» — против `references/fact-discipline.md`. Непроверяемое — проверь,
   опусти или пометь неуверенность.
3. **Структура.** Сломай machine-симметрию: одинаковую длину абзацев, резюме в конце
   каждого раздела, шаблон «**Термин:** —» на весь документ, мотивационный эпилог.

## Несокращаемый минимум (держи в голове всегда)

Маркеры, которые лезут чаще всего — не пиши их вовсе:

- **«Стоит/важно отметить»**, **«играет ключевую роль»**, **«позволяет/обеспечивает»** —
  скажи сам факт с механизмом, а не оборот о факте.
- **«является»** связкой в КАЖДОМ предложении (механический повтор) — варьируй форму;
  умеренная связка, вводящая определение, допустима (`voice.md`).
- **«не только…, но и…»**, **триколоны по три**, **«не просто X, а Y»** — подставная
  симметрия; оставь одно содержательное утверждение.
- **«Давайте рассмотрим»**, **вступительный абзац о важности темы**, **резюме в конце
  каждого раздела**, **«Подводя итог… Удачи!»** — начинай и заканчивай фактом.
- **Список «**Термин:** — пояснение» на весь документ** — чередуй абзац / таблицу /
  код / простой список.
- **Декоративные эмодзи в заголовках**, **избыточный болд** (0–2 на абзац).

Регистр — живой технический разбор (`references/voice.md`):

- Веди читателя за собой: повествование от «мы» / «я», разбор в настоящем времени
  («повернём — посмотрим»), прямое обращение на «вы» и предвосхищение его вопроса.
- Рваный ритм (длинное разъяснение + короткая фраза-посадка), дозированная аналогия и
  ирония, риторический вопрос — только если за ним идёт ответ.
- «В меру»: без тяжёлого сленга («железка», «грабли»), мемов, панибратского «ты» и
  восклицаний через слово. Это всё ещё учебный материал — точные термины и глубина на месте.
- Не сваливайся обратно в канцелярит: «осуществляется», «производится», «в целях»,
  безличный лекторский тон без автора и читателя.

Точность — без исключений:

- Сигнатуры, имена API, ключи Gradle/Manifest — **дословно из reference**, не по памяти.
- Версия ↔ API level ↔ кодовое имя — **таблицей целиком**, не по одной строке (ловит сдвиг ±1).
- Никаких выдуманных **«в N раз быстрее»** и дефолтов «по логике» — цитата или опусти число.
- Deprecated-статус — **проверь бейдж** перед тем, как советовать API.
- Не уверен — **пометь или опусти** (как именно помечать, не скатываясь в ИИ-хедж, — в `fact-discipline.md`). Честная неопределённость лучше уверенной выдумки.

## Чек-лист перед сдачей

- [ ] Факты в горячих зонах сверены по первоисточнику (`fact-discipline.md`); числа/версии/сигнатуры — из reference, не по памяти.
- [ ] **(режим A)** Глава проходит чек полноты (`completeness.md`): механизм, edge cases, trade-offs, failure modes, follow-up-вопросы. В режиме B — только если просили дописать по сути.
- [ ] Текст прогнан против `ai-markers.md`; маркеры заменены фактами, а не удалены вхолостую.
- [ ] Регистр — живой технический разбор (`voice.md`): «мы»/«вы»-повествование, рваный ритм, дозированная ирония; без канцелярита-воды с одной стороны и без сленга/мемов/восклицаний через слово — с другой.
- [ ] Нет machine-симметрии: длина абзацев рваная, резюме только где есть нетривиальный вывод, форматы подачи перемешаны.
- [ ] Болд точечный; декоративных эмодзи нет; нет мотивационного эпилога.

## Файлы

- `references/fact-discipline.md` — протокол точности и горячие зоны галлюцинаций.
- `references/completeness.md` — критерии технической полноты + быстрый чек главы.
- `references/ai-markers.md` — полный каталог ИИ-маркеров (рус. лексика и синтаксис, структура, англ.) с заменами.
- `references/voice.md` — регистр изложения: живой технический разбор, его приёмы и соотношение с анти-маркерами (и сухой академизм, и перебор — мимо).
