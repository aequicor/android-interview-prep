# Техническая полнота главы (Middle+/Senior)

Поверхностный разбор пересказывает API. Полный — объясняет, как механизм работает, где
он протекает на границах и какими исключениями падает в проде, чтобы кандидат пережил
follow-up-вопросы. Ниже — что должно быть в
серьёзной главе. Не каждый пункт нужен в каждой теме, но отсутствие пункта должно быть
осознанным, а не от незнания.

- **Механизм «под капотом», а не пересказ API.** Внутренняя реализация, иначе глава
  равна документации и не помогает диагностировать нестандартный баг.
  ✗ «ViewModel переживает поворот; наследуешь от ViewModel, получаешь через by viewModels()».
  ✓ ViewModel переживает смену конфигурации, потому что на recreate() Activity кладёт его
  в NonConfigurationInstances, передаваемый в новую Activity; держит ViewModelStore,
  привязанный к ViewModelStoreOwner; при настоящем уничтожении → ViewModelStore.clear() →
  onCleared(). Отсюда утечка Context во ViewModel: Store живёт дольше Activity. Поворот от
  финального закрытия отличают через isChangingConfigurations().

- **Edge cases и gotchas — где абстракция протекает.** Счастливый путь знает джун; на
  границах падает прод и строятся каверзные вопросы.
  ✗ «StateFlow всегда отдаёт последнее значение — бери вместо LiveData».
  ✓ StateFlow конфлейтит и сравнивает через equals(): два равных значения подряд →
  второй emit проглочен. Для одноразовых событий (навигация, snackbar) это баг — нужен
  SharedFlow(replay=0)/Channel. И .value неатомарен для read-modify-write → .update {}.

- **Trade-offs и когда НЕ использовать.** Цена и границы применимости. Интервьюер почти
  всегда спрашивает «а минусы? когда не возьмёшь?».
  ✗ «Используй Hilt для DI — удобно».
  ✓ Hilt хорош для типового графа Android-компонентов, но завязан на kapt/ksp (растёт
  сборка), генерит по аннотациям (хуже отладка графа), плохо ложится на динамические
  зависимости и multi-module с feature-флагами. Для библиотеки/KMP — Koin или manual DI.
  Ось выбора: compile-time safety против гибкости и скорости сборки.

- **Разрушение мифов и заученных полу-правд.** Иначе кандидат уверенно повторит миф и
  завалится.
  ✗ «launch и async создают корутины; async возвращает результат».
  ✓ Миф «async всегда параллелит» — параллельность даёт отложенный await, не сам async:
  два async с немедленным await идут последовательно. Миф «GlobalScope удобен для фона» —
  игнорирует структурную конкурентность, не отменяется с экраном, течёт. Миф
  «withContext(Dispatchers.IO) делает код асинхронным» — он лишь переключает поток.

- **Сравнение с конкретной альтернативой.** Тема не должна висеть в вакууме; вопросы
  часто формулируют как X-vs-Y.
  ✗ «Coroutines — способ писать асинхронный код».
  ✓ Корутины vs RxJava: оба про асинхронность, но корутины дают последовательный код
  (suspend вместо цепочек), дёшевы (стейт-машина, не поток), структурную отмену из
  коробки; Rx сильнее в композиции потоков событий и backpressure, но тяжелее и легче
  течёт без управления подписками. Flow закрывает реактивную часть, но cold-by-default и
  без backpressure-протокола Rx.

- **Связь с lifecycle / потоками / памятью + владение и teardown.** В Android ~80% багов
  (утечки, ANR, краши после поворота) живут на этих осях: кто держит ресурс и когда он
  снимается.
  ✗ «Подпишись через observe; корутину запусти CoroutineScope(Dispatchers.IO).launch {}».
  ✓ observe(lifecycleOwner) снимает наблюдателя в DESTROYED и молчит в STOPPED — во
  фрагменте нужен viewLifecycleOwner, не this: view живёт короче фрагмента (backstack),
  подписка на this течёт и крашит двойной доставкой. Симметрично: CoroutineScope(...).launch
  ни к чему не привязан и не отменяется — утечка; нужен viewModelScope/lifecycleScope.

- **Эволюция между версиями.** Без истории читатель тянет устаревшие практики и не
  понимает, почему в легаси «так».
  ✗ «Для фоновых задач — сервисы».
  ✓ До API 21 — свободные Service/AlarmManager; Android 6 (Doze) режет фон; API 26 —
  background execution limits, startService в фоне → IllegalStateException, нужен
  startForegroundService + нотификация за 5 сек; канон — WorkManager. Android 12 — запрет
  старта foreground-сервиса из фона; 13 — рантайм-разрешение на нотификации.

- **Код, иллюстрирующий суть (сломанный → почему → исправленный).** Сниппет из доков
  ничего не доказывает; код демонстрирует саму неочевидность через контраст.
  ✗ `lifecycleScope.launch { val data = repo.load(); textView.text = data }` как «пример
  использования корутин».
  ✓ Сниппет показывает отмену: `while(true){}` не отменяется (нет suspension point) →
  нужен ensureActive()/yield(); рядом — почему очистка в finally при suspend-вызове
  требует withContext(NonCancellable). Каждая строка про cooperative cancellation.

- **Реальные вопросы интервьюера + развилки.** Это методичка для собеса: нужны
  формулировки разной глубины и куда копают после первого ответа.
  ✗ «Q: что такое ViewModel? A: класс для UI-данных».
  ✓ Q: чем ViewModel лучше onSaveInstanceState? → а процесс убит в фоне, переживёт? (нет —
  только SavedStateHandle) → чем SavedStateHandle отличается от поля во ViewModel? → куда
  сериализуется Bundle и почему нельзя класть большое (TransactionTooLargeException, ~1 МБ
  на Binder-транзакцию)? Глава ведёт по всей цепочке, не до первого ответа.

- **Failure modes: симптомы, исключения, диагностика.** Senior знает, *какими*
  исключениями тема падает в проде и как дебажить. Это другой ракурс, чем edge cases: как
  ломается и как чинить инцидент.
  ✗ «RecyclerView показывает список через адаптер и ViewHolder».
  ✓ `Inconsistency detected: IndexOutOfBounds` — данные изменены не на main или notify*
  рассинхронен; мерцание — нет stableIds/DiffUtil; утечка — анонимный listener в
  ViewHolder; чужие данные после рециклинга — состояние не сброшено в onBindViewHolder.
  Корень — нарушение «адаптер и данные мутируются только из main и только через notify».

- **Производительность: стоимость, аллокации, чем мерить.** На Middle+ ждут мышление о
  цене конструкции.
  ✗ «Compose быстрее старого UI, потому что современный».
  ✓ Скорость Compose определяет рекомпозиция (Composition → Layout → Draw): переплата,
  когда рекомпозируется лишнее. Рычаги: стабильность типов (нестабильный параметр →
  функция не skippable), remember, derivedStateOf, key() в списках. Аллокации в теле
  @Composable бьют по GC (вызов на каждый кадр). Мерить: Layout Inspector (счётчик
  рекомпозиций), Macrobenchmark/baseline profiles.

- **Потокобезопасность: видимость, атомарность, happens-before.** Без разбора того, что
  именно НЕ гарантировано, глава создаёт ложную уверенность.
  ✗ «Чтобы переменная была потокобезопасной — @Volatile».
  ✓ @Volatile даёт видимость и запрет переупорядочивания, но **не** атомарность: counter++
  через volatile — всё равно гонка (read-modify-write), нужен AtomicInteger/synchronized/
  Mutex. В корутинах блокирующий synchronized нельзя держать через suspension point —
  Mutex.withLock. Double-checked locking без volatile сломан. Идиоматичный синглтон —
  object или by lazy.

- **Контракт API: nullability, исключения, идемпотентность, совместимость.** Зрелый
  разбор фиксирует, что гарантировано / что бросает / что nullable / что меняется при
  обновлении.
  ✗ «Парси JSON через Gson — он сам превратит строку в объект».
  ✓ Ловушки Gson в Kotlin: создаёт объекты через Unsafe в обход конструктора → non-null
  поле без значения в JSON станет null (NPE позже и не там), игнорируются default-значения.
  kotlinx.serialization уважает nullability и defaults, требует @Serializable на
  компиляции — ошибки контракта всплывают раньше. Gson — легаси-выбор с риском молчаливых
  null.

## Быстрый чек главы

Пройди по готовой главе. Если на половину — «нет», глава джуновская.

- [ ] Есть механизм «как работает», а не только «что это».
- [ ] Названы edge cases / где абстракция протекает.
- [ ] Есть trade-offs и явное «когда НЕ использовать».
- [ ] Разобран хотя бы один распространённый миф по теме.
- [ ] Есть сравнение с конкретной альтернативой (не «бывают и другие»).
- [ ] Показана связь с lifecycle / потоком / памятью; названо, кто владеет ресурсом и когда он снимается.
- [ ] Отмечено, что изменилось между версиями (если применимо).
- [ ] Код демонстрирует суть темы (контраст «сломано → почему → как надо»), а не Hello World.
- [ ] Есть реальные вопросы интервьюера с follow-up-цепочкой.
- [ ] Названы failure modes: конкретные исключения/симптомы и как диагностировать.
