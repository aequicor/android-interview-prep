# Дисциплина фактов — анти-галлюцинации

Техтекст врёт не равномерно, а в предсказуемых зонах. Правило одно: **для каждой
горячей зоны — конкретная проверка по первоисточнику. Если проверить нельзя —
пометь неуверенность или опусти. Не достраивай «по аналогии».**

Первоисточники по приоритету: `developer.android.com` и `kotlinlang.org` (reference,
не блог) → официальные release notes → книги в `books/` (grep по `.md`) → KEEP /
AOSP-исходники → проверенные доклады.
Агрегаторы, старые статьи и Stack Overflow — повод проверить, а не источник истины.

Книги в `books/` — специализированные первоисточники для тем Kotlin / Coroutines /
Compose. Перед написанием любого утверждения по этим темам — grep:

```bash
grep -n "ключевое_слово" books/kotlin-in-action-2ed.md | head -20
grep -rni "ключевое_слово" books/*.md | head -20
```

Что есть в `books/`:
- `kotlin-in-action-2ed.md` — Kotlin In Action 2nd ed (2024): язык, типы, лямбды, DSL, generics
- `kotlin-coroutines-deep-dive-2ed.md` — Kotlin Coroutines Deep Dive 2nd ed: Flow, channels, structured concurrency, testing
- `compose-internals.md` — Jetpack Compose Internals: компилятор, slot table, Snapshot, recomposition

## Горячие зоны (где галлюцинация чаще всего)

- **Сигнатуры методов — дословно из reference, не по памяти.** Имена и порядок
  параметров, границы дженериков, вид аргумента (значение vs лямбда). Код
  «компилируется в голове», но сигнатура не та.
  ✗ «getOrElse(key, default) вернёт default, если ключа нет» (это getOrDefault).
  ✓ С kotlinlang.org дословно: `getOrDefault(key: K, defaultValue: V): V` и
  `getOrElse(key: K, defaultValue: () -> V): V`. Рядом: getOrElse — лямбда (ленивая),
  getOrDefault — готовое значение.

- **Платформенные бейджи stdlib (JVM-only / common / Native).** Методичка выдаёт
  JVM-функцию за «функцию Kotlin» — на KMP/Native не соберётся.
  ✗ «getOrDefault есть у любого Map в Kotlin».
  ✓ getOrDefault помечен «JVM only» (из java.util.Map). Кроссплатформенно — getOrElse.
  Платформа не проверена — не писать «работает везде».

- **API level ↔ кодовое имя ↔ версия Android.** Тройное соответствие съезжает на ±1
  и ломает все примеры с `SDK_INT`.
  ✗ «Android 14 — это API 33, TIRAMISU».
  ✓ По `Build.VERSION_CODES`: API 33 = TIRAMISU = Android 13; API 34 =
  UPSIDE_DOWN_CAKE = Android 14; API 35 = VANILLA_ICE_CREAM = Android 15. Дать таблицу
  целиком — соседние строки ловят сдвиг.

- **Дефолты и гарантированные минимумы.** Звучат «логично» — поэтому выдумываются,
  и теряется гарантированный минимум.
  ✗ «Dispatchers.Default — ровно по потоку на ядро».
  ✓ Дословно: число потоков Dispatchers.Default равно числу ядер CPU, **но не меньше
  двух**. Указать оба условия. Не подтверждено — «уточните в доке вашей версии».

- **Никаких выдуманных множителей «в N раз быстрее».** В офиц. доке таких цифр почти
  нет; интервьюер пробивает вопросом «откуда число?».
  ✗ «inline ускоряет в 5–10 раз».
  ✓ Док про inline даёт только качественное: устраняет аллокацию объектов функций и
  виртуальные вызовы. Писать механизм, не цифру. Числа нет в источнике — опустить.

- **Граница configuration change vs process death (ViewModel, onCleared).**
  Спрашивают почти всегда; концептуальная ошибка дорого стоит.
  ✗ «ViewModel переживает и поворот, и убийство процесса; onCleared() гарантированно
  вызовется при смерти процесса».
  ✓ ViewModel кэширует состояние через configuration changes; для process death нужен
  SavedStateHandle. onCleared() — при штатном уничтожении ViewModelStoreOwner; при
  убийстве процесса вызов **не гарантирован**. Формулировать буквально: config change —
  да, process death — нет.

- **Контракт интерфейса vs деталь реализации.** Перенос свойства реализации на
  интерфейс — содержательная ошибка.
  ✗ «Map в Kotlin сохраняет порядок вставки».
  ✓ Интерфейс Map порядок обхода **не** гарантирует; порядок вставки даёт
  LinkedHashMap. Всегда называть источник гарантии.

- **lateinit: полный список запретов + точное имя исключения.** Длинный список теряет
  пункты; имя исключения и синтаксис проверки реконструируются неверно.
  ✗ «lateinit можно на любое поле, включая Int и val; кидает NullPointerException;
  проверка — lateinit.isInitialized()».
  ✓ По docs/properties: только `var`, тип non-nullable и не примитив, нельзя в
  primary-конструкторе, без кастомных get/set. Кидает
  `kotlin.UninitializedPropertyAccessException` (не NPE). Проверка —
  `this::prop.isInitialized` (property reference, без скобок).

- **Имена флагов / Gradle-DSL / Manifest под конкретную версию.** Ключи AGP/Gradle
  меняются между версиями и различаются в Groovy/Kotlin DSL; выдуманное имя не
  подхватится сборкой.
  ✗ «в build.gradle: `minimumSdk = 24`, `enableR8 = true`».
  ✓ В нынешнем DSL — `minSdk`; R8 включён по умолчанию в release, отдельного
  `enableR8` нет. Указывать ключ под конкретную версию AGP и нужный DSL, цитируя
  пример из доки. Не изобретать camelCase по аналогии.

- **Устаревшие API и актуальная замена.** Методички тиражируют мёртвые API как живые
  либо называют неверную замену — это сразу датирует автора.
  ✗ «Для фоновой работы — AsyncTask, стандартный способ».
  ✓ AsyncTask помечен `@Deprecated`. Корутины/Executor, для отложенной работы —
  WorkManager. Для каждого deprecated-API — пометка + рекомендованная доком замена.

- **StateFlow конфлейтит значения (equals), а не «отдаёт каждое».** Звучит
  правдоподобно, поэтому проверку пропускают.
  ✗ «collect у StateFlow получает каждое присвоенное значение».
  ✓ StateFlow конфлейтит и пропускает равные по equals — промежуточные значения могут
  теряться. Любое утверждение о доставке/порядке — цитатой из reference или пометкой
  «нужно проверить».

- **Inline-модификаторы (crossinline/noinline) — определение дословно.** Память путает,
  какой что запрещает; неверная трактовка ломает реальный код.
  ✗ «crossinline разрешает non-local return».
  ✓ По docs/inline-functions: crossinline, наоборот, **запрещает** non-local return для
  лямбды, вызываемой из другого контекста; noinline исключает лямбду из инлайнинга.
  Определение каждого — дословно + минимальный компилируемый пример.

- **Даты релизов и «когда появилось».** Память выдаёт уверенно и неверно; дата легко
  проверяется собеседником.
  ✗ «Compose стал стабильным в 2020, в 1.0».
  ✓ По release notes: Compose 1.0 stable — июль 2021. Не подтверждено — обтекаемо
  («стабилизировался около 2021») или опустить.

- **Атрибуция «официальных рекомендаций» — по источнику.** Без ссылки мнение автора
  превращается в выдуманную позицию вендора.
  ✗ «Google официально требует один Activity на приложение».
  ✓ Гайд по навигации описывает single-Activity как один из поддерживаемых подходов, не
  предписанный. Нет ссылки — «распространённая практика», а не «Google требует».
  Любую кавычку — дословно с URL.

## Общий протокол

1. **Разделяй уровни уверенности.** Проверенный факт / общеизвестно в стандарте /
   моя догадка / не уверен. В тексте остаётся только первое и второе; остальное —
   проверить или убрать.
2. **Числа, даты, версии, имена — режим максимальной подозрительности.** Это зона №1.
   Нет первоисточника под рукой — опусти число и оставь механизм.
3. **Код проверяй до вставки.** Сигнатуры — copy-paste из reference, не из памяти.
   Сомнительный сниппет — собрать/прогнать.
4. **Гарантия языка/стандарта ≠ поведение конкретной реализации.** Всегда называй,
   *что* гарантирует поведение (интерфейс, класс, версия).
5. **При неуверенности честная формулировка сильнее выдуманной точности.** «Уточните в
   доке вашей версии» лучше уверенно неверной цифры. Для высоких ставок — отдельный
   проход ревьюером-субагентом на фактическую корректность.
6. **Как помечать неуверенность — и не превратить пометку в ИИ-хедж.** Адресная
   неопределённость — это факт о факте, а не размытие: называй, *что* не подтверждено и
   *где* свериться («поведение X не подтверждено в reference на момент версии N —
   проверьте перед использованием»), а не «обычно в большинстве случаев работает». В
   рабочем черновике ставь grep-токен `[CHECK: что проверить]`; вычитка снимает все токены
   перед сдачей — в финальный текст книги заглушки не попадают. Честная адресная пометка ≠
   хедж «в значительной степени / often» из ai-markers.md.
7. **Источники в длинном тексте.** Дословную цитату давай с точной ссылкой рядом; общие
   факты главы собирай в блок «Источники» в её конце (URL developer.android.com /
   kotlinlang.org, для версионно-зависимых фактов — с датой обращения). Разметку списка
   источников бери из interview-prep-author, не изобретай свою.
