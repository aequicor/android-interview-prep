Потоки и память JVM

Базовый слой всей конкурентности: поток как нить исполнения, гонки данных, взаимное исключение и — главное — модель памяти JVM (видимость, happens-before, переупорядочивание). Здесь же неблокирующая альтернатива блокировкам — CAS и атомики. Это фундамент, на котором затем строятся Looper/Handler, пулы потоков и корутины; на собесе проверяют не знание слова «поток», а понимание, почему без синхронизации код некорректен.

Middle Senior Частый вопрос

Поток и Runnable: носитель и задача

Поток (Thread) — это нить исполнения, которой ОС выделяет квант процессорного времени. В Java поток создаётся через Thread + Runnable, но напрямую плодить потоки — антипаттерн: каждый стоит примерно 512 КБ–1 МБ стека, а переключение контекста требует перехода в режим ядра. На практике используют пулы (Executor) или корутины — к ним отдельные разделы.

Thread и Runnable
// Runnable — задача; Thread — носитель исполнения
val task = Runnable { println("работаю в ${Thread.currentThread().name}") }
val t = Thread(task, "worker-1").apply { start() }  // start(), НЕ run()!
t.join()  // блокирующе ждём завершения
⚠️
Классический вопрос-ловушка. Вызов run() исполнит код в текущем потоке (это обычный метод), а новый поток создаёт только start(). Повторный start() на одном объекте кинет IllegalThreadStateException.

Поток бывает демоном (isDaemon = true): JVM завершится, не дожидаясь демон-потоков. Фоновые служебные потоки (таймеры, планировщики) обычно делают демонами, чтобы они не держали процесс живым.

Поток упал с непойманным исключением — что случится с остальной программой и кто это увидит? ★

Непойманное исключение завершает только сам этот поток, а не всю JVM — если, конечно, он не последний non-daemon. Перед смертью поток отдаёт исключение своему UncaughtExceptionHandler: сначала ищется тот, что повешен через Thread.setUncaughtExceptionHandler(), иначе берётся handler группы потоков (ThreadGroup), который по умолчанию печатает стектрейс в System.err. Глобальный fallback на всё приложение ставится через Thread.setDefaultUncaughtExceptionHandler().

Отсюда коварство: исключение в фоновом потоке часто «молча» уходит в логи, а не роняет приложение, и поймать его из потока-родителя через try/catch нельзя — у каждого потока свой стек. На Android именно через дефолтный handler крэш-репортеры (Crashlytics) и системный обработчик перехватывают падения фоновых потоков.

Гонка данных и synchronized

Когда два потока без синхронизации читают-пишут общее изменяемое состояние, результат недетерминирован. Операция count++ не атомарна — это три действия (read–modify–write), которые могут переплестись.

class Counter {
    @Volatile private var count = 0              // volatile НЕ спасёт от гонки инкремента!
    fun inc() { count++ }                        // read, +1, write — не атомарно
    fun get() = count
}
// 2 потока × 100_000 инкрементов → итог почти всегда < 200_000

Почему count++ не атомарна

Запись count++ компилируется не в одно неделимое действие, а в три отдельных шага над полем count (оно лежит в общей памяти, в куче):

  • Чтение (read). Значение count копируется из памяти в локальную ячейку потока — в байткоде JVM это getfield, кладущий значение на стек операндов. С этого момента поток работает с копией, а не с самим полем.
  • Инкремент (modify). К копии прибавляется единица (iadd). Поле в памяти при этом не затрагивается — меняется только локальное значение внутри потока.
  • Запись (write). Результат записывается обратно в поле (putfield). Только теперь новое значение становится видно остальным потокам.
count++              // одна строка — но три шага под капотом:
val tmp  = count     // read:   значение поля → в локальную копию потока
val next = tmp + 1   // modify: +1 над копией, поле в памяти не тронуто
count = next         // write:  результат → обратно в поле

Опасны не сами шаги, а зазоры между ними. Планировщик вправе вытеснить поток в любой момент, в том числе между чтением и записью; на многоядерном процессоре два потока выполняют эти шаги и вовсе одновременно на разных ядрах. «Просто прочитать и записать» недостаточно именно потому, что между чтением и записью значение поля успевает устареть: если оба потока прочитали count раньше, чем любой из них записал результат, они прибавят единицу к одному и тому же исходному числу и запишут одинаковое значение — вторая запись затрёт первую.

ШагПоток AПоток Bcount в памяти
1read → A = 41—41
2—read → B = 4141
3+1 → A = 42—41
4—+1 → B = 4241
5write 42—42
6—write 4242
⚠️
Потерянное обновление (lost update). Оба потока отработали по разу, но из 41 вышло 42 вместо 43: инкремент потока A перезаписан потоком B, у которого на руках было устаревшее значение. Одна такая накладка теряет ровно один инкремент — на сотне тысяч итераций они копятся, оттого итог из примера выше почти всегда заметно меньше 200_000. @Volatile здесь не спасает: он делает каждое отдельное чтение и запись видимыми сразу, но не связывает тройку read–modify–write в одно неделимое действие — зазор между чтением и записью остаётся.

synchronized: взаимное исключение + видимость

synchronized захватывает монитор объекта: только один поток внутри критической секции. Важно: монитор даёт не только атомарность, но и видимость — на входе в блок поток видит все изменения, сделанные до выхода из блока другим потоком (happens-before).

class SafeCounter {
    private var count = 0
    private val lock = Any()
    fun inc() = synchronized(lock) { count++ }   // взаимное исключение
    fun get() = synchronized(lock) { count }     // и видимость свежего значения
}

В Kotlin нет ключевого слова synchronized как в Java — есть inline-функция synchronized(lock) { … } и аннотация @Synchronized для методов. Под капотом это те же байткод-инструкции monitorenter/monitorexit.

Что такое reentrancy у synchronized и почему рекурсивный вызов synchronized-метода на том же объекте не вешает поток? ★

Мониторы в JVM реентерабельны: монитор хранит не просто флаг «занят», а счётчик удержаний и идентификатор владеющего потока. Когда поток, уже владеющий монитором, снова входит в synchronized на том же объекте, monitorenter лишь увеличивает счётчик — он не блокирует поток. На каждом monitorexit счётчик уменьшается, и монитор освобождается только когда счётчик дойдёт до нуля. Поэтому @Synchronized-метод спокойно вызывает другой synchronized-метод того же объекта без самоблокировки.

Практический подвох в другом: синхронизация по public-объекту (this, по классу) опасна — чужой код может захватить тот же монитор и устроить deadlock или starvation, о которых вы не подозреваете. Поэтому синхронизируются по приватному final-объекту-локу, как в примере выше с private val lock = Any().

Модель памяти: volatile и happens-before

Без синхронизации JIT и CPU вправе кэшировать переменную в регистре или кэше ядра — поток может «не увидеть» запись из другого потока сколь угодно долго. volatile гарантирует, что чтение идёт из главной памяти, а запись туда же сбрасывается, и запрещает переупорядочивание операций вокруг переменной.

Гарантияvolatilesynchronized / атомики
Видимость (visibility)✅ да✅ да
Запрет реордеринга✅ да✅ да
Атомарность compound-операций (x++)❌ нет✅ да
Взаимное исключение❌ нет✅ да (только synchronized/Lock)

Над чем работает volatile: поле, а не объект за ним

@Volatile ставится на поле любого типа, не только примитивного — модификатор просто помечает backing field. Но гарантия (видимость, запрет реордеринга, атомарность 64-битного доступа) распространяется ровно на значение самого поля: для примитива — на число, для ссылочного типа — на ссылку (на какой объект смотрит переменная), но не на содержимое объекта за ней.

class Box {
    @Volatile var i: Int = 5      // примитив int — volatile над самим числом
    @Volatile var n: Int? = 5     // ссылка на java.lang.Integer (бокс) — volatile над ссылкой
    @Volatile var s = State(0)    // ссылка на изменяемый объект — volatile над ссылкой
}
data class State(var x: Int)

val b = Box()

// Видимость самого поля — есть во всех трёх случаях:
b.i = 10          // другой поток гарантированно увидит 10
b.n = 10          // увидит новую ссылку на Integer(10)
b.s = State(7)    // увидит новую ссылку + безопасно опубликованный объект

// Здесь volatile уже не помогает:
b.i++             // read-modify-write — по-прежнему не атомарно
b.s.x = 7         // мутация поля ВНУТРИ объекта: x не volatile → запись может быть не видна

Сам тип примитива важен в одном месте. По спецификации JVM 64-битные long и double без volatile можно читать и писать двумя 32-битными половинами — другой поток рискует увидеть «разорванное» значение. @Volatile (или AtomicLong) возвращает атомарность доступа. У int, boolean и ссылок отдельные чтения и записи атомарны и без него — volatile добавляет им лишь видимость и запрет реордеринга.

⚠️
Ссылка — не объект за ней. volatile даёт безопасную публикацию новой ссылки, но не потокобезопасную мутацию объекта. Переприсваивание поля (b.s = State(7)) другой поток увидит целиком — вместе с полями опубликованного объекта; мутацию поля внутри уже опубликованного объекта (b.s.x = 7) — нет, если само поле не volatile. Чтобы защитить внутреннее изменяемое состояние, volatile/atomic ставят на сами поля объекта, доступ к ним синхронизируют или используют неизменяемый объект, который только переприсваивают.

Happens-before — формальное отношение из Java Memory Model: если действие A «происходит-до» B, то эффекты A видны в B. Ключевые правила: разблокировка монитора happens-before следующей блокировки того же монитора; запись в volatile happens-before последующего чтения этой же переменной; Thread.start() happens-before кода запущенного потока; код потока happens-before возврата из join(); запись в final-поле в конструкторе happens-before чтения ссылки на объект после конструктора.

💡
Канонический пример volatile — флаг остановки. @Volatile var running = true в одном потоке, цикл while (running) в другом. Без volatile цикл может крутиться вечно, не увидев running = false: компилятор вправе поднять чтение из цикла, решив, что переменная не меняется.

happens-before — это не «всем видно одно значение»

Частое заблуждение: «happens-before значит, что переменную видят все одинаково». На деле ребро happens-before связывает не одну переменную, а две точки во времени — и переносит через себя всю память. Запись в volatile работает как release-барьер, чтение той же переменной — как acquire-барьер. Когда читатель увидел значение, записанное под release, ему гарантированно видно и всё, что писатель сделал до этой записи, включая обычные, не-volatile поля.

Ребро happens-before от volatile-write к volatile-read той же переменной переносит читателю все предыдущие записи писателя, а не только саму volatile-переменную

Double-checked locking: почему поле обязано быть volatile

Классический пример, где acquire/release решает всё, — ленивый синглтон через double-checked locking. Запись instance = Heavy() не атомарна: под капотом это (1) выделить память, (2) выполнить конструктор, (3) записать ссылку в поле. JIT и CPU вправе переставить (2) и (3) местами. Тогда другой поток на первой проверке (без захвата лока) увидит ненулевую ссылку на ещё не до конца сконструированный объект.

sequenceDiagram participant A as "Поток A (создаёт)" participant M as "Память (поле instance)" participant B as "Поток B (читает)" A->>M: 1. allocate (выделить память) A->>M: 3. instance = ref (реордеринг: ДО конструктора!) B->>M: читает instance != null B-->>B: работает с НЕинициализированным объектом A->>A: 2. выполнить конструктор (поздно) Note over B: без volatile B видит «полуобъект»
Без volatile запись ссылки может опередить конструктор; volatile (release на записи, acquire на чтении) запрещает этот реордеринг и гарантирует безопасную публикацию

А с volatile запись ссылки получает release-семантику: к моменту, когда другой поток увидит ненулевой instance, все поля объекта уже записаны. Это и есть безопасная публикация. Без volatile DCL официально «broken» (см. «The Double-Checked Locking is Broken» declaration). В Kotlin его руками и не пишут — берут object (инициализация в <clinit>, потокобезопасна по гарантии загрузчика классов) или by lazy.

Зачем в double-checked locking поле обязано быть volatile и что конкретно сломается без него? ★

instance = Heavy() — это не атомарная публикация, а три шага: выделить память, выполнить конструктор, записать ссылку в поле. JIT/CPU вправе переставить запись ссылки перед завершением конструктора. Тогда поток, который на быстром пути (без лока) проверяет instance != null, увидит ненулевую ссылку на ещё не достроенный объект и начнёт читать его поля до того, как конструктор их инициализировал.

volatile запрещает этот реордеринг: запись ссылки получает release-семантику, чтение — acquire, и к моменту видимости ненулевой ссылки все поля гарантированно записаны — безопасная публикация. На практике в Kotlin DCL руками не пишут: object Heavy (инициализация в статическом блоке, потокобезопасна) или by lazy { Heavy() } с режимом SYNCHRONIZED делают то же самое без шанса ошибиться.

Deadlock, livelock, starvation

ПроблемаСутьКак лечат
DeadlockПотоки взаимно ждут блокировки друг друга (A держит L1, ждёт L2; B наоборот). Прогресса нет.Единый порядок захвата локов; tryLock с таймаутом; меньше локов.

Самая частая причина deadlock на практике — захват двух локов в разном порядке в разных местах кода. Универсальная профилактика: установить глобальный порядок захвата (например, по System.identityHashCode объектов) и всегда брать локи в нём.

Четыре условия Коффмана

Deadlock возникает только когда выполнены сразу все четыре условия Коффмана. Хорошая новость: чтобы deadlock стал невозможен, достаточно сломать любое одно из них — отсюда и весь арсенал приёмов.

Все четыре условия нужны одновременно; разрыв любого одного делает deadlock невозможным
Можно ли повесить поток одним локом и чем такой случай отличается от классического deadlock на двух локах? ★

Классический deadlock требует минимум двух потоков, двух ресурсов и циклического ожидания (полный набор условий Коффмана). Но повесить поток можно и одним локом — если лок не реентерабельный или используется неправильно. synchronized и ReentrantLock реентерабельны, а вот Semaphore(1) — нет: повторный acquire() тем же потоком заблокирует его на самом себе. У ReentrantReadWriteLock самоблокировка возникает при попытке апгрейда read→write (поток держит read и просит write — никто не отпустит read).

У самого ReentrantLock поток повесится, если вызвать lock() дважды, а unlock() — один раз: счётчик удержаний останется больше нуля, и лок никогда не освободится для других. Ровно поэтому withLock {} (то есть try/finally с unlock()) обязателен — он гарантирует строго один unlock на каждый lock.

wait / notify — низкоуровневая координация

Методы Object.wait()/notify()/notifyAll() работают только внутри synchronized на том же мониторе. wait() отпускает монитор и засыпает; notify() будит один ждущий поток, notifyAll() — все. Главное правило — проверять условие в цикле while, а не if (защита от ложных пробуждений, spurious wakeups, и от перехвата условия другим потоком).

// простой блокирующий буфер на одну ячейку
class OneCell<T> {
    private var value: T? = null
    private val lock = Object()

    fun put(v: T) = synchronized(lock) {
        while (value != null) lock.wait()        // ждём, пока освободится
        value = v
        lock.notifyAll()                         // будим потребителей
    }
    fun take(): T = synchronized(lock) {
        while (value == null) lock.wait()        // while, НЕ if!
        val v = value!!; value = null
        lock.notifyAll(); v
    }
}
ℹ️
В реальном коде вместо wait/notify берут высокоуровневые примитивы из java.util.concurrent: ReentrantLock + Condition, BlockingQueue, CountDownLatch, Semaphore. Но на собесе про wait/notify спрашивают, чтобы проверить понимание мониторов.
В чём практическая разница между notify() и notifyAll() и как notify() может «потерять» сигнал? ★

notify() будит ровно один произвольный ждущий поток, notifyAll() — все. Опасность notify() вылезает, когда на одном мониторе ждут потоки по разным условиям. Скажем, producer ждёт «буфер не полон», consumer — «буфер не пуст». notify() может разбудить «не тот» поток: он перепроверит своё условие в while, увидит, что оно по-прежнему ложно, и снова уйдёт в wait() — а нужный поток так и не проснулся. Сигнал пропал, система зависла.

Поэтому с одним монитором и несколькими условиями безопасен только notifyAll(). Правильная альтернатива — ReentrantLock с отдельным Condition на каждое условие (notFull/notEmpty): тогда signal() будит поток именно нужной категории, без «грома стада». Ровно по этой причине задача №2 в этой главе использует два Condition.

CAS и lock-free

Блокировки дороги (переключение в ядро, парковка потоков). Альтернатива — неблокирующие алгоритмы поверх аппаратной инструкции CAS (Compare-And-Swap). CPU предоставляет её как атомарную команду (CMPXCHG на x86, LL/SC на ARM): «сравни значение в памяти с ожидаемым; если совпало — запиши новое; верни признак успеха». Всё это — атомарно, без локов.

CAS-retry loop: читаем, вычисляем, пробуем атомарно записать; при провале повторяем

Атомики поверх CAS

Классы AtomicInteger, AtomicLong, AtomicReference инкапсулируют CAS-цикл. incrementAndGet() внутри — это спин-цикл с CAS, который HotSpot инлайнит в нативную инструкцию.

val counter = java.util.concurrent.atomic.AtomicInteger(0)
counter.incrementAndGet()                 // атомарный ++ без synchronized

// ручной CAS-цикл — суть incrementAndGet под капотом:
fun AtomicInteger.addCustom(delta: Int): Int {
    while (true) {
        val cur = get()
        val next = cur + delta
        if (compareAndSet(cur, next)) return next   // успех — выходим
        // иначе кто-то опередил — повторяем
    }
}
💡
Под высокой конкуренцией CAS-цикл «крутится» вхолостую и проигрывает. Для счётчиков с частой записью и редким чтением берут LongAdder: он распыляет инкременты по ячейкам (striping) и суммирует только при чтении — меньше contention.

ABA-проблема

CAS проверяет только значение, не историю. Если поток прочитал A, затем другой поток успел сделать A → B → A, наш CAS «успешно» сработает, хотя состояние менялось. Для стеков/очередей на ссылках это приводит к порче структуры.

Решение — версионирование: AtomicStampedReference хранит пару (ссылка, stamp-счётчик); CAS проверяет и ссылку, и марку, поэтому цикл A→B→A поднимет stamp и наш CAS провалится.

val ref = java.util.concurrent.atomic.AtomicStampedReference("A", 0)
val stampHolder = IntArray(1)
val cur = ref.get(stampHolder)            // читаем значение + версию
val stamp = stampHolder[0]
// сработает, только если и значение, и версия не изменились:
ref.compareAndSet(cur, "C", stamp, stamp + 1)

VarHandle — современная замена Unsafe

Исторически атомики опирались на внутренний sun.misc.Unsafe. С Java 9 появился VarHandle (java.lang.invoke) — безопасный типизированный доступ к полям с атомарными операциями и явными режимами памяти (getAcquire, setRelease, compareAndSet). Это легитимный способ строить свои lock-free структуры без приватного API.

ℹ️
В Kotlin-мире атомики прячутся внутри корутин: kotlinx.atomicfu и поля Job/каналов реализованы на CAS. Понимание CAS помогает объяснить, почему отмена и обновление состояния корутины потокобезопасны без блокировок.
CAS называют «lock-free», но не «wait-free» — в чём разница и какой прогресс CAS-цикл реально гарантирует? ★

Lock-free гарантирует прогресс системы в целом: хотя бы один поток завершит операцию за конечное число шагов, и нет блокировок, которые могут зависнуть навсегда. Но конкретный поток при этом может голодать (starvation): его CAS будет проваливаться сколь угодно долго, если соседи постоянно опережают его и меняют значение между его чтением и попыткой записи.

Wait-free — более сильная гарантия: каждый поток завершает операцию за конечное число своих шагов, независимо от остальных. CAS-цикл incrementAndGet() — lock-free, но не wait-free: под высоким contention он крутит retry вхолостую и жжёт CPU. Поэтому для счётчиков с частой записью берут LongAdder (striping по ячейкам) вместо AtomicLong — он снижает конкуренцию, хотя и не делает алгоритм wait-free.

Почему не «new Thread»: мост к пулам и корутинам

Прямое создание потоков плохо масштабируется: память на стеки, стоимость переключения контекста в ядре, отсутствие управления жизненным циклом. Отсюда два направления, которым посвящены отдельные разделы блока «Многопоточность»:

  • Пулы потоков (Executor, ThreadPoolExecutor) — переиспользуют ограниченный набор потоков под поток задач. См. раздел «Executor и пулы потоков».
  • Корутины — лёгкая абстракция исполнения поверх потоков: приостановка не блокирует поток, поэтому десятки тысяч корутин живут на пуле из нескольких потоков. См. раздел «Корутины и Flow».
  • На главном потоке Android планированием задач занимается петля событий Looper/Handler — см. «Looper и Handler».
ℹ️
Точная формулировка. Корутины не «быстрее» потоков сами по себе — они дешевле по памяти и удобнее по модели исполнения и отмены. Все они в итоге исполняются на потоках; разница в том, сколько потоков нужно и как ими управляют.

Память и GC на Android (ART)

Все потоки делят одну кучу, и за неё отвечает сборщик мусора. На Android рантайм — ART (заменил Dalvik в 5.0 Lollipop), а с версии 8.0 (Oreo) дефолтный сборщик — Concurrent Copying (CC): он generational (большинство объектов умирает молодыми, поэтому young-поколение чистится чаще и дешевле) и concurrent (объекты перемещает параллельно работе приложения, а корректность чтения ссылок во время копирования держит read-barrier, а не остановка всех потоков). От HotSpot на сервере ART отличается прежде всего жёсткими лимитами: куча каждого приложения ограничена (десятки–сотни МБ, зависит от устройства), а при нехватке памяти за дело берётся не GC, а системный Low Memory Killer — он убивает фоновые процессы целиком, по приоритету.

Главный практический риск — утечки. GC не освободит объект, на который тянется цепочка сильных ссылок от живого корня (GC root). Классика на Android: Activity, удержанная статикой, синглтоном, долгоживущим потоком или Handler с отложенным сообщением — экран давно закрыт, а его дерево View и Context висят в памяти. Лечится тем, что долгоживущие объекты не держат ссылок на UI, а если держат — берут applicationContext или WeakReference. Как ловить такие утечки инструментами (LeakCanary, heap dump, профайлер памяти) — в главе «Профилирование».

Почему выделение объекта в куче из нескольких потоков обычно НЕ требует синхронизации, хотя куча — общий ресурс? ★

Начиная с Android 8 (Oreo) дефолтный сборщик — Concurrent Copying с bump-pointer аллокатором RegionTLAB. Каждому потоку приложения выдаётся собственный TLAB (thread-local allocation buffer) — кусок региона кучи. Аллокация внутри TLAB сводится к сдвигу указателя top, и делается она без синхронизации, потому что буфер принадлежит одному потоку — гонки за общий указатель просто нет.

Синхронизация (через CAS или лок) нужна только в редкий момент, когда TLAB исчерпан и поток просит у аллокатора новый буфер. Этим и объясняется, почему создание объектов хорошо масштабируется по потокам. Сам CC при этом перемещает объекты конкурентно, а корректность чтения ссылок во время копирования держит read-barrier, а не stop-the-world.

Жизненный цикл потока и кооперативная отмена

У потока шесть состояний (Thread.State), и собеседующему важно, что вы различаете два «спящих»: BLOCKED и WAITING. В BLOCKED поток попадает, когда ждёт захвата монитора на входе в synchronized; в WAITING/TIMED_WAITING — когда сам отпустил монитор и ждёт сигнала (wait(), join(), park(), await()).

Почему stop()/suspend() устарели

Старые методы Thread.stop(), suspend(), resume() — deprecated и опасны. stop() мог убить поток в любой точке, оставив объекты в несогласованном состоянии (например, на середине обновления инварианта) и не отпустив мониторы. suspend() замораживал поток, не освобождая локи, — прямая дорога к deadlock. Поэтому остановка потока в Java — только кооперативная: попросить поток завершиться, а решает он сам.

Кооперативная отмена через interrupt
val worker = Thread {
    while (!Thread.currentThread().isInterrupted) {
        try {
            doWork()
            Thread.sleep(100)            // реагирует на interrupt → InterruptedException
        } catch (e: InterruptedException) {
            // флаг сброшен броском исключения — восстанавливаем его и выходим
            Thread.currentThread().interrupt()
            break
        }
    }
}
worker.start()
worker.interrupt()                       // выставляем флаг прерывания, не убиваем поток

interrupt() лишь выставляет флаг; сам поток обязан его проверять (isInterrupted()) и завершаться. Блокирующие методы (sleep, wait, join, BlockingQueue.take) реагируют на флаг, бросая InterruptedException и сбрасывая его. Глотать это исключение нельзя: либо завершайте работу, либо восстанавливайте флаг через Thread.currentThread().interrupt(), иначе вышестоящий код потеряет сигнал отмены. Тот же кооперативный принцип лежит в основе отмены корутин (isActive/ensureActive()) — см. «Корутины и Flow».

Чем interrupted() отличается от isInterrupted()? ★

Оба читают флаг прерывания, но различаются в двух вещах. Thread.interrupted() — статический метод: он работает с текущим потоком и сбрасывает флаг (после вызова флаг снова false). thread.isInterrupted() — метод экземпляра: проверяет флаг произвольного потока и не сбрасывает его.

Отсюда типичная ловушка: цикл while (!Thread.interrupted()) в одной строке и проверяет, и снимает флаг — повторная проверка ниже по коду уже ничего не увидит. Если флаг нужно сохранить (например, чтобы пробросить отмену выше), берут isInterrupted() либо вручную восстанавливают его после interrupted().

ReentrantLock и java.util.concurrent: чем locks богаче монитора

synchronized прост и надёжен — JVM сама отпустит монитор при выходе из блока, даже при исключении. Но монитор негибок: нельзя поставить таймаут на ожидание, нельзя прервать ждущий поток, нельзя выбрать справедливость. Когда это нужно, в дело идёт java.util.concurrent.locks.

ReentrantLock с таймаутом
import java.util.concurrent.locks.ReentrantLock
import java.util.concurrent.TimeUnit
import kotlin.concurrent.withLock

val lock = ReentrantLock(/* fair = */ false)

fun safeUpdate() = lock.withLock {       // withLock = try { lock() } finally { unlock() }
    // критическая секция
}

fun tryUpdate(): Boolean {
    if (!lock.tryLock(50, TimeUnit.MILLISECONDS)) return false  // не вешаемся навсегда
    try { /* работа */ ; return true } finally { lock.unlock() }
}

ReentrantLock даёт то, чего нет у монитора: tryLock() (в том числе с таймаутом — спасение от deadlock), lockInterruptibly() (прерываемое ожидание), несколько Condition на один лок и опциональную справедливость. Плата — обязательный try/finally { unlock() } (его инкапсулирует withLock {}): забыли unlock() — лок утёк навсегда.

ReadWriteLock и координаторы

ReentrantReadWriteLock разводит читателей и писателей: много читателей одновременно, писатель — эксклюзивно. Подвох — нельзя апгрейдить read→write (поток держит read и просит write, но write ждёт освобождения всех read, включая собственный → самоблокировка). StampedLock (Java 8) добавляет оптимистичное чтение без блокировки вообще. Для координации потоков, а не защиты данных, есть готовые примитивы:

ПримитивДля чего

SemaphoreОграничить число параллельных доступов к ресурсу.
PhaserБарьер с динамическим числом участников и фазами.
Почему non-fair ReentrantLock по умолчанию и обычно быстрее справедливого? ★

Справедливый (fair) лок выдаёт его строго в порядке очереди (FIFO): любой пришедший поток встаёт в хвост и ждёт. Non-fair допускает barging — «влезание»: если в момент освобождения лока подвернулся поток, который только что попросил его, он может схватить лок сразу, не вставая в очередь и не дожидаясь разбуженного из хвоста.

Это выгодно по производительности: разбудить припаркованный поток дорого (context switch, перепланирование), а пока он просыпается, «влезший» поток успеет войти и выйти из короткой критической секции. Меньше переключений контекста — выше throughput. Расплата — теоретическая возможность starvation для ждущих в очереди, поэтому fair-режим оставляют на случай, когда важна предсказуемая задержка, а не пропускная способность.

Безопасная публикация и неизменяемость

Синхронизация — не единственная стратегия потокобезопасности. Вторая, часто более дешёвая, — не делить изменяемое состояние вовсе: публиковать объекты безопасно и предпочитать неизменяемость. «Опубликовать» объект — значит сделать ссылку на него видимой другому потоку. Сделать это безопасно (так, чтобы второй поток увидел объект целиком, а не полуинициализированным) можно четырьмя способами:

  • из статического инициализатора (загрузчик классов даёт happens-before);
  • через volatile-поле или AtomicReference;
  • через final-поле, записанное в конструкторе;
  • под защитой лока (synchronized/Lock).

Гарантия final-полей

JMM даёт final-полям особую гарантию: значения, записанные в них в конструкторе, видны любому потоку, прочитавшему ссылку на объект после завершения конструктора — без всякой синхронизации. Поэтому корректно сконструированный неизменяемый объект (все поля final/val, this не «утёк» наружу) можно безопасно публиковать через обычное, даже не-volatile поле. На этой гарантии стоит потокобезопасность String, обёрток и data class с val.

Неизменяемость вместо блокировок
// Все поля val — объект неизменяем. Безопасно делить между потоками без локов.
data class Config(val url: String, val retries: Int)

// "Изменение" = создание новой копии, старая не трогается:
val next = current.copy(retries = current.retries + 1)
// другой поток либо видит старый Config целиком, либо новый — но не «полуобъект»

Где нет изменяемого общего состояния — там нет и гонки. data class с val и copy(), thread confinement (объект живёт в одном потоке, как при ThreadLocal), effectively immutable (объект больше не меняют после публикации) снимают целый класс ошибок, которые иначе пришлось бы закрывать локами.

Почему публикация объекта с не-final изменяемыми полями через обычное поле опасна, даже если его больше не меняют? ★

Гарантия видимости при публикации распространяется только на final-поля. Если поля объекта обычные (var без volatile), то даже когда вы записали ссылку в обычное (не-volatile) поле и больше объект не трогаете, JMM не обещает, что другой поток увидит записанные в конструкторе значения этих полей. Без ребра happens-before чтение ссылки и чтение полей за ней не упорядочены — читатель вправе увидеть ссылку уже ненулевой, а поля — ещё в дефолтных нулях (так называемая «частично сконструированная» видимость).

Объект становится effectively immutable и безопасным без синхронизации только при безопасной публикации: через final-поле, volatile, статический инициализатор или под локом. Либо сделайте поля final/val и получите гарантию автоматически.

Частые вопросы интервьюера

В чём разница между volatile и synchronized? Middle

volatile даёт только видимость и запрет реордеринга для одной переменной, но не атомарность compound-операций и не взаимное исключение. synchronized даёт всё сразу: взаимное исключение, атомарность критической секции и видимость (через happens-before на мониторе). Правило: флаг/публикация ссылки — volatile; составные инварианты и инкременты — synchronized или атомики.

Что такое happens-before и зачем оно? Senior

Это отношение из Java Memory Model, определяющее, какие записи одного потока гарантированно видны другому. Без него JIT/CPU вправе переупорядочивать и кэшировать операции, и поток может «не увидеть» чужую запись. Ключевые рёбра: unlock→lock того же монитора, write→read одной volatile, start()→тело потока, тело потока→join(), запись в final→чтение после конструктора. JMM — это контракт, на котором стоит вся потокобезопасность.

Объясните CAS и ABA-проблему. Senior

CAS (Compare-And-Swap) — атомарная инструкция CPU «сравни с ожидаемым и, если совпало, запиши новое». На ней строят lock-free структуры (атомики) с retry-циклом. ABA: поток прочитал A, другой успел A→B→A, и наш CAS ошибочно «успешен», хотя состояние менялось (опасно для стеков/очередей на ссылках). Решение — версионирование: AtomicStampedReference хранит марку-счётчик и сравнивает её вместе со значением.

Почему count++ небезопасен даже с @Volatile? Middle+

volatile обеспечивает видимость отдельных чтений и записей, но count++ — это три отдельных действия (read, +1, write). Между чтением и записью другой поток может вклиниться со своим инкрементом, и одно из обновлений потеряется. Нужна атомарность всей тройки: AtomicInteger.incrementAndGet() (CAS-цикл) или synchronized.

Как предотвратить deadlock из двух локов? Middle+

Базовый приём — единый глобальный порядок захвата локов: если все места кода всегда берут L1 раньше L2, циклического ожидания не возникнет. Когда естественного порядка нет, его задают искусственно (например, по идентичности объекта). Альтернатива — tryLock с таймаутом: не удалось взять второй лок за отведённое время — отпустить первый и повторить. Чем меньше локов и уже критические секции, тем меньше риск.

Зачем условие в wait() проверять в while, а не if? Senior

Из-за ложных пробуждений (spurious wakeups) и из-за того, что между notify и фактическим перезахватом монитора другой поток мог изменить условие. while перепроверяет предикат после пробуждения и снова уходит в wait(), если условие ещё не выполнено. С if поток продолжил бы исполнение при невыполненном условии — типичная гонка.

Что такое ThreadLocal и почему на Android он часто течёт? Senior

ThreadLocal<T> даёт каждому потоку свою копию значения — обращения thread-confined, синхронизация не нужна. Хранится значение не в самом ThreadLocal, а в поле threadLocals каждого Thread (это ThreadLocalMap, где ключ — слабая ссылка на сам объект ThreadLocal, а значение — сильная). Утечка вылезает в пулах потоков: поток живёт долго, карта не очищается, и пока поток жив, значение (а через него Activity/Context) удерживается сильной ссылкой — даже если сам ThreadLocal уже собран. Лечится явным remove() в finally. Типичный кейс — SimpleDateFormat или контекст запроса, привязанные к потокам пула.

Что такое false sharing и почему запись в РАЗНЫЕ поля тормозит потоки? Senior

Кэш процессора оперирует не байтами, а строками (cache line, обычно 64 байта). Если два независимых поля лежат в одной кэш-строке и два потока на разных ядрах пишут каждый в своё поле, протокол когерентности (MESI) вынужден гонять всю строку между ядрами и инвалидировать кэш — поля «делят» строку, хотя логически независимы. Это false sharing: корректность цела, но производительность падает в разы. Лечат паддингом (раздвинуть поля на размер строки) или аннотацией @Contended. Ровно из-за false sharing LongAdder со striping выравнивает свои ячейки по кэш-строкам.

Чем daemon-поток отличается от обычного? Middle

JVM работает, пока жив хотя бы один non-daemon (пользовательский) поток. Daemon-потоки (isDaemon = true) JVM убивает принудительно при завершении последнего пользовательского потока, не дожидаясь их и не выполняя в них finally. main по умолчанию non-daemon, новые потоки наследуют статус от создателя. Поэтому обычный поток с while(true) удержит JVM живой даже после возврата из main, а служебные потоки (таймеры, планировщики) делают daemon, чтобы они не мешали завершению.

Можно ли остановить поток через Thread.stop() и как прервать его правильно? Middle+

stop(), suspend(), resume() — deprecated и опасны: stop() мог прервать поток в любой точке, оставив объекты в несогласованном состоянии и не отпустив мониторы. Правильно — кооперативная отмена через флаг: interrupt() выставляет его, а сам поток обязан проверять (isInterrupted()) и завершаться. Блокирующие методы (sleep, wait, join, take) реагируют на флаг броском InterruptedException; ловить его нужно правильно — завершиться либо восстановить флаг через currentThread().interrupt(). На том же принципе построена отмена корутин.

Задача 1. Напишите потокобезопасный счётчик без synchronized и без блокировок — только с помощью атомарных операций. Реализуйте метод increment() и get(). Затем объясните: почему реализация с @Volatile var count = 0 и count++ не является потокобезопасной, даже если два потока инкрементируют переменную строго по очереди (без перекрытия)?

Показать решение 
import java.util.concurrent.atomic.AtomicInteger

class LockFreeCounter {
    private val count = AtomicInteger(0)

    fun increment() {
        // compareAndSet в цикле — суть incrementAndGet под капотом
        while (true) {
            val cur = count.get()
            if (count.compareAndSet(cur, cur + 1)) return
        }
        // то же самое одной строкой:
        // count.incrementAndGet()
    }

    fun get(): Int = count.get()
}

fun main() {
    val counter = LockFreeCounter()
    val threads = List(10) {
        Thread { repeat(100_000) { counter.increment() } }
    }
    threads.forEach { it.start() }
    threads.forEach { it.join() }
    println(counter.get()) // всегда 1_000_000
}

Объяснение: @Volatile гарантирует видимость каждого отдельного чтения и записи, но не связывает три шага count++ (read, modify, write) в одно неделимое действие. Даже при «строгой очерёдности» на уровне JIT два потока могут читать одно и то же значение до того, как любой из них запишет результат — lost update гарантирован на любом многоядерном CPU без барьера взаимного исключения. AtomicInteger.incrementAndGet() выполняет CAS-цикл, который на x86 компилируется в атомарную инструкцию LOCK XADD. Сложность O(1) амортизировано.

Задача 2. Реализуйте паттерн «производитель – потребитель» на ReentrantLock и Condition (без BlockingQueue). Буфер — массив фиксированного размера capacity. Производитель блокируется, когда буфер полон; потребитель — когда буфер пуст. Гарантируйте отсутствие busy-wait и корректное завершение при вызове close().

Показать решение 
import java.util.concurrent.locks.ReentrantLock
import kotlin.concurrent.withLock

class BoundedBuffer<T>(private val capacity: Int) {
    private val buf = ArrayDeque<T>(capacity)
    private val lock = ReentrantLock()
    private val notFull  = lock.newCondition()
    private val notEmpty = lock.newCondition()
    @Volatile private var closed = false

    fun put(item: T) = lock.withLock {
        // while, не if — защита от spurious wakeups и перехвата слота
        while (buf.size == capacity && !closed) notFull.await()
        if (closed) return
        buf.addLast(item)
        notEmpty.signal()
    }

    fun take(): T? = lock.withLock {
        while (buf.isEmpty() && !closed) notEmpty.await()
        if (buf.isEmpty()) return null   // closed + пусто
        val item = buf.removeFirst()
        notFull.signal()
        item
    }

    fun close() = lock.withLock {
        closed = true
        notFull.signalAll()   // разблокируем производителей
        notEmpty.signalAll()  // разблокируем потребителей
    }
}

Объяснение: два Condition на одном локе — стандартная замена пары wait/notifyAll. notFull.await() атомарно отпускает лок и паркует поток; notFull.signal() будит ровно одного ждущего производителя, не устраивая «гром стада» (thundering herd), как при notifyAll. Проверка условия в while обязательна: после пробуждения поток конкурирует за лок с другими и к моменту его захвата условие может снова оказаться ложным (spurious wakeup или другой поток успел занять слот). Сложность put/take — O(1) по времени удержания лока.

Задача 3. Найдите потенциальный deadlock в коде ниже и исправьте его, не убирая блокировки и не меняя логику перевода. Объясните, почему именно такой порядок захвата локов приводит к взаимоблокировке, и опишите универсальное правило предотвращения.

class Account(val id: Int, var balance: Double) {
    val lock = Any()
}

// ПРОБЛЕМНЫЙ КОД — возможен deadlock
fun transfer(from: Account, to: Account, amount: Double) {
    synchronized(from.lock) {
        synchronized(to.lock) {
            require(from.balance >= amount) { "Недостаточно средств" }
            from.balance -= amount
            to.balance  += amount
        }
    }
}
Показать решение 
// ИСПРАВЛЕННЫЙ КОД — единый порядок захвата по id
fun transfer(from: Account, to: Account, amount: Double) {
    // Всегда захватываем лок с меньшим id первым.
    // Нет цикличности → нет deadlock.
    val (first, second) = if (from.id < to.id) from to to else to to from

    synchronized(first.lock) {
        synchronized(second.lock) {
            require(from.balance >= amount) { "Недостаточно средств" }
            from.balance -= amount
            to.balance  += amount
        }
    }
}

// Альтернатива без изменения порядка: tryLock с таймаутом
import java.util.concurrent.locks.ReentrantLock
import java.util.concurrent.TimeUnit

class Account2(val id: Int, var balance: Double) {
    val lock = ReentrantLock()
}

fun transferTryLock(from: Account2, to: Account2, amount: Double): Boolean {
    while (true) {
        if (from.lock.tryLock(50, TimeUnit.MILLISECONDS)) {
            try {
                if (to.lock.tryLock(50, TimeUnit.MILLISECONDS)) {
                    try {
                        require(from.balance >= amount)
                        from.balance -= amount
                        to.balance  += amount
                        return true
                    } finally { to.lock.unlock() }
                }
            } finally { from.lock.unlock() }
        }
        // не удалось захватить оба лока — отступаем и пробуем снова
        Thread.yield()
    }
}

Объяснение: deadlock возникает, когда поток A вызывает transfer(alice, bob, 100.0), а поток B одновременно — transfer(bob, alice, 50.0). A держит alice.lock и ждёт bob.lock; B держит bob.lock и ждёт alice.lock — циклическое ожидание. Универсальное правило: если нужно захватить несколько локов, всегда делать это в одном и том же глобальном порядке (здесь — по id). Тогда цикл «A ждёт B, B ждёт A» невозможен по определению: чтобы возник цикл, кому-то пришлось бы захватывать локи в обратном порядке, что правило запрещает. Альтернатива — tryLock с таймаутом и отступом, но она сложнее и не устраняет livelock при высокой конкуренции.

Задача 4. Реализуйте потокобезопасный ленивый синглтон на double-checked locking так, чтобы getInstance() не входил в synchronized после первой инициализации. Объясните, какую строку нельзя убрать и почему без volatile паттерн некорректен. Покажите идиоматичную Kotlin-альтернативу без ручного DCL.

Показать решение 
class Heavy private constructor() {
    companion object {
        @Volatile private var instance: Heavy? = null   // volatile обязателен
        private val lock = Any()

        fun getInstance(): Heavy {
            val cached = instance
            if (cached != null) return cached            // быстрый путь — без лока
            return synchronized(lock) {
                // вторая проверка: пока ждали лок, другой поток мог уже создать
                instance ?: Heavy().also { instance = it }
            }
        }
    }
}

// Идиоматичный Kotlin — без ручного DCL:
object HeavySingleton                                    // init в <clinit>, потокобезопасно
// или:
class Service {
    companion object {
        val instance: Service by lazy { Service() }      // SYNCHRONIZED по умолчанию
    }
}

Объяснение: @Volatile убрать нельзя. Без него запись instance = Heavy() может стать видимой до завершения конструктора (реордеринг store) — поток на быстром пути увидит ненулевую ссылку на полусобранный объект. Вторая проверка внутри synchronized тоже обязательна: между первой проверкой и захватом лока другой поток мог уже создать instance, и без перепроверки мы создали бы второй экземпляр. В реальном Kotlin DCL руками не пишут: object инициализируется в статическом блоке (потокобезопасность гарантирует загрузчик классов), а by lazy с режимом SYNCHRONIZED делает то же самое.

Задача 5. Напишите класс, который из N потоков собирает счётчики событий по ключу (Map<String, Long>) без потери инкрементов и без блокировки на каждый инкремент. Сравните три варианта и объясните, почему merge/compute атомарны, а get-then-put — нет.

Показать решение 
import java.util.concurrent.ConcurrentHashMap
import java.util.concurrent.atomic.LongAdder

// Вариант 1 — корректно, но единый лок = bottleneck:
val m1 = HashMap<String, Long>()
fun inc1(k: String) = synchronized(m1) { m1[k] = (m1[k] ?: 0) + 1 }

// Вариант 2 — атомарная композиция на уровне бакета, без глобального лока:
val m2 = ConcurrentHashMap<String, Long>()
fun inc2(k: String) { m2.merge(k, 1L) { a, b -> a + b } }

// Вариант 3 — лучший под высокой записью: striping внутри LongAdder:
val m3 = ConcurrentHashMap<String, LongAdder>()
fun inc3(k: String) { m3.computeIfAbsent(k) { LongAdder() }.increment() }
fun total(k: String): Long = m3[k]?.sum() ?: 0

Объяснение: var c = map[k]; map[k] = c + 1 — это две раздельные операции, между ними другой поток вклинится со своим инкрементом → lost update. ConcurrentHashMap.merge/compute выполняют read-modify-write атомарно под защитой бакета (lock striping/CAS внутри CHM), поэтому корректны без глобального лока. LongAdder распыляет инкременты по нескольким ячейкам (striping), снижая contention и false sharing; читается через sum() — это самый быстрый вариант, когда запись частая, а чтение редкое.
