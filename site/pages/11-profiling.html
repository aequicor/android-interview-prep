StrictMode — дешёвый ранний детектор

App.kt
class App : Application() {
    override fun onCreate() {
        super.onCreate()
        if (BuildConfig.DEBUG) {
            StrictMode.setThreadPolicy(
                StrictMode.ThreadPolicy.Builder()
                    .detectDiskReads().detectDiskWrites()
                    .detectNetwork()            // I/O на main thread
                    .penaltyLog()               // в проде — никогда не penaltyDeath()
                    .build()
            )
            StrictMode.setVmPolicy(
                StrictMode.VmPolicy.Builder()
                    .detectLeakedClosableObjects()  // незакрытые Cursor/Stream
                    .detectActivityLeaks()
                    .penaltyLog()
                    .build()
            )
        }
    }
}

Память: утечки, GC и heap dump

Утечка в JVM/ART — это объект, который больше не нужен, но удерживается живой ссылкой от GC root, поэтому сборщик мусора не может его освободить. На Android самый дорогой «лист» такой цепочки — Activity/Fragment/View: они тянут за собой весь UI-граф и порой десятки мегабайт. Накопление → рост памяти → паузы GC → в пределе OutOfMemoryError.

Типичные источники утечек

Не отписанный listener / callbackВнешний объект держит ссылку на слушателя, привязанного к экрануСнимать в onStop/onDestroyView; lifecycle-aware

Как искать: LeakCanary → leak trace

LeakCanary ставит «сторож» (ObjectWatcher) на объекты, которые обязаны умереть (Activity после onDestroy, Fragment-view, ViewModel). Если объект жив спустя задержку и принудительный GC, число удержанных объектов превышает порог — LeakCanary снимает heap dump (.hprof), парсит его движком Shark и для каждого удержанного объекта строит кратчайший путь сильных ссылок от GC root — это и есть leak trace. В трейсе подсвечена подозрительная ссылка (~~~), с которой и начинается фикс.

flowchart LR GC["GC Root (static / thread / JNI)"] --> S["Cache.ctx : static field"] S --> A["DetailActivity (LEAKING)"] A --> V["DecorView"] V --> BMP["Bitmap ~12 MB (нативная память)"] classDef leak fill:#3a1d1d,stroke:#e06,stroke-width:2px,color:#fff; class A leak
Leak trace: цепочка сильных ссылок от GC root к утекшей Activity. Рвём слабое звено — static-ссылку
⚠️
WeakReference — не серебряная пуля. Слабая ссылка не мешает GC собрать объект, но если по цепочке есть хоть одна сильная ссылка — утечка останется. WeakReference уместна для кэшей и callback-ов на UI, но не «затыкает» архитектурную проблему удержания.

Память bitmap и heap dump вручную

С Android 8.0 пиксели bitmap живут в нативной куче (вне Java-heap), но всё равно учитываются как память процесса — большие незакрытые/непереиспользуемые bitmap быстро приводят к OOM. В Profiler делаем Capture heap dump: смотрим вкладку Apps Heap, сортируем по Retained Size, ищем дубли Activity (счётчик > 1 после ротаций — почти всегда утечка) и крупные массивы байт.

Старт приложения: cold / warm / hot

Метрика, которую видит пользователь и Play Store (Android Vitals). Различают три сценария по тому, сколько работы уже сделано.

ТипСостояние процессаЧто делаетсяЦена
ColdПроцесса нетFork zygote → Application.onCreate → inflate → first drawМаксимальная; цель < 500 мс
WarmПроцесс жив, Activity уничтоженаПересоздать Activity, заново нарисоватьСредняя
HotПроцесс и Activity живыВернуть на передний планМинимальная; цель < 100 мс

Замеряют две точки: TTID (Time To Initial Display — первый кадр) и TTFD (Time To Full Display — когда контент реально готов, репортится через reportFullyDrawn()).

flowchart LR Z["Zygote fork"] --> P["Process create"] P --> AC["Application.onCreate"] AC --> A["Activity create + inflate"] A --> M["measure / layout / draw"] M --> TTID["Первый кадр (TTID)"] TTID --> D["Загрузка данных"] D --> TTFD["reportFullyDrawn (TTFD)"]
Путь cold start. Всё на критическом пути растягивает TTID — особенно тяжёлый Application.onCreate

Что нельзя делать на старте (и на main thread)

🛑
Главный убийца cold start. Тяжёлая инициализация в Application.onCreate: сеть, чтение диска/БД, синхронная инициализация SDK, рефлексия, «прогрев» аналитики. Всё это блокирует main thread до первого кадра. Любой блокирующий I/O на UI-потоке → ANR-риск и jank.

App Startup library — отложенная и упорядоченная инициализация

Многие библиотеки втихую добавляют свой ContentProvider, чтобы инициализироваться на старте — каждый провайдер стоит времени. Jetpack App Startup заменяет их одним общим провайдером и даёт явный граф зависимостей инициализаторов, часть из которых можно сделать ленивыми.

WorkManagerInitializer.kt
class AnalyticsInitializer : Initializer<Analytics> {
    override fun create(context: Context): Analytics =
        Analytics.init(context.applicationContext)
    // зависит от логгера — App Startup сам выстроит порядок
    override fun dependencies(): List<Class<out Initializer<*>>> =
        listOf(LoggerInitializer::class.java)
}

// ленивую инициализацию запускаем вручную, когда фича реально нужна:
val analytics = AppInitializer.getInstance(context)
    .initializeComponent(AnalyticsInitializer::class.java)

Baseline Profiles — главный «бесплатный» ускоритель

Baseline Profile — это список «горячих» классов и методов (старт, навигация, скролл), который кладётся в APK/AAB. На установке ART делает по нему AOT-компиляцию (Profile Guided Optimization), минуя интерпретацию и JIT для этих путей. По данным Google это даёт ~30% к скорости выполнения с первого запуска (у Google Maps — ~30% к среднему времени старта). Профиль генерится Macrobenchmark-тестом (BaselineProfileRule) и работает с первого запуска и после каждого обновления.

StartupBenchmark.kt
@RunWith(AndroidJUnit4::class)
class StartupBenchmark {
    @get:Rule val rule = MacrobenchmarkRule()

    @Test fun coldStartup() = rule.measureRepeated(
        packageName = "com.example.app",
        metrics = listOf(StartupTimingMetric()),
        iterations = 10,
        startupMode = StartupMode.COLD          // COLD / WARM / HOT
    ) {
        pressHome()
        startActivityAndWait()                   // ждём первый кадр
    }
}
💡
Связка для собеса. Macrobenchmark измеряет старт, Baseline Profile ускоряет его, а App Startup + отложенная инициализация убирают лишнюю работу с критического пути. Назвав все три вместе, ты показываешь системное понимание.

Производительность UI: jank и бюджет кадра

Чтобы анимация была плавной при 60 Гц, каждый кадр должен быть готов за ≈16,6 мс; для 90 Гц — ~11 мс, для 120 Гц — ~8 мс. Не уложился — Choreographer пропускает кадр (dropped frame), пользователь видит рывок. Это и есть jank. Кадр, опоздавший очень сильно (> ~700 мс), — frozen frame, его отдельно ругает Android Vitals.

sequenceDiagram participant V as VSYNC participant C as Choreographer participant U as UI Thread participant R as RenderThread V->>C: сигнал кадра (каждые 16мс / 8мс) C->>U: doFrame() U->>U: input + animation + measure/layout/draw U->>R: sync дерева отрисовки R->>R: GPU rendering + buffer swap Note over U,R: уложились в бюджет → плавно, опоздали → dropped frame
Пайплайн кадра. Затык чаще всего на UI Thread (тяжёлый layout/recomposition) или из-за overdraw на GPU

Частые причины jank и фиксы

  • Overdraw — один пиксель красится несколько раз (вложенные фоны). Включи Debug GPU Overdraw: цель — минимум красного. Убирай лишние background, плоскую иерархию.
  • Работа на main thread в момент кадра — парсинг, БД, аллокации в onBindViewHolder/composable. Выноси в фон, кэшируй.
  • RecyclerView — стабильные id, DiffUtil вместо notifyDataSetChanged(), лёгкий onBindViewHolder, нет вложенных скроллов без prefetch.
  • Compose — лишние перекомпозиции: стабильные типы и key в LazyColumn, чтение состояния как можно ниже по дереву, derivedStateOf, remember тяжёлых вычислений, @Immutable/@Stable.
DiffUtil.kt
// ✅ ListAdapter сам считает дельту в фоне — нет полной перерисовки списка
class UserAdapter : ListAdapter<User, UserVH>(DIFF) {
    companion object {
        val DIFF = object : DiffUtil.ItemCallback<User>() {
            override fun areItemsTheSame(a: User, b: User) = a.id == b.id      // та же сущность?
            override fun areContentsTheSame(a: User, b: User) = a == b         // изменилось ли содержимое?
        }
    }
}
Recomposition.kt
@Composable
fun Feed(items: List<User>, onClick: (String) -> Unit) {
    LazyColumn {
        items(items, key = { it.id }) { user ->     // key стабилизирует элементы
            UserRow(user, onClick)
        }
    }
}

// дорогой расчёт не должен идти на каждой рекомпозиции
val sorted = remember(items) { items.sortedBy { it.name } }

В Layout Inspector два столбца — counts перекомпозиций и skip-ов: если у composable счётчик растёт при скролле, а данные не менялись — кандидат на стабилизацию. Подтверждай эффект на release Macrobenchmark с FrameTimingMetric.

Размер APK/AAB: R8 и shrinking

R8 — стандартный компилятор-оптимизатор Android (заменил ProGuard), работает за один проход прямо над DEX и делает четыре вещи: tree shaking (удаляет недостижимый код), optimization (инлайнинг, слияние классов, удаление мёртвых веток), obfuscation (короткие имена → меньше строковая таблица) и desugaring. Включается в release одной строкой.

app/build.gradle.kts
android {
    buildTypes {
        release {
            isMinifyEnabled = true          // R8: shrinking + obfuscation + optimization
            isShrinkResources = true        // выкинуть неиспользуемые ресурсы (только с minify)
            proguardFiles(
                getDefaultProguardFile("proguard-android-optimize.txt"),
                "proguard-rules.pro"        // keep-правила для рефлексии/Gson/JNI
            )
        }
    }
}
⚠️
Подводный камень R8. Код, к которому обращаются по имени (рефлексия, Gson/Moshi по полям, JNI, имена в манифесте), R8 может удалить или переименовать → краш только в release. Лекарство — точечные -keep-правила. Для деобфускации стектрейсов сохраняй mapping.txt и загружай его в Play Console.

Resource shrinking, App Bundle и динамические модули

После code shrinking R8/AGP удаляет неиспользуемые ресурсы (строки, drawable). Android App Bundle (.aab) — формат публикации: Play собирает оптимизированные split-APK под конкретное устройство (его плотность, ABI, язык), так что пользователь не качает чужие xxxhdpi-ассеты и нативные либы под другие ABI. Dynamic Feature Modules грузятся по требованию (Play Feature Delivery) — крупные редкие фичи не утяжеляют базовую установку.

Найти, что именно раздувает сборку, помогает APK Analyzer (Build → Analyze APK): он показывает вклад DEX, ресурсов и нативных либ и умеет делать diff двух APK — удобно ловить рост размера между релизами.

Сеть и батарея

Радиомодуль — один из главных потребителей энергии: после передачи он ещё держится в высоком энергосостоянии («tail energy»). Поэтому дешевле пачка запросов раз в N минут, чем десять частых, и обязателен кэш, чтобы не ходить в сеть зря.

  • Батчинг и дедупликация запросов; HTTP-кэш OkHttp; не поллить, если есть push.
  • WorkManager constraints — откладывать фоновую работу до Wi-Fi, зарядки, idle; система сама батчит по Doze-окнам.
  • Doze / App Standby — в режиме ожидания система душит сеть, alarm и wakelock-и; рассчитывай на отложенное выполнение, а не на «прямо сейчас в фоне».
  • Energy Profiler — ищет лишние wakelock-и, частые alarm и запросы геолокации.
SyncWork.kt
val constraints = Constraints.Builder()
    .setRequiredNetworkType(NetworkType.UNMETERED)   // только Wi-Fi
    .setRequiresCharging(true)                        // и на зарядке
    .build()

val sync = PeriodicWorkRequestBuilder<SyncWorker>(6, TimeUnit.HOURS)
    .setConstraints(constraints)                      // система сама подберёт Doze-окно
    .build()

WorkManager.getInstance(context).enqueueUniquePeriodicWork(
    "sync", ExistingPeriodicWorkPolicy.KEEP, sync
)

Частые вопросы интервьюера

Приложение долго стартует — твои действия по шагам? Middle+

1) Замерить воспроизводимо: Macrobenchmark StartupTimingMetric в режиме COLD на release-сборке, плюс посмотреть Android Vitals. 2) Снять system trace в Profiler/Perfetto и разобрать критический путь: что тяжёлого в Application.onCreate, какие ContentProvider-ы инициализируются. 3) Фиксы: убрать с main thread синхронный I/O и прогрев SDK, перевести на ленивую инициализацию через App Startup, подключить Baseline Profile. 4) Перемерить тем же бенчмарком и сравнить TTID/TTFD. Ключевое — назвать cold/warm/hot и подтвердить эффект цифрой.

Что такое утечка памяти в Android и как её найти? Middle

Утечка — объект, который не нужен, но удерживается живой ссылкой от GC root, поэтому не собирается GC; на Android особенно дорого течёт Activity/View. Поиск: в debug — LeakCanary, который сам снимает heap dump и строит leak trace (кратчайший путь сильных ссылок от root к объекту); вручную — Memory Profiler, Capture heap dump, сортировка по Retained Size и поиск дублей Activity (счётчик > 1 после ротаций). Дальше по trace находим удерживающую ссылку и рвём её.

Что такое Baseline Profiles и сколько они дают? Senior

Это список горячих классов/методов критических сценариев (старт, навигация, скролл), упакованный в AAB. На установке ART делает по нему AOT-компиляцию (Profile Guided Optimization), пропуская интерпретацию и JIT для этих путей — выигрыш с первого запуска и после каждого апдейта, без изменения кода. По данным Google — порядка 30% к скорости выполнения горячего кода (у Maps ~30% к среднему старту). Генерируются Macrobenchmark-правилом BaselineProfileRule, эффект подтверждается тем же Macrobenchmark.

Чему равен бюджет на кадр и что такое jank? Middle

При 60 Гц — ~16,6 мс на кадр, при 120 Гц — ~8 мс. Если UI/Render не успели подготовить кадр к VSYNC, Choreographer его пропускает (dropped frame), пользователь видит рывок — это jank; очень опоздавший кадр (> ~700 мс) — frozen frame. Чинят выносом тяжёлой работы с main thread, снижением overdraw, оптимизацией списков (DiffUtil) и перекомпозиций в Compose. Замеры — Perfetto FrameTimeline и Macrobenchmark FrameTimingMetric.

В чём разница между Macrobenchmark и Microbenchmark? Senior

Microbenchmark меряет крошечный кусок кода (наносекунды): отдельный поток JVM, прогрев, отсев JIT, повторы — для сравнения двух реализаций алгоритма/парсера. Macrobenchmark меряет сквозные сценарии (старт, скролл) на реальном устройстве и обязательно на release/profileable-сборке, выдаёт StartupTiming/FrameTiming и трейсы. Правило: debug-замеры производительности бессмысленны, потому что debug отключает оптимизации и медленнее в разы.

Что делает R8 и какие у него подводные камни? Middle+

R8 в одном проходе делает shrinking (удаление недостижимого кода), optimization (инлайнинг, слияние), obfuscation (короткие имена) и desugaring, плюс с ним работает resource shrinking. Подводный камень — код, доступный по имени (рефлексия, Gson/Moshi, JNI, классы из манифеста): R8 может его удалить/переименовать, и краш проявится только в release. Решение — точечные -keep-правила; для читаемых стектрейсов хранить и загружать mapping.txt.

Почему bitmap так часто причина OOM и как с этим бороться? Middle+

Bitmap — это width × height × байт_на_пиксель: полноэкранная картинка ARGB_8888 — мегабайты. С Android 8 пиксели в нативной куче, но всё равно считаются памятью процесса, и пара несжатых картинок легко даёт OOM. Лечение: декодировать с inSampleSize/downscale под размер View, использовать загрузчики (Coil/Glide) с их кэшем и пулом, выбирать RGB_565 где допустимо, не держать ссылки на bitmap уничтоженных экранов. В heap dump такие байтовые массивы видны по большому Retained Size.

Зачем WorkManager constraints и причём тут Doze? Senior

Constraints (Wi-Fi, зарядка, idle, не-low-battery) позволяют отложить фоновую работу до выгодного момента и сэкономить батарею — система батчит такие задачи. Doze (когда экран выключен и устройство неподвижно) принудительно глушит сеть, alarm и wakelock-и, выпуская работу окнами обслуживания; App Standby душит редко используемые приложения. Вывод для архитектуры: фоновая синхронизация должна быть отложенной и идемпотентной, а не рассчитанной на «выполнить ровно сейчас». Для немедленной критичной работы — foreground service или expedited work.
