Executor и пулы потоков

Прямое создание потоков (new Thread) не масштабируется, поэтому работу отдают пулу. На собесе по ThreadPoolExecutor чаще всего «топят» на одном: в каком порядке пул создаёт потоки и ставит задачи в очередь. Этот порядок контринтуитивен, и именно его здесь разбираем — вместе с очередями, политиками отказа, планировщиком и тем, почему AsyncTask устарел.

Middle Senior Частый вопрос

Executor, ExecutorService и задачи

Executor — минимальный интерфейс с единственным методом execute(Runnable): он отделяет отправку задачи от того, как и на каком потоке она исполнится. ExecutorService расширяет его до полноценного фреймворка: приём Callable (задача с результатом), возврат Future, групповые операции и управляемое завершение.

ТипРоль
RunnableЗадача без результата (run()), не бросает проверяемых исключений.
Callable<V>Задача с результатом (call(): V), может бросить исключение.
Future<V>Дескриптор результата: get() (блокирует), cancel(), isDone().
Executorexecute(Runnable) — отправить задачу на исполнение.
ExecutorServicesubmit(), invokeAll(), shutdown(), awaitTermination().

val future = pool.submit(java.util.concurrent.Callable { heavyCompute() })  // Future<T>
val result = future.get()        // блокирующе ждём результат (или исключение в ExecutionException)
pool.shutdown()                  // корректное завершение

Путь исключения зависит от способа отправки: execute() кричит в лог, submit() прячет ошибку до get()
Чем поведение при исключении различается у submit() и execute(), и куда «уходит» исключение, если результат Future никто не запросил? ★

При execute(Runnable) необработанное исключение всплывает как uncaught и попадает в UncaughtExceptionHandler потока. Поток при этом завершается, а пул создаёт ему замену — то есть ошибку вы хотя бы увидите в логе.

А вот submit() оборачивает задачу в FutureTask: исключение перехватывается и сохраняется как результат. Future.get() перебросит его обёрнутым в ExecutionException (исходная причина — в getCause()). Подвох в том, что если результат submit() никто не вызовет через get(), исключение просто проглатывается — оно не всплывёт и не залогируется, баг останется невидимым. Поэтому для fire-and-forget задач берите либо execute(), либо переопределяйте afterExecute() и доставайте исключение из Future руками.

Запусти: execute() печатает «поймали uncaught», а submit() без get() молчит — исключение просто исчезло.

import java.util.concurrent.* fun main() { val pool = Executors.newFixedThreadPool(1) as ThreadPoolExecutor pool.threadFactory = ThreadFactory { r -> Thread(r).apply { setUncaughtExceptionHandler { _, e -> println("поймали uncaught: ${e.message}") } } } pool.execute { throw RuntimeException("из execute") } // всплывёт в handler потока pool.submit { throw RuntimeException("из submit") } // осядет в Future, get() не зовём pool.shutdown() pool.awaitTermination(2, TimeUnit.SECONDS) println("конец: про исключение из submit мы так и не узнали") }

ThreadPoolExecutor: параметры

Фабрики Executors — это обёртки над настраиваемым ThreadPoolExecutor. Его полный конструктор (Java SE):

ThreadPoolExecutor(
    int corePoolSize,                 // сколько потоков держать живыми, даже если простаивают
    int maximumPoolSize,              // верхний предел числа потоков
    long keepAliveTime,               // сколько лишний (сверх core) поток ждёт задачу перед смертью
    TimeUnit unit,                    // единица keepAliveTime
    BlockingQueue<Runnable> workQueue, // очередь задач до их исполнения
    ThreadFactory threadFactory,      // как создавать поток (имя, приоритет, daemon)
    RejectedExecutionHandler handler  // что делать, когда пул и очередь переполнены
)

По умолчанию (без allowCoreThreadTimeOut) keepAliveTime касается только потоков сверх corePoolSize: core-потоки живут постоянно. Если задать corePoolSize == maximumPoolSize, получится пул фиксированного размера.

corePoolSize=4, но пул простаивает — все 4 потока уже созданы и «греются»? И как заставить core-потоки умирать при простое? ★

Нет: core-потоки создаются лениво, по мере поступления задач, а не в конструкторе. Пока не пришла ни одна задача, в пуле ноль потоков. Прогреть их заранее можно вызовом prestartAllCoreThreads() (или prestartCoreThread() по одному).

По умолчанию core-потоки не завершаются по keepAliveTime — живут вечно. Чтобы они тоже умирали при простое, нужен allowCoreThreadTimeOut(true), и при этом keepAliveTime обязан быть ненулевым (иначе бросится IllegalArgumentException). Это частый приём для пулов, которые должны схлопываться до нуля потоков в idle — например, чтобы фоновый пул не держал ресурсы, пока приложение свёрнуто.

Запусти: сразу после создания пула потоков ноль, и только prestartAllCoreThreads() поднимает их заранее.

import java.util.concurrent.* fun main() { val pool = ThreadPoolExecutor( 4, 4, 0L, TimeUnit.MILLISECONDS, LinkedBlockingQueue() ) println("после конструктора: ${pool.poolSize} потоков") // 0 — core создаются лениво pool.execute { Thread.sleep(50) } println("после одной задачи: ${pool.poolSize} потоков") // 1 — поднялся ровно один pool.prestartAllCoreThreads() println("после prestart: ${pool.poolSize} потоков") // 4 — прогрели весь core pool.shutdown() }

Модель роста пула (ключевой вопрос)

Главная контринтуитивность: пул предпочитает очередь созданию нового потока. Порядок, дословно по документации Oracle, такой:

  1. Потоков меньше corePoolSize → создаётся новый поток, даже если другие простаивают.
  2. Потоков ≥ corePoolSize → задача ставится в очередь.
  3. Очередь переполнена и потоков меньше maximumPoolSize → создаётся новый (не-core) поток.
  4. Очередь переполнена и достигнут maximumPoolSize → задача отклоняется (RejectedExecutionHandler).
flowchart TB S["execute(task)"] --> A{"Меньше core потоков?"} A -->|да| T1["Создать поток"] A -->|нет| B{"Очередь не полна?"} B -->|да| Q["Поставить в очередь"] B -->|нет| C{"Меньше max потоков?"} C -->|да| T2["Создать поток"] C -->|нет| R["Отклонить (handler)"]
Порядок принятия задачи: сначала core-потоки, затем очередь, затем рост до max, и лишь потом отказ
⚠️
Следствие, на котором ловят. Новый поток сверх corePoolSize создаётся только когда очередь полна. С неограниченной очередью шаг 3 не наступает никогда — значит maximumPoolSize просто игнорируется, и пул не вырастет за пределы core, сколько бы задач ни пришло.
Технически как именно пул решает «очередь полна»? Что вызывается на BlockingQueue и почему SynchronousQueue всегда «полна»? ★

На шаге постановки в очередь пул вызывает workQueue.offer(task) — неблокирующий вариант, который возвращает false, если поставить нельзя. Именно false (а не блокировка) трактуется как «очередь полна» и запускает попытку создать не-core поток.

У SynchronousQueue ёмкость нулевая: offer() возвращает true только если прямо сейчас есть поток, уже ждущий в take(); иначе сразу false. Поэтому в cached-пуле каждая задача либо мгновенно подхватывается простаивающим потоком, либо offer падает и создаётся новый поток — отсюда и «взрывной» рост. У неограниченной LinkedBlockingQueue offer() не возвращает false никогда, поэтому ветка роста до maximumPoolSize недостижима — это и есть механика, объясняющая предыдущий callout.

Запусти: на пустом SynchronousQueue offer() возвращает false (некому передать), а у неограниченной LinkedBlockingQueue — всегда true.

import java.util.concurrent.* fun main() { val sync = SynchronousQueue<Int>() // никто не ждёт в take() — передать некому, значит «полна» println("SynchronousQueue.offer = ${sync.offer(1)}") // false → пул создаст новый поток val linked = LinkedBlockingQueue<Int>() println("LinkedBlockingQueue.offer = ${linked.offer(1)}") // true → задача просто легла в очередь println("в очереди лежит: ${linked.size}") // 1 }

Очередь определяет поведение пула

Поскольку рост потоков завязан на «полноту очереди», именно тип BlockingQueue задаёт характер пула.

ОчередьПоведение пулаРиск
SynchronousQueue (прямая передача)Очередь не хранит задачи — каждая требует свободный или новый поток. Это режим newCachedThreadPool.Неограниченный рост числа потоков, если задачи приходят быстрее, чем исполняются.
LinkedBlockingQueue без ёмкости (неограниченная)Задачи копятся в очереди; потоков не больше corePoolSize (max не действует). Это режим newFixedThreadPool.Неограниченный рост очереди под перегрузкой → OutOfMemoryError.
ArrayBlockingQueue (ограниченная)И очередь, и число потоков ограничены — защита от исчерпания ресурсов.Сложнее тюнить: размер очереди и max приходится подбирать вместе.
PriorityBlockingQueue в ThreadPoolExecutor: даст ли это приоритетное исполнение, и какие два подвоха тут вылезают? ★

Формально да — задачи будут выниматься из очереди по приоритету (через compareTo), но с оговорками.

Во-первых, PriorityBlockingQueue неограниченна, поэтому пул, как и с LinkedBlockingQueue, никогда не вырастет за corePoolSize и может копить бэклог до OOM. Во-вторых, submit() оборачивает задачу в FutureTask, который не реализует Comparable, — очередь бросит ClassCastException. Нужно либо подавать через execute() свои Comparable-Runnable, либо переопределять newTaskFor(), чтобы FutureTask нёс приоритет. И помните: приоритет действует только на ожидающие в очереди задачи — уже запущенную он не вытеснит.

Запусти: положить «свой» Comparable-объект в PriorityBlockingQueue можно, а вот FutureTask от submit() роняет очередь с ClassCastException.

import java.util.concurrent.* fun main() { val pq = PriorityBlockingQueue<Any>() pq.offer(5) // Int — Comparable, всё хорошо println("положили Int: ok, голова = ${pq.peek()}") val ft = FutureTask(Callable { 42 }) // ровно то, во что submit() оборачивает задачу try { pq.offer(ft) // очередь попытается сравнить элементы через compareTo } catch (e: ClassCastException) { println("FutureTask в PriorityBlockingQueue: ${e.javaClass.simpleName} — он не Comparable") } }

Фабрики Executors и их ловушки

Executors даёт готовые конфигурации. Удобно, но у «школьных» фабрик есть документированные подводные камни — поэтому в проде чаще конфигурируют ThreadPoolExecutor вручную с ограниченной очередью.

Про виртуальные потоки. Java 21 добавила newVirtualThreadPerTaskExecutor() (Project Loom). Но это фича JVM; среда исполнения Android (ART) виртуальные потоки не предоставляет, поэтому в Android-проекте на неё не рассчитывают — лёгкую конкурентность там дают корутины.
Сколько потоков ставить в пул: какая формула для CPU-bound и для IO-bound задач, и откуда в ней берётся коэффициент? ★

Для чисто CPU-bound работы оптимум — около числа доступных ядер (Runtime.getRuntime().availableProcessors(), иногда +1, чтобы перекрыть редкие page-fault'ы). Больше потоков лишь добавят накладных расходов на переключение контекста, не ускорив счёт.

Для IO-bound задач потоков нужно заметно больше: большую часть времени поток блокирован в ожидании сети или диска и не грузит CPU. Классическая оценка (Brian Goetz, «Java Concurrency in Practice»): N = ядра × (1 + время_ожидания / время_счёта). При соотношении ожидание/счёт = 9 на 8 ядрах получится около 80 потоков. На Android это и есть аргумент в пользу корутин: они мультиплексируют тысячи suspend-задач на маленький пул вместо того, чтобы плодить дорогие ОС-потоки. Подробнее про формулу — в параграфе «Сколько потоков заводить».

Политики отказа

Когда задачу нельзя ни поставить в очередь, ни запустить (достигнут maximumPoolSize при полной ограниченной очереди), срабатывает RejectedExecutionHandler. Четыре готовые политики:

ПолитикаЧто делает
AbortPolicy (по умолчанию)Бросает RejectedExecutionException.
CallerRunsPolicyЗадачу выполняет вызывающий поток — естественный «тормоз» темпа отправки.
DiscardPolicyМолча отбрасывает задачу.
DiscardOldestPolicyВыбрасывает самую старую задачу из очереди и повторяет попытку.

CallerRunsPolicy — частый ответ на «как сделать мягкий backpressure без OOM»: когда пул перегружен, поток-отправитель сам исполняет задачу и временно перестаёт принимать новые.

У CallerRunsPolicy есть неочевидный побочный эффект помимо «замедления отправителя». Какой, и где он стреляет? ★

Пока поток-отправитель сам выполняет отклонённую задачу, он не может подавать новые задачи в пул — отсюда и backpressure. Но если отправитель один-единственный (цикл, читающий из сокета, или Main/UI-поток), он на время задачи перестаёт делать свою основную работу. На UI-потоке это прямой ANR или фриз — поэтому отдавать CallerRunsPolicy пулу, в который шлёт задачи главный поток, опасно.

Второй подвох: CallerRunsPolicy выполнит задачу даже после shutdown() пула (проверка идёт в самом handler). А DiscardOldestPolicy при работе с очередью-приоритетом выбросит не «самую старую по времени», а голову очереди — для PriorityQueue это самый приоритетный элемент, что обычно противоположно ожиданию.

Запусти: при перегрузке часть задач исполняется на main — это и есть отправитель, делающий чужую работу вместо подачи новых.

import java.util.concurrent.* fun main() { val pool = ThreadPoolExecutor( 1, 1, 0L, TimeUnit.MILLISECONDS, ArrayBlockingQueue(1), // тесная очередь: переполнится мгновенно ThreadPoolExecutor.CallerRunsPolicy() ) repeat(6) { pool.execute { Thread.sleep(20) println("задача $it на потоке ${Thread.currentThread().name}") } } pool.shutdown() pool.awaitTermination(2, TimeUnit.SECONDS) // часть строк будет «на потоке main» — отправитель сам выполнил отклонённые задачи }

ScheduledExecutorService: отложенное и периодическое

ScheduledExecutorService добавляет отложенный запуск и периодику. Два периодических метода различаются точкой отсчёта интервала:

МетодКогда стартует следующий запуск
scheduleAtFixedRateПо «сетке»: initialDelay, затем +period, +2·period… — интервал от начала предыдущего запуска. Если задача дольше периода, запуски не накладываются, а идут подряд.
scheduleWithFixedDelayЧерез фиксированную паузу после завершения предыдущего запуска. Длительность задачи сдвигает следующий старт.
 
val ses = java.util.concurrent.Executors.newScheduledThreadPool(1)
// каждые 10 c от НАЧАЛА запуска:
ses.scheduleAtFixedRate({ poll() }, 0, 10, java.util.concurrent.TimeUnit.SECONDS)
// 10 c паузы ПОСЛЕ завершения предыдущего:
ses.scheduleWithFixedDelay({ sync() }, 0, 10, java.util.concurrent.TimeUnit.SECONDS)
⚠️
Необработанное исключение убивает периодику. Если периодическая задача бросит исключение наружу, последующие запуски подавляются (молча перестают выполняться). Поэтому тело периодической задачи оборачивают в try/catch, иначе «таймер тихо умрёт».
Задача длиннее периода: у fixedRate запуски жмутся к временной сетке впритык, у fixedDelay между ними всегда выдержана пауза delay
Создали newScheduledThreadPool(1) с одной разовой задачей через 10 минут. Можно ли получить эту задачу обратно из shutdownNow() прямо сейчас, и что внутри хранит этот пул? ★

Да: внутри ScheduledThreadPoolExecutor использует DelayedWorkQueue — специализированную неограниченную очередь-кучу, упорядоченную по времени срабатывания. Ещё не наступившая отложенная задача всё это время лежит в этой очереди (а не «спит» в потоке), поэтому shutdownNow() вернёт её в списке невыполненных Runnable.

Из-за неограниченности DelayedWorkQueue параметры maximumPoolSize и политика очереди у scheduled-пула фактически не работают — рост за core не происходит, как и с обычной неограниченной очередью. Отмена через ScheduledFuture.cancel() помечает задачу, но по умолчанию она остаётся в очереди до своего времени; убрать её сразу можно, включив setRemoveOnCancelPolicy(true).

Запусти: задача назначена на 10 минут вперёд, но shutdownNow() прямо сейчас достаёт её обратно из очереди невыполненной.

import java.util.concurrent.* fun main() { val ses = Executors.newScheduledThreadPool(1) ses.schedule({ println("сработаю не скоро") }, 10, TimeUnit.MINUTES) val pending = ses.shutdownNow() // отложенная задача всё это время лежит в DelayedWorkQueue println("вернулось невыполненных задач: ${pending.size}") // 1 — её время ещё не наступило }

Завершение пула

МетодПоведение
shutdown()Упорядоченное завершение: новые задачи не принимаются, ранее отправленные — доисполняются.
shutdownNow()Пытается прервать активные задачи (через Thread.interrupt() — best-effort) и возвращает List<Runnable> задач, которые не успели стартовать.
awaitTermination(t, unit)Блокирует до завершения всех задач после shutdown, таймаута или прерывания. true — завершился, false — истёк таймаут.
ℹ️
shutdownNow() прерывает best-effort: задача, которая не реагирует на interrupt (нет проверки isInterrupted, не ловит InterruptedException), может не остановиться. Прерывание — это кооперативный механизм, как и отмена корутин.
stateDiagram-v2 [*] --> RUNNING RUNNING --> SHUTDOWN: shutdown() RUNNING --> STOP: shutdownNow() SHUTDOWN --> STOP: shutdownNow() SHUTDOWN --> TIDYING: очередь и пул пусты STOP --> TIDYING: пул пуст TIDYING --> TERMINATED: terminated() TERMINATED --> [*] note right of TERMINATED awaitTermination() разблокируется здесь end note
Жизненный цикл пула. Без shutdown()/shutdownNow() он навсегда в RUNNING — TERMINATED не наступит, и awaitTermination провисит весь таймаут
После shutdown() пул надо «дождаться» — почему один awaitTermination() без предварительного shutdown() будет ждать вечно, и как выглядит канонический двухфазный shutdown? ★

awaitTermination() блокирует, пока пул не перейдёт в состояние TERMINATED, а туда он попадает только после shutdown()/shutdownNow() и завершения всех задач. Без вызова shutdown() пул остаётся в RUNNING и принимает новые задачи — TERMINATED не наступит никогда, поэтому awaitTermination просто провисит весь таймаут и вернёт false.

Канонический шаблон из документации:

pool.shutdown()
try {
    if (!pool.awaitTermination(60, TimeUnit.SECONDS)) {
        pool.shutdownNow()                              // дёрнуть зависшие
        pool.awaitTermination(60, TimeUnit.SECONDS)     // дождаться ещё раз
    }
} catch (e: InterruptedException) {
    pool.shutdownNow()
    Thread.currentThread().interrupt()                  // восстановить флаг
}

Запусти: без shutdown() ожидание упирается в таймаут и возвращает false; после shutdown() то же ожидание сразу даёт true.

import java.util.concurrent.* fun main() { val pool = Executors.newSingleThreadExecutor() pool.submit { /* мгновенная задача */ } // пул всё ещё в RUNNING — TERMINATED не наступит, ждём весь таймаут впустую println("без shutdown: ${pool.awaitTermination(300, TimeUnit.MILLISECONDS)}") // false pool.shutdown() println("после shutdown: ${pool.awaitTermination(1, TimeUnit.SECONDS)}") // true }

Почему AsyncTask устарел и что вместо

AsyncTask объявлен устаревшим в API 30 (Android 11). Документация прямо рекомендует вместо него «стандартные java.util.concurrent или утилиты конкурентности Kotlin» (то есть корутины). Исторические проблемы AsyncTask стали хрестоматийными:

  • Утечки памяти: нестатический внутренний класс держит ссылку на Activity; при повороте экрана задача переживает уничтожение и течёт.
  • Проблемы при смене конфигурации: результат может прийти на «мёртвую» Activity → NullPointerException или потерянный апдейт.
  • Один последовательный исполнитель по умолчанию (с Honeycomb) — задачи не параллелятся.
  • Неудобная обработка ошибок и навязанный жизненный цикл коллбэков.
🛑
Что вместо. Для in-process фоновой работы — ExecutorService или корутины (привязка к lifecycleScope/viewModelScope с авто-отменой решает и утечки, и читаемость). См. раздел «Корутины и Flow».

Граница с WorkManager

Важно не путать две задачи. Executor/корутины — это in-process работа: она умирает вместе с процессом приложения. WorkManager — для отложенной и гарантированной работы, которая обязана выполниться, даже если пользователь ушёл с экрана, приложение закрылось или устройство перезагрузилось (синхронизация, выгрузка логов, периодические задачи). Не используйте пул потоков для гарантированной фоновой работы и WorkManager — для мгновенных внутрипроцессных вычислений. Разбор инструментов фоновой работы — в главе «Фоновая работа».

Корутина на Dispatchers.IO — это «свой поток на корутину»? Что под капотом этого диспетчера и чем он отличается от голого ExecutorService? ★

Нет. Dispatchers.IO — это разделяемый пул поверх Dispatchers.Default, с лимитом по умолчанию 64 потока (или числу ядер, если их больше), причём Default и IO делят общий набор рабочих потоков. Тысячи корутин мультиплексируются на эти потоки: пока корутина приостановлена (на сетевом вызове), её поток свободен для другой корутины — поэтому корутины дешевле, чем «поток на задачу» у ExecutorService.

Любой ExecutorService можно превратить в диспетчер через asCoroutineDispatcher(), а ограниченную параллельность внутри IO задают через limitedParallelism(n). Принципиальная разница: пул потоков параллелит блокирующие задачи, а корутины ещё и кооперативно отдают поток на точках приостановки, не занимая его на время ожидания.

Запусти: тысяча корутин с delay проходит на горстке потоков IO — а не на тысяче.

import kotlinx.coroutines.* import java.util.concurrent.ConcurrentHashMap.newKeySet fun main() = runBlocking { val threads = newKeySet<String>() coroutineScope { repeat(1000) { launch(Dispatchers.IO) { delay(20) // приостановка освобождает поток threads.add(Thread.currentThread().name) } } } // 1000 suspend-задач мультиплексировались на десятки потоков, а не на 1000 println("1000 корутин выполнились на ${threads.size} потоках") }

Future: get, отмена и переход к CompletableFuture

В таблице выше Future мелькнул одной строкой, а на собесе по нему гоняют отдельно. Это дескриптор ещё не готового результата, и весь его API — про то, как этот результат забрать или отменить задачу.

get() без аргументов блокирует вызывающий поток до готовности. Что он может бросить:

Исключение из get()Когда
ExecutionExceptionЗадача упала с исключением; исходная причина — в getCause().
CancellationExceptionЗадачу отменили до или во время исполнения.
InterruptedExceptionПрервали поток, ждущий в get(); сама задача при этом продолжит работать.
TimeoutExceptionТолько у get(timeout, unit): вышел срок ожидания, задача остаётся работать.

Отмена — cancel(mayInterruptIfRunning). Если задача ещё не стартовала, её просто выкидывают из очереди. Если уже бежит: cancel(true) шлёт исполняющему потоку interrupt() (кооперативно — задача должна на него реагировать), а cancel(false) уже запущенную не трогает вовсе. После успешной отмены isCancelled() и isDone() вернут true.

Чего голому Future не хватает

Из обычного Future нельзя «подписаться» на готовность — только заблокироваться в get() или крутить isDone() в опросе. Цепочки без блокировки не собрать. Это и закрывает CompletableFuture (Java 8): неблокирующая композиция thenApply/thenCompose/thenCombine, обработка ошибок exceptionally/handle, объединение нескольких через allOf/anyOf и явный выбор executor'а на каждый шаг.

val cf = CompletableFuture
    .supplyAsync({ fetchUser(id) }, pool)         // в нашем пуле
    .thenApply { user -> user.name.uppercase() }  // неблокирующая трансформация
    .exceptionally { "anonymous" }                // фолбэк при ошибке
// результат заберём позже — get() блокирует только если уже нужен прямо сейчас

В Kotlin-коде на Android ту же роль играют корутины с suspend и async/await — поэтому CompletableFuture часто спрашивают как мост к пониманию структурированной конкурентности. Подробнее — в главе «Корутины и Flow».

Чем CompletableFuture лучше обычного Future на практике, и где он берёт потоки по умолчанию? ★

Обычный Future умеет только блокирующий get(), isDone() и cancel() — из него нельзя собрать неблокирующую цепочку. CompletableFuture добавляет композицию, обработку ошибок и объединение нескольких фьючерсов.

Тонкость: если у supplyAsync/thenApplyAsync не передать свой executor, работа по умолчанию уходит на ForkJoinPool.commonPool(). Блокирующая операция там надолго займёт общий поток и затормозит всё, что им пользуется (включая параллельные стримы), — поэтому для IO всегда передавайте собственный пул.

Запусти: без своего executor имя потока выдаёт ForkJoinPool.commonPool — тот самый общий пул.

import java.util.concurrent.* fun main() { val cf = CompletableFuture.supplyAsync { Thread.currentThread().name // где же мы выполняемся? } println(cf.get()) // ForkJoinPool.commonPool-worker-... — executor не передавали val mine = Executors.newSingleThreadExecutor { Thread(it, "мой-пул") } val cf2 = CompletableFuture.supplyAsync({ Thread.currentThread().name }, mine) println(cf2.get()) // «мой-пул» — со своим executor работа уходит туда mine.shutdown() }

Сколько потоков заводить: sizing пула

«Сколько потоков в пуле?» — один из самых частых практических вопросов, и ответ «побольше» неверный. Отправная точка — число доступных ядер: Runtime.getRuntime().availableProcessors(). Дальше всё зависит от того, чем заняты задачи.

CPU-bound

Если задача считает (парсинг, шифрование, обработка картинки) и почти не ждёт, оптимум — около числа ядер, иногда +1, чтобы перекрыть редкие page-fault'ы. Лишние потоки сверх ядер счёт не ускорят: они лишь добавят переключений контекста и борьбы за кэш.

IO-bound

Если задача большую часть времени ждёт сеть или диск, поток в это время простаивает и не грузит CPU — значит потоков нужно заметно больше ядер. Классическая оценка из «Java Concurrency in Practice» (Brian Goetz):

ℹ️
N = ядра × (1 + W/C), где W — среднее время ожидания задачи, C — время счёта. Чем больше доля ожидания, тем выше множитель. Для 8 ядер при W/C = 9 получаем 8 × 10 = 80 потоков.
val cores = Runtime.getRuntime().availableProcessors()
val cpuPool = Executors.newFixedThreadPool(cores)          // CPU-bound: ~ядра
val ioPool  = Executors.newFixedThreadPool(cores * 10)     // IO-bound: при W/C≈9

Отсюда же — мотивация корутин на Android. Раздувать пул до 80 ОС-потоков ради IO дорого: каждый поток ест память под стек и нагружает планировщик. Корутины на Dispatchers.IO мультиплексируют тысячи suspend-задач на маленький пул, отдавая поток на время ожидания, — и формула «поток на ожидающую задачу» становится не нужна.

Почему просто «поставить 200 потоков на всякий случай» — плохой ответ даже для IO-задач? ★

Каждый ОС-поток резервирует адресное пространство под стек (порядка сотен КБ), и планировщику есть что переключать. До какого-то предела рост числа потоков на IO помогает, но дальше выигрыша нет: упрёшься либо в реальный лимит внешнего ресурса (пул соединений к серверу, диск), либо в накладные расходы самого планировщика.

Поэтому число берут из ожидаемого W/C и из ёмкости ресурса, а не «с запасом». А когда ожидающих задач реально тысячи — это сигнал, что нужны не потоки, а корутины: они держат столько же конкурентных операций на пуле в десяток потоков.

Память и happens-before в пулах

Отдали задачу в пул, она что-то посчитала и записала в общий объект — увидит ли отправитель свежее значение? Без правильной синхронизации не факт: на это и ловят на Senior-уровне.

Хорошая новость в том, что java.util.concurrent даёт happens-before гарантии. Действия в потоке до submit()/execute() видны коду задачи. Действия задачи до её завершения видны потоку, который успешно дождался её через Future.get(). Поэтому передавать результат через Future безопасно — никаких volatile и synchronized вокруг него не нужно.

⚠️
Где гарантия пропадает. Если задача пишет в общий изменяемый объект, который читают мимо Future (без get() и без volatile/synchronized), happens-before не возникает. Читающий поток может увидеть устаревшее значение или вообще словить data race — баг, который воспроизводится «через раз».
var result = 0                          // обычное поле, без volatile

val f = pool.submit { result = compute() }
f.get()                                 // happens-before: запись result видна
println(result)                         // OK — увидим свежее значение

pool.submit { result = compute() }      // get() не вызываем
println(result)                         // data race: можем прочитать устаревшее
Результат я беру через Future.get(), но задача ещё и пишет в общий список параллельно с другими задачами. Этого достаточно для корректности? ★

Нет. Happens-before от get() связывает только задачу и того, кто её дождался, — он ничего не говорит про доступ между параллельными задачами к общему списку. Если несколько задач одновременно пишут в обычный ArrayList, это data race с непредсказуемым результатом вплоть до порчи внутренней структуры.

Для общей изменяемой коллекции под конкурентным доступом берут потокобезопасную структуру (ConcurrentHashMap, CopyOnWriteArrayList) или внешнюю синхронизацию. Future.get() решает только передачу результата от одной задачи к её ожидателю, не защищает разделяемое состояние.

Запусти несколько раз: обычный ArrayList под параллельной записью теряет элементы (не 4000), а CopyOnWriteArrayList всегда даёт ровно столько, сколько добавляли.

import java.util.concurrent.* fun fill(list: MutableList<Int>): Int { val pool = Executors.newFixedThreadPool(4) (1..4).map { pool.submit { repeat(1000) { list.add(it) } } }.forEach { it.get() } pool.shutdown() return list.size } fun main() { println("ArrayList: ${fill(ArrayList())}") // часто < 4000 — гонка println("CopyOnWriteArrayList: ${fill(CopyOnWriteArrayList())}") // всегда 4000 }

Тупики и голодание потоков в пуле

Пул может застрять сам из-за себя — без единого synchronized и без классического deadlock на двух локах. Имя этому — thread starvation deadlock.

Схема простая: задача в пуле блокируется в ожидании результата другой задачи, отправленной в тот же пул, а свободных потоков уже не осталось. Ждущий поток занят ожиданием, выполнить вложенную задачу некому — обе зависают навсегда.

val pool = Executors.newSingleThreadExecutor()   // ровно один поток
pool.submit {
    // этот поток занят; он отправляет вложенную задачу и ждёт её
    val inner = pool.submit(Callable { 42 })
    inner.get()                                  // зависнет навсегда: исполнить inner некому
}

Правило: не блокируйтесь в ожидании задачи того же ограниченного пула. Зависимые подзадачи гоняют через отдельный пул, либо собирают неблокирующей композицией (CompletableFuture.thenCompose), либо — на Android — структурированной конкурентностью корутин, где вложенный async не «съедает» поток на время ожидания.

⚠️
Тот же риск у common pool. parallelStream() и CompletableFuture без явного executor'а работают на общем ForkJoinPool.commonPool() (размер по умолчанию — ядра − 1). Блокирующая операция там надолго занимает дефицитный поток и тормозит всё остальное. Если без блокировки в нём никак, оборачивают её в ManagedBlocker или передают свой executor.
Почему этот тупик особенно коварен на пуле размером больше единицы — он же «иногда работает»? ★

Именно поэтому и коварен. На пуле из, скажем, 8 потоков вложенный submit().get() проходит, пока есть свободные потоки, — на тестах с малой нагрузкой всё зелёное. А под продакшен-нагрузкой, когда все 8 потоков одновременно встанут в ожидание своих вложенных задач, пул мгновенно защёлкивается в deadlock. Баг «плавающий» и привязан к нагрузке, что делает его трудновоспроизводимым.

Поэтому правило формулируют жёстко: ограниченный пул не должен исполнять задачи, которые блокируются в ожидании других задач этого же пула, — независимо от его размера.

Запусти: единственный поток занят ожиданием вложенной задачи — исполнить её некому, и get(timeout) упирается в TimeoutException вместо результата.

import java.util.concurrent.* fun main() { val pool = Executors.newSingleThreadExecutor() // ровно один поток val outer = pool.submit(Callable { val inner = pool.submit(Callable { 42 }) // встаёт в очередь — поток-то занят нами inner.get(500, TimeUnit.MILLISECONDS) // ждём задачу, которую некому выполнить }) try { println("результат: ${outer.get(1, TimeUnit.SECONDS)}") } catch (e: ExecutionException) { // внутри outer вылетел TimeoutException — классический starvation deadlock println("тупик: ${e.cause?.javaClass?.simpleName} — вложенной задаче не достался поток") } pool.shutdownNow() }

Executors в Android-стеке

На Android голый ExecutorService редко живёт сам по себе: результат фоновой работы почти всегда нужно вернуть на главный поток, а сам пул часто спрятан внутри библиотек. Где он реально встречается:

ExecutorService + Handler(mainLooper)

Каноничная замена связки doInBackground/onPostExecute из AsyncTask без всяких корутин: фоновый пул считает, Handler главного Looper возвращает результат в UI-поток.

val io = Executors.newFixedThreadPool(4)
val main = Handler(Looper.getMainLooper())

io.execute {
    val data = repository.load()        // фоновый поток
    main.post { textView.text = data }  // обратно в UI-поток
}

С API 28 для главного исполнителя есть готовый ContextCompat.getMainExecutor(context) — это Executor, постящий в главный Looper, его удобно отдавать API, которые принимают Executor.

HandlerThread и Looper

HandlerThread — поток с собственным Looper: фактически очередь-на-одном-потоке с гарантией последовательности. Удобно, когда нужен один выделенный поток под серию упорядоченных задач (например, запись в файл). Связку Looper/Handler подробно разбираем в главе «Looper и Handler».

Dispatchers как пул

Dispatchers.IO и Dispatchers.Default — это разделяемые пулы под капотом корутин. Свой ExecutorService превращается в диспетчер через asCoroutineDispatcher(), а ограничить параллельность внутри IO можно через limitedParallelism(n).

Свой Executor в библиотеках

Room принимает setQueryExecutor() и setTransactionExecutor() в билдере базы — так задаётся, на каком пуле выполняются запросы и транзакции. Retrofit умеет принимать callbackExecutor. Это места, где знание о пулах пригождается напрямую, даже если весь остальной код на корутинах.

Почему нельзя просто обновить View из фонового потока пула, и что именно проверяет Android? ★

Android-тулкит однопоточный: иерархия View не синхронизирована, и обращаться к ней разрешено только из потока, который её создал, — главного. При попытке тронуть View из чужого потока ViewRootImpl.checkThread() бросает CalledFromWrongThreadException («Only the original thread that created a view hierarchy can touch its views»).

Проверка появляется только после первого прохода отрисовки, поэтому из onCreate до показа экрана случайное обновление из фона иногда «прокатывает» — и тем коварнее. Правильный путь — вернуть результат на главный поток через Handler(Looper.getMainLooper()), getMainExecutor() или Dispatchers.Main.

Частые вопросы интервьюера

В каком порядке ThreadPoolExecutor создаёт потоки и использует очередь? Senior

Сначала, пока потоков меньше corePoolSize, на каждую задачу создаётся новый core-поток (даже если есть простаивающие). Когда core заполнен — задачи идут в очередь. Новый поток сверх core создаётся только если очередь переполнена и не достигнут maximumPoolSize. Если и очередь полна, и достигнут max — задача отклоняется через RejectedExecutionHandler. Главное: пул предпочитает очередь росту потоков.

Почему у newFixedThreadPool игнорируется maximumPoolSize? Senior

Потому что он использует неограниченную LinkedBlockingQueue. Новый не-core поток создаётся лишь когда очередь полна, а неограниченная очередь не переполняется никогда — значит шаг роста до max недостижим, и потоков всегда ровно corePoolSize. Обратная сторона — очередь может расти до OOM под перегрузкой.

Чем scheduleAtFixedRate отличается от scheduleWithFixedDelay? Middle+

scheduleAtFixedRate отсчитывает период от начала предыдущего запуска (фиксированная «сетка» времени): initialDelay, +period, +2·period… Если задача длиннее периода, запуски не накладываются, а выстраиваются подряд. scheduleWithFixedDelay выдерживает паузу после завершения задачи, поэтому её длительность сдвигает следующий старт. И помните: необработанное исключение в периодической задаче подавляет все последующие запуски.

Что вернёт shutdownNow() и гарантирует ли остановку? Middle+

Он возвращает List<Runnable> — задачи, которые ещё не начали исполняться. Активные задачи он пытается прервать через Thread.interrupt(), но это best-effort: задача, не проверяющая флаг прерывания и не ловящая InterruptedException, может не остановиться. Гарантированной «жёсткой» остановки потока в Java нет — прерывание кооперативно.

Какую политику отказа взять, чтобы не словить OOM под нагрузкой? Senior

Сначала — ограниченную очередь (а не неограниченную), иначе отказа не наступит вовсе и память переполнится. Затем, для мягкого backpressure, — CallerRunsPolicy: при переполнении задачу исполняет поток-отправитель, что замедляет темп подачи и даёт пулу разгрести бэклог. AbortPolicy (по умолчанию) бросает исключение — подходит, если перегрузку нужно явно заметить.

Почему AsyncTask устарел и что вместо него? Middle

Устарел в API 30 (Android 11): утечки через ссылку на Activity, проблемы при смене конфигурации, неудобная обработка ошибок, последовательный исполнитель по умолчанию. Документация рекомендует java.util.concurrent или корутины. Для гарантированной фоновой работы, переживающей закрытие приложения, — отдельный инструмент, WorkManager; пул потоков и корутины живут лишь в пределах процесса.

Что произойдёт, если вызвать Future.get() на ещё выполняющейся задаче, и можно ли его прервать? Middle+

get() без аргументов блокирует вызывающий поток до готовности результата. Если задача упадёт с исключением — перебросит его в ExecutionException (причина в getCause()); если её отменили — бросит CancellationException. Если прервут (interrupt) сам поток, ждущий в get(), — бросится InterruptedException, при этом задача в пуле продолжит выполняться. Чтобы не виснуть навсегда, берут get(timeout, unit): по истечении срока он бросит TimeoutException, оставив задачу работать.

Что такое ForkJoinPool и common pool, и почему важно не блокировать его потоки? Senior

ForkJoinPool — пул с work-stealing: каждый поток держит свою деку задач, а простаивающий «крадёт» задачи у загруженного, что хорошо для рекурсивных divide-and-conquer алгоритмов (fork/join). У него есть общий статический ForkJoinPool.commonPool() (размер по умолчанию — ядра − 1), на котором по умолчанию работают параллельные стримы (.parallelStream()) и CompletableFuture без явного executor'а. Опасность: блокирующая задача (IO, sleep, get) надолго занимает поток common pool и тормозит всё, что им пользуется. Поэтому для блокирующих операций передают свой executor или применяют ManagedBlocker.

Почему создавать new Thread() на каждую задачу — плохо, и что именно это съедает? Middle

Каждый ОС-поток дорог: под стек резервируется порядка сотен КБ адресного пространства, плюс затраты ядра на создание, планирование и переключение контекста. При тысячах коротких задач время на создание потока превышает саму работу, а память на стеки приводит к OutOfMemoryError («unable to create new native thread»). Пул решает это переиспользованием фиксированного набора потоков и очередью задач. Это же — мотивация корутин на Android: тысячи suspend-задач живут на маленьком пуле, не выделяя поток на каждую.

Задачи

Задача 1. Реализуйте пул потоков, который принимает не более 100 задач в очередь, запускает до 4 потоков, а при перегрузке замедляет отправителя вместо выброса исключения. Продемонстрируйте конфигурацию и объясните, почему newFixedThreadPool(4) здесь не подходит.

Показать решение 
import java.util.concurrent.*

fun buildBoundedPool(): ExecutorService =
    ThreadPoolExecutor(
        2,                                        // corePoolSize: 2 постоянных потока
        4,                                        // maximumPoolSize: до 4 при перегрузке
        60L, TimeUnit.SECONDS,                    // keepAlive для не-core потоков
        ArrayBlockingQueue(100),                  // ограниченная очередь: ключевой параметр
        Executors.defaultThreadFactory(),
        ThreadPoolExecutor.CallerRunsPolicy()     // backpressure: отправитель сам выполнит задачу
    )

fun main() {
    val pool = buildBoundedPool()
    repeat(200) { i ->
        pool.submit { Thread.sleep(50); println("task $i on ${Thread.currentThread().name}") }
    }
    pool.shutdown()
    pool.awaitTermination(30, TimeUnit.SECONDS)
}

Объяснение: newFixedThreadPool(4) использует неограниченную LinkedBlockingQueue — шаг роста до maximumPoolSize никогда не достигается, а очередь растёт в память до OOM. Здесь ArrayBlockingQueue(100) ограничивает бэклог; когда и очередь, и 4 потока заняты, CallerRunsPolicy замедляет отправителя вместо исключения. Сложность добавления задачи O(1), памяти O(queue\_size).

Задача 2. Напишите код с ScheduledExecutorService, который раз в 5 секунд опрашивает некий ресурс. Внутри периодической задачи может выброситься исключение. Покажите, как не дать таймеру «тихо умереть», и поясните разницу между scheduleAtFixedRate и scheduleWithFixedDelay для задачи длиннее интервала.

Показать решение 
import java.util.concurrent.*

fun startPolling(intervalSec: Long = 5): ScheduledFuture<*> {
    val scheduler = Executors.newScheduledThreadPool(1)

    return scheduler.scheduleAtFixedRate(
        {
            try {
                pollResource()          // может бросить исключение
            } catch (e: Exception) {
                // Без этого catch любое исключение подавит все последующие запуски
                System.err.println("poll failed: ${e.message}")
            }
        },
        0L, intervalSec, TimeUnit.SECONDS
    )
}

fun pollResource() {
    println("polling at ${System.currentTimeMillis()}")
    if (Math.random() < 0.3) throw RuntimeException("transient error")
}

Объяснение: scheduleAtFixedRate отсчитывает интервал от начала предыдущего запуска — если задача занимает 7 секунд при периоде 5 с, следующий запуск стартует сразу по завершению (накладываться не дадут). scheduleWithFixedDelay выдерживает паузу после завершения, поэтому время между запусками всегда не меньше delay. Главное: исключение, пролетевшее наружу из задачи, молча отменяет все последующие запуски — обёртка в try/catch обязательна.

Задача 3. Дан список из N URL-адресов. Скачайте содержимое каждого параллельно в пуле из 8 потоков, дождитесь всех результатов и верните список строк в исходном порядке. Обработайте ситуацию, когда один из запросов упал с исключением: остальные должны завершиться, а в результирующем списке на месте сбойного URL должна стоять строка "ERROR".

Показать решение 
import java.util.concurrent.*

fun fetchAll(urls: List<String>): List<String> {
    val pool = Executors.newFixedThreadPool(8)
    try {
        val futures: List<Future<String>> = urls.map { url ->
            pool.submit(Callable {
                fetchUrl(url)           // бросает IOException при сбое
            })
        }
        return futures.map { future ->
            try {
                future.get(10, TimeUnit.SECONDS)
            } catch (e: Exception) {
                // ExecutionException оборачивает исключение из задачи;
                // TimeoutException — таймаут ожидания
                "ERROR"
            }
        }
    } finally {
        pool.shutdown()
    }
}

fun fetchUrl(url: String): String {
    // имитация: в реальности — HttpURLConnection или OkHttp
    if (url.contains("bad")) throw java.io.IOException("unreachable: $url")
    return "content of $url"
}

fun main() {
    val urls = listOf("https://a.com", "https://bad.url", "https://b.com")
    println(fetchAll(urls))   // [content of https://a.com, ERROR, content of https://b.com]
}

Объяснение: submit(Callable) возвращает Future немедленно; задачи стартуют параллельно. future.get() блокирует до готовности и бросает ExecutionException, если задача упала — это позволяет поймать ошибку отдельно для каждого URL, не прерывая остальные. Порядок сохраняется, потому что мы итерируемся по списку futures в том же порядке, что и urls.

Задача 4. Реализуйте функцию runWithTimeout(task: () -> T, timeoutMs: Long): T?, которая выполняет блокирующую задачу в общем пуле и возвращает её результат, но если та не уложилась в timeoutMs — возвращает null и отменяет задачу, не давая ей утечь. Корректно обработайте ExecutionException, TimeoutException и InterruptedException.

Показать решение 
import java.util.concurrent.*

private val pool = Executors.newCachedThreadPool()   // общий, не закрываем на каждый вызов

fun <T> runWithTimeout(task: () -> T, timeoutMs: Long): T? {
    val future: Future<T> = pool.submit(Callable { task() })
    return try {
        future.get(timeoutMs, TimeUnit.MILLISECONDS)
    } catch (e: TimeoutException) {
        future.cancel(true)                          // шлём interrupt задаче
        null
    } catch (e: ExecutionException) {
        throw e.cause ?: e                           // разворачиваем настоящую причину
    } catch (e: InterruptedException) {
        future.cancel(true)
        Thread.currentThread().interrupt()           // восстанавливаем флаг прерывания
        null
    }
}

Объяснение: get(timeout) ждёт результат не дольше срока. По TimeoutException зовём cancel(true) — но это лишь посылает interrupt, реально остановится только кооперативная задача, проверяющая флаг. ExecutionException оборачивает исключение из задачи, поэтому пробрасываем cause. При InterruptedException (прервали нас самих) отменяем задачу и восстанавливаем флаг прерывания, чтобы не «съесть» сигнал. Пул общий и между вызовами не закрывается; накладные на задачу — O(1).

Задача 5. Напишите простой пул потоков с нуля (без ThreadPoolExecutor): N worker-потоков, потокобезопасная очередь, метод submit(Runnable) и graceful shutdown(), дожидающийся доисполнения уже принятых задач. Объясните, как воркеры засыпают на пустой очереди без busy-wait.

Показать решение 
import java.util.concurrent.*
import java.util.concurrent.atomic.AtomicBoolean

class MiniPool(n: Int) {
    private val queue = LinkedBlockingQueue<Runnable>()
    private val running = AtomicBoolean(true)
    private val workers = (1..n).map { id ->
        Thread({
            while (running.get() || queue.isNotEmpty()) {
                // poll с таймаутом: блокирует на пустой очереди — никакого busy-wait,
                // но просыпается проверить флаг running при shutdown
                val task = queue.poll(200, TimeUnit.MILLISECONDS) ?: continue
                runCatching { task.run() }   // исключение воркера не должно ронять поток
            }
        }, "mini-worker-$id").apply { start() }
    }

    fun submit(task: Runnable) {
        if (!running.get()) throw RejectedExecutionException("pool is shut down")
        queue.put(task)
    }

    fun shutdown() {                 // graceful: добираем очередь, потом выходим
        running.set(false)
        workers.forEach { it.join() }
    }
}

Объяснение: LinkedBlockingQueue.poll(timeout) блокирует поток на пустой очереди вместо холостого цикла (busy-wait), а таймаут даёт воркеру шанс периодически перечитать флаг running и завершиться. Условие running || queue.isNotEmpty() обеспечивает graceful shutdown: после shutdown() воркеры доедают остаток очереди и только потом выходят, а join() дожидается каждого. Альтернатива таймауту — «отравленная» задача (poison pill): кладём в очередь специальный маркер, увидев который воркер завершается. Сама BlockingQueue даёт happens-before между put и poll, поэтому задача видит данные, записанные до постановки в очередь.
