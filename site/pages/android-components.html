Платформа Android

Android-компоненты и межпроцессное взаимодействие

Четыре базовых компонента Android-приложения: Activity (экран/точка входа), Service (служба для фоновой работы), BroadcastReceiver (приёмник системных и приложенческих событий), ContentProvider (поставщик данных), их жизненные циклы, ограничения фоновой работы в новых версиях ОС (операционной системы) и то, как процессы общаются через Binder и AIDL (Android Interface Definition Language — язык описания интерфейса для IPC, межпроцессного взаимодействия). Тема, по которой видно, понимаешь ли ты платформу «под капотом», а не только Jetpack Compose (декларативный инструментарий пользовательского интерфейса) поверх неё.

Middle Senior Частый вопрос

Activity: жизненный цикл и состояние

Activity проходит через колбэки, парные по смыслу: onCreate/onDestroy (всё время жизни), onStart/onStop (видимость), onResume/onPause (фокус, foreground). Считывать эти пары — половина успеха: ресурсы, требующие парного освобождения, привязывают к симметричным колбэкам.

Жизненный цикл Activity. После onPause экран ещё виден; после onStop — уже нет

onSaveInstanceState и восстановление

Система может убить процесс в фоне ради памяти. Чтобы UI восстановился «как был», непосредственно перед потенциальным уничтожением (после onStop, до Android P — после onPause) вызывается onSaveInstanceState(Bundle). Bundle сериализуется и переживает смерть процесса. Ограничения: только небольшие данные (он маршалится через Binder, лимит ~1 МБ на всю транзакцию), типы должны быть Parcelable/примитивы.

MainActivity.kt
class MainActivity : ComponentActivity() {
    private var counter = 0

    override fun onCreate(savedInstanceState: Bundle?) {
        super.onCreate(savedInstanceState)
        // восстановление транзиентного UI-состояния
        counter = savedInstanceState?.getInt(KEY_COUNTER) ?: 0
    }

    override fun onSaveInstanceState(outState: Bundle) {
        super.onSaveInstanceState(outState)
        outState.putInt(KEY_COUNTER, counter)   // только маленькие данные!
    }

    companion object { private const val KEY_COUNTER = "counter" }
}
💡
Три уровня живучести состояния. ViewModel переживает поворот, но не смерть процесса. onSaveInstanceState/SavedStateHandle переживает смерть процесса, но мало места. Постоянное хранилище (Room/DataStore) — для всего остального. На собесе различай эти три уровня чётко.

Configuration changes

Поворот экрана, смена языка, темы, размера окна — это configuration change. По умолчанию система уничтожает и пересоздаёт Activity, чтобы перечитать ресурсы под новую конфигурацию. Можно перехватить вручную через android:configChanges + onConfigurationChanged(), но это считается «костылём» — теряется автоматическая подгрузка ресурсов. Правильный путь — поднимать состояние в ViewModel, а UI делать пересоздаваемым (Compose это и делает).

Launch modes и back stack

Launch mode определяет, как новая Activity ложится в task (стек задач). Управляется атрибутом android:launchMode либо флагами Intent (FLAG_ACTIVITY_*).

⚠️
Грабли singleTask/singleInstance. Они меняют поведение кнопки «Назад» и могут «вынести» Activity в чужой/отдельный task, ломая ожидаемую навигацию. На практике 95% экранов — standard/singleTop. Если интервьюер спрашивает про singleInstance «по жизни» — почти всегда правильный ответ «избегаю, кроме особых случаев».

Intent: explicit и implicit

Explicit — указан конкретный класс-получатель (внутри своего приложения). Implicit — описано действие (action + category + data), а система через intent-filter находит подходящий компонент (часто в другом приложении).

Intent.kt
// Explicit — знаем класс
startActivity(Intent(this, DetailActivity::class.java).apply {
    putExtra("id", 42)
})

// Implicit — описываем намерение, система выбирает приложение
val share = Intent(Intent.ACTION_SEND).apply {
    type = "text/plain"
    putExtra(Intent.EXTRA_TEXT, "Привет с собеса")
}
// безопасно: проверка, что есть кому обработать
if (share.resolveActivity(packageManager) != null) startActivity(share)

ActivityResult API

Старый startActivityForResult()/onActivityResult() устарел: ручной разбор requestCode, потеря результата при смерти процесса. Современный ActivityResult API регистрирует контракт заранее, лайфсайкл-безопасен и типизирован.

PickImage.kt
// регистрируем ДО onCreate-резолва, переживает пересоздание Activity
private val pickImage = registerForActivityResult(
    ActivityResultContracts.GetContent()
) { uri: Uri? ->
    uri?.let { viewModel.onImagePicked(it) }
}

// запуск без requestCode-магии
fun chooseImage() = pickImage.launch("image/*")
Почему onSaveInstanceState вызывается не всегда, когда Bundle НЕ переживёт ситуацию, и как менялся его порядок относительно onStop? ★

Он срабатывает только когда система сама может уничтожить Activity ради последующего восстановления: при configuration change или когда процесс убивают в фоне ради памяти. При явном finish() или нажатии Back его не зовут — экран уходит навсегда, восстанавливать нечего.

Порядок относительно onStop сменился: до Android P (API 28) onSaveInstanceState шёл перед onStop, с Android P — после. Это и разрешило безопасно коммитить FragmentTransaction внутри него (раньше — IllegalStateException).

Сам Bundle лежит не на диске, а в памяти системного процесса (ActivityManagerService). Поэтому он переживает смерть вашего процесса, но не перезагрузку устройства и не свайп приложения из recents. И всё это маршалится через Binder — отсюда тот же лимит транзакции ~1 МБ и TransactionTooLargeException на тяжёлом Bundle.

Что переживает поворот, а что — смерть процесса

На собесе это валит даже опытных: «ViewModel переживает process death» — миф. Разнести два события и три механизма хранения помогает одна матрица. Строки — где вы держите состояние, столбцы — какое событие проверяем.

МеханизмConfiguration change (поворот)Process death (kill в фоне)Свайп из recentsОбъём
ViewModelпереживаеттеряетсятеряетсялюбой (в памяти)
onSaveInstanceState / SavedStateHandleпереживаетпереживаеттеряется~1 МБ, Parcelable
Room / DataStoreпереживаетпереживаетпереживаетне ограничен
Поворот переживают все три; process death — только сериализованное; свайп из recents — только постоянное хранилище
⚠️
Как воспроизвести process death. Поворотом его не проверишь — процесс жив, и баг прячется. Включи Developer Options → «Don't keep activities» или убей процесс через adb shell am kill <package>, свернув приложение. Тогда система пересоздаёт процесс и проходит путь с восстановлением Bundle. Типичная расплата за неучтённый process death: пустой экран после возврата или потерянный введённый текст.

SavedStateHandle внутри ViewModel пишет в тот же системный Bundle, что и onSaveInstanceState — то есть это и есть способ дать ViewModel пережить process death, не выходя из ViewModel. Кладите туда только маленькие ключи (выбранный id, поисковый запрос), а не весь экран.

Если SavedStateHandle сериализуется в тот же Bundle, что и onSaveInstanceState, можно ли положить туда StateFlow со списком на тысячи элементов? ★

Нет. SavedStateHandle переживает process death именно потому, что его содержимое уходит в системный Bundle, а тот маршалится через Binder с лимитом транзакции ~1 МБ. Тяжёлый список переполнит буфер и даст TransactionTooLargeException — причём не там, где вы пишете, а позже, когда система решит сохранить состояние.

Граница ответственности простая: в SavedStateHandle — идентификаторы и параметры экрана (что выбрано, какой фильтр), а сами данные восстанавливаются из Room/сети по этим ключам. Это ровно тот случай «передавай id, а не данные», что и в IPC.

Fragment: lifecycle и viewLifecycleOwner

Fragment — переиспользуемая часть UI внутри Activity. Ключевое отличие от Activity: у фрагмента два разных жизненных цикла — самого фрагмента и его View. View может уничтожаться и пересоздаваться (например, при уходе в back stack), пока сам фрагмент жив.

Этап FragmentЭтап ViewЧто делаем
onAttach → onCreate—Инициализация, аргументы
onCreateViewView созданНадуть/получить layout
onViewCreatedviewLifecycleOwner активенПодписки на UI, биндинг
onDestroyViewView уничтоженОбнулить binding, отписки
onDestroy → onDetach—Финальная очистка
🛑
Почему viewLifecycleOwner критичен. Если подписаться на Flow/LiveData, указав this (lifecycle фрагмента) вместо viewLifecycleOwner, подписка переживёт уничтожение View. После onDestroyView придёт новое значение — и колбэк обратится к уже мёртвой View (или старому binding) → NullPointerException или утечка. Правило: всё, что трогает View, подписывай на viewLifecycleOwner.
ProfileFragment.kt
override fun onViewCreated(view: View, savedInstanceState: Bundle?) {
    super.onViewCreated(view, savedInstanceState)
    // привязка к жизни VIEW, а не фрагмента
    viewLifecycleOwner.lifecycleScope.launch {
        viewLifecycleOwner.repeatOnLifecycle(Lifecycle.State.STARTED) {
            viewModel.state.collect { render(it) }
        }
    }
}

FragmentManager и Fragment Result API

FragmentManager управляет транзакциями (add/replace/remove) и back stack фрагментов. Для передачи данных между фрагментами без прямых ссылок (которые ведут к связности и крашам) используют Fragment Result API: один фрагмент кладёт результат в менеджер, другой подписывается по строковому ключу.

FragmentResult.kt
// Фрагмент-получатель: слушатель привязан к viewLifecycleOwner
setFragmentResultListener("filterKey") { _, bundle ->
    val sort = bundle.getString("sort")
    viewModel.applySort(sort)
}

// Фрагмент-отправитель (например, BottomSheet)
setFragmentResult("filterKey", bundleOf("sort" to "price_asc"))
Почему getViewLifecycleOwner() бросает IllegalStateException вне окна onCreateView…onDestroyView, и чем подписка на него отличается от подписки на сам фрагмент при back stack? ★

viewLifecycleOwner существует только пока у фрагмента есть View — между onCreateView и onDestroyView. Обратитесь к нему в onCreate или после onDestroyView — получите IllegalStateException: Can't access the Fragment View's LifecycleOwner when getView() is null.

Когда фрагмент уходит в back stack через replace(), сам он остаётся в состоянии CREATED, а его View уничтожается. viewLifecycleOwner переходит в DESTROYED, и его наблюдатели снимаются. Наблюдатели, привязанные к самому фрагменту (this), остаются живыми. При возврате View пересоздаётся — и viewLifecycleOwner уже новый объект.

Отсюда правило: подписки на View ставим в onViewCreated на текущий viewLifecycleOwner. Подпишете на this — при возврате из back stack получите вторую, дублирующую подписку, и старая будет дёргать уже мёртвый binding.

Service: started, foreground и bound

Service — компонент без UI для долгой работы. Важно: Service по умолчанию работает в главном потоке — это не «фоновый поток», а лишь компонент без интерфейса. Тяжёлую работу всё равно уносят в корутины/поток.

Foreground service и типы в Android 14

Если задача должна выполняться, пока пользователь явно видит её (плеер, навигация, запись), используют foreground service с постоянным уведомлением. Начиная с Android 14 (API 34), для каждого FGS обязательно объявить android:foregroundServiceType в манифесте и запросить соответствующее разрешение FOREGROUND_SERVICE_*. Запуск без подходящего типа кидает SecurityException/MissingForegroundServiceTypeException.

TrackingService.kt
class TrackingService : Service() {
    override fun onStartCommand(intent: Intent?, flags: Int, startId: Int): Int {
        val notification = buildNotification()
        // на Android 10+ передаём тип явно при старте
        if (Build.VERSION.SDK_INT >= Build.VERSION_CODES.Q) {
            startForeground(ID, notification, FOREGROUND_SERVICE_TYPE_LOCATION)
        } else {
            startForeground(ID, notification)
        }
        return START_STICKY  // система перезапустит при убийстве
    }
    override fun onBind(intent: Intent?): IBinder? = null
}

Ограничения фоновой работы

С Android 8 (Oreo) приложение в фоне не может свободно стартовать обычные сервисы (Background Execution Limits): попытка startService() из фона кидает IllegalStateException. С Android 12 запрещён и старт самого FGS из фона (за исключениями). Поэтому «просто запустить сервис, когда захочу» больше не работает.

💡
WorkManager — современная замена. Для гарантированной, отложенной фоновой работы (синхронизация, выгрузка логов, ретраи) сегодня используют WorkManager: он переживает перезагрузку, уважает Doze/ограничения, выбирает движок (JobScheduler) под версию ОС и поддерживает constraints (сеть/зарядка). FGS — только для работы, которую прямо сейчас видит пользователь.
SyncWorker.kt
class SyncWorker(ctx: Context, params: WorkerParameters) : CoroutineWorker(ctx, params) {
    override suspend fun doWork(): Result = try {
        repository.sync()
        Result.success()
    } catch (e: IOException) {
        Result.retry()   // WorkManager сам применит backoff
    }
}

// запуск с условиями
val req = OneTimeWorkRequestBuilder<SyncWorker>()
    .setConstraints(Constraints(requiredNetworkType = NetworkType.CONNECTED))
    .build()
WorkManager.getInstance(context).enqueue(req)

Doze и App Standby

Doze включается, когда устройство долго лежит без движения с выключенным экраном: система «замораживает» сетевой доступ, откладывает джобы/алармы и собирает их в редкие «окна обслуживания». App Standby ограничивает приложения, которыми давно не пользовались (бакеты active/working set/frequent/rare/restricted). Вывод для кандидата: нельзя рассчитывать на точное время фоновых задач — используй WorkManager, а для критичной точности по времени — setExactAndAllowWhileIdle() с разрешением (и осторожно).

Глубже про WorkManager, FGS-типы и точные алармы — в отдельной главе Фоновая работа; здесь — только то, что нужно для разговора о компонентах.

Чем START_STICKY, START_NOT_STICKY и START_REDELIVER_INTENT отличаются по тому, что прилетит в onStartCommand после убийства сервиса, и почему START_STICKY коварен? ★

Возврат onStartCommand говорит системе, как поступить, если она убила сервис и появилась возможность перезапустить.

  • START_STICKY — сервис пересоздаётся, но intent == null (последний Intent заново не передаётся). Годится для плеера, который сам держит состояние; коварство в том, что про проверку intent на null забывают и ловят NPE.
  • START_NOT_STICKY — система не перезапускает сервис, пока не придёт новый startService. Для работы, которую бессмысленно возобновлять.
  • START_REDELIVER_INTENT — сервис пересоздаётся и получает последний Intent заново. Для работы, которую обязательно доделать (докачка файла).

Отдельно: с Android 8 убитый в фоне START_STICKY-сервис может вообще не перезапуститься из-за Background Execution Limits. Нужны гарантии — это WorkManager, а не надежда на sticky.

WorkManager под капотом

В callout выше WorkManager назван «современной заменой сервисам» — но на собесе про фоновую работу спрашивают глубже, чем «есть такой класс». Разберём, как он даёт гарантии и чем управляется. Полная глава — Фоновая работа; здесь — суть для вопросов про компоненты.

Где он хранит работу и почему она переживает ребут

Запросы лежат в собственной SQLite-базе (через Room), а не в памяти процесса. Поэтому работа переживает перезапуск приложения и перезагрузку устройства: после ребута её восстанавливает RescheduleReceiver, ловящий BOOT_COMPLETED. Под капотом на разных версиях WorkManager выбирает движок — JobScheduler (API 23+) либо, для старых, связку AlarmManager + BroadcastReceiver.

Уникальная работа и политики конфликта

Чтобы не плодить дубли синхронизации, работу ставят уникальной — через enqueueUniqueWork / enqueueUniquePeriodicWork с именем и политикой:

ПолитикаЧто делает при конфликте имёнКогда
REPLACEОтменить существующую, поставить новуюАктуальна только последняя задача
KEEPОставить существующую, новую проигнорироватьХватит одной запущенной
APPEND / APPEND_OR_REPLACEДобавить в конец цепочкиЗадачи должны идти по очереди

Цепочки, expedited, constraints

Цепочки строят через beginWith().then(); output одной Worker автоматически становится input следующей. Expedited work (setExpedited, с 2.7) — для важной, инициированной пользователем работы, которую надо начать почти сразу, но коротко. Система даёт ей повышенный приоритет в рамках квоты foreground-времени; когда квота исчерпана, запрос откатывается по политике (RUN_AS_NON_EXPEDITED_WORK_REQUEST или DROP). На API < 31 expedited реализуется через foreground service, поэтому нужен getForegroundInfo() с уведомлением; на API 31+ — через expedited job в JobScheduler без уведомления.

SyncChain.kt
val constraints = Constraints(
    requiredNetworkType = NetworkType.CONNECTED,
    requiresBatteryNotLow = true
)
val download = OneTimeWorkRequestBuilder<DownloadWorker>()
    .setConstraints(constraints)
    .setExpedited(OutOfQuotaPolicy.RUN_AS_NON_EXPEDITED_WORK_REQUEST)
    .setBackoffCriteria(BackoffPolicy.EXPONENTIAL, 30, TimeUnit.SECONDS)
    .build()
val process = OneTimeWorkRequestBuilder<ProcessWorker>().build()

WorkManager.getInstance(context)
    .beginUniqueWork("sync", ExistingWorkPolicy.KEEP, download)
    .then(process)   // output download → input process
    .enqueue()
flowchart TD A["Нужна фоновая работа"] --> B{"Пользователь видит её сейчас?"} B -->|"да: плеер, навигация"| C["Foreground service (+ тип, Android 14)"] B -->|"нет"| D{"Надо доделать гарантированно?"} D -->|"да, можно подождать"| E["WorkManager"] D -->|"да, начать почти сразу"| F["WorkManager expedited"] D -->|"точное время"| G["AlarmManager (exact)"] E --> H["constraints + backoff + unique work"] F --> H
Выбор механизма: видимость сейчас → FGS, отложенная гарантия → WorkManager, точное время → AlarmManager
⚠️
Где WorkManager не годится. Точное время срабатывания (будильник, напоминание в 9:00) — это AlarmManager с setExactAndAllowWhileIdle, а не WorkManager, который собирает работу в окна и уважает Doze. Мгновенная видимая пользователю работа — foreground service. WorkManager — про «рано или поздно, но точно».
Чем expedited work отличается от обычного запроса и от foreground service, и что произойдёт, когда исчерпается квота foreground-времени? ★

От обычного запроса expedited отличается приоритетом: он начинается почти сразу и игнорирует часть отложек, но рассчитан на короткую работу (минуты). От FGS — тем, что это всё ещё управляемая WorkManager отложенно-гарантированная работа, а не процесс, который пользователь видит и контролирует прямо сейчас.

Квота foreground-времени общая на приложение. Когда она исчерпана, новый expedited-запрос не падает молча: он откатывается по OutOfQuotaPolicy — либо выполнится как обычный неускоренный запрос (RUN_AS_NON_EXPEDITED_WORK_REQUEST), либо будет отброшен (DROP). На API < 31 расход квоты особенно заметен, потому что под капотом expedited там — это foreground service.

BroadcastReceiver

Компонент-подписчик на системные (BOOT_COMPLETED, заряд, сеть) или собственные события. Регистрируется двумя способами с принципиально разной семантикой.

Ограничение implicit broadcast (Android 8+)

Чтобы пачка приложений не «просыпалась» от каждого системного события, начиная с Android 8 нельзя регистрировать implicit броадкасты в манифесте. Исключения остались для важных событий, которые приходят редко: ACTION_BOOT_COMPLETED, ACTION_LOCALE_CHANGED, ACTION_MY_PACKAGE_REPLACED, события медиа-носителей и др. (полный список — «Implicit Broadcast Exceptions»). Для всего остального: регистрируй ресивер динамически, пока приложение активно, либо используй explicit-броадкаст с указанием пакета/компонента.

NetworkReceiver.kt
class NetworkReceiver(val onChange: (Boolean) -> Unit) : BroadcastReceiver() {
    override fun onReceive(context: Context, intent: Intent) {
        // onReceive выполняется в main-потоке и кратко: тяжёлое — в WorkManager
        onChange(isOnline(context))
    }
}

// динамическая регистрация на время видимости экрана
override fun onStart() {
    super.onStart()
    registerReceiver(receiver, IntentFilter(ConnectivityManager.CONNECTIVITY_ACTION))
}
override fun onStop() {
    super.onStop()
    unregisterReceiver(receiver)   // обязательно — иначе утечка
}
⚠️
LocalBroadcastManager — deprecated. Он опирался на Binder (механизм IPC) ради сообщений внутри одного процесса — это избыточно и поощряло нарушение слоёв («любой компонент слышит любое событие»). Современная замена для внутреннего event-bus — SharedFlow/StateFlow (или LiveData): лайфсайкл-безопасно, типизированно, без накладных расходов Binder.
Сколько времени есть у onReceive() до ANR, почему нельзя просто запустить корутину прямо в нём, и что меняет goAsync()? ★

onReceive() идёт на главном потоке, и у него жёсткий лимит ~10 секунд для foreground-броадкаста — дальше система объявляет ANR и может убить процесс.

Просто запустить корутину или Thread ненадёжно: как только onReceive вернёт управление, процесс считается неактивным и его могут убить до завершения работы, а сам объект ресивера перестаёт быть валидным. goAsync() возвращает PendingResult и говорит системе «я ещё не закончил» — процесс держат живым чуть дольше (тот же бюджет ~10 сек, не бесконечно), а по завершении обязателен pendingResult.finish().

Это для коротких операций. Для длительной работы из ресивера канонично — enqueue в WorkManager (подробнее в главе Фоновая работа), а не goAsync.

ContentProvider и Context

ContentProvider

Стандартный механизм предоставить структурированные данные другим приложениям (или процессам). Внутри одного приложения он почти не нужен — там проще Room/репозиторий. Он оправдан, когда: данные читают чужие приложения; нужно интегрироваться с системой (контакты, календарь, медиатека, FileProvider для безопасной раздачи файлов); требуется SyncAdapter/виджеты, которым нужен provider.

Доступ — через ContentResolver по URI вида content://authority/path/id. Права разграничиваются android:readPermission/writePermission и временными URI permissions (FLAG_GRANT_READ_URI_PERMISSION) — так FileProvider отдаёт один файл без выдачи доступа ко всему.

ContentResolver.kt
// чтение контактов через системный ContentProvider
val cursor = contentResolver.query(
    ContactsContract.Contacts.CONTENT_URI,
    arrayOf(ContactsContract.Contacts.DISPLAY_NAME),
    null, null, null
)
cursor?.use {
    while (it.moveToNext()) {
        val name = it.getString(0)
    }
}

// безопасная раздача своего файла другому приложению
val uri = FileProvider.getUriForFile(context, "$packageName.fileprovider", file)
val intent = Intent(Intent.ACTION_VIEW).apply {
    setDataAndType(uri, "application/pdf")
    addFlags(Intent.FLAG_GRANT_READ_URI_PERMISSION)  // только этот URI
}

Context: application vs activity и утечки

Context — «ручка» к ресурсам, ассетам, системным сервисам и запуску компонентов. Есть два важных вида: Application Context (живёт всё время процесса) и Activity Context (умирает с экраном, знает про тему/окно).

НужноКакой Context
Надуть UI, показать диалог, тема экранаActivity (у Application нет темы)
Singleton, БД, DI-граф, долгоживущий объектApplication
Старт сервиса/получение системного сервисаЛюбой (предпочти Application для долгих)
🛑
Классическая утечка. Передать Activity Context в долгоживущий объект (синглтон, статика, фоновый поток, корутина, которая переживает экран). Activity не сможет быть собрана GC, пока её держит «вечный» объект → утечка всего дерева View. Правило: для всего, что живёт дольше экрана, бери applicationContext. Для UI — Activity, но не сохраняй её ссылку надолго.
Почему ContentProvider.onCreate() вызывается раньше Application.onCreate(), как этим пользуются библиотеки и чем это бьёт по холодному старту? ★

При старте процесса система сперва создаёт Application (конструктор и attachBaseContext), затем зовёт onCreate() всех объявленных ContentProvider, и только потом — Application.onCreate(). То есть провайдеры инициализируются в окне между attach и onCreate приложения.

Этим пользовались для автоинициализации SDK без кода пользователя: библиотека объявляет служебный провайдер в манифесте (через manifest merger), и его onCreate поднимает SDK ещё до вашего Application.onCreate. Так раньше стартовали WorkManager и Firebase.

Минус — каждый такой провайдер замедляет cold start: их onCreate выполняются последовательно на главном потоке. Ради этого появился androidx.startup: один InitializationProvider агрегирует все Initializer вместо россыпи провайдеров. Отключить чужую автоинициализацию можно через tools:node="remove" для нужного провайдера в манифесте.

Безопасность компонентов

Senior-собес почти всегда трогает безопасность межкомпонентного взаимодействия, а тема легко проседает: компоненты по неосторожности экспортируют, PendingIntent отдают мутабельным. Разберём три точки, где это спрашивают.

android:exported и изменение Android 12

android:exported определяет, могут ли другие приложения запускать ваш компонент. Экспортированный компонент — это поверхность атаки: любой может прислать ему Intent. Раньше при наличии intent-filter компонент по умолчанию считался exported=true, и это часто упускали. С Android 12 (API 31) для любого компонента с intent-filter атрибут android:exported надо указывать явно, иначе приложение не соберётся. Правило: exported=false для внутренних компонентов; для exported=true — проверка прав через permission и недоверие входящему Intent.

PendingIntent и обязательный флаг изменяемости

PendingIntent — токен, который вы отдаёте чужому коду (системе, нотификации, виджету), позволяя выполнить действие от вашего имени и с вашими правами. Опасность: если он мутабельный, а обёрнутый Intent имеет пустой компонент, получатель может его переписать (Intent redirection) и заставить ваше приложение сделать нежелательное. С Android 12 (API 31) при создании PendingIntent обязательно явно указывать FLAG_IMMUTABLE или FLAG_MUTABLE — иначе IllegalArgumentException. По умолчанию берут FLAG_IMMUTABLE; FLAG_MUTABLE — только когда системе реально нужно дописать данные (inline-reply в нотификации, Bubbles).

Notify.kt
val intent = Intent(this, DetailActivity::class.java).apply {
    putExtra("id", itemId)
}
val pending = PendingIntent.getActivity(
    this, 0, intent,
    // по умолчанию — IMMUTABLE; флаг обязателен с API 31
    PendingIntent.FLAG_IMMUTABLE or PendingIntent.FLAG_UPDATE_CURRENT
)

Проверка вызывающего на стороне сервиса

Если сервис всё же экспортирован, на стороне Binder можно узнать, кто пришёл: Binder.getCallingUid() / getCallingPid() возвращают UID/PID реального вызывающего на время транзакции. По UID сверяют подпись или package и отказывают чужим. У ContentProvider та же логика — плюс временные URI permissions (FLAG_GRANT_READ_URI_PERMISSION, см. выше), чтобы отдать один файл, а не весь источник.

🛑
Экспортированный компонент без проверки. exported=true без permission и без валидации Intent означает, что любое приложение на устройстве может дёргать ваш сервис/Activity с произвольными данными. Для внутренних компонентов — всегда exported=false; для межприложенческих — protectionLevel="signature" у кастомного permission, чтобы доступ имели только ваши же подписанные приложения.
Гарантирует ли FLAG_IMMUTABLE, что получатель вообще ничего не сможет добавить к PendingIntent, и зачем тогда системе FLAG_MUTABLE? ★

Immutable значит, что обёрнутый Intent нельзя переписать через fillIn(): компонент, action, data и уже заданные extras зафиксированы. Это и закрывает Intent redirection — чужой код не подменит назначение.

Но есть сценарии, где системе нужно дописать данные в момент срабатывания: inline-reply в нотификации возвращает введённый пользователем текст через RemoteInput, Bubbles и некоторые системные UI тоже доклеивают свои extras. Для них и существует FLAG_MUTABLE — но это узкие, осознанные случаи, а дефолт остаётся immutable.

IPC: Binder под капотом

Процессы в Android изолированы (каждое приложение — свой Linux-процесс и UID). Чтобы они общались, нужен IPC. Основной механизм — Binder: это и есть транспорт почти всего системного API (вызов ActivityManager, startActivity через system_server и т.д.).

Binder реализован как драйвер ядра /dev/binder. На стороне клиента ты держишь proxy (реализует интерфейс, но внутри упаковывает вызов), на стороне сервера — stub (распаковывает и вызывает реальный метод). Клиент зовёт transact(), драйвер копирует данные в адресное пространство сервера и вызывает onTransact(). Сервер обрабатывает вызовы в пуле binder-потоков (по умолчанию до ~16), поэтому удалённые методы могут выполняться параллельно и не на main-потоке.

sequenceDiagram participant C as Клиент (Proxy) participant K as Драйвер /dev/binder participant S as Сервер (Stub) C->>C: marshalling args в Parcel C->>K: transact(code, data, reply) Note over C: вызывающий поток блокируется K->>K: копирует Parcel в адр. пр-во сервера K->>S: onTransact(code, data, reply) S->>S: распаковка + реальный метод S-->>K: пишет результат в reply K-->>C: разблокировка + reply C->>C: unmarshalling результата
Один синхронный Binder-вызов: proxy → драйвер → stub и обратно
ℹ️
Почему Binder, а не сокеты/pipe. Binder делает копирование один раз (через драйвер), знает про процессы/потоки/UID (можно проверить Binder.getCallingUid()), переносит ссылки на объекты между процессами и интегрирован с моделью безопасности Android. Это быстрее и безопаснее обычного двойного копирования через сокет.

Смерть сервера и linkToDeath

Клиент удалённого сервиса держит proxy на чужой процесс — а тот может умереть в любой момент. Чтобы узнать об этом не на следующем упавшем вызове, а сразу, регистрируют death recipient: binder.linkToDeath(recipient, 0). Колбэк DeathRecipient.binderDied() срабатывает, когда процесс-владелец Binder умер, и клиент успевает сбросить состояние и переподключиться. Без этого после смерти сервера любой transact бросит DeadObjectException (наследник RemoteException).

stateDiagram-v2 [*] --> Unbound Unbound --> Bound: bindService() Bound --> Alive: onServiceConnected() Alive --> ServerDied: binderDied() / DeadObjectException ServerDied --> Bound: переподключение (rebind) Alive --> Unbound: unbindService() note right of ServerDied сработал death recipient, proxy больше не валиден end note
Жизнь связи клиент–сервис: death recipient ловит смерть сервера и запускает reconnect
Зачем клиенту linkToDeath и что случится, если синхронный AIDL-вызов сделать с главного потока, а сервер в этот момент повиснет? ★

linkToDeath даёт death-notification: при смерти удалённого процесса вызывается binderDied(), и клиент узнаёт о разрыве до того, как упадёт на следующем transact с DeadObjectException. Это позволяет аккуратно сбросить кеш proxy и переподключиться.

Синхронный AIDL-вызов блокирует вызывающий поток до ответа сервера. Сделаете его с главного потока, а сервер зависнет — ваш UI-поток встанет, и через ~5 секунд прилетит ANR. Поэтому удалённые синхронные вызовы делают с фонового потока. Дополнительно: вложенные Binder-вызовы могут выполняться повторно-входяще (reentrant) на том же binder-потоке.

AIDL, Messenger и процессы

AIDL: интерфейс между процессами

Писать proxy/stub руками тяжело — для этого есть AIDL (Android Interface Definition Language). Описываешь интерфейс в .aidl, генератор создаёт Stub (наследуешь на сервере) и Proxy (получаешь на клиенте через asInterface()). Данные пересекают границу процесса через маршалинг в Parcel; кастомные классы должны быть Parcelable.

IRemoteService.aidl
interface IRemoteService {
    int add(int a, int b);
    // oneway — не блокирует клиента, без возврата результата ("fire and forget")
    oneway void log(String message);
}
RemoteService.kt
class RemoteService : Service() {
    // Stub сгенерирован из .aidl; работает в binder-потоке
    private val binder = object : IRemoteService.Stub() {
        override fun add(a: Int, b: Int): Int = a + b
        override fun log(message: String) { Log.d("IPC", message) }
    }
    override fun onBind(intent: Intent?): IBinder = binder
}

// клиент в другом процессе
private val connection = object : ServiceConnection {
    override fun onServiceConnected(name: ComponentName?, service: IBinder?) {
        val remote = IRemoteService.Stub.asInterface(service)  // получили proxy
        val sum = remote.add(2, 3)                             // синхронный IPC
    }
    override fun onServiceDisconnected(name: ComponentName?) {}
}

Модификатор oneway делает вызов асинхронным: клиент не ждёт ответа, метод не может возвращать значение. По умолчанию AIDL-вызовы синхронны и блокируют вызывающий поток до ответа.

Messenger: очередь поверх Binder

Если параллелизм не нужен и хватает последовательной обработки сообщений, проще Messenger: он строит очередь Message поверх одного Binder и обрабатывает их по одному в Handler. Меньше кода, чем AIDL, но нет параллельных вызовов и сложных сигнатур — только Message.

MessengerНет (очередь сообщений)Простой обмен сообщениями, не нужна многопоточность

Процессы и цена IPC

Компонент можно вынести в отдельный процесс через android:process (например, тяжёлый/падучий код изолировать от UI-процесса). Но за это платят: каждый IPC-вызов в разы дороже обычного вызова метода (маршалинг + копирование + переключение контекста), данные нужно делать Parcelable, отладка усложняется.

🛑
Лимит транзакции Binder ~1 МБ. Буфер транзакций один на процесс и невелик (порядка 1 МБ, делится между всеми параллельными транзакциями). Попытка передать большой Bundle/Parcelable (огромный список, битмап) через Intent/AIDL/onSaveInstanceState кидает TransactionTooLargeException. Лечение: не передавай большие данные через IPC — передавай идентификатор/URI, а сами данные доставай из общего хранилища (БД, файл, ContentProvider) или через shared memory.

Альтернативы Binder для IPC: ContentProvider (CRUD-данные), доменные сокеты/LocalSocket (потоковые данные, портирование «как на сервере»), файлы/MemoryFile/ashmem (большие объёмы через shared memory). Но для типичного «дай мне API другого приложения/сервиса» канонично именно AIDL поверх Binder.

flowchart TB A["Нужно IPC?"] --> B{"Что передаём?"} B --> |"Богатый API, методы"| C["AIDL (Binder)"] B --> |"Простые сообщения"| D["Messenger"] B --> |"Данные / CRUD"| E["ContentProvider"] B --> |"Большой поток байт"| F["Сокеты / shared memory"] C --> G["Parcelable, лимит ~1MB"] D --> G
Выбор механизма IPC под задачу
Гарантирует ли oneway-вызов неблокирование и порядок доставки, и почему он всё равно может бросить TransactionTooLargeException, хотя клиент «не ждёт ответа»? ★

oneway делает вызов асинхронным: клиент не блокируется и не получает возврата (метод обязан быть void). В пределах одного интерфейса oneway-вызовы от одного клиента доставляются по порядку, но между разными интерфейсами/binder-объектами порядок не гарантирован.

«Не ждёт ответа» не значит «без маршалинга». Аргументы всё равно сериализуются в Parcel и копируются в буфер транзакции получателя ещё на стороне вызова — поэтому слишком большой аргумент переполняет буфер и бросает TransactionTooLargeException синхронно у вызывающего, до того как сервер начнёт обработку. Плюс у oneway отдельный, меньший лимит буфера, так что крупные данные через него тем более не гоняют — передают URI/идентификатор.

Частые вопросы интервьюера

Что вызывается при повороте экрана и как сохранить состояние? Middle

Поворот — это configuration change: Activity по умолчанию уничтожается (onPause → onStop → onSaveInstanceState → onDestroy) и пересоздаётся (onCreate → onStart → onResume). Состояние сохраняют по трём уровням: ViewModel (переживает поворот, но не смерть процесса), onSaveInstanceState/SavedStateHandle (переживает смерть процесса, но только маленькие Parcelable), постоянное хранилище (Room/DataStore — всё остальное). Перехватывать поворот через configChanges вручную считается антипаттерном.

Зачем нужен viewLifecycleOwner во фрагменте? Middle+

У фрагмента два жизненных цикла: самого фрагмента и его View. View может пересоздаваться (back stack), пока фрагмент жив. Если подписку на Flow/LiveData, которая трогает View, привязать к lifecycle фрагмента (this), она переживёт onDestroyView и при новом эмите обратится к уничтоженной View или старому binding → NPE/утечка. Поэтому всё, что работает с View, подписываем на viewLifecycleOwner.

Чем foreground service отличается от WorkManager и что выбрать? Middle+

Foreground service — для работы, которую пользователь видит прямо сейчас (плеер, навигация, запись), с постоянным уведомлением; с Android 14 обязателен foregroundServiceType + соответствующее разрешение. WorkManager — для гарантированной отложенной работы (синхронизация, выгрузка, ретраи): переживает перезагрузку, уважает Doze и constraints, сам выбирает движок. Правило: «нужно показать прогресс/удержать систему сейчас» → FGS; «надо доделать рано или поздно, можно подождать» → WorkManager.

Почему с Android 8 нельзя ловить implicit broadcast в манифесте? Senior

Раньше системное событие (например, изменение сети) будило десятки приложений разом, расходуя батарею и память. С Android 8 манифест-регистрация implicit-броадкастов запрещена (кроме списка исключений — BOOT_COMPLETED, LOCALE_CHANGED, события медиа-носителей и т.п., которые приходят редко). Решения: регистрировать ресивер динамически, пока приложение активно; использовать explicit-броадкаст с указанием компонента; либо подписываться на изменения через WorkManager constraints или ConnectivityManager.NetworkCallback.

Что такое Binder и почему он, а не сокеты? Senior

Binder — ядро IPC в Android (драйвер /dev/binder), транспорт почти всего системного API. Клиент держит proxy, сервер — stub; вызов идёт через transact()/onTransact(), драйвер копирует Parcel в память сервера, обработка — в пуле binder-потоков (отсюда параллелизм и не-main-поток). Преимущества над сокетами: одно копирование, знание о процессах/UID (проверка прав getCallingUid()), перенос ссылок на объекты, интеграция с моделью безопасности. AIDL — это генератор proxy/stub поверх Binder.

Откуда берётся TransactionTooLargeException? Senior

Буфер транзакций Binder один на процесс и невелик (порядка 1 МБ, делится между всеми параллельными транзакциями). Если через Intent extras, AIDL-вызов или onSaveInstanceState протащить большой объект (огромный список, битмап), маршалинг переполняет буфер → исключение. Чаще всего ловится при onSaveInstanceState с тяжёлым Bundle. Лечение: не гонять большие данные через IPC — передавать идентификатор/URI, а данные доставать из общего хранилища (Room, файл, ContentProvider).

В чём разница AIDL и Messenger? Middle+

Оба работают поверх Binder. AIDL даёт полноценный типизированный интерфейс с методами и поддерживает параллельные вызовы (binder thread pool) — нужно, когда много клиентов или богатый API; требует думать о потокобезопасности. Messenger строит очередь Message поверх одного Binder и обрабатывает их последовательно в Handler — меньше кода, нет многопоточности, подходит для простого обмена сообщениями. Эвристика: нужен параллелизм/сложный API → AIDL, иначе → Messenger.

Какой Context брать и где утечки? Middle

Application Context живёт весь процесс — его берут для синглтонов, БД, DI-графа, всего, что переживает экран. Activity Context знает про тему/окно — нужен для надувания UI и диалогов, но умирает с экраном. Утечка возникает, когда Activity Context (или View/Activity) сохраняют в долгоживущем объекте (статика, синглтон, фоновый поток/корутина) — Activity не может быть собрана GC, утекает всё дерево View. Правило: дольше экрана — только applicationContext.

Чем плох LocalBroadcastManager и чем заменить? Middle+

Он deprecated: использовал Binder (механизм IPC) для сообщений внутри одного процесса — избыточно по накладным расходам и поощрял глобальный event-bus, где любой компонент слышит любое событие (нарушение слоёв, сложно отслеживать). Современная замена — реактивные потоки: SharedFlow/StateFlow в общем источнике (репозиторий/ViewModel) или LiveData. Это лайфсайкл-безопасно, типизированно и имеет понятный scope.

В чём разница между configuration change и process death, и что спасает в каждом случае? Middle+

Два разных события, которые путают. Configuration change (поворот, смена темы/языка/размера окна) — Activity пересоздаётся, но процесс жив; ViewModel переживает его (хранится в ViewModelStore через NonConfigurationInstance), onSaveInstanceState тоже срабатывает. Process death — систему убивает весь процесс в фоне ради памяти; ViewModel при этом умирает (частый миф, что переживает). Переживает только сериализованное: Bundle onSaveInstanceState или SavedStateHandle во ViewModel (он пишет в тот же Bundle). Проверять process death надо принудительно: «Don't keep activities» или adb shell am kill, а не поворотом. Большое и постоянное — всегда в Room/DataStore.

Как работает deep link и при чём тут taskAffinity / FLAG_ACTIVITY_NEW_TASK? Senior

Deep link — implicit Intent с data-URI, который intent-filter (data со scheme/host/path, action VIEW, category BROWSABLE + DEFAULT) сопоставляет с Activity. App Links — это verified deep links (через assetlinks.json и autoVerify), открывающиеся без диалога выбора. taskAffinity задаёт, к какой задаче «тянется» Activity: по умолчанию affinity = package name, поэтому все Activity в одной задаче. При запуске с FLAG_ACTIVITY_NEW_TASK система ищет существующую задачу с подходящей affinity и кладёт Activity туда (или создаёт новую), а не в задачу вызывающего. Сочетание taskAffinity + launchMode singleTask/singleInstance и даёт неожиданное поведение Back — частый источник багов навигации.

Что такое expedited work в WorkManager, чем оно отличается от обычного и от FGS? Senior

Expedited work (с WorkManager 2.7, setExpedited) — для важной, инициированной пользователем работы, которую надо начать почти сразу, но коротко (минуты). Система даёт повышенный приоритет в рамках квоты foreground-времени; при исчерпании квоты запрос откатывается по политике (RUN_AS_NON_EXPEDITED_WORK_REQUEST или DROP). На API < 31 реализуется через foreground service, поэтому нужен getForegroundInfo() с уведомлением; на API 31+ — через expedited job в JobScheduler без уведомления. От обычной работы отличается приоритетом, от FGS — тем, что это всё ещё отложенно-гарантированная работа под управлением WorkManager, а не процесс, который пользователь видит прямо сейчас.

Как WorkManager обеспечивает уникальность и гарантированность работы и где хранит её? Middle+

Уникальная работа задаётся через enqueueUniqueWork / enqueueUniquePeriodicWork с именем и политикой конфликта: REPLACE (отменить старую), KEEP (оставить существующую), APPEND / APPEND_OR_REPLACE (в конец цепочки) — это решает классические дубли синхронизации. Гарантированность: запросы лежат в собственной SQLite-базе (Room), поэтому работа переживает перезагрузку и перезапуск приложения — после ребута её восстанавливает RescheduleReceiver. Под капотом движок выбирается по API: JobScheduler (23+) или, для старых, AlarmManager + BroadcastReceiver. Output одной Worker автоматически передаётся как input следующей в цепочке (beginWith().then()).

Чем PendingIntent опасен и зачем FLAG_IMMUTABLE / FLAG_MUTABLE? Senior

PendingIntent — токен, который вы отдаёте чужому коду (системе, нотификации, виджету), позволяя выполнить действие от вашего имени и с вашими правами. Опасность: если он мутабельный и обёрнутый Intent имеет пустой компонент, получатель может его переписать (Intent redirection) и заставить ваше приложение сделать нежелательное. С Android 12 (API 31) при создании PendingIntent обязательно явно указывать FLAG_IMMUTABLE или FLAG_MUTABLE, иначе IllegalArgumentException. По умолчанию — FLAG_IMMUTABLE; FLAG_MUTABLE только когда системе реально нужно дописать данные (inline-reply, Bubbles).

Зачем android:exported и что изменилось в Android 12 для компонентов с intent-filter? Middle+

android:exported определяет, могут ли другие приложения запускать ваш компонент. Экспортированный — поверхность атаки: любое приложение может прислать ему Intent. Раньше при наличии intent-filter компонент по умолчанию считался exported=true, что часто упускали. С Android 12 (API 31) для любого компонента с intent-filter атрибут android:exported нужно указывать явно, иначе приложение не соберётся. Правило: exported=false для внутренних компонентов; для exported=true — проверка прав (permission) и недоверие входящему Intent. У ContentProvider отдельно — осторожность с grantUriPermissions.

Задачи

Задача 1. Напишите Fragment, который подписывается на Flow<List<String>> из ViewModel и обновляет RecyclerView-адаптер. При этом подписка должна автоматически останавливаться, когда View уничтожена, и возобновляться при пересоздании View (например, при возврате из back stack). Покажите корректный код в onViewCreated.

Показать решение 
class ItemsFragment : Fragment(R.layout.fragment_items) {
    private val viewModel: ItemsViewModel by viewModels()
    private var binding: FragmentItemsBinding? = null

    override fun onViewCreated(view: View, savedInstanceState: Bundle?) {
        super.onViewCreated(view, savedInstanceState)
        binding = FragmentItemsBinding.bind(view)
        val adapter = ItemsAdapter()
        binding!!.recyclerView.adapter = adapter

        // viewLifecycleOwner — жизнь View, а не фрагмента
        viewLifecycleOwner.lifecycleScope.launch {
            viewLifecycleOwner.repeatOnLifecycle(Lifecycle.State.STARTED) {
                viewModel.items.collect { list ->
                    adapter.submitList(list)
                }
            }
        }
    }

    override fun onDestroyView() {
        super.onDestroyView()
        binding = null  // обнуляем, чтобы не держать мёртвую View
    }
}

Объяснение: repeatOnLifecycle(STARTED) автоматически отменяет корутину при переходе ниже STARTED (onStop) и перезапускает при возврате. Привязка к viewLifecycleOwner гарантирует, что подписка живёт ровно столько, сколько живёт View — не дольше. Если вместо viewLifecycleOwner передать this (lifecycle фрагмента), подписка переживёт onDestroyView и при следующем эмите обратится к уничтоженному binding → NPE.

Задача 2. Реализуйте foreground service для запущенной тренировки (fitness-tracking), который работает на Android 14+. Service должен: стартовать из Activity, показывать постоянное уведомление с каналом, корректно объявить тип в манифесте и запросить нужное разрешение. Покажите манифест и минимальный код сервиса.

Показать решение 
// AndroidManifest.xml (фрагмент)
// <uses-permission android:name="android.permission.FOREGROUND_SERVICE" />
// <uses-permission android:name="android.permission.FOREGROUND_SERVICE_HEALTH" />
// <uses-permission android:name="android.permission.POST_NOTIFICATIONS" />
//
// <service
//     android:name=".WorkoutService"
//     android:foregroundServiceType="health"
//     android:exported="false" />

class WorkoutService : Service() {

    override fun onStartCommand(intent: Intent?, flags: Int, startId: Int): Int {
        val channelId = createNotificationChannel()
        val notification = NotificationCompat.Builder(this, channelId)
            .setContentTitle("Тренировка идёт")
            .setContentText("Время: 00:00")
            .setSmallIcon(R.drawable.ic_run)
            .setOngoing(true)
            .build()

        if (Build.VERSION.SDK_INT >= Build.VERSION_CODES.Q) {
            startForeground(
                NOTIFICATION_ID,
                notification,
                ServiceInfo.FOREGROUND_SERVICE_TYPE_HEALTH
            )
        } else {
            startForeground(NOTIFICATION_ID, notification)
        }

        return START_STICKY
    }

    override fun onBind(intent: Intent?): IBinder? = null

    private fun createNotificationChannel(): String {
        val id = "workout_channel"
        if (Build.VERSION.SDK_INT >= Build.VERSION_CODES.O) {
            val channel = NotificationChannel(
                id, "Тренировка", NotificationManager.IMPORTANCE_LOW
            )
            getSystemService(NotificationManager::class.java).createNotificationChannel(channel)
        }
        return id
    }

    companion object {
        const val NOTIFICATION_ID = 1001

        fun start(context: Context) {
            val intent = Intent(context, WorkoutService::class.java)
            // startForegroundService — обязателен при старте FGS из foreground-контекста
            context.startForegroundService(intent)
        }
    }
}

Объяснение: на Android 14 (API 34) тип health обязателен для фитнес-трекинга — без него startForeground() кидает MissingForegroundServiceTypeException. START_STICKY просит систему перезапустить сервис, если она его убьёт (Intent будет null в следующем onStartCommand). Уведомление с IMPORTANCE_LOW не издаёт звук и подходит для постоянного статуса.

Задача 3. Предскажите, что выведет следующий код при первом запуске Activity, а затем при повороте экрана. Объясните порядок колбэков и почему counter сохраняется (или нет) после поворота.

Показать решение 
class PredictActivity : AppCompatActivity() {
    private var counter = 0

    override fun onCreate(savedInstanceState: Bundle?) {
        super.onCreate(savedInstanceState)
        counter = savedInstanceState?.getInt("counter") ?: 0
        Log.d("LC", "onCreate counter=$counter")
    }

    override fun onStart()   { super.onStart();   Log.d("LC", "onStart") }
    override fun onResume()  { super.onResume();  Log.d("LC", "onResume")  }
    override fun onPause()   { super.onPause();   Log.d("LC", "onPause")   }
    override fun onStop()    { super.onStop();    Log.d("LC", "onStop")    }
    override fun onDestroy() { super.onDestroy(); Log.d("LC", "onDestroy") }

    override fun onSaveInstanceState(out: Bundle) {
        super.onSaveInstanceState(out)
        counter++
        out.putInt("counter", counter)
        Log.d("LC", "onSaveInstanceState counter=$counter")
    }
}

Первый запуск: onCreate counter=0 → onStart → onResume.

При повороте: onPause → onStop → onSaveInstanceState counter=1 → onDestroy → onCreate counter=1 → onStart → onResume.

Объяснение: поворот — configuration change; система уничтожает и пересоздаёт Activity. Перед уничтожением вызывается onSaveInstanceState (после onStop начиная с Android P), который сохраняет Bundle в памяти процесса. При пересоздании Bundle передаётся в onCreate, поэтому counter равен 1. Ключевое: onSaveInstanceState вызывается только при системном уничтожении (поворот, смерть процесса), но не при явном finish().

Задача 4. В onReceive ресивера нужно сделать короткий сетевой/дисковый запрос (несколько секунд) и записать результат. Покажите два корректных подхода: через goAsync() для очень короткой работы и через WorkManager для работы, которая может занять дольше. Объясните, почему нельзя просто запустить корутину в onReceive.

Показать решение 
// Подход 1 — goAsync(): для очень короткой работы (бюджет всё равно ~10 сек)
class ShortWorkReceiver : BroadcastReceiver() {
    override fun onReceive(context: Context, intent: Intent) {
        val pending = goAsync()
        CoroutineScope(Dispatchers.IO).launch {
            try {
                doShortWork(context)
            } finally {
                pending.finish()   // обязательно: иначе процесс висит → ANR
            }
        }
    }
}

// Подход 2 — WorkManager: для работы, которая может занять дольше
class EnqueueReceiver : BroadcastReceiver() {
    override fun onReceive(context: Context, intent: Intent) {
        val req = OneTimeWorkRequestBuilder<SyncWorker>()
            .setConstraints(Constraints(requiredNetworkType = NetworkType.CONNECTED))
            .build()
        WorkManager.getInstance(context).enqueue(req)  // возврат мгновенный
    }
}

Объяснение: onReceive идёт на главном потоке с лимитом ~10 секунд до ANR. Просто запустить launch без goAsync() нельзя: как только onReceive вернёт управление, процесс считается idle и может быть убит до завершения корутины, а сам объект ресивера больше не валиден. goAsync() держит процесс живым чуть дольше (тот же бюджет ~10 сек), и finish() обязателен. Для работы, которая может занять дольше или должна пережить смерть процесса, — WorkManager.enqueue: возврат из onReceive мгновенный, а работу WorkManager доведёт сам.

Задача 5. Activity с launchMode="singleTop" уже открыта Intent'ом A. Приходит push и запускает её же Intent'ом B с новыми extras. Напишите код, который корректно обработает оба Intent'а, и объясните, какой колбэк сработает во втором случае и почему чтение getIntent() в onCreate даст устаревшие данные.

Показать решение 
class SearchActivity : AppCompatActivity() {

    override fun onCreate(savedInstanceState: Bundle?) {
        super.onCreate(savedInstanceState)
        handleIntent(intent)   // первый запуск: Intent A из getIntent()
    }

    override fun onNewIntent(intent: Intent) {
        super.onNewIntent(intent)
        setIntent(intent)      // ВАЖНО: иначе getIntent() вернёт старый A
        handleIntent(intent)   // повторный запуск: Intent B
    }

    private fun handleIntent(intent: Intent) {
        val query = intent.getStringExtra("query")
        // ... отрисовать по query
    }
}

Объяснение: при singleTop, если Activity уже на вершине стека, повторный запуск не создаёт новый экземпляр и не зовёт onCreate — приходит onNewIntent(intent) (между onPause и onResume). При этом getIntent() по-прежнему возвращает старый Intent A, пока вы не вызовете setIntent(intent). Классическая ловушка с уведомлениями: extras из второго пуша теряются, если читать только в onCreate. Решение — общий handleIntent, вызываемый и в onCreate (из getIntent()), и в onNewIntent после setIntent.
