Android-компоненты и межпроцессное взаимодействие

Четыре базовых компонента Android-приложения: Activity (экран/точка входа), Service (служба для фоновой работы), BroadcastReceiver (приёмник системных и приложенческих событий), ContentProvider (поставщик данных), их жизненные циклы, ограничения фоновой работы в новых версиях ОС (операционной системы) и то, как процессы общаются через Binder и AIDL (Android Interface Definition Language — язык описания интерфейса для IPC, межпроцессного взаимодействия). Тема, по которой видно, понимаешь ли ты платформу «под капотом», а не только Jetpack Compose (декларативный инструментарий пользовательского интерфейса) поверх неё.

Middle Senior Частый вопрос

Activity: жизненный цикл и состояние

Activity проходит через колбэки, парные по смыслу: onCreate/onDestroy (всё время жизни), onStart/onStop (видимость), onResume/onPause (фокус, foreground). Считывать эти пары — половина успеха: ресурсы, требующие парного освобождения, привязывают к симметричным колбэкам.

stateDiagram-v2 [*] --> Created: onCreate() Created --> Started: onStart() Started --> Resumed: onResume() Resumed --> Started: onPause() Started --> Stopped: onStop() Stopped --> Started: onRestart() + onStart() Stopped --> Destroyed: onDestroy() Destroyed --> [*] note right of Resumed RESUMED: на экране, в фокусе end note note right of Started STARTED: видна, но без фокуса (диалог, split-screen) end note
Жизненный цикл Activity. После onPause экран ещё виден; после onStop — уже нет

onSaveInstanceState и восстановление

Система может убить процесс в фоне ради памяти. Чтобы UI восстановился «как был», непосредственно перед потенциальным уничтожением (после onStop, до Android P — после onPause) вызывается onSaveInstanceState(Bundle). Bundle сериализуется и переживает смерть процесса. Ограничения: только небольшие данные (он маршалится через Binder, лимит ~1 МБ на всю транзакцию), типы должны быть Parcelable/примитивы.

MainActivity.kt
class MainActivity : ComponentActivity() {
    private var counter = 0

    override fun onCreate(savedInstanceState: Bundle?) {
        super.onCreate(savedInstanceState)
        // восстановление транзиентного UI-состояния
        counter = savedInstanceState?.getInt(KEY_COUNTER) ?: 0
    }

    override fun onSaveInstanceState(outState: Bundle) {
        super.onSaveInstanceState(outState)
        outState.putInt(KEY_COUNTER, counter)   // только маленькие данные!
    }

    companion object { private const val KEY_COUNTER = "counter" }
}
💡
Три уровня живучести состояния. ViewModel переживает поворот, но не смерть процесса. onSaveInstanceState/SavedStateHandle переживает смерть процесса, но мало места. Постоянное хранилище (Room/DataStore) — для всего остального. На собесе различай эти три уровня чётко.

Configuration changes

Поворот экрана, смена языка, темы, размера окна — это configuration change. По умолчанию система уничтожает и пересоздаёт Activity, чтобы перечитать ресурсы под новую конфигурацию. Можно перехватить вручную через android:configChanges + onConfigurationChanged(), но это считается «костылём» — теряется автоматическая подгрузка ресурсов. Правильный путь — поднимать состояние в ViewModel, а UI делать пересоздаваемым (Compose это и делает).

Launch modes и back stack

Launch mode определяет, как новая Activity ложится в task (стек задач). Управляется атрибутом android:launchMode либо флагами Intent (FLAG_ACTIVITY_*).

Launch modeПоведениеКогда применять
standardВсегда новый экземпляр, можно несколько в стекеПо умолчанию, обычные экраны
singleTopЕсли уже на вершине стека — переиспользуется, прилетает onNewIntent()Экран поиска, уведомления
singleTaskОдин экземпляр в системе; при повторном запуске стек над ним очищается (onNewIntent())Главный/«домашний» экран приложения
singleInstanceКак singleTask, но в отдельной task в одиночествеЗвонилка, launcher, очень редко
⚠️
Грабли singleTask/singleInstance. Они меняют поведение кнопки «Назад» и могут «вынести» Activity в чужой/отдельный task, ломая ожидаемую навигацию. На практике 95% экранов — standard/singleTop. Если интервьюер спрашивает про singleInstance «по жизни» — почти всегда правильный ответ «избегаю, кроме особых случаев».

Intent: explicit и implicit

Explicit — указан конкретный класс-получатель (внутри своего приложения). Implicit — описано действие (action + category + data), а система через intent-filter находит подходящий компонент (часто в другом приложении).

Intent.kt
// Explicit — знаем класс
startActivity(Intent(this, DetailActivity::class.java).apply {
    putExtra("id", 42)
})

// Implicit — описываем намерение, система выбирает приложение
val share = Intent(Intent.ACTION_SEND).apply {
    type = "text/plain"
    putExtra(Intent.EXTRA_TEXT, "Привет с собеса")
}
// безопасно: проверка, что есть кому обработать
if (share.resolveActivity(packageManager) != null) startActivity(share)

ActivityResult API

Старый startActivityForResult()/onActivityResult() устарел: ручной разбор requestCode, потеря результата при смерти процесса. Современный ActivityResult API регистрирует контракт заранее, лайфсайкл-безопасен и типизирован.

PickImage.kt
// регистрируем ДО onCreate-резолва, переживает пересоздание Activity
private val pickImage = registerForActivityResult(
    ActivityResultContracts.GetContent()
) { uri: Uri? ->
    uri?.let { viewModel.onImagePicked(it) }
}

// запуск без requestCode-магии
fun chooseImage() = pickImage.launch("image/*")

Fragment: lifecycle и viewLifecycleOwner

Fragment — переиспользуемая часть UI внутри Activity. Ключевое отличие от Activity: у фрагмента два разных жизненных цикла — самого фрагмента и его View. View может уничтожаться и пересоздаваться (например, при уходе в back stack), пока сам фрагмент жив.

Этап FragmentЭтап ViewЧто делаем
onAttach → onCreate—Инициализация, аргументы
onCreateViewView созданНадуть/получить layout
onViewCreatedviewLifecycleOwner активенПодписки на UI, биндинг
onDestroyViewView уничтоженОбнулить binding, отписки
onDestroy → onDetach—Финальная очистка
🛑
Почему viewLifecycleOwner критичен. Если подписаться на Flow/LiveData, указав this (lifecycle фрагмента) вместо viewLifecycleOwner, подписка переживёт уничтожение View. После onDestroyView придёт новое значение — и колбэк обратится к уже мёртвой View (или старому binding) → NullPointerException или утечка. Правило: всё, что трогает View, подписывай на viewLifecycleOwner.
ProfileFragment.kt
override fun onViewCreated(view: View, savedInstanceState: Bundle?) {
    super.onViewCreated(view, savedInstanceState)
    // привязка к жизни VIEW, а не фрагмента
    viewLifecycleOwner.lifecycleScope.launch {
        viewLifecycleOwner.repeatOnLifecycle(Lifecycle.State.STARTED) {
            viewModel.state.collect { render(it) }
        }
    }
}

FragmentManager и Fragment Result API

FragmentManager управляет транзакциями (add/replace/remove) и back stack фрагментов. Для передачи данных между фрагментами без прямых ссылок (которые ведут к связности и крашам) используют Fragment Result API: один фрагмент кладёт результат в менеджер, другой подписывается по строковому ключу.

FragmentResult.kt
// Фрагмент-получатель: слушатель привязан к viewLifecycleOwner
setFragmentResultListener("filterKey") { _, bundle ->
    val sort = bundle.getString("sort")
    viewModel.applySort(sort)
}

// Фрагмент-отправитель (например, BottomSheet)
setFragmentResult("filterKey", bundleOf("sort" to "price_asc"))

Service: started, foreground и bound

Service — компонент без UI для долгой работы. Важно: Service по умолчанию работает в главном потоке — это не «фоновый поток», а лишь компонент без интерфейса. Тяжёлую работу всё равно уносят в корутины/поток.

Foreground service и типы в Android 14

Если задача должна выполняться, пока пользователь явно видит её (плеер, навигация, запись), используют foreground service с постоянным уведомлением. Начиная с Android 14 (API 34), для каждого FGS обязательно объявить android:foregroundServiceType в манифесте и запросить соответствующее разрешение FOREGROUND_SERVICE_*. Запуск без подходящего типа кидает SecurityException/MissingForegroundServiceTypeException.

TrackingService.kt
class TrackingService : Service() {
    override fun onStartCommand(intent: Intent?, flags: Int, startId: Int): Int {
        val notification = buildNotification()
        // на Android 10+ передаём тип явно при старте
        if (Build.VERSION.SDK_INT >= Build.VERSION_CODES.Q) {
            startForeground(ID, notification, FOREGROUND_SERVICE_TYPE_LOCATION)
        } else {
            startForeground(ID, notification)
        }
        return START_STICKY  // система перезапустит при убийстве
    }
    override fun onBind(intent: Intent?): IBinder? = null
}

Ограничения фоновой работы

С Android 8 (Oreo) приложение в фоне не может свободно стартовать обычные сервисы (Background Execution Limits): попытка startService() из фона кидает IllegalStateException. С Android 12 запрещён и старт самого FGS из фона (за исключениями). Поэтому «просто запустить сервис, когда захочу» больше не работает.

💡
WorkManager — современная замена. Для гарантированной, отложенной фоновой работы (синхронизация, выгрузка логов, ретраи) сегодня используют WorkManager: он переживает перезагрузку, уважает Doze/ограничения, выбирает движок (JobScheduler) под версию ОС и поддерживает constraints (сеть/зарядка). FGS — только для работы, которую прямо сейчас видит пользователь.
SyncWorker.kt
class SyncWorker(ctx: Context, params: WorkerParameters) : CoroutineWorker(ctx, params) {
    override suspend fun doWork(): Result = try {
        repository.sync()
        Result.success()
    } catch (e: IOException) {
        Result.retry()   // WorkManager сам применит backoff
    }
}

// запуск с условиями
val req = OneTimeWorkRequestBuilder<SyncWorker>()
    .setConstraints(Constraints(requiredNetworkType = NetworkType.CONNECTED))
    .build()
WorkManager.getInstance(context).enqueue(req)

Doze и App Standby

Doze включается, когда устройство долго лежит без движения с выключенным экраном: система «замораживает» сетевой доступ, откладывает джобы/алармы и собирает их в редкие «окна обслуживания». App Standby ограничивает приложения, которыми давно не пользовались (бакеты active/working set/frequent/rare/restricted). Вывод для кандидата: нельзя рассчитывать на точное время фоновых задач — используй WorkManager, а для критичной точности по времени — setExactAndAllowWhileIdle() с разрешением (и осторожно).

Компонент-подписчик на системные (BOOT_COMPLETED, заряд, сеть) или собственные события. Регистрируется двумя способами с принципиально разной семантикой.

Ограничение implicit broadcast (Android 8+)

Чтобы пачка приложений не «просыпалась» от каждого системного события, начиная с Android 8 нельзя регистрировать implicit броадкасты в манифесте. Исключения остались для важных событий, которые приходят редко: ACTION_BOOT_COMPLETED, ACTION_LOCALE_CHANGED, ACTION_MY_PACKAGE_REPLACED, события медиа-носителей и др. (полный список — «Implicit Broadcast Exceptions»). Для всего остального: регистрируй ресивер динамически, пока приложение активно, либо используй explicit-броадкаст с указанием пакета/компонента.

NetworkReceiver.kt
class NetworkReceiver(val onChange: (Boolean) -> Unit) : BroadcastReceiver() {
    override fun onReceive(context: Context, intent: Intent) {
        // onReceive выполняется в main-потоке и кратко: тяжёлое — в WorkManager
        onChange(isOnline(context))
    }
}

// динамическая регистрация на время видимости экрана
override fun onStart() {
    super.onStart()
    registerReceiver(receiver, IntentFilter(ConnectivityManager.CONNECTIVITY_ACTION))
}
override fun onStop() {
    super.onStop()
    unregisterReceiver(receiver)   // обязательно — иначе утечка
}
⚠️
LocalBroadcastManager — deprecated. Он опирался на Binder (механизм IPC) ради сообщений внутри одного процесса — это избыточно и поощряло нарушение слоёв («любой компонент слышит любое событие»). Современная замена для внутреннего event-bus — SharedFlow/StateFlow (или LiveData): лайфсайкл-безопасно, типизированно, без накладных расходов Binder.

ContentProvider и Context

ContentProvider

Стандартный механизм предоставить структурированные данные другим приложениям (или процессам). Внутри одного приложения он почти не нужен — там проще Room/репозиторий. Он оправдан, когда: данные читают чужие приложения; нужно интегрироваться с системой (контакты, календарь, медиатека, FileProvider для безопасной раздачи файлов); требуется SyncAdapter/виджеты, которым нужен provider.

Доступ — через ContentResolver по URI вида content://authority/path/id. Права разграничиваются android:readPermission/writePermission и временными URI permissions (FLAG_GRANT_READ_URI_PERMISSION) — так FileProvider отдаёт один файл без выдачи доступа ко всему.

ContentResolver.kt
// чтение контактов через системный ContentProvider
val cursor = contentResolver.query(
    ContactsContract.Contacts.CONTENT_URI,
    arrayOf(ContactsContract.Contacts.DISPLAY_NAME),
    null, null, null
)
cursor?.use {
    while (it.moveToNext()) {
        val name = it.getString(0)
    }
}

// безопасная раздача своего файла другому приложению
val uri = FileProvider.getUriForFile(context, "$packageName.fileprovider", file)
val intent = Intent(Intent.ACTION_VIEW).apply {
    setDataAndType(uri, "application/pdf")
    addFlags(Intent.FLAG_GRANT_READ_URI_PERMISSION)  // только этот URI
}

Context: application vs activity и утечки

Context — «ручка» к ресурсам, ассетам, системным сервисам и запуску компонентов. Есть два важных вида: Application Context (живёт всё время процесса) и Activity Context (умирает с экраном, знает про тему/окно).

НужноКакой Context
Надуть UI, показать диалог, тема экранаActivity (у Application нет темы)
Singleton, БД, DI-граф, долгоживущий объектApplication
Старт сервиса/получение системного сервисаЛюбой (предпочти Application для долгих)
🛑
Классическая утечка. Передать Activity Context в долгоживущий объект (синглтон, статика, фоновый поток, корутина, которая переживает экран). Activity не сможет быть собрана GC, пока её держит «вечный» объект → утечка всего дерева View. Правило: для всего, что живёт дольше экрана, бери applicationContext. Для UI — Activity, но не сохраняй её ссылку надолго.

IPC: Binder под капотом

Процессы в Android изолированы (каждое приложение — свой Linux-процесс и UID). Чтобы они общались, нужен IPC. Основной механизм — Binder: это и есть транспорт почти всего системного API (вызов ActivityManager, startActivity через system_server и т.д.).

Binder реализован как драйвер ядра /dev/binder. На стороне клиента ты держишь proxy (реализует интерфейс, но внутри упаковывает вызов), на стороне сервера — stub (распаковывает и вызывает реальный метод). Клиент зовёт transact(), драйвер копирует данные в адресное пространство сервера и вызывает onTransact(). Сервер обрабатывает вызовы в пуле binder-потоков (по умолчанию до ~16), поэтому удалённые методы могут выполняться параллельно и не на main-потоке.

sequenceDiagram participant C as Клиент (Proxy) participant K as Драйвер /dev/binder participant S as Сервер (Stub) C->>C: marshalling args в Parcel C->>K: transact(code, data, reply) Note over C: вызывающий поток блокируется K->>K: копирует Parcel в адр. пр-во сервера K->>S: onTransact(code, data, reply) S->>S: распаковка + реальный метод S-->>K: пишет результат в reply K-->>C: разблокировка + reply C->>C: unmarshalling результата
Один синхронный Binder-вызов: proxy → драйвер → stub и обратно
ℹ️
Почему Binder, а не сокеты/pipe. Binder делает копирование один раз (через драйвер), знает про процессы/потоки/UID (можно проверить Binder.getCallingUid()), переносит ссылки на объекты между процессами и интегрирован с моделью безопасности Android. Это быстрее и безопаснее обычного двойного копирования через сокет.

AIDL, Messenger и процессы

AIDL: интерфейс между процессами

Писать proxy/stub руками тяжело — для этого есть AIDL (Android Interface Definition Language). Описываешь интерфейс в .aidl, генератор создаёт Stub (наследуешь на сервере) и Proxy (получаешь на клиенте через asInterface()). Данные пересекают границу процесса через маршалинг в Parcel; кастомные классы должны быть Parcelable.

IRemoteService.aidl
interface IRemoteService {
    int add(int a, int b);
    // oneway — не блокирует клиента, без возврата результата ("fire and forget")
    oneway void log(String message);
}
RemoteService.kt
class RemoteService : Service() {
    // Stub сгенерирован из .aidl; работает в binder-потоке
    private val binder = object : IRemoteService.Stub() {
        override fun add(a: Int, b: Int): Int = a + b
        override fun log(message: String) { Log.d("IPC", message) }
    }
    override fun onBind(intent: Intent?): IBinder = binder
}

// клиент в другом процессе
private val connection = object : ServiceConnection {
    override fun onServiceConnected(name: ComponentName?, service: IBinder?) {
        val remote = IRemoteService.Stub.asInterface(service)  // получили proxy
        val sum = remote.add(2, 3)                             // синхронный IPC
    }
    override fun onServiceDisconnected(name: ComponentName?) {}
}

Модификатор oneway делает вызов асинхронным: клиент не ждёт ответа, метод не может возвращать значение. По умолчанию AIDL-вызовы синхронны и блокируют вызывающий поток до ответа.

Messenger: очередь поверх Binder

Если параллелизм не нужен и хватает последовательной обработки сообщений, проще Messenger: он строит очередь Message поверх одного Binder и обрабатывает их по одному в Handler. Меньше кода, чем AIDL, но нет параллельных вызовов и сложных сигнатур — только Message.

Процессы и цена IPC

Компонент можно вынести в отдельный процесс через android:process (например, тяжёлый/падучий код изолировать от UI-процесса). Но за это платят: каждый IPC-вызов в разы дороже обычного вызова метода (маршалинг + копирование + переключение контекста), данные нужно делать Parcelable, отладка усложняется.

🛑
Лимит транзакции Binder ~1 МБ. Буфер транзакций один на процесс и невелик (порядка 1 МБ, делится между всеми параллельными транзакциями). Попытка передать большой Bundle/Parcelable (огромный список, битмап) через Intent/AIDL/onSaveInstanceState кидает TransactionTooLargeException. Лечение: не передавай большие данные через IPC — передавай идентификатор/URI, а сами данные доставай из общего хранилища (БД, файл, ContentProvider) или через shared memory.

Альтернативы Binder для IPC: ContentProvider (CRUD-данные), доменные сокеты/LocalSocket (потоковые данные, портирование «как на сервере»), файлы/MemoryFile/ashmem (большие объёмы через shared memory). Но для типичного «дай мне API другого приложения/сервиса» канонично именно AIDL поверх Binder.

Частые вопросы интервьюера

Что вызывается при повороте экрана и как сохранить состояние? Middle

Поворот — это configuration change: Activity по умолчанию уничтожается (onPause → onStop → onSaveInstanceState → onDestroy) и пересоздаётся (onCreate → onStart → onResume). Состояние сохраняют по трём уровням: ViewModel (переживает поворот, но не смерть процесса), onSaveInstanceState/SavedStateHandle (переживает смерть процесса, но только маленькие Parcelable), постоянное хранилище (Room/DataStore — всё остальное). Перехватывать поворот через configChanges вручную считается антипаттерном.

Зачем нужен viewLifecycleOwner во фрагменте? Middle+

У фрагмента два жизненных цикла: самого фрагмента и его View. View может пересоздаваться (back stack), пока фрагмент жив. Если подписку на Flow/LiveData, которая трогает View, привязать к lifecycle фрагмента (this), она переживёт onDestroyView и при новом эмите обратится к уничтоженной View или старому binding → NPE/утечка. Поэтому всё, что работает с View, подписываем на viewLifecycleOwner.

Чем foreground service отличается от WorkManager и что выбрать? Middle+

Foreground service — для работы, которую пользователь видит прямо сейчас (плеер, навигация, запись), с постоянным уведомлением; с Android 14 обязателен foregroundServiceType + соответствующее разрешение. WorkManager — для гарантированной отложенной работы (синхронизация, выгрузка, ретраи): переживает перезагрузку, уважает Doze и constraints, сам выбирает движок. Правило: «нужно показать прогресс/удержать систему сейчас» → FGS; «надо доделать рано или поздно, можно подождать» → WorkManager.

Почему с Android 8 нельзя ловить implicit broadcast в манифесте? Senior

Раньше системное событие (например, изменение сети) будило десятки приложений разом, расходуя батарею и память. С Android 8 манифест-регистрация implicit-броадкастов запрещена (кроме списка исключений — BOOT_COMPLETED, LOCALE_CHANGED, события медиа-носителей и т.п., которые приходят редко). Решения: регистрировать ресивер динамически, пока приложение активно; использовать explicit-броадкаст с указанием компонента; либо подписываться на изменения через WorkManager constraints или ConnectivityManager.NetworkCallback.

Что такое Binder и почему он, а не сокеты? Senior

Binder — ядро IPC в Android (драйвер /dev/binder), транспорт почти всего системного API. Клиент держит proxy, сервер — stub; вызов идёт через transact()/onTransact(), драйвер копирует Parcel в память сервера, обработка — в пуле binder-потоков (отсюда параллелизм и не-main-поток). Преимущества над сокетами: одно копирование, знание о процессах/UID (проверка прав getCallingUid()), перенос ссылок на объекты, интеграция с моделью безопасности. AIDL — это генератор proxy/stub поверх Binder.

Откуда берётся TransactionTooLargeException? Senior

Буфер транзакций Binder один на процесс и невелик (порядка 1 МБ, делится между всеми параллельными транзакциями). Если через Intent extras, AIDL-вызов или onSaveInstanceState протащить большой объект (огромный список, битмап), маршалинг переполняет буфер → исключение. Чаще всего ловится при onSaveInstanceState с тяжёлым Bundle. Лечение: не гонять большие данные через IPC — передавать идентификатор/URI, а данные доставать из общего хранилища (Room, файл, ContentProvider).

В чём разница AIDL и Messenger? Middle+

Оба работают поверх Binder. AIDL даёт полноценный типизированный интерфейс с методами и поддерживает параллельные вызовы (binder thread pool) — нужно, когда много клиентов или богатый API; требует думать о потокобезопасности. Messenger строит очередь Message поверх одного Binder и обрабатывает их последовательно в Handler — меньше кода, нет многопоточности, подходит для простого обмена сообщениями. Эвристика: нужен параллелизм/сложный API → AIDL, иначе → Messenger.

Какой Context брать и где утечки? Middle

Application Context живёт весь процесс — его берут для синглтонов, БД, DI-графа, всего, что переживает экран. Activity Context знает про тему/окно — нужен для надувания UI и диалогов, но умирает с экраном. Утечка возникает, когда Activity Context (или View/Activity) сохраняют в долгоживущем объекте (статика, синглтон, фоновый поток/корутина) — Activity не может быть собрана GC, утекает всё дерево View. Правило: дольше экрана — только applicationContext.

Чем плох LocalBroadcastManager и чем заменить? Middle+

Он deprecated: использовал Binder (механизм IPC) для сообщений внутри одного процесса — избыточно по накладным расходам и поощрял глобальный event-bus, где любой компонент слышит любое событие (нарушение слоёв, сложно отслеживать). Современная замена — реактивные потоки: SharedFlow/StateFlow в общем источнике (репозиторий/ViewModel) или LiveData. Это лайфсайкл-безопасно, типизированно и имеет понятный scope.
