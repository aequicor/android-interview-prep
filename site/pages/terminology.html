Проектирование и ООП

ТерминОпределение
АбстракцияОписание объекта через минимальное множество наблюдаемых свойств, достаточных для работы с ним, при сокрытии всех прочих деталей реализации.
РеализацияКонкретное воплощение абстракции, удовлетворяющее всем обязательствам её контракта.
КонтрактТройка ⟨предусловие, постусловие, инварианты⟩, которой обязана удовлетворять каждая реализация типа или метода.
ИнвариантЛогический предикат, истинный для любого наблюдаемого состояния объекта: до вызова каждого публичного метода и после его завершения.
ЭвристикаПравило принятия решений, дающее приемлемый результат в типичных случаях без гарантии оптимальности.
Связанность (coupling)Степень, в которой корректность модуля A зависит от внутреннего устройства модуля B.
Сплочённость (cohesion)Степень, в которой все элементы модуля служат единственной чётко определённой ответственности.
is-aОтношение подтипирования T₁ <: T₂: любой экземпляр T₁ может использоваться там, где ожидается T₂ (принцип подстановки Лисков).
has-aОтношение агрегации: объект A содержит ссылку на объект B как часть своего состояния.
can-doОтношение удовлетворения интерфейса: тип A предоставляет все операции, объявленные интерфейсом I, независимо от иерархии наследования.
ПолиморфизмСвойство операции работать с операндами разных типов. Подтиповой (динамический) — диспетчеризация по фактическому типу в рантайме; параметрический (статический) — единый код для всех типов, удовлетворяющих ограничению.
ИнкапсуляцияСвязывание состояния объекта с операциями над ним при ограничении прямого доступа к состоянию извне с целью сохранения инвариантов.
Dependency Injection (DI)Паттерн, при котором зависимости объекта передаются ему вызывающим кодом, а не создаются объектом самостоятельно.
Inversion of Control (IoC)Принцип, при котором управление потоком выполнения и создание объектов передаётся инфраструктурному компоненту, а не прикладному коду.
BoilerplateКод, структурно обязательный для работы системы, но не содержащий специфичной для задачи логики.
Побочный эффект (side effect)Любое изменение наблюдаемого состояния системы, произошедшее при вычислении выражения помимо формирования возвращаемого значения.
Чистая функция (pure function)Функция, удовлетворяющая двум условиям: (1) детерминирована — равные аргументы всегда дают равный результат; (2) не производит побочных эффектов.
ИдемпотентностьСвойство операции f: f(f(x)) = f(x) для любого допустимого x; повторное применение не изменяет результат.
GoFGang of Four — четыре автора «Design Patterns: Elements of Reusable Object-Oriented Software» (1994), систематизировавших 23 структурных, порождающих и поведенческих паттерна.

Язык Kotlin и JVM

ТерминОпределение
Стирание типов (type erasure)Механизм компиляции JVM, при котором параметры обобщённых типов удаляются из байткода; в рантайме все параметризованные типы представлены стёртым базовым типом.
reifiedМодификатор параметра типа inline-функции, предписывающий компилятору подставить фактический тип в каждое место вызова; делает параметр доступным как T::class и допускает is T.
Платформенный тип (T!)Тип значения из кода без аннотаций nullability (типично — Java); компилятор Kotlin не определяет его nullable-статус и перекладывает ответственность на разработчика.
Неизменяемость (immutability)Свойство объекта: ни одно его наблюдаемое поле не изменяется после завершения конструктора.
== vs ===== — структурное равенство, делегирует к equals(); === — ссылочное равенство, истинно тогда и только тогда, когда оба операнда указывают на один объект в памяти.
Контракт equals/hashCodeОбязательное свойство: a.equals(b) → a.hashCode() == b.hashCode(). Обратное необязательно. Нарушение делает объект некорректным ключом в хеш-коллекциях.
Боксинг (boxing)Преобразование значения примитивного типа JVM в экземпляр соответствующего ссылочного типа-обёртки, сопровождаемое аллокацией в heap.
Ленивая инициализацияСтратегия вычисления значения: отложить его до первого обращения, после чего кэшировать результат для всех последующих обращений.
Higher-order functionФункция, принимающая другую функцию в качестве аргумента или возвращающая функцию как значение.
suspend / Continuationsuspend-функция компилируется в конечный автомат с явной передачей Continuation<T> — объекта, инкапсулирующего оставшуюся часть вычисления; каждая точка приостановки соответствует состоянию автомата.
Делегирование (by)Механизм компилятора: реализация методов интерфейса или тела свойства генерируется как прямые вызовы к объекту-делегату.

Конкурентность и многопоточность

ТерминОпределение
Гонка (race condition)Ситуация, при которой корректность программы зависит от относительного порядка операций нескольких потоков, а этот порядок не зафиксирован синхронизацией.
АтомарностьСвойство операции: с точки зрения любого другого потока она выполняется мгновенно и неделимо — наблюдается либо полностью до, либо полностью после.
CAS (Compare-And-Swap)Атомарная инструкция: если текущее значение ячейки равно ожидаемому, заменить его новым; в противном случае — ничего не делать. Возвращает признак успеха.
ABA-проблемаДефект CAS: между чтением ожидаемого значения A и самим CAS другой поток изменил ячейку A→B→A; операция ошибочно считает, что изменений не было.
Thread-safetyСвойство объекта или операции: при произвольном конкурентном доступе из нескольких потоков без внешней синхронизации инварианты объекта сохраняются, а результаты соответствуют некоторому последовательному порядку операций.
volatileГарантия модели памяти: запись в переменную устанавливает отношение happens-before со всеми последующими чтениями той же переменной. Не обеспечивает атомарность составных операций.
happens-beforeЧастичный порядок на множестве операций в модели памяти: A happens-before B означает, что все записи, сделанные до A включительно, видимы после B.
DeadlockСостояние системы потоков, при котором каждый поток заблокирован в ожидании ресурса, удерживаемого другим потоком из того же множества; прогресс невозможен без внешнего вмешательства.
Structured concurrencyМодель, в которой время жизни дочерних асинхронных задач строго ограничено временем жизни порождающего scope; отмена и ошибки распространяются по дереву задач детерминированно.
BackpressureСостояние потокового конвейера, при котором скорость производства элементов превышает скорость их потребления; требует явной стратегии: буферизации, сброса или замедления источника.
МемоизацияОптимизация чистой функции: сохранять возвращаемое значение, индексированное аргументами, и возвращать кэшированный результат при повторном вызове с теми же аргументами.

Производительность и данные

ТерминОпределение
Latency vs ThroughputLatency — время от инициации операции до получения её результата. Throughput — количество операций, завершаемых системой в единицу времени. Метрики независимы и могут противоречить друг другу.
Cold / warm / hot startCold: процесс приложения отсутствует, требуется полная инициализация. Warm: процесс существует, но целевая Activity уничтожена. Hot: Activity присутствует в back stack и восстанавливается без пересоздания.
JankПропуск кадра вывода, возникающий когда время подготовки кадра превышает бюджет: ~16 мс при 60 Гц или ~8 мс при 120 Гц.
Утечка памяти (memory leak)Объект, более не используемый прикладным кодом, остаётся достижимым из GC root через непреднамеренную ссылку; сборщик мусора не освобождает занятую им память.
GC (сборщик мусора)Подсистема рантайма, автоматически идентифицирующая и освобождающая heap-память объектов, недостижимых из GC roots.
Single source of truth (SSOT)Архитектурный принцип: каждый элемент состояния имеет ровно один авторитетный источник; все остальные компоненты получают его значение только оттуда и не хранят собственных копий.
Offline-firstАрхитектурный подход: мутации применяются к локальному хранилищу немедленно и синхронно; репликация на сервер происходит асинхронно при наличии сети.
ПагинацияРазбиение упорядоченного набора результатов на конечные по размеру страницы, загружаемые по мере необходимости, а не целиком.
Debounce / ThrottleDebounce: эмитировать значение только после паузы не менее T мс после последнего события. Throttle: эмитировать не более одного значения за интервал T мс, игнорируя остальные.
Кэш hit/missHit: запрошенный элемент найден в кэше, обращение к источнику не требуется. Miss: элемент в кэше отсутствует; необходимо обращение к источнику с последующим сохранением результата.

Сеть и безопасность

ТерминОпределение
TLS handshakeПротокол установления защищённого соединения: стороны согласовывают версию и cipher suite, выполняют обмен ключами (ECDHE) и аутентификацию сервера по X.509-сертификату, после чего переходят к симметричному шифрованию трафика.
Симметричное vs асимметричное шифрованиеСимметричное: один и тот же ключ используется для шифрования и дешифрования. Асимметричное: пара ключей — открытый и закрытый; знание открытого ключа не позволяет восстановить закрытый.
Certificate / public-key pinningОграничение доверия на стороне клиента: принимать только заранее зафиксированный сертификат или публичный ключ, отвергая все прочие, включая выданные доверенными CA.
mTLS (mutual TLS)Расширение TLS, при котором сертификат предъявляет не только сервер, но и клиент; обе стороны взаимно верифицируют подлинность друг друга.
ХешДетерминированная функция H: {0,1}* → {0,1}ⁿ фиксированной выходной длины, обладающая свойствами: устойчивость к прообразу, устойчивость ко второму прообразу, устойчивость к коллизиям.
HMACКонструкция кода аутентификации сообщений на основе хеш-функции: HMAC(K, m) = H((K ⊕ opad) ‖ H((K ⊕ ipad) ‖ m)). Подтверждает целостность m и то, что отправитель владеет ключом K.
Token (JWT)Самодостаточный токен в формате JSON, подписанный (JWS) или зашифрованный (JWE); подпись верифицируется получателем локально без обращения к эмитенту.
Android KeystoreСистемный сервис, реализующий хранение и использование криптографических ключей в изолированном аппаратном окружении (TEE или StrongBox); ключевой материал не экспортируется в адресное пространство приложения.
ОбфускацияПреобразование программы, сохраняющее её семантику, но снижающее понятность исходного или промежуточного представления для обратного проектирования.
MITM (man-in-the-middle)Атака, при которой злоумышленник встраивается в канал связи между двумя сторонами, перехватывает сообщения и опционально модифицирует их, оставаясь незамеченным ни одной из сторон.

Задачи

Задача 1. Приложение падает с IllegalStateException: Can not perform this action after onSaveInstanceState при показе диалога из колбэка сети. Объясни причину и напиши обёртку над commitAllowingStateLoss, которая безопасно показывает DialogFragment только тогда, когда Activity гарантированно жива.

Показать решение 
fun FragmentActivity.showDialogSafely(
    dialog: DialogFragment,
    tag: String
) {
    if (isFinishing || isDestroyed) return
    // lifecycle.currentState < STARTED означает, что onSaveInstanceState уже вызван
    if (!lifecycle.currentState.isAtLeast(Lifecycle.State.STARTED)) return
    dialog.show(supportFragmentManager, tag)
}

// Пример вызова из колбэка Retrofit (уже на главном потоке):
lifecycleScope.launch {
    val result = api.fetchData()
    showDialogSafely(ResultDialog.newInstance(result), "result")
}

Объяснение: Android сохраняет back stack во время onSaveInstanceState; коммит транзакции после этой точки приводит к краши, потому что сохранённое состояние уже зафиксировано. Проверка Lifecycle.State.STARTED гарантирует, что мы находимся в промежутке между onStart и onSaveInstanceState. commitAllowingStateLoss только глушит исключение, но не фиксирует транзакцию надёжно — диалог может просто потеряться после рестора.

Задача 2. Воспроизведи и исправь утечку памяти: Activity содержит Handler(Looper.getMainLooper()) с отложенным сообщением на 30 секунд. Покажи, почему Handler удерживает Activity, и напиши корректную реализацию с WeakReference.

Показать решение 
// ПЛОХО: анонимный Handler захватывает this (Activity) в замыкании
class LeakyActivity : AppCompatActivity() {
    private val handler = Handler(Looper.getMainLooper())

    override fun onStart() {
        super.onStart()
        handler.postDelayed({
            // this — неявная ссылка на LeakyActivity
            title = "Done"
        }, 30_000)
    }
}

// ХОРОШО: явный WeakReference + отмена в onStop
class SafeActivity : AppCompatActivity() {

    private val handler = Handler(Looper.getMainLooper())
    private val ref = WeakReference(this)

    private val updateTitle = Runnable {
        ref.get()?.title = "Done"
    }

    override fun onStart() {
        super.onStart()
        handler.postDelayed(updateTitle, 30_000)
    }

    override fun onStop() {
        super.onStop()
        handler.removeCallbacks(updateTitle) // гарантируем отмену
    }
}

Объяснение: Handler кладёт Runnable в очередь главного Looper. Если лямбда захватывает Activity напрямую, GC не может собрать её до срабатывания таймера — это классическая утечка через GC root (MessageQueue). WeakReference разрывает сильную ссылку; removeCallbacks в onStop страхует от случаев, когда Activity уходит до срабатывания.

Задача 3. Предскажи вывод программы и объясни, почему он именно такой. Затем исправь её так, чтобы вывод был true.

data class User(val id: Int, val name: String)

fun main() {
    val a = User(1, "Alice")
    val b = User(1, "Alice")
    val map = HashMap<User, String>()
    map[a] = "admin"

    a.javaClass.getDeclaredField("id").let {
        it.isAccessible = true
        it.set(a, 99)  // мутируем поле id через рефлексию
    }

    println(map.containsKey(b))  // ???
}
Показать решение 
// Вывод: false
//
// Причина: hashCode объекта a вычислен при вставке в HashMap (id=1).
// После мутации id=99 его hashCode изменился — HashMap ищет a в другом бакете
// и не находит. b имеет id=1 и правильный hashCode, но в этом бакете нет записи.
//
// Исправление: не мутировать ключи хеш-коллекций.
// Если ключи должны меняться — удали старую запись перед мутацией и вставь заново.

fun main() {
    val a = User(1, "Alice")
    val b = User(1, "Alice")
    val map = HashMap<User, String>()
    map[a] = "admin"

    map.remove(a)                       // убираем до мутации
    a.javaClass.getDeclaredField("id").let {
        it.isAccessible = true
        it.set(a, 99)
    }
    // теперь a.id=99, b.id=1 — разные объекты, ищем b напрямую
    map[b] = "admin"

    println(map.containsKey(b))         // true
}

Объяснение: контракт HashMap требует, чтобы hashCode ключа не изменялся, пока объект находится в коллекции. Data-классы Kotlin вычисляют hashCode из всех val-параметров; рефлексия обходит это ограничение и приводит к «потере» ключа в коллекции — классическое нарушение контракта equals/hashCode.
