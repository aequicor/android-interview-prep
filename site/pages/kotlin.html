Middle Senior Частый вопрос

Null-safety: система типов против NPE

Kotlin делит типы на nullable (String?) и non-null (String) на уровне компилятора. Это не рантайм-проверка, а статическая гарантия: нельзя присвоить null в non-null переменную, нельзя вызвать метод на nullable-ссылке без явной проверки — компилятор не пропустит.

NullSafety.kt
val a: String = "ok"
// val b: String = null            // ❌ не компилируется
val c: String? = null              // ✅ nullable

val len1 = c?.length               // safe call → Int? (null, если c == null)
val len2 = c?.length ?: 0          // Elvis: подставит 0 при null
val len3 = c!!.length              // !! → NullPointerException, если c == null

c?.let { s -> println(s.length) }  // блок выполнится только при c != null

Платформенные типы из Java

Тип без аннотаций @Nullable/@NonNull Kotlin видит как платформенный String!: компилятор не знает, nullable он или нет, и отключает проверки. Разработчик вправе присвоить его в non-null переменную — и получить NPE в неожиданном месте.

⚠️
Подводный камень. val name: String = javaObject.getName() компилируется, даже если Java-метод вернёт null. NPE прилетит при присваивании, а стектрейс укажет на Kotlin-код, а не на Java-источник. На границе с Java явно указывайте nullable-тип или добавьте корректные Java-аннотации @Nullable/@NonNull.

Где именно компилятор вставляет runtime-проверки

Для чистого Kotlin null-safety работает статически. Но в байткод компилятор добавляет assertion там, где не может доказать безопасность на этапе компиляции:

lateinit var vs val … by lazy

Оба паттерна откладывают инициализацию, но по-разному. lateinit var — обещание компилятору «присвою до первого чтения»; поддерживается только для var с non-null типом, который не компилируется в JVM-примитив. На JVM backing field ссылочного lateinit изначально содержит null; getter проверяет это состояние и бросает UninitializedPropertyAccessException. by lazy — делегат: при первом обращении вызывается лямбда инициализации, результат кешируется в объекте-делегате.

Кто инициализируетВнешний код (DI, onCreate)Лямбда внутри объявления
ПотокобезопасностьНетПо умолчанию SYNCHRONIZED
Проверка инициализации::prop.isInitializedНе требуется — значение всегда есть после первого чтения
💡
Почему boxing не помогает. Int может бокситься в Integer в nullable-типах, дженериках, коллекциях или при присваивании в Any. Но lateinit var count: Int остаётся свойством типа Int, а не Integer?, поэтому компилятор отклоняет такую декларацию. Для поздней инициализации non-null числа используйте Delegates.notNull<Int>() или явный Int? с проверкой.
LateinitVsLazy.kt
class Screen {
    lateinit var presenter: Presenter       // приходит извне (DI / onCreate)

    val formatter: DateFormatter by lazy {  // SYNCHRONIZED по умолчанию
        DateFormatter(Locale.getDefault())
    }

    fun render() {
        if (::presenter.isInitialized) presenter.attach()
        println(formatter.now())   // lazy-блок отработает только один раз
    }
}
ℹ️
Режимы lazy. LazyThreadSafetyMode.SYNCHRONIZED — блокировка; одно вычисление гарантировано. PUBLICATION — лямбда может выполниться несколько раз, но в поле запишется результат первого завершившегося. NONE — без синхронизации, быстрее всего; безопасен только при однопоточном доступе (типично для UI-свойств).

lateinit в Activity и Fragment

В Activity lateinit var binding оправдан: binding создаётся в onCreate() и живёт до onDestroy() вместе с самой Activity. У Fragment иная модель: объект Fragment может оставаться в backstack, тогда как его View уже уничтожена в onDestroyView(). Использовать lateinit для view binding во Fragment нельзя — его невозможно обнулить, и он будет держать устаревшее дерево View.

ActivityVsFragmentBinding.kt
class ProfileActivity : AppCompatActivity() {
    private lateinit var binding: ActivityProfileBinding

    override fun onCreate(savedInstanceState: Bundle?) {
        super.onCreate(savedInstanceState)
        binding = ActivityProfileBinding.inflate(layoutInflater)
        setContentView(binding.root)
    }
}

class ProfileFragment : Fragment(R.layout.profile) {
    private var _binding: FragmentProfileBinding? = null
    private val binding get() = _binding!!   // backing property

    override fun onViewCreated(view: View, savedInstanceState: Bundle?) {
        _binding = FragmentProfileBinding.bind(view)
    }

    override fun onDestroyView() {
        _binding = null   // обрываем ссылку на View-дерево
        super.onDestroyView()
    }
}
Почему компилятор даёт smart cast для локальной val, но отказывает для nullable var-свойства, даже если строкой выше ты проверил его на null? ★

Smart cast безопасен только тогда, когда компилятор может доказать, что между проверкой и использованием значение не изменится. Локальная val это гарантирует — её нельзя переприсвоить. А мутабельное свойство (var) — особенно открытое, с кастомным геттером или объявленное в другом модуле — между проверкой if (x != null) и обращением x.length вполне может вернуть уже другое: другой поток переприсвоил поле, либо кастомный геттер при втором вызове отдаст null.

Поэтому компилятор отказывает с формулировкой «Smart cast to 'String' is impossible, because 'x' is a mutable property that could have been changed by this time». Обходы простые: скопировать в локальную val (val v = x; if (v != null) v.length), завернуть в ?.let { }, либо взять на себя ответственность через !!.

Классы: data, sealed, value, object

data class: что именно генерируется

Компилятор по свойствам первичного конструктора генерирует: equals()/hashCode() (структурное равенство), toString(), copy() и функции component1()…componentN() для деструктуризации. Свойства, объявленные в теле класса, в эти методы не попадают — ни в сравнение, ни в копирование.

DataClass.kt
data class User(val id: Long, val name: String) {
    var note: String = ""   // ⚠️ не учитывается в equals/hashCode/copy/toString
}

val u1 = User(1, "Ann")
val u2 = u1.copy(name = "Bob")        // copy: меняем часть полей, id остаётся
val (id, name) = u1                   // деструктуризация через component1/component2
println(u1 == User(1, "Ann"))         // true — структурное равенство

data class неявно final. Разрешить наследование невозможно без нарушения контракта equals: сравнение экземпляра базового класса с экземпляром подкласса нарушало бы симметричность. Реализовывать интерфейсы data class может.

🛑
Мутабельные поля в data class. Если поле первичного конструктора объявлено var, то изменение этого поля после добавления объекта в HashMap/HashSet сменит его hashCode, и элемент «потеряется» — найти или удалить его будет невозможно. Поля первичного конструктора в data class объявляйте val.

sealed class / sealed interface и исчерпывающий when

Sealed-тип — закрытое множество наследников, известных компилятору на этапе сборки. До Kotlin 1.5 все наследники должны были находиться в одном файле. С Kotlin 1.5 достаточно одного пакета внутри одного модуля — это открыло возможность разбивать большие иерархии по файлам. Исчерпывающий when без ветки else — главный практический бонус: добавьте нового наследника, и компилятор укажет на все незакрытые when.

UiState.kt
sealed interface UiState {
    data object Loading : UiState           // Kotlin 1.9+: toString() == "Loading"
    data class Success(val data: List<User>) : UiState
    data class Error(val message: String) : UiState
}

fun render(state: UiState) = when (state) {   // без else — компилятор знает все ветки
    UiState.Loading      -> showSpinner()
    is UiState.Success   -> showList(state.data)
    is UiState.Error     -> showError(state.message)
}
flowchart BT Loading["Loading: data object"] -.-> UiState["UiState: sealed interface"] Success["Success: data class"] -.-> UiState Error["Error: data class"] -.-> UiState
Закрытая иерархия sealed interface — компилятор знает всех наследников, исчерпывающий when без else

enum vs sealed

enum — фиксированный набор константных экземпляров одного типа: у всех одинаковая структура, различаются значения полей. sealed — набор разных типов, каждый со своей структурой и состоянием. Нужен общий контракт без данных — enum; ветки несут разные поля — sealed.

Вопрос со звёздочкой: что увидит Java

sealed не превращается в enum. На JVM sealed class остаётся абстрактным классом, sealed interface — интерфейсом, а его ветки компилируются в обычные классы: data class остаётся финальным классом с полями и сгенерированными методами, object — финальным singleton-классом с INSTANCE. Исчерпывающий when — проверка Kotlin-компилятора.

Разница появляется на целевом байткоде. При jvmTarget >= 17 компилятор пишет в class-файл атрибут PermittedSubclasses, поэтому Java 17 видит разрешённых наследников почти как в Java sealed-иерархии с permits. При target ниже 17 такого JVM-ограничения нет: Java видит обычный интерфейс или абстрактный класс; для Kotlin-кода закрытость иерархии проверяет Kotlin-компилятор по метаданным Kotlin.

JavaView.java (упрощённо)
// jvmTarget < 17: обычный интерфейс/классы + Kotlin metadata
public interface UiState {}

public final class Loading implements UiState {
    public static final Loading INSTANCE = new Loading();
    private Loading() {}
}

public final class Success implements UiState {
    // поля, getters, equals/hashCode/toString/copy
}

// jvmTarget >= 17: в class-файле есть PermittedSubclasses;
// Java 17 видит корень примерно так:
public sealed interface UiState permits Loading, Success, Error {}

object, companion object, data object

object — синглтон с ленивой потокобезопасной инициализацией, которую гарантирует загрузчик JVM-класса. companion object — единственный компаньон класса: через него обращаются к фабричным методам и константам. data object (Kotlin 1.9+) добавляет осмысленный toString() вида "Loading" вместо стандартного "Loading@1f5a02e" — полезно при логировании sealed-веток без состояния.

Objects.kt
object AppConfig { const val BASE_URL = "https://api.example.com" }  // синглтон

class User private constructor(val id: Long) {
    companion object {
        fun create(id: Long) = User(id)   // вызывается как User.create(1)
    }
}

data object Loading                    // toString() == "Loading"

@JvmInline value class

Оборачивает одно значение, но в рантайме компилятор по возможности разворачивает его в само значение — нет аллокации обёртки. Типобезопасность без накладных расходов: UserId и OrderId нельзя перепутать на уровне типов, хотя оба хранят Long.

ValueClass.kt
@JvmInline value class UserId(val value: Long)
@JvmInline value class OrderId(val value: Long)

fun load(id: UserId) { /* ... */ }
// load(OrderId(5))  // ❌ разные типы — компилятор не пропустит
⚠️
Когда boxing всё же происходит. Если value class используется как nullable (UserId?), в дженериках, в коллекциях или передаётся в Java-код — компилятор упаковывает его в реальный объект. Выигрыш в отсутствии аллокаций сохраняется только на прямых путях без дженериков и nullable.

Что копирует copy() — глубоко или поверхностно

copy() делает поверхностную копию: создаёт новый экземпляр и переносит в него ссылки на те же вложенные объекты. Если внутри лежит изменяемая коллекция, оригинал и копия будут указывать на один и тот же список — мутация через копию заденет оригинал.

Shallow copy: два разных объекта делят один вложенный список — мутация видна обоим

Второй нюанс: copy() собирает новый объект только по параметрам первичного конструктора и игнорирует свойства из тела класса — они инициализируются заново. Так что data class с состоянием в теле после copy() это состояние теряет. Для настоящей независимости вложенные изменяемые структуры копируйте вручную, а поля держите val.

Почему copy() у data class небезопасен с изменяемыми полями и что именно он копирует — глубоко или поверхностно? ★

copy() делает поверхностную (shallow) копию: новый экземпляр получает ссылки на те же вложенные объекты. Если внутри val items: MutableList<…>, то original и original.copy() ссылаются на ОДИН список — добавишь элемент через копию, и он появится в оригинале.

Плюс copy() переносит один-в-один только параметры первичного конструктора и не трогает свойства, объявленные в теле класса — те инициализируются заново по своему инициализатору. Поэтому состояние в теле после копирования сбрасывается. Безопасный data class — это val-поля и иммутабельные типы внутри; если вложена изменяемая структура, её надо клонировать вручную.

Тип-система: Any, Unit, Nothing

Три специальных типа формируют вершину и дно иерархии Kotlin и регулярно появляются на собеседованиях — особенно Nothing, который многие знают только по названию.

Any

Any — корень иерархии всех non-nullable типов, включая примитивные (Int, Boolean). В Java Object является суперклассом только ссылочных типов; int и Integer — принципиально разные сущности. В Kotlin этого разрыва нет. Под капотом Any компилируется в java.lang.Object; присваивание примитива в Any приводит к boxing.

flowchart BT N["Nothing"] --> String["String"] N --> Int["Int"] N --> T["...прочие типы"] String --> Any["Any"] Int --> Any T --> Any Any --> AQ["Any?"] Null["null"] --> AQ
Any? — вершина; Nothing — дно; null — элемент nullable-вселенной; Nothing подтип каждого типа

Any является non-nullable. Переменная типа Any не может содержать null; для этого нужна Any? — общий суперттип всех типов вообще. Три метода toString(), equals() и hashCode() унаследованы всеми классами именно от Any. Остальные методы java.lang.Object (например, wait() и notify()) через Any недоступны — нужно явное приведение к Object.

Unit

Unit соответствует void в Java, но, в отличие от него, является полноценным типом с единственным значением, которое тоже называется Unit. Это открывает возможность подставлять Unit как аргумент дженерика — в Java для этого пришлось бы заводить отдельный интерфейс или использовать Void с явным return null.

Unit.kt
interface Processor<T> { fun process(): T }

// Java потребовал бы Void и явного return null; Kotlin вставляет return Unit неявно
class NoResultProcessor : Processor<Unit> {
    override fun process() { /* делаем что-то */ }
}

Если функция объявлена без возвращаемого типа или с блочным телом и без return, компилятор подставляет : Unit автоматически. На JVM такая функция компилируется в void-метод, пока не используется как аргумент дженерика — тогда возвращается реальный Unit-объект.

Nothing

Nothing — нижний тип (bottom type): у него нет значений вообще. Применяется как возвращаемый тип функций, которые никогда не завершаются нормально — всегда бросают исключение или уходят в бесконечный цикл. Компилятор учитывает это при анализе потока управления: ветка с Nothing-функцией считается недостижимой.

Nothing.kt
fun fail(message: String): Nothing {
    throw IllegalStateException(message)
}

// Компилятор знает, что после fail() код не выполняется;
// поэтому address выводится как ненулевой String, а не String?
val address = company.address ?: fail("Нет адреса")
println(address.city)   // smart cast: address: String

// Стандартная error() из stdlib тоже возвращает Nothing
val value = map["key"] ?: error("Ключ не найден")
ℹ️
Nothing является подтипом каждого типа — поэтому выражение throw совместимо с любым ожидаемым типом. Конструкция val x: String = throw RuntimeException() компилируется: throw имеет тип Nothing, а Nothing — подтип String.
Какой тип у emptyList() и почему один и тот же объект можно присвоить и в List<String>, и в List<User>? ★

emptyList() возвращает List<Nothing> — внутри это синглтон-объект EmptyList. Nothing — нижний тип, подтип любого типа, а List ковариантен по элементу (List<out E>). Из ковариантности следует: раз Nothing — подтип String, то List<Nothing> — подтип List<String>. Поэтому одну и ту же пустую коллекцию-синглтон можно безопасно присвоить в List<String>, List<User> и куда угодно — без аллокации новой.

Тот же приём с Nothing работает для веток sealed без полезной нагрузки: data object Loading : Result<Nothing> подходит под Result<любой T> именно потому, что Result ковариантен, а Nothing — подтип всего.

Extension-функции, inline и операторные конвенции

Extension'ы — статическая диспетчеризация

Extension-функция не модифицирует класс и не имеет доступа к его private-членам. Компилятор превращает её в статический метод, где приёмник передаётся первым аргументом. Отсюда два следствия: резолюция статическая — по объявленному типу переменной, а не по рантайм-классу; если у класса есть метод-член с той же сигнатурой, член всегда победит.

Extensions.kt
fun String.shout(): String = uppercase() + "!"
// ~ компилируется в: static String shout(String $receiver) { ... }

open class A
class B : A()
fun A.who() = "A"
fun B.who() = "B"

val ref: A = B()
println(ref.who())   // "A" — резолвится по СТАТИЧЕСКОМУ типу ref (A), не по B!

inline, noinline, crossinline

Каждая лямбда без inline — это объект (Function-инстанс): аллокация в куче + вызов через виртуальный invoke(). На горячих путях (цепочки операций с коллекциями) это создаёт давление на GC. Модификатор inline подставляет тело функции и тела лямбд прямо в точку вызова — лямбда исчезает как объект. Исключение: если лямбда захватывает переменные из окружающего контекста, SAM-конвертация с Java по-прежнему создаёт объект для каждого вызова.

МодификаторЧто делаетЗачем
inline (на функции)Встраивает тело функции и лямбд в точку вызоваУбрать аллокацию лямбд; открывает reified и non-local return
noinline (на параметре)Конкретную лямбду НЕ встраиватьКогда лямбду нужно сохранить или передать дальше как объект
crossinline (на параметре)Встроить, но запретить non-local returnЛямбда вызывается из другого контекста (например, из Runnable)
⚠️
Обратная сторона inline. Встраивание дублирует байткод в каждой точке вызова — большая inline-функция раздувает методы и может выйти за лимиты JVM. Модификатор оправдан для небольших higher-order функций и там, где нужны reified или non-local return.

reified-дженерики

На JVM дженерики стираются (type erasure): в рантайме тип T неизвестен, нельзя написать is T или T::class. В inline-функции конкретный тип известен в точке вызова, поэтому reified T «материализуется» — компилятор подставляет реальный класс прямо в байткод.

Reified.kt
inline fun <reified T> Gson.fromJson(json: String): T =
    fromJson(json, T::class.java)   // T известен — type erasure обойдён

val user: User = gson.fromJson(jsonString)   // без передачи Class<T> вручную

inline fun <reified T> List<*>.filterByType(): List<T> =
    filterIsInstance<T>()   // стандартная реализация через reified

Операторные конвенции и componentN

В Kotlin ряд операторов компилируется в вызов конвенционального метода с ключевым словом operator. Так реализуется, в частности, деструктуризация: val (a, b) = pair — это val a = pair.component1(); val b = pair.component2(). data class генерирует componentN-функции автоматически по каждому свойству первичного конструктора.

Operators.kt
data class Vector(val x: Double, val y: Double) {
    operator fun plus(other: Vector) = Vector(x + other.x, y + other.y)
    operator fun times(factor: Double) = Vector(x * factor, y * factor)
}

val v = Vector(1.0, 2.0) + Vector(3.0, 4.0)   // вызывает operator plus
println(v * 2.0)                                // вызывает operator times

// Для не-data-класса componentN пишутся вручную
class Point(val x: Int, val y: Int) {
    operator fun component1() = x
    operator fun component2() = y
}

for ((x, y) in listOf(Point(1, 2), Point(3, 4))) { println("$x, $y") }

Конвенция invoke делает объект вызываемым как функцию — полезна в DSL и при моделировании функциональных объектов с состоянием:

InvokeConvention.kt
class Multiplier(private val factor: Int) {
    operator fun invoke(value: Int) = value * factor
}

val triple = Multiplier(3)
println(triple(5))   // 15 — вызов через invoke без явного .invoke()
Что напечатает val s: String? = null; println(s.toString()) — упадёт с NPE или нет? ★

Напечатает строку "null" и не упадёт. Extension можно объявить на nullable-приёмнике: fun Any?.toString(): String — именно такая функция и есть в stdlib. Поскольку extension компилируется в статический метод, где приёмник — обычный первый аргумент, передать туда null совершенно законно: это не вызов метода на объекте, а вызов статической функции с аргументом null.

Внутри тела this имеет тип Any?, и его надо проверять руками. Канонический Any?.toString() для null-приёмника возвращает "null" вместо падения. Поэтому в примере вызывается extension, а не «метод на объекте», и NPE не возникает.

Scope-функции: let / run / with / apply / also

Пять функций различаются по двум осям: как они ссылаются на объект (this — неявный приёмник vs аргумент it) и что возвращают (сам объект vs результат лямбды). Понимание этих осей снимает большую часть путаницы.

ФункцияКонтекстВозвращаетКогда применять
letitРезультат лямбдыNull-safe цепочки (?.let), трансформация значения
runthisРезультат лямбдыКонфигурация + вычисление результата в одном блоке
withthisРезультат лямбдыНесколько вызовов одного объекта (не extension)
applythisСам объектНастройка объекта (builder-стиль), возврат самого объекта

ScopeFunctions.kt
// apply — настроить и вернуть объект (builder-стиль)
val paint = Paint().apply {
    color = Color.RED
    isAntiAlias = true
}

// let — выполнить только при non-null + получить результат
val length: Int = nullableName?.let { it.trim().length } ?: 0

// also — вклиниться с побочным эффектом, не разрывая цепочку
val users = repository.load()
    .also { Log.d("TAG", "loaded ${it.size}") }
    .filter { it.isActive }

// run — конфигурация + результат
val message = StringBuilder().run {
    append("Hello, ")
    append(name)
    toString()
}
💡
Практическое правило. apply и also возвращают объект — пишите в цепочку при настройке или side-эффектах. let, run, with возвращают результат лямбды — используйте, когда нужно вычислить нечто на основе объекта. let vs run: если объект нужен во внешней области видимости под другим именем — let { it }; если переопределяете this внутри блока — run.

Вся путаница снимается, если держать в голове ровно два вопроса — что вернуть и как обращаться к объекту:

flowchart TD Start["Нужна scope-функция"] --> Q1{"Что вернуть?"} Q1 -->|"сам объект"| Q2{"Как к объекту?"} Q1 -->|"результат лямбды"| Q3{"Как к объекту?"} Q2 -->|"this"| Apply["apply"] Q2 -->|"it"| Also["also"] Q3 -->|"this"| RunWith["run / with"] Q3 -->|"it"| Let["let"]
Две оси выбора: возвращаемое значение × способ обращения к приёмнику
Чем ?.let{} отличается от ?.also{}, если внутри блока вернуть null, и почему apply иногда молча проглатывает баг? ★

let возвращает результат лямбды, поэтому x?.let { transform(it) } даёт null двумя путями: либо x был null (блок не выполнился), либо блок сам вернул null. also же возвращает приёмник и игнорирует значение лямбды — x?.also { it.log() } всегда вернёт исходный x (или null, если x был null), что бы ни насчитали внутри.

Подвох apply: он тоже возвращает приёмник, а не результат блока. Напишешь по инерции val n = sb.apply { length } — получишь не длину, а сам StringBuilder, и компилятор промолчит, тип-вывод проглотит ошибку. Правило: считаешь значение из объекта — let/run/with; настраиваешь объект и возвращаешь его — apply/also.

Дженерики и вариативность

По умолчанию дженерики в Kotlin инвариантны: Box<String> не является подтипом Box<Any>, даже при String : Any. Инвариантность безопасна, но иногда избыточно строга — на помощь приходят out и in.

МодификаторВариативностьРольПравило подтипирования
out TКовариантностьProducer (только отдаёт T)Box<String> → подтип Box<Any>
in TКонтравариантностьConsumer (только принимает T)Box<Any> → подтип Box<String>
(без модификатора)ИнвариантностьИ читает, и пишетПодтипирования по T нет

Мнемоника PECS из Java работает и в Kotlin: Producer — Extends (out), Consumer — Super (in). out запрещает использовать T во входных позициях — нельзя «положить». in запрещает использовать T в выходных позициях — нельзя «достать» как T, только как Any?.

Variance.kt
// out: только производит T → ковариантность
interface Source<out T> { fun next(): T }
val anySource: Source<Any> = object : Source<String> { override fun next() = "x" }  // ✅

// in: только потребляет T → контравариантность
interface Sink<in T> { fun put(item: T) }
val stringSink: Sink<String> = object : Sink<Any> { override fun put(item: Any) {} } // ✅

// star-projection: тип не важен, но безопасность сохраняется
fun printSize(box: Source<*>) { /* читаем как Any?, конкретный T неизвестен */ }

Declaration-site vs use-site variance

out/in на самом типе (interface Source<out T>) — это declaration-site variance: вариативность фиксирована и работает везде, где используется тип. Kotlin поддерживает и use-site variance: указать out/in прямо в месте использования на обычном инвариантном типе.

UseSiteVariance.kt
// MutableList инвариантен, но при передаче только для чтения добавляем out
fun copyAll(from: MutableList<out Any>, to: MutableList<Any>) {
    // from.add("x")  // ❌ out запрещает запись
    to.addAll(from)    // ✅ читать безопасно
}

// in — передаём только для записи
fun fillWith(list: MutableList<in String>, value: String) {
    list.add(value)    // ✅
    // val s: String = list[0]  // ❌ in запрещает чтение как String
}
ℹ️
Star-projection <*> — «какой-то конкретный, но неизвестный тип». Не то же, что <Any?>: MutableList<*> можно читать (элементы приходят как Any?), но нельзя в него писать — компилятор не знает реальный T. По семантике эквивалентно <out Any?> для чтения и <in Nothing> для записи.

PECS: где можно читать, а где писать

Интуиция out = «отдаёт», in = «принимает» закрепляется одной таблицей — что разрешено в каждой роли:

ТипЧитать как TПисать TКем выступает
Producer<out T>данетисточник: только отдаёт
Box<T> (инвариант)дадаи читает, и пишет
Consumer<in T>нет (только как Any?)даприёмник: только принимает
Почему один параметр типа не может быть одновременно out и in, и где компилятор запретит out T во входной позиции? ★

out T означает «T появляется только в out-позициях» — как возвращаемые значения, но не как типы параметров методов; in T — ровно наоборот. Это не стилистика, а гарантия типобезопасности. Будь у ковариантного Producer<out T> метод fun consume(t: T), через Producer<Any> = producerOfString можно было бы передать туда Int и сломать рантайм. Поэтому компилятор выдаёт «Type parameter T is declared as 'out' but occurs in 'in' position».

Исключение — приватные члены: они не видны снаружи, и проверку вариантности можно ослабить через @UnsafeVariance. Классика — Function1<in P, out R>: контравариантен по аргументу и ковариантен по результату. Две разные роли требуют двух разных параметров; один параметр обе сразу нести не может.

Generic-функции, ограничения и where

Вариативность — это про подтипирование. А есть отдельный вопрос, который любят задавать рядом: как вообще объявить дженерик-функцию и как ограничить её параметр. У generic-класса параметр живёт на уровне всего класса (class Box<T>); у generic-функции — только на время одного вызова (fun <T> singletonList(item: T)), на каждый вызов T выводится заново.

По умолчанию верхняя граница параметра — Any?. То есть T может оказаться nullable, и это частый недосмотр: fun <T> T.orThrow() примет и null. Чтобы сузить, ставят upper bound через двоеточие.

Constraints.kt
// Одно ограничение — в угловых скобках. Теперь T обязан быть Comparable
fun <T : Comparable<T>> max(a: T, b: T): T = if (a > b) a else b

// T : Any убирает nullable из верхней границы
fun <T : Any> T.orThrow(msg: String): T = this   // null сюда уже не передать

// Несколько ограничений — одно в <>, остальные в where
fun <T> copyWhenReady(source: T): T
    where T : CharSequence,
          T : Comparable<T> {
    // внутри доступны методы обоих интерфейсов
    return source
}

Когда ограничение одно — оно пишется в <>. Когда их несколько — первое всё равно в скобках, а дальше идёт where со списком. Все ограничения проверяются на компиляции, поэтому внутри функции T уже несёт методы перечисленных интерфейсов. С reified (см. inline и reified) это сочетается: inline fun <reified T : Activity> и ограничивает тип, и материализует его в байткоде.

ℹ️
@UnsafeVariance как escape hatch. Если вариантность параметра мешает в одном узком месте (например, метод contains(e: T) в ковариантной коллекции, где e используется только для сравнения и наружу не утекает), аннотация @UnsafeVariance отключает проверку именно для этой позиции. Это берёт безопасность на себя — применяйте, только когда действительно доказали, что нарушения типов не будет.
Что выведет компилятор для fun <T> T.f() без явной границы — допустим ли там nullable, и как это поправить? ★

Верхняя граница по умолчанию — Any?, а не Any. Значит, для fun <T> T.f() внутри тела this имеет тип T, который может быть nullable, и вызвать f() на null компилятор разрешит. Это типичный источник скрытых NPE в «универсальных» хелперах.

Поправка — явная граница T : Any: fun <T : Any> T.f(). Теперь T гарантированно non-null, и попытка nullable.f() не скомпилируется. Тот же приём в stdlib: многие функции вроде checkNotNull и ряд builder-хелперов ограничивают T : Any именно чтобы не пропустить null вглубь.

Делегирование: by для классов и свойств

Делегирование класса

Конструкция class C : Iface by impl заставляет компилятор сгенерировать все методы интерфейса как форвардинг к impl. Это композиция вместо наследования без boilerplate: реализуем интерфейс, опираясь на готовый объект, и при желании переопределяем отдельные методы.

ClassDelegation.kt
interface Repository { fun load(): List<User> }

class CachingRepository(
    private val origin: Repository
) : Repository by origin {           // load() автоматически форвардируется в origin
    private var cache: List<User>? = null
    override fun load(): List<User> = cache ?: origin.load().also { cache = it }
}

Делегаты свойств

Свойство может делегировать чтение/запись объекту, у которого есть операторные функции getValue/setValue. Стандартные делегаты из stdlib:

ДелегатПоведение

Delegates.observable(init){ p, old, new -> }Колбэк после каждого изменения var
Delegates.vetoable(init){ p, old, new -> Bool }Колбэк до изменения; false отменяет присваивание

import kotlin.properties.Delegates

class Form {
    var name: String by Delegates.observable("") { _, old, new ->
        println("name: $old -> $new")
    }
    var age: Int by Delegates.vetoable(0) { _, _, new -> new in 0..150 }
}

// Кастомный делегат: любой объект с getValue/setValue
class Prefs(private val sp: SharedPreferences) {
    operator fun getValue(ref: Any?, p: KProperty<*>): String =
        sp.getString(p.name, "") ?: ""
    operator fun setValue(ref: Any?, p: KProperty<*>, value: String) =
        sp.edit().putString(p.name, value).apply()
}

class Settings(prefs: SharedPreferences) {
    var token: String by Prefs(prefs)   // делегат берёт имя свойства как ключ
}
💡
provideDelegate. Если делегату нужна информация о свойстве (например, его имя) в момент создания, а не при чтении/записи, можно добавить оператор provideDelegate. Компилятор вызовет его при инициализации класса и передаст KProperty до первого getValue. Это позволяет, например, проверить имя свойства заранее или зарегистрировать его в реестре.
При class C : Iface by impl внутренний вызов одного метода интерфейса из другого попадёт в твоё переопределение в обёртке или в реализацию делегата? ★

Попадёт в реализацию делегата, мимо твоего переопределения. class C : Iface by impl генерирует форвардинг-методы, каждый из которых просто дёргает impl.method(). Если внутри impl один метод интерфейса вызывает другой метод того же интерфейса, он вызовет реализацию внутри самого impl — про существование обёртки C делегат не знает, ссылки на её this у него нет.

То есть открытой рекурсии и виртуальной диспетчеризации через обёртку здесь нет: переопределив метод в C, ты не «перехватываешь» внутренние вызовы делегата. Частый сюрприз — думаешь, что подменил поведение, а делегат продолжает звать свою старую реализацию.

Интероп с Java: коллекции, SAM и аннотации

SAM-конвертация

Java API полон функциональных интерфейсов (Single Abstract Method): Runnable, Callable, Comparator, слушатели в Android. Kotlin позволяет передавать лямбду туда, где ожидается SAM-интерфейс — компилятор автоматически оборачивает её в анонимный класс-реализацию. Если лямбда не захватывает переменные из внешнего контекста, анонимный объект создаётся один раз и переиспользуется.

SAM.kt
// Java: void postponeComputation(int delay, Runnable r)
postponeComputation(1000) { println(42) }  // лямбда → Runnable; один объект на программу

// Если лямбда захватывает переменную — новый объект на каждый вызов
fun handleComputation(id: String) {
    postponeComputation(1000) { println(id) }  // new Runnable на каждый вызов
}

// SAM-конструктор — явное преобразование для сохранения инстанса
val allDone = Runnable { println("all done") }
button.setOnClickListener(allDone)   // один и тот же объект при повторном использовании
ℹ️
fun interface в Kotlin. Для Kotlin-SAM используйте ключевое слово fun interface (Kotlin 1.4+). Обычный интерфейс с одним методом SAM-конвертацию не получит — только Java-интерфейсы и Kotlin fun interface. Это намеренно: без явного маркера компилятор не может вывести, какой метод считать «тем самым».
FunInterface.kt
fun interface Predicate<T> { fun test(value: T): Boolean }

fun findFirst(list: List<Int>, pred: Predicate<Int>): Int? =
    list.firstOrNull { pred.test(it) }

val result = findFirst(listOf(1, 2, 3)) { it > 1 }   // лямбда → Predicate<Int>

List vs MutableList и Sequence

List в Kotlin — read-only интерфейс, а не гарантия иммутабельности: за ним может скрываться изменяемый ArrayList. Read-only означает «через эту ссылку менять нельзя», а не «объект неизменяем».

Операции над List (map, filter) энергичные: каждая создаёт промежуточную коллекцию. Sequence — ленивый конвейер: элементы протягиваются по одному, промежуточных коллекций нет, и терминальная операция запускает обработку.

flowchart LR A["Элемент 1"] --> F1{filter} F1 -->|прошёл| M1["map"] M1 --> T["toList"] A2["Элемент 2"] --> F2{filter} F2 -->|отброшен| X["стоп"] A3["Элемент 3"] --> F3{filter} F3 -->|прошёл| M3["map"] M3 --> T
Sequence: каждый элемент проходит всю цепочку по одному — без промежуточных списков
Sequence.kt
// List: 2 промежуточных списка, оба полностью обрабатываются
val r1 = (1..1_000_000).toList()
    .map { it * 2 }       // список из 1 000 000
    .filter { it > 10 }   // ещё один список
    .take(3)              // берём 3

// Sequence: ленивый конвейер — обработается несколько элементов, затем стоп
val r2 = (1..1_000_000).asSequence()
    .map { it * 2 }
    .filter { it > 10 }
    .take(3)              // short-circuit: остановится после 3 подходящих
    .toList()
💡
Когда что. Sequence быстрее на больших данных, длинных цепочках и при коротком замыкании (take/first/find) — экономит память. На маленьких коллекциях Sequence медленнее из-за оверхеда итераторов — обычный List предпочтительнее.

@Throws — checked-исключения для Java

В Kotlin нет checked exceptions: любая функция может бросить что угодно, ничего не объявляя. Когда такой метод зовут из Java, Java-компилятор не знает, что он бросает, скажем, IOException, и не заставит её обрабатывать (а попытка её поймать даст «exception is never thrown»). Аннотация @Throws(IOException::class) дописывает throws в сигнатуру байткода — Java видит checked-исключение и может/обязана его обработать. На сам Kotlin-код она не влияет, это чисто интероп-аннотация, как @JvmStatic.

Throws.kt
@Throws(IOException::class)
fun readConfig(path: String): String = File(path).readText()
// Java увидит: String readConfig(String path) throws IOException
Почему передача Kotlin List<String> в Java-метод, который делает list.add(...), компилируется, но падает в рантайме? ★

Read-only List и MutableList — разные интерфейсы только на стороне Kotlin; на уровне JVM оба отображаются в один java.util.List. Когда Kotlin-List пересекает границу, Java видит обычный java.util.List со всеми мутирующими методами (add, remove), и запретить это компилятор не может.

Если под капотом лежал реально неизменяемый список (например, результат listOf(...)), вызов add бросит UnsupportedOperationException в рантайме. Это «mutation leak» через интероп: Kotlin-гарантия read-only не переживает переход в Java. Защита — отдавать в Java явную копию (ArrayList(list)) либо чётко документировать контракт.

Равенство, иммутабельность, const val

== — структурное равенство: компилируется в a?.equals(b) ?: (b === null). === — ссылочное равенство: один объект в куче. Для data class equals сгенерирован по полям первичного конструктора, поэтому == сравнивает значения.

Equality.kt
val a = User(1, "Ann")
val b = User(1, "Ann")
println(a == b)    // true  — структурно равны (data class equals)
println(a === b)   // false — разные объекты в памяти

// Контракт: if (a == b) → a.hashCode() == b.hashCode()
// Нарушите — объект «потеряется» в HashMap/HashSet
🛑
Контракт equals/hashCode. Равные объекты обязаны иметь равный hashCode. Если переопределить только equals, объект исчезнет из HashMap/HashSet. Никогда не кладите в hash-коллекцию data class с var-полями в первичном конструкторе: изменив поле после вставки, вы сделаете объект ненаходимым.

Контракт equals и ручной hashCode

Для data class всё генерируется, но не-data классы (и собеседование) требуют написать руками — и тут важен сам контракт. equals обязан быть рефлексивным (x.equals(x) — true), симметричным (a.equals(b) ⇔ b.equals(a)), транзитивным (a==b и b==c ⇒ a==c), консистентным (повторные вызовы дают тот же результат, пока объект не менялся) и x.equals(null) — всегда false.

Самая частая ловушка — симметричность при наследовании. Поэтому класс сверяют через javaClass != other.javaClass, а не через is: с is наследник мог бы считаться равным базовому, но обратное сравнение дало бы false — асимметрия.

ManualEquals.kt
class Money(val amount: Long, val currency: String) {
    override fun equals(other: Any?): Boolean {
        if (this === other) return true
        if (other == null || javaClass != other.javaClass) return false
        other as Money
        return amount == other.amount && currency == other.currency
    }

    override fun hashCode(): Int {
        var result = amount.hashCode()
        result = 31 * result + currency.hashCode()   // нечётное простое 31
        return result
    }
}

Каноничная схема hashCode: взять хеш первого значимого поля и для каждого следующего делать result = 31 * result + field.hashCode(). Множитель 31 — нечётное простое, минимизирует коллизии и оптимизируется в сдвиг (31 * x == (x << 5) - x). Главный инвариант: a == b ⇒ a.hashCode() == b.hashCode() (обратное не требуется). В обычном коде проще Objects.hash(...) — он чуть медленнее из-за vararg-массива и boxing, но читаемее.

Иммутабельность: val ≠ immutable

val запрещает переприсваивание ссылки, но не делает объект неизменяемым. val list = mutableListOf(1) — список можно менять, нельзя лишь присвоить новую ссылку. Настоящая иммутабельность — это неизменяемый тип: read-only интерфейс List, data class с только val-полями в первичном конструкторе.

const val vs val

const val — compile-time константа: значение встраивается непосредственно в байткод в местах использования (инлайнинг). Допустимо только для примитивов и String, только на верхнем уровне или в object/companion object. Обычный val вычисляется в рантайме и хранится в поле с геттером.

Const.kt
object Api {
    const val VERSION = "v1"                        // зашивается в байткод (инлайн)
    val timestamp = System.currentTimeMillis()       // рантайм — const здесь невозможен
}

typealias vs value class

typealias — это только псевдоним для существующего типа, нового типа он не создаёт и типобезопасности не даёт: typealias UserId = Long и обычный Long полностью взаимозаменяемы, перепутать UserId и OrderId компилятор не помешает. Он нужен для читаемости — сократить длинный generic-тип или развести одноимённые классы из разных пакетов. value class (см. выше в разделе о классах), наоборот, создаёт отдельный тип с проверкой на компиляции, но без аллокации обёртки на прямых путях.

Typealias.kt
typealias Handler = (Event, Context) -> Unit   // читаемость для длинного типа
typealias UserId = Long                          // НЕ новый тип — не защищает от путаницы

// нужна типобезопасность, а не только читаемость → value class
@JvmInline value class OrderId(val value: Long)
Сгенерированный equals у data class сравнивает поля через == или ===, и что будет с полем Double, равным NaN, или с -0.0 против 0.0? ★

Сгенерированный equals сравнивает каждое поле структурно (через equals с null-безопасностью), а не по ссылке. Но для Double/Float это даёт неинтуитивный результат, потому что компилятор использует семантику java.lang.Double.equals, а не примитивное ==.

Отсюда два сюрприза. Примитивное NaN == NaN — false, а вот два data class с полем NaN окажутся РАВНЫ: boxed Double.equals считает NaN равным NaN. И наоборот: 0.0 и -0.0 примитивно равны, но как поля data class они НЕ равны (Double.equals их различает), и hashCode у них разный. Поэтому data class с floating-point полями ведёт себя в HashSet/HashMap не так, как ждёшь от обычного ==.

Частые вопросы интервьюера

Чем lateinit отличается от by lazy? Middle+

lateinit var — для var с non-null ссылочным типом; Int/Boolean нельзя, даже если эти значения могут бокситься в других контекстах. Инициализируется извне (DI, onCreate), не потокобезопасен. До инициализации бросает UninitializedPropertyAccessException, проверка — ::prop.isInitialized.

by lazy — для val, любого типа; инициализируется сам при первом чтении лямбдой, кеширует результат; по умолчанию SYNCHRONIZED.

Правило выбора: значение приходит снаружи и может меняться → lateinit; вычисляется один раз и лениво → by lazy. Для view binding во Fragment lateinit не подходит: View живёт короче Fragment, binding нужно обнулять в onDestroyView().

Что такое Nothing и зачем он в типовой системе? Senior

Nothing — нижний тип (bottom type): у него нет значений. Используется как возвращаемый тип функций, которые никогда не возвращают управление — всегда бросают исключение или зависают в бесконечном цикле (fun fail(…): Nothing, стандартная error()).

Nothing является подтипом каждого типа в Kotlin. Это позволяет компилятору считать ветку с Nothing-функцией недостижимой и применять smart cast: val addr = company.address ?: fail("нет адреса") → addr выводится как String, а не String?. По той же причине выражение throw совместимо с любым ожидаемым типом.

В чём разница Unit и void из Java? Middle

void в Java — это не тип, а ключевое слово-метка «функция ничего не возвращает». Его нельзя использовать как аргумент дженерика: попытка написать Processor<void> провалится.

Unit в Kotlin — полноценный тип с единственным значением (тоже Unit). Его можно подставить как аргумент дженерика: class Processor : Action<Unit> и не нужен явный return. На JVM Unit-функция компилируется в void-метод, пока не используется в дженерическом контексте.

Что именно генерирует data class и почему его нельзя наследовать? Middle

По свойствам первичного конструктора генерируются equals, hashCode, toString, copy и функции componentN. Свойства в теле класса в них не входят. data class неявно final: наследование нарушило бы контракт equals (симметричность при сравнении экземпляра базового и дочернего типа). Реализовывать интерфейсы data class может.

Что такое SAM-конвертация и когда объект создаётся повторно? Middle+

SAM-конвертация (Single Abstract Method) — автоматическое преобразование лямбды в экземпляр Java-интерфейса с одним абстрактным методом (Runnable, Callable и т. п.). Компилятор генерирует анонимный класс-реализацию.

Если лямбда не захватывает переменные из внешнего контекста, анонимный объект создаётся один раз и переиспользуется между вызовами. Если лямбда захватывает переменные — новый объект при каждом вызове, поскольку захваченные значения хранятся в полях объекта. Для Kotlin-интерфейсов SAM работает только с fun interface (Kotlin 1.4+).

Когда NPE появляется при Java→Kotlin interop? Senior

Java API без nullability-аннотаций возвращает платформенный тип T!. Kotlin разрешает трактовать его и как T, и как T?. При выборе non-null T компилятор вставляет runtime assertion. Поэтому val s: String = java.getName() падает сразу после Java-вызова, если тот вернул null. То же при передаче platform value в non-null Kotlin-параметр. Лечение: явно принимать T? и обрабатывать null, либо добавлять Java-аннотации @Nullable/@NonNull.

Во что компилируется extension-функция? Полиморфна ли она? Senior

В статический метод, где приёмник — первый аргумент. Она не модифицирует класс и не видит его private-членов. Резолвится статически — по объявленному типу переменной, а не по рантайм-классу. Поэтому extension не участвует в полиморфизме: если у A и B : A есть одноимённые extension'ы и переменная объявлена как val ref: A = B(), вызовется версия для A. Если у класса есть метод-член с той же сигнатурой, член всегда побеждает extension.

Зачем inline и что такое reified? Senior

Лямбда без inline — это объект (Function-инстанс): аллокация + вызов через виртуальный invoke(). На горячих путях это нагрузка на GC. inline встраивает тело функции и лямбд в точку вызова — лямбда исчезает как объект. Также открывает non-local return и reified-дженерики.

На JVM дженерики стираются: в рантайме T неизвестен, нельзя is T или T::class. В inline-функции конкретный тип подставляется в точку вызова, поэтому reified T материализуется — компилятор знает его и вставляет в байткод. Минус inline: дублирование байткода; оправдан для небольших функций.

Что такое операторные конвенции и как работает деструктуризация? Middle+

Kotlin преобразует ряд операторов в вызовы конвенциональных методов с ключевым словом operator: + → plus, * → times, вызов объекта → invoke, и т. д.

Деструктуризация val (a, b) = obj — сахар над val a = obj.component1(); val b = obj.component2(). data class генерирует componentN-функции автоматически по свойствам первичного конструктора. Для собственного класса достаточно объявить operator fun component1() = … и т. д. Деструктуризация в for-цикле и лямбдах работает так же.

Когда Sequence быстрее List, а когда медленнее? Middle+

Sequence вычисляется лениво и поэлементно: цепочка map.filter.first() не создаёт промежуточных коллекций и обрывается при коротком замыкании. Это выигрывает на больших/бесконечных источниках и длинных цепочках с take/first/find.

На маленьких коллекциях List быстрее: операции над массивами кэш-дружелюбны, а у Sequence есть оверхед на итераторы и потенциальный boxing примитивов. Правило: менее 1000 элементов и короткая цепочка — List; иначе — взвесить.

В чём разница == и ===? Middle

== — структурное равенство, компилируется в a?.equals(b) ?: (b === null). === — ссылочное равенство (тот же объект в куче). Для data class equals сгенерирован по полям первичного конструктора.

Контракт: если a == b, то a.hashCode() == b.hashCode(). Нарушение ломает HashMap/HashSet: объект кладётся в одну корзину, а ищется в другой — и «исчезает».

Перечисли контракт equals и напиши корректный hashCode вручную для не-data класса. Middle+

equals обязан быть рефлексивным (x.equals(x)), симметричным (a.equals(b) ⇔ b.equals(a)), транзитивным (a==b и b==c ⇒ a==c), консистентным (повторные вызовы стабильны, пока объект не менялся) и x.equals(null) == false. Чтобы не сломать симметричность при наследовании, класс проверяют через other?.javaClass != javaClass, а не через is.

Каноничный hashCode: начать с хеша первого поля и для каждого следующего делать result = 31 * result + field.hashCode(). Нечётное простое 31 минимизирует коллизии и сворачивается в сдвиг. Главный инвариант: равные объекты ⇒ равный hashCode; обратное не требуется. В проде проще Objects.hash(...) или data class, но на собесе просят руками.

Чем generic-функция отличается от generic-класса и зачем upper bound и where? Middle+

У generic-класса параметр живёт на уровне всего класса (class Box<T>), у generic-функции — только на время вызова (fun <T> singletonList(item: T)), T выводится заново на каждый вызов. Верхняя граница по умолчанию — Any? (то есть nullable допустим).

Чтобы сузить, пишут upper bound: fun <T : Comparable<T>> max(...). Несколько ограничений — одно в угловых скобках, остальные в where: fun <T> copy(...) where T : CharSequence, T : Comparable<T>. Это даёт доступ к методам обоих интерфейсов внутри функции и проверяется на компиляции.

Почему лямбда в Kotlin может менять захваченную var, а в Java — нет? Middle+

Локальная функция (объявленная внутри другой) и лямбда видят локальные переменные внешней функции — это замыкание. В Java лямбда захватывает только effectively final переменные. Kotlin позволяет читать И изменять захваченную var: под капотом компилятор оборачивает такую переменную в объект-ссылку (Ref.IntRef / Ref.ObjectRef) и захватывает ссылку, а не значение — поэтому изменения видны снаружи.

Цена — лишняя аллокация обёртки и риск гонок при доступе из разных потоков. Отдельный подводный камень: лямбда, захватившая this или Context и пережившая владельца (колбэк, статическая ссылка), держит его и течёт по памяти.

В каком порядке выполняются init-блоки, инициализаторы свойств и вторичный конструктор? Middle+

Сначала отрабатывает первичный конструктор: инициализаторы свойств и init-блоки выполняются в текстовом порядке сверху вниз, как единый блок. Поэтому init не может читать свойство, объявленное ниже него. Вторичный конструктор обязан делегировать первичному через this(...), и тело первичного (со всеми init) выполняется ДО тела вторичного.

Классическая ловушка наследования: при создании наследника тело базового конструктора выполняется раньше инициализаторов наследника. Если open-метод, вызванный в базовом init, переопределён в наследнике, он увидит ещё не инициализированные поля наследника (null/0). Отложить помогает lateinit или by lazy.

Задача 1. Реализуй функцию parseUser, которая принимает JSON-строку через библиотеку с Java API — метод JavaParser.parse(json) возвращает platform type User! и может вернуть null. Функция должна вернуть non-null User либо пользователя по умолчанию — без использования !! и без риска NPE.

Показать решение 
data class User(val id: Long, val name: String)

// Имитация Java-класса с platform type (возвращает User! — может быть null)
object JavaParser {
    @JvmStatic
    fun parse(json: String): User? =
        if (json.isBlank()) null else User(1L, "Parsed")
}

fun parseUser(json: String): User {
    val parsed: User? = JavaParser.parse(json)  // явно принимаем как nullable
    return parsed ?: User(0L, "default")         // Elvis вместо !!
}

fun main() {
    println(parseUser("{...}"))  // User(id=1, name=Parsed)
    println(parseUser(""))       // User(id=0, name=default)
}

Объяснение: platform type User! позволяет присвоить значение в User — тогда компилятор вставляет runtime assertion и NPE прилетает неожиданно. Безопасный путь: явно указать тип как User? и обработать null через ?:. Сложность O(1).

Задача 2. Есть sealed interface Result<out T> с ветками Success(val data: T), Error(val message: String) и Loading. Напиши extension-функцию fold, которая принимает три лямбды (по одной на каждую ветку) и возвращает значение типа R. Покажи, почему else здесь не нужен и что случится при добавлении новой ветки в sealed.

Показать решение 
sealed interface Result<out T> {
    data class Success<T>(val data: T) : Result<T>
    data class Error(val message: String) : Result<Nothing>
    data object Loading : Result<Nothing>
}

fun <T, R> Result<T>.fold(
    onSuccess: (T) -> R,
    onError: (String) -> R,
    onLoading: () -> R
): R = when (this) {
    is Result.Success -> onSuccess(data)
    is Result.Error   -> onError(message)
    Result.Loading    -> onLoading()
    // else не нужен — компилятор знает всех наследников sealed
}

fun main() {
    val r: Result<List<String>> = Result.Success(listOf("a", "b"))
    val text = r.fold(
        onSuccess = { "Loaded: ${it.size}" },
        onError   = { "Err: $it" },
        onLoading = { "Wait..." }
    )
    println(text)  // Loaded: 2
}

Объяснение: when по sealed-типу без else — исчерпывающий. Добавишь ветку Empty — код перестанет компилироваться, пока не добавишь её в fold. Ковариантный параметр out T позволяет Result<String> быть подтипом Result<Any>.

Задача 3. Есть список из 10 000 строк. Нужно отфильтровать те, что начинаются на «A», привести к верхнему регистру и взять первые 5. Напиши два варианта — через List и через Sequence. Объясни, сколько элементов обработает каждый вариант и почему в данном случае Sequence выигрывает по памяти и времени.

Показать решение 
val words = List(10_000) { i -> if (i % 100 == 0) "Apple$i" else "banana$i" }

// List — энергичные операции, две промежуточные коллекции
val listResult = words
    .filter { it.startsWith("A") }   // список из ~100 элементов
    .map { it.uppercase() }           // ещё один список из ~100
    .take(5)                          // берём 5 из уже готового

// Sequence — ленивый конвейер, короткое замыкание
val seqResult = words.asSequence()
    .filter { it.startsWith("A") }
    .map { it.uppercase() }
    .take(5)
    .toList()   // останавливается, как только нашлось 5

println(listResult == seqResult)  // true

Объяснение: вариант с List дважды проходит по всем 10 000 элементов и создаёт две промежуточные коллекции. Sequence пропускает каждый элемент сразу через всю цепочку и останавливается после 5 подходящих — реально обрабатывает около 500 элементов. Выигрыш тем заметнее, чем больше данных и чем раньше срабатывает take/first.

Задача 4. Дан не-data класс Money(val amount: Long, val currency: String). Реализуй equals и hashCode вручную так, чтобы они удовлетворяли контракту и корректно работали в HashSet. Объясни, почему для проверки класса используется javaClass-сравнение, а не is Money.

Показать решение 
class Money(val amount: Long, val currency: String) {
    override fun equals(other: Any?): Boolean {
        if (this === other) return true
        if (other == null || javaClass != other.javaClass) return false
        other as Money
        return amount == other.amount && currency == other.currency
    }

    override fun hashCode(): Int {
        var result = amount.hashCode()
        result = 31 * result + currency.hashCode()
        return result
    }
}

fun main() {
    val set = hashSetOf(Money(100, "USD"))
    println(Money(100, "USD") in set)   // true — равные объекты, равный hashCode
}

Объяснение: javaClass != other.javaClass (а не other !is Money) защищает симметричность — с is наследник Money мог бы считаться равным Money, но обратное сравнение Money.equals(наследник) дало бы false, и контракт ломается. Инвариант: равные объекты ⇒ равный hashCode, иначе HashSet кладёт в одну корзину, а ищет в другой.

Задача 5. Напиши inline-функцию measure, которая принимает блок () -> T, замеряет время его выполнения через System.nanoTime() и возвращает пару (результат, миллисекунды). Объясни, что даёт inline здесь и почему без него был бы лишний объект-лямбда.

Показать решение 
inline fun <T> measure(block: () -> T): Pair<T, Long> {
    val start = System.nanoTime()
    val result = block()
    val ms = (System.nanoTime() - start) / 1_000_000
    return result to ms
}

fun main() {
    val (value, ms) = measure { (1..1_000_000).sum() }
    println("$value за $ms мс")
}

Объяснение: inline встраивает тело block прямо в точку вызова — лямбда не создаётся как Function0-объект (нет аллокации в куче и виртуального invoke()), а замер не включает оверхед вызова. Бонус: внутри block разрешён non-local return — return из measure { ... } выйдет из внешней функции. Аналог в stdlib — measureTimedValue/measureNanoTime.

Задача 6. Реализуй кастомный делегат свойства nonBlank(default: String), который при попытке записать пустую/blank-строку оставляет предыдущее значение. Используй operator getValue/setValue. Покажи применение: var name by nonBlank("Аноним").

Показать решение 
import kotlin.reflect.KProperty

class NonBlankDelegate(private var value: String) {
    operator fun getValue(thisRef: Any?, prop: KProperty<*>): String = value
    operator fun setValue(thisRef: Any?, prop: KProperty<*>, newValue: String) {
        if (newValue.isNotBlank()) value = newValue   // blank игнорируем
    }
}

fun nonBlank(default: String) = NonBlankDelegate(default)

class Profile {
    var name: String by nonBlank("Аноним")
}

fun main() {
    val p = Profile()
    p.name = "   "      // игнорируется
    println(p.name)     // Аноним
    p.name = "Ира"
    println(p.name)     // Ира
}

Объяснение: делегат хранит состояние в себе, getValue/setValue обязаны быть operator-функциями с сигнатурой (thisRef, KProperty). Альтернатива через stdlib — Delegates.vetoable(default) { _, _, new -> new.isNotBlank() }: возврат false отменяет присваивание.
