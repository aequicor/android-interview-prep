Безопасность и качество

Безопасность

Криптография, Android Keystore (хранилище криптографических ключей), биометрия, защита от реверс-инжиниринга (обратного анализа) и OWASP Mobile Top 10 (Open Worldwide Application Security Project — открытый проект безопасности приложений; список десяти главных мобильных рисков). На собесе проверяют не знание API (программных интерфейсов), а понимание модели угроз: что злоумышленник может, где граница доверия и какими контрмерами вы её держите.

Middle Senior Частый вопрос

Криптография: симметрия, асимметрия, хеши, MAC

Три кита, которые нужно чётко разграничивать: шифрование (конфиденциальность, обратимо при наличии ключа), хеширование (целостность/отпечаток, необратимо) и MAC/подпись (аутентичность + целостность).

Симметричное vs асимметричное

КритерийСимметричное (AES)Асимметричное (RSA / EC)
КлючиОдин общий секрет на шифр/дешифрПара: публичный (шифрует/проверяет) + приватный (дешифрует/подписывает)
СкоростьОчень быстро (аппаратное AES-NI)На порядки медленнее, объём данных ограничен размером ключа

На практике используют гибридную схему: данные шифруют быстрым AES, а сам AES-ключ — асимметрично (или хранят в Keystore). Так работает TLS и «envelope encryption».

Почему AES-GCM, а не AES-CBC

GCM — это AEAD (Authenticated Encryption with Associated Data): он одновременно шифрует и аутентифицирует. На выходе — шифртекст + тег целостности (16 байт). При дешифровании, если данные подменили, тег не сойдётся и операция упадёт. CBC даёт только конфиденциальность — без MAC он уязвим к padding-oracle и битовым манипуляциям. Поэтому современный выбор по умолчанию — AES-256-GCM.

⚠️
Главный подводный камень GCM — повтор IV/nonce. Для одного и того же ключа нельзя дважды использовать один IV: это полностью ломает безопасность (восстанавливается ключ аутентификации). Генерируйте IV случайно (12 байт) на каждое шифрование и храните его рядом с шифртекстом — он не секрет.
AesGcmCipher.kt
object AesGcm {
    private const val GCM_TAG_BITS = 128
    private const val IV_SIZE = 12  // 96 бит — рекомендация для GCM

    fun encrypt(plain: ByteArray, key: SecretKey): ByteArray {
        val cipher = Cipher.getInstance("AES/GCM/NoPadding")
        cipher.init(Cipher.ENCRYPT_MODE, key)         // IV сгенерируется провайдером
        val iv = cipher.iv                            // 12 байт, не секрет
        val ct = cipher.doFinal(plain)                // шифртекст + 16-байтный тег
        return iv + ct                                // склеиваем IV || ciphertext||tag
    }

    fun decrypt(blob: ByteArray, key: SecretKey): ByteArray {
        val iv = blob.copyOfRange(0, IV_SIZE)
        val ct = blob.copyOfRange(IV_SIZE, blob.size)
        val cipher = Cipher.getInstance("AES/GCM/NoPadding")
        cipher.init(Cipher.DECRYPT_MODE, key, GCMParameterSpec(GCM_TAG_BITS, iv))
        return cipher.doFinal(ct)   // бросит AEADBadTagException, если данные подменили
    }
}

Хеши, соль, HMAC и пароли

SHA-256 — криптографический хеш: фиксированный отпечаток, лавинный эффект, необратимость. Но «голый» хеш пароля небезопасен: одинаковые пароли дают одинаковый хеш (уязвимо к rainbow-таблицам). Поэтому добавляют соль (уникальная случайная строка на каждый пароль) и используют медленные KDF — Argon2 / scrypt / PBKDF2 — чтобы перебор был дорогим. HMAC — это хеш с секретным ключом: даёт аутентичность сообщения (нельзя подделать без ключа), в отличие от простого SHA-256.

Целостность файла / отпечатокSHA-256Необратимый хеш, лавинный эффект
Аутентичность сообщения с общим ключомHMAC-SHA-256Подделать без ключа невозможно
Хранение пароляArgon2id / PBKDF2 + сольМедленно → перебор дорог

Не катайте свою крипту. Используйте проверенные примитивы и высокоуровневые обёртки (Tink, Jetpack Security). «Roll your own crypto» — классический способ выстрелить себе в ногу (неправильный режим, реюз nonce, утечка по таймингам).

Android Keystore: ключи, которые не покидают устройство

Главная идея Keystore: ключевой материал генерируется и используется в защищённой среде и не извлекается в адресное пространство приложения. Вы передаёте Keystore данные на шифрование/подпись и получаете результат — но сам ключ «потрогать» нельзя даже с root в обычном процессе. Это превращает кражу ключа в кражу доступа (которую можно отозвать), а не файла.

TEE и StrongBox — где живёт ключ

УровеньГде исполняетсяСтойкость
Software-backedВ процессе/ОС (старые девайсы, fallback)Слабая: ключ в RAM, доступен при компрометации ОС
TEE (hardware-backed)Trusted Execution Environment — изолированная зона того же SoCВысокая: ключ вне Android OS, но делит «железо» с основным миром
StrongBoxОтдельный аппаратный Secure Element (своя CPU/RAM/flash)Максимальная: физическая изоляция, защита от side-channel и tampering

Запросить StrongBox — setIsStrongBoxBacked(true). Если устройство его не поддерживает для заданного алгоритма/размера, провайдер бросит StrongBoxUnavailableException — обработайте и откатитесь на TEE. Проверить, что ключ реально в железе, можно через KeyInfo.isInsideSecureHardware() (на новых API — через getSecurityLevel()).

KeyGenParameterSpec — описываем политику ключа

KeystoreKeyFactory.kt
fun getOrCreateAesKey(alias: String): SecretKey {
    val ks = KeyStore.getInstance("AndroidKeyStore").apply { load(null) }
    (ks.getKey(alias, null) as? SecretKey)?.let { return it }

    val gen = KeyGenerator.getInstance(
        KeyProperties.KEY_ALGORITHM_AES, "AndroidKeyStore"
    )
    val spec = KeyGenParameterSpec.Builder(
        alias,
        KeyProperties.PURPOSE_ENCRYPT or KeyProperties.PURPOSE_DECRYPT
    )
        .setBlockModes(KeyProperties.BLOCK_MODE_GCM)
        .setEncryptionPaddings(KeyProperties.ENCRYPTION_PADDING_NONE)
        .setKeySize(256)
        .setIsStrongBoxBacked(true)          // SE, если доступен (иначе ловим исключение)
        .setUserAuthenticationRequired(true) // использовать ключ только после аутентификации
        .setUserAuthenticationParameters(
            0,                               // 0 = требовать аутентификацию на каждую операцию
            KeyProperties.AUTH_BIOMETRIC_STRONG
        )
        // ключ инвалидируется при добавлении нового отпечатка/лица:
        .setInvalidatedByBiometricEnrollment(true)
        .build()

    return try {
        gen.init(spec); gen.generateKey()
    } catch (e: StrongBoxUnavailableException) {
        gen.init(KeyGenParameterSpec.Builder(spec).setIsStrongBoxBacked(false).build())
        gen.generateKey()
    }
}
⚠️
Инвалидация ключей с user-auth. Ключ, привязанный к биометрии/lockscreen, становится невалидным при изменении набора биометрии (если setInvalidatedByBiometricEnrollment(true)) или снятии экрана блокировки. Готовьтесь ловить KeyPermanentlyInvalidatedException при дешифровании: пересоздать ключ и попросить пользователя заново ввести/сохранить данные. Это частая причина «внезапных» крашей в проде.

Jetpack Security и Tink — обёртки

Чтобы не писать boilerplate, есть высокоуровневые библиотеки. Jetpack Security (EncryptedSharedPreferences, EncryptedFile) использует Keystore под капотом и шифрует пары ключ/значение и файлы. Под капотом она построена на Tink — крипто-библиотеке Google с безопасными дефолтами (AEAD, ротация ключей через keyset, защита от типичных ошибок). Учтите: EncryptedSharedPreferences в Jetpack Security 1.x переведена в deprecated; для нового кода рассматривайте Tink напрямую или собственную обёртку поверх Keystore.

"обёрнут KEK" .-> F["Android Keystore (TEE / StrongBox)"] F -. "KEK не покидает железо" .-> G["Secure Element"] B -. "берёт DEK у Keystore" .-> E
Envelope encryption: данные шифруются DEK внутри приложения, а DEK защищён KEK из Keystore, который физически не покидает TEE/StrongBox

Биометрия: BiometricPrompt и CryptoObject

BiometricPrompt (из androidx.biometric) — единая системная точка для отпечатка/лица/радужки с авто-fallback на PIN/пароль. Но важнейший нюанс уровня Senior: сам факт «прошёл биометрию» — слабая гарантия. Логический флаг «authenticated = true» легко обойти хуком (Frida) или подменой результата. Сильная схема — привязать крипто-операцию к биометрии через CryptoObject: ключ Keystore становится доступен только после успешной аутентификации, и без неё данные физически не расшифровать.

Классы аутентификаторов

AuthenticatorЧто этоCryptoObject
BIOMETRIC_STRONG (Class 3)Стойкая биометрия (низкий spoof/false-accept rate)Да — можно привязывать ключи
BIOMETRIC_WEAK (Class 2)Слабая (напр. часть face-unlock)Нет — нельзя использовать с ключом

Для крипто-операций используют только BIOMETRIC_STRONG (или + DEVICE_CREDENTIAL). Комбинацию BIOMETRIC_STRONG | DEVICE_CREDENTIAL с CryptoObject нельзя применять на API 28–29 (известное ограничение). Если операция должна быть строго «только биометрия» — не добавляйте DEVICE_CREDENTIAL в разрешённые аутентификаторы.

Полный цикл: ключ ↔ биометрия ↔ Cipher

BiometricSecret.kt
// 1) Проверяем доступность сильной биометрии
fun canUseStrongBiometrics(ctx: Context): Boolean {
    val mgr = BiometricManager.from(ctx)
    return mgr.canAuthenticate(BiometricManager.Authenticators.BIOMETRIC_STRONG) ==
            BiometricManager.BIOMETRIC_SUCCESS
}

// 2) Показываем prompt, передавая Cipher через CryptoObject
fun decryptWithBiometric(
    activity: FragmentActivity,
    cipher: Cipher,                 // init(DECRYPT_MODE, key, GCMParameterSpec(...))
    payload: ByteArray,
    onResult: (ByteArray?) -> Unit
) {
    val prompt = BiometricPrompt(activity,
        ContextCompat.getMainExecutor(activity),
        object : BiometricPrompt.AuthenticationCallback() {
            override fun onAuthenticationSucceeded(r: BiometricPrompt.AuthenticationResult) {
                // r.cryptoObject?.cipher разблокирован TEE именно для этой сессии
                val unlocked = r.cryptoObject?.cipher ?: return onResult(null)
                onResult(unlocked.doFinal(payload))
            }
            override fun onAuthenticationError(code: Int, msg: CharSequence) = onResult(null)
        })

    val info = BiometricPrompt.PromptInfo.Builder()
        .setTitle("Подтвердите личность")
        .setSubtitle("Доступ к зашифрованным данным")
        .setAllowedAuthenticators(BiometricManager.Authenticators.BIOMETRIC_STRONG)
        .setNegativeButtonText("Отмена")  // обязателен, если нет DEVICE_CREDENTIAL
        .build()

    prompt.authenticate(info, BiometricPrompt.CryptoObject(cipher))
}
💡
Почему CryptoObject сильнее «флага». Без него вы доверяете булеву результату prompt — а его можно подделать инструментацией. С CryptoObject успешная биометрия разблокирует сам ключ в TEE: если злоумышленник не прошёл биометрию, cipher.doFinal() провалится, потому что ключ недоступен. Аутентификация становится частью крипто-операции, а не отдельным «да/нет».

Реверс-инжиниринг и защита: Frida и контрмеры

Раздел про defensive security: как кандидат объясняет угрозы со стороны атакующего и какие контрмеры строит. Не про написание эксплойтов — про понимание, что приложение на чужом устройстве работает во враждебной среде.

Что такое Frida (концептуально)

Frida — это фреймворк динамической инструментации. Идея: подключиться к уже запущенному процессу и в рантайме «хукнуть» функции — перехватить вызов, прочитать/подменить аргументы и возвращаемое значение, выполнить свой код. Это даёт атакующему (обычно на рутованном устройстве или эмуляторе) мощный инструмент: можно заставить функцию isDeviceRooted() всегда возвращать false, отключить проверку сертификата, вынуть ключ или токен из памяти, подменить результат «успешной» биометрии. Работает без исходников — хукаются как Java/Kotlin-методы, так и нативные функции по адресу.

flowchart LR A["Атакующий: root / эмулятор"] --> B["Frida server в процессе"] B --> C{"Хук метода в рантайме"} C -->|"перехват"| D["checkSsl() -> true"] C -->|"перехват"| E["isRooted() -> false"] C -->|"чтение памяти"| F["Вынуть токен / ключ"] G["Контрмеры (RASP)"] -.->|"детект Frida-сервера"| B G -.->|"проверка целостности"| C
Концептуально: инструментация перехватывает вызовы в рантайме; защита (RASP) пытается обнаружить присутствие хуков и факт компрометации среды

Что обычно пытаются обойти: SSL/certificate pinning (чтобы читать трафик через MITM-прокси), root/emulator detection, проверки целостности и подписи APK, лицензионные и платёжные проверки. Важно понимать: на достаточно подготовленном устройстве любая клиентская защита в итоге обходима — цель контрмер не «сделать невозможным», а «сделать дорого и заметно».

Ключевой принцип. Никогда не полагайтесь только на клиентскую проверку для критичных решений (платёж прошёл, пользователь не на руте, лицензия валидна). Клиент сообщает сигнал — а финальное решение принимает сервер, сопоставляя его с Play Integrity и собственной бизнес-логикой. Это единственная защита, которую нельзя отключить хуком на устройстве.

Play Integrity API

Заменил устаревший SafetyNet Attestation. Возвращает подписанный Google вердикт о целостности: устройства (MEETS_BASIC_INTEGRITY / MEETS_DEVICE_INTEGRITY / MEETS_STRONG_INTEGRITY), приложения (подлинный ли APK из Play, не перепакован), аккаунта. MEETS_STRONG_INTEGRITY — высший уровень: требует hardware-backed подтверждения загрузки (verified boot) и свежих патчей безопасности; рутованные устройства с разблокированным загрузчиком его обычно не проходят. Критично: вердикт нужно проверять на вашем сервере (валидировать подпись токена), иначе ответ можно подделать на клиенте.

Где НЕ хранить секреты

Частый вопрос-ловушка: «куда положить API-ключ?». Правильный ответ начинается с «зависит, какой секрет, и в идеале — не на клиенте». Конкретика:

  • Не в коде / не в BuildConfig / не в strings.xml / не в gradle.properties, попавших в APK. Всё это тривиально достаётся декомпиляцией (BuildConfig — это просто константы в classes.dex). Обфускация имён их не прячет — значения остаются строками.
  • Не в обычных SharedPreferences и не в логах. На рутованном устройстве и в бэкапах это читается.
  • Высокочувствительные секреты (ключи к платёжкам, мастер-ключи) — только на сервере; клиент получает короткоживущие токены.
  • Если секрет всё же нужен на устройстве (например, токен сессии) — шифруйте через Keystore (EncryptedFile/Tink), привязывайте к user-auth, держите время жизни минимальным.
SecureTokenStore.kt — токен под Keystore, не в Prefs
class SecureTokenStore(context: Context) {
    private val keyAlias = "session_token_key"

    fun save(token: String) {
        val key = KeystoreKeyFactory.getOrCreateAesKey(keyAlias) // см. раздел Keystore
        val blob = AesGcm.encrypt(token.toByteArray(), key)       // IV||ct||tag
        File(context.filesDir, "session.bin").writeBytes(blob)    // на диске — только шифртекст
    }

    fun load(): String? {
        val f = File(context.filesDir, "session.bin")
        if (!f.exists()) return null
        return try {
            val key = KeystoreKeyFactory.getOrCreateAesKey(keyAlias)
            String(AesGcm.decrypt(f.readBytes(), key))
        } catch (e: KeyPermanentlyInvalidatedException) {
            f.delete(); null   // ключ инвалидирован (сменилась биометрия) — чистим и просим перелогин
        }
    }
}
⚠️
Backup и логи. Отключите авто-бэкап для файлов с секретами (android:allowBackup="false" или правила dataExtractionRules), иначе зашифрованный blob и/или метаданные утекут в облачный бэкап. И никогда не логируйте токены/PII — логи читаются на устройстве и попадают в краш-репорты.

Безопасный IPC, WebView и сеть

Экспортированные компоненты и permissions

Каждый exported-компонент (Activity/Service/Receiver/Provider) — это точка входа извне, доступная другим приложениям. С API 31 атрибут android:exported обязателен явно. Правила: экспортируйте только то, что действительно публично; защищайте чувствительные компоненты custom permission с protectionLevel="signature" (вызывать сможет только приложение с вашей подписью); валидируйте все входящие Intent как недоверенный ввод; не передавайте секреты через широковещательные интенты.

AndroidManifest.xml
<!-- своя подпись-разрешение: вызовет только приложение с тем же signing-ключом -->
<permission android:name="com.app.permission.SYNC"
    android:protectionLevel="signature" />

<service
    android:name=".SyncService"
    android:exported="false" />        <!-- по умолчанию закрыт -->

<activity
    android:name=".DeepLinkActivity"
    android:exported="true">           <!-- открыт намеренно: ловит deep link -->
    <intent-filter android:autoVerify="true">
        <action android:name="android.intent.action.VIEW" />
        <data android:scheme="https" android:host="app.example.com" />
    </intent-filter>
</activity>

Валидация deep link

Deep link / App Link — это недоверенный вход: ссылку может сформировать кто угодно. Используйте Android App Links (autoVerify="true" + assetlinks.json на домене) — это криптографически привязывает ссылку к вашему домену, исключая перехват кастомной схемы другим приложением. Всегда валидируйте параметры (никаких слепых редиректов по URL из ссылки, проверка прав на запрошенный ресурс, защита от open-redirect и инъекций).

Риски WebView

WebView — частый источник дыр. Ключевые правила:

  • addJavascriptInterface — самое опасное. Исторически (до API 17) позволял RCE через рефлексию. Сейчас экспонируйте только методы с @JavascriptInterface, минимизируйте поверхность, не подключайте интерфейс к недоверенному контенту.
  • Не включайте setJavaScriptEnabled(true), если не нужно; не грузите произвольные URL; отключите доступ к файловой системе (setAllowFileAccess(false)).
  • Включите Safe Browsing (setSafeBrowsingEnabled(true), API 26+).
  • Загружайте только HTTPS; cleartext в WebView запрещайте через network security config.

Network Security Config и certificate pinning

Network Security Configuration (XML, API 24+) — декларативный способ задать сетевую политику без правки кода: запретить cleartext, ограничить доверенные CA, настроить pinning. Плюс в том, что правила применяются ко всему сетевому стеку, включая WebView, автоматически. Certificate pinning заставляет приложение доверять только конкретному сертификату/публичному ключу сервера — даже валидный, но «чужой» сертификат (подставленный MITM-прокси или скомпрометированным CA) будет отклонён. Пинят обычно публичный ключ (SPKI), а не весь сертификат, и закладывают backup-pin на случай ротации.

res/xml/network_security_config.xml
<network-security-config>
    <!-- запрещаем нешифрованный трафик глобально -->
    <base-config cleartextTrafficPermitted="false" />

    <domain-config>
        <domain includeSubdomains="true">api.example.com</domain>
        <pin-set expiration="2026-12-31">
            <!-- основной pin (SPKI публичного ключа) -->
            <pin digest="SHA-256">k3X...основной...=</pin>
            <!-- backup-pin на случай ротации сертификата -->
            <pin digest="SHA-256">9aF...запасной...=</pin>
        </pin-set>
    </domain-config>
</network-security-config>
🛑
Pinning без backup-pin — мина замедленного действия. Если запинить один сертификат и он ротируется/истечёт, все клиенты потеряют связь с сервером, а починить можно только обновлением приложения. Всегда закладывайте запасной pin (на будущий ключ) и expiration. Альтернатива в коде — OkHttp CertificatePinner, но конфиг покрывает и WebView.

Частые вопросы интервьюера

В чём разница симметричного и асимметричного шифрования и когда что применять? Middle

Симметричное (AES) — один общий ключ, быстро, для шифрования объёмных данных «на диске» или полезной нагрузки. Асимметричное (RSA/EC) — пара ключей (публичный/приватный), медленно, для обмена ключами, цифровой подписи и TLS-handshake. На практике — гибридно: данные шифруют AES, а сам AES-ключ передают/защищают асимметрично (envelope encryption). Для Android: данные — AES-256-GCM, а ключ защищён Keystore.

Зачем хранить ключ в Keystore, если он «всё равно на устройстве»? Middle+

Потому что в Keystore ключ не извлекается в память приложения: он живёт в TEE/StrongBox, а вы лишь передаёте данные на шифрование/подпись. Даже с root в обычном процессе нельзя скопировать ключевой материал — можно лишь временно получить доступ к операции, который отзывается (сменой биометрии, переустановкой). Это превращает кражу «файла-ключа» в кражу доступа, плюс позволяет привязать ключ к биометрии. Хранить же ключ в SharedPreferences/файле — значит отдать его при первом дампе памяти или бэкапе.

Чем CryptoObject лучше простого флага «биометрия пройдена»? Senior

Булев результат prompt — это логическая проверка, которую легко обойти хуком (Frida перехватит callback и вернёт «успех»). CryptoObject привязывает крипто-операцию к биометрии: успешная аутентификация разблокирует сам ключ в TEE для этой сессии. Если пользователь не прошёл биометрию, ключ недоступен и cipher.doFinal() упадёт — расшифровать данные физически нельзя. Аутентификация становится частью операции, а не отдельным «да/нет». Для этого нужен ключ с setUserAuthenticationRequired(true) и authenticator BIOMETRIC_STRONG.

Что такое Frida и как от неё защищаться? Senior

Frida — фреймворк динамической инструментации: подключается к запущенному процессу и хукает функции в рантайме (перехват аргументов/результата, выполнение своего кода). Атакующий обходит SSL pinning, root detection, integrity-проверки, вынимает токены из памяти. Защита эшелонированная: детект инструментации (наличие Frida-сервера, подозрительные библиотеки) через RASP; обфускация R8; критичные строки шифруются, чувствительная логика — в native. Но главное — не доверять клиенту: финальные решения (целостность, лицензия, платёж) принимать на сервере по вердикту Play Integrity. На подготовленном устройстве клиентскую защиту в итоге обходят — цель в том, чтобы поднять стоимость и заметность атаки.

Можно ли хранить API-ключ в BuildConfig или strings.xml? Middle+

Нет. BuildConfig — это просто константы в classes.dex, а strings.xml — ресурсы; и то и другое тривиально достаётся декомпиляцией (apktool/jadx), обфускация значения-строки не прячет. Высокочувствительные секреты должны жить на сервере, а клиент получать короткоживущие токены. Если секрет действительно нужен на устройстве — шифровать через Keystore (EncryptedFile/Tink), привязывать к user-auth и минимизировать TTL. И помнить, что любой секрет в APK — это уже не секрет, а «обфусцированная константа».

Что такое certificate pinning и какие у него риски? Middle+

Pinning заставляет приложение доверять только конкретному сертификату/публичному ключу сервера, а не любому валидному из системного хранилища. Это защищает от MITM через скомпрометированный или подставленный CA. Пинят обычно SPKI (публичный ключ), потому что он переживает обновление сертификата. Главный риск — отсутствие backup-pin: при ротации/истечении ключа без запасного pin все клиенты теряют связь, и чинится это только релизом приложения. Поэтому всегда закладывают backup-pin и срок действия. Настраивают через Network Security Config (покрывает и WebView) или OkHttp CertificatePinner.

Почему AES-GCM предпочтительнее AES-CBC и в чём опасность GCM? Senior

GCM — это AEAD: шифрует и одновременно аутентифицирует (тег целостности). При подмене шифртекста дешифрование падает (AEADBadTagException). CBC даёт только конфиденциальность, без MAC уязвим к padding-oracle и битовым манипуляциям. Опасность GCM — повтор IV/nonce под одним ключом: это катастрофа, восстанавливается ключ аутентификации и теряется конфиденциальность. Поэтому IV (12 байт) генерируют случайно на каждое шифрование и хранят рядом с шифртекстом (он не секрет). Никогда не используйте фиксированный/счётчиковый IV без гарантии уникальности.

Как обезопасить exported-компоненты и deep link? Middle+

Экспортируйте только то, что реально публично (с API 31 android:exported обязателен явно). Чувствительные компоненты защищайте custom permission с protectionLevel="signature" — вызовет только приложение с вашей подписью. Все входящие Intent и параметры deep link трактуйте как недоверенный ввод и валидируйте. Для ссылок используйте Android App Links (autoVerify + assetlinks.json), что криптографически привязывает домен и не даёт перехватить кастомную схему. Не передавайте секреты через broadcast и не делайте слепых редиректов по URL из ссылки.

Чем TEE отличается от StrongBox? Senior

TEE (Trusted Execution Environment) — изолированный «безопасный мир» на том же SoC, что и основная ОС: ключи исполняются вне Android, но делят физическое железо. StrongBox — отдельный аппаратный Secure Element со своими CPU/RAM/flash: физическая изоляция, защита от side-channel и аппаратного вскрытия — максимальная стойкость. Запрашивается setIsStrongBoxBacked(true); если не поддержан — StrongBoxUnavailableException, нужно откатиться на TEE. Проверить уровень — KeyInfo.getSecurityLevel()/isInsideSecureHardware(). StrongBox требуется для самых чувствительных ключей (платежи, identity), для большинства задач достаточно TEE.

Задачи

Задача 1. Вам дали код, который шифрует пользовательские данные и сохраняет их на диск. Найдите уязвимость и перепишите метод так, чтобы он использовал Android Keystore с AES-256-GCM, генерировал уникальный IV на каждое шифрование и корректно обрабатывал инвалидацию ключа при смене биометрии.

Показать решение 
// ❌ Уязвимый исходный код
object BrokenEncryptor {
    private val KEY = SecretKeySpec("hardcoded_key_123".toByteArray(), "AES")
    private val FIXED_IV = ByteArray(12) { 0 }  // один IV для всех шифровок

    fun encrypt(data: ByteArray): ByteArray {
        val cipher = Cipher.getInstance("AES/GCM/NoPadding")
        cipher.init(Cipher.ENCRYPT_MODE, KEY, GCMParameterSpec(128, FIXED_IV))
        return cipher.doFinal(data)  // не сохраняет IV рядом с данными
    }
}

// ✅ Правильная реализация
class SecureEncryptor(private val alias: String) {

    private fun getOrCreateKey(): SecretKey {
        val ks = KeyStore.getInstance("AndroidKeyStore").apply { load(null) }
        (ks.getKey(alias, null) as? SecretKey)?.let { return it }

        val gen = KeyGenerator.getInstance(KeyProperties.KEY_ALGORITHM_AES, "AndroidKeyStore")
        val spec = KeyGenParameterSpec.Builder(
            alias,
            KeyProperties.PURPOSE_ENCRYPT or KeyProperties.PURPOSE_DECRYPT
        )
            .setBlockModes(KeyProperties.BLOCK_MODE_GCM)
            .setEncryptionPaddings(KeyProperties.ENCRYPTION_PADDING_NONE)
            .setKeySize(256)
            .setUserAuthenticationRequired(true)
            .setUserAuthenticationParameters(0, KeyProperties.AUTH_BIOMETRIC_STRONG)
            .setInvalidatedByBiometricEnrollment(true)
            .build()
        gen.init(spec)
        return gen.generateKey()
    }

    fun encrypt(data: ByteArray): ByteArray {
        val key = getOrCreateKey()
        val cipher = Cipher.getInstance("AES/GCM/NoPadding")
        cipher.init(Cipher.ENCRYPT_MODE, key)   // IV генерируется случайно провайдером
        val iv = cipher.iv                       // 12 байт, не секрет
        val ct = cipher.doFinal(data)
        return iv + ct                           // формат: IV(12) || CT || Tag(16)
    }

    fun decrypt(blob: ByteArray): ByteArray? {
        return try {
            val key = getOrCreateKey()
            val iv = blob.copyOfRange(0, 12)
            val ct = blob.copyOfRange(12, blob.size)
            val cipher = Cipher.getInstance("AES/GCM/NoPadding")
            cipher.init(Cipher.DECRYPT_MODE, key, GCMParameterSpec(128, iv))
            cipher.doFinal(ct)
        } catch (e: KeyPermanentlyInvalidatedException) {
            // ключ инвалидирован: новая биометрия — пересоздать и попросить re-auth
            null
        }
    }
}

Объяснение: исходный код содержит три уязвимости: хардкод ключа в памяти приложения (тривиально извлекается), фиксированный IV (ломает GCM — восстанавливается ключ аутентификации), отсутствие обработки инвалидации ключа. Исправление переносит ключ в Keystore (физически не извлекаем), генерирует уникальный IV на каждую операцию и ловит KeyPermanentlyInvalidatedException.

Задача 2. Ваш коллега написал BiometricAuth-обёртку. При тест-ревью вы обнаружили, что защита обходится одним Frida-скриптом, не затрагивая крипто-операции. Перепишите метод authenticate() так, чтобы данные физически нельзя было расшифровать без успешной биометрии, — используйте CryptoObject.

Показать решение 
// ❌ Уязвимая реализация — флаг можно подделать хуком
class WeakBiometricAuth(private val activity: FragmentActivity) {

    private var isAuthenticated = false   // Frida легко делает это true

    fun authenticate(onSuccess: () -> Unit) {
        val prompt = BiometricPrompt(activity,
            ContextCompat.getMainExecutor(activity),
            object : BiometricPrompt.AuthenticationCallback() {
                override fun onAuthenticationSucceeded(r: BiometricPrompt.AuthenticationResult) {
                    isAuthenticated = true
                    onSuccess()   // вызывается и при хукнутом флаге
                }
                override fun onAuthenticationError(code: Int, msg: CharSequence) {}
            })
        prompt.authenticate(BiometricPrompt.PromptInfo.Builder()
            .setTitle("Войти")
            .setAllowedAuthenticators(BiometricManager.Authenticators.BIOMETRIC_STRONG)
            .setNegativeButtonText("Отмена")
            .build())
    }
}

// ✅ Сильная схема — биометрия разблокирует ключ в TEE
class StrongBiometricDecryptor(
    private val activity: FragmentActivity,
    private val keyAlias: String
) {
    // Ключ должен быть создан с setUserAuthenticationRequired(true), BIOMETRIC_STRONG
    fun decryptWithBiometric(
        encryptedBlob: ByteArray,   // формат: IV(12) || CT || Tag(16)
        onResult: (ByteArray?) -> Unit
    ) {
        val ks = KeyStore.getInstance("AndroidKeyStore").apply { load(null) }
        val key = ks.getKey(keyAlias, null) as? SecretKey
            ?: return onResult(null)

        val iv = encryptedBlob.copyOfRange(0, 12)
        val cipher = Cipher.getInstance("AES/GCM/NoPadding")
        cipher.init(Cipher.DECRYPT_MODE, key, GCMParameterSpec(128, iv))
        // cipher ещё «заблокирован» — TEE не пустит до реальной биометрии

        val prompt = BiometricPrompt(activity,
            ContextCompat.getMainExecutor(activity),
            object : BiometricPrompt.AuthenticationCallback() {
                override fun onAuthenticationSucceeded(r: BiometricPrompt.AuthenticationResult) {
                    val unlockedCipher = r.cryptoObject?.cipher
                        ?: return onResult(null)
                    // только здесь ключ разблокирован TEE для этой сессии
                    val plain = try {
                        unlockedCipher.doFinal(encryptedBlob.copyOfRange(12, encryptedBlob.size))
                    } catch (e: Exception) { null }
                    onResult(plain)
                }
                override fun onAuthenticationError(code: Int, msg: CharSequence) = onResult(null)
            })

        prompt.authenticate(
            BiometricPrompt.PromptInfo.Builder()
                .setTitle("Подтвердите личность")
                .setAllowedAuthenticators(BiometricManager.Authenticators.BIOMETRIC_STRONG)
                .setNegativeButtonText("Отмена")
                .build(),
            BiometricPrompt.CryptoObject(cipher)   // ← вот вся суть
        )
    }
}

Объяснение: слабая версия доверяет булеву флагу — Frida перехватит onAuthenticationSucceeded и вызовет его напрямую. В сильной версии CryptoObject(cipher) передаётся в prompt: успешная биометрия разблокирует ключ в TEE именно для этого cipher-объекта. Без реальной биометрии cipher.doFinal() бросит исключение, потому что ключ физически недоступен. Хук на callback ничего не даст — данные всё равно не расшифруются.

Задача 3. Приложение использует Certificate Pinning через Network Security Config. После ротации сертификата на сервере все пользователи получили сетевую ошибку и пришлось срочно выкатывать хотфикс. Объясните ошибку в конфиге и напишите исправленный вариант, который выдержит плановую ротацию без аварийного релиза.

Показать решение 
// ❌ Уязвимый конфиг — один pin, нет expiration, нет backup
// res/xml/network_security_config_broken.xml:
//
// <network-security-config>
//     <domain-config>
//         <domain includeSubdomains="true">api.example.com</domain>
//         <pin-set>
//             <pin digest="SHA-256">oldKey...=</pin>
//         </pin-set>
//     </domain-config>
// </network-security-config>
//
// При ротации сертификата на сервере oldKey становится невалидным →
// все клиенты получают SSLPeerUnverifiedException, связь рвётся.

// ✅ Правильный конфиг — основной pin + backup + expiration + запрет cleartext
// res/xml/network_security_config.xml:
//
// <network-security-config>
//     <base-config cleartextTrafficPermitted="false" />
//
//     <domain-config>
//         <domain includeSubdomains="true">api.example.com</domain>
//         <pin-set expiration="2027-06-01">
//             <!-- основной: текущий публичный ключ (SPKI SHA-256) -->
//             <pin digest="SHA-256">currentKey...=</pin>
//             <!-- backup: следующий ключ, уже готов, но ещё не активен -->
//             <pin digest="SHA-256">nextKey...=</pin>
//         </pin-set>
//     </domain-config>
// </network-security-config>

// Для OkHttp-вариант (если нужно в коде):
fun buildSecureClient(): OkHttpClient {
    val pinner = CertificatePinner.Builder()
        .add("api.example.com",
            "sha256/currentKey...=",   // основной
            "sha256/nextKey...=")      // backup на ротацию
        .build()
    return OkHttpClient.Builder()
        .certificatePinner(pinner)
        .build()
}

// Алгоритм получения pin для своего домена:
// openssl s_client -connect api.example.com:443 | \
//   openssl x509 -pubkey -noout | \
//   openssl pkey -pubin -outform der | \
//   openssl dgst -sha256 -binary | base64

Объяснение: без backup-pin ротация сертификата на сервере мгновенно разрывает связь для всех клиентов, у которых нет обновлённого приложения. Правильная схема: всегда держать два pin — текущий и следующий (его генерируют заранее, до ротации). Пинить нужно SPKI (публичный ключ), а не весь сертификат — тогда обновление сертификата при том же ключе не ломает pin. Атрибут expiration заставляет устаревший конфиг игнорировать pin-проверку (защита от «вечного» lockout), что стимулирует регулярные обновления приложения.
