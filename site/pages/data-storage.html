Данные и сеть

Хранение данных

Где и как Android-приложение хранит состояние: от key-value (ключ-значение: SharedPreferences и DataStore) до реляционных БД (баз данных: Room/SQLite и SQLCipher) и файлов (scoped storage — ограниченное хранилище, MediaStore — медиакаталог, FileProvider — поставщик файлов). На собесе ждут не перечень API (программных интерфейсов), а понимание trade-offs (компромиссов): транзакционность, потокобезопасность, шифрование и миграции.

Middle Senior Частый вопрос

SharedPreferences: наследие и его минусы

SharedPreferences — простое key-value-хранилище поверх XML-файла в приватной директории приложения. Долго был дефолтом для настроек, но в современных проектах его считают legacy. Понимать его всё ещё нужно: его минусы — это ровно те причины, по которым появился DataStore.

apply() vs commit() — классический вопрос

val prefs = context.getSharedPreferences("settings", Context.MODE_PRIVATE)

// commit(): синхронно пишет на диск в ТЕКУЩЕМ потоке, возвращает Boolean (успех/нет)
val ok: Boolean = prefs.edit().putBoolean("dark", true).commit()

// apply(): мгновенно обновляет in-memory, запись на диск — асинхронно в фоне;
// возвращает Unit, об ошибке узнать НЕЛЬЗЯ
prefs.edit().putBoolean("dark", true).apply()

⚠️
«apply не блокирует» — полуправда. Сама запись уходит в фон, но Android регистрирует её в очереди и дожидается завершения всех pending-apply() в момент Activity.onPause()/onStop() и в QueuedWork. На медленном диске или при множестве записей это становится источником ANR. Именно про это любят спросить «а точно ли apply бесплатный?».

Почему уходят на DataStore

  • Синхронное чтение. Первое обращение к любому ключу читает и парсит весь XML в память — на main-потоке это блокирующая операция (риск StrictMode-violation и подлагиваний).
  • Нет нормальной обработки ошибок. apply() молча проглатывает сбои записи; повреждённый файл может привести к потере данных без уведомления.
  • Нет транзакционности. Несколько put в одном edit() не образуют атомарной транзакции с гарантиями консистентности на уровне типобезопасности — легко получить рассогласованное состояние.
  • Нет реактивности из коробки. Есть только OnSharedPreferenceChangeListener с ручной отпиской; нет Flow, нет типобезопасности, нет работы с корутинами.
Потокобезопасен ли SharedPreferences и почему getSharedPreferences с одним именем всегда возвращает тот же объект? Хватает ли этого для read-modify-write из разных потоков? ★

На уровне отдельной операции — да. Внутри хранится карта значений под собственной блокировкой, а getSharedPreferences кэширует один экземпляр на имя файла в статической карте ContextImpl. Поэтому два вызова с одним именем дают тот же объект и одну общую блокировку: параллельные getString/putString по отдельности друг друга не покорёжат.

А вот транзакционности это не даёт. Между getString и последующим putString другой поток успеет вклиниться — классическая гонка read-modify-write с потерянным обновлением. apply() из разных потоков сериализует записи на диск через QueuedWork, но порядок относительно других edit() не гарантирует. Между процессами всё ещё хуже: MODE_MULTI_PROCESS объявлен deprecated и работает ненадёжно. Нужно согласованное составное состояние — берите DataStore с атомарным edit{}.

DataStore: современная замена

Jetpack DataStore — рекомендованное Google key-value/типизированное хранилище. Полностью асинхронное на корутинах и Flow, транзакционное, безопасно обрабатывает повреждение данных и поставляется в двух вариантах.

Preferences DataStore: чтение через Flow, запись через edit

SettingsDataStore.kt
// один DataStore на процесс — через property-делегат на Context
val Context.dataStore by preferencesDataStore(name = "settings")

class SettingsRepository(private val context: Context) {
    private object Keys { val DARK = booleanPreferencesKey("dark_theme") }

    // чтение — холодный Flow, эмитит при каждом изменении
    val darkTheme: Flow<Boolean> = context.dataStore.data
        .catch { e ->                                  // обработка повреждения файла
            if (e is IOException) emit(emptyPreferences()) else throw e
        }
        .map { prefs -> prefs[Keys.DARK] ?: false }

    // запись — suspend, весь блок edit{} транзакционен и атомарен
    suspend fun setDarkTheme(enabled: Boolean) {
        context.dataStore.edit { prefs -> prefs[Keys.DARK] = enabled }
    }
}

Proto DataStore: типобезопасный объект

UserPrefsSerializer.kt
// UserPrefs генерируется из user_prefs.proto
object UserPrefsSerializer : Serializer<UserPrefs> {
    override val defaultValue: UserPrefs = UserPrefs.getDefaultInstance()
    override suspend fun readFrom(input: InputStream): UserPrefs =
        try { UserPrefs.parseFrom(input) }
        catch (e: InvalidProtocolBufferException) { throw CorruptionException("proto", e) }
    override suspend fun writeTo(t: UserPrefs, output: OutputStream) = t.writeTo(output)
}

val Context.userPrefs: DataStore<UserPrefs> by dataStore(
    fileName = "user_prefs.pb",
    serializer = UserPrefsSerializer
)
💡
Транзакционность на деле. Блок edit { } (или updateData { } у Proto) выполняется атомарно: DataStore читает текущее значение, применяет переданную трансформацию и сериализует результат под единой блокировкой. Параллельные записи сериализуются, что исключает гонки и «потерянные обновления», которых не избежать в SharedPreferences.

Миграция со SharedPreferences

DataStore умеет автоматически перенести существующие SharedPreferences при первом обращении через SharedPreferencesMigration — данные читаются один раз, переносятся в DataStore, старый файл удаляется.

migration.kt
val Context.dataStore by preferencesDataStore(
    name = "settings",
    produceMigrations = { ctx ->
        listOf(SharedPreferencesMigration(ctx, "old_settings_prefs"))
    }
)
// Первое чтение/запись DataStore выполнит миграцию ключей из old_settings_prefs
// и пометит её завершённой, удалив исходный XML.
Что будет, если создать два экземпляра DataStore на один файл в процессе, или открыть файл из двух процессов? Как DataStore держит консистентность? ★

DataStore требует ровно один экземпляр на файл в пределах процесса — именно поэтому его заводят через property-делегат на уровне приложения, а не создают на каждый экран. Второй экземпляр на тот же файл получит IllegalStateException с текстом «There are multiple DataStores active for the same file»: внутренняя in-memory-блокировка и кэш перестают защищать от гонок, и DataStore честно об этом сообщает, а не молча портит данные.

Между процессами консистентности нет вовсе — DataStore спроектирован как single-process и для IPC не годится (в отличие от ContentProvider). Атомарность одного процесса держится на Mutex: updateData/edit читают текущее значение, применяют трансформацию, сериализуют результат во временный файл и атомарно подменяют основной через rename. Параллельные записи выстраиваются в очередь — ровно то, чего не хватало SharedPreferences.

Шифрованное хранилище и Keystore

Для чувствительных данных (токены, PII) нужно шифрование в покое (at rest): даже если файл вытащат с устройства (root, бэкап), без ключа он бесполезен. Корень доверия — Android Keystore: ключи генерируются и хранятся в аппаратно-защищённой среде (TEE/StrongBox), их нельзя экспортировать — приложение лишь использует их для шифрования/дешифрования.

EncryptedSharedPreferences и статус Jetpack Security

Исторически Google предлагал EncryptedSharedPreferences и EncryptedFile из библиотеки Jetpack Security (androidx.security:security-crypto) поверх Google Tink. Ключи-обёртки (master key) хранятся в Keystore.

EncryptedPrefs.kt (legacy)
val masterKey = MasterKey.Builder(context)
    .setKeyScheme(MasterKey.KeyScheme.AES256_GCM)
    .build()

val securePrefs = EncryptedSharedPreferences.create(
    context, "secure_tokens", masterKey,
    EncryptedSharedPreferences.PrefKeyEncryptionScheme.AES256_SIV,
    EncryptedSharedPreferences.PrefValueEncryptionScheme.AES256_GCM

securePrefs.edit().putString("auth_token", token).apply()
🛑
Jetpack Security в maintenance, EncryptedSharedPreferences deprecated. EncryptedSharedPreferences (и весь androidx.security:security-crypto) помечен deprecated начиная с 1.1.0-alpha07 (апрель 2025) — статус закреплён и в стабильной 1.1.0. Известные проблемы: StrictMode-violations на main-потоке (синхронный init) и «keyset corruption» на некоторых OEM. Для новых проектов лучше выбирать альтернативы — см. ниже.

Чем заменить: шифрованный DataStore (Tink) и Keystore напрямую

Современный подход — отделить хранение от шифрования: класть данные в (Proto) DataStore, а сам поток байт шифровать через актуальную версию Tink перед записью на диск. Это даёт контроль над схемой ключей и обходит баги старого Tink в Jetpack Security. Альтернативы-обёртки: сообщество выпустило, например, Ackee Guardian как преемника Jetpack Security; на низком уровне можно использовать KeyGenParameterSpec + Cipher с ключом из Keystore.

KeystoreCipher.kt
// Генерируем неэкспортируемый AES-ключ прямо в Android Keystore
fun getOrCreateKey(alias: String): SecretKey {
    val ks = KeyStore.getInstance("AndroidKeyStore").apply { load(null) }
    (ks.getEntry(alias, null) as? KeyStore.SecretKeyEntry)?.let { return it.secretKey }

    val gen = KeyGenerator.getInstance(KeyProperties.KEY_ALGORITHM_AES, "AndroidKeyStore")
    gen.init(
        KeyGenParameterSpec.Builder(alias,
            KeyProperties.PURPOSE_ENCRYPT or KeyProperties.PURPOSE_DECRYPT)
            .setBlockModes(KeyProperties.BLOCK_MODE_GCM)
            .setEncryptionPaddings(KeyProperties.ENCRYPTION_PADDING_NONE)
            .build()
    )
    return gen.generateKey()   // приватный материал не покидает TEE
}

Шифрование самой БД — SQLCipher

«Encrypted SQLite» из вопроса кандидата — это про SQLCipher: расширение SQLite, прозрачно шифрующее весь файл БД (AES-256) по парольной фразе. Подключается к Room через SupportFactory (openHelperFactory) — DAO и запросы не меняются.

EncryptedRoom.kt
// passphrase лучше держать как ключ в Keystore, а не строкой в коде
val passphrase: ByteArray = SQLiteDatabase.getBytes(secretChars)
val factory = SupportFactory(passphrase)

val db = Room.databaseBuilder(context, AppDatabase::class.java, "encrypted.db")
    .openHelperFactory(factory)   // вся БД шифруется прозрачно
    .build()
⚠️
Ключ — слабое звено. Шифрование бесполезно, если passphrase зашит в коде или лежит в обычных prefs. Его хранят как ключ в Android Keystore или производят из биометрии/PIN. Важно помнить: шифрование защищает от извлечения файла, но не от рантайм-доступа на скомпрометированном (rooted) устройстве, где процесс держит ключ в памяти.

Когда ключ Keystore внезапно «протухает»

Ключ, созданный с setUserAuthenticationRequired(true), привязан к экрану блокировки. Снимет пользователь блокировку — ключ инвалидируется навсегда, и при следующем использовании прилетает KeyPermanentlyInvalidatedException. По умолчанию такой ключ ещё и инвалидируется при добавлении нового отпечатка или лица — это setInvalidatedByBiometricEnrollment(true) (значение по умолчанию). Логика защитная: если злоумышленник дописал свою биометрию, он не должен получить доступ к старым данным. Передадите false — ключ переживёт ре-энролл ценой ослабления модели угроз.

Жизненный цикл ключа Keystore: переход в «Инвалидирован» бросает KeyPermanentlyInvalidatedException — приложение обязано его ловить
⚠️
Ловите KeyPermanentlyInvalidatedException. При init Cipher или использовании ключа поймайте это исключение, удалите старый алиас (keyStore.deleteEntry), сгенерируйте новый ключ и считайте ранее зашифрованные им данные безвозвратно потерянными — пользователя придётся пере-аутентифицировать. Для аппаратной изоляции есть setIsStrongBoxBacked(true): ключ живёт в отдельном Secure Element (API 28+), наличие StrongBox проверяют через FEATURE_STRONGBOX_KEYSTORE с фолбэком.
Почему ключ Android Keystore вдруг становится невалидным и как это связано с биометрией? Что делают setUserAuthenticationRequired и setInvalidatedByBiometricEnrollment? ★

Ключ с setUserAuthenticationRequired(true) завязан на secure lock screen. Снимет пользователь блокировку экрана — ключ инвалидируется навсегда, и попытка его использовать кинет KeyPermanentlyInvalidatedException. По умолчанию такой ключ инвалидируется и при добавлении новой биометрии — это поведение setInvalidatedByBiometricEnrollment(true) (дефолт). Смысл защиты: тот, кто дописал свой отпечаток, не должен открыть данные, зашифрованные до этого.

Передав false, можно оставить ключ валидным при ре-энролле — но это сознательное ослабление. Поэтому приложение обязано ловить исключение, пересоздавать ключ и помечать зашифрованные им данные потерянными. Отдельно: setIsStrongBoxBacked(true) переносит ключ в выделенный Secure Element (API 28+), а не только в TEE; наличие StrongBox проверяют через FEATURE_STRONGBOX_KEYSTORE, потому что он есть не на всех устройствах.

Room и SQLite: реляционное хранилище

Для структурированных данных со связями и сложными запросами — Room, official ORM-обёртка над SQLite. «Голый» SQLite работает со строковыми SQL без проверок, требует ручного маппинга Cursor и легко даёт ошибки в рантайме. Room проверяет SQL на этапе компиляции, генерирует boilerplate, интегрируется с корутинами/Flow и даёт инструменты для миграций.

Сущность, DAO, база

UserDao.kt
@Entity(
    tableName = "users",
    indices = [Index(value = ["email"], unique = true)]   // ускоряет поиск + уникальность
)
data class User(
    @PrimaryKey(autoGenerate = true) val id: Long = 0,
    val name: String,
    val email: String
)

@Dao
interface UserDao {
    // реактивный запрос: эмитит новый список при любом изменении таблицы
    @Query("SELECT * FROM users ORDER BY name")
    fun observeAll(): Flow<List<User>>

    @Query("SELECT * FROM users WHERE id = :id")
    suspend fun getById(id: Long): User?

    @Insert(onConflict = OnConflictStrategy.REPLACE)
    suspend fun upsert(user: User): Long

    // атомарная операция из нескольких шагов
    @Transaction
    suspend fun replaceAll(users: List<User>) {
        deleteAll(); insertAll(users)
    }

    @Query("DELETE FROM users") suspend fun deleteAll()
    @Insert suspend fun insertAll(users: List<User>)
}

Миграции схемы

При изменении схемы повышаешь version и описываешь переход Migration(from, to) с явным SQL. Room экспортирует схему в JSON (room.schemaLocation) — это «история» БД, по которой работает тест миграций. Для простых случаев есть @AutoMigration; если объявлены оба пути на одну версию, приоритет у ручной миграции.

migrations.kt
val MIGRATION_1_2 = object : Migration(1, 2) {
    override fun migrate(db: SupportSQLiteDatabase) {
        // используем ПОЛНЫЕ запросы, а не константы DAO (схема могла измениться)
        db.execSQL("ALTER TABLE users ADD COLUMN created_at INTEGER NOT NULL DEFAULT 0")
        db.execSQL("CREATE INDEX IF NOT EXISTS idx_users_email ON users(email)")
    }
}

Room.databaseBuilder(context, AppDatabase::class.java, "app.db")
    .addMigrations(MIGRATION_1_2)
    // .fallbackToDestructiveMigration()  // ОПАСНО: дропнет данные при отсутствии пути
    .build()
⚠️
fallbackToDestructiveMigration — не в проде. Если для версии нет Migration, Room по умолчанию падает с IllegalStateException. Деструктивный fallback «решает» это пересозданием БД с потерей всех данных пользователя. Допустимо для кэша/dev-сборок, недопустимо для пользовательских данных. Не забывай обновлять индексы внутри миграций.

FTS, отношения и тестирование

Для полнотекстового поиска — @Fts4-сущности (виртуальная таблица FTS, поиск по MATCH вместо LIKE %...%). Связи описывают через @Relation/@Embedded и читают в @Transaction-методах. Миграции тестируют через MigrationTestHelper: создают БД на старой версии, вставляют данные сырым SQL, открывают с новой версией и валидируют, что данные на месте и схема совпала.

MigrationTest.kt
@get:Rule val helper = MigrationTestHelper(
    InstrumentationRegistry.getInstrumentation(),
    AppDatabase::class.java
)

@Test fun migrate1To2_keepsData() {
    helper.createDatabase(TEST_DB, 1).apply {
        execSQL("INSERT INTO users (id, name, email) VALUES (1, 'Ann', 'a@x.io')")
        close()
    }
    // runMigrationsAndValidate проверит схему по экспортированному JSON
    helper.runMigrationsAndValidate(TEST_DB, 2, true, MIGRATION_1_2)
}

Как Flow-запрос узнаёт об изменении: InvalidationTracker

Реактивность Room держится на InvalidationTracker. При сборке БД Room создаёт служебную таблицу room_table_modification_log и вешает на каждую наблюдаемую таблицу SQL-триггеры на INSERT/UPDATE/DELETE, помечающие tableId изменённым. Flow- или LiveData-запрос регистрирует наблюдателя на нужные таблицы; после коммита транзакции трекер через refreshVersionsAsync опрашивает лог, находит затронутые таблицы и перезапускает привязанные запросы на query-executor — оттуда и приходит новый эмит.

sequenceDiagram participant W as "Writer (INSERT)" participant T as "SQL-триггер" participant I as "InvalidationTracker" participant Q as "QueryExecutor" participant F as "Flow-collector" W->>T: commit транзакции T->>I: пометить tableId в логе I->>I: refreshVersionsAsync (опрос лога) I->>Q: перезапустить подписанный запрос Q->>F: новый эмит результата
Путь от записи в таблицу до нового значения во Flow: инвалидация идёт на уровне таблицы, не строки
ℹ️
Инвалидация на уровне таблицы. Триггер срабатывает на любое изменение в таблице, а не на ваш конкретный WHERE. Поэтому Flow переэмитит даже когда правка не затронула выбранные строки — отсюда «лишние» перерисовки. Гасят их distinctUntilChanged() (требует data class с корректным equals) или более узкими наблюдаемыми сущностями/@DatabaseView.
Как Flow-запрос Room узнаёт, что таблицу изменили, и переэмитит результат? Почему он может сработать «лишний раз»? ★

За реактивность отвечает InvalidationTracker. При сборке БД Room создаёт таблицу room_table_modification_log и вешает на каждую наблюдаемую таблицу триггеры на INSERT/UPDATE/DELETE — они помечают tableId изменённым. Flow-запрос регистрирует наблюдателя; после коммита транзакции трекер периодически (refreshVersionsAsync) опрашивает лог, находит затронутые таблицы и перезапускает привязанные запросы на query-executor, эмитя новое значение.

Нюанс под капотом: инвалидация идёт на уровне таблицы, а не строки и не WHERE-условия. Любое изменение в таблице переэмитит ваш Flow, даже если выбранные строки не тронуты. Отсюда лишние переэмиты — глушат их через distinctUntilChanged() или сужением наблюдаемой сущности.

Потокобезопасность Room: WAL, single-writer, main-thread

Один из самых частых senior-вопросов про Room — про конкурентность: можно ли читать во время записи, сколько писателей одновременно, почему запрос нельзя дёргать с главного потока. Всё это вытекает из модели SQLite, поверх которой Room лишь добавляет эргономику.

Журнальный режим: rollback journal vs WAL

SQLite по умолчанию использует rollback journal: перед записью старые страницы копируются в -journal-файл, и на время записи писатель блокирует читателей. Room на устройствах с API 16+ включает WAL (Write-Ahead Logging): изменения сначала пишутся в отдельный -wal-файл, а в основную БД вливаются позже отдельной операцией checkpoint. Выигрыш — читатели больше не блокируют писателя и наоборот: несколько потоков читают согласованный снимок, пока один пишет.

ℹ️
Single-writer остаётся. WAL снимает блокировку чтение↔запись, но писатель по-прежнему один в каждый момент. Две параллельные записи сериализуются — и долгая транзакция-писатель не вешает читателей, но вешает других писателей. Это объясняет, почему «база тормозит на запись», а чтение при этом летает.

Запросы и главный поток

По умолчанию Room запрещает синхронные запросы на главном потоке и бросает IllegalStateException («Cannot access database on the main thread»): дисковый I/O на UI-потоке — это подлагивания и риск ANR. allowMainThreadQueries() снимает защиту, но это анти-паттерн — держат его разве что в тестах. Правильный путь: suspend-функции DAO и Flow/LiveData-запросы выполняются вне главного потока на внутренних executor'ах Room.

⚠️
withContext не управляет потоком suspend-DAO. Частый источник недопонимания: suspend-метод DAO использует собственные TransactionExecutor/QueryExecutor Room, поэтому обернуть его в withContext(Dispatchers.IO) бессмысленно — диспетчер не определяет, на каком потоке реально пойдёт запрос. Room сам уводит работу в свой пул.
Что такое режим WAL в SQLite, использует ли его Room по умолчанию и как он влияет на конкурентность чтения и записи? ★

WAL — журнальный режим, при котором изменения сперва пишутся в отдельный -wal-файл, а не перезаписывают страницы основной БД, и периодически вливаются туда операцией checkpoint. Room включает WAL по умолчанию на API 16+. Главный выигрыш: читатели больше не блокируют писателя и наоборот — несколько потоков читают параллельно согласованный снимок (до начала текущей записи), пока один пишет.

Ограничение никуда не делось: писатель в каждый момент один, две параллельные записи сериализуются. Цена WAL — дополнительные файлы -wal и -shm рядом с БД и необходимость checkpoint. Отсюда практический вывод: долгая транзакция-писатель не вешает запросы на чтение, но вешает другие записи.

Связи и сложные типы: TypeConverter, @Relation, @Upsert

Room — это SQLite, а SQLite знает только NULL, INTEGER, REAL, TEXT, BLOB. Всё остальное (Date, enum, список) надо превращать в эти типы — и тут начинаются нюансы, которые любят на собесе.

TypeConverter: как хранить Date, enum, List

Для нестандартного типа пишут @TypeConverter — пару функций «объект ↔ примитив», зарегистрированных через @TypeConverters на БД, DAO или поле.

Converters.kt
class Converters {
    @TypeConverter fun dateToLong(d: Date?): Long? = d?.time
    @TypeConverter fun longToDate(v: Long?): Date? = v?.let { Date(it) }

    // enum по ИМЕНИ, а не по ordinal — устойчиво к перестановке констант
    @TypeConverter fun statusToString(s: Status): String = s.name
    @TypeConverter fun stringToStatus(v: String): Status = Status.valueOf(v)
}
⚠️
Три грабли TypeConverter. (1) Список как JSON в одной колонке убивает возможность фильтровать/индексировать по элементам — это анти-нормализация, чаще нужна отдельная таблица со связью. (2) enum по ordinal ломается при изменении порядка констант — храните по name. (3) Конвертер вызывается на каждой строке: тяжёлая сериализация бьёт по производительности больших выборок.

@Relation и обязательность @Transaction при чтении

Связи описывают через @Embedded (вложить поля) и @Relation (подтянуть дочерние сущности). Под капотом Room делает для этого несколько запросов — родитель, затем дети — поэтому метод обязан быть @Transaction, иначе между запросами таблицу изменят и снимок окажется рассогласованным.

OrderDao.kt
data class OrderWithItems(
    @Embedded val order: Order,
    @Relation(parentColumn = "id", entityColumn = "orderId")
    val items: List<OrderItem>
)

@Dao interface OrderDao {
    // @Transaction нужен даже на ЧТЕНИИ: Room делает 2 запроса (заказ + позиции)
    @Transaction
    @Query("SELECT * FROM orders WHERE id = :id")
    suspend fun getOrderWithItems(id: Long): OrderWithItems
}

OnConflictStrategy и @Upsert

OnConflictStrategy задаёт поведение при нарушении ограничения (обычно UNIQUE/PRIMARY KEY): ABORT (дефолт — откат и исключение), REPLACE, IGNORE. Коварен именно REPLACE для @Insert: он реализован как DELETE старой строки + INSERT новой, а не UPDATE.

🛑
REPLACE при внешних ключах теряет данные. Если на таблице стоит ON DELETE CASCADE, удаление старой строки каскадно сносит связанные дочерние записи — и они не вернутся при вставке новой. Плюс REPLACE меняет rowid, ломая наблюдателей по конкретному id. С Room 2.4 есть @Upsert: настоящий UPDATE при конфликте по первичному ключу и INSERT иначе, без удаления строки. Для обновления существующих записей @Upsert обычно правильнее, чем @Insert(REPLACE).
Чем опасен OnConflictStrategy.REPLACE при внешних ключах и почему @Upsert часто корректнее? ★

REPLACE для @Insert — это не UPDATE, а DELETE старой строки + INSERT новой. При ON DELETE CASCADE по внешним ключам удаление старой строки каскадно сносит связанные дочерние записи, и они не восстановятся при вставке новой — частая скрытая потеря данных. Вдобавок меняется rowid, что ломает наблюдателей, завязанных на конкретный id.

@Upsert (Room 2.4+) делает настоящий UPDATE при конфликте по первичному ключу и INSERT иначе, не удаляя строку — каскад не срабатывает, rowid сохраняется. Поэтому для «вставить или обновить» по существующему ключу берут @Upsert, а не @Insert(onConflict = REPLACE).

Индексы под капотом: когда помогают, когда вредят

В главе индекс пока показан только как «ускоряет поиск + уникальность». На senior-уровне ждут понимания обратной стороны: индекс — это отдельная B-tree-структура, которую SQLite обязан обновлять при каждом INSERT/UPDATE/DELETE затронутой колонки, и которая занимает место.

Когда индекс вредит

  • Тяжёлая запись, редкие выборки. На такой таблице лишние индексы замедляют запись и раздувают БД, окупаясь редко.
  • Низкая кардинальность. Индекс по колонке вроде done BOOLEAN почти бесполезен — оптимизатор всё равно предпочтёт full scan.

Составной и covering индекс

Составной индекс полезен, когда запросы фильтруют/сортируют по префиксу его колонок именно в этом порядке — leftmost-prefix rule. Covering index содержит все колонки запроса, чтобы SQLite ответил, не заглядывая в саму таблицу.

Feed.kt
@Entity(
    tableName = "posts",
    // помогает запросам WHERE user_id = ? ORDER BY created_at
    indices = [Index(value = ["user_id", "created_at"])]
)
data class Post(
    @PrimaryKey val id: Long,
    val user_id: Long,
    val created_at: Long,
    val body: String
)
💡
Проверяй планом, а не на глаз. Реальный план запроса смотрят командой EXPLAIN QUERY PLAN — она покажет, использован ли индекс или идёт SCAN TABLE. Room на этапе компиляции предупреждает о потенциально неэффективных запросах (full table scan) через RoomWarnings. Баланс простой: больше индексов — быстрее чтение, медленнее запись и больше места.
Когда индекс в SQLite/Room вредит, а не помогает? Что такое covering index и зачем составной индекс? ★

Индекс — отдельная B-tree, которую SQLite обновляет при каждом изменении затронутой колонки и которая занимает место. Поэтому на таблице с интенсивной записью и редкими выборками лишние индексы замедляют запись и раздувают БД. Индекс по колонке низкой кардинальности (boolean done) почти бесполезен — оптимизатор всё равно выберет full scan.

Составной индекс Index(value=["user_id","created_at"]) полезен, когда запросы фильтруют/сортируют по этому префиксу колонок именно в таком порядке (leftmost-prefix rule). Covering index содержит все колонки запроса, чтобы ответить, не заглядывая в таблицу. План проверяют через EXPLAIN QUERY PLAN, а Room на компиляции предупреждает о full table scan через RoomWarnings.

Scoped storage (Android 10+)

До Android 10 с разрешением WRITE_EXTERNAL_STORAGE приложение видело весь общий раздел через прямые пути. Scoped storage это закрыл: приложение имеет доступ только к своей app-specific-директории и к медиа, которые само создало. Для остального — MediaStore (медиа) и Storage Access Framework (документы). С Android 11 (API 30) приложения, таргетящие 30+, обязаны использовать scoped storage; прямой File-доступ к чужим файлам не работает.

MediaStore: запись изображения без разрешений

SaveImage.kt
val values = ContentValues().apply {
    put(MediaStore.Images.Media.DISPLAY_NAME, "photo.jpg")
    put(MediaStore.Images.Media.MIME_TYPE, "image/jpeg")
    put(MediaStore.Images.Media.RELATIVE_PATH, "Pictures/MyApp")
}
val uri = resolver.insert(MediaStore.Images.Media.EXTERNAL_CONTENT_URI, values)
uri?.let { resolver.openOutputStream(it)?.use { os -> bitmap.compress(JPEG, 95, os) } }
// На Android 10+ запись своих медиа в общий раздел НЕ требует разрешений

FileProvider: безопасный обмен файлами

Делиться файлом из приватной директории через file:// запрещено (с Android 7 это FileUriExposedException). FileProvider отдаёт content://-URI с временным грантом доступа — безопасно для другого приложения (камера, «Поделиться»).

ShareFile.kt
val file = File(context.cacheDir, "report.pdf")
val uri = FileProvider.getUriForFile(context, "${context.packageName}.fileprovider", file)

val share = Intent(Intent.ACTION_SEND).apply {
    type = "application/pdf"
    putExtra(Intent.EXTRA_STREAM, uri)
    addFlags(Intent.FLAG_GRANT_READ_URI_PERMISSION)   // временный доступ получателю
}
context.startActivity(Intent.createChooser(share, "Поделиться"))

FileProvider заменяет небезопасный file:// на content:// с временным разрешением
Почему запись в MediaStore на Android 10 идёт через IS_PENDING, и зачем приложению с targetSdk 30+ может всё-таки понадобиться MANAGE_EXTERNAL_STORAGE? ★

На Android 10+ новое медиа через MediaStore рекомендуется сначала вставить с MediaStore.MediaColumns.IS_PENDING = 1: пока флаг выставлен, файл виден только создавшему его приложению. После записи флаг сбрасывают в 0, и файл появляется у остальных. Это даёт атомарность публикации — недозаписанный файл не мелькнёт в галерее. Для редактирования или удаления чужих медиа на Android 11+ система возвращает RecoverableSecurityException, и приложение показывает системный диалог через MediaStore.createWriteRequest/createDeleteRequest, чтобы получить согласие пользователя.

MANAGE_EXTERNAL_STORAGE (All files access) даёт широкий доступ ко всему общему разделу в обход scoped storage. Но Google Play разрешает его лишь узкому классу приложений — файловые менеджеры, бэкап-утилиты, антивирусы. Для обычного приложения это не путь: заявку на такое разрешение отклонят.

Auto Backup и безопасность в покое

Android Auto Backup по умолчанию (для targetSdk ≥ 23) сохраняет до 25 МБ данных приложения (включая SharedPreferences, файлы, БД) в облако пользователя и восстанавливает при переустановке. На собесе важно знать: бэкап может «утащить» чувствительные данные — управляй этим через android:fullBackupContent/android:dataExtractionRules (Android 12+), исключая токены и шифрованные хранилища, либо отключай android:allowBackup="false" для секретов.

💡
Принцип «защита в покое». Любые секреты — только в шифрованном виде с ключом из Keystore; индексы и БД с PII — через SQLCipher; в правилах бэкапа явно исключают чувствительные файлы. Шифрование защищает извлечённый файл, но не рантайм на rooted-устройстве — поэтому не стоит хранить больше, чем нужно (data minimization).

Auto Backup vs Key/Value Backup и перенос между устройствами

Видов бэкапа два. Auto Backup (для targetSdk ≥ 23) сам копирует файлы, БД и SharedPreferences целиком — до квоты (по умолчанию около 25 МБ на приложение) в Google Drive пользователя, без единой строки кода. Key/Value Backup (старый BackupAgent / Backup API) требует ручной реализации агента и копирует только явно зарегистрированные данные. С Android 12 (API 31) правила разделили: android:dataExtractionRules задаёт разные наборы исключений для облачного бэкапа (cloud-backup) и для прямого переноса между устройствами (device-transfer) — D2D локален и менее рискован, чем облако.

data_extraction_rules.xml
<data-extraction-rules>
    <cloud-backup>
        <exclude domain="sharedpref" path="auth_secure.xml"/>
        <exclude domain="file" path="tokens/"/>
    </cloud-backup>
    <device-transfer>
        <!-- при D2D токены можно переносить: канал локальный -->
    </device-transfer>
</data-extraction-rules>
⚠️
Ключи Keystore не переносятся. Аппаратные ключи привязаны к железу и при переносе на новое устройство теряются — а значит, зашифрованные ими данные на новом устройстве не расшифровать. Вывод: чувствительные файлы исключают из cloud-backup, а зашифрованные данные проектируют так, чтобы их можно было регенерировать (повторный логин, повторная синхронизация). android:allowBackup="false" — грубый выключатель сразу для всего.
Чем Auto Backup отличается от Key/Value Backup и почему важно различать D2D-перенос и облачный бэкап при обращении с секретами? ★

Auto Backup (для targetSdk ≥ 23) автоматически копирует файлы, БД и SharedPreferences целиком (до квоты ~25 МБ на приложение) в Google Drive пользователя без участия кода. Key/Value Backup (BackupAgent / старый Backup API) требует ручной реализации агента и копирует только явно зарегистрированные данные.

С Android 12 (API 31) android:dataExtractionRules позволяет задать разные наборы исключений для cloud-backup и device-transfer, потому что D2D-перенос локален и менее рискован, чем облако. Практический вывод: чувствительные файлы (токены, ключи) исключают из cloud-backup явными exclude, а ключи Keystore в принципе невосстановимы на новом устройстве — они привязаны к железу и при переносе теряются, поэтому зашифрованные ими данные надо уметь регенерировать.

Частые вопросы интервьюера

В чём разница между apply() и commit()? Middle

commit() пишет на диск синхронно в текущем потоке и возвращает Boolean — успех записи. apply() мгновенно обновляет in-memory и сбрасывает на диск асинхронно, возвращает Unit, об ошибке узнать нельзя. На UI-потоке предпочитают apply(), чтобы не блокировать UI. Нюанс, который ценят: apply() не совсем бесплатен — pending-записи дожидаются в QueuedWork на onPause()/onStop() и могут давать ANR при частых записях.

Почему индустрия уходит со SharedPreferences на DataStore? Middle+

Четыре причины: (1) синхронное чтение — первый доступ парсит весь XML на вызывающем потоке, риск блокировки UI; (2) нет сигнала об ошибках — apply() молча проглатывает сбои; (3) нет транзакционности и типобезопасности — легко получить рассогласованное состояние; (4) нет реактивности на корутинах — только громоздкий listener. DataStore решает всё это: полностью async на Flow/корутинах, атомарный edit{}, безопасная обработка повреждения файла, авто-миграция со SharedPreferences.

Preferences DataStore или Proto DataStore — когда что? Middle+

Preferences — динамические ключи без схемы: быстро для простых флагов и при миграции со SharedPreferences, но опечатка в ключе/типе вылезет в рантайме. Proto — строгая схема через protobuf и сгенерированные классы: полная типобезопасность, удобно для структурированного объекта настроек, ценой .proto-файла и сериализатора. Правило: простые независимые значения → Preferences; связный типизированный объект → Proto.

Как зашифровать данные? Что с EncryptedSharedPreferences? Senior

Корень доверия — Android Keystore: ключи генерируются в TEE/StrongBox и неэкспортируемы. Раньше для key-value использовали EncryptedSharedPreferences из Jetpack Security (поверх Tink). Сейчас важно знать: библиотека в maintenance, а EncryptedSharedPreferences deprecated (с security-crypto:1.1.0-alpha07, апрель 2025; причины — StrictMode-violations, keyset corruption на части OEM). Современный подход — Proto DataStore + актуальный Tink (отделяем хранение от шифрования) либо обёртки-преемники / прямое использование Keystore + Cipher. Для шифрования всей БД — SQLCipher.

Как зашифровать SQLite/Room без переписывания запросов? Senior

Через SQLCipher — он прозрачно шифрует весь файл БД (AES-256) по парольной фразе. В Room подключается одной строкой: создаётся SupportFactory(passphrase) и передаётся в openHelperFactory(). DAO, сущности и запросы не меняются. Критично: passphrase нельзя зашивать в коде или класть в обычные prefs — его хранят как ключ в Keystore или производят из биометрии/PIN. Шифрование защищает от извлечения файла, но не от рантайм-доступа на rooted-устройстве.

Как сделать миграцию схемы Room без потери данных? Senior

Повысить version и добавить объект Migration(from, to) с явным SQL (ALTER TABLE, обновление индексов). Включить экспорт схемы в JSON (room.schemaLocation) — это история БД для проверки. Тестировать через MigrationTestHelper: создать БД старой версии, вставить данные, прогнать runMigrationsAndValidate. Для простых случаев — @AutoMigration; при объявлении обоих путей на версию приоритет у ручного. Использовать полные запросы, а не константы DAO. Категорически не использовать fallbackToDestructiveMigration() для пользовательских данных — он дропает БД.

Что такое scoped storage и зачем он? Middle+

С Android 10 (API 29) приложение по умолчанию имеет доступ только к своей app-specific-директории и к медиа, которые само создало; прямой доступ ко всему общему разделу через WRITE_EXTERNAL_STORAGE закрыт. Для общих медиа — MediaStore (свои файлы — без разрешений), для документов — Storage Access Framework. С Android 11 (API 30) для targetSdk 30+ это обязательно. Цель — приватность: приложения не могут читать/менять чужие файлы без явного согласия пользователя.

Зачем FileProvider и почему нельзя отдать file:// напрямую? Middle

С Android 7 (API 24) передача file://-URI другому приложению кидает FileUriExposedException: чужой процесс не имеет прав на песочницу приложения, а голый путь — это утечка деталей файловой системы. FileProvider (наследник ContentProvider) выдаёт content://-URI и через флаг FLAG_GRANT_READ_URI_PERMISSION даёт получателю временный доступ ровно к этому файлу. Безопасный способ делиться файлами с камерой, «Поделиться» и т.п.

Почему Room, а не голый SQLite? Middle

Room проверяет SQL на этапе компиляции (ошибка у разработчика, а не у пользователя), автоматически маппит строки в объекты, генерирует boilerplate, даёт suspend-DAO и реактивные Flow-запросы (эмитят при изменении таблицы), а также инструменты миграций с экспортом схемы и тест-хелпером. Голый SQLite — строковый SQL без проверок, ручной Cursor-маппинг, никакой реактивности и полностью ручные миграции. Room — это тот же SQLite, но с безопасностью и эргономикой.

Чем транзакция SQLite отличается от аннотации @Transaction в Room? Middle+

@Transaction оборачивает тело DAO-метода в одну транзакцию SQLite (BEGIN/COMMIT, при исключении — ROLLBACK), давая атомарность составных операций. Но аннотация нужна и там, где записи нет: при чтении @Relation/@Embedded Room делает несколько отдельных запросов (родитель + дети), и без @Transaction между ними таблицу могут изменить — снимок окажется рассогласованным. То есть это и про consistency на чтении, а не только про откат на записи. На голом SQLite то же делают через beginTransaction()/setTransactionSuccessful()/endTransaction().

Как Room хранит типы, которых нет в SQLite (Date, enum, List)? Middle+

SQLite знает только NULL, INTEGER, REAL, TEXT, BLOB. Для остального пишут @TypeConverter — пару функций объект↔примитив, зарегистрированных через @TypeConverters. Типичные кейсы: Date↔Long (epoch millis), enum↔String, List↔JSON-строка. Грабли: хранение списка как JSON в одной колонке убивает фильтрацию/индексацию по элементам (часто лучше отдельная таблица); enum по name устойчивее, чем по ordinal (тот ломается при перестановке констант); конвертер вызывается на каждой строке — тяжёлая сериализация бьёт по большим выборкам.

Можно ли выполнять Room-запросы на главном потоке и что делает allowMainThreadQueries()? Middle+

По умолчанию Room запрещает синхронные запросы на главном потоке и бросает IllegalStateException («Cannot access database on the main thread») — дисковый I/O на UI грозит подлагиваниями и ANR. allowMainThreadQueries() снимает защиту, но это анти-паттерн (разве что для тестов). Правильно: suspend-функции DAO идут на внутреннем Executor Room, Flow/LiveData-запросы тоже считываются вне главного потока. Важно: suspend-DAO использует собственный executor Room, поэтому withContext не определяет, на каком потоке реально пойдёт запрос.

Зачем DataStore возвращает Flow при чтении, если значение по сути одно? Middle+

data — холодный Flow: при подписке отдаёт текущее значение и переэмитит при каждой успешной записи, чтобы UI обновлялся без ручных listener'ов. Получить значение разово можно через .first(), но делать это через runBlocking на главном потоке плохо — первое чтение читает и десериализует файл с диска, блокируя поток. updateData{} (низкоуровневый API под edit{}) выполняет read-modify-write атомарно под Mutex и сам suspend. Ключевое отличие от SharedPreferences.getString: у DataStore нет «дешёвого синхронного геттера» — это сознательное решение, чтобы исключить блокирующий I/O на UI.

Задача 1. У вас есть Room-база v1 с таблицей notes(id INTEGER, text TEXT). В v2 нужно добавить колонку created_at INTEGER NOT NULL с дефолтом 0 и создать индекс по ней. Напишите объект Migration(1, 2) и подключите его в базу. После этого в тесте через MigrationTestHelper проверьте, что строка, вставленная в v1, не потерялась.

Показать решение 
// --- Migration ---
val MIGRATION_1_2 = object : Migration(1, 2) {
    override fun migrate(db: SupportSQLiteDatabase) {
        db.execSQL(
            "ALTER TABLE notes ADD COLUMN created_at INTEGER NOT NULL DEFAULT 0"
        )
        db.execSQL(
            "CREATE INDEX IF NOT EXISTS idx_notes_created_at ON notes(created_at)"
        )
    }
}

// --- AppDatabase ---
@Database(entities = [Note::class], version = 2, exportSchema = true)
abstract class AppDatabase : RoomDatabase() {
    abstract fun noteDao(): NoteDao
}

fun buildDb(context: Context) =
    Room.databaseBuilder(context, AppDatabase::class.java, "notes.db")
        .addMigrations(MIGRATION_1_2)
        .build()

// --- MigrationTestHelper (instrumented test) ---
@get:Rule
val helper = MigrationTestHelper(
    InstrumentationRegistry.getInstrumentation(),
    AppDatabase::class.java
)

@Test fun migrate1To2_rowSurvives() {
    helper.createDatabase(TEST_DB, 1).apply {
        execSQL("INSERT INTO notes (id, text) VALUES (42, 'hello')")
        close()
    }
    val db = helper.runMigrationsAndValidate(TEST_DB, 2, true, MIGRATION_1_2)
    val cursor = db.query("SELECT id, text, created_at FROM notes WHERE id = 42")
    assert(cursor.moveToFirst())
    assert(cursor.getString(1) == "hello")
    assert(cursor.getLong(2) == 0L)
    cursor.close()
}

Объяснение: миграция использует полный SQL без ссылок на константы DAO — схема могла измениться, константы туда не подходят. runMigrationsAndValidate сравнивает итоговую схему с JSON-экспортом (нужен включённый room.schemaLocation) и роняет тест при расхождении. fallbackToDestructiveMigration() здесь нет — это намеренно: без миграции база упадёт с IllegalStateException, что лучше чем молчаливая потеря данных.

// Исправленный AuthManager.kt
class AuthManager(private val context: Context) {

    // (1) Чувствительный токен нельзя держать в открытом SharedPreferences.
    //     Используем DataStore + шифрование; сам DataStore — suspend/Flow, не блокирует UI.
    private val Context.tokenStore by preferencesDataStore(name = "auth_secure")

    private object Keys { val TOKEN = stringPreferencesKey("token") }

    suspend fun saveToken(token: String) {
        // edit{} атомарен и выполняется в IO-диспетчере DataStore, не блокирует main
        context.tokenStore.edit { it[Keys.TOKEN] = token }
    }

    // (2) file:// запрещён с Android 7 (FileUriExposedException).
    //     FileProvider даёт content:// с временным грантом.
    fun shareReport(file: File) {
        val uri = FileProvider.getUriForFile(
            context,
            "${context.packageName}.fileprovider",
            file
        )
        val intent = Intent(Intent.ACTION_SEND).apply {
            putExtra(Intent.EXTRA_STREAM, uri)
            type = "application/pdf"
            addFlags(Intent.FLAG_GRANT_READ_URI_PERMISSION)
        }
        context.startActivity(Intent.createChooser(intent, null))
    }

    // (3) DataStore читается через Flow — нет блокировки main-потока.
    //     Собираем в ViewModel/coroutineScope, не в onCreate напрямую.
    val tokenFlow: Flow<String?> = context.tokenStore.data
        .catch { e -> if (e is IOException) emit(emptyPreferences()) else throw e }
        .map { it[Keys.TOKEN] }
}

Объяснение трёх проблем: (1) commit() на main-потоке — синхронная запись, блокирует UI; к тому же открытый SharedPreferences не шифрует токен. (2) Uri.fromFile() кидает FileUriExposedException на API 24+ — нужен FileProvider. (3) Первое чтение SharedPreferences парсит весь XML на вызывающем потоке; в onCreate это блокировка main-потока, StrictMode-violation и потенциальный ANR.

Задача 3. Реализуйте DAO-метод, который атомарно заменяет весь список задач в таблице todos: удаляет старые и вставляет новые так, чтобы при сбое посередине база не осталась в промежуточном состоянии (ни пустой, ни с половиной строк). Также добавьте реактивный запрос, который отдаёт список незавершённых задач как Flow.

Показать решение 
@Entity(tableName = "todos")
data class Todo(
    @PrimaryKey(autoGenerate = true) val id: Long = 0,
    val title: String,
    val done: Boolean = false
)

@Dao
interface TodoDao {

    // Реактивный запрос: Room переэмитит при любом изменении таблицы todos
    @Query("SELECT * FROM todos WHERE done = 0 ORDER BY id")
    fun observePending(): Flow<List<Todo>>

    @Query("DELETE FROM todos")
    suspend fun deleteAll()

    @Insert
    suspend fun insertAll(todos: List<Todo>)

    // @Transaction гарантирует, что оба suspend-вызова выполняются в одной транзакции.
    // Если insertAll() бросит исключение — deleteAll() откатится автоматически.
    @Transaction
    suspend fun replaceAll(todos: List<Todo>) {
        deleteAll()
        insertAll(todos)
    }
}

Объяснение: без @Transaction два отдельных suspend-вызова — это две независимые транзакции; сбой между ними оставит базу пустой. Аннотация оборачивает весь метод в одну транзакцию SQLite: либо оба шага успешны, либо откат. Flow<List<Todo>> — холодный поток; Room сам переэмитит значение после каждого INSERT/DELETE в таблице todos, без ручных подписок или Listener.

Задача 4. Есть реактивный запрос val items: Flow<List<Item>> = dao.observeAll(), и UI дёргается лишний раз: список перерисовывается даже при изменении поля, которое не относится к выбранным строкам. Объясните причину и приведите минимальную правку, чтобы лишних перерисовок не было.

Показать решение 
// Причина: InvalidationTracker инвалидирует на уровне ТАБЛИЦЫ, а не строки/WHERE.
// Любой INSERT/UPDATE/DELETE в таблице items переэмитит Flow.

// Правка 1: distinctUntilChanged() в репозитории — одинаковый список дальше не пройдёт.
// Требует data class Item с корректным equals (сравнение по содержимому).
val items: Flow<List<Item>> = dao.observeAll().distinctUntilChanged()

// Правка 2 (если в таблице много нерелевантных изменений):
// сузить наблюдаемую сущность через @DatabaseView или вынести

Задача 6. Ключ Android Keystore с setUserAuthenticationRequired(true) используется для шифрования токена. После того как пользователь добавил новый отпечаток пальца, расшифровка падает с KeyPermanentlyInvalidatedException. Объясните причину и напишите корректную обработку.

Показать решение 
// Причина: ключ по умолчанию создан с setInvalidatedByBiometricEnrollment(true) —
// при добавлении новой биометрии он инвалидируется навсегда.

fun decryptToken(alias: String, data: ByteArray, iv: ByteArray): ByteArray? {
    return try {
        val key = getKey(alias) ?: return null
        Cipher.getInstance("AES/GCM/NoPadding").run {
            init(Cipher.DECRYPT_MODE, key, GCMParameterSpec(128, iv))
            doFinal(data)
        }
    } catch (e: KeyPermanentlyInvalidatedException) {
        // ключ мёртв: удаляем алиас, данные считаем потерянными
        KeyStore.getInstance("AndroidKeyStore").apply { load(null) }.deleteEntry(alias)
        getOrCreateKey(alias)          // новый ключ для будущих записей
        null                           // вызывающий код заставит пере-логиниться
    }
}

Объяснение: ключ с setUserAuthenticationRequired(true) по умолчанию инвалидируется при ре-энролле биометрии (защита от того, что чужая добавленная биометрия откроет старые данные). Ловим KeyPermanentlyInvalidatedException при использовании Cipher, удаляем старый алиас, генерируем новый ключ и помечаем зашифрованные старым ключом данные безвозвратно потерянными — пользователя надо пере-аутентифицировать. Альтернатива по требованиям продукта: при генерации передать setInvalidatedByBiometricEnrollment(false), чтобы ключ переживал ре-энролл ценой ослабления модели угроз.
