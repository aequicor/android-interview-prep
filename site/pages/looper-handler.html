Looper и Handler

Главный поток Android — это петля событий: Looper бесконечно достаёт сообщения из MessageQueue и отдаёт их Handler на обработку. На этом механизме держатся отрисовка кадров, доставка событий ввода и переключение корутин в Dispatchers.Main. На собесе проверяют, понимаете ли вы, почему долгая работа на главном потоке приводит к ANR и как сообщение попадает из фонового потока в UI.

Middle Senior Частый вопрос

Главный поток и петля событий

UI в Android — однопоточный: только главный (UI) поток вправе трогать View. Под ним крутится петля событий: Looper в цикле достаёт сообщения из MessageQueue и отдаёт их Handler на обработку. Любая долгая работа на этом потоке = пропуск кадров и риск ANR.

flowchart LR H["Handler.post / sendMessage"] --> Q["MessageQueue"] Q --> L["Looper.loop извлекает Message"] L --> D["Handler.dispatchMessage"] D --> UI["Обновление View на главном потоке"]
Петля событий: Handler кладёт сообщения в очередь, Looper последовательно их обрабатывает

Главный Looper приложения создаёт сама среда выполнения: метод ActivityThread.main() вызывает Looper.prepareMainLooper(), а затем Looper.loop() — и этот вызов не возвращается, пока живёт процесс. Поэтому весь жизненный цикл приложения происходит «внутри» петли главного потока.

Looper: один на поток

Обычный поток петли не имеет. Чтобы её завести, в этом потоке вызывают Looper.prepare() (создаёт Looper и его MessageQueue), а затем Looper.loop(). Ключевые гарантии:

  • Один Looper на поток. Повторный prepare() в том же потоке бросит RuntimeException("Only one Looper may be created per thread"). Связка хранится в ThreadLocal.
  • Один MessageQueue на Looper. Looper.myLooper() возвращает Looper текущего потока (или null), Looper.getMainLooper() — главный.
  • Главный Looper нельзя остановить. Он создан с запретом выхода, поэтому quit() на нём бросает IllegalStateException("Main thread not allowed to quit.").
// Превратить обычный поток в loop-поток вручную (учебный пример):
class LooperThread : Thread() {
    var handler: Handler? = null
    override fun run() {
        Looper.prepare()                       // создаём Looper + MessageQueue
        handler = Handler(Looper.myLooper()!!) { msg -> /* обработка */ true }
        Looper.loop()                          // блокируемся на обработке очереди
    }
}
ℹ️
Останавливать собственный Looper следует через quitSafely() (с API 18): он доставит уже наступившие по времени сообщения и завершит цикл, тогда как quit() обрывает очередь немедленно и может потерять сообщения. После завершения Looper отправка через Handler вернёт false.

MessageQueue: очередь по времени, а не FIFO

MessageQueue хранит сообщения упорядоченными по времени доставки (Message.when на базе SystemClock.uptimeMillis()), а не в порядке отправки. Отложенное сообщение вставляется в нужное место списка, и петля заберёт его, лишь когда подойдёт его время. Поэтому postDelayed(r, 100) и сразу post(r2) выполнятся в порядке r2, затем r.

Когда подходящих по времени сообщений нет, поток петли не крутит холостой цикл: он засыпает в нативном коде (через epoll на дескрипторе-eventfd), не потребляя CPU. Новое сообщение или наступление срока ближайшего отложенного будит поток. Это и позволяет главному потоку «ничего не делать» между кадрами без нагрузки на батарею.

💡
IdleHandler. Через MessageQueue.addIdleHandler { … } можно повесить колбэк, который вызывается, когда очередь опустела (или все оставшиеся сообщения отложены на будущее). Удобно для низкоприоритетной работы «когда поток свободен»; вернёте true — обработчик останется, false — снимется.

Handler: отправка и обработка сообщений

Handler привязывается к Looper в момент создания: он берёт Looper текущего потока и кэширует его очередь. Если в потоке нет подготовленного Looper, конструктор бросит RuntimeException. Через Handler переключаются в нужный поток: создав его с главным Looper, можно из фона выполнить код на UI.

Переключение в главный поток
// Handler на главный поток — вернуться в UI из фонового потока
val mainHandler = Handler(Looper.getMainLooper())
mainHandler.post { textView.text = "готово" }

// отложенная задача (база времени — uptimeMillis):
mainHandler.postDelayed({ hideLoader() }, 2_000)

// снять задачу, пока не выполнилась (например, при уходе с экрана):
mainHandler.removeCallbacks(/* тот же Runnable */ ::hideLoader)

Методы post* — это обёртки над sendMessage*: Runnable упаковывается в Message, у которого он становится колбэком. Сами Message берут из пула — Message.obtain() или handler.obtainMessage() вместо new Message(), чтобы не аллоцировать объекты на каждый кадр; после доставки сообщение перерабатывается (recycle()) автоматически.

⚠️
removeMessages/removeCallbacks и проверки hasMessages сканируют всю очередь — это O(n) по числу сообщений. Документация прямо предупреждает: на горячем пути их лучше избегать. Также помните: removeMessages(0) снесёт все сообщения без явного what, включая запощенные Runnable.

dispatchMessage: порядок обработки

Когда Looper извлёк сообщение, он вызывает msg.target.dispatchMessage(msg) — то есть отдаёт его тому Handler, что отправлял. Внутри — фиксированный порядок из трёх веток:

// упрощённо, как в исходниках Handler:
fun dispatchMessage(msg: Message) {
    if (msg.callback != null) {
        msg.callback.run()                 // 1) Runnable, отправленный через post()
    } else {
        if (mCallback != null &&          // 2) Handler.Callback из конструктора
            mCallback.handleMessage(msg)) {
            return                          // вернул true — обработку прекращаем
        }
        handleMessage(msg)                  // 3) переопределённый handleMessage()
    }
}

Сначала проверяется собственный колбэк сообщения (Runnable из post), затем — Handler.Callback, переданный в конструктор (если он вернул true, дальше не идём), и только потом — переопределённый метод handleMessage(). Знание этого порядка — типичный senior-вопрос.

Почему конструктор Handler() устарел

Безаргументный Handler() и Handler(Callback) помечены устаревшими начиная с Android 11 (API 30). Причина — неявный выбор Looper из текущего потока порождает трудноуловимые баги:

  • операции молча теряются, если Looper того потока уже завершился;
  • падения, если Handler иногда создаётся в потоке без активного Looper;
  • гонки — поток, к которому привязался Handler, оказывается не тем, что предполагал автор.
🛑
Как правильно. Указывайте Looper явно — Handler(Looper.getMainLooper()) — или используйте Executor. Если нужна именно привязка к текущему потоку, пишите Handler(Looper.myLooper()), чтобы намерение было очевидно читателю.

HandlerThread: фоновый поток с очередью

HandlerThread — это Thread с уже подготовленным Looper: удобно для последовательной фоновой обработки (например, запись в файл или БД из одного потока без гонок). Его getLooper() блокируется, пока Looper не инициализируется, поэтому Handler можно создавать сразу после start().

val ht = HandlerThread("io-worker").apply { start() }
val bgHandler = Handler(ht.looper)        // getLooper() дождётся готовности
bgHandler.post { writeToDisk() }          // последовательная фоновая работа
// при остановке:
ht.quitSafely()                            // доставит наступившие сообщения и завершит петлю
ℹ️
Сама документация HandlerThread советует прибегать к нему, только если нужно работать именно с Handler-API; в остальных случаях предпочтительны Executor/ExecutorService (см. «Executor и пулы потоков») или корутины.

Барьеры синхронизации, vsync и Dispatchers.Main

Помимо обычных, очередь поддерживает асинхронные сообщения и барьеры синхронизации. Барьер — это служебное сообщение без получателя (target == null); пока он в голове очереди, обычные (синхронные) сообщения «придерживаются», а исполняются только асинхронные. Это нужно, чтобы отрисовка кадра по сигналу vsync не задерживалась прочей работой.

Именно так устроена отрисовка: после View.invalidate() система ставит на очередь главного потока барьер, а колбэк отрисовки от Choreographer кладёт асинхронным сообщением, привязанным к следующему vsync. Когда кадр готов, барьер снимается.

ℹ️
Граница публичного API. Пометить сообщение асинхронным можно публично — Message.setAsynchronous(true) (API 22). А вот поставить барьер публичного API нет: postSyncBarrier()/removeSyncBarrier() скрыты (@hide). Это внутренний механизм платформы; знать его полезно для понимания, но вызывать в приложении нельзя.

Связь с корутинами: на Android Dispatchers.Main реализован поверх Handler к главному Looper. Диспетчеризация — это handler.post(block), а возобновление после delay — postDelayed с отменой через removeCallbacks. Причём используется асинхронный Handler (в обход vsync-барьера). Поэтому «переключиться в Main» — это, по сути, положить задачу в очередь главного потока. Подробнее о корутинах — в разделе «Корутины и Flow».

Связь с ANR: почему нельзя «грузить» главный поток

Поскольку обработка ввода и отрисовка идут через очередь главного потока, любая долгая операция в handleMessage/Runnable блокирует доставку следующих сообщений. Система отслеживает это и показывает диалог ANR (Application Not Responding):

  • Ввод не обработан за 5 секунд (нажатие, касание) — основной и самый известный порог из официальной документации по ANR.
  • У BroadcastReceiver — свои тайм-ауты, заданные системными константами (порядка 10 секунд для foreground-приёма, до минуты для фонового), которые ещё масштабируются на медленном железе.
⚠️
Вывод. Кадр при 60 fps — это ~16 мс на главном потоке; всё, что дольше, роняет частоту кадров (jank), а блокировка на секунды ведёт к ANR. Тяжёлую работу уводят в фон (Executor, HandlerThread, корутины на Dispatchers.IO/Default) и возвращаются на главный поток только для обновления View.

Частые вопросы интервьюера

Как связаны Looper, Handler и MessageQueue? Middle

Looper владеет одной MessageQueue и в цикле (loop()) достаёт из неё сообщения по времени доставки. Handler привязан к конкретному Looper: он кладёт Message/Runnable в его очередь (post/sendMessage) и он же их обрабатывает (dispatchMessage). На поток приходится не больше одного Looper (хранится в ThreadLocal), но Handler-ов на один Looper может быть много.

Как из фонового потока обновить UI? Middle

Переключиться на главный поток через его очередь: Handler(Looper.getMainLooper()).post { … } (или view.post { … }, activity.runOnUiThread { … }, а в корутинах — withContext(Dispatchers.Main), который под капотом делает тот же post). Напрямую трогать View из фона нельзя — будет CalledFromWrongThreadException.

Очередь сообщений — это FIFO? Middle+

Нет. MessageQueue упорядочена по времени доставки Message.when (база — uptimeMillis), а не по порядку отправки. Сообщение без задержки получает текущее время и встаёт после уже «просроченных»; отложенное — в позицию по своему времени. Поэтому postDelayed может выполниться позже отправленного позже, но «немедленного» сообщения.

В каком порядке Handler обрабатывает сообщение? Senior

dispatchMessage идёт по трём веткам: (1) если у сообщения есть свой callback (это Runnable из post()) — выполняется он; (2) иначе, если у Handler задан Callback из конструктора и его handleMessage вернул true — обработка завершается; (3) иначе вызывается переопределённый handleMessage(). То есть post-Runnable имеет приоритет, затем конструкторный Callback, затем метод класса.

Почему new Handler() без аргументов — плохо? Middle+

Он неявно берёт Looper текущего потока, и это источник багов: задачи молча теряются (если тот Looper уже завершился), приложение падает (если потока без Looper) или возникают гонки из-за привязки не к тому потоку. Поэтому конструктор устарел с Android 11 (API 30). Правильно — указывать Looper явно: Handler(Looper.getMainLooper()) или использовать Executor.

Что блокирует главный поток, когда сообщений нет? Senior

Ничего не «крутится» вхолостую: MessageQueue.next() уходит в нативный сон через epoll на дескрипторе-eventfd до тех пор, пока не появится новое сообщение или не наступит время ближайшего отложенного. Запись в этот дескриптор (например, при post из другого потока) будит петлю. Поэтому простаивающий главный поток не нагружает CPU и батарею, оставаясь готовым мгновенно обработать ввод.

Задачи

Задача 1. Найдите утечку памяти в коде ниже. Объясните, почему она возникает, и предложите исправление.

Показать решение 
// Исходный код с утечкой:
class MainActivity : AppCompatActivity() {
    private val handler = Handler(Looper.getMainLooper())

    override fun onCreate(savedInstanceState: Bundle?) {
        super.onCreate(savedInstanceState)
        handler.postDelayed({
            textView.text = "загрузка завершена"
        }, 30_000)
    }
}

// Исправление — отменяем задачу при уничтожении Activity:
class MainActivity : AppCompatActivity() {
    private val handler = Handler(Looper.getMainLooper())
    private val updateTask = Runnable {
        textView.text = "загрузка завершена"
    }

    override fun onCreate(savedInstanceState: Bundle?) {
        super.onCreate(savedInstanceState)
        handler.postDelayed(updateTask, 30_000)
    }

    override fun onDestroy() {
        super.onDestroy()
        handler.removeCallbacks(updateTask)
    }
}

Объяснение: лямбда, переданная в postDelayed, захватывает неявную ссылку на this (Activity), потому что обращается к textView. Если Activity закроют до истечения 30 секунд, система не сможет её собрать — Runnable удерживает ссылку через MessageQueue. Решение — хранить задачу как отдельный объект и убирать её через removeCallbacks в onDestroy.

Задача 2. Реализуйте класс SerialWorker, который принимает задачи (Runnable) и выполняет их строго последовательно в фоновом потоке. Создание нового потока на каждую задачу запрещено. Обеспечьте корректную остановку.

Показать решение 
class SerialWorker {
    private val thread = HandlerThread("serial-worker").apply { start() }
    private val handler = Handler(thread.looper)

    fun submit(task: Runnable) {
        handler.post(task)
    }

    fun submitDelayed(task: Runnable, delayMs: Long) {
        handler.postDelayed(task, delayMs)
    }

    fun cancel(task: Runnable) {
        handler.removeCallbacks(task)
    }

    /** Завершает после обработки уже поставленных в очередь задач. */
    fun shutdown() {
        thread.quitSafely()
    }
}

Объяснение: HandlerThread создаёт поток с уже готовым Looper — заводить его вручную через Looper.prepare() не нужно. Handler, привязанный к его looper, гарантирует последовательное выполнение: новая задача запустится, только когда предыдущая завершится. quitSafely() дождётся задач с наступившим временем доставки, quit() — обрубил бы очередь немедленно.

Задача 3. Предскажите порядок вывода для кода ниже. Объясните, почему именно такой.

Показать решение 
val handler = Handler(Looper.getMainLooper())

handler.postDelayed({ Log.d("Q", "A — delayed 100ms") }, 100)
handler.post { Log.d("Q", "B — immediate") }
handler.postDelayed({ Log.d("Q", "C — delayed 50ms") }, 50)
handler.post { Log.d("Q", "D — immediate") }

// Порядок вывода (при условии, что все четыре вызова выполнены
// подряд, до того как Looper успел обработать очередь):
// B, D, C, A

Объяснение: MessageQueue упорядочена по Message.when (время доставки на базе uptimeMillis), а не по порядку отправки. «Немедленные» сообщения B и D получают время ~T+0 и встают в очередь перед C (T+50) и A (T+100). Среди немедленных порядок вставки сохраняется: B зашёл первым — выйдет первым. Итог: B → D → C → A.
