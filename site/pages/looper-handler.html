Looper и Handler

Главный поток Android — это петля событий: Looper бесконечно достаёт сообщения из MessageQueue и отдаёт их Handler на обработку. На этом механизме держатся отрисовка кадров, доставка событий ввода и переключение корутин в Dispatchers.Main. На собесе проверяют, понимаете ли вы, почему долгая работа на главном потоке приводит к ANR и как сообщение попадает из фонового потока в UI.

Middle Senior Частый вопрос

Главный поток и петля событий

UI в Android — однопоточный: только главный (UI) поток вправе трогать View. Под ним крутится петля событий: Looper в цикле достаёт сообщения из MessageQueue и отдаёт их Handler на обработку. Любая долгая работа на этом потоке = пропуск кадров и риск ANR.

flowchart LR H["Handler.post / sendMessage"] --> Q["MessageQueue"] Q --> L["Looper.loop извлекает Message"] L --> D["Handler.dispatchMessage"] D --> UI["Обновление View на главном потоке"]
Петля событий: Handler кладёт сообщения в очередь, Looper последовательно их обрабатывает

Главный Looper приложения создаёт сама среда выполнения: метод ActivityThread.main() вызывает Looper.prepareMainLooper(), а затем Looper.loop() — и этот вызов не возвращается, пока живёт процесс. Поэтому весь жизненный цикл приложения происходит «внутри» петли главного потока.

Если Looper.loop() — бесконечный цикл, который не возвращается, почему он сам по себе не вызывает ANR? Поток же «вечно занят». ★

ANR измеряет не то, что поток крутится в loop(), а то, успевает ли он вовремя забрать и обработать конкретное сообщение — событие ввода, broadcast и так далее. Почти всё время loop() спит в MessageQueue.next() на нативном epoll и не ест CPU: спать — это не «быть занятым».

ANR случается, когда обработка одной итерации — ваш handleMessage или Runnable — длится так долго, что следующее сообщение (например, доставка input-события) не успевает обработаться за свой порог: 5 секунд для ввода. Виноват не цикл, а долгий колбэк внутри одной итерации, который не даёт вернуться в next() за следующим сообщением. Сторож отслеживает именно задержку обработки: InputDispatcher для приложений, Watchdog для system_server — а не факт, что поток сидит в loop().

Looper: один на поток

Обычный поток петли не имеет. Чтобы её завести, в этом потоке вызывают Looper.prepare() (создаёт Looper и его MessageQueue), а затем Looper.loop(). Ключевые гарантии:

  • Один Looper на поток. Повторный prepare() в том же потоке бросит RuntimeException("Only one Looper may be created per thread"). Связка хранится в ThreadLocal.
  • Один MessageQueue на Looper. Looper.myLooper() возвращает Looper текущего потока (или null), Looper.getMainLooper() — главный.
  • Главный Looper нельзя остановить. Он создан с запретом выхода, поэтому quit() на нём бросает IllegalStateException("Main thread not allowed to quit.").
// Превратить обычный поток в loop-поток вручную (учебный пример):
class LooperThread : Thread() {
    var handler: Handler? = null
    override fun run() {
        Looper.prepare()                       // создаём Looper + MessageQueue
        handler = Handler(Looper.myLooper()!!) { msg -> /* обработка */ true }
        Looper.loop()                          // блокируемся на обработке очереди
    }
}
ℹ️
Останавливать собственный Looper следует через quitSafely() (с API 18): он доставит уже наступившие по времени сообщения и завершит цикл, тогда как quit() обрывает очередь немедленно и может потерять сообщения. После завершения Looper отправка через Handler вернёт false.
Можно ли создать Handler на потоке без Looper, и что именно бросится — при создании или при первой отправке? ★

Бросится сразу в конструкторе Handler, а не при отправке. Конструктор без явного Looper делает Looper.myLooper() для текущего потока, получает null и бросает RuntimeException("Can't create handler inside thread … that has not called Looper.prepare()"). Дело в том, что Handler привязывается к Looper и кэширует его MessageQueue в момент создания — проверка происходит немедленно.

Чтобы Handler жил на обычном потоке, в этом потоке сначала вызывают Looper.prepare(), потом создают Handler, и только затем Looper.loop() запускает обработку. Главный поток — исключение: его Looper уже подготовлен через prepareMainLooper() в ActivityThread.main(), поэтому Handler(Looper.getMainLooper()) работает откуда угодно.

MessageQueue: очередь по времени, а не FIFO

MessageQueue хранит сообщения упорядоченными по времени доставки (Message.when на базе SystemClock.uptimeMillis()), а не в порядке отправки. Отложенное сообщение вставляется в нужное место списка, и петля заберёт его, лишь когда подойдёт его время. Поэтому postDelayed(r, 100) и сразу post(r2) выполнятся в порядке r2, затем r.

Когда подходящих по времени сообщений нет, поток петли не крутит холостой цикл: он засыпает в нативном коде (через epoll на дескрипторе-eventfd), не потребляя CPU. Новое сообщение или наступление срока ближайшего отложенного будит поток. Это и позволяет главному потоку «ничего не делать» между кадрами без нагрузки на батарею.

💡
IdleHandler. Через MessageQueue.addIdleHandler { … } можно повесить колбэк, который вызывается, когда очередь опустела (или все оставшиеся сообщения отложены на будущее). Удобно для низкоприоритетной работы «когда поток свободен»; вернёте true — обработчик останется, false — снимется.
Чем sendMessageAtFrontOfQueue() отличается от post() и почему документация называет его опасным? ★

sendMessageAtFrontOfQueue() (и postAtFrontOfQueue()) ставит сообщение в самое начало очереди с временем доставки 0, минуя сортировку по Message.when — оно обработается следующим, перед всеми уже стоящими. Документация прямо предупреждает: метод «следует использовать очень редко — он легко приводит к голоданию очереди (starvation), дедлокам и другим побочным эффектам».

Если постить так регулярно, обычные сообщения — включая отрисовку и ввод — рискуют вообще не дойти до обработки. И это не то же самое, что асинхронное сообщение в обход барьера: там играет тип сообщения, а здесь — позиция в очереди. В прикладном коде он почти никогда не нужен; обычный post() с правильными приоритетами безопаснее.

Handler: отправка и обработка сообщений

Handler привязывается к Looper в момент создания: он берёт Looper текущего потока и кэширует его очередь. Если в потоке нет подготовленного Looper, конструктор бросит RuntimeException. Через Handler переключаются в нужный поток: создав его с главным Looper, можно из фона выполнить код на UI.

Переключение в главный поток
// Handler на главный поток — вернуться в UI из фонового потока
val mainHandler = Handler(Looper.getMainLooper())
mainHandler.post { textView.text = "готово" }

// отложенная задача (база времени — uptimeMillis):
mainHandler.postDelayed({ hideLoader() }, 2_000)

// снять задачу, пока не выполнилась (например, при уходе с экрана):
mainHandler.removeCallbacks(/* тот же Runnable */ ::hideLoader)

Методы post* — это обёртки над sendMessage*: Runnable упаковывается в Message, у которого он становится колбэком. Сами Message берут из пула — Message.obtain() или handler.obtainMessage() вместо new Message(), чтобы не аллоцировать объекты на каждый кадр; после доставки сообщение перерабатывается (recycle()) автоматически.

⚠️
removeMessages/removeCallbacks и проверки hasMessages сканируют всю очередь — это O(n) по числу сообщений. Документация прямо предупреждает: на горячем пути их лучше избегать. Также помните: removeMessages(0) снесёт все сообщения без явного what, включая запощенные Runnable.
Почему рекомендуют Message.obtain() вместо new Message(), и что будет, если вручную вызвать recycle() на сообщении, которое ещё в очереди? ★

Message держит глобальный пул переиспользуемых объектов — связный список через поле next, ограниченный MAX_POOL_SIZE = 50. obtain() берёт сообщение из пула, а после доставки Looper сам вызывает recycleUnchecked() и возвращает объект обратно. На горячем пути (например, по сообщению на кадр) это убирает аллокации и нагрузку на GC; new Message() пул не использует.

Вручную вызвать recycle() на сообщении, которое уже отправлено и ещё лежит в очереди, нельзя — будет IllegalStateException("This message cannot be recycled because it is still in use"). Сообщение помечено флагом FLAG_IN_USE с момента энкью и до конца доставки. Переработка идёт автоматически; ручной recycle() нужен только для сообщений, которые вы создали, но так и не отправили.

dispatchMessage: порядок обработки

Когда Looper извлёк сообщение, он вызывает msg.target.dispatchMessage(msg) — то есть отдаёт его тому Handler, что отправлял. Внутри — фиксированный порядок из трёх веток:

// упрощённо, как в исходниках Handler:
fun dispatchMessage(msg: Message) {
    if (msg.callback != null) {
        msg.callback.run()                 // 1) Runnable, отправленный через post()
    } else {
        if (mCallback != null &&          // 2) Handler.Callback из конструктора
            mCallback.handleMessage(msg)) {
            return                          // вернул true — обработку прекращаем
        }
        handleMessage(msg)                  // 3) переопределённый handleMessage()
    }
}

Сначала проверяется собственный колбэк сообщения (Runnable из post), затем — Handler.Callback, переданный в конструктор (если он вернул true, дальше не идём), и только потом — переопределённый метод handleMessage(). Знание этого порядка — типичный senior-вопрос.

flowchart TD M["Looper отдал Message"] --> C1{"msg.callback != null?"} C1 -->|да| R["Выполнить Runnable из post"] C1 -->|нет| C2{"Callback из конструктора вернул true?"} C2 -->|да| S["Стоп: сообщение поглощено"] C2 -->|нет| H["handleMessage() класса"]
Три ветки dispatchMessage по убыванию приоритета: post-Runnable → Callback → handleMessage
Конструкторный Handler.Callback вернул false. Что произойдёт дальше и в чём смысл возвращаемого значения? ★

Возврат из Handler.Callback.handleMessage(msg) работает как «поглотил ли я сообщение». Вернул true — dispatchMessage завершается, переопределённый handleMessage() класса не вызывается. Вернул false — обработка продолжается и управление уходит в handleMessage(msg) самого класса.

Так на готовый Handler можно навесить Callback-перехватчик и пофактно решать: перехватить сообщение или пропустить дальше в штатную обработку. Важная оговорка — ветка с Callback вообще достигается, только если у сообщения нет собственного callback (Runnable из post): тот имеет наивысший приоритет, и тогда ни Callback, ни handleMessage не зовутся.

Почему конструктор Handler() устарел

Безаргументный Handler() и Handler(Callback) помечены устаревшими начиная с Android 11 (API 30). Причина — неявный выбор Looper из текущего потока порождает трудноуловимые баги:

  • операции молча теряются, если Looper того потока уже завершился;
  • падения, если Handler иногда создаётся в потоке без активного Looper;
  • гонки — поток, к которому привязался Handler, оказывается не тем, что предполагал автор.
🛑
Как правильно. Указывайте Looper явно — Handler(Looper.getMainLooper()) — или используйте Executor. Если нужна именно привязка к текущему потоку, пишите Handler(Looper.myLooper()), чтобы намерение было очевидно читателю.
Что вернёт post(), если Looper, к которому привязан Handler, уже завершён через quit()/quitSafely()? ★

Вернёт false — и никакого исключения. После quit() очередь помечается как «не принимающая» сообщения: enqueueMessage возвращает false, и это значение прокидывается наружу из post/sendMessage. Сообщение в очередь не попадёт и обработано не будет.

Это ровно тот молчаливый сбой, ради защиты от которого и устарел безаргументный Handler(). Классическая ловушка с HandlerThread: остановили поток через quitSafely(), потом запостили задачу — она тихо потерялась. В граничных сценариях возвращаемое значение post стоит проверять, а не игнорировать.

HandlerThread: фоновый поток с очередью

HandlerThread — это Thread с уже подготовленным Looper: удобно для последовательной фоновой обработки (например, запись в файл или БД из одного потока без гонок). Его getLooper() блокируется, пока Looper не инициализируется, поэтому Handler можно создавать сразу после start().

val ht = HandlerThread("io-worker").apply { start() }
val bgHandler = Handler(ht.looper)        // getLooper() дождётся готовности
bgHandler.post { writeToDisk() }          // последовательная фоновая работа
// при остановке:
ht.quitSafely()                            // доставит наступившие сообщения и завершит петлю
ℹ️
Сама документация HandlerThread советует прибегать к нему, только если нужно работать именно с Handler-API; в остальных случаях предпочтительны Executor/ExecutorService (см. «Executor и пулы потоков») или корутины.
Почему getLooper() может вернуть null сразу после start(), и как это решено внутри класса? ★

start() лишь планирует запуск потока, а run() с Looper.prepare() выполнится асинхронно — между ними есть гонка. Внутри HandlerThread.getLooper() её закрывают через wait/notifyAll: если поток уже жив (isAlive()), но mLooper ещё null, метод блокируется в synchronized-блоке на wait(). А run() после Looper.prepare() выставляет mLooper и делает notifyAll() — будит ожидающих. Поэтому вызывать getLooper() сразу после start() безопасно: он сам дождётся готовности.

Но если поток так и не стартовали (start() не вызвали), getLooper() вернёт null без блокировки. Отсюда правило: всегда start() перед обращением к looper, иначе Handler(thread.looper) упадёт на null.

Барьеры синхронизации, vsync и Dispatchers.Main

Помимо обычных, очередь поддерживает асинхронные сообщения и барьеры синхронизации. Барьер — это служебное сообщение без получателя (target == null); пока он в голове очереди, обычные (синхронные) сообщения «придерживаются», а исполняются только асинхронные. Это нужно, чтобы отрисовка кадра по сигналу vsync не задерживалась прочей работой.

Именно так устроена отрисовка: после View.invalidate() система ставит на очередь главного потока барьер, а колбэк отрисовки от Choreographer кладёт асинхронным сообщением, привязанным к следующему vsync. Когда кадр готов, барьер снимается.

ℹ️
Граница публичного API. Пометить сообщение асинхронным можно публично — Message.setAsynchronous(true) (API 22). А вот поставить барьер публичного API нет: postSyncBarrier()/removeSyncBarrier() скрыты (@hide). Это внутренний механизм платформы; знать его полезно для понимания, но вызывать в приложении нельзя.

Связь с корутинами: на Android Dispatchers.Main реализован поверх Handler к главному Looper. Диспетчеризация — это handler.post(block), а возобновление после delay — postDelayed с отменой через removeCallbacks. Причём используется асинхронный Handler (в обход vsync-барьера). Поэтому «переключиться в Main» — это, по сути, положить задачу в очередь главного потока. Подробнее о корутинах — в разделе «Корутины и Flow».

Барьер придерживает синхронные сообщения, а асинхронное (кадр от Choreographer) проходит к Looper в обход
Что физически случится с очередью главного потока, если на ней повиснет sync-барьер и его забудут снять? ★

Барьер — служебное сообщение с target == null в голове очереди. Пока он там, MessageQueue.next() пропускает все синхронные (обычные) сообщения и отдаёт Looper только асинхронные — помеченные setAsynchronous(true). Если барьер не снять через removeSyncBarrier(), обычные сообщения — большинство ваших post() и handleMessage — будут вечно придерживаться и никогда не обработаются, хотя асинхронные (кадры от Choreographer) продолжат идти.

Снаружи это выглядит дико: «UI завис, но анимации идут». Редкий, но реальный класс багов в платформенном и библиотечном коде. В норме систему страхует то, что Choreographer снимает барьер по завершении кадра; постить барьер из приложения нельзя — postSyncBarrier()/removeSyncBarrier() помечены @hide.

Связь с ANR: почему нельзя «грузить» главный поток

Поскольку обработка ввода и отрисовка идут через очередь главного потока, любая долгая операция в handleMessage/Runnable блокирует доставку следующих сообщений. Система отслеживает это и показывает диалог ANR (Application Not Responding):

  • Ввод не обработан за 5 секунд (нажатие, касание) — основной и самый известный порог из официальной документации по ANR.
  • У BroadcastReceiver — свои тайм-ауты, заданные системными константами (порядка 10 секунд для foreground-приёма, до минуты для фонового), которые ещё масштабируются на медленном железе.
⚠️
Вывод. Кадр при 60 fps — это ~16 мс на главном потоке; всё, что дольше, роняет частоту кадров (jank), а блокировка на секунды ведёт к ANR. Тяжёлую работу уводят в фон (Executor, HandlerThread, корутины на Dispatchers.IO/Default) и возвращаются на главный поток только для обновления View.

Порог 5 секунд — самый известный, но не единственный. У broadcast, service и content-provider свои тайм-ауты, и все эти колбэки крутятся на главном потоке того же процесса — то есть в той же очереди, что и UI. Поэтому корень всегда один: долгий колбэк блокирует Looper. Подробнее о компонентах и фоне — в главе «Фоновая работа».

Тип ANRПорог foregroundПорог backgroundГде выполняется колбэк
Input (ввод)5 с—Главный поток
BroadcastReceiver~10 с~60 сГлавный поток
Service~20 с~200 сГлавный поток
ContentProvider~10 с—Главный поток

На медленном железе пороги масштабируются

Значения выше — базовые; система умножает их на коэффициент для слабых устройств, так что точные секунды плавают. Знание разных типов отличает кандидата, который видел реальные ANR-трейсы, от того, кто помнит только «5 секунд».

BroadcastReceiver.onReceive() и Service.onStartCommand() идут на главном потоке. Почему тяжёлая работа в onReceive может вызвать ANR в совсем другой Activity? ★

Колбэки receiver, service и обычно content-provider работают на главном потоке того же процесса — то есть в той же единственной очереди главного Looper, что и UI. Долгий onReceive() занимает поток и не возвращает управление в loop(), поэтому уже стоящие в очереди input-сообщения активной Activity не успевают обработаться за свой порог (5 с) — и ANR диагностируется «в Activity», хотя настоящая причина в receiver.

У каждого компонента ещё и свой тайм-аут (см. таблицу выше): ввод — 5 с; foreground-broadcast — 10 с, background — 60 с; foreground-service — 20 с, background — 200 с; content-provider — 10 с. Вывод один: любую заметную работу в этих колбэках уводят с главного потока, а сам колбэк держат коротким.

Утечки через Handler: static-класс и WeakReference

Самый частый senior-вопрос по теме — и любимое предупреждение Lint: «This Handler class should be static or leaks might occur». Корень в том, что нестатический внутренний (или анонимный) Handler держит неявную ссылку на внешний объект, обычно Activity. Пока в очереди лежит хоть одно неотправленное сообщение — особенно отложенное через postDelayed, — оно через цепочку удержания не даёт собрать Activity даже после её уничтожения.

flowchart LR Q["MessageQueue"] --> M["Message"] M -->|"target"| H["Handler (внутренний)"] H -->|"неявный this"| A["Activity (уничтожена)"]
Цепочка утечки: отложенное сообщение держит Handler, тот — внешнюю Activity

Отложенные задачи опаснее всего: postDelayed(r, 60_000) живёт в очереди минуту независимо от того, что экран давно закрыли. Лечений три, и они дополняют друг друга.

Static-вложенный Handler с WeakReference
class MyActivity : AppCompatActivity() {
    private class UiHandler(activity: MyActivity) : Handler(Looper.getMainLooper()) {
        private val ref = WeakReference(activity)
        override fun handleMessage(msg: Message) {
            val activity = ref.get() ?: return   // Activity уже собрана — выходим
            activity.textView.text = "готово"
        }
    }

    private val handler = UiHandler(this)

    override fun onDestroy() {
        super.onDestroy()
        handler.removeCallbacksAndMessages(null) // снять всё, что отправлял этот Handler
    }
}
  • Static inner + WeakReference. Статический класс не держит внешний this, а слабая ссылка позволяет собрать Activity; в handleMessage проверяем ref.get() на null.
  • Очистка очереди в onDestroy(). removeCallbacksAndMessages(null) снимает все сообщения и Runnable этого Handler — отложенные задачи перестают удерживать что-либо.
  • Lifecycle-aware скоупы. На современном Android ручной Handler чаще заменяют на lifecycleScope.launch { delay(60_000); … } или viewModelScope — они сами отменяются при уничтожении lifecycle.
⚠️
Нестатический + WeakReference не спасает. Если класс Handler остался нестатическим, он всё равно держит сильную неявную ссылку на внешний объект — никакая слабая ссылка внутри этого не отменит. Слабой должна быть именно ссылка на Activity внутри static-класса.
Чем removeCallbacksAndMessages(null) отличается от removeCallbacks/removeMessages? ★

removeCallbacks(r) снимает конкретный Runnable, removeMessages(what) — сообщения с конкретным what, а removeCallbacksAndMessages(token) — все сообщения и Runnable с заданным токеном. Переданный null означает «снять вообще всё, что отправлял этот Handler» — удобный способ полностью очистить его очередь в onDestroy.

Две оговорки. Во-первых, это O(n)-скан очереди. Во-вторых, с null снесутся и «чужие» отложенные задачи этого же Handler, если их не различать токеном, — поэтому когда нужно точечно, постят с токеном (post(r, token)) и чистят по нему.

База времени postDelayed: uptimeMillis против реального времени

Каверзный senior-вопрос: «сработает ли postDelayed(_, 10_000), если устройство за это время уснёт?». Ответ — нет. postDelayed и Message.when опираются на SystemClock.uptimeMillis() — монотонные миллисекунды с момента загрузки, которые не идут во время глубокого сна (deep sleep) и не зависят от того, перевёл ли пользователь системные часы.

Отсюда практическая граница: postDelayed гарантирует «через N мс активного времени», а не «через N мс настенного времени». Уснуло устройство — таймер фактически встал. Для задач, которые должны срабатывать по реальному времени и будить устройство, Handler не годится: нужен AlarmManager (например, setExactAndAllowWhileIdle) или WorkManager. Поэтому на postDelayed нельзя строить точные напоминания — подробнее в главе «Фоновая работа».

currentTimeMillis()ДаДаНастенное время, метки в БД
Почему интервал нельзя замерять через System.currentTimeMillis(), а для таймеров берут uptimeMillis? ★

currentTimeMillis() — настенные часы: пользователь, NTP-синхронизация или смена часового пояса могут прыгнуть ими вперёд или назад прямо посреди замера, и интервал получится отрицательным или абсурдно большим. Для длительностей нужны монотонные часы, которые только растут, — uptimeMillis() или elapsedRealtime().

Очередь сообщений строится именно на uptimeMillis() по той же причине плюс ещё одной: таймеры не должны «оживать» во сне. Если бы Message.when считался от elapsedRealtime, после долгого сна все накопившиеся отложенные сообщения выстрелили бы залпом при пробуждении. uptimeMillis, замирая в deep sleep, держит относительный порядок и задержки между сообщениями корректными.

Частые вопросы интервьюера

Как связаны Looper, Handler и MessageQueue? Middle

Looper владеет одной MessageQueue и в цикле (loop()) достаёт из неё сообщения по времени доставки. Handler привязан к конкретному Looper: он кладёт Message/Runnable в его очередь (post/sendMessage) и он же их обрабатывает (dispatchMessage). На поток приходится не больше одного Looper (хранится в ThreadLocal), но Handler-ов на один Looper может быть много.

Как из фонового потока обновить UI? Middle

Переключиться на главный поток через его очередь: Handler(Looper.getMainLooper()).post { … } (или view.post { … }, activity.runOnUiThread { … }, а в корутинах — withContext(Dispatchers.Main), который под капотом делает тот же post). Напрямую трогать View из фона нельзя — будет CalledFromWrongThreadException.

Очередь сообщений — это FIFO? Middle+

Нет. MessageQueue упорядочена по времени доставки Message.when (база — uptimeMillis), а не по порядку отправки. Сообщение без задержки получает текущее время и встаёт после уже «просроченных»; отложенное — в позицию по своему времени. Поэтому postDelayed может выполниться позже отправленного позже, но «немедленного» сообщения.

В каком порядке Handler обрабатывает сообщение? Senior

dispatchMessage идёт по трём веткам: (1) если у сообщения есть свой callback (это Runnable из post()) — выполняется он; (2) иначе, если у Handler задан Callback из конструктора и его handleMessage вернул true — обработка завершается; (3) иначе вызывается переопределённый handleMessage(). То есть post-Runnable имеет приоритет, затем конструкторный Callback, затем метод класса.

Почему new Handler() без аргументов — плохо? Middle+

Он неявно берёт Looper текущего потока, и это источник багов: задачи молча теряются (если тот Looper уже завершился), приложение падает (если потока без Looper) или возникают гонки из-за привязки не к тому потоку. Поэтому конструктор устарел с Android 11 (API 30). Правильно — указывать Looper явно: Handler(Looper.getMainLooper()) или использовать Executor.

Что блокирует главный поток, когда сообщений нет? Senior

Ничего не «крутится» вхолостую: MessageQueue.next() уходит в нативный сон через epoll на дескрипторе-eventfd до тех пор, пока не появится новое сообщение или не наступит время ближайшего отложенного. Запись в этот дескриптор (например, при post из другого потока) будит петлю. Поэтому простаивающий главный поток не нагружает CPU и батарею, оставаясь готовым мгновенно обработать ввод.

Какие бывают типы ANR кроме «ввод не обработан за 5 секунд»? Middle+

Их несколько, и пороги разные. KeyDispatchTimeout — ввод не обработан за 5 с (самый известный). BroadcastTimeout — onReceive не завершился: ~10 с для foreground, ~60 с для background. ServiceTimeout — колбэки Service: ~20 с для foreground, ~200 с для background. ContentProviderTimeout — ~10 с. Все эти компоненты крутятся на главном потоке процесса, поэтому корень один: долгий колбэк блокирует очередь главного Looper. На медленном железе пороги масштабируются.

Как сделать низкоприоритетную работу «когда поток освободится», и чем это лучше postDelayed? Senior

Через Looper.myQueue().addIdleHandler { … }. IdleHandler вызывается, когда очередь опустела или в ней остались только отложенные на будущее сообщения — то есть в момент реального простоя, а не по таймеру. Возврат true оставляет обработчик активным (сработает при следующем простое), false — снимает его после одного раза. Это точнее, чем postDelayed с угаданной задержкой: idle-колбэк не отнимает время у срочной работы и не рискует выстрелить посреди отрисовки кадра. Платформа сама использует это, например, для отложенной подсказки GC и ленивой инициализации.

Задача 1. Найдите утечку памяти в коде ниже. Объясните, почему она возникает, и предложите исправление.

Показать решение 
// Исходный код с утечкой:
class MainActivity : AppCompatActivity() {
    private val handler = Handler(Looper.getMainLooper())

    override fun onCreate(savedInstanceState: Bundle?) {
        super.onCreate(savedInstanceState)
        handler.postDelayed({
            textView.text = "загрузка завершена"
        }, 30_000)
    }
}

// Исправление — отменяем задачу при уничтожении Activity:
class MainActivity : AppCompatActivity() {
    private val handler = Handler(Looper.getMainLooper())
    private val updateTask = Runnable {
        textView.text = "загрузка завершена"
    }

    override fun onCreate(savedInstanceState: Bundle?) {
        super.onCreate(savedInstanceState)
        handler.postDelayed(updateTask, 30_000)
    }

    override fun onDestroy() {
        super.onDestroy()
        handler.removeCallbacks(updateTask)
    }
}

Объяснение: лямбда, переданная в postDelayed, захватывает неявную ссылку на this (Activity), потому что обращается к textView. Если Activity закроют до истечения 30 секунд, система не сможет её собрать — Runnable удерживает ссылку через MessageQueue. Решение — хранить задачу как отдельный объект и убирать её через removeCallbacks в onDestroy.

Задача 2. Реализуйте класс SerialWorker, который принимает задачи (Runnable) и выполняет их строго последовательно в фоновом потоке. Создание нового потока на каждую задачу запрещено. Обеспечьте корректную остановку.

Показать решение 
class SerialWorker {
    private val thread = HandlerThread("serial-worker").apply { start() }
    private val handler = Handler(thread.looper)

    fun submit(task: Runnable) {
        handler.post(task)
    }

    fun submitDelayed(task: Runnable, delayMs: Long) {
        handler.postDelayed(task, delayMs)
    }

    fun cancel(task: Runnable) {
        handler.removeCallbacks(task)
    }

    /** Завершает после обработки уже поставленных в очередь задач. */
    fun shutdown() {
        thread.quitSafely()
    }
}

Объяснение: HandlerThread создаёт поток с уже готовым Looper — заводить его вручную через Looper.prepare() не нужно. Handler, привязанный к его looper, гарантирует последовательное выполнение: новая задача запустится, только когда предыдущая завершится. quitSafely() дождётся задач с наступившим временем доставки, quit() — обрубил бы очередь немедленно.

Задача 3. Предскажите порядок вывода для кода ниже. Объясните, почему именно такой.

Показать решение 
val handler = Handler(Looper.getMainLooper())

handler.postDelayed({ Log.d("Q", "A — delayed 100ms") }, 100)
handler.post { Log.d("Q", "B — immediate") }
handler.postDelayed({ Log.d("Q", "C — delayed 50ms") }, 50)
handler.post { Log.d("Q", "D — immediate") }

// Порядок вывода (при условии, что все четыре вызова выполнены
// подряд, до того как Looper успел обработать очередь):
// B, D, C, A

Объяснение: MessageQueue упорядочена по Message.when (время доставки на базе uptimeMillis), а не по порядку отправки. «Немедленные» сообщения B и D получают время ~T+0 и встают в очередь перед C (T+50) и A (T+100). Среди немедленных порядок вставки сохраняется: B зашёл первым — выйдет первым. Итог: B → D → C → A.

Задача 4. Реализуйте debounce на главном потоке: класс Debouncer, у которого submit(action: () -> Unit) откладывает выполнение action на 300 мс, а если submit вызвали снова до истечения окна — предыдущий вызов отменяется и выполнится только последний. Используйте Handler главного потока. Добавьте cancel().

Показать решение 
class Debouncer(private val delayMs: Long = 300) {
    private val handler = Handler(Looper.getMainLooper())
    private var pending: Runnable? = null

    fun submit(action: () -> Unit) {
        pending?.let { handler.removeCallbacks(it) }   // отменяем предыдущий
        val r = Runnable { action() }
        pending = r
        handler.postDelayed(r, delayMs)
    }

    fun cancel() {
        pending?.let { handler.removeCallbacks(it) }
        pending = null
    }
}

Объяснение: каждая лямбда — новый объект Runnable, поэтому отменять надо именно по сохранённой ссылке pending, а не «по лямбде». На каждый submit снимаем прошлый отложенный Runnable и ставим новый — в очереди живёт максимум один. Альтернатива — postDelayed(r, token, time) с одним токеном и removeCallbacksAndMessages(token). Снятие — O(n) по очереди, для debounce это приемлемо.

Задача 5. Внутри Activity создан анонимный Handler, который через postDelayed(r, 60_000) обновляет TextView. Перепишите код так, чтобы при повороте экрана или уничтожении Activity не было утечки памяти. Объясните, какая именно ссылка удерживает Activity.

Показать решение 
// Утечка: анонимный Handler держит неявный this (Activity);
// отложенное сообщение в очереди → Message.target = Handler → Activity.

// Вариант 1 — static-вложенный класс + WeakReference + очистка очереди:
class MyActivity : AppCompatActivity() {
    private class UiHandler(a: MyActivity) : Handler(Looper.getMainLooper()) {
        private val ref = WeakReference(a)
        override fun handleMessage(msg: Message) {
            ref.get()?.textView?.text = "обновлено"
        }
    }
    private val handler = UiHandler(this)

    override fun onCreate(s: Bundle?) {
        super.onCreate(s)
        handler.sendEmptyMessageDelayed(0, 60_000)
    }
    override fun onDestroy() {
        super.onDestroy()
        handler.removeCallbacksAndMessages(null)
    }
}

// Вариант 2 (идиоматичный) — lifecycle-aware скоуп, отменяется сам:
class MyActivity : AppCompatActivity() {
    override fun onCreate(s: Bundle?) {
        super.onCreate(s)
        lifecycleScope.launch {
            delay(60_000)
            textView.text = "обновлено"
        }
    }
}

Объяснение: цепочка удержания — Message в MessageQueue → Message.target = Handler → анонимный Handler держит неявный this Activity. Отложенное сообщение живёт в очереди минуту независимо от Activity, поэтому одной слабой ссылки мало без статического класса. lifecycleScope снимает проблему целиком: корутина отменяется при уничтожении lifecycle, и колбэк на мёртвом экране не выполнится.
