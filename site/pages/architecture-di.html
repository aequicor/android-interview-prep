Архитектура и DI

Как структурировать Android-приложение, чтобы оно жило годами: паттерны представления MVVM/MVI/MVP (Model-View-ViewModel, Model-View-Intent, Model-View-Presenter — модели архитектуры экрана), Clean Architecture (чистая архитектура), гайд Google по архитектуре, multi-module (многомодульность) и DI (dependency injection, внедрение зависимостей) на Hilt. Тема почти гарантированно всплывёт на Middle+/Senior (уверенный/старший уровень) — ждут не определений, а trade-offs (компромиссов) и опыта из боевых проектов.

Middle Senior Частый вопрос

Паттерны представления: MVVM, MVI, MVP

Все три отвечают на вопрос «как связать UI с логикой и состоянием». Разница — в направлении потока данных и в том, как хранится и обновляется состояние экрана.

MVP (legacy)

Presenter держит ссылку на пассивный View (через интерфейс) и императивно командует им: view.showLoading(), view.showError(). Двусторонняя связь, ручное управление жизненным циклом, View-интерфейсы с десятками методов. Сегодня встречается в легаси на Android Views; на новых проектах вытеснен MVVM/MVI, потому что плохо дружит с конфиг-изменениями и Compose.

MVVM

ViewModel выставляет наблюдаемое состояние (StateFlow/LiveData), View подписывается и рендерит. ViewModel не знает о View — связь односторонняя на чтение состояния, плюс вызовы методов VM на события. Состояние может быть как одним объектом, так и набором отдельных полей. Это рекомендация Google по умолчанию.

ProfileViewModel.kt (MVVM)
class ProfileViewModel(
    private val getProfile: GetProfileUseCase
) : ViewModel() {

    private val _state = MutableStateFlow(ProfileUiState())
    val state: StateFlow<ProfileUiState> = _state.asStateFlow()

    fun onRefresh() = viewModelScope.launch {
        _state.update { it.copy(isLoading = true) }
        val result = getProfile()
        _state.update {
            it.copy(isLoading = false, profile = result.getOrNull(), error = result.exceptionOrNull())
        }
    }
}

MVI

Эволюция MVVM в сторону строгого UDF. Ключевые элементы:

  • Единый immutable State — весь экран описан одним неизменяемым объектом (single source of truth для UI).
  • Intent (часто называют Action/Event) — единственный способ запросить изменение. View эмитит интенты, не трогает состояние напрямую.
  • Reducer — чистая функция (State, Action) -> State: по текущему состоянию и интенту возвращает новое. Легко тестировать, можно «перематывать» историю (time-travel debugging).
  • Side-effects — одноразовые эффекты (навигация, тост, snackbar), которые не должны жить в state. Отдаются через отдельный канал (Channel/SharedFlow), иначе при повороте экрана тост повторится.
CounterViewModel.kt (MVI)
data class CounterState(val count: Int = 0, val isLoading: Boolean = false)

sealed interface CounterAction {
    data object Increment : CounterAction
    data object Decrement : CounterAction
    data object Save : CounterAction
}

sealed interface CounterEffect {            // side-effects
    data class ShowToast(val text: String) : CounterEffect
    data object NavigateBack : CounterEffect
}

class CounterViewModel(private val repo: CounterRepository) : ViewModel() {

    private val _state = MutableStateFlow(CounterState())
    val state: StateFlow<CounterState> = _state.asStateFlow()

    private val _effects = Channel<CounterEffect>(Channel.BUFFERED)
    val effects = _effects.receiveAsFlow()    // одноразовые, не часть state

    fun onAction(action: CounterAction) {
        _state.update { reduce(it, action) }   // чистый редьюсер
        if (action is CounterAction.Save) save()
    }

    private fun reduce(s: CounterState, a: CounterAction): CounterState = when (a) {
        CounterAction.Increment -> s.copy(count = s.count + 1)
        CounterAction.Decrement -> s.copy(count = s.count - 1)
        CounterAction.Save      -> s.copy(isLoading = true)
    }

    private fun save() = viewModelScope.launch {
        repo.save(state.value.count)
        _state.update { it.copy(isLoading = false) }
        _effects.send(CounterEffect.ShowToast("Сохранено"))
    }
}

MVI: однонаправленный цикл. Состояние течёт вниз, интенты — вверх, side-effects идут отдельным каналом

Честный тезис для собеса. MVI — это MVVM с дисциплиной: то же дерево (ViewModel + observable state), но с жёсткими правилами (immutable state, интенты, редьюсер, отдельные эффекты). Почему одноразовые события (навигация, snackbar) нельзя класть в StateFlow, и чем Channel надёжнее SharedFlow(replay=0) для их доставки? ★

StateFlow конфлейтит и всегда переигрывает последнее значение новому подписчику. После поворота экрана UI повторно «выполнит» событие — снова откроет экран, снова покажет тост. Для эффекта это катастрофа: эффект должен сработать ровно один раз.

SharedFlow(replay=0) не реигрывает, но это горячий broadcast-поток. Если на момент emit подписчика нет (UI ушёл в фон), событие теряется; а при нескольких коллекторах оно доставится каждому — нежелательный fan-out. Channel через receiveAsFlow() семантически точнее: он буферизует, пока подписчика нет, и доставляет каждый элемент ровно один раз ровно одному получателю.

Поэтому каноничный носитель side-effects — Channel(Channel.BUFFERED).receiveAsFlow(), а не StateFlow и не SharedFlow. Тонкость: даже с Channel подписка через repeatOnLifecycle(STARTED) может пропустить эффект, эмитнутый пока экран в STOPPED, — отсюда и нужен буфер, чтобы элемент дождался нового коллектора.

Вывод. StateFlow — для состояния, которое должно переживать поворот; Channel.receiveAsFlow() — для эффекта, который должен сработать один раз и не повториться. SharedFlow(replay=0) уместен, когда событие действительно широковещательное и потеря допустима.
Когда SharedFlow всё-таки лучше Channel для событий? ★

Когда событие по смыслу broadcast: его должны получить все активные подписчики, а не один. Пример — глобальный сигнал «токен протух, разлогиньтесь», на который реагируют сразу несколько экранов. Channel отдал бы его только первому, кто успел подписаться, остальные бы его не увидели. Здесь нужен именно fan-out SharedFlow. Для навигации и snackbar внутри одного экрана fan-out, наоборот, вреден — нужен Channel.

Clean Architecture: слои и правило зависимостей

Clean Architecture (Роберт Мартин) делит код на концентрические слои. Главное — правило зависимостей: исходный код может зависеть только в направлении внутрь, к более стабильной политике. Внутренние слои ничего не знают о внешних.

Стрелки зависимостей направлены внутрь. domain не знает ни про Android, ни про Retrofit/Room. data реализует интерфейсы, объявленные в domain (DIP)

Роли слоёв

  • presentation — ViewModel + UI. Зависит от domain (вызывает UseCase'ы), оперирует UI-моделями.
  • domain — чистый Kotlin-модуль без зависимостей от Android. Содержит UseCase, доменные модели и интерфейсы репозиториев. Это сердце бизнес-логики.
  • data — реализации репозиториев, источники (network/database), DTO/Entity и мапперы. Зависит от domain (реализует его интерфейсы), но domain про data ничего не знает.

UseCase, Repository и маппинг моделей

Каждый слой имеет свою модель: UserDto (data, формат API), UserEntity (data, Room), User (domain), UserUi (presentation, готовая к показу). Мапперы изолируют слои: смена формата API не доходит до UI.

domain-слой (чистый Kotlin)
// domain/model — без аннотаций фреймворков
data class User(val id: String, val name: String, val isPremium: Boolean)

// domain/repository — интерфейс, реализация живёт в data
interface UserRepository {
    suspend fun getUser(id: String): Result<User>
}

// domain/usecase — одна бизнес-операция, удобно мокать и тестировать
class GetUserUseCase(private val repo: UserRepository) {
    suspend operator fun invoke(id: String): Result<User> = repo.getUser(id)
}
data-слой: реализация + маппинг
// data/dto — формат сети
data class UserDto(val id: String, val full_name: String, val plan: String)

// data/mapper — изолирует формат API от домена
fun UserDto.toDomain() = User(
    id = id,
    name = full_name,
    isPremium = plan == "premium"
)

// data/repository — реализует контракт domain, внедряется через DI
class UserRepositoryImpl(
    private val api: UserApi,
    private val dao: UserDao
) : UserRepository {
    override suspend fun getUser(id: String): Result<User> = runCatching {
        api.fetchUser(id).toDomain()
    }
}
⚠️
Подводный камень. Не тащите UserDto или Room-Entity в ViewModel — это «протечка» слоя data в presentation. Тогда смена бэкенда заставит править UI. Мапперы — не бюрократия, а граница, которая держит изменения локально. На простом CRUD-экране слой UseCase иногда схлопывают (VM зовёт репозиторий напрямую) — это допустимый прагматичный компромисс, но будьте готовы его обосновать.
Где физически объявлен интерфейс репозитория — в domain или в data — и что сломается, если перенести его в data? ★

Интерфейс репозитория живёт в domain, а его реализация (RepositoryImpl) — в data. Это инверсия зависимостей (DIP) и одновременно «dependency inversion» из правила зависимостей Clean Architecture: data зависит от domain и реализует его контракт, а domain про data ничего не знает. Стрелка зависимости от конкретики направлена к абстракции.

Перенесите интерфейс в data — и domain.UseCase будет вынужден импортировать data, стрелка развернётся наружу. domain потеряет независимость от Android/Retrofit/Room: его уже не собрать отдельным чистым Kotlin-модулем и не протестировать без моков инфраструктуры. В multi-module это видно буквально — модуль :domain не должен иметь :data в своём build.gradle.

Гайд Google по архитектуре приложений

Официальная рекомендация Google — слоистая архитектура из UI layer, опционального domain layer и data layer, построенная на двух принципах: single source of truth (SSOT) и unidirectional data flow (UDF). Это «облегчённая» и прагматичная версия идей Clean Architecture.

UI layer и state holder

UI-слой рисует состояние и шлёт события наверх. State holder (обычно ViewModel) — класс, который производит UI-state и держит логику его формирования. UDF здесь буквально: state течёт вниз в UI, события (нажатия) текут вверх в state holder, где данные меняются и выставляются как immutable-тип.

Domain layer (опциональный)

Слой UseCase'ов. Нужен, когда логика сложна или переиспользуется несколькими ViewModel. UseCase инкапсулирует одну операцию и комбинирует данные из нескольких репозиториев. Если логики мало — слой можно опустить, VM ходит в data напрямую.

Data layer, SSOT и offline-first

Data-слой состоит из репозиториев и источников данных. Репозиторий — публичный API слоя, агрегирует источники, разрешает конфликты и выставляет единый источник истины. Offline-first: локальная БД (Room) становится SSOT, сеть лишь обновляет её, а UI всегда читает из БД. Так приложение работает без сети и не мигает при рефетче.

Offline-first репозиторий (Flow из БД + sync из сети)
class ArticleRepository(
    private val api: ArticleApi,
    private val dao: ArticleDao
) {
    // SSOT — UI всегда читает из БД, это горячий источник истины
    fun observeArticles(): Flow<List<Article>> =
        dao.observeAll().map { entities -> entities.map { it.toDomain() } }

    // сеть лишь обновляет БД; UI обновится сам через Flow
    suspend fun refresh() {
        val remote = api.fetchArticles()
        dao.upsertAll(remote.map { it.toEntity() })
    }
}
Чем по гайду Google ViewModel отличается от обычного state holder, и какие виды state и логики он разделяет? ★

Google различает два типа state holder. ViewModel-класс — держатель business logic state: доступ к данным, правила приложения, переживает конфиг-изменения. Обычный plain-класс, запоминаемый в композиции (через remember), — держатель UI logic state: как показывать state (форматирование, видимость), состояние элемента вроде объекта скролла.

State тоже двух родов: screen UI state — что показать, продукт бизнес-логики, неизменяемый, его выставляет ViewModel; и UI element state — состояние самих элементов (раскрытие меню, позиция скролла), которое живёт в UI-слое. ViewModel выставляет screen UI state и не должен держать ссылок на View/Context уровня Activity.

Главная мысль для собеса: ViewModel — не «контейнер для всего», а именно держатель бизнес-логики и screen state. Часть состояния правильно оставить в UI-слое, а не тащить в ViewModel.

Многомодульность

Монолитный :app на большом проекте упирается в скорость сборки и спутанные зависимости. Решение — разбить на Gradle-модули. Типичная классификация (как в эталонном Google-проекте Now in Android):

  • :app — точка входа, склейка фич, навигационный граф верхнего уровня.
  • :feature:* — самодостаточные фичи (home, profile, settings). Зависят от :core, но не друг от друга.
  • :core:* — переиспользуемая инфраструктура: :core:ui, :core:designsystem, :core:common, :core:network, :core:database.
  • :core:data / :core:domain — общие репозитории и UseCase'ы.

Граф модулей. Фичи не зависят друг от друга; общее вынесено в :core. Это держит граф ацикличным и позволяет параллельную сборку

Навигация между модулями

Фичи не знают друг о друге, поэтому прямой вызов экрана из соседнего модуля невозможен. Решения: навигационный граф собирается в :app (он видит все фичи), а каждая фича экспонирует свой NavGraphBuilder-extension. Маршруты к чужим фичам передаются как абстракция (например, через интерфейс-навигатор в :core или type-safe route, объявленный в общем модуле).

Gradle convention plugins

Без них каждый модуль дублирует одинаковый build.gradle.kts (compileSdk, Compose, Kotlin-опции). Convention plugin — это переиспользуемая build-логика в композитной сборке build-logic: настройка описывается один раз и применяется строкой. Именно так сделан Now in Android.

build.gradle.kts фичи — одна строка вместо десятков
plugins {
    id("nowinandroid.android.feature")      // convention plugin
    id("nowinandroid.android.library.compose")
}

dependencies {
    implementation(projects.core.data)
    implementation(projects.core.designsystem)
}
// весь общий setup (compileSdk, Hilt, Compose, тест-раннеры)
// живёт внутри convention-плагина, а не копируется в каждый модуль
В чём разница между api и implementation в Gradle и как неверный выбор бьёт по инкрементальной сборке? ★

implementation делает зависимость видимой только внутри объявившего модуля; api экспортирует её транзитивно всем, кто зависит от модуля. Разница в сборке решающая: при изменении ABI зависимости, объявленной через api, Gradle перекомпилирует все модули вниз по цепочке — потребители видят её в своём compile-classpath. Через implementation перекомпилируется только сам модуль; потребители не трогаются, пока не меняется публичный API самого модуля.

Поэтому дефолт — implementation, а api берут только когда тип действительно фигурирует в публичной сигнатуре модуля (возвращается или принимается в его API). :feature-модули прячут детали за implementation, чтобы не протекали транзитивные зависимости и не раздувался граф пересборки. api оправдан в :core:designsystem или :core:model, чьи типы намеренно публичны для всех фич. Подробнее про механику сборки — в главе Gradle и сборка.

DI: Hilt — компоненты, скоупы, граф

Dependency Injection — передача зависимостей объекту извне вместо их создания внутри. Это практическая реализация DIP: верхние слои зависят от абстракций, а конкретные реализации поставляет контейнер. Hilt — надстройка над Dagger: он генерирует за вас стандартный граф компонентов, привязанный к жизненным циклам Android (Application, Activity, ViewModel, Fragment), и кодогенерацию точек входа. Под капотом всё тот же compile-time DI Dagger с проверкой графа на этапе компиляции.

@Module, @Provides и @Binds

@Provides — когда вы сами конструируете объект (например, чужой класс из библиотеки). @Binds — когда нужно связать интерфейс с реализацией; компилируется эффективнее (без тела метода), метод абстрактный, модуль обычно abstract.

NetworkModule.kt / RepositoryModule.kt
@Module
@InstallIn(SingletonComponent::class)
object NetworkModule {
    @Provides @Singleton
    fun provideRetrofit(): Retrofit = Retrofit.Builder()
        .baseUrl("https://api.example.com/")
        .addConverterFactory(Json.asConverterFactory("application/json".toMediaType()))
        .build()

    @Provides @Singleton
    fun provideUserApi(retrofit: Retrofit): UserApi = retrofit.create(UserApi::class.java)
}

@Module
@InstallIn(SingletonComponent::class)
abstract class RepositoryModule {
    // @Binds: связать интерфейс с реализацией, без ручного конструирования
    @Binds @Singleton
    abstract fun bindUserRepository(impl: UserRepositoryImpl): UserRepository
}

Manual DI / Service Locator — когда уместно

Hilt не всегда обязателен. Manual DI (передаём зависимости руками через конструкторы/фабрики) подходит для маленьких приложений, библиотек (не навязывать пользователю Dagger) и учебных проектов. Service Locator (глобальный реестр, у которого запрашивают зависимости) проще, но скрывает зависимости класса (их не видно в сигнатуре) и хуже тестируется — обычно считается анти-паттерном по сравнению с конструкторной инъекцией. Правило: предпочитайте конструкторную инъекцию; Hilt берите, когда граф вырастает и ручная проводка становится болью.

На повороте рвутся только Activity/Fragment-скоупы (серая дорожка). Singleton и ActivityRetained/ViewModel переживают поворот и гаснут лишь на настоящем закрытии
Что произойдёт на этапе компиляции при циклической зависимости в графе Hilt, и как разорвать цикл через Provider/Lazy? ★

Dagger/Hilt строит граф на этапе компиляции и при прямом цикле A→B→A выдаёт ошибку компиляции «Found a dependency cycle» — рантайм-сбоя не будет, сборка просто упадёт.

Разорвать цикл, не трогая дизайн, можно ленивой инъекцией: вместо Foo инжектить dagger.Lazy<Foo> или javax.inject.Provider<Foo> на одной из сторон. Тогда Dagger откладывает создание узла, и в момент конструирования цикл не замыкается: Provider создаёт по требованию, Lazy — лениво один раз. Работает это потому, что Provider<T>/Lazy<T> доступны в графе для любого биндинга без отдельной декларации.

Правильнее, конечно, устранить цикл архитектурно — он обычно сигнал о нарушении направления зависимостей. Но на собесе ждут именно знания про Lazy/Provider как штатный приём.

Навигация: Jetpack Navigation (Compose)

С Navigation 2.8.0 type-safe API стабильны: маршруты описываются не строками, а @Serializable-типами (через Kotlin Serialization). Это убирает хрупкие строковые роуты и даёт компайл-тайм проверку аргументов.

Type-safe routes + аргументы + deep link
@Serializable data object HomeRoute
@Serializable data class ProfileRoute(val userId: String)   // типобезопасный аргумент

@Composable
fun AppNavHost(navController: NavHostController) {
    NavHost(navController, startDestination = HomeRoute) {
        composable<HomeRoute> {
            HomeScreen(onUser = { id -> navController.navigate(ProfileRoute(id)) })
        }
        composable<ProfileRoute>(
            // deep link на ту же type-safe цель
            deepLinks = listOf(navDeepLink<ProfileRoute>(basePath = "https://app.example.com/user"))
        ) { backStackEntry ->
            val route: ProfileRoute = backStackEntry.toRoute()   // достаём типизированно
            ProfileScreen(userId = route.userId)
        }
    }
}
  • Типобезопасные маршруты: навигация принимает объект ProfileRoute("42"), библиотека сама сериализует/десериализует аргументы с сохранением типов.
  • Передача аргументов: складываются в data-class маршрута, читаются через backStackEntry.toRoute<T>() — без ручного парсинга строк и багов с ключами.
  • Deep links: navDeepLink<T> связывает внешний URL с той же type-safe целью; внешняя ссылка открывает нужный экран с аргументами.

Back stack и single-activity

Современная рекомендация Google — single-activity: одна Activity-хост, а экраны — это destinations внутри одного NavHost (composable или фрагменты). Тогда у приложения единый back stack, общий ViewModelStoreOwner на уровне Activity для расшаривания состояния, единообразные deep links и меньше накладных расходов на пересоздание Activity. NavController ведёт стек: navigate() кладёт запись, системный «назад» снимает её. Управление стеком — через popUpTo (срезать до нужной точки), inclusive (включить саму цель), launchSingleTop (не плодить дубликат вершины).

Multi-activity всё ещё оправдан для по-настоящему независимых частей: отдельный флоу логина, экран-«запуск по интенту» извне, изоляция тяжёлого legacy. Подвох на собесе: при single-activity конфиг-изменения переживает не Activity, а ViewModel и back stack — поэтому скоупы (например ActivityRetained) и записи back stack становятся центральными.

Scoping ViewModel на back stack entry

Каждая запись стека — NavBackStackEntry — сама выступает ViewModelStoreOwner. По умолчанию hiltViewModel() в composable<Route> скоупит ViewModel на запись этого destination: создаётся при входе, очищается (onCleared) при попе из стека. Чтобы ViewModel пережил несколько экранов одного флоу (мастер-визард), его скоупят на родительский граф:

Расшаренный ViewModel на nested graph
@Composable
fun WizardStep(navController: NavHostController) {
    // backStackEntry графа-родителя, а не текущего экрана
    val parentEntry = remember(navController) {
        navController.getBackStackEntry<WizardGraph>()
    }
    // один инстанс на весь флоу: жив, пока в стеке есть запись графа
    val vm: WizardViewModel = hiltViewModel(viewModelStoreOwner = parentEntry)
}

Без этого каждый шаг получит свой инстанс, и общее состояние флоу не сохранится между экранами. SavedStateHandle при этом автоматически наполняется аргументами навигации соответствующей записи.

App Links, custom scheme и deferred deep links

Deep link бывает двух родов. Custom scheme (myapp://profile/42) ловит только ваше приложение, но не открывается из браузера и любой может его подделать. HTTPS App Links (https://app.example.com/user/42) — это обычный веб-URL, который Android может открыть прямо в приложении, минуя диалог выбора, если домен подтверждён.

Подтверждение делается через android:autoVerify="true" на <intent-filter> и файл assetlinks.json, выложенный по адресу https://домен/.well-known/assetlinks.json с отпечатком (SHA-256) подписи приложения. Система проверяет соответствие при установке — и тогда ссылки на этот домен открываются в приложении без диалога.

AndroidManifest.xml — verified App Link
<intent-filter android:autoVerify="true">
    <action android:name="android.intent.action.VIEW" />
    <category android:name="android.intent.category.DEFAULT" />
    <category android:name="android.intent.category.BROWSABLE" />
    <data android:scheme="https" android:host="app.example.com" />
</intent-filter>
ℹ️
Deferred deep links. Если приложение ещё не установлено, обычный deep link открывает браузер. «Отложенная» ссылка сохраняет цель (через стор или сторонний сервис), и после установки приложение читает её при первом запуске и ведёт пользователя на нужный экран. В самой платформе из коробки этого нет — нужен внешний механизм (Play Install Referrer или SDK вроде ветки диплинков).
Как заскоупить один ViewModel на nested navigation graph, и почему hiltViewModel() без скоупа на graph entry этого не даёт? ★

По умолчанию hiltViewModel() в composable<Route> скоупит ViewModel на NavBackStackEntry этого destination — он создаётся при входе на экран и очищается при попе из back stack. Каждый экран получает свой инстанс.

Чтобы один ViewModel пережил несколько экранов флоу, его скоупят на родительский граф: берут navController.getBackStackEntry(parentRoute) и передают как viewModelStoreOwner в hiltViewModel(). Тогда инстанс живёт, пока в стеке есть запись графа-родителя, и общий для всех экранов флоу; при выходе из графа очищается целиком. Без этого расшаренное состояние визарда не сохранится между шагами, потому что у каждого экрана будет отдельный ViewModelStore.

Reactive-слой: Flow/StateFlow как контракт

Связь между слоями реактивна. Контракт data → domain → UI описывается через Flow: репозиторий отдаёт «холодный» Flow данных (например, из Room), UseCase его трансформирует, ViewModel превращает в «горячий» StateFlow для UI. Так обновления (новый ответ сети записан в БД) автоматически доезжают до экрана.

Flow по слоям: cold во data, hot в presentation
// data: cold Flow из источника истины (Room)
fun ArticleRepository.observe(): Flow<List<Article>> = dao.observeAll().map { ... }

// domain: трансформация бизнес-правилом
class ObserveUnread(private val repo: ArticleRepository) {
    operator fun invoke(): Flow<List<Article>> =
        repo.observe().map { list -> list.filter { !it.isRead } }
}

// presentation: cold -> hot StateFlow, готовый к подписке UI
class FeedViewModel(observeUnread: ObserveUnread) : ViewModel() {
    val state: StateFlow<List<Article>> = observeUnread()
        .stateIn(viewModelScope, SharingStarted.WhileSubscribed(5_000), emptyList())
}
💡
Зачем StateFlow на границе с UI. Он всегда имеет текущее значение (UI получает state сразу при подписке) и конфлейтит обновления. SharingStarted.WhileSubscribed(5_000) держит upstream живым 5 секунд после ухода подписчика — переживает поворот экрана, но отписывается, когда экран действительно закрыт. Это стандартная связка для экспонирования состояния из ViewModel.

stateIn + WhileSubscribed(5000): при повороте новый подписчик успевает в окно 5 с — upstream не рестартует; при реальном уходе с экрана пауза истекает и upstream гаснет
Чем грозит SharingStarted.Lazily/Eagerly вместо WhileSubscribed(5000), и почему именно 5000 мс? ★

Eagerly и Lazily никогда не останавливают upstream. После первого запуска подписка на источник (Room, сеть) держится до конца жизни viewModelScope, даже когда экран ушёл в фон — лишняя работа и расход батареи. Eagerly вдобавок стартует ещё до появления подписчика.

WhileSubscribed(stopTimeoutMillis) останавливает upstream через заданную паузу после ухода последнего подписчика и перезапускает при новом. 5000 мс — эмпирический буфер: при повороте старый коллектор отписывается, а новый подписывается не мгновенно — пересоздание Activity/Composition занимает доли секунды. Таймаут в 5 с перекрывает этот разрыв: upstream не успевает остановиться и перезапуститься (нет лишнего рефетча из сети), но при настоящем уходе с экрана через 5 с подписка корректно гасится.

Дополнительно с WhileSubscribed учитывают replayExpirationMillis (по умолчанию Long.MAX_VALUE), чтобы во время паузы последнее значение не сбрасывалось на initial.

Тестирование архитектуры и DI

Главный заявленный смысл слоёв и конструкторной инъекции — тестируемость. На собесе это спрашивают почти всегда: «как тестировать каждый слой», «зачем инжектить Dispatchers», «как подменить модуль Hilt». Чем чище границы, тем меньше хаков в тестах.

Слой за слоем

domain тестируется проще всего — это чистый Kotlin без Android: UseCase покрывают юнит-тестами с подменённым (fake/mock) интерфейсом репозитория, без Robolectric и инструментальных тестов. ViewModel тестируют, подменяя UseCase и проверяя последовательность UiState. Repository — с fake-источниками: in-memory DAO и MockWebServer для сети.

domain-тест: UseCase с fake-репозиторием
class GetUserUseCaseTest {
    private val repo = FakeUserRepository()           // подменили интерфейс
    private val useCase = GetUserUseCase(repo)

    @Test
    fun `returns user from repository`() = runTest {
        repo.stub(User(id = "1", name = "Ann", isPremium = true))
        val result = useCase("1")
        assertEquals("Ann", result.getOrNull()?.name)
    }
}

Почему хардкод ломает тест

Два частых анти-паттерна делают класс непроверяемым. Первый — Dispatchers.IO прямо в теле корутины: в тесте его не подменить на TestDispatcher, и runTest не контролирует виртуальное время. Лечится инъекцией диспетчера через qualifier (см. @IoDispatcher выше). Второй — new/прямое создание зависимости внутри класса: её нечем заменить на fake. Конструкторная инъекция обе проблемы снимает.

ViewModel-тест: TestDispatcher + последовательность state
@Test
fun `emits loading then content`() = runTest {
    val vm = FeedViewModel(repo, testDispatcher)      // диспетчер инжектится
    vm.state.test {                                   // Turbine
        assertEquals(FeedUiState(isLoading = true), awaitItem())
        assertEquals(expectedContent, awaitItem())
    }
}

Hilt в инструментальных тестах

Когда тест всё же инструментальный, Hilt даёт штатную подмену графа: @HiltAndroidTest на классе, @UninstallModules снимает продакшен-модуль, а @TestInstallIn ставит вместо него тестовый с fake-биндингами. Никаких рефлексий и ручной проводки — граф собирается из тестовых модулей.

💡
Тезис. Интерфейсы на границах слоёв + конструкторная инъекция = подмена без хаков. Если для теста класса приходится поднимать половину приложения или мокать статику — это сигнал, что зависимости спрятаны (хардкод, service locator), а не приходят через конструктор. Про сами корутинные тесты подробнее — в главе Фоновая работа.
Почему юнит-тест UseCase не требует Robolectric, а тест ViewModel иногда требует? ★

UseCase в правильной Clean-архитектуре — чистый Kotlin без единого Android-импорта, поэтому его тест гоняется на JVM мгновенно, без эмулятора и без Robolectric. ViewModel формально тоже не зависит от Android-UI, но тянет androidx.lifecycle.ViewModel и часто SavedStateHandle; для них хватает обычного JVM-теста, если не трогать Looper/LiveData с главным потоком. Robolectric нужен только когда в тест проникают настоящие Android-классы (Context, ресурсы, LiveData.observeForever с main looper). Если ViewModel тащит за собой Robolectric — обычно в нём протекла Android-зависимость, которую стоило вынести.

SavedStateHandle и его лимит

SavedStateHandle персистится в saved instance state Bundle, поэтому переживает process death, и Hilt автоматически наполняет его аргументами навигации. Но Bundle ограничен по размеру: крупные данные дают TransactionTooLargeException. Поэтому в SavedStateHandle кладут ключи и идентификаторы, а тяжёлые данные восстанавливают из источника истины.

SavedStateHandle: ключ переживает process death
@HiltViewModel
class ProfileViewModel @Inject constructor(
    private val savedState: SavedStateHandle,        // Hilt даёт автоматически
    private val repo: UserRepository
) : ViewModel() {
    // userId переживёт убийство процесса — он в Bundle
    private val userId: String = savedState.toRoute<ProfileRoute>().userId

    // сам профиль не храним в полях — перечитаем из SSOT после рестарта
    val state: StateFlow<User?> = repo.observeUser(userId)
        .stateIn(viewModelScope, SharingStarted.WhileSubscribed(5_000), null)
}

Отсюда и устойчивость offline-first: когда SSOT — локальная БД, после рестарта процесса UI просто перечитывает её, и восстанавливать руками нечего. Самый надёжный способ пережить process death — не держать критичные данные только в MutableStateFlow, а опираться на Room/DataStore.

Чем rememberSaveable в Compose отличается от SavedStateHandle и где проходит граница? ★

Оба пишут в один и тот же saved instance state Bundle и оба переживают process death — разница в том, чьё состояние они хранят. SavedStateHandle живёт в ViewModel и держит state бизнес-уровня (id выбранной сущности, черновик формы), который переживает и поворот, и убийство процесса. rememberSaveable хранит UI element state самого компонента (позиция скролла, раскрытие, введённый, но не отправленный текст), привязанный к месту в дереве композиции. Граница ровно та же, что у Google между business logic state и UI element state: бизнес-данные — в SavedStateHandle, эфемерное состояние виджета — в rememberSaveable. Оба упираются в лимит Bundle.

Обработка ошибок сквозь слои

«Где обрабатывать ошибки и как доносить их до UI» — прямое продолжение разговора про слои и SSOT. Главный принцип: исключения не должны неуправляемо «пробивать» слои до UI.

Распространённый подход — репозиторий ловит сетевые/IO-ошибки и возвращает Result<T> (или собственный sealed Resource/Either с Loading/Success/Error), а не бросает наружу. Вызывающий сам решает, показать ошибку или проглотить, и кэш при этом не стирается. domain может мапить технические ошибки в доменные (NoConnection, Unauthorized), чтобы UI не разбирал HttpException.

sealed Resource как контракт результата
sealed interface Resource<out T> {
    data object Loading : Resource<Nothing>
    data class Success<T>(val data: T) : Resource<T>
    data class Error(val cause: DomainError) : Resource<Nothing>
}

sealed interface DomainError {            // технические ошибки -> доменные
    data object NoConnection : DomainError
    data object Unauthorized : DomainError
    data class Unknown(val message: String?) : DomainError
}

Куда ошибка попадает в UI

Тут важно различать два рода ошибок. Постоянная ошибка (баннер «нет сети», который висит, пока проблема не решена) — это часть screen UI state: поле error, которое переживает поворот. Одноразовая ошибка (snackbar «не удалось сохранить») — side-effect: её отдают через Channel, иначе после поворота она реигрывается из StateFlow и всплывёт повторно (тот же механизм, что для навигации в первом параграфе).

⚠️
Подвох. Ошибку-«показать тост» нельзя держать в постоянном state — после поворота тост повторится. Для постоянного баннера, наоборот, место именно в state, а не в одноразовом канале, иначе после пересоздания экрана баннер исчезнет. Граница та же, что между state и side-effect.

Связь с offline-first прямая: ошибка сети не должна стирать кэш. Запись в БД делается только на успешном ответе — тогда при сбое UI продолжает показывать старые данные, а ошибку получает отдельно (см. задачу 3 и новую задачу про refresh).

Когда Result/runCatching в репозитории — плохая идея, и что с CancellationException? ★

Главная ловушка: runCatching ловит Throwable, включая CancellationException. Если корутину отменили (экран ушёл), исключение отмены попадёт в Result.failure и будет проглочено как «обычная ошибка» — структурированная конкурентность сломается, отмена не прорастёт вверх. Поэтому в suspend-коде либо ловят конкретные типы (IOException, HttpException), либо перебрасывают отмену вручную: catch (e: CancellationException) { throw e } перед общим catch. Второй момент: Result хорош как контракт одной операции, но не как способ протащить состояние загрузки сквозь весь поток — для потоковых данных естественнее sealed Resource с явным Loading, а не оборачивать каждый элемент Flow в Result.

Частые вопросы интервьюера

Чем MVI отличается от MVVM и когда выбрать MVI? Middle+

MVVM выставляет наблюдаемое состояние, которое VM может обновлять разными методами; состояние может быть раздроблено на поля. MVI добавляет дисциплину: один immutable State, изменения только через Intent, состояние пересчитывается чистым reducer'ом (State, Action) -> State, а одноразовые эффекты (тост, навигация) идут отдельным каналом, а не в state. Выбирают MVI для сложных экранов с многими взаимозависимыми флагами (предсказуемость, тестируемый редьюсер, возможность реплея), для простых экранов это лишний boilerplate. По сути MVI — это строгий вариант UDF поверх той же связки ViewModel+observable state.

Объясните правило зависимостей в Clean Architecture. Middle+

Зависимости направлены только внутрь — к более стабильной политике. presentation зависит от domain, data зависит от domain, но domain не знает ни о ком. Ключ — DIP: domain объявляет интерфейс Repository, а data его реализует; стрелка зависимости от реализации идёт к абстракции в домене. Поэтому domain — чистый Kotlin без Android/Retrofit/Room, его легко тестировать и переиспользовать. Каждый слой имеет свои модели (DTO/Entity/Domain/Ui) и мапперы между ними, чтобы изменения формата не протекали наружу.

Hilt — это что поверх Dagger? Что он добавляет? Middle+

Hilt — официальная надстройка над Dagger для Android. Dagger даёт compile-time DI с проверкой графа, но требует вручную писать компоненты и связывать их с жизненными циклами. Hilt генерирует стандартный набор компонентов (Singleton, ActivityRetained, ViewModel, Activity, Fragment, View, Service), привязывает их к Android-жизненным циклам, добавляет аннотации @AndroidEntryPoint, @HiltViewModel, @InstallIn и упрощает точки входа. Под капотом — тот же Dagger и тот же граф; Hilt лишь убирает шаблонный код по его настройке.

Разница @ActivityScoped и @ActivityRetainedScoped? Senior

@ActivityScoped привязан к конкретному экземпляру Activity — при повороте экрана Activity пересоздаётся, и скоупнутый объект тоже. @ActivityRetainedScoped живёт в ActivityRetainedComponent, который переживает конфиг-изменения (от первого создания Activity до её настоящего закрытия), как и ViewModel. Поэтому для состояния, которое должно пережить поворот, берут ActivityRetained, а для привязки к экземпляру Activity — ActivityScoped. ViewModelComponent — потомок ActivityRetained, поэтому VM-скоупнутые зависимости тоже переживают поворот.

Зачем дробить приложение на модули? Что это даёт сборке? Senior

Главные выгоды: скорость инкрементальной сборки (Gradle собирает независимые модули параллельно и кэширует — правка одного модуля не пересобирает всё), инкапсуляция (через internal и явные API нельзя случайно дёрнуть внутренности чужой фичи), переиспользование (:core:* подключается многими фичами), контроль архитектуры (граф физически запрещает циклы и протечки слоёв) и параллельная работа команд. Фичи не зависят друг от друга, общее выносится в core. Чтобы не дублировать build-конфиги, применяют Gradle convention plugins (как в Now in Android).

В чём смысл single source of truth и offline-first? Middle+

SSOT — у каждого типа данных один авторитетный источник; все остальные читают из него и не имеют права независимо его менять. В offline-first роль SSOT играет локальная БД (Room): UI всегда читает из БД через Flow, а сеть лишь обновляет БД. Плюсы: приложение работает без сети, данные консистентны, нет «мигания» при рефетче, проще дебажить. Репозиторий инкапсулирует разрешение конфликтов между источниками и выставляет наружу единый поток данных.

Когда manual DI / service locator уместнее Hilt? Senior

Manual DI (конструкторная проводка вручную, фабрики) хорош для маленьких приложений, библиотек (не навязывать пользователю Dagger) и прототипов. Service locator проще завести, но скрывает зависимости класса (их не видно в конструкторе) и хуже тестируется — обычно считается анти-паттерном. Hilt окупается, когда граф большой и завязан на Android-жизненные циклы: он даёт compile-time проверку и убирает рутину. Универсальное правило — предпочитать конструкторную инъекцию, а инструмент (ручной/Hilt) выбирать по размеру графа.

Как инжектить зависимости в WorkManager Worker? Middle+

WorkManager сам инстанцирует Worker, поэтому обычная конструкторная инъекция не сработает. Нужен @HiltWorker + @AssistedInject: Context и WorkerParameters приходят от фреймворка через @Assisted, а остальные зависимости (репозитории и т.п.) Hilt поставляет из графа как обычно. В Application подключается HiltWorkerFactory через Configuration.Provider (или удаляется дефолтный инициализатор WorkManager в манифесте).

Почему StateFlow на границе ViewModel→UI, а не cold Flow? Senior

StateFlow — горячий и всегда имеет текущее значение: подписавшийся UI сразу получает актуальный state, без повторного запуска upstream на каждого коллектора. Cold Flow перезапускал бы пайплайн под каждую подписку. Стандарт — собрать состояние через stateIn(scope, SharingStarted.WhileSubscribed(5000), initial): это конфлейтит обновления, шарит один upstream между подписчиками и держит его живым ещё 5 секунд после ухода последнего — чтобы пережить поворот экрана, но не работать, когда экран закрыт. Внутри слоёв data/domain, наоборот, держат cold Flow как «рецепт» данных.

Single-Activity vs Multi-Activity: что выбрать сегодня и почему? Middle+

Современная рекомендация Google — Single-Activity: одна Activity-хост, навигация между экранами через Navigation Component (фрагменты или composable-destinations). Плюсы: единый back stack и предсказуемая навигация, общий ViewModelStoreOwner на уровне Activity для расшаривания state, проще передавать аргументы и делать deep links, меньше накладных на пересоздание Activity и работу с Intent. Multi-Activity оправдан для по-настоящему независимых частей: отдельный флоу логина, экран-«запуск по интенту» извне, изоляция тяжёлого legacy. Подвох: при Single-Activity конфиг-изменения переживает уже не Activity, а ViewModel и back stack — поэтому скоупы (ActivityRetained) и записи back stack становятся центральными.

Что такое multibindings (@IntoSet/@IntoMap) и где они реально нужны? Senior

Multibindings собирают вклад нескольких модулей в одну коллекцию: @IntoSet добавляет элемент в Set<T>, @IntoMap — в Map<K, T> по ключу (через @MapKey-аннотацию вроде @StringKey/@ClassKey). Инжектируется готовый Set<T>/Map<K, Provider<T>>, который дополняется из разных модулей, не трогая место потребления — это open/closed на практике. Типичные применения: набор инициализаторов приложения, плагины аналитики, регистрация ViewModel'ей по ключу-классу (multibinding-фабрика в Dagger без Hilt), список интерсепторов OkHttp. В multi-module особенно ценно: feature-модуль «вкладывает» свой инициализатор в общий Set, а :app о нём ничего не знает.

Как пробросить аргумент в ViewModel: SavedStateHandle vs assisted injection? Senior

Если аргумент приходит из навигации, прокидывать руками не надо: при @HiltViewModel + Navigation Hilt кладёт аргументы destination в SavedStateHandle, и ViewModel читает их через savedStateHandle.get<String>("userId") или type-safe savedStateHandle.toRoute<ProfileRoute>(). К тому же SavedStateHandle переживает process death (лежит в Bundle). Assisted injection (@AssistedInject + @AssistedFactory) нужен, когда параметр не из навигации и не сериализуется в Bundle — рантайм-объект, переданный из другого экрана через память. Правило: аргумент навигации → SavedStateHandle; рантайм-зависимость, которую нельзя положить в Bundle → assisted. Хардкодить значение в конструктор @HiltViewModel нельзя — Hilt должен уметь построить VM без вашего участия.

Чем grouping по слоям отличается от grouping по фичам при модуляризации? Middle+

Layer-based (модули :data, :domain, :ui на весь проект) прост, но плохо масштабируется: фича размазана по трём модулям, команды конфликтуют, граница инкапсуляции слабая. Feature-based / по доменам (как в Now in Android: :feature:home, :feature:profile, каждый со своими ui+presentation поверх общих :core) даёт сильную инкапсуляцию (internal внутри фичи), параллельную работу команд и независимую сборку. На практике берут гибрид: фичи изолированы по вертикали, инфраструктура вынесена в :core:* (network, database, designsystem, common). Фичи не зависят друг от друга — общаются через :app (навигационный граф) и абстракции в :core. Подвох: модуляризация ради сборки без продуманных границ фич часто только усложняет граф, не ускоряя его.

@HiltAndroidApp

        initializers.forEach { it.init() }
    }
}

@JvmSuppressWildcards обязателен: иначе Kotlin сгенерирует Set<? extends AppInitializer>, и Dagger не найдёт биндинг. Главный поинт — новый модуль добавляет инициализатор, не трогая Application (open/closed): :app не знает, кто и что вложил в Set.

Задача 6. Дан репозиторий, который при ошибке сети бросает исключение наружу и при этом перезаписывает кэш пустым списком. Исправьте: ошибка должна доходить до вызывающего как Result, а локальный кэш не должен стираться при сетевом сбое.

Показать решение 
// БЫЛО (сломано): кэш затирается до проверки успеха
suspend fun refresh() {
    val remote = runCatching { api.fetch() }.getOrDefault(emptyList())
    dao.upsertAll(remote.map { it.toEntity() })   // при сбое затрёт кэш пустым
}

// СТАЛО: upsert только на успехе, ошибка — наружу как Result
suspend fun refresh(): Result<Unit> = runCatching {
    val remote = api.fetch()                        // бросит IOException/HttpException
    dao.upsertAll(remote.map { it.toEntity() })     // не выполнится при сбое
}

// observeAll() остаётся холодным Flow из БД (SSOT) — кэш цел
fun observeAll(): Flow<List<Article>> =
    dao.observeAll().map { it.map { e -> e.toDomain() } }

Ключ: upsertAll внутри runCatching выполняется только после успешного api.fetch(). Бросил fetch — runCatching поймал, upsertAll не вызвался, кэш цел. ViewModel: repo.refresh().onFailure { ... } отдаёт ошибку в side-effect-канал или в error-поле state. Тонкость: внутри suspend не глотайте CancellationException — при общем catch её перебрасывают вручную.
