Инкапсуляция: состояние меняется только через контракт OOP

Инкапсуляция — не тайник для полей. Объект сам отвечает за правила изменения своего состояния: внешний код вызывает публичный метод, а объект уже проверяет инварианты и не даёт записать произвольное значение напрямую.

UserViewModel.kt
class UserViewModel : ViewModel() {
    // изменяемое состояние — приватно
    private val _state = MutableStateFlow(UserUiState())
    // наружу — только неизменяемый контракт для чтения
    val state: StateFlow<UserUiState> = _state.asStateFlow()

    fun onNameChange(name: String) {
        _state.update { it.copy(name = name) }   // мутируем только изнутри
    }
}
Наследование: что такое is-a OOP

is-a означает «разновидность». Если CardPayment наследует/реализует PaymentMethod, его можно использовать везде, где ожидают PaymentMethod. Если подтип требует особых проверок или ломает ожидания базового типа, это уже проблема LSP.

interface PaymentMethod {
    fun pay(amount: Long): Result<Unit>
}

class CardPayment : PaymentMethod {
    override fun pay(amount: Long): Result<Unit> = runCatching { /* card */ }
}
Полиморфизм: override и перегрузка методов OOP

Динамический полиморфизм — один контракт, разные реализации, выбранные в рантайме через override. Статический полиморфизм — перегрузка: несколько методов с одним именем, но разными параметрами; выбор делается компилятором.

interface Formatter { fun format(value: Long): String }
class DateFormatter : Formatter { override fun format(value: Long) = "date=$value" }
class MoneyFormatter : Formatter { override fun format(value: Long) = "$value ₽" }

fun log(message: String) { /* overload #1 */ }
fun log(error: Throwable) { /* overload #2 */ }
Абстракция: interface vs abstract class OOP

interface — контракт роли: класс может реализовать много интерфейсов, но интерфейс не хранит состояние через backing field. abstract class — базовый неполный класс: его нельзя создать напрямую, он может иметь конструктор, состояние, обычные и абстрактные члены.

Важная точность: если в классе есть абстрактный метод или свойство, сам класс обязан быть abstract. Но абстрактный класс не обязан иметь хотя бы один абстрактный метод: его можно пометить abstract, чтобы запретить прямое создание и заставить наследоваться от него.

abstract class BaseViewModel<S>(initial: S) : ViewModel() {
    protected val _state = MutableStateFlow(initial)
    val state: StateFlow<S> = _state.asStateFlow()

    abstract fun onRetry()
}
Kotlin-класс по умолчанию final. Зачем так сделали и как это связано с наследованием и хрупким базовым классом? ★

В Kotlin классы и их методы закрыты для наследования по умолчанию: чтобы наследоваться или переопределять, нужны явные open или abstract. Это прямое следование совету Джошуа Блоха из Effective Java — «Design and document for inheritance or else prohibit it». Наследование от класса, который под него не проектировали, легко ломает инварианты, как только меняется база: это и есть проблема хрупкого базового класса.

Final-по-умолчанию заставляет автора осознанно открыть точку расширения, а не оставить её открытой случайно. Интерфейсы и их методы при этом всегда открыты — там переопределение реализации это нормальный сценарий. Практический эффект простой: язык подталкивает к композиции и явным контрактам вместо случайной иерархии.

Запусти: классика хрупкого базового класса — добавили 3 элемента, а счётчик показывает 6. addAll базы внутри зовёт add, а наследник его переопределил. Именно это ловит final по умолчанию.

// базу не проектировали под наследование: addAll внутри зовёт add open class CountingBag { var added = 0 private set open fun add(x: Int) { added++ } open fun addAll(xs: List<Int>) { xs.forEach { add(it) } } // self-вызов add } // наследник «логично» считает свои добавления — и ломает базу class LoudBag : CountingBag() { override fun add(x: Int) { println("add $x"); super.add(x) } override fun addAll(xs: List<Int>) { xs.forEach { add(it) }; super.addAll(xs) } } fun main() { val bag = LoudBag() bag.addAll(listOf(1, 2, 3)) println("added = ${bag.added}") // 6, а не 3 — двойной учёт через self-call базы }

Композиция, иерархия, расширение

Грань такая: наследование описывает is-a («разновидность»), композиция описывает has-a («имеет/использует»), интерфейс часто описывает роль can-do («умеет делать»). Ошибка начинается там, где роль или зависимость превращают в жёсткую иерархию.

Когда передавать зависимость, а когда создавать внутри

Через конструктор передают то, что влияет на поведение класса, требует подмены в тестах, имеет внешний ресурс или lifecycle: Repository, Api, Clock, Logger, Dispatcher. Внутри класса допустимо создавать маленькие локальные value-объекты без внешнего состояния: например, DTO для результата или простой formatter, если его не нужно конфигурировать и мокать.

// лучше: зависимость видна и подменяема
class ProfileViewModel(
    private val loadProfile: LoadProfileUseCase
) : ViewModel()

// хуже: класс сам знает, как собрать сеть, и его трудно тестировать
class ProfileViewModel : ViewModel() {
    private val api = Retrofit.Builder().build().create(ProfileApi::class.java)
}

При делегировании через by метод делегата не видит override класса-обёртки. Почему, и к какому багу это приводит? ★

Делегирование class A(b: B) : I by b генерирует методы интерфейса I, которые просто зовут тот же метод на сохранённой ссылке b. Делегат b ничего не знает об обёртке A. Если A переопределяет метод m(), а другой метод делегата внутри себя вызывает m() — вызовется реализация из b, а не override из A. Это не «динамический» self, как при наследовании: b и A — разные объекты, и виртуальная диспетчеризация идёт по b.

Эффект: попытка частично подменить поведение через override поверх делегата молча не работает для внутренних self-вызовов делегата. Это фундаментальное отличие композиции-делегирования от наследования и причина, почему by нельзя использовать как замену наследованию там, где нужен переопределяемый self-call.

Запусти: greet() на обёртке зовёт name(), но печатает «Базовый» — self-вызов делегата идёт по самому делегату, а не по override обёртки.

interface Greeter { fun name(): String fun greet(): String } class Base : Greeter { override fun name() = "Базовый" override fun greet() = "Привет, ${name()}!" // self-вызов внутри делегата } class Loud(origin: Greeter) : Greeter by origin { override fun name() = "ГРОМКИЙ" // пытаемся подменить } fun main() { val g = Loud(Base()) println(g.name()) // ГРОМКИЙ — прямой вызов попал в override println(g.greet()) // forward в делегат; его name() остался "Базовый" }

SOLID — пять принципов

SOLID — пять принципов объектного проектирования. Их удобно отвечать по схеме: определение, симптом нарушения, Android-пример, паттерны/архитектурные приёмы, которые помогают принцип реализовать.

ПринципЧем обычно реализуется в архитектуре
SRPMVVM, UseCase/Interactor, Repository, Mapper, Formatter, отдельные Android-компоненты по ответственности.
OCPStrategy, Factory Method, Template Method, sealed-иерархии, polymorphism вместо большого when.
LSPDesign by contract, маленькие интерфейсы, композиция вместо опасного наследования, корректные sealed-типы.

DIPDependency Injection, Hilt/Dagger, Clean Architecture, Repository interface, Adapter для внешних SDK.

S — Single Responsibility (единственная ответственность)

У класса должна быть одна причина для изменения. ViewModel, который ходит в сеть, парсит JSON, маппит DTO, форматирует дату и держит UI-state, меняется по слишком многим причинам.

Паттерны/приёмы: MVVM, UseCase/Interactor, Repository, Mapper, Formatter, отдельные сервисы по одной ответственности.

Плохо vs хорошо
// ❌ God-object: сеть + маппинг + форматирование
class ProfileViewModel {
    fun load() { /* http, json parse, date format, ui state ... */ }
}

// ✅ VM отвечает за UI-state, UseCase — за сценарий, repo — за данные
class ProfileViewModel(
    private val loadProfile: LoadProfileUseCase
) : ViewModel() {
    fun load() { /* вызов use case + reduce в UiState */ }
}

class LoadProfileUseCase(
    private val repo: ProfileRepository,
    private val mapper: ProfileMapper
)
★
Тонкость. Если меняется способ загрузки профиля — API, cache, Room, fallback — это причина менять repository/use case, а не ViewModel. Если меняется отображение состояния экрана — это причина менять ViewModel/UI-state.

O — Open/Closed (открыт для расширения, закрыт для изменения)

Добавлять поведение — через новые типы/стратегии, а не правкой существующего when.

Паттерны/приёмы: Strategy, Factory Method, Template Method, polymorphism, sealed-иерархии с контролируемым расширением.

// ✅ новый способ оплаты = новый класс, старый код не трогаем
interface PaymentMethod { fun pay(amount: Long): Result<Unit> }
class CardPayment : PaymentMethod { /* ... */ }
class SbpPayment  : PaymentMethod { /* ... */ }

class Checkout(private val method: PaymentMethod) {
    fun confirm(sum: Long) = method.pay(sum)
}

L — Liskov Substitution (подстановка Барбары Лисков)

Подтип не должен ломать контракт базового типа. Классика — Square наследует Rectangle: установка ширины меняет высоту, и код, рассчитывающий на независимость сторон, ломается.

Паттерны/приёмы: Design by contract, композиция вместо опасного наследования, маленькие интерфейсы, корректные sealed-типы.

⚠️
Симптом нарушения LSP. Подкласс кидает UnsupportedOperationException на методе родителя или требует проверок if (x is Subtype) у вызывающего. Решение — пересмотреть иерархию (часто в пользу композиции/интерфейсов).

I — Interface Segregation (разделение интерфейсов)

Лучше много тонких интерфейсов, чем один «толстый». Клиент не должен зависеть от методов, которые не использует.

Паттерны/приёмы: role interfaces, Facade для крупного API, отдельные listener/callback интерфейсы под конкретных клиентов.

Связь с безопасностью: ISP помогает ограничивать доступ по ролям. В контексте БД не всем клиентам нужен полный Repository: экрану отчётов достаточно Readable, экрану редактирования — Writable, синхронизатору — отдельный Syncable. Это тот же принцип минимальных привилегий: меньше методов в контракте — меньше случайных и опасных операций. См. также раздел «Безопасное хранение секретов».

// ❌ "бог-интерфейс"
interface Repository { fun read(); fun write(); fun sync(); fun export() }

// ✅ тонкие роли
interface Readable { fun read() }
interface Writable { fun write() }
interface Syncable { fun sync() }

D — Dependency Inversion (инверсия зависимостей)

Модули верхнего уровня зависят от абстракций, а не от деталей. Это фундамент DI/Hilt и Clean Architecture: domain определяет интерфейс Repository, data его реализует.

Паттерны/приёмы: Dependency Injection, Hilt/Dagger, Repository interface, Adapter для внешнего SDK, Factory для сборки реализаций.

DIP: стрелка зависимости направлена к абстракции, реализация «подключается» снизу
// domain — знает только абстракцию
interface UserRepository { suspend fun user(id: String): User }
class GetUser(private val repo: UserRepository) {
    suspend operator fun invoke(id: String) = repo.user(id)
}
// data — деталь, реализует контракт; внедряется через DI (Hilt)
class UserRepositoryImpl(private val api: Api) : UserRepository {
    override suspend fun user(id: String) = api.fetch(id).toDomain()
}

Сформулируй LSP формально через пред- и постусловия. Почему переопределение, которое только усиливает проверку входных данных, уже нарушает принцип? ★

Формально (по Лисков — Уинг): подтип не должен усиливать предусловия и не должен ослаблять постусловия и инварианты базового типа; история-свойство и набор исключений тоже должны сохраняться. Усиление предусловия — это когда подтип принимает более узкий набор входов, чем база. Скажем, базовый метод принимал любой String, а override бросает исключение на пустой строке. Клиент, который писался под контракт базы, законно передаёт значение, валидное для базового типа, — и ломается.

Клиент при этом не делает ничего «неправильного»: он опирается на опубликованный контракт. Поэтому LSP — про поведенческую совместимость, а не только про совпадение сигнатур: компилятор подстановку разрешит, а рантайм-контракт сломается. Параметры в override должны быть контравариантны, а возвращаемый тип — ковариантен (Kotlin поддерживает ковариантность возвращаемого типа при переопределении).

Запусти: клиент написан под контракт Validator (принимает любую строку) и падает на подтипе, который усилил предусловие до «непустая».

open class Validator { // контракт базы: принимает любую строку, включая пустую open fun normalize(input: String): String = input.trim() } class StrictValidator : Validator() { override fun normalize(input: String): String { require(input.isNotEmpty()) { "пустая строка запрещена" } // усилили предусловие return input.trim() } } // клиент писался под Validator и законно передаёт "" fun process(v: Validator): String = v.normalize("") fun main() { println("Validator: ok, вернул [${process(Validator())}]") // ok, [] println("Strict: ${runCatching { process(StrictValidator()) }}") // Failure: контракт сломан }

Singleton vs public static val: в чём отличие? Middle

Singleton — это один экземпляр типа + глобальная точка доступа + возможность держать поведение и состояние за контролируемым API. В Kotlin object создаёт такой экземпляр лениво и потокобезопасно при первом доступе. public static val или const val — это просто поле/константа, а не объект с жизненным циклом, контрактом и инкапсулированным поведением. @Singleton в Hilt — не языковой singleton, а scope: один instance внутри DI-компонента.

Kotlin object потокобезопасен при ленивой инициализации. Что именно это гарантирует и зачем тогда знать про double-checked locking? ★

Kotlin object компилируется в Java-класс со статическим финальным полем INSTANCE, которое создаётся в статическом инициализаторе. Потокобезопасность здесь даёт сама JVM: загрузка и инициализация класса проходят под блокировкой инициализации класса (JLS §12.4), поэтому INSTANCE создаётся ровно один раз и виден всем потокам без ручной синхронизации. Это идиома holder / lazy-class-init.

Double-checked locking с volatile нужен в другом случае — когда singleton зависит от рантайм-параметра (например, требует Context) и поэтому не может быть простым object: пишется ручная фабрика getInstance(context). Без volatile и двойной проверки тут возможны гонка и публикация недоинициализированного объекта. То есть object закрывает беспараметрический случай, а DCL — параметризованный; знать надо оба, потому что на Android параметризованный singleton встречается постоянно.

Запусти: 50 потоков лезут к object одновременно — инициализатор печатает «init» ровно один раз, и все видят один и тот же экземпляр.

import java.util.concurrent.Executors import java.util.concurrent.ConcurrentHashMap object Config { init { println("init Config") } // выполнится РОВНО один раз val id = System.nanoTime() } fun main() { val pool = Executors.newFixedThreadPool(8) val seen = ConcurrentHashMap.newKeySet<Long>() repeat(50) { pool.submit { seen.add(Config.id) } } // гонка за первой инициализацией pool.shutdown() Thread.sleep(200) println("разных экземпляров: ${seen.size}") // 1 — JVM сериализует class-init }

Strategy, State, Decorator — паттерны на композиции

Три паттерна, которые на собесе путают чаще остальных. Это прямой мостик от «композиция лучше наследования» к конкретике: все три прячут варьируемое поведение за интерфейсом, но решают разные задачи.

Strategy — алгоритм за интерфейсом, выбор снаружи

Алгоритм выносим за интерфейс, а конкретную реализацию выбирает и подставляет клиент. Способ оплаты, способ сортировки, загрузчик картинок — стратегии не знают друг о друге и не переключают себя сами.

interface ImageLoader { suspend fun load(url: String): Bitmap }
class CoilLoader : ImageLoader { /* ... */ }
class GlideLoader : ImageLoader { /* ... */ }

// клиент получает стратегию извне (через конструктор/DI)
class Avatar(private val loader: ImageLoader) {
    suspend fun show(url: String) = loader.load(url)
}

State — поведение меняется по внутреннему состоянию

Объект меняет поведение при смене внутреннего состояния, и сами состояния обычно знают про переходы — переключают контекст на следующее состояние. В Android это чаще всего sealed-иерархия UiState и when по ней.

sealed interface UiState {
    data object Loading : UiState
    data class Content(val items: List<Item>) : UiState
    data class Error(val message: String) : UiState
}

fun render(state: UiState) = when (state) {
    UiState.Loading      -> showSpinner()
    is UiState.Content   -> showList(state.items)
    is UiState.Error     -> showError(state.message)
}

Decorator — наращиваем поведение обёрткой

Декоратор оборачивает объект и добавляет поведение динамически, сохраняя тот же интерфейс. Декораторы комбинируются в рантайме в любом порядке: логирование + кэш + ретрай поверх одного клиента. Наследование дало бы класс под каждую комбинацию фич — комбинаторный взрыв. В Android это OkHttp Interceptor (цепочка декораторов запроса) и ContextWrapper.

interface ApiClient { suspend fun get(url: String): String }

class LoggingApiClient(private val delegate: ApiClient) : ApiClient {
    override suspend fun get(url: String): String {
        Log.d("API", "-> $url")
        return delegate.get(url).also { Log.d("API", "<- $it") }
    }
}

// обёртки складываются как угодно — все реализуют ApiClient
val client = LoggingApiClient(RealApiClient())
Decorator и Proxy реализуют тот же интерфейс и оборачивают объект. В чём разница? ★

Совпадает структура — оба держат ссылку на обёрнутый объект того же типа и делегируют ему. Разница в намерении. Decorator добавляет новое поведение поверх существующего и рассчитан на стекирование: несколько декораторов навешивают независимые фичи (лог, кэш, ретрай). Proxy контролирует доступ к объекту, не меняя его обязанности: ленивое создание, проверка прав, удалённый вызов, подсчёт ссылок.

Грубая проверка: если объект оборачивают, чтобы он «умел больше», — это декоратор; если чтобы решать, можно ли и когда обращаться к настоящему объекту, — это прокси. На Android к прокси ближе ленивая инициализация тяжёлого ресурса и обёртки безопасности, к декоратору — цепочка Interceptor.

Когда SOLID мешает: over-engineering и YAGNI

Сильный сигнал для Senior — не догматичное применение SOLID. SOLID толкает к абстракциям, YAGNI запрещает строить лишнее, и конфликт между ними реальный. Абстракция не бесплатна: за неё платят лишним слоем косвенности и тем, что по коду труднее ходить.

Критерий ввода абстракции. Вводи её, когда есть вторая реализация, реальная потребность подменить зависимость в тесте или граница реально нестабильна. Иначе KISS/YAGNI важнее «галочки SOLID». SOLID — эвристики, а не закон.
Где проходит граница между «здоровой абстракцией» и over-engineering, если требований ещё нет? ★

Полезный критерий — «правило трёх» и стоимость отката. Пока вариация одна, абстракция спекулятивна: вы угадываете ось изменения, ещё не зная её. Появилась вторая реализация — гипотеза подтвердилась, абстракцию вводят по факту, а не на упреждение. Это дешевле, потому что рефакторинг к интерфейсу из конкретного класса — механическая операция, а вот ломать неверно угаданную абстракцию дорого: она уже расползлась по вызовам.

Отдельно стоит «абстракция ради теста»: если зависимость и так подменяема (передаётся через конструктор, имеет fake-реализацию), интерфейс может не понадобиться — мок можно сделать и поверх открытого класса. Граница проходит там, где косвенность начинает стоить дороже, чем гибкость, которую она даёт прямо сейчас, а не гипотетически.

Чем абстрактный класс отличается от интерфейса в Kotlin? Middle

Интерфейс описывает роль/контракт, может иметь реализации методов и абстрактные свойства, но не хранит состояние через backing field. Класс может реализовать много интерфейсов. Абстрактный класс нельзя создать напрямую; он может иметь конструктор, состояние, обычные и абстрактные члены, но наследоваться можно только от одного класса. Нужна роль — интерфейс. Нужна общая база со state/constructor — abstract class.

Приведи нарушение SRP из своего проекта и как исправил. Middle+

Хороший ответ — конкретная история: «ViewModel грузил данные, маппил DTO в UI-модель и форматировал даты. При смене формата дат приходилось трогать VM. Вынес маппинг в Mapper, форматирование — в DateFormatter, сеть — в Repository. VM стал тонким оркестратором, покрылся unit-тестами через моки».

Где в Android нарушается принцип Лисков? Senior

Пример: иерархия View, где подкласс переопределяет метод и кидает исключение или игнорирует вызов, ломая ожидания клиента. Или коллекции: List, возвращающий UnsupportedOperationException на add() (immutable view) — формально подтип, но контракт MutableList нарушает. Решение — корректная иерархия типов (Kotlin разделяет List и MutableList именно поэтому).

Как SOLID связан с DI и Clean Architecture? Senior

DIP — прямой фундамент DI: верхние слои зависят от интерфейсов, а конкретные реализации внедряются контейнером (Hilt/Dagger). Clean Architecture применяет DIP на уровне слоёв: domain объявляет интерфейсы репозиториев, data их реализует, зависимость направлена внутрь — к политике, а не к деталям. OCP и ISP помогают расширять слои без правок существующего кода.

Какие принципы SOLID нарушает Context и почему? Senior

Context в Android — framework boundary, поэтому его нельзя оценивать как обычный прикладной класс. Но если такой API сделать в своём коде, он выглядел бы как нарушение SRP и ISP: через него доступны ресурсы, сервисы, запуск Activity, broadcast, файлы, preferences и многое другое, значит у него много причин для изменения и клиенты зависят от огромного интерфейса. DIP нарушается, когда domain/use case напрямую зависит от Context; лучше передать узкий контракт: StringProvider, PermissionChecker, FileStore.

Почему «композиция лучше наследования»? Всегда ли? Middle+

Композиция даёт слабую связанность, гибкость (поведение можно подменить, в т.ч. в рантайме) и тестируемость. Наследование жёстко привязывает подкласс к реализации базового (проблема хрупкого базового класса). Но наследование уместно при истинном «is-a» и стабильной иерархии (например, sealed-классы состояний). Правило: «is-a» → наследование, «has-a/can-do» → композиция.

Что такое coupling и cohesion? Middle

Coupling — степень зависимости модулей друг от друга (хотим низкую). Cohesion — насколько элементы внутри модуля связаны общей задачей (хотим высокую). Хороший дизайн: low coupling + high cohesion. SOLID, по сути, инструменты для достижения именно этого баланса.

Что такое diamond problem и как Kotlin его решает для интерфейсов с дефолтными методами? Senior

Diamond problem возникает, когда класс наследует один и тот же метод по двум путям и неясно, чью реализацию брать. Множественного наследования классов в Kotlin нет, но два интерфейса могут дать дефолтную реализацию одного метода. Kotlin в этом случае требует, чтобы класс-наследник явно переопределил метод, иначе ошибка компиляции. Внутри override доступен квалифицированный super-вызов super<Interface>.method(), которым выбирают конкретную реализацию или комбинируют обе. Неоднозначность решается явно на стороне наследника, а не молчаливым выбором компилятора.

Зачем нужны абстрактные классы, если есть интерфейсы с дефолтными методами? Middle+

Интерфейсы с дефолтными методами дают разделяемое поведение, но не дают разделяемое изменяемое состояние: у интерфейса нет backing field, свойство в нём либо абстрактно, либо вычисляется через геттер без хранения. Абстрактный класс может иметь конструктор, поля-состояние и инициализацию и гарантирует единственную базу. Берём abstract class, когда нескольким наследникам нужна общая база с хранимым состоянием и общим конструктором (BaseViewModel с _state). Берём интерфейс, когда нужна роль/контракт, которую класс совмещает с другими. Ещё отличие: добавить дефолтный метод в интерфейс — не сломать существующих реализаторов, что важно для эволюции публичного API.

Глубокая иерархия наследования — почему это плохо и какая метрика это ловит? Middle+

Глубокая иерархия (большой DIT, depth of inheritance tree) усиливает связанность: изменение в базовом классе каскадом задевает всех потомков, поведение объекта размазано по цепочке, и труднее удержать LSP на каждом уровне. Чем глубже дерево, тем сложнее понять и протестировать лист, потому что его реальное поведение — сумма всех предков. Лечится это через «favor composition over inheritance»: вместо нового уровня иерархии выносим поведение в отдельный объект и внедряем его. На практике держат иерархии плоскими (1–2 уровня), а варьируемость прячут в композицию/стратегии.

Пример нарушения OCP в типичном Android-коде и как его убрать? Middle

Классика — большой when по типу: рендер разных типов элементов ленты или обработка аналитических событий через when (item.type), который растёт при каждом новом типе и правится в одном месте. Это нарушает OCP: добавление типа = модификация существующего кода. Убирают полиморфизмом/Strategy — каждый тип реализует общий интерфейс (делегаты в RecyclerView через AdapterDelegate или sealed-тип с собственным рендером), и добавление типа становится добавлением класса. Нюанс: с sealed-иерархией when по веткам тоже придётся расширять, зато компилятор подсветит все неполные when — это контролируемое, а не молчаливое нарушение.

Задача 1. Дан класс OrderProcessor, который в одном методе делает всё: загружает заказ из сети, считает скидку, форматирует чек и отправляет письмо. Нарушение SRP очевидно. Разбейте класс на несколько с правильно распределённой ответственностью, сохранив рабочий публичный API. Допущение: сеть и email — заглушки (TODO()), акцент на структуру.

Показать решение 
data class Order(val id: String, val amount: Long)

// загрузка заказа — одна причина меняться: сменился источник данных
class OrderRepository {
    fun load(id: String): Order = TODO("сеть/БД")
}

// правила скидки — одна причина: изменились правила бизнеса
class DiscountCalculator {
    fun discount(order: Order): Long =
        if (order.amount > 10_000L) order.amount / 10 else 0L
}

// форматирование — одна причина: изменился формат чека
class ReceiptFormatter {
    fun format(order: Order, discount: Long): String =
        "Заказ ${order.id}: итого ${order.amount - discount} ₽"
}

// уведомления — одна причина: сменился транспорт (email → push)
class Notifier {
    fun send(address: String, message: String): Unit = TODO("SMTP/FCM")
}

// оркестратор — тонкий, только координирует
class OrderProcessor(
    private val repo: OrderRepository,
    private val discount: DiscountCalculator,
    private val formatter: ReceiptFormatter,
    private val notifier: Notifier
) {
    fun process(orderId: String, email: String) {
        val order = repo.load(orderId)
        val disc  = discount.discount(order)
        val text  = formatter.format(order, disc)
        notifier.send(email, text)
    }
}

Объяснение: каждый класс меняется ровно по одной причине. OrderProcessor знает только о координации — сам ничего не вычисляет и не форматирует. При смене формата чека трогаем только ReceiptFormatter; при смене правил скидки — только DiscountCalculator. Все зависимости передаются через конструктор, что упрощает тестирование через моки.

Задача 2. Реализуйте иерархию фигур с методом area(). Требование: добавление новой фигуры не должно менять существующий код (OCP). Начальный набор: Circle и Rectangle. Покажите, как добавить Triangle без правки кода, который считает суммарную площадь списка фигур.

Показать решение 
// абстракция — стабильный контракт, закрытый для изменений
interface Shape {
    fun area(): Double
}

// каждая фигура — расширение, а не правка существующего
class Circle(private val radius: Double) : Shape {
    override fun area() = Math.PI * radius * radius
}

class Rectangle(private val w: Double, private val h: Double) : Shape {
    override fun area() = w * h
}

// новая фигура — просто новый класс; totalArea() не трогаем
class Triangle(private val base: Double, private val height: Double) : Shape {
    override fun area() = 0.5 * base * height
}

// код верхнего уровня зависит от Shape, а не от конкретных классов
fun totalArea(shapes: List<Shape>): Double = shapes.sumOf { it.area() }

fun main() {
    val shapes: List<Shape> = listOf(
        Circle(5.0),
        Rectangle(4.0, 6.0),
        Triangle(3.0, 8.0)      // добавили без правки totalArea
    )
    println(totalArea(shapes))  // 114.539...
}

Объяснение: totalArea() открыт для расширения (новые типы фигур) и закрыт для изменения — он никогда не правится при добавлении фигуры. Полиморфизм через interface — классический приём реализации OCP. Альтернатива: sealed interface Shape с when, но тогда каждое новое плечо потребует правки when, что нарушает OCP.

Задача 3. Реализуйте паттерн Observer вручную (без Flow и LiveData): есть источник событий EventBus и несколько подписчиков. Подписчик должен уметь отписаться. Покажите использование. Бонус: объясните, какую проблему решает этот паттерн и чем отличается от прямого колбэка.

Показать решение 
// контракт наблюдателя
fun interface Observer<T> {
    fun onEvent(event: T)
}

// источник событий — хранит подписчиков и рассылает
class EventBus<T> {
    private val observers = mutableListOf<Observer<T>>()

    fun subscribe(observer: Observer<T>) {
        observers.add(observer)
    }

    fun unsubscribe(observer: Observer<T>) {
        observers.remove(observer)
    }

    fun emit(event: T) {
        // копируем список: подписчик может отписаться внутри onEvent
        observers.toList().forEach { it.onEvent(event) }
    }
}

// использование
fun main() {
    val bus = EventBus<String>()

    val logger = Observer<String> { println("LOG: $it") }
    val analytics = Observer<String> { println("ANALYTICS: $it") }

    bus.subscribe(logger)
    bus.subscribe(analytics)

    bus.emit("user_login")       // оба получают событие

    bus.unsubscribe(analytics)
    bus.emit("purchase")         // только logger
}

Объяснение: Observer разрывает прямую зависимость источника от конкретных потребителей — источник не знает, кто и сколько подписчиков. Прямой колбэк предполагает одного получателя, жёстко вшитого в источник; Observer поддерживает N подписчиков и позволяет добавлять/удалять их в рантайме. В Android этот же принцип реализован через LiveData и StateFlow/SharedFlow, которые добавляют lifecycle-осведомлённость и потокобезопасность.

Задача 4. Реализуйте паттерн Decorator: базовый ApiClient с методом suspend fun get(url: String): String и два декоратора поверх него — LoggingApiClient (логирует запрос/ответ) и RetryingApiClient (повторяет до N раз при исключении). Покажите, что декораторы комбинируются в любом порядке и оба реализуют тот же интерфейс. Объясните, почему это лучше иерархии подклассов.

Показать решение 
interface ApiClient {
    suspend fun get(url: String): String
}

class RealApiClient : ApiClient {
    override suspend fun get(url: String): String = TODO("реальный http")
}

// декоратор 1: логирование
class LoggingApiClient(private val delegate: ApiClient) : ApiClient {
    override suspend fun get(url: String): String {
        Log.d("API", "-> $url")
        val response = delegate.get(url)
        Log.d("API", "<- $response")
        return response
    }
}

// декоратор 2: повтор при ошибке
class RetryingApiClient(
    private val delegate: ApiClient,
    private val times: Int = 3
) : ApiClient {
    override suspend fun get(url: String): String {
        repeat(times - 1) {
            runCatching { return delegate.get(url) }
        }
        return delegate.get(url)   // последняя попытка пробрасывает ошибку
    }
}

// сборка: обёртки складываются в любом порядке
val client: ApiClient = RetryingApiClient(LoggingApiClient(RealApiClient()))

Объяснение: все три класса реализуют ApiClient, поэтому декораторы складываются как угодно и сохраняют единый интерфейс для клиента. Наследование потребовало бы отдельный класс под каждую комбинацию (LoggingRetryingApiClient и т.д.) — комбинаторный взрыв при росте числа независимых фич. Decorator добавляет поведение в рантайме без правки базового класса (OCP).

Задача 5. Дан тип Bird с методом fly(). В иерархии появляется Penguin, который не летает, и его fly() бросает UnsupportedOperationException. Покажите клиент, который из-за этого ломается (нарушение LSP), и перепроектируйте иерархию так, чтобы нарушения не было.

Показать решение 
// ❌ нарушение LSP
interface Bird { fun fly() }
class Penguin : Bird {
    override fun fly() = throw UnsupportedOperationException("пингвины не летают")
}
// клиент законно работает с Bird — и падает на Penguin
fun makeAllFly(birds: List<Bird>) = birds.forEach { it.fly() }

// ✅ разделяем способности через интерфейсы-роли (ISP + LSP)
interface Bird { fun eat() }
interface Flyable { fun fly() }

class Sparrow : Bird, Flyable {
    override fun eat() { /* ... */ }
    override fun fly() { /* ... */ }
}
class Penguin : Bird {
    override fun eat() { /* ... */ }
}

// клиенту нужен полёт — он принимает Flyable, Penguin туда не попадает
fun makeAllFly(birds: List<Flyable>) = birds.forEach { it.fly() }

Объяснение: LSP-проблема — это сигнал, что иерархия выбрана неверно. «Умеет летать» — это can-do (роль/интерфейс), а не is-a Bird. После разделения подстановка больше не ломается: клиент, которому нужен полёт, работает с List<Flyable>, и нелетающий Penguin в этот контракт просто не входит — компилятор не даст передать его туда.
