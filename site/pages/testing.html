Тестирование

Пирамида тестов, unit-тесты (модульные тесты) на JUnit (фреймворк тестирования) и MockK (библиотека моков), тестирование корутин через kotlinx-coroutines-test, Flow (поток данных Kotlin) через Turbine, UI-тесты (тесты пользовательского интерфейса) на Espresso и Compose, скриншот-тесты и борьба с flaky-тестами (нестабильными тестами). На Middle+/Senior (уверенный/старший уровень) ждут не «знаю про JUnit», а понимание что и на каком уровне тестировать и почему.

Middle Senior Частый вопрос

Fakes vs mocks: тестируемость через DI

Чтобы изолировать класс, его зависимости подменяют тест-дублёрами. Два главных подхода:

  • Mock — объект с заданным поведением и проверкой взаимодействий (verify: «метод вызван 1 раз с такими аргументами»). Создаётся библиотекой (MockK/Mockito).
  • Fake — рабочая лёгкая реализация интерфейса (например, in-memory репозиторий). Ведёт себя «как настоящий», но без сети/БД.
💡
Гугл рекомендует fakes. Для своих абстракций (репозитории, data source) предпочитайте fakes — тесты проверяют состояние/результат (state verification), а не последовательность вызовов, и не ломаются при рефакторинге внутренностей. Mocks хороши для проверки побочных эффектов (логирование, аналитика) и для внешних типов, которые сложно подделать.

Всё это работает только если зависимости внедряются (constructor injection), а не создаются внутри (new/object). Поэтому DI — фундамент тестируемости: в тесте конструктор получает fake/mock вместо боевой реализации.

Тестируемый дизайн через конструктор
// ✅ зависимости снаружи — в тесте подменим
class ProfileViewModel(
    private val repo: ProfileRepository,
    private val dispatcher: CoroutineDispatcher = Dispatchers.IO
) : ViewModel()

// ❌ зависимость «зашита» внутри — не подменить, тест полезет в сеть
class BadViewModel : ViewModel() {
    private val repo = ProfileRepositoryImpl(RetrofitClient.create())
}
Google в гайдах давно отошёл от слова «пирамида» и рисует тестовую матрицу. Что не так с пирамидой как метафорой? ★

Пирамида по уровням (unit / integration / UI) сваливает в одну ось две независимые вещи: scope теста — сколько кода он трогает (одна функция, модуль, всё приложение) — и его fidelity, то есть где он бежит (мок на JVM против реального девайса). Из-за этого слипания «integration» превращается в размытую категорию, и команда бесконечно спорит, куда положить конкретный тест.

Свежий гайд (testing fundamentals, Now in Android) описывает тесты по двум осям сразу — subject under test от мелкого к крупному и среда исполнения от local JVM к instrumented — и советует брать самый дешёвый уровень, который ещё даёт нужную уверенность. Сама идея «много дешёвого внизу» жива, но проценты 70/20/10 — ориентир под конкретный проект, а не догма.

Unit-тесты: JUnit, MockK vs Mockito

База — JUnit. JUnit4 (@Test, @Before, @Rule, @RunWith) до сих пор стандарт в Android, т.к. инструментальные раннеры и многие правила завязаны на него. JUnit5 (Jupiter) — модульный, с @BeforeEach, @Nested, @ParameterizedTest, расширениями вместо правил; на JVM-тестах удобен, но на androidTest нужен мост.

MockK — Kotlin-first библиотека моков

MockK понимает Kotlin: мокает final-классы (в Kotlin всё final по умолчанию), object, расширения, и главное — suspend-функции через coEvery/coVerify. Mockito этого «из коробки» не умел (нужен mockito-kotlin + mockito-inline), поэтому в Kotlin-проектах MockK — выбор по умолчанию.

ВозможностьMockKMockito (+ mockito-kotlin)

suspend-функцииcoEvery / coVerifyобходные пути
object / static / extensionmockkObject, mockkStaticограниченно

Ключевые конструкции MockK:

  • mockk<Type>() — строгий мок (вызов незастабленного метода → ошибка); relaxed = true — возвращает дефолты для всех методов; relaxUnitFun = true — релаксит только Unit-функции.
  • every { mock.foo() } returns x — стаб обычной функции; coEvery { mock.load() } returns y — стаб suspend.
  • verify { ... } / coVerify { ... } — проверка вызовов (exactly = 1, wasNot Called, порядок через verifyOrder).
  • slot<T>() + capture(slot) — захват аргумента, переданного в мок, для последующих ассертов (slot.captured).
RepositoryTest.kt — MockK: every / coEvery / verify / slot
class AnalyticsLoggerTest {

    private val tracker = mockk<Tracker>(relaxed = true) // не стабим каждый метод
    private val repo = mockk<UserRepository>()
    private val logger = AnalyticsLogger(repo, tracker)

    @Test
    fun `logs event with captured user id`() = runTest {
        // Arrange: coEvery — мок suspend-функции
        coEvery { repo.user("42") } returns User(id = "42", name = "Ann")

        val slot = slot<String>()                 // ловушка для аргумента
        every { tracker.send(capture(slot)) } just Runs

        // Act
        logger.openProfile("42")

        // Assert: состояние слота + факт вызова
        assertEquals("profile_opened:42", slot.captured)
        coVerify(exactly = 1) { repo.user("42") }
        verify(exactly = 1) { tracker.send(any()) }
    }
}

LiveData: правило InstantTaskExecutorRule

Architecture Components (LiveData) по умолчанию постят значения через главный поток и ArchTaskExecutor. В JVM-тесте Looper'а нет, поэтому без правила setValue/наблюдатели не отработают синхронно. InstantTaskExecutorRule заставляет все фоновые задачи Architecture Components выполняться синхронно в том же потоке — наблюдатель получает значение немедленно.

ViewModelLiveDataTest.kt
class CounterViewModelTest {
    // выполняет задачи Arch Components синхронно — иначе LiveData молчит в JVM-тесте
    @get:Rule
    val instantRule = InstantTaskExecutorRule()

    @Test
    fun `increment updates liveData`() {
        val vm = CounterViewModel()
        vm.increment()
        assertEquals(1, vm.count.getOrAwaitValue()) // helper читает текущее значение
    }
}
⚠️
Не наблюдайте без жизненного цикла. В тестах LiveData используют observeForever (или helper getOrAwaitValue() из гайдов Google), потому что нет LifecycleOwner. Не забывайте removeObserver, чтобы не текли коллбеки между тестами.

Почему suspend-функцию нельзя застабить обычным every { } returns, и что MockK делает под капотом для coEvery? ★

После компиляции suspend-функция получает скрытый последний параметр Continuation и возвращает не объявленный тип, а Any? — либо реальное значение, либо маркер COROUTINE_SUSPENDED. Лямбда обычного every { } не помечена suspend, поэтому внутри неё suspend-функцию просто не вызвать — код не скомпилируется.

coEvery/coVerify — это suspend-обёртки над тем же механизмом записи вызовов. Их блок объявлен suspend, и MockK исполняет его в собственном корутинном контексте (через runBlocking), перехватывает вызов с учётом Continuation и подставляет результат. То есть запись вызова одна и та же — co-версии лишь умеют дёрнуть suspend-функцию внутри блока.

Тестирование корутин: kotlinx-coroutines-test

Корутины асинхронны, а тест должен быть детерминированным. Библиотека kotlinx-coroutines-test даёт виртуальное время: delay не ждёт реально, а двигает виртуальные часы планировщика. Точка входа — runTest: запускает тело как корутину, по завершении проверяет, что не осталось «висящих» (не дождавшихся) корутин и что не было необработанных исключений.

Управление виртуальным временем (есть на TestScope внутри runTest):

  • advanceUntilIdle() — прокрутить время, пока есть запланированные задачи (выполнить всё, что в очереди).
  • advanceTimeBy(ms) — прокрутить на конкретный интервал (полезно тестировать таймауты/debounce).
  • runCurrent() — выполнить только задачи, запланированные на текущий момент.
  • testScheduler.currentTime — текущее виртуальное время.

Подмена Dispatchers.Main

На JVM нет Android Main Looper'а, поэтому viewModelScope (он на Dispatchers.Main.immediate) упадёт. Решение — глобально подменить Main на TestDispatcher через Dispatchers.setMain() в @Before и вернуть через Dispatchers.resetMain() в @After. Удобно вынести в JUnit-правило.

MainDispatcherRule.kt
@OptIn(ExperimentalCoroutinesApi::class)
class MainDispatcherRule(
    val dispatcher: TestDispatcher = StandardTestDispatcher()
) : TestWatcher() {
    override fun starting(d: Description) = Dispatchers.setMain(dispatcher)
    override fun finished(d: Description) = Dispatchers.resetMain()
}
💡
Один планировщик на тест. Передавайте в setMain тот же TestDispatcher, что инжектите в код. Тогда у runTest и боевых корутин общий TestCoroutineScheduler — виртуальное время синхронно, а advanceUntilIdle() двигает все корутины.

runTest со Standard откладывает дочерние корутины — почему же тело самого теста бежит сразу? И что будет, если корутина внутри зависнет навсегда? ★

Тело runTest исполняется на UnconfinedTestDispatcher, поэтому идёт eagerly до первой точки приостановки. Отсюда асимметрия: сам блок теста бежит немедленно, а вот корутины, которые вы стартуете внутри через launch/async (они наследуют StandardTestDispatcher из TestScope), уходят в очередь планировщика и ждут advanceUntilIdle()/runCurrent().

Зависнуть навсегда runTest не даст: у него есть таймаут по реальному времени — по умолчанию 60 секунд, настраивается параметром timeout. По истечении тест падает с UncompletedCoroutinesError и дампом висящих корутин. А для фона, который не должен блокировать финал (сбор StateFlow, бесконечный продюсер), есть backgroundScope: он на виртуальном времени, но его завершения runTest не ждёт.

Тестирование Flow: Turbine и StateFlow

Flow асинхронен и может быть бесконечным (особенно StateFlow/SharedFlow), поэтому собрать его в список через toList() не выйдет — он не завершится. Ручной сбор в launch с буфером многословен и хрупок. Turbine (Cash App) даёт лаконичный DSL: flow.test { ... } запускает сбор, а внутри вы по одному снимаете эмиссии.

  • awaitItem() — дождаться и вернуть следующий элемент (снимает его из очереди).
  • awaitComplete() / awaitError() — дождаться завершения/ошибки потока.
  • expectMostRecentItem() — взять самое свежее значение, отбросив промежуточные (важно при конфляции StateFlow).
  • cancelAndIgnoreRemainingEvents() — закрыть подписку, если хвост не интересует (нужно для бесконечных потоков).
  • ensureAllEventsConsumed() — гарантия, что не осталось непроверенных эмиссий.
SearchViewModelTest.kt — Turbine + StateFlow
@OptIn(ExperimentalCoroutinesApi::class)
class SearchViewModelTest {

    @get:Rule val mainRule = MainDispatcherRule()
    private val repo = mockk<SearchRepository>()

    @Test
    fun `emits Loading then Content`() = runTest {
        coEvery { repo.search("kt") } returns listOf("Kotlin")
        val vm = SearchViewModel(repo)

        vm.uiState.test {
            assertEquals(UiState.Idle, awaitItem())   // начальное значение StateFlow
            vm.onQuery("kt")
            assertEquals(UiState.Loading, awaitItem())
            assertEquals(UiState.Content(listOf("Kotlin")), awaitItem())
            cancelAndIgnoreRemainingEvents()          // StateFlow бесконечен
        }
    }
}
⚠️
Конфляция StateFlow. StateFlow может «склеить» быстрые подряд идущие значения — промежуточное состояние Turbine не увидит, и тест на строгую последовательность станет flaky. Если важна именно цепочка состояний, иногда проще ассертить итог через .value или expectMostRecentItem(); Google рекомендует относиться к StateFlow как к держателю состояния и проверять текущее значение.
Тест StateFlow через Turbine иногда пропускает Loading, хотя код «работает». В чём причина и как сделать тест детерминированным? ★

StateFlow conflated и хранит только последнее значение. Если несколько emit/setValue успели пройти до того, как Turbine сделал первый awaitItem(), промежуточные состояния схлопнутся, и коллектор увидит лишь самое свежее. На UnconfinedTestDispatcher продюсер легко проскакивает Idle → Loading → Content ещё до подписки — отсюда «то ловит, то нет».

Лечение: подписаться (запустить flow.test { } и снять первый awaitItem() на начальном значении) до триггера; либо взять StandardTestDispatcher для пошагового контроля; либо вообще не проверять промежуточную цепочку на StateFlow, а ассертить терминальное .value / expectMostRecentItem(). Если последовательность состояний — это контракт, моделируйте её обычным cold Flow или SharedFlow без конфляции, а событийный поток — через Channel/SharedFlow с replay=0.

Instrumented / UI: Espresso, Compose, UiAutomator

Эти тесты лежат в androidTest/ и бегут на устройстве/эмуляторе через AndroidJUnitRunner. Они проверяют реальный UI, поэтому медленнее и более хрупки — держите их немного, на ключевые сценарии.

Espresso (классические View)

Три кита Espresso: ViewMatchers (найти view — withId, withText), ViewActions (действие — click(), typeText()), ViewAssertions (проверка — matches(isDisplayed())). Главная фишка — автосинхронизация: Espresso сам ждёт, пока главный поток освободится и очередь сообщений опустеет, прежде чем выполнить действие.

LoginScreenTest.kt (Espresso)
@RunWith(AndroidJUnit4::class)
class LoginScreenTest {
    @get:Rule val rule = ActivityScenarioRule(LoginActivity::class.java)

    @Test fun emptyPassword_showsError() {
        onView(withId(R.id.email)).perform(typeText("a@b.c"), closeSoftKeyboard())
        onView(withId(R.id.loginBtn)).perform(click())
        onView(withId(R.id.error))
            .check(matches(withText("Введите пароль")))
    }
}

Idling Resources. Espresso «видит» только главный поток. Для фоновых задач (свой пул потоков, не-корутинный async) он не знает, когда они закончились, и тест мигает. IdlingResource — мост: вы сообщаете Espresso «ресурс занят/свободен», и он ждёт. На современном стеке с корутинами это нужно редко, но классический вопрос на собесе.

Compose UI testing

Compose тестируется через семантическое дерево (semantics), а не через view-иерархию. Правило createComposeRule() (или createAndroidComposeRule<Activity>()) задаёт контент и даёт API «найти узел → действие/проверка»: onNodeWithText, onNodeWithTag (по Modifier.testTag), onNodeWithContentDescription; ассерты assertIsDisplayed(), assertTextEquals(); действия performClick(), performTextInput(). Compose синхронизирует тест с рекомпозицией автоматически (waitForIdle под капотом).

CounterScreenTest.kt (Compose)
class CounterScreenTest {
    @get:Rule val composeRule = createComposeRule()

    @Test fun click_increments_counter() {
        composeRule.setContent { CounterScreen() }

        composeRule.onNodeWithText("0").assertIsDisplayed()
        composeRule.onNodeWithTag("inc").performClick()   // Modifier.testTag("inc")
        composeRule.onNodeWithText("1").assertIsDisplayed()
    }
}
💡
testTag для устойчивости. Привязка к видимому тексту ломается при локализации/правках копирайта. Для стабильных селекторов вешайте Modifier.testTag("…") и ищите через onNodeWithTag. Текстовые матчеры оставляйте там, где проверяете именно контент.

UiAutomator, Robolectric, screenshot-тесты

  • UiAutomator — чёрный ящик на уровне всей системы: умеет взаимодействовать с другими приложениями, шторкой, настройками. Для кросс-приложенческих E2E и системных диалогов (разрешения), куда Espresso не дотягивается.
  • Robolectric — запускает Android-зависимый код прямо на JVM без эмулятора (через «тени» — shadows). Те же тесты могут идти быстрее без девайса; удобно для кода, что трогает Context/ресурсы, но не нуждается в реальном рендере.
  • Screenshot-тесты — фиксируют эталонный рендер и сравнивают пиксели при каждом прогоне, ловя визуальные регрессии. Paparazzi (Cash App) рендерит Compose/View на JVM без устройства; Roborazzi делает скриншоты поверх Robolectric. Плюс — официальный Compose Preview Screenshot Testing.

Что Compose-тест синхронизирует сам, а что нет

Compose ждёт свои механизмы автоматически, но всё, что крутится мимо его графа, для теста невидимо — и тут нужен registerIdlingResource на ComposeTestRule (Compose поддерживает IdlingResource так же, как Espresso).

Compose дождётся самНужен IdlingResource / waitUntil
РекомпозицияФоновый поток вне Compose
Анимации на withFrameNanosКорутина на Dispatchers.IO
Snapshot-stateСторонняя очередь, callback API
Корутины на RecomposerWorkManager, внешний таймер
Compose-тест синхронизируется с рекомпозицией сам. Какой класс асинхронной работы он НЕ видит, и чем mainClock/waitForIdle отличаются от waitUntil? ★

Compose синхронизируется со своими механизмами: рекомпозицией, анимациями на withFrameNanos, корутинами на Recomposer. Работу вне этого графа — фоновый поток, не-Compose корутину на Dispatchers.IO, стороннюю очередь — он не отслеживает, и тут нужен registerIdlingResource.

У теста есть виртуальные часы composeTestRule.mainClock. По умолчанию autoAdvance=true: rule сам проматывает кадры и анимации до idle на каждом действии, а waitForIdle() ждёт, пока нет ожидающих рекомпозиций и анимаций. Выставив mainClock.autoAdvance=false, можно вручную двигать advanceTimeBy и проверять промежуточные кадры анимации. Разница в том, что waitForIdle ждёт «успокоения» самого Compose, а waitUntil(condition) крутит цикл, пока ваше произвольное условие не станет истинным — для работы, которую rule сам не дождётся.

Хорошие практики и пример ViewModel-теста

AAA: Arrange-Act-Assert

Структурируйте тело теста в три блока: Arrange (подготовка моков/входных данных), Act (одно действие — то, что проверяем), Assert (проверка результата/состояния). Один тест — одна проверяемая мысль; имя теста описывает поведение (`returns error when network fails`).

flowchart LR A["Arrange
моки, входные данные"] --> B["Act
один вызов SUT"] B --> C["Assert
состояние / результат"] C --> D{Детерминирован?} D -->|нет| E["Чинить: время, потоки,
порядок, рандом"] D -->|да| F["Зелёный, стабильный"]
AAA + проверка детерминизма как защита от flaky

Борьба с flaky-тестами

Flaky — тест, который то проходит, то падает без изменений кода. Главные причины и лечение:

Причина flakyЛечение
Реальное время / Thread.sleep / таймаутыВиртуальное время (advanceUntilIdle), убрать sleep
Гонки потоков, Dispatchers.MainОдин TestDispatcher + setMain, инжект диспетчеров
Зависимость от порядка тестов / общий стейтИзолировать: новый объект в @Before, clearAllMocks()
Сеть/БД/файлы (внешний мир)Fakes, MockWebServer, Room in-memory
Недетерминизм UI (анимации, асинхрон)Idling Resources, отключить анимации, testTag
Случайность / текущая датаИнжектить Random(seed), Clock

Что и на каком уровне тестировать

  • Логика (UseCase, Mapper, ViewModel, валидаторы) → unit, максимально много.
  • Интеграция (Repository ↔ Room ↔ сеть) → integration с in-memory БД и MockWebServer.
  • Ключевые пользовательские пути (логин, оплата) → немного UI/E2E.
  • Не тестируйте геттеры/сеттеры, фреймворк и тривиальный код — это шум.
ℹ️
Покрытие — ориентир, не цель. JaCoCo/Kover показывают % покрытых строк, но 100% покрытия не равно отсутствию багов: можно «прошить» код тестом без единого ассерта. Смотрите на покрытие веток и осмысленность проверок. На CI гоняйте unit на каждый PR (быстро), instrumented — на ночных/мердж-прогонах.

Полный пример: ViewModel с корутинами и моками

Собираем всё вместе: MainDispatcherRule, runTest, MockK (coEvery), Turbine и проверка состояний StateFlow через AAA.

ProfileViewModelTest.kt
@OptIn(ExperimentalCoroutinesApi::class)
class ProfileViewModelTest {

    @get:Rule val mainRule = MainDispatcherRule()        // подмена Main на TestDispatcher
    private val repo = mockk<ProfileRepository>()
    private lateinit var vm: ProfileViewModel

    @Before fun setUp() { vm = ProfileViewModel(repo) }
    @After  fun tearDown() { clearAllMocks() }           // нет утечки стабов между тестами

    @Test
    fun `load success - Loading then Content`() = runTest {
        // Arrange
        coEvery { repo.profile("42") } returns Profile("42", "Ann")

        // Act + Assert через Turbine
        vm.state.test {
            assertEquals(ProfileState.Idle, awaitItem())
            vm.load("42")
            assertEquals(ProfileState.Loading, awaitItem())
            advanceUntilIdle()                            // промотать корутину загрузки
            assertEquals(ProfileState.Content("Ann"), awaitItem())
            cancelAndIgnoreRemainingEvents()
        }
        coVerify(exactly = 1) { repo.profile("42") }
    }

    @Test
    fun `load failure - emits Error`() = runTest {
        // Arrange: репозиторий бросает
        coEvery { repo.profile(any()) } throws IOException("offline")

        // Act
        vm.load("42")
        advanceUntilIdle()

        // Assert: проверяем итоговое состояние держателя
        assertEquals(ProfileState.Error("offline"), vm.state.value)
    }
}
Что такое тавтологичный тест и почему verify над мокнутым методом — запах? Когда verify оправдан, а когда нет? ★

Тавтологичный тест — это когда вы мокаете зависимость, заставляете её вернуть X, а потом через verify проверяете, что код вызвал ровно то, что вы сами и запрограммировали. Проверяется не поведение, а ваша же настройка мока, и тест повторяет реализацию один в один: поменяли внутренний вызов — тест упал, хотя поведение не изменилось. Это brittle test.

verify оправдан только для побочных эффектов без наблюдаемого результата-состояния: отправка аналитики, запись в лог, инвалидация кэша, отправка письма — там факт вызова и есть единственное доказательство эффекта. Всё, что возвращает значение или меняет состояние (репозиторий отдал данные → ViewModel выставил UiState), проверяйте через результат (state verification) на fake, а не через verify. Правило простое: один verify на команду-эффект; если он дублирует ассерт состояния — он лишний.

Тестирование событий: SharedFlow и Channel

Глава выше разбирала StateFlow — держатель состояния. Но навигация, снэкбары, тосты — это события, а не состояние, и тестируются принципиально иначе. Событие должно сработать ровно один раз: показать StateFlow для one-shot плохо, потому что при повороте экрана коллектор переподпишется и получит последнее значение заново — снэкбар выскочит второй раз. Поэтому события возят через SharedFlow с replay=0 или Channel.

replay=0: подписка должна идти до эмиссии

SharedFlow с replay=0 горячий и не хранит историю. Подписчик, пришедший после emit, событие просто не увидит. В тесте это значит: сбор надо запустить до триггера. Через Turbine flow.test { } подписывается синхронно внутри блока, либо вынесите коллектор в backgroundScope и дайте ему реально подписаться через runCurrent() до вызова действия.

NavEventsTest.kt — SharedFlow replay=0
@Test
fun `onSave success emits Back`() = runTest {
    coEvery { repo.save() } returns Unit
    val vm = EditViewModel(repo)

    val received = mutableListOf<NavEvent>()
    backgroundScope.launch { vm.events.toList(received) } // подписались...
    runCurrent()                                          // ...и дали подписке сработать

    vm.onSave()                                           // ...только теперь триггер
    advanceUntilIdle()

    assertEquals(NavEvent.Back, received.single())
}
⚠️
Порядок «подписался → событие». Если эмитить до того, как коллектор реально встал в очередь планировщика, событие потеряется и тест станет flaky. На StandardTestDispatcher между launch { collect } и эмиссией всегда вставляйте runCurrent() — иначе подписка ещё не активна.

Channel + receiveAsFlow

Альтернатива — Channel (обычно Channel.BUFFERED), выставленный наружу как receiveAsFlow(). У него другая семантика доставки: событие буферизуется и достаётся одному коллектору ровно один раз, даже если подписка пришла чуть позже эмиссии. Это удобнее для строго one-shot событий, но Channel по своей природе single-consumer — два экрана на один поток не подпишутся. В тесте его собирают так же: через Turbine или коллектор в backgroundScope.

Почему конфляция у SharedFlow с replay=0 не работает как у StateFlow, и какой это даёт риск в тесте? ★

StateFlow всегда держит одно последнее значение и схлопывает промежуточные — потерять можно «середину». У SharedFlow с replay=0 и достаточным extraBufferCapacity конфляции нет: каждое событие доставляется отдельно. Зато появляется другой риск — гонка подписки: пока коллектор не подписался, эмиссия уходит в никуда (нет ни буфера replay, ни ожидания). В тесте это лечится порядком «сначала запустить сбор и дать ему подписаться (runCurrent()), потом эмитить». То есть у StateFlow вы боретесь с конфляцией, а у событийного SharedFlow — с порядком подписки.

Как измерить качество тестов: мутационное тестирование

Покрытие (JaCoCo / Kover) считает, какие строки и ветки исполнились во время прогона. Но исполнение — не проверка: тест, который дёргает код без единого ассерта, даёт 100% покрытия и ловит ноль багов. Нужен инструмент, который меряет не охват, а силу самих утверждений.

Killed vs survived: мутационный score

Мутационное тестирование (PIT / Pitest) намеренно вносит мелкие правки в продакшн-код — мутации: + на -, > на >=, выкинуть вызов, инвертировать условие — и прогоняет тесты на каждой. Если тесты упали, мутацию «убили»: тест эффективен. Если прошли — мутация «выжила»: тест слеп к этому изменению. Mutation score = доля убитых мутаций. Это прямая мера качества ассертов, а не строк. Цена — медленный прогон (тесты гоняются на каждую мутацию), поэтому мутационку применяют точечно к критичной логике.

МетрикаЧто реально меряетСлепое пятно
Покрытие строкКакие строки исполнилисьИсполнение ≠ проверка
Покрытие ветвейОбе ветки if пройденыВетка пройдена без ассерта
Mutation scoreЛовят ли ассерты реальные измененияМедленно; не для всего кода

Анти-паттерны плохого теста

  • Тавтологичный тест — verify своей же настройки мока; проверяет конфигурацию, а не поведение.
  • Тест без ассерта — «прошивает» код ради покрытия, ничего не утверждает.
  • Тест на приватную реализацию — лезет во внутренности, падает при любом рефакторинге.
  • Over-mocking — мокнуто всё подряд; тест дублирует код один в один.

Хороший тест проверяет поведение и контракт, переживает рефакторинг внутренностей, несёт один смысл и читается как спецификация: «при ошибке сети показываем Error» — а не «вызван метод X».

Почему 100% покрытия не значит «нет багов», но и мутационный score не делают целью на весь код? ★

Покрытие показывает исполнение, а баг — это неверная логика, которую тест должен поймать ассертом. Можно прогнать каждую строку и ничего не утверждать — отсюда зелёные 100% при дырявых тестах. Мутационный score честнее: он проверяет, замечают ли ассерты подмену логики.

Но гнать мутационку на весь код невыгодно: каждая мутация — отдельный полный прогон тестов, время растёт мультипликативно. Поэтому её включают точечно — на ядро бизнес-логики, расчёты, критичные ветвления, — а на тривиальном коде (DTO, маппинги) держат обычное покрытие веток. И score тоже не самоцель: 100% убитых мутаций на геттерах никому не нужны.

CI/CD для Android

На Senior/Lead спрашивают не «знаю ли я JUnit», а как держать тесты быстрыми и стабильными в конвейере. Типовой pipeline идёт от дешёвого к дорогому: lint (статический анализ, detekt/ktlint) → unit на JVM → instrumented на эмуляторе → подпись и дистрибуция. Каждый шаг — гейт: упал ранний и дешёвый, до дорогого instrumented дело не доходит. Сборкой и подписью подробно занимается глава Gradle и сборка; здесь — про тестовую часть конвейера.

на эмуляторе"] I --> S["подпись AAB"] S --> D["дистрибуция"]
Pipeline: дешёвые быстрые гейты первыми, instrumented и дистрибуция — в конце

Изоляция и параллелизм instrumented-тестов

  • AndroidX Test Orchestrator запускает каждый instrumented-тест в своём процессе Instrumentation. Общий стейт и синглтоны не текут между тестами, краш одного теста не валит весь класс, можно делать clearPackageData между прогонами. Цена — рестарт процесса на каждый тест, то есть медленнее.
  • Sharding — разбивка набора по нескольким эмуляторам/раннерам (numShards/shardIndex) для параллелизма. На практике поверх этого берут Gradle Managed Devices или Firebase Test Lab.
  • Screenshot-тесты в CI работают в двух режимах: record (записать эталоны) и verify (сравнить с эталоном). Эталоны хранят в репозитории; больная точка — рендер на разных API и плотностях, поэтому фиксируют конкретное окружение.

Кэш Gradle и флейки

В CI критичен кэш Gradle (build cache + кэш зависимостей) — без него каждый прогон пересобирает всё с нуля и конвейер ползёт. Флейки в instrumented — отдельная боль: политика повторов (retry) маскирует симптом, но не лечит причину, и допустима разве что как временный буфер на инфраструктурную нестабильность эмулятора. Где гонять что: unit на каждый PR (быстро, дёшево), instrumented — на merge или ночных прогонах (дорого, медленно).

🛑
Retry — пластырь, а не лечение. Автоповтор упавшего теста скрывает флейк, и команда перестаёт ему доверять: зелёный прогон с тремя ретраями ничего не гарантирует. Сначала найдите источник недетерминизма (время, гонки, общий стейт), а retry оставьте только для заведомо инфраструктурных сбоев эмулятора.
Зачем Orchestrator, если sharding и так распараллеливает тесты — они решают одну задачу? ★

Нет, это ортогональные вещи. Sharding отвечает за скорость: режет набор тестов на части и гоняет их на нескольких эмуляторах параллельно. Orchestrator отвечает за изоляцию: запускает каждый тест в отдельном процессе Instrumentation, чтобы статика, синглтоны и общий стейт не протекали между тестами, а краш одного не уносил остальные.

Их используют вместе: sharding раскидывает тесты по раннерам, а Orchestrator внутри каждого раннера изолирует тесты друг от друга. Без Orchestrator один «грязный» тест в шарде отравит соседей по тому же процессу — и параллелизм лишь быстрее размножит флейки.

Частые вопросы интервьюера

В чём разница между mock, stub и fake? Middle

Stub — возвращает заранее заданные значения (просто «заглушка» ответа). Mock — стаб + проверка взаимодействий (был ли вызван метод, сколько раз, с какими аргументами — behavior verification). Fake — упрощённая, но рабочая реализация интерфейса (например, in-memory репозиторий вместо БД). Гайдлайны Google советуют для собственных абстракций предпочитать fakes и проверять состояние/результат, а mocks приберечь для побочных эффектов и внешних типов.

Почему в Kotlin-проектах берут MockK, а не Mockito? Middle+

MockK задумана под Kotlin: нативно мокает final-классы (а в Kotlin всё final по умолчанию), object, статические методы и расширения, и — главное — suspend-функции через coEvery/coVerify. Mockito исторически Java-ориентирован: для тех же задач нужны mockito-inline и mockito-kotlin, а корутины — обходными путями. Плюс у MockK идиоматичный Kotlin-DSL (every { } returns, slot, relaxed).

Как протестировать ViewModel с корутинами? Зачем подменять Dispatchers.Main? Senior

viewModelScope работает на Dispatchers.Main.immediate, а в JVM-тесте нет Android-Looper'а — корутина упадёт с «Module with the Main dispatcher had failed to initialize». Поэтому в @Before вызывают Dispatchers.setMain(testDispatcher), а в @After — resetMain() (обычно через правило MainDispatcherRule). Тело теста оборачивают в runTest (виртуальное время), при StandardTestDispatcher двигают корутины через advanceUntilIdle(). Важно использовать один TestDispatcher и для setMain, и для инжекта в код — тогда планировщик общий.

StandardTestDispatcher vs UnconfinedTestDispatcher — что выбрать? Senior

StandardTestDispatcher (дефолт runTest) откладывает запуск новых корутин — они встают в очередь планировщика, и вы явно проматываете время (advanceUntilIdle/runCurrent). Это даёт точный контроль и возможность увидеть промежуточные состояния (Loading). UnconfinedTestDispatcher выполняет корутины сразу до первой приостановки — меньше церемоний, ближе к «реальному» поведению, но промежуточные состояния можно проскочить. Standard — когда важна последовательность; Unconfined — для простых случаев «запустил и проверил итог».

Почему StateFlow нельзя протестировать через toList()? Чем помогает Turbine? Middle+

StateFlow бесконечен (всегда есть текущее значение и нет завершения), поэтому toList() никогда не вернётся — тест зависнет. Turbine запускает сбор внутри flow.test { } и позволяет снимать эмиссии по одной через awaitItem(), а в конце закрыть подписку (cancelAndIgnoreRemainingEvents()). Нюанс: StateFlow конфлирует быстрые значения, поэтому строгая проверка цепочки состояний может быть flaky — часто надёжнее ассертить .value или expectMostRecentItem().

Зачем нужен InstantTaskExecutorRule? Middle

LiveData и другие Architecture Components диспетчеризуют работу через ArchTaskExecutor и главный поток. В JVM-тесте главного потока/Looper'а нет, поэтому setValue и уведомления наблюдателей не отрабатывают синхронно — ассерт видит старое значение. InstantTaskExecutorRule заменяет исполнитель на синхронный (всё выполняется немедленно в текущем потоке), и LiveData ведёт себя предсказуемо. Наблюдают в тестах через observeForever (нет LifecycleOwner).

Espresso или Compose-тесты — чем отличаются под капотом? Senior

Espresso работает с view-иерархией: матчит View по id/тексту и синхронизируется с очередью сообщений главного потока (ждёт idle). Compose не имеет View — тесты ходят по семантическому дереву (semantics tree), которое описывает доступность и состояние UI; узлы ищут через onNodeWithText/onNodeWithTag, а синхронизация идёт с циклом рекомпозиции. В гибридных экранах есть интероп: createAndroidComposeRule позволяет смешивать Espresso и Compose-ассерты.

Чем Robolectric отличается от instrumented-тестов и зачем скриншот-тесты? Senior

Robolectric подменяет Android-классы «тенями» (shadows) и исполняет Android-зависимый код прямо на JVM — без эмулятора, поэтому быстрее и стабильнее для логики, трогающей Context/ресурсы; но это не настоящий рендер на реальном устройстве. Instrumented-тесты идут на девайсе/эмуляторе и проверяют поведение «как у пользователя». Скриншот-тесты (Paparazzi на JVM, Roborazzi поверх Robolectric, Compose Preview Screenshot Testing) фиксируют эталонный рендер и ловят визуальные регрессии, которые функциональные тесты не замечают (сдвиги, цвета, отступы, тёмная тема).

Что делать с flaky-тестом? Middle+

Не «ретраить вслепую» и не @Ignore — найти источник недетерминизма. Чаще всего это: реальное время/Thread.sleep (→ виртуальное время), гонки потоков и неподменённый Main (→ единый TestDispatcher), общий стейт между тестами (→ изоляция, clearAllMocks, новый SUT в @Before), обращение к сети/БД (→ fakes/MockWebServer/in-memory Room), случайность и текущая дата (→ инжект Random(seed)/Clock), асинхронный UI и анимации (→ Idling Resources, отключить анимации). Flaky-тест хуже отсутствия теста: он подрывает доверие к зелёному прогону.

Зачем нужен backgroundScope в runTest и чем он отличается от обычного launch внутри теста? Senior

Обычный launch внутри runTest стартует дочернюю корутину в TestScope, и по правилам структурной конкуренции runTest дождётся её завершения перед концом теста. Если корутина бесконечна (сбор StateFlow, продюсер с while(true)), тест провисит до 60-секундного таймаута и упадёт с UncompletedCoroutinesError. backgroundScope — специальный scope внутри runTest: тоже на виртуальном времени, но его завершения runTest не ждёт и автоматически отменяет в конце. В него выносят всё фоновое, что нужно тесту, но не должно блокировать финал: коллекторы StateFlow/SharedFlow, длительные продюсеры. Это идиоматичная замена ручного collectJob + cancel.

Чем spyk отличается от mockk и когда оправдан частичный мок? Middle+

mockk создаёт полный дублёр: методы по умолчанию не реализованы (строгий мок кидает, relaxed возвращает дефолты), реальный код класса не выполняется. spyk оборачивает реальный объект: по умолчанию вызываются настоящие методы, а застабить можно лишь отдельные через every. Spy полезен, когда надо подменить один-два метода у в остальном рабочего объекта — например, замокать только обращение к сети у класса с кучей чистой логики. Но spy — запах: обычно он сигналит, что класс делает слишком много и его пора разбить или вынести зависимость в конструктор. На практике предпочитают fake целиком или mock интерфейса, а spy держат для legacy, который нельзя переписать.

Как тестировать код, который читает текущее время или генерирует случайность, не делая тест flaky? Middle+

Нельзя завязываться на System.currentTimeMillis()/LocalDateTime.now()/Random() напрямую — результат недетерминирован. Время инжектят через java.time.Clock: в проде Clock.systemDefaultZone(), в тесте Clock.fixed(instant, zone), а код берёт время как clock.instant(). Случайность — через инжектируемый Random(seed) с фиксированным семенем: последовательность повторяема. Генераторы UUID/ID выносят за интерфейс и подменяют fake-ом. Это частный случай общего принципа: любой недетерминированный источник (время, рандом, локаль, файловая система) — внешняя зависимость, которую внедряют, а не дёргают статически.

Задачи

Задача 1. Напишите unit-тест для функции login(email, password) в LoginViewModel. Функция вызывает suspend-метод AuthRepository.login(), при успехе переводит StateFlow uiState в Success, при IOException — в Error("offline"). Тест должен проверить оба сценария, использовать MockK для мокирования репозитория и Turbine для проверки последовательности состояний.

Показать решение 
@OptIn(ExperimentalCoroutinesApi::class)
class LoginViewModelTest {

    @get:Rule val mainRule = MainDispatcherRule()
    private val repo = mockk<AuthRepository>()
    private lateinit var vm: LoginViewModel

    @Before fun setUp() { vm = LoginViewModel(repo) }
    @After  fun tearDown() { clearAllMocks() }

    @Test
    fun `login success - Loading then Success`() = runTest {
        coEvery { repo.login("a@b.c", "pass") } returns User(id = "1")

        vm.uiState.test {
            assertEquals(LoginState.Idle, awaitItem())
            vm.login("a@b.c", "pass")
            assertEquals(LoginState.Loading, awaitItem())
            advanceUntilIdle()
            assertEquals(LoginState.Success, awaitItem())
            cancelAndIgnoreRemainingEvents()
        }
        coVerify(exactly = 1) { repo.login("a@b.c", "pass") }
    }

    @Test
    fun `login failure - emits Error`() = runTest {
        coEvery { repo.login(any(), any()) } throws IOException("offline")

        vm.login("a@b.c", "pass")
        advanceUntilIdle()

        assertEquals(LoginState.Error("offline"), vm.uiState.value)
    }
}

Объяснение: MainDispatcherRule подменяет Dispatchers.Main на TestDispatcher — без этого viewModelScope упадёт в JVM-тесте. advanceUntilIdle() прокручивает виртуальное время и позволяет suspend-вызову завершиться. Turbine фиксирует промежуточное состояние Loading — через прямой ассерт vm.uiState.value его легко проскочить при конфляции StateFlow. clearAllMocks() в @After изолирует стабы между тестами.

Задача 2. Дан SearchViewModel: поле query: MutableStateFlow<String> преобразуется через debounce(300) + flatMapLatest { repo.search(it) } в results: StateFlow<List<String>>. Напишите тест, который проверяет, что при вводе двух запросов подряд с интервалом менее 300 мс репозиторий вызывается ровно один раз — с последним запросом. Используйте advanceTimeBy.

Показать решение 
@OptIn(ExperimentalCoroutinesApi::class)
class SearchViewModelDebounceTest {

    @get:Rule val mainRule = MainDispatcherRule()
    private val repo = mockk<SearchRepository>()

    @Test
    fun `debounce - repo called once with last query`() = runTest {
        coEvery { repo.search(any()) } returns emptyList()
        val vm = SearchViewModel(repo)

        vm.query.value = "ko"
        advanceTimeBy(100)          // 100 мс — меньше debounce, запрос ещё не ушёл
        vm.query.value = "kotlin"
        advanceTimeBy(301)          // перешагнуть debounce (300 мс) для последнего запроса

        coVerify(exactly = 0) { repo.search("ko") }
        coVerify(exactly = 1) { repo.search("kotlin") }
    }
}

Объяснение: advanceTimeBy(ms) двигает виртуальные часы TestCoroutineScheduler без реального ожидания. После 100 мс debounce-окно ещё не истекло — первый запрос «ko» отменяется оператором flatMapLatest, когда появляется «kotlin». После 301 мс debounce завершается, и только второй запрос доходит до репозитория. coVerify(exactly = 0) гарантирует отсутствие лишнего сетевого вызова.

Задача 3. Напишите Compose UI-тест для экрана CounterScreen: экран показывает число (начальное значение 0) и кнопку с testTag("increment"). После трёх нажатий на кнопку на экране должно отображаться число «3». Тест должен использовать createComposeRule и проверить состояние через семантическое дерево без обращения к ViewModel напрямую.

Показать решение 
class CounterScreenTest {

    @get:Rule val composeRule = createComposeRule()

    @Test
    fun `three clicks - counter shows 3`() {
        composeRule.setContent { CounterScreen() }

        // начальное состояние
        composeRule.onNodeWithText("0").assertIsDisplayed()

        // три нажатия
        repeat(3) {
            composeRule.onNodeWithTag("increment").performClick()
        }

        // итоговое состояние
        composeRule.onNodeWithText("3").assertIsDisplayed()
        composeRule.onNodeWithText("0").assertDoesNotExist()
    }
}

Объяснение: Compose-тесты ходят по семантическому дереву, а не по view-иерархии, поэтому onNodeWithTag устойчив к изменениям текста. createComposeRule автоматически синхронизируется с рекомпозицией после каждого performClick() — дополнительного waitForIdle() не нужно. assertDoesNotExist() на «0» исключает ситуацию, когда оба текста случайно видны одновременно (например, при анимации).

Задача 4. ViewModel шлёт навигацию через событийный канал events: SharedFlow<NavEvent> (это MutableSharedFlow с replay=0). Метод onSave() при успехе репозитория эмитит NavEvent.Back. Напишите тест на успешный сценарий так, чтобы не потерять событие из-за replay=0, и объясните, почему подписка обязана идти до вызова onSave().

Показать решение 
@OptIn(ExperimentalCoroutinesApi::class)
class NavEventsTest {

    @get:Rule val mainRule = MainDispatcherRule()
    private val repo = mockk<EditRepository>()

    @Test
    fun `onSave success emits Back`() = runTest {
        coEvery { repo.save() } returns Unit
        val vm = EditViewModel(repo)

        // вариант через backgroundScope: подписались ДО триггера
        val received = mutableListOf<NavEvent>()
        backgroundScope.launch { vm.events.toList(received) }
        runCurrent()                      // дать коллектору реально подписаться

        vm.onSave()                       // только теперь эмитим
        advanceUntilIdle()

        assertEquals(NavEvent.Back, received.single())
    }

    // Альтернатива через Turbine — test{} подписывается синхронно:
    @Test
    fun `onSave success emits Back via turbine`() = runTest {
        coEvery { repo.save() } returns Unit
        val vm = EditViewModel(repo)

        vm.events.test {
            vm.onSave()
            advanceUntilIdle()
            assertEquals(NavEvent.Back, awaitItem())
            cancelAndIgnoreRemainingEvents()
        }
    }
}

Объяснение: при replay=0 подписчик, пришедший после emit, событие не увидит — буфера истории нет. Поэтому коллектор (backgroundScope.launch { collect } или Turbine test{}) должен стартовать до триггера. С StandardTestDispatcher между запуском сбора и эмиссией нужен runCurrent(), иначе подписка ещё не активна и событие уйдёт в никуда. backgroundScope здесь удобен тем, что бесконечный toList не заблокирует завершение теста — его отменят автоматически.

Задача 5. Напишите свой минимальный MainDispatcherRule на базе TestWatcher: в starting() — Dispatchers.setMain(dispatcher), в finished() — resetMain(), с параметром TestDispatcher (по умолчанию UnconfinedTestDispatcher). Затем покажите тест ViewModel, чей init-блок грузит данные через viewModelScope и выставляет state в Content без ручного advanceUntilIdle. Объясните, почему с Unconfined advance не нужен, а со Standard — нужен.

Показать решение 
@OptIn(ExperimentalCoroutinesApi::class)
class MainDispatcherRule(
    val dispatcher: TestDispatcher = UnconfinedTestDispatcher()
) : TestWatcher() {
    override fun starting(d: Description) = Dispatchers.setMain(dispatcher)
    override fun finished(d: Description) = Dispatchers.resetMain()
}

@OptIn(ExperimentalCoroutinesApi::class)
class FeedViewModelTest {

    @get:Rule val mainRule = MainDispatcherRule()   // Unconfined по умолчанию
    private val repo = mockk<FeedRepository>()

    @Test
    fun `init loads data into Content`() {
        coEvery { repo.load() } returns listOf("a", "b")

        val vm = FeedViewModel(repo)               // init { viewModelScope.launch { ... } }

        // никакого advanceUntilIdle — Unconfined уже всё выполнил
        assertEquals(FeedState.Content(listOf("a", "b")), vm.state.value)
    }
}

Объяснение: UnconfinedTestDispatcher выполняет корутину из init eagerly до первой приостановки, поэтому к моменту ассерта state уже обновлён. Со StandardTestDispatcher корутина отложена в очередь планировщика — нужен runTest { advanceUntilIdle() } (и тот же scheduler у rule и кода), иначе на момент проверки state ещё Idle. Это та же асимметрия «тело eagerly, дети в очереди», только применённая к загрузке в конструкторе.
