HTTP: методы, статусы, семантика

HTTP — текстовый (в HTTP/2 — бинарный) запрос-ответный протокол. Запрос = метод + URL + заголовки + тело; ответ = статус + заголовки + тело. Ключевые свойства методов, которые спрашивают: безопасность (safe — не меняет состояние) и идемпотентность (повтор даёт тот же эффект).

Почему это важно: повторять (retry) автоматически можно только идемпотентные запросы. Повторный POST «создать заказ» создаст дубль — поэтому для него вводят ключ идемпотентности (Idempotency-Key), а OkHttp по умолчанию не ретраит не-идемпотентные методы при разрыве.

Кэширование: ETag и Cache-Control

Сервер отдаёт Cache-Control: max-age=60 (сколько секунд ответ «свежий») и/или валидатор ETag — хэш версии ресурса. Когда срок истёк, клиент шлёт условный запрос с If-None-Match: "etag". Если ресурс не менялся, сервер отвечает 304 Not Modified без тела — экономия трафика и батареи.

// Запрос -> первый ответ
// < HTTP/1.1 200 OK
// < ETag: "v3-abc"
// < Cache-Control: max-age=60

// Повторный запрос после устаревания
// > GET /profile
// > If-None-Match: "v3-abc"
// < HTTP/1.1 304 Not Modified   (тело не передаётся — берём из кэша)

Дополнительно: no-store — не кэшировать вообще; no-cache — кэшировать, но всегда перепроверять; private — только клиентский кэш, не прокси. Cookie/Set-Cookie переносят сессионное состояние, но в мобильных API чаще используют токены в Authorization: Bearer (stateless), а не cookie.

HTTP/2 — общий фундамент: бинарные фреймы, мультиплексирование (много запросов в одном TCP-соединении без head-of-line блокировки уровня HTTP), сжатие заголовков HPACK, server push. OkHttp поддерживает HTTP/2 автоматически (через ALPN при TLS) — на одном соединении параллельно идут десятки запросов, что снижает overhead на установку TCP/TLS.

PUT идемпотентен, а PATCH — нет. Но если PATCH присылает полный объект целиком, он же повторяется без вреда. Так почему спецификация всё равно считает PATCH не идемпотентным, и где это «кусается» при ретраях? ★

Идемпотентность — это свойство метода по спецификации (RFC 9110), а не конкретного тела запроса. PATCH объявлен не идемпотентным, потому что семантика патча в общем случае относительная: JSON Patch с операцией add в массив, «увеличь счётчик на 1», «допиши в конец». Сервер и любой прокси на пути не вправе предполагать, что ваш конкретный PATCH на самом деле заменяет ресурс целиком — для них метод не идемпотентен по объявлению. PUT же идемпотентен, потому что заменяет ресурс целиком: повтор даёт то же состояние.

Практическое следствие — про ретраи. Автоматический повтор (OkHttp retryOnConnectionFailure, ретраи прокси и инфраструктуры) безопасен для PUT/DELETE/GET. А для PATCH повтор после неясного исхода — таймаут уже после того, как сервер применил изменение, — может применить операцию дважды. Поэтому даже «полнообъектный» PATCH защищают Idempotency-Key, а не формой тела: клиент не контролирует, как метод трактует промежуточная инфраструктура.

TLS/HTTPS: рукопожатие и доверие

HTTPS = HTTP поверх TLS. TLS решает три задачи: конфиденциальность (шифрование), целостность (MAC/AEAD), аутентификация сервера (сертификат). Главная идея, которую ждут на собесе: асимметричная криптография используется только при рукопожатии, чтобы безопасно договориться об общем симметричном ключе сессии, а дальше весь трафик шифруется быстрым симметричным алгоритмом (AES-GCM/ChaCha20).

Скоростьбыстро (массив данных)медленно (дорогая математика)

Рукопожатие (TLS 1.3)

sequenceDiagram participant C as Клиент participant S as Сервер C->>S: ClientHello (версии TLS, cipher suites, key_share, client random) S->>S: Выбор cipher suite, генерация ключа сессии S->>C: ServerHello (выбранный suite, key_share, server random) S->>C: Certificate + CertificateVerify (зашифровано) S->>C: Finished Note over C: Проверка цепочки сертификата до доверенного CA C->>S: Finished Note over C,S: Установлен общий симметричный ключ — данные шифруются C->>S: HTTP-запрос (зашифрован, application data) S->>C: HTTP-ответ (зашифрован)
TLS 1.3: 1-RTT — ключ согласуется за один круг, после ServerHello всё зашифровано (включая сам сертификат)

В TLS 1.3 клиент шлёт свой key_share (половину эфемерного Diffie-Hellman) уже в ClientHello, поэтому handshake укладывается в один round-trip (в TLS 1.2 их было два). Общий секрет вычисляется по DH — приватные ключи по сети не передаются, что даёт forward secrecy: даже компрометация приватного ключа сервера в будущем не расшифрует старые сессии.

Сертификаты и цепочка доверия

Сертификат (X.509) связывает домен с публичным ключом и подписан удостоверяющим центром (CA). Клиент проверяет цепочку: листовой сертификат сервера → промежуточный CA → корневой CA, который уже лежит в системном trust store устройства. Если подпись валидна, домен совпадает и срок не истёк — серверу доверяют.

MITM. Если злоумышленник подсунет свой CA в trust store (rooted-устройство, корпоративный прокси, малварь), он сможет расшифровать «защищённый» трафик. Защита прикладного уровня от этого — pinning.

Certificate pinning vs public-key pinning, mTLS

OkHttp CertificatePinner на самом деле пинит именно SPKI-хэш (base64 SHA-256), несмотря на имя класса. Обязательно нужен backup-пин (резервный ключ): иначе при экстренной смене ключа приложение потеряет связь до выката апдейта.

val pinner = CertificatePinner.Builder()
    .add("api.example.com", "sha256/AAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAA=")
    .add("api.example.com", "sha256/BBBBBBBBBBBBBBBBBBBBBBBBBBBBBBBBBBBBBBBBBBB=") // backup
    .build()

val client = OkHttpClient.Builder()
    .certificatePinner(pinner)
    .build()

mTLS (mutual TLS) — двусторонняя аутентификация: не только клиент проверяет сервер, но и сервер требует клиентский сертификат. Применяется в банковских/корпоративных API и B2B — приложение предъявляет свой сертификат из KeyStore.

Зачем в ClientHello отправляется расширение SNI открытым текстом, в чём из-за этого дыра в приватности, и что её закрывает в TLS 1.3? ★

SNI (Server Name Indication) — это имя хоста прямо в ClientHello. Оно нужно, чтобы сервер с несколькими доменами на одном IP выбрал правильный сертификат до того, как узнает, что внутри HTTP-запроса — ведь Host-заголовок поедет уже зашифрованным. Курица и яйцо: чтобы согласовать шифрование, сервер должен знать, какой сертификат показать, а узнаёт он это из открытого SNI.

Дыра: в TLS 1.2 и базовом 1.3 SNI идёт открытым текстом. Пассивный наблюдатель — провайдер, DPI на границе сети — видит, к какому домену вы подключаетесь, даже не расшифровывая трафик. На этом строят блокировки и слежку «по доменам». Закрывает это ECH (Encrypted Client Hello, бывший ESNI) — расширение поверх TLS 1.3: оно шифрует внутренний ClientHello публичным ключом, который клиент берёт из DNS-записи (HTTPS/SVCB RR). На Android поддержка ECH зависит от сетевого стека и серверной стороны (Cloudflare и др.) — это уже не часть рукопожатия 1.3 по умолчанию.

OkHttp: клиент, интерсепторы, пул

OkHttpClient — тяжёлый объект: держит пул соединений, пул потоков (dispatcher) и кэш. Создавай его один раз (синглтон/DI) и переиспользуй через newBuilder() — так клиенты делят connection pool и кэш.

NetworkModule.kt
val client = OkHttpClient.Builder()
    .connectTimeout(10, TimeUnit.SECONDS)   // установка TCP+TLS
    .readTimeout(15, TimeUnit.SECONDS)      // ожидание байтов ответа
    .writeTimeout(15, TimeUnit.SECONDS)     // отправка тела
    .callTimeout(30, TimeUnit.SECONDS)      // суммарный лимит на весь вызов
    .retryOnConnectionFailure(true)         // retry при сбое соединения (idempotent)
    .connectionPool(ConnectionPool(5, 5, TimeUnit.MINUTES))
    .cache(Cache(File(cacheDir, "http"), 10L * 1024 * 1024)) // 10 МБ
    .addInterceptor(AuthInterceptor(tokenStore))   // application
    .addNetworkInterceptor(StethoLikeLogger())     // network
    .build()

Это любимый вопрос. Интерсептор — это Interceptor с методом intercept(chain), который может изменить запрос, вызвать chain.proceed() и изменить ответ. Разница — в месте в цепочке.

class AuthInterceptor(private val tokens: TokenStore) : Interceptor {
    override fun intercept(chain: Interceptor.Chain): Response {
        val request = chain.request().newBuilder()
            .header("Authorization", "Bearer ${tokens.access()}")
            .build()
        val response = chain.proceed(request)

        // прозрачное обновление токена на 401 (без рекурсии)
        if (response.code == 401 && tokens.refresh()) {
            response.close()
            val retried = request.newBuilder()
                .header("Authorization", "Bearer ${tokens.access()}")
                .build()
            return chain.proceed(retried)
        }
        return response
    }
}
⚠️
Authenticator vs Interceptor. Для реактивного обновления токена на 401 каноничнее OkHttpClient.authenticator(): OkHttp сам вызовет его при 401 и встроит ретрай, исключив гонки. В интерсепторе обновление токена легко получить дубли запросов — синхронизируй доступ к refresh.

Connection pool переиспользует keep-alive соединения: не тратится время на TCP+TLS для каждого запроса. EventListener даёт тайминги по фазам (DNS, connect, TLS, время до первого байта) — основа сетевых метрик и диагностики «почему медленно».

    override fun responseHeadersStart(call: Call) { /* TTFB */ }
}

Порядок цепочки интерсепторов

Между вашими application- и network-интерсепторами OkHttp вставляет встроенные звенья. Из их порядка следует, почему application-интерсептор вызывается один раз и не видит ни редиректы, ни gzip, а network — на каждый хоп.

flowchart TB App["Приложение"] --> AppI["Ваши application-интерсепторы"] AppI --> Bridge["BridgeInterceptor (gzip, Host, Content-Length)"] Bridge --> CacheI["CacheInterceptor (хит / мисс)"] CacheI --> Connect["ConnectInterceptor (сокет, TLS)"] Connect --> NetI["Ваши network-интерсепторы"] NetI --> CallServer["CallServerInterceptor"] CallServer --> Net(["Сеть"]) Net --> CallServer CacheI -. "хит: ответ из кэша" .-> AppI
Запрос идёт сверху вниз, ответ — обратно. До BridgeInterceptor нет gzip, до ConnectInterceptor нет соединения

Поэтому HttpLoggingInterceptor на addInterceptor видит уже распакованное тело и не видит Accept-Encoding/Content-Encoding — их добавляет и снимает BridgeInterceptor ниже. Чтобы увидеть реальные «по проводу» заголовки и каждый редирект, логгер вешают на addNetworkInterceptor.

Десять корутин одновременно дёргают один эндпоинт через один OkHttpClient. Сколько TCP-соединений к хосту откроется и почему именно столько? И что произойдёт с 11-м запросом? ★

По умолчанию Dispatcher держит maxRequests = 64 и maxRequestsPerHost = 5. К одному хосту одновременно выполняется максимум 5 запросов, остальные ждут в очереди readyAsyncCalls. Дальше всё зависит от версии протокола:

  • HTTP/1.1 — одно соединение обслуживает один запрос за раз, поэтому под пять параллельных запросов откроется до пяти отдельных TCP+TLS-соединений.
  • HTTP/2 (согласован через ALPN при TLS) — все запросы к хосту мультиплексируются в одном соединении, и узким местом раньше становится maxRequestsPerHost, а не число сокетов.

ConnectionPool по умолчанию хранит до 5 idle-соединений и закрывает их после 5 минут простоя, переиспользуя keep-alive. 11-й запрос (если первые 5 ещё идут на том же хосте по HTTP/1.1) просто ждёт освобождения слота — никакой ошибки, только лишняя латентность. Поднять параллелизм можно через dispatcher.maxRequestsPerHost.

Retrofit: конвертеры, suspend, обёртки

Retrofit превращает аннотированный Kotlin-интерфейс в реализацию HTTP-клиента. Под капотом — OkHttp; сверху — конвертеры (тело ↔ объект) и call-адаптеры (тип возврата: Call, suspend, Flow, кастомный Result).

Api.kt
interface UserApi {
    @GET("users/{id}")
    suspend fun getUser(@Path("id") id: String): UserDto          // тело напрямую

    @GET("users/{id}")
    suspend fun getUserResponse(@Path("id") id: String): Response<UserDto> // + код/заголовки

    @POST("users")
    suspend fun create(@Body body: CreateUserDto): UserDto
}

val retrofit = Retrofit.Builder()
    .baseUrl("https://api.example.com/")
    .client(okHttpClient)
    .addConverterFactory(json.asConverterFactory("application/json".toMediaType()))
    .build()

val api = retrofit.create<UserApi>()

suspend-функции: начиная с Retrofit 2.6, метод можно объявить suspend — Retrofit сам адаптирует Call в корутину и переключает на IO-диспетчер OkHttp. Возврат UserDto — бросит исключение на не-2xx; возврат Response<UserDto> — отдаст код и тело, давая контроль над ошибками.

Оборачивание в Result/sealed

Голые исключения в data-слое неудобны. Два пути: (1) кастомный CallAdapter, чтобы интерфейс возвращал ApiResult<T> прямо из Retrofit; (2) обернуть вызов в репозитории через runCatching/try-catch и маппить в доменный sealed-тип. Второй проще и переносим между источниками — его и покажем в следующем разделе.

💡
DTO ≠ domain. Из API возвращайте *Dto (поля как в JSON), а в репозитории маппьте в доменную модель. Так контракт сервера не протекает в UI, а смена формата не ломает весь код.
Retrofit-метод объявлен suspend и вызывается из viewModelScope. Корутину отменяют (ушли с экрана). Что физически происходит с уже улетевшим HTTP-запросом и на чём именно подвешен механизм отмены? ★

Для suspend-функции Retrofit под капотом адаптирует Call в корутину через suspendCancellableCoroutine и регистрирует invokeOnCancellation { call.cancel() }. Когда родительский scope отменяет корутину, бросается CancellationException, отрабатывает этот колбэк, OkHttp вызывает Call.cancel() — сокет закрывается, текущий блокирующий ввод-вывод выбрасывает IOException внутри OkHttp. То есть отмена не «магическая»: запрос реально рвётся на транспортном уровне, а не просто игнорируется результат.

Нюанс, на котором ловят: сервер мог уже получить и обработать запрос (создать заказ) до разрыва. Отмена на клиенте не откатывает серверный эффект — для не-идемпотентных операций это не страховка. И ещё: CancellationException нельзя глотать в safeApiCall, иначе ensureActive и структурированный параллелизм сломаются.

Обработка ошибок и устойчивость

Сетевой вызов падает множеством способов: нет связи (IOException/UnknownHostException), HTTP-ошибка (HttpException / не-2xx), таймаут (SocketTimeoutException), ошибка парсинга (SerializationException), отмена (CancellationException). Цель — поймать всё это в одном месте и отдать наверх предсказуемый доменный результат, а не «сырое» исключение.

DataResult.kt
sealed interface DataResult<out T> {
    data class Success<T>(val data: T) : DataResult<T>
    sealed interface Failure : DataResult<Nothing> {
        data class Network(val cause: Throwable) : Failure         // нет связи/таймаут
        data class Http(val code: Int, val body: String?) : Failure // 4xx/5xx
        data class Parse(val cause: Throwable) : Failure           // битый JSON
        data class Unknown(val cause: Throwable) : Failure
    }
}
safeApiCall.kt
suspend fun <T> safeApiCall(block: suspend () -> T): DataResult<T> =
    try {
        DataResult.Success(block())
    } catch (e: CancellationException) {
        throw e                          // отмену НЕ глотаем — пробрасываем
    } catch (e: HttpException) {
        DataResult.Failure.Http(e.code(), e.response()?.errorBody()?.string())
    } catch (e: IOException) {
        DataResult.Failure.Network(e)    // нет сети/таймаут/DNS
    } catch (e: SerializationException) {
        DataResult.Failure.Parse(e)
    } catch (e: Throwable) {
        DataResult.Failure.Unknown(e)
    }
🛑
Никогда не глотайте CancellationException. При отмене корутины (ушли с экрана) исключение должно пробрасываться, иначе сломается структурированный параллелизм и корутина «зависнет». Ловите его первым и делайте throw.

Retry с экспоненциальным backoff

Ретраить стоит транзиентные ошибки (сеть, 5xx, 429) и только идемпотентные запросы. Backoff с джиттером не даёт «толпе» клиентов синхронно долбить сервер после сбоя.

suspend fun <T> retry(
    times: Int = 3,
    initialDelay: Long = 500,
    maxDelay: Long = 8000,
    factor: Double = 2.0,
    block: suspend () -> T,
): T {
    var delayMs = initialDelay
    repeat(times - 1) {
        try { return block() }
        catch (e: CancellationException) { throw e }
        catch (e: IOException) { /* транзиентная — повторяем */ }
        delay(delayMs + Random.nextLong(0, 200))      // + джиттер
        delayMs = (delayMs * factor).toLong().coerceAtMost(maxDelay)
    }
    return block()  // последняя попытка — без перехвата
}
ℹ️
Отмена и оффлайн. Отмену даёт сама корутина: viewModelScope отменит запрос при очистке VM, OkHttp прервёт сокет. Оффлайн — стратегия SSOT: репозиторий отдаёт кэш (Room) и помечает данные «устаревшими», фоновая синхронизация догоняет (single source of truth).

Когда ретраить, а когда нет

«Слепой» ретрай легко превращается в проблему — поэтому решение о повторе стоит проводить как дерево с явными отсечками.

Почему «слепой» retry с экспоненциальным backoff может усугубить отказ сервера (retry storm), и какие три механизма этого избегают — что добавить помимо джиттера? ★

Если все клиенты при 5xx синхронно ретраят, нагрузка на уже перегруженный сервер не падает, а множится — это retry storm, амплификация: восстановиться ему становится только тяжелее. Джиттер размазывает повторы во времени, но не уменьшает их число — это лечит синхронность, а не объём. Поэтому добавляют три вещи:

  • Circuit breaker — после серии отказов цепь «размыкается», и запросы какое-то время вообще не идут на бэкенд (он остывает), а UI сразу получает ошибку.
  • Retry budget / token bucket — лимит на долю запросов, которые вообще разрешено ретраить (например не больше 10–20% трафика), чтобы повторы не превысили исходную нагрузку.
  • Уважение Retry-After из заголовка ответа на 429/503 — сервер сам говорит, когда вернуться, и это приоритетнее собственного backoff.

И базовое: ретраить только идемпотентные запросы и только транзиентные ошибки (сеть, 429, 5xx), а не 4xx.

Главный риск Gson. Он создаёт объект через Unsafe, минуя конструктор Kotlin: non-null поле может оказаться null, default-значение проигнорируется — и приложение упадёт с NPE в неожиданном месте. На new-проектах берут kotlinx.serialization или Moshi.
В kotlinx.serialization у поля есть значение по умолчанию, и сервер это поле не прислал. Поле подставится из default. Но если поле пришло как явный null, а тип non-null с default — будет ли использован default? От чего это зависит? ★

Это два разных случая, и путать их — классическая ошибка. Если ключа в JSON вообще нет — kotlinx.serialization берёт значение по умолчанию из конструктора (декодер опирается на дефолты, и в отличие от Gson их нельзя выключить). А вот если ключ присутствует, но значение явный null, а свойство объявлено non-null с default — по умолчанию это ошибка десериализации (MissingFieldException / несоответствие типа), и default не применяется.

Чтобы null трактовался как «использовать default», нужен флаг Json { coerceInputValues = true } — он приводит недопустимый null (и неизвестные значения enum) к значению по умолчанию. Отсюда разделение: ignoreUnknownKeys спасает от лишних полей сервера, coerceInputValues — от «грязных» null'ов. Разные настройки под разные проблемы.

WebSockets и реалтайм: Upgrade, ping/pong, реконнект

HTTP — запрос-ответный: сервер молчит, пока его не спросят. Для чатов, котировок, статусов заказа это неудобно — нужен канал, по которому сервер сам шлёт сообщения. WebSocket поднимается поверх того же TCP через HTTP Upgrade-рукопожатие: клиент шлёт обычный GET с заголовком Upgrade: websocket, сервер отвечает 101 Switching Protocols, и дальше сокет переходит в полнодуплексный режим. wss:// — то же самое, но поверх TLS.

В OkHttp WebSocket поднимается через client.newWebSocket(request, listener), а WebSocketListener отдаёт колбэки onOpen / onMessage / onClosing / onClosed / onFailure. Закрытие — webSocket.close(1000, reason).

ChatSocket.kt
fun chatMessages(client: OkHttpClient, url: String): Flow<String> = callbackFlow {
    val request = Request.Builder().url(url).build()
    val listener = object : WebSocketListener() {
        override fun onMessage(ws: WebSocket, text: String) { trySend(text) }
        override fun onClosed(ws: WebSocket, code: Int, reason: String) { close() }
        override fun onFailure(ws: WebSocket, t: Throwable, r: Response?) { close(t) }
    }
    val ws = client.newWebSocket(request, listener)
    awaitClose { ws.close(1000, null) }   // отмена Flow -> закрываем сокет
}

Ping/pong и детект «тихо умершего» сокета

NAT и прокси умеют рвать соединение без FIN — для приложения сокет «как будто живой», но сообщения не доходят. OkHttp умеет автоматический ping через pingInterval(20, TimeUnit.SECONDS): ping/pong — control-фреймы, которые проверяют, что канал жив, и быстро ловят такой разрыв.

Реконнект и коды закрытия

Мобильная сеть рвёт соединения часто, поэтому reconnect пишут вручную в onFailure/onClosed с экспоненциальным backoff и джиттером. Смотрят на код закрытия: 1000 — нормальное закрытие (реконнектить не нужно), 1011 — ошибка сервера (реконнектить стоит). Вся работа — не на main-потоке, один OkHttpClient на всё.

Скорее всего соединение разорвал NAT или прокси без отправки FIN — на уровне TCP клиент об этом не узнаёт, сокет в состоянии «как будто открыт». Колбэк onFailure не сработает, пока вы сами не попробуете что-то отправить и не упрётесь в таймаут. Это «half-open» соединение.

Ловят это pingInterval(...): OkHttp шлёт ping-фреймы и ждёт pong; если pong не пришёл за интервал, соединение признаётся мёртвым и вызывается onFailure — после чего отрабатывает ваш reconnect с backoff. Без активного ping приложение узнаёт о разрыве только в момент следующей отправки, и для realtime-экрана это та самая «минута тишины».

HTTP/3 и QUIC: транспорт нового поколения

HTTP/2 мультиплексирует потоки в одном TCP-соединении, но у TCP остаётся head-of-line blocking на транспортном уровне: потеря одного пакета задерживает доставку всех мультиплексированных потоков, пока пакет не перешлют. На мобильных сетях с потерями это бьёт особенно больно — один потерянный сегмент тормозит сразу все запросы в соединении.

HTTP/3 работает поверх QUIC — транспорта на UDP, где потоки независимы: потеря пакета в одном потоке не блокирует другие. QUIC объединяет TCP- и TLS-рукопожатие в одно (1-RTT на установку, а с 0-RTT — мгновенное возобновление сессии) и умеет connection migration: соединение переживает смену сети (Wi-Fi → LTE) по Connection ID, не разрываясь при переходе.

HTTP/2 уже мультиплексирует потоки в одном соединении. Тогда какую именно проблему решает HTTP/3, если на уровне приложения head-of-line blocking в HTTP/2 уже нет? ★

HTTP/2 убрал head-of-line blocking на уровне приложения: запросы больше не ждут друг друга в очереди HTTP. Но потоки HTTP/2 едут внутри одного TCP-соединения, а TCP — это упорядоченный байтовый поток. Если теряется один TCP-сегмент, ядро не отдаёт приложению ничего из последующих байтов, пока сегмент не перешлют — даже если те байты относятся к совсем другому, «здоровому» потоку. То есть HoL-блокировка просто опустилась на транспортный уровень.

QUIC решает это тем, что сам управляет потоками поверх UDP и нумерует пакеты по каждому потоку отдельно: потеря в одном потоке не задерживает другие. Плюс бонусы, которых у TCP нет в принципе — 0-RTT возобновление и connection migration по Connection ID при смене сети. На мобильных потери и переключение Wi-Fi/LTE — норма, поэтому выигрыш заметен именно там.

Кэширование OkHttp на практике: офлайн, CacheControl, Vary

Семантику ETag/Cache-Control мы разобрали со стороны сервера. Теперь клиентский слой: OkHttp умеет дисковый кэш, но включается он не сам собой. Нужен сконфигурированный Cache, и кэшируются только ответы с валидными Cache-Control/ETag — если сервер их не прислал, кэш молчит.

CacheModule.kt
val cache = Cache(File(context.cacheDir, "http"), 10L * 1024 * 1024) // 10 МБ

// network-интерсептор: навязать клиентский max-age, если сервер не задал
val rewriteCache = Interceptor { chain ->
    chain.proceed(chain.request()).newBuilder()
        .header("Cache-Control", "public, max-age=60")
        .build()
}

// application-интерсептор: офлайн — отдавать устаревший кэш вместо ошибки
val offlineCache = Interceptor { chain ->
    var request = chain.request()
    if (!hasNetwork()) {
        request = request.newBuilder()
            .cacheControl(CacheControl.Builder().onlyIfCached().maxStale(7, TimeUnit.DAYS).build())
            .build()
    }
    chain.proceed(request)
}

val client = OkHttpClient.Builder()
    .cache(cache)
    .addNetworkInterceptor(rewriteCache)
    .addInterceptor(offlineCache)
    .build()

Помимо этого есть готовые директивы: CacheControl.FORCE_CACHE берёт только из кэша (вернёт 504, если данных нет), FORCE_NETWORK игнорирует кэш. Заголовок Vary говорит кэшу, по каким заголовкам запроса различать варианты ответа (Vary: Accept-Encoding, Authorization) — без него кэш может вернуть «чужой» вариант, например ответ для другого пользователя.

⚠️
Один кэш на клиент. Cache потокобезопасен, но один и тот же каталог нельзя делить между несколькими OkHttpClient — они затрут данные друг друга. Диагностика хита/мисса — через response.cacheResponse и response.networkResponse: хит = есть cacheResponse и нет networkResponse.
Настроили Cache, сервер шлёт Cache-Control: max-age=60, но в офлайне OkHttp всё равно отдаёт ошибку вместо кэша. Почему, и что добавить? ★

Обычный max-age=60 делает ответ «свежим» только 60 секунд. Через минуту запись в кэше есть, но считается устаревшей, и без сети OkHttp не имеет права её отдать — он попытается сходить на сервер, упрётся в отсутствие сети и вернёт IOException. Кэш не пустой, просто «протух» по правилам.

Чинит это application-интерсептор, который офлайн переписывает запрос на CacheControl с onlyIfCached() и maxStale(...): можно брать из кэша, даже если запись устарела на N дней. Тогда OkHttp отдаёт старую запись вместо ошибки. Важно вешать это именно как application-интерсептор (он видит запрос до решения о сети) и проверять наличие сети самому — кэш сам по себе про офлайн-политику ничего не знает.

Network Security Configuration: декларативная политика доверия

Pinning через CertificatePinner живёт в коде и работает только для запросов через конкретный клиент. Он применяется ко всему сетевому стеку приложения и описывает доверие политикой, а не кодом.

Pinning без плана ротации = «кирпич». NSC требует expiration у pin-set: после этой даты pinning отключается — это защита от ситуации «сертификат сменили, а пины забыли». Любой pinning без backup-пина и плана обновления рискует отрезать пользователей при смене сертификата до выката апдейта.
Поставили pinning через NSC, всё работало. Через год сертификат на сервере планово сменили — и приложение у старых пользователей продолжило работать, хотя пины не совпадают. Как так? ★

Сработала защита от «кирпича»: у pin-set в NSC обязателен атрибут expiration. После этой даты Android просто перестаёт применять pinning для домена и откатывается к обычной проверке цепочки до системных CA. Если дата истечения уже прошла к моменту смены сертификата, новый валидный сертификат пройдёт штатную проверку, и приложение продолжит работать, несмотря на несовпадение пинов.

Это палка о двух концах: с одной стороны — страховка от того, что забытые пины навсегда отрежут пользователей; с другой — после expiration вы теряете защиту от MITM, ради которой pinning и ставили. Поэтому expiration выбирают с запасом под цикл обновления приложения, держат backup-пин и обновляют пины релизом заранее, а не полагаются на их «протухание».

Внутренности OkHttp: Dispatcher, ConnectionPool, DNS

Senior-вопросы про сеть быстро уходят «под капот» клиента: сколько соединений откроется, как ограничить параллелизм, можно ли подменить DNS. Три узла, которые стоит держать в голове.

Dispatcher — очередь и параллелизм

Dispatcher управляет асинхронными вызовами: maxRequests = 64 (всего одновременно) и maxRequestsPerHost = 5 (на один хост) по умолчанию. Вызовы сверх лимита ждут в readyAsyncCalls и переезжают в runningAsyncCalls, когда освобождается слот. У диспетчера свой ExecutorService — можно подменить.

ConnectionPool — keep-alive

ConnectionPool по умолчанию хранит до 5 idle-соединений и закрывает их после 5 минут простоя. Живое соединение переиспользуется (keep-alive), экономя TCP+TLS на каждом запросе. По HTTP/1.1 одно соединение = один запрос за раз; по HTTP/2 в одном соединении мультиплексируются все запросы к хосту — поэтому число реальных сокетов зависит от протокола.

DNS — свой резолвер

На одном хосте по HTTP/2 запустили 30 параллельных запросов через OkHttp. Сколько TCP-соединений откроется и упрётся ли что-то в maxRequestsPerHost? ★

По HTTP/2 все запросы к одному хосту мультиплексируются в одном TCP+TLS-соединении — отдельный сокет под каждый запрос не нужен. Так что соединение в норме будет одно (плюс, возможно, ещё одно, пока первое устанавливается).

А вот maxRequestsPerHost = 5 в Dispatcher ограничивает именно число одновременно выполняемых вызовов к хосту, независимо от того, что транспорт легко тянет больше в одном HTTP/2-соединении. Из 30 запросов одновременно «в полёте» будет 5, остальные 25 ждут в очереди readyAsyncCalls. То есть в HTTP/2-сценарии искусственным узким местом становится диспетчер, а не сокеты — и поднимать тут надо maxRequestsPerHost, а не пул соединений.

Загрузка изображений: Coil и Glide

Картинки в списке — отдельная сетевая задача со своими ловушками: один и тот же URL запрашивают десятки раз при скролле, bitmap'ы съедают память, а view переиспользуются в RecyclerView/LazyColumn. Coil и Glide решают это многоуровневым кэшем и привязкой загрузки к жизненному циклу view.

Многоуровневый кэш

Оба держат два уровня: memory cache (декодированные bitmap'ы в LruCache — мгновенно при повторном показе) и disk cache (сырые байты ответа, переживают перезапуск). Запрос идёт по цепочке: память → диск → сеть. Glide дополнительно делает bitmap pooling — переиспользует уже выделенные bitmap'ы вместо новых аллокаций, снижая работу GC при быстром скролле.

Downsampling под размер view

Главная экономия памяти — не грузить картинку 4000×3000 в ImageView размером 200×200. Библиотека измеряет целевой view и декодирует bitmap под его размер (downsampling через inSampleSize). Отсюда классическая проблема: если у view на момент запроса ширина = 0 (ещё не прошёл layout), библиотека не знает целевой размер — Glide ждёт layout-прохода, а в Compose размер задают через модификатор. Поэтому грузить картинку в ноль-размерный контейнер без явного размера — частый баг «ничего не отображается» или «грузится оригинал».

Когда RecyclerView переиспользует строку под новый элемент (или LazyColumn пересобирает item), старая загрузка должна отмениться — иначе в ячейку прилетит чужая картинка. Библиотеки привязывают запрос к target/lifecycle и отменяют его при rebind автоматически.

В Compose картинка через AsyncImage то грузит оригинал на 12 МП, то не отображается вовсе, хотя URL верный. Что не так с размером и как Coil вообще узнаёт, под какой размер декодировать? ★

Coil декодирует bitmap под размер цели, а размер он берёт из ограничений компоновки (constraints) того места, куда рисует AsyncImage. Если контейнер не задаёт ширину/высоту — например AsyncImage стоит в Row без веса и без size, и на момент запроса ширина равна 0 или «бесконечности», — Coil не может вычислить целевой размер. В итоге он либо падает на оригинал (грузит все 12 МП в память), либо рисует в нулевой бокс, и картинки «нет».

Лечится явным размером: Modifier.size(...) / fillMaxWidth().aspectRatio(...) — чтобы у композабла были конечные constraints до загрузки. Это прямой аналог проблемы width = 0 в RecyclerView, где Glide ждёт layout-прохода: библиотека не магически знает, насколько уменьшать bitmap — ей нужен измеренный размер цели, иначе downsampling не срабатывает.

Частые вопросы интервьюера

Чем отличается idempotent от safe? Какие методы можно ретраить? Middle

Safe — метод не меняет состояние на сервере (GET, HEAD). Idempotent — повтор даёт тот же результат (GET, PUT, DELETE, но не POST/PATCH). Автоматический retry безопасен только для идемпотентных запросов: повторный POST «создать заказ» создаст дубль. Для не-идемпотентных вводят Idempotency-Key, по которому сервер дедуплицирует.

Расскажи, что происходит при TLS-рукопожатии. Senior

Клиент шлёт ClientHello (версии TLS, cipher suites, случайное число, в TLS 1.3 — key_share). Сервер выбирает шифр, отвечает ServerHello + свой key_share, присылает сертификат и Finished. Клиент проверяет цепочку сертификата до доверенного корневого CA, и обе стороны по Diffie-Hellman вычисляют общий симметричный ключ. Дальше весь трафик шифруется симметрично (AES-GCM). Асимметрия — только для аутентификации и согласования ключа; эфемерный DH даёт forward secrecy. В TLS 1.3 всё это — за 1 round-trip.

Чем application-интерсептор отличается от network в OkHttp? Middle+

Application-интерсептор стоит между приложением и ядром OkHttp: вызывается ровно один раз на запрос (даже при кэш-хите), не видит редиректы, не имеет доступа к Connection. Идеален для auth-заголовков и верхнеуровневого логирования. Network-интерсептор — между ядром и сетью: вызывается на каждый редирект/retry, видит реальные «по проводу» заголовки (например Accept-Encoding: gzip), имеет доступ к Connection (IP, TLS). Подходит для логирования по хопам и метрик. Из кэша — не вызывается вообще.

Certificate pinning или public-key pinning — что выберешь и почему? Senior

Public-key (SPKI) pinning. Он пинит хэш публичного ключа, а не всего сертификата, поэтому переживает плановую пере-выписку сертификата с тем же ключом — не требует релиза приложения. OkHttp CertificatePinner как раз пинит SPKI. Обязательно добавляю backup-пин (резервный ключ): без него экстренная смена ключа отрежет всех пользователей до выката апдейта. Pinning защищает от MITM даже при скомпрометированном/подложном CA в trust store.

Как обработать ошибку сети, чтобы приложение не падало? Middle+

Оборачиваю вызов в safeApiCall, который ловит HttpException (4xx/5xx), IOException (нет связи/таймаут/DNS), SerializationException (парсинг) и маппит в sealed DataResult.Failure.{Http, Network, Parse, Unknown}. CancellationException ловлю первым и пробрасываю. UI получает предсказуемый тип и решает, что показать: ретрай, «нет интернета», экран ошибки. На транзиентных ошибках — retry с экспоненциальным backoff и джиттером, только для идемпотентных запросов.

Почему нельзя глотать CancellationException? Senior

Отмена корутины реализована через бросок CancellationException. Если поймать его в общем catch (e: Exception) и не пробросить, родительский scope не узнает об отмене — структурированный параллелизм сломается, корутина «повиснет», ресурсы (сокет, курсор) не освободятся. Поэтому CancellationException ловят отдельной первой веткой и делают throw, либо используют currentCoroutineContext().ensureActive().

ETag и Cache-Control — как работает условный запрос? Middle

Cache-Control: max-age=N задаёт время «свежести» ответа — пока оно не истекло, клиент берёт данные из кэша без запроса. Когда истекло, клиент шлёт условный запрос с If-None-Match: "<etag>" (ETag — версия/хэш ресурса). Если ресурс не менялся, сервер отвечает 304 Not Modified без тела, и клиент использует кэш — экономия трафика и батареи. OkHttp с настроенным Cache делает это прозрачно.

Зачем HTTP/2 и что такое мультиплексирование? Middle+

HTTP/1.1 на одном TCP-соединении обрабатывает запросы по очереди (head-of-line блокировка), поэтому браузеры/клиенты открывали несколько соединений. HTTP/2 ввёл бинарные фреймы и мультиплексирование: множество запросов и ответов идут параллельно в одном TCP-соединении, разбитые на фреймы и переплетённые. Плюс сжатие заголовков HPACK и server push. Итог — меньше соединений, меньше overhead на TCP/TLS, ниже латентность. OkHttp включает HTTP/2 автоматически через ALPN при TLS.

kotlinx.serialization, Moshi или Gson — что и почему? Senior

Для нового кода — kotlinx.serialization (compile-time, без рефлексии, понимает default-значения, null-safety и sealed-полиморфизм) или Moshi с codegen. Gson — в режиме поддержки: он использует рефлексию и Unsafe, создаёт объект минуя конструктор Kotlin, поэтому игнорирует default-значения и может поставить null в non-null поле → NPE. Это его главный подводный камень. Производительность и безопасность типов на стороне kotlinx/Moshi.

Запрос завис на 40 секунд, хотя connectTimeout = 10 с. Как такое возможно и какой таймаут это ловит? Middle+

connectTimeout (по умолчанию 10 с) ограничивает только установку TCP+TLS-соединения, а не получение ответа. После успешного коннекта сервер может «думать» сколько угодно — это область readTimeout (ожидание следующего байта, по умолчанию 10 с) и writeTimeout (отправка тела). Но и они применяются к отдельному чтению/записи: если сервер шлёт по байту раз в 9 секунд, readTimeout не сработает, а суммарно запрос тянется минутами. Жёсткий лимит на весь вызов — включая DNS, коннект, редиректы, повторы и чтение тела — даёт callTimeout (по умолчанию 0 = выключен), его и ставят в проде. Ещё retryOnConnectionFailure может незаметно удваивать время, переподключаясь к другому IP.

Почему вы почти никогда не увидите Accept-Encoding: gzip в своём логирующем application-интерсепторе? Senior

BridgeInterceptor — встроенное звено OkHttp между application- и network-слоем. Если вы сами не задали Accept-Encoding, он добавляет Accept-Encoding: gzip, а когда приходит сжатый ответ — прозрачно распаковывает его и убирает заголовки Content-Encoding и Content-Length. Поскольку BridgeInterceptor стоит ниже application-интерсепторов, ваш HttpLoggingInterceptor на addInterceptor видит уже распакованное тело и не видит этих заголовков. Чтобы залогировать реальные «по проводу» сжатые заголовки, вешайте логгер как network-интерсептор (addNetworkInterceptor). Это прямое следствие порядка цепочки.

Почему для рефреша токена на 401 предпочтительнее Authenticator, чем интерсептор? Senior

OkHttp Authenticator вызывается ядром реактивно — именно при 401 (для прокси — 407), и OkHttp сам встраивает повтор запроса с новыми заголовками, отслеживая responseCount, чтобы не зациклиться. В интерсепторе ретрай и защиту от рекурсии вы пишете руками. Общая для обоих проблема — гонка: при просроченном токене несколько запросов одновременно ловят 401 и кидаются рефрешить. Решение: синхронизировать рефреш (Mutex/synchronized) и внутри критической секции сравнить токен из заголовка пришедшего запроса с актуальным в хранилище — если они уже разные, другой поток успел обновить, и надо просто взять новый токен без повторного refresh. В Authenticator возврат null означает «сдаться» — отдать 401 наверх (например разлогинить).

Задача 1. Напишите Interceptor для OkHttp, который при получении 401 однократно обновляет токен через TokenRefresher.refresh(): String? и повторяет запрос с новым токеном. Если refresh() вернул null — просто отдайте оригинальный ответ 401. Параллельные запросы не должны вызывать refresh() несколько раз одновременно.

Показать решение 
class TokenInterceptor(
    private val tokenStore: TokenStore,
    private val refresher: TokenRefresher,
) : Interceptor {

    private val lock = Any()

    override fun intercept(chain: Interceptor.Chain): Response {
        val response = chain.proceed(withBearer(chain.request()))
        if (response.code != 401) return response

        val newToken: String? = synchronized(lock) {
            // Если пока ждали блокировку кто-то уже обновил токен — повторяем без refresh
            val current = tokenStore.access()
            val fromHeader = chain.request().header("Authorization")
            if (fromHeader != null && fromHeader != "Bearer $current") {
                current
            } else {
                refresher.refresh()?.also { tokenStore.save(it) }
            }
        }

        return if (newToken != null) {
            response.close()
            chain.proceed(withBearer(chain.request(), newToken))
        } else {
            response
        }
    }

    private fun withBearer(req: okhttp3.Request, token: String = tokenStore.access()) =
        req.newBuilder().header("Authorization", "Bearer $token").build()
}

Объяснение: блок synchronized(lock) гарантирует, что при одновременном 401 от нескольких корутин обновление токена произойдёт ровно один раз — остальные потоки дождутся и подхватят уже готовый токен. Проверка того, совпадает ли заголовок с текущим токеном в хранилище, отсекает повторный refresh(), если первый поток уже всё обновил.

Задача 2. Реализуйте функцию fetchWithRetry для suspend-вызова Retrofit. Функция должна повторять запрос при IOException или HTTP-коде 429/5xx — но не более 3 попыток. Между попытками — экспоненциальный backoff: 1 с → 2 с → 4 с. При любом другом HTTP-коде или при CancellationException — не ретраить, пробросить как есть.

Показать решение 
suspend fun <T> fetchWithRetry(
    maxAttempts: Int = 3,
    block: suspend () -> retrofit2.Response<T>,
): retrofit2.Response<T> {
    var attempt = 0
    var delayMs = 1_000L
    while (true) {
        attempt++
        try {
            val response = block()
            val code = response.code()
            if (code == 429 || code in 500..599) {
                if (attempt >= maxAttempts) return response
                kotlinx.coroutines.delay(delayMs)
                delayMs *= 2
                continue
            }
            return response
        } catch (e: kotlinx.coroutines.CancellationException) {
            throw e                           // отмену не трогаем
        } catch (e: java.io.IOException) {
            if (attempt >= maxAttempts) throw e
            kotlinx.coroutines.delay(delayMs)
            delayMs *= 2
        }
    }
}

Объяснение: CancellationException пробрасывается первым — иначе structured concurrency сломается. HTTP 429/5xx и IOException считаются транзиентными, по ним делаем до maxAttempts попыток. На последней попытке при транзиентной ошибке возвращаем ответ (или пробрасываем исключение) без дальнейшего ретрая. 4xx (кроме 429) — ошибка клиента, ретраить бессмысленно.

Задача 4. Реализуйте OkHttp Authenticator для рефреша access-токена на 401, устойчивый к гонке: при одновременном 401 от нескольких параллельных запросов refresh() должен вызваться ровно один раз, остальные — подхватить уже обновлённый токен. Если refresh не удался — верните null (сдаться). Защитите от бесконечного цикла повторов.

Показать решение 
class TokenAuthenticator(
    private val store: TokenStore,
    private val refresher: Refresher,
) : Authenticator {

    private val lock = Any()

    override fun authenticate(route: Route?, response: Response): Request? {
        // защита от цикла: если повторов уже >= 2 — сдаёмся
        if (responseCount(response) >= 2) return null

        synchronized(lock) {
            val current = store.access()
            val failedToken = response.request.header("Authorization")

            // другой поток уже обновил токен, пока мы ждали lock
            if (failedToken != null && failedToken != "Bearer $current") {
                return response.request.newBuilder()
                    .header("Authorization", "Bearer $current")
                    .build()
            }

            val fresh = refresher.refresh() ?: return null  // не вышло — отдать 401 наверх
            store.save(fresh)
            return response.request.newBuilder()
                .header("Authorization", "Bearer $fresh")
                .build()
        }
    }

    private fun responseCount(response: Response): Int {
        var r: Response? = response.priorResponse
        var count = 1
        while (r != null) { count++; r = r.priorResponse }
        return count
    }
}

Объяснение: ключевой приём — внутри synchronized сравнить токен из заголовка упавшего запроса (failedToken) с текущим в хранилище. Если они уже различаются, значит другой поток успел обновить токен, пока мы ждали блокировку — тогда просто берём актуальный токен без повторного refresh(). Так refresh реально выполняется один раз. responseCount по цепочке priorResponse отсекает бесконечный цикл, а возврат null = «сдаться», OkHttp отдаст 401 наверх.

Задача 5. Напишите на OkHttp WebSocket-клиент с авто-реконнектом: при onFailure или ненормальном закрытии (код ≠ 1000) переподключаться с экспоненциальным backoff (1с → 2с → 4с → … → макс 30с) и джиттером; при успешном onOpen счётчик backoff сбрасывается. При close(1000) — не реконнектить. Отдавайте входящие сообщения через Flow.

Показать решение 
fun webSocketFlow(
    client: OkHttpClient,
    request: Request,
    scope: CoroutineScope,
): Flow<String> = callbackFlow {
    var attempt = 0
    var current: WebSocket? = null

    fun connect() {
        current = client.newWebSocket(request, object : WebSocketListener() {
            override fun onOpen(ws: WebSocket, response: Response) { attempt = 0 }
            override fun onMessage(ws: WebSocket, text: String) { trySend(text) }
            override fun onClosing(ws: WebSocket, code: Int, reason: String) {
                if (code == 1000) close() else scheduleReconnect()
            }
            override fun onFailure(ws: WebSocket, t: Throwable, r: Response?) {
                scheduleReconnect()
            }
        })
    }

    fun scheduleReconnect() {
        val delayMs = minOf(1_000L shl attempt, 30_000L) + Random.nextLong(0, 1_000)
        attempt++
        scope.launch { delay(delayMs); connect() }
    }

    connect()
    awaitClose { current?.close(1000, null) }
}

Объяснение: onOpen сбрасывает attempt — после успешного коннекта backoff начинается заново. 1_000L shl attempt даёт 1с→2с→4с…, minOf(..., 30_000L) ставит потолок, Random.nextLong(0, 1_000) добавляет джиттер. Код закрытия 1000 = нормально, канал закрываем без реконнекта; всё остальное и onFailure запускают повтор. awaitClose закрывает сокет при отмене Flow. На клиенте полезен pingInterval(...) для детекта мёртвого сокета, и один OkHttpClient переиспользуется.
