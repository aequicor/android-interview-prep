Когда вы возьмёте Profiler, а когда Macrobenchmark, и почему замер на debug-сборке бессмысленен?

Показать разбор

Профайлер — интерактивный поиск причины «почему медленно» здесь и сейчас. Macrobenchmark даёт воспроизводимую цифру для регрессий в CI («стало ли медленнее на N мс»). Гонять бенчмарк надо на release/profileable-сборке: debug отключает R8, не применяет Baseline Profile и держит ART в интерпретаторе/JIT, поэтому его цифры в разы хуже пользовательских и нерепрезентативны.

    override fun onCreate() {
        super.onCreate()
        if (BuildConfig.DEBUG) {
            StrictMode.setThreadPolicy(
                StrictMode.ThreadPolicy.Builder()
                    .detectDiskReads().detectDiskWrites()
                    .detectNetwork()            // I/O на main thread

                    .detectActivityLeaks()
                    .penaltyLog()
                    .build()
            )
        }
    }
}
Чем системный трейс (Perfetto / system trace) принципиально отличается от method tracing в CPU Profiler, и почему jank разбирают именно им? ★

Method tracing (Trace Java Methods / Sample Java Methods) инструментирует каждый вход и выход метода в вашем процессе. Это само по себе добавляет немалый overhead и искажает тайминги: найти hot-метод им можно, а понять, чем в этот момент занята система, — нет.

System trace через Perfetto пишет события ядра и всех процессов с очень низким overhead: потоки и их состояния (Running / Runnable / Sleeping / Uninterruptible), VSYNC и Choreographer, FrameTimeline с пометкой janky-кадров, биндер-транзакции, GC-паузы, работу RenderThread. Для jank это решающе: важно увидеть не только свой код, но и почему поток не выполнялся — ждал блокировку, вытеснен планировщиком, упёрся в биндер или в GC.

Поэтому jank и frozen frames разбирают по system trace, а method/sample tracing берут точечно — когда уже ясно, что время горит в коде вашего процесса. Свои участки подсвечивают через androidx.tracing: trace("label") { ... }.

Память: утечки, GC и heap dump

Утечка в JVM/ART — это объект, который больше не нужен, но удерживается живой ссылкой от GC root, поэтому сборщик мусора не может его освободить. На Android самый дорогой «лист» такой цепочки — Activity/Fragment/View: они тянут за собой весь UI-граф и порой десятки мегабайт. Накопление → рост памяти → паузы GC → в пределе OutOfMemoryError.

Как искать: LeakCanary → leak trace

LeakCanary ставит «сторож» (ObjectWatcher) на объекты, которые обязаны умереть (Activity после onDestroy, Fragment-view, ViewModel). Если объект жив спустя задержку и принудительный GC, число удержанных объектов превышает порог — LeakCanary снимает heap dump (.hprof), парсит его движком Shark и для каждого удержанного объекта строит кратчайший путь сильных ссылок от GC root — это и есть leak trace. В трейсе подсвечена подозрительная ссылка (~~~), с которой и начинается фикс.

Leak trace: цепочка сильных ссылок от GC root к утекшей Activity. Рвём слабое звено — static-ссылку
⚠️
WeakReference — не серебряная пуля. Слабая ссылка не мешает GC собрать объект, но если по цепочке есть хоть одна сильная ссылка — утечка останется. WeakReference уместна для кэшей и callback-ов на UI, но не «затыкает» архитектурную проблему удержания.

Память bitmap и heap dump вручную

С Android 8.0 пиксели bitmap живут в нативной куче (вне Java-heap), но всё равно учитываются как память процесса — большие незакрытые/непереиспользуемые bitmap быстро приводят к OOM. В Profiler делаем Capture heap dump: смотрим вкладку Apps Heap, сортируем по Retained Size, ищем дубли Activity (счётчик > 1 после ротаций — почти всегда утечка) и крупные массивы байт.

Как устроен GC в ART и почему «много мелких аллокаций» вредит плавности UI сильнее, чем одна большая утечка? ★

С Android 8 ART по умолчанию использует Concurrent Copying (CC) GC, а с Android 10 он стал ещё и поколенческим: большинство объектов умирает молодыми, поэтому частые молодые сборки обходят только недавно созданные объекты, не трогая старшее поколение. Сборка идёт конкурентно с приложением, так что Stop-The-World пауз почти нет — обычно две короткие паузы порядка единиц миллисекунд.

Из-за этого настоящая проблема плавности — не длина STW, а GC pressure. Чем больше аллокаций в горячем пути (в onBindViewHolder, на каждой рекомпозиции, автобоксинг, временные строки и коллекции), тем чаще запускается GC и тем больше он суммарно отъедает у кадра, дробя его и вызывая jank. Поэтому в горячих участках избегают лишних аллокаций: пулинг объектов, переиспользование буферов, отказ от автобоксинга.

Одна крупная утечка опасна иначе: она растит RSS и в пределе даёт OutOfMemoryError, но сама по себе не дёргает кадры.

Почему поток мелких аллокаций в onBindViewHolder бьёт по плавности сильнее, чем одна крупная утечка?

Показать разбор

С Android 8 ART использует Concurrent Copying GC (с Android 10 поколенческий): паузы конкурентные и короткие, поэтому проблема плавности — не длина STW, а GC pressure. Чем больше аллокаций в горячем пути, тем чаще запускается young GC и тем больше он суммарно отъедает у кадра, дробя его и давая jank. Крупная утечка опасна иначе: растит RSS и в пределе даёт OOM, но сама кадры не дёргает.

Эта схема объясняет связку из вопроса выше наглядно: молодые объекты собираются часто и дёшево, до старшего поколения доживают единицы, а паузы — конкурентные и короткие. Отсюда вывод: бьёт по кадру не длина одной паузы, а частота молодых сборок под потоком аллокаций.

Где смотреть GC. Частоту и длительность сборок видно в Memory Profiler (всплески на графике) и в Perfetto отдельными GC-слайсами на потоке HeapTaskDaemon. Резкий рост частоты young GC при скролле — прямой сигнал лишних аллокаций в горячем пути.
Когда ART выбирает не CC GC, и что меняет режим low-memory? ★

Concurrent Copying — дефолт, но ART не один на все случаи. На устройствах с малой памятью система может перейти на более компактный, но менее «плавный» сборщик и агрессивнее уплотнять кучу: цель смещается с минимальных пауз на минимальный footprint. Копирующий сборщик платит за скорость удвоением полупространства, поэтому там, где памяти в обрез, выгоднее экономия места ценой более заметных пауз.

Практический вывод для собеса: нельзя считать «паузы всегда по паре миллисекунд». На бюджетных устройствах под давлением памяти GC дороже и заметнее — поэтому перформанс-проверки гоняют не только на флагмане, но и на слабом железе, где GC pressure проявляется раньше.

Чем TTID отличается от TTFD и зачем вообще нужен ручной вызов reportFullyDrawn()?

Показать разбор

TTID — время до первого отрисованного кадра, его система определяет сама. TTFD — момент, когда контент реально готов (данные подгружены, плейсхолдеры заменены); этот момент знает только приложение, поэтому его сообщают вызовом reportFullyDrawn(). Иначе можно показать пустой каркас за 200 мс (отличный TTID), но тянуть данные 2 секунды (плохой TTFD) — а пользователь видит именно второе.

Метрика, которую видит пользователь и Play Store (Android Vitals). Различают три сценария по тому, сколько работы уже сделано.

WarmПроцесс жив, Activity уничтоженаПересоздать Activity, заново нарисоватьСредняя

Что нельзя делать на старте (и на main thread)

🛑
Главный убийца cold start. Тяжёлая инициализация в Application.onCreate: сеть, чтение диска/БД, синхронная инициализация SDK, рефлексия, «прогрев» аналитики. Всё это блокирует main thread до первого кадра. Любой блокирующий I/O на UI-потоке → ANR-риск и jank.

App Startup library — отложенная и упорядоченная инициализация

Многие библиотеки втихую добавляют свой ContentProvider, чтобы инициализироваться на старте — каждый провайдер стоит времени. Jetpack App Startup заменяет их одним общим провайдером и даёт явный граф зависимостей инициализаторов, часть из которых можно сделать ленивыми.

WorkManagerInitializer.kt
class AnalyticsInitializer : Initializer<Analytics> {
    override fun create(context: Context): Analytics =
        Analytics.init(context.applicationContext)
    // зависит от логгера — App Startup сам выстроит порядок
    override fun dependencies(): List<Class<out Initializer<*>>> =
        listOf(LoggerInitializer::class.java)
}

// ленивую инициализацию запускаем вручную, когда фича реально нужна:
val analytics = AppInitializer.getInstance(context)
    .initializeComponent(AnalyticsInitializer::class.java)

Baseline Profiles — главный «бесплатный» ускоритель

Baseline Profile — это список «горячих» классов и методов (старт, навигация, скролл), который кладётся в APK/AAB. На установке ART делает по нему AOT-компиляцию (Profile Guided Optimization), минуя интерпретацию и JIT для этих путей. По данным Google это даёт ~30% к скорости выполнения с первого запуска (у Google Maps — ~30% к среднему времени старта). Профиль генерится Macrobenchmark-тестом (BaselineProfileRule) и работает с первого запуска и после каждого обновления.

StartupBenchmark.kt
@RunWith(AndroidJUnit4::class)
class StartupBenchmark {
    @get:Rule val rule = MacrobenchmarkRule()

    @Test fun coldStartup() = rule.measureRepeated(
        packageName = "com.example.app",
        metrics = listOf(StartupTimingMetric()),
        iterations = 10,
        startupMode = StartupMode.COLD          // COLD / WARM / HOT
    ) {
        pressHome()
        startActivityAndWait()                   // ждём первый кадр
    }
}
💡
Связка для собеса. Macrobenchmark измеряет старт, Baseline Profile ускоряет его, а App Startup + отложенная инициализация убирают лишнюю работу с критического пути. Назвав все три вместе, вы показываете системное понимание.
Почему Baseline Profile ускоряет старт уже с первого запуска, а без него первые запуски медленные? ★

Без Baseline Profile свежеустановленное приложение исполняется в интерпретаторе. ART накапливает собственный профиль горячих методов локально и компилирует их AOT только в фоне — обычно когда устройство на зарядке и в idle. То есть первые запуски остаются медленными, пока JIT и фоновая компиляция не прогреются.

Baseline Profile кладёт список горячих классов и методов прямо в AAB, и при установке (а также после каждого обновления) ART делает по нему AOT-компиляцию заранее — Profile Guided Optimization. Поэтому ускорение есть уже на первом холодном старте; Google называет порядка 30% к скорости выполнения горячего кода.

Дополнительно Play может раздавать агрегированный Cloud Profile, собранный с других пользователей, но он применяется не мгновенно. Профиль в артефакте даёт эффект сразу и детерминированно. Проверять выигрыш нужно Macrobenchmark с CompilationMode.None() против CompilationMode.Partial(BaselineProfileMode.Require).

Производительность UI: jank и бюджет кадра

Перед чтением к повторению сегодня

Почему высокий средний FPS не противоречит ощутимому jank, и какую метрику измерять вместо среднего?

Показать разбор

FPS как среднее за секунду прячет отдельные провалы: можно держать 58 кадров в среднем, но один кадр длиной 120 мс даст заметный рывок — пользователь чувствует именно худшие кадры. Поэтому измеряют не средний FPS, а перцентили длительности кадров (P90/P99) и долю janky/frozen-кадров; их и отдают FrameTimingMetric в Macrobenchmark и FrameTimeline в Perfetto.

Чтобы анимация была плавной при 60 Гц, каждый кадр должен быть готов за ≈16,6 мс; для 90 Гц — ~11 мс, для 120 Гц — ~8 мс. Не уложился — Choreographer пропускает кадр (dropped frame), пользователь видит рывок. Это и есть jank. Кадр, опоздавший очень сильно (> ~700 мс), — frozen frame, его отдельно ругает Android Vitals.

sequenceDiagram participant V as VSYNC participant C as Choreographer participant U as UI Thread participant R as RenderThread V->>C: сигнал кадра (каждые 16мс / 8мс) C->>U: doFrame() U->>U: input + animation + measure/layout/draw U->>R: sync дерева отрисовки R->>R: GPU rendering + buffer swap Note over U,R: уложились в бюджет → плавно, опоздали → dropped frame
Пайплайн кадра. Затык чаще всего на UI Thread (тяжёлый layout/recomposition) или из-за overdraw на GPU

Частые причины jank и фиксы

  • Overdraw — один пиксель красится несколько раз (вложенные фоны). Включите Debug GPU Overdraw: цель — минимум красного. Уберите лишние background и держите иерархию плоской.
  • Работа на main thread в момент кадра — парсинг, БД, аллокации в onBindViewHolder/composable. Выноси в фон, кэшируй.
  • RecyclerView — стабильные id, DiffUtil вместо notifyDataSetChanged(), лёгкий onBindViewHolder, нет вложенных скроллов без prefetch.
  • Compose — лишние перекомпозиции: стабильные типы и key в LazyColumn, чтение состояния как можно ниже по дереву, derivedStateOf, remember тяжёлых вычислений, @Immutable/@Stable.

Механика кадра — отдельно. Как именно VSYNC, Choreographer и RenderThread складываются в конвейер кадра, подробно разобрано в главе Рендеринг и кадры. Здесь нас интересует профилирование: где конвейер встаёт и как это измерить.
Что именно делает Choreographer при пропуске кадра, и почему высокий средний FPS не противоречит ощутимому jank? ★

Choreographer подписан на сигнал VSYNC от дисплея и на каждый вызывает doFrame(). Внутри по порядку обрабатываются input → animation (включая колбэки) → measure/layout/draw, после чего дерево отрисовки синхронизируется в RenderThread. Если doFrame() не уложился в бюджет (≈16,6 мс при 60 Гц), кадр к ближайшему VSYNC не готов — на экране остаётся прежний, и в logcat появляется «Skipped N frames! The application may be doing too much work on its main thread».

FPS как среднее за секунду это прячет: можно держать 58 кадров в среднем, но один кадр длиной 120 мс даёт заметный рывок. Пользователь чувствует именно худшие кадры, а не среднее. Поэтому метрика — не средний FPS, а перцентили длительности кадров (P90/P99) и доля janky/frozen frames; их и отдают FrameTimingMetric в Macrobenchmark и FrameTimeline в Perfetto.

ANR: типы, пороги и разбор трейса

Перед чтением к повторению сегодня

Почему синхронный I/O в onReceive у BroadcastReceiver грозит ANR, и как правильно вынести работу?

Показать разбор

И onReceive, и колбэки Service по умолчанию исполняются на главном потоке, а у broadcast жёсткий порог (10 с в foreground). Тяжёлый onReceive не только сам рискует упереться в этот таймаут, но и блокирует обработку ввода. Работу выносят в goAsync(), корутину или WorkManager, не держа main thread.

ANR (Application Not Responding) система рисует, когда главный поток слишком долго не реагирует — за этим следит watchdog в ActivityManager. Это не отдельный баг, а симптом: main thread заблокирован, и ОС предлагает пользователю закрыть приложение. На собесе по перформансу ANR спрашивают почти всегда, причём конкретику: пороги и как читать трейс.

Типы ANR и пороги

⚠️
Broadcast и Service крутятся на main thread. И onReceive, и колбэки Service по умолчанию исполняются на главном потоке. Поэтому «тяжёлый» onReceive не только сам рискует ANR, но и блокирует обработку ввода — выноси работу в goAsync(), корутину или WorkManager.

Как читать ANR-trace

В трейсе главное — состояние main thread в момент зависания:

  • В своём коде (стек в ваших методах, состояние RUNNABLE) — ищем тяжёлую операцию: парсинг, БД, цикл, регэксп.
  • В Object.wait / park или держит-ждёт монитор — дедлок или contention на блокировке: два потока ждут друг друга.
  • В биндер-транзакции — синхронный IPC: ждём ответа другого процесса (часто system_server под нагрузкой).

Диагностика в проде

ANR редко ловится локально, поэтому стартуют с Android Vitals в Play Console: там агрегированы ANR-rate и сами трейсы со стеком всех потоков. Прямо из приложения пост-фактум причину даёт ApplicationExitInfo (getHistoricalProcessExitReasons, REASON_ANR); на устройстве есть /data/anr/traces.txt; сторонние APM (Crashlytics, Sentry) умеют захватывать ANR. Локально воспроизводят нагрузкой, медленным хранилищем и StrictMode с penaltyLog(), а корень ищут в Perfetto.

Чем frozen frame отличается от ANR, и почему длинный кадр — это не всегда ANR? ★

Frozen frame — это кадр, который рисовался дольше ~700 мс: метрика плавности из Android Vitals. Он портит ощущение, но приложение продолжает реагировать. ANR — это другой порог и другой watchdog: главный поток не обработал ввод за 5 секунд (или компонент не уложился в свой таймаут). Frozen frame может случиться без единого ANR, и наоборот — ANR бывает без рендера вообще (например, зависший onReceive).

Связь между ними причинная: и то, и другое — следствие занятого main thread. Длинная работа на UI-потоке сперва даёт цепочку janky/frozen кадров, а если затягивается за 5 секунд и в это время прилетает тап — превращается в ANR. Поэтому чинят их одинаково (вынос работы с main thread), но измеряют разными метриками: FrameTimeline для кадров, ANR-rate для зависаний.

Размер APK/AAB: R8 и shrinking

Перед чтением к повторению сегодня

Включили isMinifyEnabled = true — и приложение падает только в release. Какой класс кода чаще всего ломает R8 и как это лечить?

Показать разбор

R8 удаляет недостижимый код и переименовывает классы/члены. Код, к которому обращаются по имени, — рефлексия, Gson/Moshi по полям, kotlinx.serialization, JNI, классы из манифеста — для R8 выглядит «недостижимым», и он его выпиливает или переименовывает, отсюда краш только в release. Лечение — точечные -keep-правила; для читаемых стектрейсов хранить и загружать mapping.txt в Play Console.

R8 — стандартный компилятор-оптимизатор Android (заменил ProGuard), работает за один проход прямо над DEX и делает четыре вещи: tree shaking (удаляет недостижимый код), optimization (инлайнинг, слияние классов, удаление мёртвых веток), obfuscation (короткие имена → меньше строковая таблица) и desugaring. Включается в release одной строкой.

app/build.gradle.kts
android {
    buildTypes {
        release {
            isMinifyEnabled = true          // R8: shrinking + obfuscation + optimization
            isShrinkResources = true        // выкинуть неиспользуемые ресурсы (только с minify)
            proguardFiles(
                getDefaultProguardFile("proguard-android-optimize.txt"),
                "proguard-rules.pro"        // keep-правила для рефлексии/Gson/JNI
            )
        }
    }
}
⚠️
Подводный камень R8. Код, к которому обращаются по имени (рефлексия, Gson/Moshi по полям, JNI, имена в манифесте), R8 может удалить или переименовать → краш только в release. Лекарство — точечные -keep-правила. Для деобфускации стектрейсов сохраняй mapping.txt и загружай его в Play Console.

Resource shrinking, App Bundle и динамические модули

После code shrinking R8/AGP удаляет неиспользуемые ресурсы (строки, drawable). Android App Bundle (.aab) — формат публикации: Play собирает оптимизированные split-APK под конкретное устройство (его плотность, ABI, язык), так что пользователь не качает чужие xxxhdpi-ассеты и нативные либы под другие ABI. Dynamic Feature Modules грузятся по требованию (Play Feature Delivery) — крупные редкие фичи не утяжеляют базовую установку.

Найти, что именно раздувает сборку, помогает APK Analyzer (Build → Analyze APK): он показывает вклад DEX, ресурсов и нативных либ и умеет делать diff двух APK — удобно ловить рост размера между релизами.

Чем full mode R8 отличается от ProGuard-совместимого режима, и какой класс багов он добавляет в release? ★

R8 по умолчанию работает в full mode (android.enableR8.fullMode=true) и оптимизирует агрессивнее ProGuard: например, может предполагать, что у класса нет конструктора по умолчанию, удалять и переименовывать классы без видимых обращений, вольнее обращаться с реализациями интерфейсов и дженериками. ProGuard-совместимый режим делает более консервативные допущения.

Плата за агрессивность — краши только в release у кода, к которому обращаются по имени: рефлексия, сериализация (Gson/Moshi через TypeToken и поля, kotlinx.serialization), DI по рефлексии, классы из манифеста, JNI. Без точечных -keep/-keepclassmembers/-keepattributes (Signature, RuntimeVisibleAnnotations) такой код выпиливается или переименовывается — и падает NoSuchMethodException, NPE или невалидный JSON.

Поэтому release-сборку обязательно прогоняют end-to-end, а стектрейсы деобфусцируют через mapping.txt в Play Console. Современные библиотеки поставляют consumer-keep-правила прямо в AAR, чтобы это не приходилось писать руками.

Сеть и батарея

Радиомодуль — один из главных потребителей энергии: после передачи он ещё держится в высоком энергосостоянии («tail energy»). Поэтому дешевле пачка запросов раз в N минут, чем десять частых, и обязателен кэш, чтобы не ходить в сеть зря.

  • Батчинг и дедупликация запросов; HTTP-кэш OkHttp; не поллить, если есть push.
  • WorkManager constraints — откладывать фоновую работу до Wi-Fi, зарядки, idle; система сама батчит по Doze-окнам.
  • Doze / App Standby — в режиме ожидания система душит сеть, alarm и wakelock-и; рассчитывай на отложенное выполнение, а не на «прямо сейчас в фоне».
  • Energy Profiler — ищет лишние wakelock-и, частые alarm и запросы геолокации.
SyncWork.kt
val constraints = Constraints.Builder()
    .setRequiredNetworkType(NetworkType.UNMETERED)   // только Wi-Fi
    .setRequiresCharging(true)                        // и на зарядке
    .build()

val sync = PeriodicWorkRequestBuilder<SyncWorker>(6, TimeUnit.HOURS)
    .setConstraints(constraints)                      // система сама подберёт Doze-окно
    .build()

WorkManager.getInstance(context).enqueueUniquePeriodicWork(
    "sync", ExistingPeriodicWorkPolicy.KEEP, sync
)
ℹ️
Подробнее о фоне. Doze, App Standby, expedited work и выбор между WorkManager и foreground service разобраны в главе Фоновая работа. Здесь это в фокусе профилирования: лишние wakelock-и и сетевые «будильники» как источник разряда.
Что происходит с фоновой работой и сетью в Doze, и почему «просто запустить WorkManager» не гарантирует немедленного выполнения? ★

В Doze (экран выключен, устройство неподвижно и не на зарядке достаточно долго) система откладывает сетевой доступ, jobs/syncs и обычные alarm-ы, а wakelock-и игнорируются. Накопленную работу она выпускает периодическими maintenance-окнами, интервал между которыми по мере углубления Doze растёт. App Standby по тому же принципу душит редко используемые приложения.

Поэтому WorkManager с обычным OneTime/PeriodicWorkRequest гарантирует выполнение, но не «прямо сейчас»: задача запустится в ближайшее подходящее окно с учётом constraints. Для несрочной синхронизации это и есть правильное поведение — она должна быть отложенной и идемпотентной.

Если работа действительно срочная и инициирована пользователем — нужен expedited work (setExpedited, под капотом foreground-исполнение с квотой) или полноценный foreground service. Для гарантированного времени — setAndAllowWhileIdle/setExactAndAllowWhileIdle у AlarmManager, но злоупотребление ими бьёт по батарее и ограничено системой.

Системный трейс на практике: читаем Perfetto

Назвать Perfetto в списке инструментов — половина дела. Senior-вопрос звучит иначе: «снял трейс — что дальше?». System trace берут вместо method tracing, когда нужен низкий overhead и видна вся система, а не только ваш процесс. Снимают его из Profiler (System Trace) или через perfetto CLI, открывают в ui.perfetto.dev.

Что искать в трейсе

  • Состояния потоков — Running (на CPU), Runnable (готов, но не на CPU), Sleeping, Uninterruptible (D, обычно блокирующий I/O). Главный сигнал — почему поток не Running.
  • VSYNC и Choreographer.doFrame — рамки кадров; рядом FrameTimeline помечает кадры как Janky/Frozen с причиной.
  • RenderThread — если затык там, дело в GPU/отрисовке (overdraw, тяжёлые шейдеры), а не в вашем коде.
  • GC-слайсы и binder-транзакции — паузы GC дробят кадр; долгая binder-транзакция = синхронный IPC, ждём другой процесс.

Типовые паттерны разбора одного janky-кадра

Нашли красный (janky) кадр на FrameTimeline — кликаем, смотрим, на чём стоял main thread в его границах. Дальше три частых исхода: поток Runnable, но не Running — его вытеснил планировщик (CPU занят другим, частый случай на слабом железе); один долгий слайс в вашем коде — несём работу в фон; ожидание биндера — синхронный вызов в другой процесс, кэшируем результат или делаем асинхронно.

💡
Свои метки в трейсе. Чтобы видеть собственные участки рядом с системными, оборачивай их в androidx.tracing: метка появится отдельным слайсом ровно на том потоке, где выполнялась.
Repo.kt
import androidx.tracing.trace

fun loadFeed(): List<Item> = trace("loadFeed") {
    val raw = trace("db.query") { dao.queryAll() }      // отдельный слайс в Perfetto
    trace("map.toUi") { raw.map { it.toUi() } }          // видно, что дороже
}
Почему method tracing нельзя доверять для замера абсолютного времени, а system trace — можно? ★

Trace Java Methods инструментирует каждый вход и выход метода, добавляя в hot-путь столько накладных расходов, что абсолютные тайминги раздуваются, а соотношение между методами искажается тем сильнее, чем чаще метод вызывается (мелкие частые методы «толстеют» непропорционально). Им хорошо находить относительно горячее место, но не утверждать «это заняло 12 мс».

System trace пишет уже существующие события ядра и заранее расставленные точки (ATrace) с минимальным overhead — он не оборачивает каждый вызов, а слушает планировщик и явные слайсы. Поэтому его длительности близки к реальным, и именно по нему меряют, уложился ли кадр в бюджет. Отсюда практика: jank и бюджет кадра — system trace; «какой из двух методов горячее в моём процессе» — sample/method tracing точечно.

Частые вопросы интервьюера

Приложение долго стартует — как вы будете разбирать проблему по шагам? Middle+

1) Замерить воспроизводимо: Macrobenchmark StartupTimingMetric в режиме COLD на release-сборке, плюс посмотреть Android Vitals. 2) Снять system trace в Profiler/Perfetto и разобрать критический путь: что тяжёлого в Application.onCreate, какие ContentProvider-ы инициализируются. 3) Фиксы: убрать с main thread синхронный I/O и прогрев SDK, перевести на ленивую инициализацию через App Startup, подключить Baseline Profile. 4) Перемерить тем же бенчмарком и сравнить TTID/TTFD. Ключевое — назвать cold/warm/hot и подтвердить эффект цифрой.

Что такое утечка памяти в Android и как её найти? Middle

Утечка — объект, который не нужен, но удерживается живой ссылкой от GC root, поэтому не собирается GC; на Android особенно дорого течёт Activity/View. Поиск: в debug — LeakCanary, который сам снимает heap dump и строит leak trace (кратчайший путь сильных ссылок от root к объекту); вручную — Memory Profiler, Capture heap dump, сортировка по Retained Size и поиск дублей Activity (счётчик > 1 после ротаций). Дальше по trace находим удерживающую ссылку и рвём её.

Что такое Baseline Profiles и сколько они дают? Senior

Это список горячих классов/методов критических сценариев (старт, навигация, скролл), упакованный в AAB. На установке ART делает по нему AOT-компиляцию (Profile Guided Optimization), пропуская интерпретацию и JIT для этих путей — выигрыш с первого запуска и после каждого апдейта, без изменения кода. По данным Google — порядка 30% к скорости выполнения горячего кода (у Maps ~30% к среднему старту). Генерируются Macrobenchmark-правилом BaselineProfileRule, эффект подтверждается тем же Macrobenchmark.

Чему равен бюджет на кадр и что такое jank? Middle

При 60 Гц — ~16,6 мс на кадр, при 120 Гц — ~8 мс. Если UI/Render не успели подготовить кадр к VSYNC, Choreographer его пропускает (dropped frame), пользователь видит рывок — это jank; очень опоздавший кадр (> ~700 мс) — frozen frame. Чинят выносом тяжёлой работы с main thread, снижением overdraw, оптимизацией списков (DiffUtil) и перекомпозиций в Compose. Замеры — Perfetto FrameTimeline и Macrobenchmark FrameTimingMetric.

В чём разница между Macrobenchmark и Microbenchmark? Senior

Microbenchmark меряет крошечный кусок кода (наносекунды): отдельный поток JVM, прогрев, отсев JIT, повторы — для сравнения двух реализаций алгоритма/парсера. Macrobenchmark меряет сквозные сценарии (старт, скролл) на реальном устройстве и обязательно на release/profileable-сборке, выдаёт StartupTiming/FrameTiming и трейсы. Правило: debug-замеры производительности бессмысленны, потому что debug отключает оптимизации и медленнее в разы.

Что делает R8 и какие у него подводные камни? Middle+

R8 в одном проходе делает shrinking (удаление недостижимого кода), optimization (инлайнинг, слияние), obfuscation (короткие имена) и desugaring, плюс с ним работает resource shrinking. Подводный камень — код, доступный по имени (рефлексия, Gson/Moshi, JNI, классы из манифеста): R8 может его удалить/переименовать, и краш проявится только в release. Решение — точечные -keep-правила; для читаемых стектрейсов хранить и загружать mapping.txt.

Почему bitmap так часто причина OOM и как с этим бороться? Middle+

Bitmap — это width × height × байт_на_пиксель: полноэкранная картинка ARGB_8888 — мегабайты. С Android 8 пиксели в нативной куче, но всё равно считаются памятью процесса, и пара несжатых картинок легко даёт OOM. Лечение: декодировать с inSampleSize/downscale под размер View, использовать загрузчики (Coil/Glide) с их кэшем и пулом, выбирать RGB_565 где допустимо, не держать ссылки на bitmap уничтоженных экранов. В heap dump такие байтовые массивы видны по большому Retained Size.

Зачем WorkManager constraints и причём тут Doze? Senior

Constraints (Wi-Fi, зарядка, idle, не-low-battery) позволяют отложить фоновую работу до выгодного момента и сэкономить батарею — система батчит такие задачи. Doze (когда экран выключен и устройство неподвижно) принудительно глушит сеть, alarm и wakelock-и, выпуская работу окнами обслуживания; App Standby душит редко используемые приложения. Вывод для архитектуры: фоновая синхронизация должна быть отложенной и идемпотентной, а не рассчитанной на «выполнить ровно сейчас». Для немедленной критичной работы — foreground service или expedited work.

Как диагностировать ANR в проде, если он не воспроизводится локально? Senior

Старт — Android Vitals в Play Console: там агрегированы ANR-rate и сами трейсы со стеком всех потоков в момент зависания. По трейсу смотрят состояние main thread: в своём коде — ищем тяжёлую операцию; в Object.wait/park или держит-ждёт монитор — дедлок или contention; в биндер-транзакции — синхронный IPC. Прямо из приложения причину пост-фактум даёт ApplicationExitInfo (getHistoricalProcessExitReasons, REASON_ANR); на устройстве — /data/anr/traces.txt; сторонние APM (Crashlytics, Sentry) умеют захватывать ANR. Локально воспроизводят нагрузкой и медленным хранилищем, ловят StrictMode с penaltyLog(), корень ищут в Perfetto.

В чём разница между TTID и TTFD и зачем нужен reportFullyDrawn()? Middle+

TTID (Time To Initial Display) — время до первого полностью отрисованного кадра, его система определяет сама. TTFD (Time To Full Display) — время до момента, когда контент реально готов к взаимодействию (данные загружены, плейсхолдеры заменены реальными элементами). Этот момент знает только приложение, поэтому его явно сообщают вызовом Activity.reportFullyDrawn(). Это важно, потому что можно показать пустой каркас за 200 мс (отличный TTID), но подтянуть данные за 2 секунды (плохой TTFD) — пользователь видит именно второе. Macrobenchmark-метрики ориентируются на оба значения; reportFullyDrawn ещё и помогает ART точнее собрать профиль горячего кода старта.

Счётчик Activity > 1 в heap dump — всегда ли это утечка? Senior

После поворота старая Activity уничтожается и должна быть собрана GC, поэтому устойчивый счётчик экземпляров одной Activity больше 1 (особенно растущий с числом ротаций) — почти всегда утечка. Но бывают ложные срабатывания: объект ещё не собран просто потому, что GC не запускался — перед снятием heap dump нужно форсировать сборку (LeakCanary делает это сам, вручную — кнопка GC в Profiler) и только потом смотреть Retained Size. Короткоживущее дублирование в момент самого перехода — норма. Достоверный признак течи — объект жив после принудительного GC и держится сильной ссылкой от GC root; именно эту цепочку показывает leak trace.

Можно ли держать LeakCanary в release-сборке? Middle+

Смысла нет: leakcanary-android по дизайну debug-only. Снятие heap dump через Debug.dumpHprofData замораживает приложение на сотни миллисекунд–секунды (Stop-The-World для всего процесса), парсинг hprof тяжёлый, а сам дамп содержит данные пользователя — в проде это и UX-фриз, и нарушение приватности. Для продакшен-детектирования у Square есть отдельный путь (ручной триггеринг или сбор только статистики), но классический автоматический режим оставляют в debug/QA. В release утечки и рост памяти ловят иначе: Android Vitals по памяти/OOM, APM, выборочные heap-метрики, нагрузочное тестирование на QA-сборке с включённым LeakCanary.

Что такое overdraw, как его измерить и какой уровень приемлем? Middle

Overdraw — когда один пиксель за кадр закрашивается несколько раз (вложенные непрозрачные фоны, перекрывающиеся View): GPU делает лишнюю работу, кадр дорожает. Меряют через Developer Options → Debug GPU Overdraw: экран раскрашивается по слоям — без цвета (0x, идеально), синий (1x), зелёный (2x), светло-красный (3x), тёмно-красный (4x+). Цель — преобладание синего, минимум красного. Чинят удалением лишних background у контейнеров, плоской иерархией, clipRect/clipToPadding, отказом от полноэкранных непрозрачных фонов поверх друг друга. В Compose аналог — лишние Box с фоном и многократные Modifier.background.

Чем профилирование debug-сборки опасно как источник выводов? Middle+

Debug-сборка отключает оптимизации и даёт заведомо неверные цифры: код не прошёл R8 (нет инлайнинга и shrinking), нет Baseline Profile, ART чаще в интерпретаторе/JIT, включён debuggable-флаг (он сам по себе замедляет ART), могут быть активны StrictMode и логирование. В итоге старт, скролл и аллокации в debug в разы хуже, чем у пользователя в release — оптимизировать «по debug» значит гонять призраков. Поэтому Macrobenchmark требует release или profileable-сборку (android:profileable), а Microbenchmark прогревает JIT и отсекает выбросы. Правило для собеса: любые перформанс-замеры — только на release/profileable, debug годится лишь для функциональной отладки и LeakCanary.

Задача 5. Напишите Macrobenchmark-тест, который сравнивает холодный старт без Baseline Profile и с ним, чтобы доказать выигрыш цифрой. Укажите, какие CompilationMode используются и почему нельзя сравнивать на debug-сборке.

Показать решение 
@RunWith(AndroidJUnit4::class)
class StartupCompilationBenchmark {
    @get:Rule val rule = MacrobenchmarkRule()

    // 1) чистый интерпретатор — как первый запуск без профиля
    @Test fun startupNoProfile() = measure(CompilationMode.None())

    // 2) применён Baseline Profile (требуем, иначе тест упадёт)
    @Test fun startupBaselineProfile() =
        measure(CompilationMode.Partial(BaselineProfileMode.Require))

    private fun measure(mode: CompilationMode) = rule.measureRepeated(
        packageName = "com.example.app",
        metrics = listOf(StartupTimingMetric()),     // timeToInitialDisplayMs
        compilationMode = mode,
        startupMode = StartupMode.COLD,
        iterations = 10                               // ≥10 для устойчивой медианы
    ) {
        pressHome()
        startActivityAndWait()
    }
}

Объяснение: два прогона одного StartupTimingMetric с разным CompilationMode — None() (как первый запуск без профиля) против Partial(BaselineProfileMode.Require) (профиль применён). Сравниваем timeToInitialDisplayMs по медиане и P90. Почему не debug: Macrobenchmark требует release или profileable non-debuggable APK — на debuggable ART работает в разы медленнее, а Baseline Profile и R8 не применяются, так что цифры нерепрезентативны (тест на debuggable выдаст ошибку). Меряем на реальном стабильном устройстве, iterations ≥ 10.
