Сеть

HTTP (HyperText Transfer Protocol — протокол передачи гипертекста) и его семантика, TLS/HTTPS (Transport Layer Security и защищённый HTTP — шифрованное соединение), рукопожатие и доверие к сертификатам, стек OkHttp + Retrofit (HTTP-клиент и клиент API, программного интерфейса), обработка сетевых ошибок и сериализация. Один из самых частых блоков: интервьюер проверяет, понимаешь ли ты, что происходит «под капотом» между нажатием кнопки и появлением данных на экране.

HTTP: методы, статусы, семантика

HTTP — текстовый (в HTTP/2 — бинарный) запрос-ответный протокол. Запрос = метод + URL + заголовки + тело; ответ = статус + заголовки + тело. Ключевые свойства методов, которые спрашивают: безопасность (safe — не меняет состояние) и идемпотентность (повтор даёт тот же эффект).

МетодНазначениеSafeИдемпотентный
GETПолучить ресурсдада
HEADGET без тела (только заголовки)дада
POSTСоздать/выполнить действиенетнет
PUTЗаменить ресурс целикомнетда
PATCHЧастичное изменениенетнет
DELETEУдалить ресурснетда

Почему это важно: повторять (retry) автоматически можно только идемпотентные запросы. Повторный POST «создать заказ» создаст дубль — поэтому для него вводят ключ идемпотентности (Idempotency-Key), а OkHttp по умолчанию не ретраит не-идемпотентные методы при разрыве.

💡
401 vs 403. 401 Unauthorized — «я не знаю, кто ты» (нет/протух токен → обнови и повтори). 403 Forbidden — «я знаю, кто ты, но тебе сюда нельзя» (повтор бессмыслен). Разное поведение на клиенте.

Кэширование: ETag и Cache-Control

Сервер отдаёт Cache-Control: max-age=60 (сколько секунд ответ «свежий») и/или валидатор ETag — хэш версии ресурса. Когда срок истёк, клиент шлёт условный запрос с If-None-Match: "etag". Если ресурс не менялся, сервер отвечает 304 Not Modified без тела — экономия трафика и батареи.

// Запрос -> первый ответ
// < HTTP/1.1 200 OK
// < ETag: "v3-abc"
// < Cache-Control: max-age=60

// Повторный запрос после устаревания
// > GET /profile
// > If-None-Match: "v3-abc"
// < HTTP/1.1 304 Not Modified   (тело не передаётся — берём из кэша)

Дополнительно: no-store — не кэшировать вообще; no-cache — кэшировать, но всегда перепроверять; private — только клиентский кэш, не прокси. Cookie/Set-Cookie переносят сессионное состояние, но в мобильных API чаще используют токены в Authorization: Bearer (stateless), а не cookie.

HTTP/2 — общий фундамент: бинарные фреймы, мультиплексирование (много запросов в одном TCP-соединении без head-of-line блокировки уровня HTTP), сжатие заголовков HPACK, server push. OkHttp поддерживает HTTP/2 автоматически (через ALPN при TLS) — на одном соединении параллельно идут десятки запросов, что снижает overhead на установку TCP/TLS.

TLS/HTTPS: рукопожатие и доверие

HTTPS = HTTP поверх TLS. TLS решает три задачи: конфиденциальность (шифрование), целостность (MAC/AEAD), аутентификация сервера (сертификат). Главная идея, которую ждут на собесе: асимметричная криптография используется только при рукопожатии, чтобы безопасно договориться об общем симметричном ключе сессии, а дальше весь трафик шифруется быстрым симметричным алгоритмом (AES-GCM/ChaCha20).

СимметричноеАсимметричное
Ключиодин общий секретпара: публичный + приватный
Скоростьбыстро (массив данных)медленно (дорогая математика)
Роль в TLSшифрование данных сессииаутентификация + обмен ключом при handshake

Рукопожатие (TLS 1.3)

sequenceDiagram participant C as Клиент participant S as Сервер C->>S: ClientHello (версии TLS, cipher suites, key_share, client random) S->>S: Выбор cipher suite, генерация ключа сессии S->>C: ServerHello (выбранный suite, key_share, server random) S->>C: Certificate + CertificateVerify (зашифровано) S->>C: Finished Note over C: Проверка цепочки сертификата до доверенного CA C->>S: Finished Note over C,S: Установлен общий симметричный ключ — данные шифруются C->>S: HTTP-запрос (зашифрован, application data) S->>C: HTTP-ответ (зашифрован)
TLS 1.3: 1-RTT — ключ согласуется за один круг, после ServerHello всё зашифровано (включая сам сертификат)

В TLS 1.3 клиент шлёт свой key_share (половину эфемерного Diffie-Hellman) уже в ClientHello, поэтому handshake укладывается в один round-trip (в TLS 1.2 их было два). Общий секрет вычисляется по DH — приватные ключи по сети не передаются, что даёт forward secrecy: даже компрометация приватного ключа сервера в будущем не расшифрует старые сессии.

Сертификаты и цепочка доверия

Сертификат (X.509) связывает домен с публичным ключом и подписан удостоверяющим центром (CA). Клиент проверяет цепочку: листовой сертификат сервера → промежуточный CA → корневой CA, который уже лежит в системном trust store устройства. Если подпись валидна, домен совпадает и срок не истёк — серверу доверяют.

flowchart LR Leaf["Сертификат сервера (домен + публичный ключ)"] --> Inter["Промежуточный CA"] Inter --> Root["Корневой CA (в trust store ОС)"] Root -.подписывает.-> Inter Inter -.подписывает.-> Leaf
Цепочка доверия: доверие «спускается» от предустановленного корневого CA к сертификату сервера
🛑
MITM. Если злоумышленник подсунет свой CA в trust store (rooted-устройство, корпоративный прокси, малварь), он сможет расшифровать «защищённый» трафик. Защита прикладного уровня от этого — pinning.

Certificate pinning vs public-key pinning, mTLS

Pinning — приложение доверяет не «любому валидному CA», а конкретному заранее зашитому отпечатку. Два варианта:

Certificate pinningPublic-key pinning (SPKI)
Что пинятхэш всего сертификатахэш публичного ключа (SubjectPublicKeyInfo)
Ротация сертификаталомается при любой пере-выпискепереживает, если ключ тот же
Рекомендацияхрупкопредпочтительно (так делает OkHttp CertificatePinner)

OkHttp CertificatePinner на самом деле пинит именно SPKI-хэш (base64 SHA-256), несмотря на имя класса. Обязательно держи backup-пин (резервный ключ): иначе при экстренной смене ключа приложение потеряет связь до выката апдейта.

val pinner = CertificatePinner.Builder()
    .add("api.example.com", "sha256/AAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAA=")
    .add("api.example.com", "sha256/BBBBBBBBBBBBBBBBBBBBBBBBBBBBBBBBBBBBBBBBBBB=") // backup
    .build()

val client = OkHttpClient.Builder()
    .certificatePinner(pinner)
    .build()

mTLS (mutual TLS) — двусторонняя аутентификация: не только клиент проверяет сервер, но и сервер требует клиентский сертификат. Применяется в банковских/корпоративных API и B2B — приложение предъявляет свой сертификат из KeyStore.

OkHttp: клиент, интерсепторы, пул

OkHttpClient — тяжёлый объект: держит пул соединений, пул потоков (dispatcher) и кэш. Создавай его один раз (синглтон/DI) и переиспользуй через newBuilder() — так клиенты делят connection pool и кэш.

NetworkModule.kt
val client = OkHttpClient.Builder()
    .connectTimeout(10, TimeUnit.SECONDS)   // установка TCP+TLS
    .readTimeout(15, TimeUnit.SECONDS)      // ожидание байтов ответа
    .writeTimeout(15, TimeUnit.SECONDS)     // отправка тела
    .callTimeout(30, TimeUnit.SECONDS)      // суммарный лимит на весь вызов
    .retryOnConnectionFailure(true)         // retry при сбое соединения (idempotent)
    .connectionPool(ConnectionPool(5, 5, TimeUnit.MINUTES))
    .cache(Cache(File(cacheDir, "http"), 10L * 1024 * 1024)) // 10 МБ
    .addInterceptor(AuthInterceptor(tokenStore))   // application
    .addNetworkInterceptor(StethoLikeLogger())     // network
    .build()

Application vs Network интерсепторы

Это любимый вопрос. Интерсептор — это Interceptor с методом intercept(chain), который может изменить запрос, вызвать chain.proceed() и изменить ответ. Разница — в месте в цепочке.

Application (addInterceptor)Network (addNetworkInterceptor)
Положениемежду приложением и ядром OkHttpмежду ядром OkHttp и сетью
Вызовов на 1 запросровно один (даже при кэш-хите)по разу на каждый редирект/retry; не вызывается при отдаче из кэша
Видит редиректынет (только финальный ответ)да (каждый «прыжок»)
Доступ к Connectionнетда (IP, TLS-параметры)
Типичное применениеauth-токен, общие заголовки, верхнеуровневое логированиелогирование «по проводу», подмена/сжатие, метрики на хоп
 
class AuthInterceptor(private val tokens: TokenStore) : Interceptor {
    override fun intercept(chain: Interceptor.Chain): Response {
        val request = chain.request().newBuilder()
            .header("Authorization", "Bearer ${tokens.access()}")
            .build()
        val response = chain.proceed(request)

        // прозрачное обновление токена на 401 (без рекурсии)
        if (response.code == 401 && tokens.refresh()) {
            response.close()
            val retried = request.newBuilder()
                .header("Authorization", "Bearer ${tokens.access()}")
                .build()
            return chain.proceed(retried)
        }
        return response
    }
}
⚠️
Authenticator vs Interceptor. Для реактивного обновления токена на 401 каноничнее OkHttpClient.authenticator(): OkHttp сам вызовет его при 401 и встроит ретрай, исключив гонки. В интерсепторе обновление токена легко получить дубли запросов — синхронизируй доступ к refresh.

Connection pool переиспользует keep-alive соединения: не тратится время на TCP+TLS для каждого запроса. EventListener даёт тайминги по фазам (DNS, connect, TLS, время до первого байта) — основа сетевых метрик и диагностики «почему медленно».

val listener = object : EventListener() {
    override fun dnsEnd(call: Call, domainName: String, results: List<InetAddress>) { /* ... */ }
    override fun secureConnectEnd(call: Call, handshake: Handshake?) { /* TLS готов */ }
    override fun responseHeadersStart(call: Call) { /* TTFB */ }
}
val client = base.newBuilder().eventListener(listener).build()

Retrofit: конвертеры, suspend, обёртки

Retrofit превращает аннотированный Kotlin-интерфейс в реализацию HTTP-клиента. Под капотом — OkHttp; сверху — конвертеры (тело ↔ объект) и call-адаптеры (тип возврата: Call, suspend, Flow, кастомный Result).

Api.kt
interface UserApi {
    @GET("users/{id}")
    suspend fun getUser(@Path("id") id: String): UserDto          // тело напрямую

    @GET("users/{id}")
    suspend fun getUserResponse(@Path("id") id: String): Response<UserDto> // + код/заголовки

    @POST("users")
    suspend fun create(@Body body: CreateUserDto): UserDto
}

val retrofit = Retrofit.Builder()
    .baseUrl("https://api.example.com/")
    .client(okHttpClient)
    .addConverterFactory(json.asConverterFactory("application/json".toMediaType()))
    .build()

val api = retrofit.create<UserApi>()

suspend-функции: начиная с Retrofit 2.6, метод можно объявить suspend — Retrofit сам адаптирует Call в корутину и переключает на IO-диспетчер OkHttp. Возврат UserDto — бросит исключение на не-2xx; возврат Response<UserDto> — отдаст код и тело, давая контроль над ошибками.

Оборачивание в Result/sealed

Голые исключения в data-слое неудобны. Два пути: (1) кастомный CallAdapter, чтобы интерфейс возвращал ApiResult<T> прямо из Retrofit; (2) обернуть вызов в репозитории через runCatching/try-catch и маппить в доменный sealed-тип. Второй проще и переносим между источниками — его и покажем в следующем разделе.

💡
DTO ≠ domain. Возвращай из API *Dto (поля как в JSON), а в репозитории маппь в доменную модель. Так контракт сервера не протекает в UI, а смена формата не ломает весь код.

Обработка ошибок и устойчивость

Сетевой вызов падает множеством способов: нет связи (IOException/UnknownHostException), HTTP-ошибка (HttpException / не-2xx), таймаут (SocketTimeoutException), ошибка парсинга (SerializationException), отмена (CancellationException). Цель — поймать всё это в одном месте и отдать наверх предсказуемый доменный результат, а не «сырое» исключение.

DataResult.kt
sealed interface DataResult<out T> {
    data class Success<T>(val data: T) : DataResult<T>
    sealed interface Failure : DataResult<Nothing> {
        data class Network(val cause: Throwable) : Failure         // нет связи/таймаут
        data class Http(val code: Int, val body: String?) : Failure // 4xx/5xx
        data class Parse(val cause: Throwable) : Failure           // битый JSON
        data class Unknown(val cause: Throwable) : Failure
    }
}
safeApiCall.kt
suspend fun <T> safeApiCall(block: suspend () -> T): DataResult<T> =
    try {
        DataResult.Success(block())
    } catch (e: CancellationException) {
        throw e                          // отмену НЕ глотаем — пробрасываем
    } catch (e: HttpException) {
        DataResult.Failure.Http(e.code(), e.response()?.errorBody()?.string())
    } catch (e: IOException) {
        DataResult.Failure.Network(e)    // нет сети/таймаут/DNS
    } catch (e: SerializationException) {
        DataResult.Failure.Parse(e)
    } catch (e: Throwable) {
        DataResult.Failure.Unknown(e)
    }
🛑
Никогда не глотай CancellationException. При отмене корутины (ушли с экрана) исключение должно пробрасываться, иначе сломается структурированный параллелизм и корутина «зависнет». Лови его первым и делай throw.

Безопасный вызов в репозитории

UserRepository.kt
class UserRepository(
    private val api: UserApi,
    private val dao: UserDao,
) {
    suspend fun user(id: String): DataResult<User> =
        when (val r = safeApiCall { api.getUser(id) }) {
            is DataResult.Success -> {
                dao.upsert(r.data.toEntity())          // кэшируем в Room
                DataResult.Success(r.data.toDomain())  // маппим DTO -> domain
            }
            is DataResult.Failure.Network -> {
                // оффлайн: отдаём кэш, если он есть
                dao.find(id)?.let { DataResult.Success(it.toDomain()) } ?: r
            }
            is DataResult.Failure -> r
        }
}

Retry с экспоненциальным backoff

Ретраить стоит транзиентные ошибки (сеть, 5xx, 429) и только идемпотентные запросы. Backoff с джиттером не даёт «толпе» клиентов синхронно долбить сервер после сбоя.

suspend fun <T> retry(
    times: Int = 3,
    initialDelay: Long = 500,
    maxDelay: Long = 8000,
    factor: Double = 2.0,
    block: suspend () -> T,
): T {
    var delayMs = initialDelay
    repeat(times - 1) {
        try { return block() }
        catch (e: CancellationException) { throw e }
        catch (e: IOException) { /* транзиентная — повторяем */ }
        delay(delayMs + Random.nextLong(0, 200))      // + джиттер
        delayMs = (delayMs * factor).toLong().coerceAtMost(maxDelay)
    }
    return block()  // последняя попытка — без перехвата
}
ℹ️
Отмена и оффлайн. Отмену даёт сама корутина: viewModelScope отменит запрос при очистке VM, OkHttp прервёт сокет. Оффлайн — стратегия SSOT: репозиторий отдаёт кэш (Room) и помечает данные «устаревшими», фоновая синхронизация догоняет (single source of truth).

Частые вопросы интервьюера

Чем отличается idempotent от safe? Какие методы можно ретраить? Middle

Safe — метод не меняет состояние на сервере (GET, HEAD). Idempotent — повтор даёт тот же результат (GET, PUT, DELETE, но не POST/PATCH). Автоматический retry безопасен только для идемпотентных запросов: повторный POST «создать заказ» создаст дубль. Для не-идемпотентных вводят Idempotency-Key, по которому сервер дедуплицирует.

Расскажи, что происходит при TLS-рукопожатии. Senior

Клиент шлёт ClientHello (версии TLS, cipher suites, случайное число, в TLS 1.3 — key_share). Сервер выбирает шифр, отвечает ServerHello + свой key_share, присылает сертификат и Finished. Клиент проверяет цепочку сертификата до доверенного корневого CA, и обе стороны по Diffie-Hellman вычисляют общий симметричный ключ. Дальше весь трафик шифруется симметрично (AES-GCM). Асимметрия — только для аутентификации и согласования ключа; эфемерный DH даёт forward secrecy. В TLS 1.3 всё это — за 1 round-trip.

Чем application-интерсептор отличается от network в OkHttp? Middle+

Application-интерсептор стоит между приложением и ядром OkHttp: вызывается ровно один раз на запрос (даже при кэш-хите), не видит редиректы, не имеет доступа к Connection. Идеален для auth-заголовков и верхнеуровневого логирования. Network-интерсептор — между ядром и сетью: вызывается на каждый редирект/retry, видит реальные «по проводу» заголовки (например Accept-Encoding: gzip), имеет доступ к Connection (IP, TLS). Подходит для логирования по хопам и метрик. Из кэша — не вызывается вообще.

Certificate pinning или public-key pinning — что выберешь и почему? Senior

Public-key (SPKI) pinning. Он пинит хэш публичного ключа, а не всего сертификата, поэтому переживает плановую пере-выписку сертификата с тем же ключом — не требует релиза приложения. OkHttp CertificatePinner как раз пинит SPKI. Обязательно добавляю backup-пин (резервный ключ): без него экстренная смена ключа отрежет всех пользователей до выката апдейта. Pinning защищает от MITM даже при скомпрометированном/подложном CA в trust store.

Как обработать ошибку сети, чтобы приложение не падало? Middle+

Оборачиваю вызов в safeApiCall, который ловит HttpException (4xx/5xx), IOException (нет связи/таймаут/DNS), SerializationException (парсинг) и маппит в sealed DataResult.Failure.{Http, Network, Parse, Unknown}. CancellationException ловлю первым и пробрасываю. UI получает предсказуемый тип и решает, что показать: ретрай, «нет интернета», экран ошибки. На транзиентных ошибках — retry с экспоненциальным backoff и джиттером, только для идемпотентных запросов.

Почему нельзя глотать CancellationException? Senior

Отмена корутины реализована через бросок CancellationException. Если поймать его в общем catch (e: Exception) и не пробросить, родительский scope не узнает об отмене — структурированный параллелизм сломается, корутина «повиснет», ресурсы (сокет, курсор) не освободятся. Поэтому CancellationException ловят отдельной первой веткой и делают throw, либо используют currentCoroutineContext().ensureActive().

ETag и Cache-Control — как работает условный запрос? Middle

Cache-Control: max-age=N задаёт время «свежести» ответа — пока оно не истекло, клиент берёт данные из кэша без запроса. Когда истекло, клиент шлёт условный запрос с If-None-Match: "<etag>" (ETag — версия/хэш ресурса). Если ресурс не менялся, сервер отвечает 304 Not Modified без тела, и клиент использует кэш — экономия трафика и батареи. OkHttp с настроенным Cache делает это прозрачно.

Зачем HTTP/2 и что такое мультиплексирование? Middle+

HTTP/1.1 на одном TCP-соединении обрабатывает запросы по очереди (head-of-line блокировка), поэтому браузеры/клиенты открывали несколько соединений. HTTP/2 ввёл бинарные фреймы и мультиплексирование: множество запросов и ответов идут параллельно в одном TCP-соединении, разбитые на фреймы и переплетённые. Плюс сжатие заголовков HPACK и server push. Итог — меньше соединений, меньше overhead на TCP/TLS, ниже латентность. OkHttp включает HTTP/2 автоматически через ALPN при TLS.

kotlinx.serialization, Moshi или Gson — что и почему? Senior

Для нового кода — kotlinx.serialization (compile-time, без рефлексии, понимает default-значения, null-safety и sealed-полиморфизм) или Moshi с codegen. Gson — в режиме поддержки: он использует рефлексию и Unsafe, создаёт объект минуя конструктор Kotlin, поэтому игнорирует default-значения и может поставить null в non-null поле → NPE. Это его главный подводный камень. Производительность и безопасность типов на стороне kotlinx/Moshi.
