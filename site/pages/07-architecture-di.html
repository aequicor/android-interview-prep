Архитектура и DI

Как структурировать Android-приложение, чтобы оно жило годами: паттерны представления MVVM/MVI/MVP (Model-View-ViewModel, Model-View-Intent, Model-View-Presenter — модели архитектуры экрана), Clean Architecture (чистая архитектура), гайд Google по архитектуре, multi-module (многомодульность) и DI (dependency injection, внедрение зависимостей) на Hilt. Тема почти гарантированно всплывёт на Middle+/Senior (уверенный/старший уровень) — ждут не определений, а trade-offs (компромиссов) и опыта из боевых проектов.

Middle Senior Частый вопрос

Паттерны представления: MVVM, MVI, MVP

Все три отвечают на вопрос «как связать UI с логикой и состоянием». Разница — в направлении потока данных и в том, как хранится и обновляется состояние экрана.

MVP (legacy)

Presenter держит ссылку на пассивный View (через интерфейс) и императивно командует им: view.showLoading(), view.showError(). Двусторонняя связь, ручное управление жизненным циклом, View-интерфейсы с десятками методов. Сегодня встречается в легаси на Android Views; на новых проектах вытеснен MVVM/MVI, потому что плохо дружит с конфиг-изменениями и Compose.

MVVM

ViewModel выставляет наблюдаемое состояние (StateFlow/LiveData), View подписывается и рендерит. ViewModel не знает о View — связь односторонняя на чтение состояния, плюс вызовы методов VM на события. Состояние может быть как одним объектом, так и набором отдельных полей. Это рекомендация Google по умолчанию.

ProfileViewModel.kt (MVVM)
class ProfileViewModel(
    private val getProfile: GetProfileUseCase
) : ViewModel() {

    private val _state = MutableStateFlow(ProfileUiState())
    val state: StateFlow<ProfileUiState> = _state.asStateFlow()

    fun onRefresh() = viewModelScope.launch {
        _state.update { it.copy(isLoading = true) }
        val result = getProfile()
        _state.update {
            it.copy(isLoading = false, profile = result.getOrNull(), error = result.exceptionOrNull())
        }
    }
}

MVI

Эволюция MVVM в сторону строгого UDF. Ключевые элементы:

  • Единый immutable State — весь экран описан одним неизменяемым объектом (single source of truth для UI).
  • Intent (часто называют Action/Event) — единственный способ запросить изменение. View эмитит интенты, не трогает состояние напрямую.
  • Reducer — чистая функция (State, Action) -> State: по текущему состоянию и интенту возвращает новое. Легко тестировать, можно «перематывать» историю (time-travel debugging).
  • Side-effects — одноразовые эффекты (навигация, тост, snackbar), которые не должны жить в state. Отдаются через отдельный канал (Channel/SharedFlow), иначе при повороте экрана тост повторится.
CounterViewModel.kt (MVI)
data class CounterState(val count: Int = 0, val isLoading: Boolean = false)

sealed interface CounterAction {
    data object Increment : CounterAction
    data object Decrement : CounterAction
    data object Save : CounterAction
}

sealed interface CounterEffect {            // side-effects
    data class ShowToast(val text: String) : CounterEffect
    data object NavigateBack : CounterEffect
}

class CounterViewModel(private val repo: CounterRepository) : ViewModel() {

    private val _state = MutableStateFlow(CounterState())
    val state: StateFlow<CounterState> = _state.asStateFlow()

    private val _effects = Channel<CounterEffect>(Channel.BUFFERED)
    val effects = _effects.receiveAsFlow()    // одноразовые, не часть state

    fun onAction(action: CounterAction) {
        _state.update { reduce(it, action) }   // чистый редьюсер
        if (action is CounterAction.Save) save()
    }

    private fun reduce(s: CounterState, a: CounterAction): CounterState = when (a) {
        CounterAction.Increment -> s.copy(count = s.count + 1)
        CounterAction.Decrement -> s.copy(count = s.count - 1)
        CounterAction.Save      -> s.copy(isLoading = true)
    }

    private fun save() = viewModelScope.launch {
        repo.save(state.value.count)
        _state.update { it.copy(isLoading = false) }
        _effects.send(CounterEffect.ShowToast("Сохранено"))
    }
}
flowchart LR View -->|Intent / Action| VM[ViewModel] VM -->|reduce| State[(Immutable State)] State -->|render| View VM -.->|side-effect| FX["Channel: toast, nav"] FX -.-> View
MVI: однонаправленный цикл. Состояние течёт вниз, интенты — вверх, side-effects идут отдельным каналом

Честный тезис для собеса. MVI — это MVVM с дисциплиной: то же дерево (ViewModel + observable state), но с жёсткими правилами (immutable state, интенты, редьюсер, отдельные эффекты). На простом экране MVI — это overhead; на экране с десятком взаимозависимых флагов он окупается предсказуемостью. Не противопоставляйте их как «или/или» — это спектр строгости UDF.

Clean Architecture: слои и правило зависимостей

Clean Architecture (Роберт Мартин) делит код на концентрические слои. Главное — правило зависимостей: исходный код может зависеть только в направлении внутрь, к более стабильной политике. Внутренние слои ничего не знают о внешних.

flowchart TB subgraph presentation["presentation (UI)"] UI[Compose / Activity] VM[ViewModel] end subgraph domain["domain (политика, чистый Kotlin)"] UC[UseCase] RI{{"Repository (интерфейс)"}} DM[Domain models] end subgraph data["data (детали)"] RImpl[RepositoryImpl] API[Retrofit API] DB[Room] DTO[DTO / Entity] end UI --> VM --> UC --> RI RImpl -. реализует .-> RI RImpl --> API RImpl --> DB RImpl --> DTO
Стрелки зависимостей направлены внутрь. domain не знает ни про Android, ни про Retrofit/Room. data реализует интерфейсы, объявленные в domain (DIP)

Роли слоёв

  • presentation — ViewModel + UI. Зависит от domain (вызывает UseCase'ы), оперирует UI-моделями.
  • domain — чистый Kotlin-модуль без зависимостей от Android. Содержит UseCase, доменные модели и интерфейсы репозиториев. Это сердце бизнес-логики.
  • data — реализации репозиториев, источники (network/database), DTO/Entity и мапперы. Зависит от domain (реализует его интерфейсы), но domain про data ничего не знает.

UseCase, Repository и маппинг моделей

Каждый слой имеет свою модель: UserDto (data, формат API), UserEntity (data, Room), User (domain), UserUi (presentation, готовая к показу). Мапперы изолируют слои: смена формата API не доходит до UI.

domain-слой (чистый Kotlin)
// domain/model — без аннотаций фреймворков
data class User(val id: String, val name: String, val isPremium: Boolean)

// domain/repository — интерфейс, реализация живёт в data
interface UserRepository {
    suspend fun getUser(id: String): Result<User>
}

// domain/usecase — одна бизнес-операция, удобно мокать и тестировать
class GetUserUseCase(private val repo: UserRepository) {
    suspend operator fun invoke(id: String): Result<User> = repo.getUser(id)
}
data-слой: реализация + маппинг
// data/dto — формат сети
data class UserDto(val id: String, val full_name: String, val plan: String)

// data/mapper — изолирует формат API от домена
fun UserDto.toDomain() = User(
    id = id,
    name = full_name,
    isPremium = plan == "premium"
)

// data/repository — реализует контракт domain, внедряется через DI
class UserRepositoryImpl(
    private val api: UserApi,
    private val dao: UserDao
) : UserRepository {
    override suspend fun getUser(id: String): Result<User> = runCatching {
        api.fetchUser(id).toDomain()
    }
}
⚠️
Подводный камень. Не тащите UserDto или Room-Entity в ViewModel — это «протечка» слоя data в presentation. Тогда смена бэкенда заставит править UI. Мапперы — не бюрократия, а граница, которая держит изменения локально. На простом CRUD-экране слой UseCase иногда схлопывают (VM зовёт репозиторий напрямую) — это допустимый прагматичный компромисс, но будьте готовы его обосновать.

Гайд Google по архитектуре приложений

Официальная рекомендация Google — слоистая архитектура из UI layer, опционального domain layer и data layer, построенная на двух принципах: single source of truth (SSOT) и unidirectional data flow (UDF). Это «облегчённая» и прагматичная версия идей Clean Architecture.

UI layer и state holder

UI-слой рисует состояние и шлёт события наверх. State holder (обычно ViewModel) — класс, который производит UI-state и держит логику его формирования. UDF здесь буквально: state течёт вниз в UI, события (нажатия) текут вверх в state holder, где данные меняются и выставляются как immutable-тип.

Domain layer (опциональный)

Слой UseCase'ов. Нужен, когда логика сложна или переиспользуется несколькими ViewModel. UseCase инкапсулирует одну операцию и комбинирует данные из нескольких репозиториев. Если логики мало — слой можно опустить, VM ходит в data напрямую.

Data layer, SSOT и offline-first

Data-слой состоит из репозиториев и источников данных. Репозиторий — публичный API слоя, агрегирует источники, разрешает конфликты и выставляет единый источник истины. Offline-first: локальная БД (Room) становится SSOT, сеть лишь обновляет её, а UI всегда читает из БД. Так приложение работает без сети и не мигает при рефетче.

flowchart TB subgraph ui["UI layer"] C[Compose UI] SH[ViewModel / State holder] end subgraph dom["Domain layer (опц.)"] U[UseCase] end subgraph dat["Data layer"] Repo[Repository = SSOT] Net[Remote source] Loc[(Local DB - источник истины)] end C -->|события вверх| SH SH -->|state вниз| C SH --> U --> Repo Repo --> Loc Net -->|sync| Loc
Слои Google-гайда. UDF: state вниз, события вверх. Offline-first: локальная БД — единый источник истины, сеть только синхронизирует
Offline-first репозиторий (Flow из БД + sync из сети)
class ArticleRepository(
    private val api: ArticleApi,
    private val dao: ArticleDao
) {
    // SSOT — UI всегда читает из БД, это горячий источник истины
    fun observeArticles(): Flow<List<Article>> =
        dao.observeAll().map { entities -> entities.map { it.toDomain() } }

    // сеть лишь обновляет БД; UI обновится сам через Flow
    suspend fun refresh() {
        val remote = api.fetchArticles()
        dao.upsertAll(remote.map { it.toEntity() })
    }
}

Многомодульность

Монолитный :app на большом проекте упирается в скорость сборки и спутанные зависимости. Решение — разбить на Gradle-модули. Типичная классификация (как в эталонном Google-проекте Now in Android):

  • :app — точка входа, склейка фич, навигационный граф верхнего уровня.
  • :feature:* — самодостаточные фичи (home, profile, settings). Зависят от :core, но не друг от друга.
  • :core:* — переиспользуемая инфраструктура: :core:ui, :core:designsystem, :core:common, :core:network, :core:database.
  • :core:data / :core:domain — общие репозитории и UseCase'ы.
flowchart TB app[":app"] --> fa[":feature:home"] app --> fb[":feature:profile"] fa --> cu[":core:ui"] fb --> cu fa --> cd[":core:data"] fb --> cd cd --> cn[":core:network"] cd --> cdb[":core:database"] cu --> cds[":core:designsystem"] cd --> cdom[":core:domain"]
Граф модулей. Фичи не зависят друг от друга; общее вынесено в :core. Это держит граф ацикличным и позволяет параллельную сборку

Плюсы

Навигация между модулями

Фичи не знают друг о друге, поэтому прямой вызов экрана из соседнего модуля невозможен. Решения: навигационный граф собирается в :app (он видит все фичи), а каждая фича экспонирует свой NavGraphBuilder-extension. Маршруты к чужим фичам передаются как абстракция (например, через интерфейс-навигатор в :core или type-safe route, объявленный в общем модуле).

Gradle convention plugins

Без них каждый модуль дублирует одинаковый build.gradle.kts (compileSdk, Compose, Kotlin-опции). Convention plugin — это переиспользуемая build-логика в композитной сборке build-logic: настройка описывается один раз и применяется строкой. Именно так сделан Now in Android.

build.gradle.kts фичи — одна строка вместо десятков
plugins {
    id("nowinandroid.android.feature")      // convention plugin
    id("nowinandroid.android.library.compose")
}

dependencies {
    implementation(projects.core.data)
    implementation(projects.core.designsystem)
}
// весь общий setup (compileSdk, Hilt, Compose, тест-раннеры)
// живёт внутри convention-плагина, а не копируется в каждый модуль

DI: Hilt — компоненты, скоупы, граф

Dependency Injection — передача зависимостей объекту извне вместо их создания внутри. Это практическая реализация DIP: верхние слои зависят от абстракций, а конкретные реализации поставляет контейнер. Hilt — надстройка над Dagger: он генерирует за вас стандартный граф компонентов, привязанный к жизненным циклам Android (Application, Activity, ViewModel, Fragment), и кодогенерацию точек входа. Под капотом всё тот же compile-time DI Dagger с проверкой графа на этапе компиляции.

Тонкость про поворот экрана. @ActivityRetainedScoped переживает конфиг-изменения (живёт от первого создания до настоящего закрытия Activity), а @ActivityScoped — нет: пересоздаётся вместе с экземпляром Activity при повороте. Путаница между ними — частая ошибка и любимый вопрос на собесе.

@Module, @Provides и @Binds

@Provides — когда вы сами конструируете объект (например, чужой класс из библиотеки). @Binds — когда нужно связать интерфейс с реализацией; компилируется эффективнее (без тела метода), метод абстрактный, модуль обычно abstract.

NetworkModule.kt / RepositoryModule.kt
@Module
@InstallIn(SingletonComponent::class)
object NetworkModule {
    @Provides @Singleton
    fun provideRetrofit(): Retrofit = Retrofit.Builder()
        .baseUrl("https://api.example.com/")
        .addConverterFactory(Json.asConverterFactory("application/json".toMediaType()))
        .build()

    @Provides @Singleton
    fun provideUserApi(retrofit: Retrofit): UserApi = retrofit.create(UserApi::class.java)
}

@Module
@InstallIn(SingletonComponent::class)
abstract class RepositoryModule {
    // @Binds: связать интерфейс с реализацией, без ручного конструирования
    @Binds @Singleton
    abstract fun bindUserRepository(impl: UserRepositoryImpl): UserRepository
}

@HiltViewModel, qualifiers и EntryPoint

Инъекция в ViewModel + qualifier
// различаем две реализации одного типа через qualifier
@Qualifier annotation class IoDispatcher
@Qualifier annotation class MainDispatcher

@Module
@InstallIn(SingletonComponent::class)
object DispatchersModule {
    @Provides @IoDispatcher fun io(): CoroutineDispatcher = Dispatchers.IO
    @Provides @MainDispatcher fun main(): CoroutineDispatcher = Dispatchers.Main
}

@HiltViewModel
class ProfileViewModel @Inject constructor(
    private val getUser: GetUserUseCase,
    @IoDispatcher private val io: CoroutineDispatcher   // нужный диспетчер
) : ViewModel()

EntryPoint нужен, чтобы достать зависимость из Hilt-графа в классе, которым Hilt не управляет (ContentProvider, кастомный класс без @AndroidEntryPoint):

EntryPoint — доступ к графу из неуправляемого класса
@EntryPoint
@InstallIn(SingletonComponent::class)
interface AnalyticsEntryPoint {
    fun analytics(): Analytics
}

// где-то в коде без @Inject:
val analytics = EntryPointAccessors
    .fromApplication(appContext, AnalyticsEntryPoint::class.java)
    .analytics()

Инъекция в Worker (@HiltWorker)

WorkManager создаёт Worker сам, поэтому конструкторная инъекция требует моста — @HiltWorker + @AssistedInject (Context и WorkerParameters приходят от фреймворка через @Assisted):

SyncWorker.kt
@HiltWorker
class SyncWorker @AssistedInject constructor(
    @Assisted appContext: Context,
    @Assisted params: WorkerParameters,
    private val repo: ArticleRepository          // обычная зависимость из графа
) : CoroutineWorker(appContext, params) {

    override suspend fun doWork(): Result {
        repo.refresh()
        return Result.success()
    }
}
// + в Application: HiltWorkerFactory подключается через Configuration.Provider

Manual DI / Service Locator — когда уместно

Hilt не всегда обязателен. Manual DI (передаём зависимости руками через конструкторы/фабрики) подходит для маленьких приложений, библиотек (не навязывать пользователю Dagger) и учебных проектов. Service Locator (глобальный реестр, у которого запрашивают зависимости) проще, но скрывает зависимости класса (их не видно в сигнатуре) и хуже тестируется — обычно считается анти-паттерном по сравнению с конструкторной инъекцией. Правило: предпочитайте конструкторную инъекцию; Hilt берите, когда граф вырастает и ручная проводка становится болью.

Навигация: Jetpack Navigation (Compose)

С Navigation 2.8.0 type-safe API стабильны: маршруты описываются не строками, а @Serializable-типами (через Kotlin Serialization). Это убирает хрупкие строковые роуты и даёт компайл-тайм проверку аргументов.

Type-safe routes + аргументы + deep link
@Serializable data object HomeRoute
@Serializable data class ProfileRoute(val userId: String)   // типобезопасный аргумент

@Composable
fun AppNavHost(navController: NavHostController) {
    NavHost(navController, startDestination = HomeRoute) {
        composable<HomeRoute> {
            HomeScreen(onUser = { id -> navController.navigate(ProfileRoute(id)) })
        }
        composable<ProfileRoute>(
            // deep link на ту же type-safe цель
            deepLinks = listOf(navDeepLink<ProfileRoute>(basePath = "https://app.example.com/user"))
        ) { backStackEntry ->
            val route: ProfileRoute = backStackEntry.toRoute()   // достаём типизированно
            ProfileScreen(userId = route.userId)
        }
    }
}
  • Типобезопасные маршруты: навигация принимает объект ProfileRoute("42"), библиотека сама сериализует/десериализует аргументы с сохранением типов.
  • Передача аргументов: складываются в data-class маршрута, читаются через backStackEntry.toRoute<T>() — без ручного парсинга строк и багов с ключами.
  • Deep links: navDeepLink<T> связывает внешний URL с той же type-safe целью; внешняя ссылка открывает нужный экран с аргументами.

Reactive-слой: Flow/StateFlow как контракт

Связь между слоями реактивна. Контракт data → domain → UI описывается через Flow: репозиторий отдаёт «холодный» Flow данных (например, из Room), UseCase его трансформирует, ViewModel превращает в «горячий» StateFlow для UI. Так обновления (новый ответ сети записан в БД) автоматически доезжают до экрана.

Flow по слоям: cold во data, hot в presentation
// data: cold Flow из источника истины (Room)
fun ArticleRepository.observe(): Flow<List<Article>> = dao.observeAll().map { ... }

// domain: трансформация бизнес-правилом
class ObserveUnread(private val repo: ArticleRepository) {
    operator fun invoke(): Flow<List<Article>> =
        repo.observe().map { list -> list.filter { !it.isRead } }
}

// presentation: cold -> hot StateFlow, готовый к подписке UI
class FeedViewModel(observeUnread: ObserveUnread) : ViewModel() {
    val state: StateFlow<List<Article>> = observeUnread()
        .stateIn(viewModelScope, SharingStarted.WhileSubscribed(5_000), emptyList())
}
💡
Зачем StateFlow на границе с UI. Он всегда имеет текущее значение (UI получает state сразу при подписке) и конфлейтит обновления. SharingStarted.WhileSubscribed(5_000) держит upstream живым 5 секунд после ухода подписчика — переживает поворот экрана, но отписывается, когда экран действительно закрыт. Это стандартная связка для экспонирования состояния из ViewModel.

Частые вопросы интервьюера

Чем MVI отличается от MVVM и когда выбрать MVI? Middle+

MVVM выставляет наблюдаемое состояние, которое VM может обновлять разными методами; состояние может быть раздроблено на поля. MVI добавляет дисциплину: один immutable State, изменения только через Intent, состояние пересчитывается чистым reducer'ом (State, Action) -> State, а одноразовые эффекты (тост, навигация) идут отдельным каналом, а не в state. Выбирают MVI для сложных экранов с многими взаимозависимыми флагами (предсказуемость, тестируемый редьюсер, возможность реплея), для простых экранов это лишний boilerplate. По сути MVI — это строгий вариант UDF поверх той же связки ViewModel+observable state.

Объясни правило зависимостей в Clean Architecture. Middle+

Зависимости направлены только внутрь — к более стабильной политике. presentation зависит от domain, data зависит от domain, но domain не знает ни о ком. Ключ — DIP: domain объявляет интерфейс Repository, а data его реализует; стрелка зависимости от реализации идёт к абстракции в домене. Поэтому domain — чистый Kotlin без Android/Retrofit/Room, его легко тестировать и переиспользовать. Каждый слой имеет свои модели (DTO/Entity/Domain/Ui) и мапперы между ними, чтобы изменения формата не протекали наружу.

Hilt — это что поверх Dagger? Что он добавляет? Middle+

Hilt — официальная надстройка над Dagger для Android. Dagger даёт compile-time DI с проверкой графа, но требует вручную писать компоненты и связывать их с жизненными циклами. Hilt генерирует стандартный набор компонентов (Singleton, ActivityRetained, ViewModel, Activity, Fragment, View, Service), привязывает их к Android-жизненным циклам, добавляет аннотации @AndroidEntryPoint, @HiltViewModel, @InstallIn и упрощает точки входа. Под капотом — тот же Dagger и тот же граф; Hilt лишь убирает шаблонный код по его настройке.

Разница @ActivityScoped и @ActivityRetainedScoped? Senior

@ActivityScoped привязан к конкретному экземпляру Activity — при повороте экрана Activity пересоздаётся, и скоупнутый объект тоже. @ActivityRetainedScoped живёт в ActivityRetainedComponent, который переживает конфиг-изменения (от первого создания Activity до её настоящего закрытия), как и ViewModel. Поэтому для состояния, которое должно пережить поворот, берут ActivityRetained, а для привязки к экземпляру Activity — ActivityScoped. ViewModelComponent — потомок ActivityRetained, поэтому VM-скоупнутые зависимости тоже переживают поворот.

Зачем дробить приложение на модули? Что это даёт сборке? Senior

Главные выгоды: скорость инкрементальной сборки (Gradle собирает независимые модули параллельно и кэширует — правка одного модуля не пересобирает всё), инкапсуляция (через internal и явные API нельзя случайно дёрнуть внутренности чужой фичи), переиспользование (:core:* подключается многими фичами), контроль архитектуры (граф физически запрещает циклы и протечки слоёв) и параллельная работа команд. Фичи не зависят друг от друга, общее выносится в core. Чтобы не дублировать build-конфиги, применяют Gradle convention plugins (как в Now in Android).

В чём смысл single source of truth и offline-first? Middle+

SSOT — у каждого типа данных один авторитетный источник; все остальные читают из него и не имеют права независимо его менять. В offline-first роль SSOT играет локальная БД (Room): UI всегда читает из БД через Flow, а сеть лишь обновляет БД. Плюсы: приложение работает без сети, данные консистентны, нет «мигания» при рефетче, проще дебажить. Репозиторий инкапсулирует разрешение конфликтов между источниками и выставляет наружу единый поток данных.

Когда manual DI / service locator уместнее Hilt? Senior

Manual DI (конструкторная проводка вручную, фабрики) хорош для маленьких приложений, библиотек (не навязывать пользователю Dagger) и прототипов. Service locator проще завести, но скрывает зависимости класса (их не видно в конструкторе) и хуже тестируется — обычно считается анти-паттерном. Hilt окупается, когда граф большой и завязан на Android-жизненные циклы: он даёт compile-time проверку и убирает рутину. Универсальное правило — предпочитать конструкторную инъекцию, а инструмент (ручной/Hilt) выбирать по размеру графа.

Как инжектить зависимости в WorkManager Worker? Middle+

WorkManager сам инстанцирует Worker, поэтому обычная конструкторная инъекция не сработает. Нужен @HiltWorker + @AssistedInject: Context и WorkerParameters приходят от фреймворка через @Assisted, а остальные зависимости (репозитории и т.п.) Hilt поставляет из графа как обычно. В Application подключается HiltWorkerFactory через Configuration.Provider (или удаляется дефолтный инициализатор WorkManager в манифесте).

Почему StateFlow на границе ViewModel→UI, а не cold Flow? Senior

StateFlow — горячий и всегда имеет текущее значение: подписавшийся UI сразу получает актуальный state, без повторного запуска upstream на каждого коллектора. Cold Flow перезапускал бы пайплайн под каждую подписку. Стандарт — собрать состояние через stateIn(scope, SharingStarted.WhileSubscribed(5000), initial): это конфлейтит обновления, шарит один upstream между подписчиками и держит его живым ещё 5 секунд после ухода последнего — чтобы пережить поворот экрана, но не работать, когда экран закрыт. Внутри слоёв data/domain, наоборот, держат cold Flow как «рецепт» данных.
