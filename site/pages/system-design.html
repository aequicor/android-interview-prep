Системный дизайн (mobile)

Как за 45 минут спроектировать клиент мобильного приложения: от сбора требований до offline-first (сначала локальные данные), синхронизации, пагинации и trade-offs (компромиссов). Требования: функциональные и нефункциональные

Функциональные — что делает фича (показать ленту, поставить лайк, отправить сообщение). Нефункциональные (NFR) — как хорошо: офлайн-доступность, latency прокрутки, размер кэша, расход батареи и трафика, поддержка слабых устройств. Именно NFR в mobile важнее всего — фиксируйте их явно.

Пользователи и нагрузка

Спросите: сколько пользователей и активных сессий, какой профиль сети (3G в метро vs Wi-Fi), какие устройства (бюджетные Android One vs флагманы), сколько данных на пользователя. Это влияет на размер локального кэша, агрессивность префетча и выбор форматов.

💡
Скажите это вслух. «Прежде чем проектировать, уточню scope и нефункциональные требования — в mobile они определяют архитектуру сильнее, чем сами фичи». Так вы сразу показываете зрелость.
Вам дали кейс «спроектировать ленту на 50 млн DAU», но интервьюер не назвал ни одной цифры. Зачем вообще делать back-of-the-envelope оценку на mobile-собесе и что конкретно она меняет в дизайне клиента? ★

Оценка нужна, чтобы решения не висели «на глаз». Из размера одной записи ленты и числа экранов вы прикидываете размер локального кэша и лимит дискового кэша Room/Coil; из частоты обновлений — интервал дельта-синхронизации и TTL; из размера страницы и prefetchDistance — сколько данных тянем за скролл.

Важно: mobile-оценка идёт не про QPS бэкенда, а про устройство. Сколько мегабайт займёт кэш, сколько трафика съест сессия в 4G, сколько wake-up'ов даст синхронизация. Конкретно на клиенте это меняет page size (обычно 20–50 элементов), агрессивность префетча, политику вытеснения и бюджет трафика на сессию.

Назвав хотя бы порядок величин — «одна карточка ~1 КБ метаданных + картинка ~50 КБ, держим ~500 в кэше = ~25 МБ» — вы показываете, что лимиты обоснованы числами, а не взяты с потолка. Подробнее про методику прикидки — в параграфе «Оценка масштаба» ниже.

Высокоуровневая архитектура

Стандарт индустрии — слоистая архитектура с однонаправленным потоком данных (UDF) и БД как единственным источником истины. Три слоя: UI (Compose + ViewModel), domain (UseCase, бизнес-правила), data (Repository поверх сети и БД). Repository прячет источники и решает, когда идти в сеть, а когда отдать кэш.

Слоистая архитектура: UI наблюдает БД через Flow, запись идёт в БД и outbox, сеть — деталь внутри Repository

Ключевая идея: UI читает только из БД (через Flow), а сеть лишь наполняет БД. Тогда экран всегда показывает что-то осмысленное, даже без интернета, и обновляется автоматически при синхронизации.

// Repository: сеть наполняет БД, UI подписан на БД (single source of truth)
class FeedRepository(
    private val api: FeedApi,
    private val dao: FeedDao,
    private val outbox: OutboxDao,
) {
    // UI всегда читает из БД — офлайн отдаёт кэш, онлайн обновляется сам
    fun feed(): Flow<List<Product>> = dao.observeFeed()

    suspend fun refresh() {
        val remote = api.getFeed()             // деталь, скрыта от UI
        dao.upsertAll(remote.map { it.toEntity() })
    }

    // запись «оптимистична»: сразу в БД + в outbox на отправку
    suspend fun like(productId: String) {
        dao.setLiked(productId, liked = true)              // UI обновится мгновенно
        outbox.enqueue(Operation.Like(productId))          // отправим, когда будет сеть
    }
}
Если UI читает только из Room через Flow, а сеть пишет в Room, то откуда экран узнаёт про состояния «идёт загрузка» и «ошибка сети»? Их ведь нет в данных БД. ★

Состояние загрузки и ошибки — это не данные домена, поэтому в основной таблице им не место. Их отдают отдельным каналом. Канонический приём: Repository возвращает данные из Room (Flow), а статус операции — отдельным потоком, и UiState во ViewModel собирает оба через combine. NetworkBoundResource из guide to architecture делает ровно это: эмитит Loading с кэшем, потом Success/Error.

У Paging 3 для этого есть встроенный CombinedLoadStates (loadState.refresh/append = Loading/Error/NotLoading) — отдельно от самих PagingData. Грубо: «истина о данных» живёт в БД, а «истина о текущей сетевой операции» — это эфемерное UI-состояние, которое не персистят.

Путать их — частая ошибка. Запишешь isLoading в Room — оно переживёт перезапуск и залипнет навсегда: пользователь увидит вечный спиннер по данным, которые давно пришли.

Удаления в дельте: тумбстоуны

Дельта по updatedSince отлично тянет созданное и изменённое, но на удалениях тихо ломается: отсутствие записи в ответе неотличимо от «не менялась». Поэтому удаление делают логическим — через тумбстоуны (soft delete): сервер хранит запись с флагом deleted=true и свежим updatedAt, она приходит в обычной дельте, а клиент по флагу удаляет её локально.

sequenceDiagram participant C as Клиент participant S as Сервер C->>S: "GET /changes?updatedSince=T" alt офлайн меньше TTL тумбстоунов S-->>C: "изменённые + тумбстоуны (deleted=true)" C->>C: "upsert живых, delete по тумбстоунам" C->>C: "сохранить новый updatedSince" else офлайн дольше TTL S-->>C: "сигнал full-resync" C->>S: "GET /snapshot (полная выгрузка)" S-->>C: "актуальный набор целиком" C->>C: "заменить локальный кэш" end
Дельта-синхронизация с тумбстоунами: удаление доезжает как запись с флагом; давно офлайн → полный ресинк

Тумбстоуны живут TTL (скажем, 30 дней), чтобы таблица не пухла. Клиент, который офлайн дольше TTL, может пропустить удаление — для таких делают полный ресинк вместо дельты. Альтернатива без тумбстоунов — сервер отдаёт список живых id за период, клиент вычисляет разницу, но это дороже по трафику.

Дельта-синхронизация по updatedSince отлично тянет созданные и изменённые записи. А как клиент узнает, что запись удалили на сервере, если её просто нет в дельте? ★

Никак — и это классическая ловушка дельта-синхронизации: отсутствие записи в ответе неотличимо от «не менялась». Удалённое навсегда зависает в кэше.

Поэтому удаление делают не физическим, а логическим — через тумбстоуны: сервер помечает запись deleted=true со свежим updatedAt, и она приходит в обычной дельте. Тумбстоуны живут TTL, чтобы таблица не пухла, но тогда клиент, который офлайн дольше TTL, пропустит удаление — для «давно не синхронизировавшихся» делают полный ресинк. На собесе важно проговорить именно проблему «удаление невидимо в дельте» — её упускают чаще всего.

Offline-first и синхронизация

Это сердце mobile system design и любимая зона вопросов. Базовый принцип: локальная БД — источник истины, сеть — слой оптимизации, а не обязательное условие работы. Пользователь должен видеть данные и совершать действия офлайн, а синхронизация происходит в фоне.

Outbox: очередь исходящих операций

Outbox — это таблица в БД, где каждая запись описывает намерение изменить данные на сервере. Она переживает перезапуск процесса и потерю сети. Воркер «дренирует» очередь, когда появляется соединение.

Outbox.kt
@Entity(tableName = "outbox")
data class OutboxOp(
    @PrimaryKey(autoGenerate = true) val localId: Long = 0,
    val type: OpType,                 // LIKE, SEND_MESSAGE, UPDATE_PROFILE...
    val payloadJson: String,
    val idempotencyKey: String,       // UUID — защита от дублей при ретраях
    val createdAt: Long,
    val attempts: Int = 0,
)

@Dao
interface OutboxDao {
    @Insert suspend fun enqueue(op: OutboxOp)
    @Query("SELECT * FROM outbox ORDER BY createdAt ASC LIMIT 50")
    suspend fun peek(): List<OutboxOp>
    @Delete suspend fun ack(op: OutboxOp)
    @Query("UPDATE outbox SET attempts = attempts + 1 WHERE localId = :id")
    suspend fun bumpAttempts(id: Long)
}

WorkManager: гарантированная фоновая синхронизация

WorkManager — правильный инструмент для отложенной, гарантированной работы: переживает рестарт приложения и перезагрузку устройства, поддерживает constraints (сеть, зарядка), экспоненциальный backoff и уникальные цепочки. Для синхронизации используем именно его, а не CoroutineScope внутри ViewModel.

SyncWorker.kt
@HiltWorker
class SyncWorker @AssistedInject constructor(
    @Assisted ctx: Context,
    @Assisted params: WorkerParameters,
    private val outbox: OutboxDao,
    private val api: SyncApi,
) : CoroutineWorker(ctx, params) {

    override suspend fun doWork(): Result {
        val ops = outbox.peek()
        for (op in ops) {
            val res = runCatching {
                api.execute(op.type, op.payloadJson, op.idempotencyKey)
            }
            when {
                res.isSuccess -> outbox.ack(op)               // отправлено
                op.attempts >= MAX_ATTEMPTS -> outbox.ack(op) // dead-letter, не зацикливаемся
                else -> { outbox.bumpAttempts(op.localId); return Result.retry() }
            }
        }
        return Result.success()
    }
    companion object { const val MAX_ATTEMPTS = 10 }
}

// планирование с constraints и backoff
fun scheduleSync(wm: WorkManager) {
    val req = OneTimeWorkRequestBuilder<SyncWorker>()
        .setConstraints(Constraints.Builder()
            .setRequiredNetworkType(NetworkType.CONNECTED).build())
        .setBackoffCriteria(BackoffPolicy.EXPONENTIAL, 30, TimeUnit.SECONDS)
        .build()
    // ExistingWorkPolicy.KEEP — не плодим параллельные синхронизации
    wm.enqueueUniqueWork("sync", ExistingWorkPolicy.KEEP, req)
}
⚠️
Идемпотентность обязательна. При ретраях запрос может выполниться дважды (сеть упала после того, как сервер уже обработал). Передавай Idempotency-Key (UUID операции) — сервер по нему отбрасывает дубликат. Без этого «двойные лайки» и «два списания» гарантированы.

Разрешение конфликтов

Конфликт — когда локальная и серверная версии разошлись (оба изменили запись офлайн). Стратегии по возрастанию сложности:

СтратегияКак работаетКогда применять
Last-Write-Wins (LWW)Побеждает запись с большим timestampПросто, ок для лайков/настроек; теряет данные при гонке
Версионирование (optimistic concurrency)Шлём version; сервер отвечает 409, если устарелаКогда важно не затереть чужие изменения
Merge по полямСливаем непересекающиеся измененияПрофиль, документ — где поля независимы

// LWW + версионирование: отправляем version, обрабатываем 409
suspend fun pushUpdate(local: ProductEntity): SyncOutcome {
    val resp = api.update(
        id = local.id,
        body = local.toDto(),
        ifMatchVersion = local.serverVersion,   // optimistic concurrency
    )
    return when (resp.code()) {
        200 -> SyncOutcome.Synced(resp.body()!!.version)
        409 -> {                                  // на сервере новее
            val server = api.get(local.id)
            // LWW: чей updatedAt больше — тот и побеждает
            if (local.updatedAt > server.updatedAt) SyncOutcome.Retry
            else SyncOutcome.OverwrittenByServer(server)  // приняли серверную
        }
        else -> SyncOutcome.Retry
    }
}
В outbox лежат три операции по одному объекту (создать черновик → отредактировать → удалить). Сеть появилась. Почему слепо отправлять их по очереди — плохо, и что с этим делают? ★

Слепой дренаж по очереди генерирует лишние раунд-трипы и упирается в зависимости: удаление нельзя отправить раньше создания, а если создание ещё не подтверждено серверным id, у правки и удаления нет валидного id для запроса.

Что делают. Первое — компакция очереди перед отправкой: три операции над одним объектом сворачиваются в одну итоговую (создать + изменить + удалить = no-op, если объект так и не уехал на сервер). Второе — порядок и зависимости: операции над одной сущностью отправляются строго последовательно (упорядочение по createdAt/seq), а после создания клиент подменяет временный локальный id на серверный во всех последующих операциях очереди.

Плюс идемпотентность (Idempotency-Key) на случай, если ответ на «создать» потерялся, но сервер объект уже создал — повтор не задвоит. Это превращает наивный FIFO-дренаж в корректный sync-движок. Задача на компакцию — в конце главы.

Пагинация: Paging 3

Бесконечная лента — почти в каждом кейсе. Библиотека Paging 3 решает подгрузку страниц, состояния загрузки/ошибки, retry и (главное для офлайна) кэширование в БД. Два режима:

  • PagingSource — когда источник один (только сеть или только БД).
  • RemoteMediator + PagingSource — слоистый источник: RemoteMediator тянет страницы из сети в Room, а PagingSource от Room отдаёт их UI. БД — источник истины, сеть дёргается только когда кэш кончился.

FeedRemoteMediator.kt
@OptIn(ExperimentalPagingApi::class)
class FeedRemoteMediator(
    private val db: AppDatabase,
    private val api: FeedApi,
) : RemoteMediator<Int, ProductEntity>() {

    override suspend fun load(
        loadType: LoadType,
        state: PagingState<Int, ProductEntity>,
    ): MediatorResult {
        val cursor = when (loadType) {
            LoadType.REFRESH -> null
            LoadType.PREPEND -> return MediatorResult.Success(endOfPaginationReached = true)
            LoadType.APPEND -> db.keysDao().get("feed")?.nextCursor
                ?: return MediatorResult.Success(endOfPaginationReached = true)
        }
        return try {
            val page = api.getFeed(cursor = cursor, limit = 20)
            db.withTransaction {
                if (loadType == LoadType.REFRESH) db.feedDao().clear()
                db.keysDao().upsert(RemoteKey("feed", page.nextCursor))
                db.feedDao().upsertAll(page.items.map { it.toEntity() })
            }
            MediatorResult.Success(endOfPaginationReached = page.nextCursor == null)
        } catch (e: IOException) {
            MediatorResult.Error(e)   // Paging покажет состояние ошибки + retry
        }
    }
}

// Сборка Pager: источник истины — Room
val pager = Pager(

Почему через Room, а не напрямую из сети? Тогда лента работает офлайн (отдаём кэш), переживает поворот экрана и убийство процесса, не дёргает сеть на каждый скролл. Это прямое следствие принципа «БД — источник истины».
Почему RemoteMediator реализован поверх Room (RemoteMediator + PagingSource из Room), а не возвращает страницы прямо в Paging? Что конкретно ломается в варианте «PagingSource напрямую из сети»? ★

В Paging 3 RemoteMediator — это не источник для UI, а «дозагрузчик кэша»: он срабатывает только когда локальных данных не хватает (LoadType.APPEND/REFRESH), пишет страницы в Room в транзакции и обновляет remote_keys. UI всегда читает из PagingSource, который строит Room (db.dao().pagingSource()). Это даёт single source of truth: лента работает офлайн, переживает поворот и смерть процесса, а сеть не дёргается на каждый скролл.

Вариант «PagingSource напрямую из сети» теряет всё это: нет офлайна, при пересоздании PagingSource всё грузится заново, и нет места, где консистентно хранить курсор.

Тонкость: запись в Room надо делать в withTransaction (очистка при REFRESH + upsert + обновление ключей атомарно), иначе между clear и insert UI мигнёт пустой лентой. И эта запись инвалидирует Room-PagingSource, что и заставляет Paging перечитать свежие данные.

Общий принцип: refresh обновляет только то, чем владеет сервер; то, что ещё «в полёте» в outbox, неприкосновенно до подтверждения. Это ровно тот edge-case, который отличает «нарисовал стрелочки» от «подумал про консистентность при гонке refresh ↔ pending-запись».

Кэш изображений

Двухуровневый кэш: memory (LRU) для мгновенного повторного показа + disk для переживания рестарта. Готовое решение — Coil/Glide, но на собесе проговорите устройство: ключ кэша (URL + размер/трансформация), вытеснение по LRU, лимиты размера, downsampling под размер ImageView/Composable (не держать 12 Мп для превью 100×100), уважение HTTP-кэш-заголовков, отмена загрузки при ресайклинге элемента списка.

ImageCache (схема LRU)
// Идея двухуровневого кэша; в проде — Coil ImageLoader с memory+disk cache
class ImageCache(maxMemBytes: Int, private val disk: DiskLruCache) {
    private val memory = object : LruCache<String, Bitmap>(maxMemBytes) {
        override fun sizeOf(key: String, value: Bitmap) = value.byteCount
    }
    suspend fun load(url: String, reqW: Int, reqH: Int): Bitmap {
        val key = "$url@${reqW}x$reqH"           // размер — часть ключа
        memory.get(key)?.let { return it }        // L1: память
        disk.get(key)?.let {                       // L2: диск
            val bmp = it.decodeSampled(reqW, reqH) // downsampling под размер
            memory.put(key, bmp); return bmp
        }
        return download(url).also {                // L3: сеть → заполняем кэши
            disk.put(key, it); memory.put(key, decode(it, reqW, reqH))
        }.let { memory.get(key)!! }
    }
}

Аналитика и трекинг событий

События нельзя терять и нельзя слать по одному (батарея, трафик). Решение: буфер в БД (events table) → батчинг по размеру/времени → отправка через WorkManager при сети. Каждое событие с eventId (UUID) для дедупликации на бэкенде, отправленные удаляем после ACK. При переполнении — дроп старых или сэмплирование. По сути это тот же outbox-паттерн, что и для записей.

Двухуровневый LRU-кэш картинок: ключ — URL плюс запрошенный размер. Почему размер обязан входить в ключ, и какой баг возникает, если ключ — просто URL? ★

Один и тот же URL рендерится в разных местах в разных размерах: превью 100×100 в списке и фуллскрин 1080px. Если ключ — только URL, в кэш попадёт первый декодированный bitmap, и все остальные места получат его же: либо размытое превью растянут на фуллскрин, либо (хуже) полноразмерный bitmap положат под превью и съедят память впустую.

Поэтому ключ memory-кэша = URL + целевой размер/трансформация (Coil так и делает: ключ учитывает size и transformations). Связанный момент — downsampling: BitmapFactory.Options.inSampleSize декодирует картинку сразу уменьшенной (степень двойки), чтобы не держать 12 Мп ради превью 100×100 — без этого OutOfMemory на бюджетных устройствах при скролле почти гарантирован. Диск-кэш при этом часто ключуют по URL (исходник), а ресайз делают при декоде.

Вы выбрали WorkManager для синхронизации и сказали «не realtime». А почему нельзя просто держать открытый WebSocket из foreground-сервиса, чтобы синк был мгновенным? В чём системная цена? ★

Постоянный сокет из foreground-сервиса не даёт ОС усыпить радио и процесс, поэтому жрёт батарею и упирается в фоновые ограничения. С Android 8.0 (API 26) фоновые сервисы ограничены, а в Doze/App Standby сеть и wake-up'ы режутся — «вечный сокет в фоне» система всё равно прибьёт или заморозит.

Правильный паттерн — гибрид. WebSocket держим только пока приложение в foreground (мгновенный realtime для активного пользователя), а для фона/убитого приложения будит FCM high-priority push, который пробивает Doze. Данные при этом обычно тянут отдельно (push как «звоночек», не как транспорт payload): у FCM data-сообщения лимит payload 4 КБ и доставка best-effort. WorkManager отвечает за гарантированную отложенную работу (дренаж outbox, дельта-синк) с backoff (минимум 10 секунд, MIN_BACKOFF_MILLIS) и constraints.

Цена realtime-сокета — батарея и борьба с background-ограничениями; цена FCM + WorkManager — задержка. Senior называет именно этот гибрид, а не одну крайность.

Real-time транспорт: WebSocket и переподключение

Дизайн чата или живой ленты почти всегда упирается в один вопрос: как доставлять обновления мгновенно. Polling простой, но при коротком интервале жжёт батарею и трафик, а при длинном даёт задержку. Для настоящего realtime берут WebSocket поверх TLS: одно персистентное двунаправленное соединение, низкая latency для доставки сообщений и статусов «доставлено/прочитано».

WebSocket сам не переподключается

Это главная ловушка. Соединение рвётся при смене сети, тайм-ауте, перезагрузке сервера — и библиотека просто отдаёт onFailure/onClosed. Реконнект приходится писать самостоятельно: экспоненциальный backoff плюс jitter (случайная добавка), иначе тысячи клиентов после падения сервера ударят синхронным штормом переподключений. Мёртвое соединение детектируют heartbeat'ом (ping/pong): нет понга за таймаут — считаем сокет мёртвым и реконнектимся. На балансировке нужны sticky sessions, чтобы клиент после подключения оставался на том же чат-сервере.

Жизненный цикл сокета: реконнект с backoff+jitter, а после него — обязательный ресинк пропущенного

Ключевой шаг — ресинк на каждом реконнекте. Пока сокета не было, могли прийти сообщения; при возврате клиент дотягивает их дельтой (messagesSince=lastSeenSeq). Отправка идёт через тот же надёжный outbox со статусами и Idempotency-Key, а порядок гарантирует монотонный clientSeq. Presence (online/typing) и last seen — отдельный, более дешёвый и «эфемерный» поток: его не персистят и не гарантируют доставку.

ChatSocketClient.kt (реконнект-цикл)
suspend fun runConnectionLoop() {
    var attempt = 0
    while (currentCoroutineContext().isActive) {
        try {
            connect()                       // suspend до закрытия сокета
            attempt = 0                     // успех — сбросили счётчик
            resyncSince(lastSeenSeq)        // дотянуть пропущенное
            listen()                        // suspend, пока сокет жив
        } catch (e: Exception) {
            val backoff = minOf(BASE * (1L << attempt), MAX_DELAY)
            val jitter = Random.nextLong(JITTER)   // ломаем синхронный шторм
            delay(backoff + jitter)
            attempt++
        }
    }
}
Зачем jitter в backoff, если экспонента и так разносит попытки во времени? ★

Экспонента разносит попытки одного клиента, но не разных. Представь: сервер упал, тысяча клиентов одновременно потеряла сокет. Без jitter все они отсчитают одинаковый backoff и ударят реконнектом ровно в одну и ту же миллисекунду — и так на каждой итерации, пока сервер не ляжет повторно от этого же шторма (thundering herd). Случайная добавка размазывает попытки по окну, и нагрузка на восстановившийся сервер становится плавной. Это дешёвая строчка кода, которая отделяет «работает на демо» от «переживает массовый обрыв».

Push-уведомления и фоновые ограничения

Сокет хорош, пока приложение на экране. В фоне ОС усыпляет радио и режет фоновую активность, поэтому держать соединение нельзя — нужен другой канал разбудки. Это FCM (Firebase Cloud Messaging).

Почему сокет в фоне нежизнеспособен

С Android 8.0 (API 26) фоновые сервисы ограничены: запустить долгоживущий сервис из фона нельзя без foreground-уведомления. А Doze и App Standby при выключенном экране откладывают сетевой доступ и wake-up'ы пачками в редкие «окна обслуживания». Открытый сокет в таком режиме либо замёрзнет, либо ОС просто прибьёт процесс. Foreground-сервис ради вечного сокета — это постоянное уведомление и быстрый расход батареи, на что пользователи и Play реагируют плохо.

FCM как «звоночек», а не транспорт

High-priority data-сообщение пробивает Doze и будит приложение. Но у data-payload лимит 4 КБ, а доставка best-effort — FCM может схлопывать одинаковые сообщения и задерживать их. Поэтому push используют как сигнал «проснись и сходи за данными», а сами данные тянут отдельным запросом (через WorkManager). Различай два типа:

Если хотите контролировать поведение в фоне (сходить за данными, построить кастомное уведомление), шлите data message, а не notification. И помните про lifecycle device-токена: он ротируется и протухает, его надо ловить в onNewToken, слать на бэкенд и дедуплицировать. Гарантированную фоновую работу подробнее разбирает глава Фоновая работа.

Пришёл notification message, а приложение в фоне — почему onMessageReceived не вызвался, и как это лечить? ★

Так задумано: у notification message в фоне уведомление в трей кладёт сама система, минуя код приложения, — onMessageReceived срабатывает только когда приложение в foreground. Если нужно перехватить пуш в фоне (сходить за данными, построить кастомное уведомление, обновить кэш), сервер должен слать data message с high-priority: тогда onMessageReceived зовётся при любом состоянии и пробивает Doze. Частая ошибка — слать notification и удивляться, что фоновая логика не отрабатывает.

Кейс: чат / мессенджер

Второй по частоте кейс после ленты. Каркас тот же (Room как SSOT, outbox, WorkManager), но появляются realtime, статусы доставки и порядок. Разберём отличия от ленты предметно.

Транспорт: гибрид

Активный чат — WebSocket (мгновенная доставка и статусы). Фон/убитое приложение — FCM будит, пропущенное дотягиваем дельтой по messagesSince при возврате в foreground. На реконнекте — обязательный ресинк. Это прямое применение двух параграфов выше.

Статусы и идемпотентность отправки

Сообщение проходит цепочку sending → sent → delivered → read. Отправка идёт через тот же outbox: записали локально со статусом sending и временным clientId, ушло на сервер — sent (сервер вернул свой serverId), ACK получателя — delivered, прочитал — read. Idempotency-ключ обязателен: если ответ на «отправить» потерялся, повтор не задвоит сообщение в чате.

Message.kt
@Entity(tableName = "messages")
data class MessageEntity(
    @PrimaryKey val clientId: String,   // UUID, он же idempotency-ключ
    val serverId: String?,              // приходит после ACK сервера
    val chatId: String,
    val text: String,
    val clientSeq: Long,                // монотонный порядок отправителя
    val serverTs: Long?,                // авторитетное время сервера
    val status: MessageStatus,          // SENDING/SENT/DELIVERED/READ/FAILED
)

enum class MessageStatus { SENDING, SENT, DELIVERED, READ, FAILED }

Порядок: clientSeq + серверный timestamp

Часы устройств врут, поэтому единого «времени» нет. Решение двухслойное: внутри одного отправителя порядок задаёт монотонный clientSeq (растёт локально), а глобальный порядок в чате — авторитетный serverTs, который проставляет сервер при приёме. UI сортирует по serverTs, а не отправленные сообщения (ещё без serverTs) клеит в хвост по clientSeq. Конфликтов почти нет: чат append-only, никто не редактирует чужие сообщения.

Presence, группы, фон

  • Presence / last seen / typing — эфемерный поток поверх сокета, доставку не гарантируем, в БД не персистим.
  • Группы и fan-out — в личке сервер шлёт обоим участникам; в группе разворачивает сообщение всем (fan-out), а клиент дедуплицирует по serverId.
  • Doze — будит группа сообщений плохо: для групп с активным трафиком сервер схлопывает пуши в «у вас N новых», а данные клиент тянет сам.
  • E2E-шифрование (кратко) — шифруем на клиенте, сервер видит только зашифрованный blob; это ломает серверный поиск и серверные превью, зато даёт приватность.

Чем чат отличается от ленты

Лента — read-mostly, pull-based, конфликты на пользовательских полях (лайк). Чат — write-heavy, push-based (realtime), append-only без конфликтов, зато с жёсткими требованиями к порядку, статусам и идемпотентности. Совпадает фундамент: локальная БД, очередь исходящих, фоновая досылка.

Два устройства одного пользователя в одном чате. Как избежать дублей сообщений и рассинхрона прочитанного? ★

Дубли гасит идемпотентность по serverId: каждое входящее клиент проверяет на наличие в БД и игнорирует повтор (сервер при fan-out шлёт сообщение на оба устройства одного пользователя). Своё отправленное сообщение, прилетевшее назад по сокету, склеивается с локальным по clientId, а не добавляется заново.

Прочитанное синхронизируют через серверный «read cursor» на чат: устройство шлёт «прочитано до serverTs=X», сервер хранит максимум и рассылает его всем устройствам пользователя. Тогда отметка «прочитано» сходится на всех экранах, даже если читали с другого телефона. Без общего курсора каждое устройство считало бы прочитанное по-своему и счётчики непрочитанных разъезжались бы.

Кейс: библиотека загрузки изображений

Иногда просят спроектировать не приложение, а библиотеку — чаще всего загрузчик картинок (свой Coil/Glide). Кейс проверяет, видите ли вы за простым load(url).into(view) координацию кэшей, декодирования, отмены и lifecycle.

Fluent API и Engine-координатор

Снаружи — текучий API: load(url).into(view). Внутри запрос проходит через Engine — координатор, который собирает результат из подсистем: проверяет memory-кэш, потом disk-кэш, потом запускает загрузку. Подсистемы: Fetcher (достать байты: сеть, файл, content URI), Decoder (байты → bitmap с downsampling под целевой размер).

Двухуровневый кэш и память

Memory-кэш — LRU по байтам bitmap, ключ = URL + целевой размер/трансформация (иначе превью и фуллскрин перепутаются). Disk-кэш — для переживания рестарта, обычно ключ по URL-исходнику. Чтобы не плодить аллокации и не дёргать GC при скролле, декод идёт в переиспользуемый buffer из bitmap pool (inBitmap). И downsampling обязателен: inSampleSize декодирует уменьшенную картинку под размер view, чтобы 12-мегапиксельное фото не заняло память ради превью 100×100.

Отмена при rebind и lifecycle

В RecyclerView/LazyColumn view переиспользуется: пока грузилась картинка для позиции 3, ViewHolder уже привязан к позиции 50. Если не отменить старый запрос, во view мелькнёт чужая картинка. Поэтому загрузку привязывают к view-тегу: новый load отменяет предыдущий на этом же view (rebind = cancel). И запрос привязывают к lifecycle: ушли с экрана — загрузка отменяется, чтобы не тратить трафик и не держать ссылку на мёртвую view.

ImageEngine.kt (упрощённо)
class ImageEngine(
    private val memory: LruCache<String, Bitmap>,
    private val disk: DiskCache,
    private val fetcher: Fetcher,
) {
    private val inFlight = mutableMapOf<ImageView, Job>()

    fun load(url: String, view: ImageView, scope: CoroutineScope) {
        inFlight.remove(view)?.cancel()          // rebind: отменяем старый запрос
        val key = "$url@${view.width}x${view.height}"
        memory.get(key)?.let { view.setImageBitmap(it); return }
        inFlight[view] = scope.launch {           // scope привязан к lifecycle
            val bytes = disk.get(url) ?: fetcher.fetch(url).also { disk.put(url, it) }
            val bmp = decodeSampled(bytes, view.width, view.height)
            memory.put(key, bmp)
            view.setImageBitmap(bmp)
        }
    }
}
Десять элементов списка грузят один и тот же URL одновременно. Как не сходить в сеть десять раз? ★

Нужен request dedup (coalescing) по in-flight запросам: Engine держит map «ключ → уже запущенный Deferred». Первый запрос на URL стартует загрузку и кладёт Deferred в map; остальные девять не запускают новую сеть, а подписываются (await()) на тот же результат. Когда загрузка завершилась — все десять получают bitmap, запись из map убирается.

Без дедупа десять одинаковых GET'ов жгут трафик и могут декодировать один bitmap десять раз. Тот же приём — request coalescing — спасает и от cache stampede на уровне данных: толпа запросов в момент истечения TTL схлопывается в один поход за свежим.

Стратегии кэширования и инвалидация

«Не обсудил инвалидацию кэша» — типовая провальная фраза на собесе. Кэш легко добавить и тяжело инвалидировать правильно, поэтому нужна понятная систематика.

Уровни кэша на клиенте

  • Память (LRU) — мгновенный повторный показ, гибнет с процессом.
  • Диск / Room — переживает рестарт, источник истины для офлайна.
  • HTTP-кэш OkHttp — прозрачный кэш по заголовкам, дешёвый слой под Retrofit.

Чем инвалидировать

Базовая стратегия — TTL: у записи есть «возраст», по истечении считаем устаревшей. Дёшево и self-healing, но грубо. Точнее — валидация по ETag/Last-Modified: клиент шлёт If-None-Match, сервер отвечает 304 Not Modified (тело не качаем, только обновляем timestamp). Где staleness критичен — добавляют точечную event/version-based инвалидацию (пуш «данные X изменились» или счётчик версии).

Для UX почти везде поверх этого работает stale-while-revalidate: сразу показываем кэш, параллельно в фоне обновляем и доливаем свежее — пользователь не ждёт сеть (см. таблицу политик чтения выше).

⚠️
Cache stampede. Когда TTL истёк, толпа запросов одновременно ломится за свежими данными и кладёт бэкенд (или дублирует работу на клиенте). Лечится request coalescing (схлопнуть параллельные запросы в один in-flight) и jitter в TTL (чтобы записи протухали не разом). Тот же приём, что и дедуп загрузок картинок.

И не забудьте про prefetch (тянуть следующую страницу заранее по prefetchDistance — плавный скролл ценой трафика) и про показ «возраста данных» пользователю (баннер «обновлено N минут назад»), чтобы staleness был честным, а не скрытым.

TTL истёк ровно в момент, когда экран открыли офлайн. Что показать — пустоту или протухший кэш? ★

Протухший кэш — почти всегда. TTL означает «пора сходить проверить свежесть», а не «данные стали ядовитыми». Если сети нет, ревалидация невозможна, и выбор между «показать вчерашнюю ленту» и «показать пустой экран» очевиден в пользу данных. Это и есть дух stale-while-revalidate: stale лучше, чем nothing.

Нюанс — честность: показав протухшее, дай пользователю понять это (баннер «офлайн, данные могут быть неактуальны» + «возраст» данных). Исключение — данные, где устаревшее опаснее пустого: баланс счёта, одноразовый токен, цена в момент оплаты. Там политика network-only и явная ошибка вместо протухшего значения.

Оценка масштаба (back-of-the-envelope)

«Прыгнул к коробкам на диаграмме без оценки и без API» — частая причина провала. Оценка на mobile-собесе — это не QPS бэкенда, а прикидка клиентских величин, из которых выводятся конкретные лимиты.

Что оцениваем

  • Размер локального кэша = размер одной записи × число удерживаемых элементов.
  • Трафик сессии = page size × число подгрузок × размер страницы + картинки.
  • Частота wake-up'ов синка = как часто будим устройство ради обновлений.

Пример на пальцах: лента

Карточка ленты — метаданные ~1 КБ + превью-картинка ~50 КБ. Держим в кэше ~500 элементов. Метаданные: 500 × 1 КБ ≈ 0.5 МБ в Room — копейки. Картинки: 500 × 50 КБ ≈ 25 МБ — вот под это и ставим явный лимит дискового кэша Coil и LRU-вытеснение. За сессию пользователь скроллит ~10 страниц по 20 элементов = 200 карточек ≈ 200 × 51 КБ ≈ 10 МБ трафика, в основном картинки. Отсюда вывод: картинки тянем в варианте под размер (WebP/AVIF), а не оригиналом.

💡
Откуда берутся числа в дизайне. Из этой прикидки прямо выводятся page size (20–50, баланс трафика и числа раунд-трипов), prefetchDistance (плавность vs лишний трафик), лимит дискового кэша и интервал дельта-синка. Назвав порядок величин, вы показываете, что лимиты обоснованы, а не взяты с потолка.
Интервьюер не дал ни одной цифры. Откуда брать числа для оценки, чтобы это не выглядело гаданием? ★

Числа вы не угадываете, а проговариваете как явные допущения и проверяете их вслух: «допустим, карточка ~1 КБ метаданных и ~50 КБ превью — это разумно для JPEG/WebP такого размера, поправьте, если не так». Интервьюер либо соглашается, либо даёт реальную цифру — и то и другое в плюс, потому что показывает, что вы думаете порядками величин, а не точными значениями.

Хорошие якоря: типичный размер JSON-записи — единицы КБ, превью-картинка — десятки КБ, экран ленты — ~10 элементов, сессия — единицы-десятки страниц. Важна не точность до байта, а то, что итоговые лимиты (кэш, page size, интервал синка) выведены из оценки, а не названы наугад.

Сценарии-минимум

  • Переживание рестарта процесса — операция в outbox остаётся после «перезапуска» (пересоздание объектов с той же БД).
  • Идемпотентность — ретрай при потерянном ответе не задваивает результат.
  • Разрешение конфликта — 409 обрабатывается по выбранной стратегии (LWW/версии).
  • Dead-letter без зацикливания — операция, превысившая лимит попыток, выкидывается, а не блокирует очередь вечно.
Как тестировать exponential backoff с задержками в минуты, не заставляя CI реально ждать? ★

Через виртуальное время runTest. Внутри TestScope работает TestDispatcher с виртуальными часами: delay(60_000) не блокирует поток, а просто двигает виртуальное время, когда вы вызываете advanceTimeBy(60_000) или advanceUntilIdle(). Тест с backoff на минуты выполняется за миллисекунды и при этом детерминированно проверяет, что повтор случился ровно через расчётную задержку.

Условие — чтобы продакшен-код брал диспетчер извне (DI), а не хардкодил Dispatchers.IO: иначе подменить часы в тесте не выйдет. Реальный Thread.sleep в sync-коде — антипаттерн именно потому, что его нельзя промотать: тест либо реально ждёт, либо его выпиливают как «медленный», и регрессии в backoff проходят незамеченными.

Частые вопросы интервьюера

С чего вы начнёте проектирование? Middle+

Не с экранов и не с кода. Сначала уточняю требования (функциональные + нефункциональные) и ограничения: офлайн нужен? сколько данных? какая сеть и устройства? Затем фиксирую scope (что НЕ делаем), и только потом рисую высокоуровневую архитектуру слоёв. В mobile нефункциональные требования — офлайн, latency, батарея, трафик — определяют дизайн сильнее самих фич, поэтому я проговариваю их явно и веду диалог, а не монолог.

Что такое offline-first и как его реализовать? Senior

Это подход, где локальная БД — источник истины, а сеть — слой оптимизации, не обязательное условие. UI читает только из БД (через Flow), поэтому работает без интернета. Записи идут оптимистично: сразу в БД + в outbox (долговечную очередь). Фоновый WorkManager с constraints дренирует outbox и подтягивает изменения дельтой. Конфликты разрешаем по стратегии (LWW/версии/merge). Обязательна идемпотентность операций (Idempotency-Key) — иначе ретраи продублируют действия.

Зачем outbox, если можно просто повторить запрос корутиной? Senior

Корутина в ViewModel умрёт вместе с экраном/процессом — операция потеряется. Outbox — это запись в БД, она переживает убийство процесса, перезагрузку устройства и долгое отсутствие сети. WorkManager гарантированно вернётся к ней позже, с backoff и constraints. То есть outbox даёт долговечность и гарантию доставки, чего сырая корутина не обеспечивает. Это классический паттерн «надёжная очередь исходящих операций».

Как разрешать конфликты при синхронизации? Senior

Зависит от данных. Для некритичных (лайк, настройка) — Last-Write-Wins по timestamp, просто и дёшево. Где важно не затереть чужое — optimistic concurrency: шлём version, сервер отвечает 409 при устаревании, мы перечитываем и решаем. Для независимых полей (профиль) — merge. Для совместного редактирования — CRDT или ручной выбор пользователем. Главное — проговорить, что LWW теряет данные при гонке, и выбрать стратегию под бизнес-критичность поля.

Почему ленту кэшируют в БД, а не грузят из сети напрямую? Middle+

Чтобы лента работала офлайн, переживала поворот экрана и убийство процесса, и не дёргала сеть на каждый скролл. Связка Paging 3 + RemoteMediator + Room: RemoteMediator тянет страницы из сети в Room, PagingSource от Room отдаёт их UI. БД — источник истины, сеть дёргается только когда кэш исчерпан. Минус — данные могут устареть, поэтому делаем pull-to-refresh и показываем «возраст» данных.

Чем оптимистичный апдейт лучше пессимистичного и какова его цена? Middle+

Оптимистичный (сразу меняем UI/БД, отправляем в фоне) даёт мгновенный отклик — критично для лайков, корзины, сообщений. Пессимистичный (ждём ответ сервера, потом меняем) ощущается медленным на плохой сети. Цена оптимизма — нужно корректно откатывать изменение при провале и объяснять пользователю, плюс сложнее логика состояний и обязательна идемпотентность. Для финансовых операций иногда осознанно выбирают пессимистичный путь ради корректности.

Как спроектировать надёжный загрузчик больших файлов? Senior

Резюмируемая загрузка по частям (chunked / resumable upload): храню в БД offset и статусы загруженных кусков, при обрыве догружаю с места, а не сначала. Запускаю через WorkManager с foreground-сервисом (длительная работа + уведомление прогресса), constraints на сеть (можно «только Wi-Fi»), экспоненциальный backoff. Идемпотентность по uploadId, сборка чанков на сервере, проверка контрольной суммы. Так загрузка переживает потерю сети, рестарт приложения и не начинается заново.

Какие нефункциональные требования специфичны для mobile? Middle

Главные: офлайн-доступность и поведение на нестабильной сети; энергопотребление (не будить устройство зря, батчить сеть); экономия трафика (дельта-синхронизация, сжатие, варианты картинок); производительность UI (60+ fps на бюджетных девайсах, отсутствие jank); размер приложения и кэша; устойчивость к убийству процесса ОС и восстановление состояния. Это всё то, чего нет в backend-дизайне и что отличает mobile-секцию.

Как не потерять аналитические события? Middle+

Тот же outbox-паттерн: пишу события в таблицу БД, батчу по размеру/времени и отправляю через WorkManager при наличии сети, удаляю после ACK. Каждому событию — eventId (UUID) для дедупликации на бэкенде (ретраи не задвоят метрики). При переполнении буфера — дроп старых или сэмплирование. Отправлять по одному событию нельзя — это бьёт по батарее и трафику.

Спроектируйте real-time чат: какой транспорт выбрать и как пережить обрыв соединения? Senior

Для активного приложения — WebSocket поверх TLS (персистентное двунаправленное соединение, низкая latency для доставки и статусов). Сокет сам не реконнектится — нужен свой reconnect с экспоненциальным backoff и jitter, плюс heartbeat/ping-pong для детекта мёртвого соединения; балансировка — sticky sessions. Отправка идёт через outbox со статусами и idempotency-ключом, порядок — монотонным clientSeq. Для фона/убитого приложения сокет держать нельзя (Doze, ограничения фоновых сервисов с API 26) — будит FCM high-priority push, а пропущенное клиент дотягивает дельтой по messagesSince. На каждом реконнекте обязателен ресинк.

Push vs WebSocket vs polling — когда что, и почему не держать сокет постоянно? Senior

Polling прост, но при коротком интервале жжёт батарею и трафик, при длинном даёт задержку — годится для редких некритичных обновлений (или long-polling как fallback). WebSocket даёт настоящий realtime, но требует foreground: в фоне ОС усыпляет радио (Doze) и режет фоновые сервисы, поэтому «вечный сокет» сажает батарею и нежизнеспособен. FCM — для фона: high-priority data-сообщение пробивает Doze, но payload ≤ 4 КБ и доставка best-effort, поэтому push используют как «звоночек разбудиться и сходить за данными». Senior-ответ — гибрид: WebSocket в foreground, FCM в фоне, дельта-ресинк на возврате, polling как деградация.

Как именно вы инвалидируете кэш? Что такое stale-while-revalidate? Senior

Базово — TTL: у записи есть «возраст», по истечении считаем устаревшей (дёшево, self-healing, но грубо). Точнее — валидация по ETag/Last-Modified: клиент шлёт If-None-Match, сервер отвечает 304 (тело не качаем). Для UX поверх берут stale-while-revalidate: сразу показываем кэш (мгновенный экран), параллельно обновляем в фоне и доливаем свежее. Явную event/version-based инвалидацию добавляют точечно, где staleness критичен. Подводные камни, которые ценят: cache stampede (толпа запросов в момент истечения TTL — лечится request coalescing и jitter в TTL) и честный показ «возраста данных».

Как удаление на сервере доезжает до офлайн-клиента при дельта-синхронизации? Senior

Через тумбстоуны (soft delete): сервер не стирает запись физически, а помечает deleted=true со свежим updatedAt, и она приходит в обычной дельте по updatedSince — клиент по флагу удаляет её локально. Тумбстоуны живут ограниченный TTL; клиента, который был офлайн дольше TTL, переводят на полный ресинк, иначе он пропустит удаление. Без тумбстоунов отсутствие записи в дельте неотличимо от «не менялась» — и удалённое навсегда зависает в кэше. Любимый каверзный вопрос: наивная дельта тихо ломается именно на удалениях.

Как протестировать sync-движок, чтобы это не было «работает на моём телефоне»? Middle+

Через runTest с виртуальным временем: backoff и таймеры проматывают advanceTimeBy/advanceUntilIdle, не ожидая реальных секунд. Сеть подменяют фейком/MockWebServer (эмуляция 5xx, 409-конфликта, таймаута, обрыва), Room тестируют на in-memory базе (inMemoryDatabaseBuilder), WorkManager — через WorkManagerTestInitHelper + TestDriver. Проверяют: операция переживает рестарт процесса, идемпотентность гасит дубль, конфликт разрешается по стратегии, dead-letter не зацикливает очередь. Ключевой посыл — детерминизм: ни одного реального sleep и ни одной реальной сети в юнит-тестах.

Задача 1. Реализуйте метод syncOutbox, который читает из OutboxDao очередь pending-операций и отправляет их через SyncApi. Если операция выполнена успешно — удалите её из очереди. Если попытка превысила лимит MAX_ATTEMPTS — тоже удалите (dead-letter). В остальных случаях увеличьте счётчик попыток и верните Result.retry(). Используйте типы из файла ниже.

Показать решение 
// Дано:
// @Entity data class OutboxOp(val localId: Long, val attempts: Int, ...)
// interface OutboxDao { suspend fun peek(): List<OutboxOp>; suspend fun ack(op: OutboxOp); suspend fun bumpAttempts(id: Long) }
// interface SyncApi { suspend fun execute(op: OutboxOp): Response<Unit> }
// sealed class Result { object Success : Result(); object Retry : Result() }

const val MAX_ATTEMPTS = 10

suspend fun syncOutbox(dao: OutboxDao, api: SyncApi): Result {
    val ops = dao.peek()
    for (op in ops) {
        val result = runCatching { api.execute(op) }
        when {
            result.isSuccess            -> dao.ack(op)
            op.attempts >= MAX_ATTEMPTS -> dao.ack(op)   // dead-letter
            else -> {
                dao.bumpAttempts(op.localId)
                return Result.Retry
            }
        }
    }
    return Result.Success
}

Объяснение: паттерн outbox гарантирует доставку через хранилище в БД — операции выживают после гибели процесса. Dead-letter по лимиту попыток не даёт цикличным ошибкам заблокировать очередь навсегда. runCatching перехватывает сетевые исключения без раскручивания стека выше. Сложность — O(n) по размеру пачки, n ≤ 50 (Limit в peek).

Задача 2. Напишите реализацию RemoteMediator.load() для ленты с курсорной пагинацией. При REFRESH — сбросить кэш и загрузить первую страницу. При APPEND — прочитать следующий курсор из remote_keys и дозагрузить. При PREPEND — вернуть «конец пагинации» (новых элементов в начале нет). Всё пишется в одной транзакции. Сигнатура: suspend fun load(loadType: LoadType, state: PagingState<Int, ProductEntity>): MediatorResult.

Показать решение 
// Дано:
// data class RemoteKey(val label: String, val nextCursor: String?)
// interface FeedApi { suspend fun getFeed(cursor: String?, limit: Int): Page }
// data class Page(val items: List<ProductDto>, val nextCursor: String?)
// AppDatabase: feedDao(), keysDao() — Room DAO

@OptIn(ExperimentalPagingApi::class)
override suspend fun load(
    loadType: LoadType,
    state: PagingState<Int, ProductEntity>,
): MediatorResult {
    val cursor: String? = when (loadType) {
        LoadType.REFRESH  -> null
        LoadType.PREPEND  -> return MediatorResult.Success(endOfPaginationReached = true)
        LoadType.APPEND   -> db.keysDao().get("feed")?.nextCursor
            ?: return MediatorResult.Success(endOfPaginationReached = true)
    }
    return try {
        val page = api.getFeed(cursor = cursor, limit = state.config.pageSize)
        db.withTransaction {
            if (loadType == LoadType.REFRESH) {
                db.feedDao().clear()
                db.keysDao().delete("feed")
            }
            db.keysDao().upsert(RemoteKey("feed", page.nextCursor))
            db.feedDao().upsertAll(page.items.map { it.toEntity() })
        }
        MediatorResult.Success(endOfPaginationReached = page.nextCursor == null)
    } catch (e: IOException) {
        MediatorResult.Error(e)
    }
}

Объяснение: транзакция вокруг clear + upsert исключает момент, когда БД пуста и UI мигает. Курсорная пагинация стабильна при вставках сервером — в отличие от offset, «прыгающей» ленты нет. MediatorResult.Error сигнализирует Paging-библиотеке показать состояние ошибки с кнопкой retry.

Задача 3. Реализуйте функцию pushUpdate: отправьте обновлённую запись на сервер с оптимистичным контролем версий. Если сервер вернул 200 — верните актуальную серверную версию. Если 409 (конфликт) — прочитайте серверную версию и применяйте LWW: если локальная запись новее по updatedAt, верните SyncOutcome.Retry; если серверная — верните SyncOutcome.OverwrittenByServer. Любой другой код — тоже Retry.

Показать решение 
// Дано:
// data class ProductEntity(val id: String, val updatedAt: Long, val serverVersion: Long, ...)
// interface SyncApi {
//     suspend fun update(id: String, body: ProductDto, ifMatchVersion: Long): Response<ServerProduct>
//     suspend fun get(id: String): ServerProduct
// }
// data class ServerProduct(val version: Long, val updatedAt: Long)
// sealed class SyncOutcome {
//     data class Synced(val newVersion: Long) : SyncOutcome()
//     data class OverwrittenByServer(val server: ServerProduct) : SyncOutcome()
//     object Retry : SyncOutcome()
// }

suspend fun pushUpdate(local: ProductEntity, api: SyncApi): SyncOutcome {
    val resp = api.update(
        id = local.id,
        body = local.toDto(),
        ifMatchVersion = local.serverVersion,
    )
    return when (resp.code()) {
        200 -> SyncOutcome.Synced(resp.body()!!.version)
        409 -> {
            val server = api.get(local.id)
            if (local.updatedAt > server.updatedAt) {
                SyncOutcome.Retry             // наша версия свежее — попробуем снова
            } else {
                SyncOutcome.OverwrittenByServer(server)  // сервер победил по LWW
            }
        }
        else -> SyncOutcome.Retry
    }
}

Объяснение: поле ifMatchVersion реализует optimistic concurrency — сервер отвечает 409, если запись уже изменена кем-то ещё. LWW по updatedAt — дешёвая стратегия разрешения конфликтов; подходит для некритичных данных вроде лайков или настроек. Для финансовых или критичных полей нужна ручная сверка или merge.

Задача 4. Реализуйте reconnect для WebSocket-клиента чата: при обрыве переподключаться с экспоненциальным backoff и jitter, ограничив задержку максимумом; при успешном подключении сбрасывать счётчик попыток и запускать дельта-ресинк (дотянуть сообщения, пришедшие пока сокета не было). Только корутины, без блокирующих sleep.

Показать решение 
// Дано:
// suspend fun connect()            — подключиться (suspend до закрытия сокета)
// suspend fun resyncSince(seq: Long) — дотянуть пропущенное
// suspend fun listen()             — слушать сокет, пока он жив
// var lastSeenSeq: Long

suspend fun runConnectionLoop() {
    var attempt = 0
    while (currentCoroutineContext().isActive) {
        try {
            connect()
            attempt = 0                          // успех — сбросили счётчик
            resyncSince(lastSeenSeq)             // ресинк на КАЖДОМ reconnect
            listen()                             // suspend, пока сокет жив
        } catch (e: Exception) {
            val backoff = minOf(BASE * (1L << attempt), MAX_DELAY)
            val jitter = Random.nextLong(JITTER) // ломаем синхронный шторм
            delay(backoff + jitter)
            attempt++
        }
    }
}

Объяснение: jitter разносит реконнекты тысяч клиентов после падения сервера, иначе они ударят синхронным штормом (thundering herd) и снова положат сервер. Ресинк на каждом подключении обязателен — без него теряются сообщения за время обрыва. Тестируется через runTest + advanceTimeBy (виртуальное время вместо реальных секунд), поэтому delay, а не Thread.sleep: блокирующий sleep нельзя промотать.

Задача 5. Напишите compactOutbox(ops): свернуть очередь outbox-операций над одной сущностью перед отправкой. Правила: CREATE затем DELETE для ещё не отправленного объекта = выкинуть обе (no-op); несколько UPDATE подряд над одним id = оставить последний (мерж payload); порядок операций над разными id и зависимости (UPDATE/DELETE не раньше CREATE того же id) сохранить.

Показать решение 
// Дано:
// enum class OpKind { CREATE, UPDATE, DELETE }
// data class Op(val entityId: String, val kind: OpKind, val payload: String?)

fun compactOutbox(ops: List<Op>): List<Op> {
    // группируем по entityId, сохраняя порядок ПЕРВОГО появления группы
    val groups = LinkedHashMap<String, MutableList<Op>>()
    for (op in ops) groups.getOrPut(op.entityId) { mutableListOf() }.add(op)

    val result = mutableListOf<Op>()
    for ((_, group) in groups) {
        val hasCreate = group.any { it.kind == OpKind.CREATE }
        val endsWithDelete = group.last().kind == OpKind.DELETE
        if (hasCreate && endsWithDelete) continue   // создан и удалён — no-op

        val out = mutableListOf<Op>()
        group.firstOrNull { it.kind == OpKind.CREATE }?.let { out += it }
        // серию UPDATE схлопываем в последний
        group.lastOrNull { it.kind == OpKind.UPDATE }?.let { out += it }
        if (group.last().kind == OpKind.DELETE) out += group.last()
        result += out
    }
    return result
}

Объяснение: LinkedHashMap держит межсущностный порядок (важно при зависимостях между сущностями). Внутри группы create+delete для неотправленного объекта схлопывается в ничто — лишний раунд-трип не уходит. Сложность O(n). Подвох за рамками задачи: после реального CREATE временный локальный id надо подменить на серверный во всех последующих операциях — это отдельный шаг id-ремаппинга на отправке.
