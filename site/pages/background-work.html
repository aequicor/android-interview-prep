Платформа Android

Фоновая работа

Как выполнить задачу, когда пользователь свернул приложение или погасил экран, — и почему система этому всеми силами мешает. Doze, App Standby, лимиты фоновых сервисов, WorkManager, foreground service, AlarmManager и FCM как способ разбудить процесс. Тема, на которой видно, отличаете ли вы «надо сделать гарантированно» от «надо сделать прямо сейчас» — это два разных инструмента.

Middle Senior Частый вопрос

Ограничения фона: Doze, App Standby, лимиты

Начнём с главного, что нужно держать в голове весь разговор: для системы батарея пользователя важнее вашей задачи. Любое приложение, ушедшее в фон, для Android — подозреваемый, который тратит CPU и сеть, пока экран погашен. Поэтому начиная с разных версий ОС платформа последовательно закручивала гайки. Если кандидат не понимает этой логики, он будет городить голые Service и Thread, которые в проде просто не запустятся.

Doze (с Android 6.0, API 23)

Телефон лежит на столе: экран погашен, не на зарядке, не двигается. Через какое-то время система вводит устройство в Doze — глубокий сон. В этом режиме сеть для приложений отрезается, WakeLock игнорируются, JobScheduler и синхронизации откладываются, обычные алармы не срабатывают. Процесс не убит — он заморожен.

Периодически система ненадолго выходит из сна — это maintenance-окно. В нём прогоняются накопившиеся задачи, алармы и синхронизации, открывается сеть. Затем снова сон, и окна идут всё реже: чем дольше телефон лежит, тем длиннее интервалы между ними. Расчёт простой — отложенную работу схлопнуть в редкие пачки, чтобы радио и CPU просыпались не по двадцать раз в час, а пару раз за ночь.

App Standby Buckets

Doze смотрит на устройство целиком. App Standby — на каждое приложение в отдельности и расставляет их по «корзинам» (buckets) по тому, как часто вы ими пользуетесь. Чем реже — тем жёстче лимиты на задачи, алармы и доступ к сети.

BucketКогдаОграничения
activeИспользуется прямо сейчас или только чтоНет
working_setВ регулярном употребленииМягкие, на задачи и алармы
frequentЧасто, но не каждый деньСильнее
rareРедкоЖёсткие, плюс лимит на сеть
restrictedЖрёт ресурсы или ведёт себя странноСамые жёсткие из «обычных»

Есть ещё особый bucket never — для установленных, но ни разу не запущенных приложений. Корзина пересматривается динамически: открыли приложение — оно подскочило в active, забросили на неделю — съехало в rare.

Background execution limits (с Android 8.0, API 26)

Это уже не про спящий телефон, а про обычную работу. Начиная с Android 8.0 действуют два жёстких правила. Первое: приложение в фоне не может держать запущенным фоновый Service — через несколько минут после ухода в фон система его остановит (точнее, startService() для фонового приложения бросит IllegalStateException). Второе: большинство неявных broadcast'ов больше нельзя ловить через манифест — статически зарегистрированный BroadcastReceiver на системное событие вроде смены коннекта просто не вызовется (есть исключения, например BOOT_COMPLETED). Подробнее про сами компоненты — в главе Компоненты и IPC.

🛑
Старый рецепт мёртв. «Запущу Service, повешу WakeLock, а внутри Thread с циклом синхронизации» — на современном Android это не работает: сервис остановят, WakeLock в Doze проигнорируют, broadcast не прилетит. Любую отложенную работу отдают планировщику (WorkManager), а не держат руками.
Почему maintenance-окна со временем становятся реже, а не идут с фиксированным периодом? ★

Это компромисс между свежестью данных и расходом батареи, и он нелинейный. Каждое окно — это пробуждение CPU и, главное, радиомодуля: подъём сети из сна стоит ощутимого заряда независимо от объёма переданного. Если телефон лежит пять минут, шанс, что пользователь вот-вот вернётся, высок — окна частые, чтобы данные были свежими. Если он лежит восемь часов (ночь), смысла будить радио каждые 10 минут нет: пользователь спит, и система растягивает интервалы, экономя десятки циклов пробуждения. То есть период не фиксирован намеренно — фиксированный давал бы либо лишний расход на коротком сне, либо неприемлемую задержку компенсировать было бы нечем. Вывод для разработчика: точное время доставки в Doze не гарантировано в принципе, и закладываться на него (через обычные алармы или периодический Work) нельзя.

WorkManager: гарантированная отложенная работа

WorkManager — ответ Google на весь зоопарк выше. Это рекомендованный способ для задач, которые должны выполниться, но не обязаны прямо сейчас: отправить накопленную аналитику, синхронизировать кэш, сжать и залить фото. Ключевое слово — гарантия: Work переживает перезапуск приложения и перезагрузку устройства, потому что состояние лежит в БД (на старых API под капотом он использует JobScheduler/AlarmManager). Когда условия выполнения наступят — задача отработает, даже если приложение к тому моменту закрыто.

Worker vs CoroutineWorker

Единица работы — класс-наследник. Простой Worker выполняет doWork() синхронно на фоновом потоке из пула. CoroutineWorker делает doWork() suspend-функцией — это идиоматичный выбор для проекта на корутинах: можно дёргать suspend-репозитории, переключать диспетчеры, корректно отменяться. Если вы уже на корутинах (см. Корутины), берите CoroutineWorker.

UploadWorker.kt
class UploadWorker(
    ctx: Context,
    params: WorkerParameters
) : CoroutineWorker(ctx, params) {

    override suspend fun doWork(): Result {
        val id = inputData.getString(KEY_PHOTO_ID) ?: return Result.failure()
        return try {
            repository.uploadPhoto(id)            // suspend-вызов
            Result.success()
        } catch (e: IOException) {
            Result.retry()                         // временный сбой — повторим
        } catch (e: SerializationException) {
            Result.failure()                       // битые данные — повтор бессмысленен
        }
    }

    companion object { const val KEY_PHOTO_ID = "photo_id" }
}

Три исхода doWork(): Result.success() — готово; Result.failure() — провал без надежды (повтор не поможет, не зовём его); Result.retry() — временная неудача, перепланировать по выбранной BackoffPolicy (по умолчанию экспоненциальная, старт 30 секунд; меньше 10 секунд задать нельзя). Различать failure и retry критично: вернёте retry() на неустранимую ошибку — словите бесконечный цикл повторов, который ест батарею; вернёте failure() на сетевой таймаут — потеряете задачу.

Constraints: условия запуска

Можно не запускать Work, пока не сложатся условия: есть сеть, идёт зарядка, устройство в простое, достаточно памяти. Система сама дождётся момента.

WorkRequest.kt
val constraints = Constraints.Builder()
    .setRequiredNetworkType(NetworkType.UNMETERED)  // только Wi-Fi
    .setRequiresCharging(true)
    .build()

val request = OneTimeWorkRequestBuilder<UploadWorker>()
    .setConstraints(constraints)
    .setInputData(workDataOf(UploadWorker.KEY_PHOTO_ID to "42"))
    .setBackoffCriteria(BackoffPolicy.EXPONENTIAL, 30, TimeUnit.SECONDS)
    .build()

OneTime vs Periodic

OneTimeWorkRequest — задача один раз. PeriodicWorkRequest — повторяющаяся, но с важным ограничением: минимальный интервал — 15 минут (тот же лимит, что у JobScheduler). Захотите «каждые 5 минут» — не выйдет, и это специально: частый периодический фон убивал бы батарею. Точное время старта периодики тоже не гарантировано — система сама выбирает момент внутри интервала, учитывая Doze и bucket.

⚠️
Periodic — не «по расписанию». WorkManager даёт гарантию выполнения, но не момента. Для «ровно в 9:00» periodic Work не годится — там нужен AlarmManager. Для «примерно раз в час, когда удобно системе» — годится идеально.

Уникальные цепочки

Чтобы не наплодить дублей одной и той же задачи (классика: пользователь жмёт «синхронизировать» десять раз), используют уникальную работу с политикой коллизии.

Enqueue.kt
WorkManager.getInstance(context).enqueueUniqueWork(
    "sync_photos",
    ExistingWorkPolicy.KEEP,   // уже есть такая работа — не трогаем
    request
)
// REPLACE — отменить старую, поставить новую
// APPEND  — поставить в очередь ПОСЛЕ предыдущей (цепочка)

// последовательная цепочка с зависимостями:
WorkManager.getInstance(context)
    .beginWith(compressWork)
    .then(uploadWork)
    .enqueue()

Наблюдение за статусом

Прогресс и итог Work читают через getWorkInfoByIdFlow() (или ...LiveData()), получая WorkInfo с текущим State.

Observe.kt
WorkManager.getInstance(context)
    .getWorkInfoByIdFlow(request.id)
    .collect { info ->
        when (info?.state) {
            WorkInfo.State.SUCCEEDED -> showDone()
            WorkInfo.State.FAILED    -> showError()
            else -> Unit
        }
    }

Expedited work

Иногда задача и гарантированная, и срочная — например, отправить сообщение, которое пользователь только что написал. Для этого есть setExpedited() (с WorkManager 2.7): система постарается запустить как можно скорее. Но у срочной работы есть системная квота времени выполнения, и на случай её исчерпания вы обязаны указать политику:

Expedited.kt
val urgent = OneTimeWorkRequestBuilder<SendMessageWorker>()
    .setExpedited(OutOfQuotaPolicy.RUN_AS_NON_EXPEDITED_WORK_REQUEST)
    .build()

RUN_AS_NON_EXPEDITED_WORK_REQUEST означает: квота кончилась — выполни как обычную работу, не срочно. Кроме того, на старых версиях ОС expedited work выполняется как foreground service, поэтому CoroutineWorker должен уметь отдать ForegroundInfo — реализуйте getForegroundInfo() с нотификацией.

Когда WorkManager НЕ подходит

Не для точного времени («ровно в 8:00 разбудить») — это AlarmManager. Не для мгновенной доставки извне («пришёл звонок, показать сейчас») — это FCM + foreground service. WorkManager — про «гарантированно, но когда система сочтёт уместным».

Почему минимальный интервал PeriodicWork — именно 15 минут, и можно ли его честно обойти? ★

15 минут — это не цифра WorkManager, а лимит нижележащего JobScheduler (getMinPeriodMillis()), на который WorkManager опирается на современных API. Причина — та же экономия пробуждений: батчинг работы по устройству имеет смысл только при достаточно длинном окне, а более частая периодика свела бы его на нет. «Честно обойти» нельзя: попросите интервал меньше — фреймворк молча поднимет его до минимума. Псевдо-обходы (цепочка OneTimeWork, где каждый воркер в конце ставит следующий с задержкой) технически работают, но это антипаттерн: вы воюете с платформой, теряете гарантии батчинга и первыми попадёте под ужесточение лимитов или в bucket restricted. Если бизнесу реально нужна минутная частота — это сигнал, что задача не фоновая: либо приложение должно быть открыто (foreground service), либо событие должно прилетать снаружи (push), а не опрашиваться по таймеру.

Foreground service и типы FGS

Foreground service (FGS) — это сервис, про который система и пользователь знают, что он работает прямо сейчас и осознанно: воспроизведение музыки, навигация, запись звука, выгрузка большого файла на глазах у пользователя. В обмен на видимость такому сервису не действуют фоновые лимиты — он живёт, пока вы его не остановите.

Обязательная нотификация

Плата за привилегию — постоянная нотификация, которую нельзя смахнуть. Пользователь всегда видит, что приложение что-то делает в фоне. Сервис обязан показать её через startForeground(), причём с дедлайном: если после startForegroundService() вы не вызовете startForeground() в течение нескольких секунд, система убьёт сервис с ANR-подобной ошибкой.

PlaybackService.kt
class PlaybackService : Service() {
    override fun onStartCommand(intent: Intent?, flags: Int, startId: Int): Int {
        val notification = buildNotification()
        // тип ОБЯЗАТЕЛЕН на Android 14+
        ServiceCompat.startForeground(
            this,
            NOTIF_ID,
            notification,
            ServiceInfo.FOREGROUND_SERVICE_TYPE_MEDIA_PLAYBACK
        )
        return START_STICKY
    }
    override fun onBind(intent: Intent?): IBinder? = null
}

Типы FGS обязательны с Android 14 (API 34)

Раньше FGS был «общего назначения». С Android 14 каждый foreground service обязан объявить тип через android:foregroundServiceType в манифесте, а под тип — запросить соответствующее разрешение. Не объявите тип — startForeground() бросит MissingForegroundServiceTypeException. Это сделано, чтобы система понимала, зачем сервис работает, и могла применять профильные правила.

ТипДля чегоРазрешение

microphoneЗапись звукаFOREGROUND_SERVICE_MICROPHONE + RECORD_AUDIO

В манифесте — оба разрешения. Сервису нужны и базовое FOREGROUND_SERVICE, и профильное FOREGROUND_SERVICE_* под тип. А сам <service> — с атрибутом android:foregroundServiceType="mediaPlayback", совпадающим с типом в startForeground().

Запуск из фона запрещён (с Android 12, API 31)

Главная грабля. С Android 12 приложение, находясь в фоне, не может стартовать foreground service — попытка кинет ForegroundServiceStartNotAllowedException. Есть исключения (приложение в недавнем взаимодействии, прилетел high-priority FCM, пользователь нажал кнопку в нотификации и т.п.), но по умолчанию из чистого фона FGS не поднять. Поэтому типовой сценарий «пришёл push → подними сервис» работает только потому, что high-priority push даёт временное окно-исключение.

Почему нельзя просто стартовать foreground service из фона по таймеру, ведь нотификация всё равно видна пользователю? ★

Потому что видимость постфактум не равна осознанности в момент запуска. До Android 12 разработчики массово эксплуатировали FGS именно как обход фоновых лимитов: поднимали «foreground»-сервис без реальной нужды в видимой длительной задаче, лишь бы обойти Doze и background execution limits, — нотификация при этом мелькала и игнорировалась. Запрет старта из фона бьёт ровно по этому: чтобы поднять FGS, нужен контекст недавнего взаимодействия пользователя или внешнего события (нажатие, high-priority push, точный аларм с разрешением). Идея — привязать привилегию к причине, а не к факту показа иконки. Отсюда же связка push + FGS: high-priority сообщение даёт системное окно-исключение именно потому, что событие пришло снаружи и, предположительно, требует немедленной видимой реакции. Если же реакция не видимая и не срочная — место задаче в WorkManager, а не в FGS.

AlarmManager: будильники и точное время

AlarmManager — единственный инструмент, который умеет «сработать в конкретный момент времени», в том числе разбудив устройство из Doze. Звучит мощно, и именно поэтому система ограничивает его сильнее всего: точный аларм — дорогое пробуждение, и злоупотребление им бьёт по батарее всех приложений.

Inexact: дёшево и батчится

По умолчанию надо брать неточные алармы. Система вольна сдвинуть их момент и схлопнуть несколько алармов разных приложений в одно пробуждение — это и есть экономия. У setInexactRepeating() нельзя задать произвольный период: только константы вроде INTERVAL_FIFTEEN_MINUTES, INTERVAL_HOUR, INTERVAL_DAY.

InexactAlarm.kt
alarmManager.setInexactRepeating(
    AlarmManager.RTC_WAKEUP,
    triggerAtMillis,
    AlarmManager.INTERVAL_DAY,   // только из набора констант
    pendingIntent
)

Exact: дорого и под разрешением

Точные алармы — setExact() и setExactAndAllowWhileIdle() (последний пробивает Doze). С Android 12 для них нужно особое разрешение, и тут две разные двери:

РазрешениеС APIВыдачаДля кого
SCHEDULE_EXACT_ALARM31 (Android 12)Спрашивается у пользователя, может быть отозваноШирокий набор кейсов
USE_EXACT_ALARM33 (Android 13)Выдаётся автоматически, не отзываетсяТолько будильники и календари

На приложениях, нацеленных на API 33+, SCHEDULE_EXACT_ALARM новым установкам по умолчанию не выдаётся — его нужно выпрашивать. USE_EXACT_ALARM выдаётся сам, но Google Play пускает с ним только приложения, где точное время — суть продукта (будильник, календарь). Поэтому перед постановкой точного аларма проверяют доступность и держат запасной путь:

ExactAlarm.kt
if (alarmManager.canScheduleExactAlarms()) {
    alarmManager.setExactAndAllowWhileIdle(  // сработает даже в Doze
        AlarmManager.RTC_WAKEUP, triggerAtMillis, pendingIntent
    )
} else {
    // разрешения нет — деградируем до неточного, не падаем
    alarmManager.setAndAllowWhileIdle(
        AlarmManager.RTC_WAKEUP, triggerAtMillis, pendingIntent
    )
}
⚠️
Exact-аларм — редкость. Календарь, будильник, таймер готовки — да. «Раз в час дёрнуть синхронизацию», «через 5 минут напомнить о корзине» — нет, это inexact или WorkManager. Если на собесе предлагаете exact alarm под обычную задачу, это красный флаг: вы не в курсе ограничений.
setExactAndAllowWhileIdle пробивает Doze — почему тогда им нельзя делать частые точные напоминания? ★

Технически он сработает в Doze, но система навешивает на него собственный троттлинг: гарантируется не больше одного срабатывания ...AllowWhileIdle примерно раз в несколько минут на приложение в глубоком сне (исторически порог был около девяти минут). Поставите десять таких алармов на ближайшую минуту — выполнится один, остальные подождут окон. То есть «пробивает Doze» означает «имеет право на редкое пробуждение», а не «игнорирует энергосбережение». Плюс с Android 12 на это ещё накладывается разрешение SCHEDULE_EXACT_ALARM/USE_EXACT_ALARM, а Google Play фильтрует, кому вообще можно его держать. Поэтому частые точные напоминания через AlarmManager — это и троттлинг, и риск отзыва разрешения, и риск отклонения в сторе. Правильный паттерн для частых событий — не опрос по таймеру, а внешний триггер (push) либо переосмысление задачи как «гарантированной, но не точной» (WorkManager).

FCM как стратегия пробуждения

До сих пор всё было про «приложение само хочет что-то сделать в фоне». Но часто событие приходит снаружи: сообщение в чате, отмена заказа, обновление цены. Опрашивать сервер по таймеру — батарейное самоубийство и борьба с лимитами. Правильный способ — пусть сервер сам разбудит приложение через push. На Android это FCM (Firebase Cloud Messaging).

FCM держит одно постоянное соединение от устройства к серверам Google, по которому приходят сообщения для всех приложений. Это снимает с вашего приложения необходимость держать свой сокет — пробуждение приходит по общему каналу.

MyFcmService.kt
class MyFcmService : FirebaseMessagingService() {
    override fun onMessageReceived(message: RemoteMessage) {
        val payload = message.data
        // окно выполнения здесь КОРОТКОЕ — не делаем долгую работу прямо тут
        val work = OneTimeWorkRequestBuilder<SyncWorker>()
            .setInputData(workDataOf("entity_id" to payload["id"]))
            .build()
        WorkManager.getInstance(this)
            .enqueueUniqueWork("sync_${payload["id"]}", ExistingWorkPolicy.KEEP, work)
    }
    override fun onNewToken(token: String) { sendTokenToServer(token) }
}

High-priority и Doze

Обычные (normal-priority) сообщения в Doze откладываются до maintenance-окна. High-priority сообщения FCM старается доставить немедленно и вправе разбудить спящее устройство, дав приложению короткое окно на обработку. Но это не бесплатно: high-priority выдаётся под реально срочные, видимые пользователю вещи (входящее сообщение, звонок); злоупотребление FCM наказывает понижением приоритета. Именно high-priority push даёт то самое окно-исключение, в котором с Android 12 можно поднять foreground service из фона.

Дедуп и идемпотентность

FCM — best-effort: сообщение может прийти дважды (ретрансляция при потере ack), может прийти с задержкой, а несколько нормальных сообщений система вправе схлопнуть, доставив только последнее. Поэтому обработчик обязан быть идемпотентным: повторный приход того же события не должен дублировать эффект. Отсюда и enqueueUniqueWork с ключом по сущности в примере выше — два одинаковых пуша дадут одну задачу, а не две.

Связка с WorkManager

Канонический паттерн виден в коде: push разбудил → onMessageReceived поставил Work → WorkManager гарантированно доделал. Почему не делать всё прямо в onMessageReceived? Окно выполнения там короткое и не гарантированное; если процесс убьют посреди сетевого запроса — работа потеряется. WorkManager же переживёт смерть процесса и доведёт дело до конца. FCM отвечает за «узнать вовремя», WorkManager — за «довести до конца». Про сетевой слой запроса — глава Сеть.

Почему FCM может «схлопнуть» несколько сообщений в одно и как с этим жить, если важно каждое событие? ★

Схлопывание (collapsing) — следствие того, что устройство может быть offline или в Doze, а FCM не держит безразмерную очередь на каждое приложение. Для сообщений с одинаковым collapse_key при недоступности устройства хранится только последнее — логика «зачем доставлять три устаревших обновления баланса, если есть свежее». Плюс normal-priority в Doze и так копятся до окна. Жить с этим так: не передавайте в payload само содержимое события как единственный источник истины. Push — это сигнал «проверь актуальное состояние», а не «вот данные». Пришёл пуш → дёрнули сервер за актуальным состоянием (через Work) → синхронизировались. Тогда даже если два промежуточных пуша схлопнулись в один, вы всё равно подтянете полное актуальное состояние с бэкенда. Если же критична доставка именно каждого дискретного события (например, лента транзакций), его нельзя хранить только в push — сервер должен отдавать недостающее при следующей синхронизации, а push лишь ускоряет её.

Куда отнести «выгрузить большой файл, начатую пользователем, которая должна доделаться даже если он свернёт приложение»? ★

Это пограничный случай, и правильный ответ — «зависит от того, видна ли задача и сколько длится», а не один инструмент. Если выгрузка короткая и пользователь следит за прогрессом — foreground service типа dataSync с прогрессом в нотификации; пока сервис виден, лимиты не действуют. Если выгрузка может быть долгой, переживать рестарт и не требует, чтобы пользователь смотрел — WorkManager с constraint на сеть, при необходимости setExpedited для срочности. На практике у WorkManager есть встроенная поддержка длительных воркеров через setForeground()/getForegroundInfo() — он сам поднимет foreground service под капотом, дав и гарантию, и видимость. То есть зрелый ответ: «беру WorkManager как каркас гарантии, а если нужна непрерывная видимая работа — перевожу воркер в foreground через ForegroundInfo, не плодя отдельный сервис руками». Это показывает, что вы понимаете, как инструменты сочетаются, а не выбираете один из списка.

Частые вопросы интервьюера

Чем WorkManager лучше старого комбо JobScheduler + AlarmManager + Service? Middle

Это единый API, который сам выбирает нижележащий механизм под версию ОС, переживает перезагрузку и смерть процесса (состояние в БД), даёт constraints, цепочки, повторы с backoff и наблюдаемость через Flow/LiveData. Раньше всё это писалось руками с развилками по API level и легко ломалось на Doze. WorkManager — рекомендованный Google способ для отложенной гарантированной работы.

В чём разница между Result.retry() и Result.failure()? Middle

retry() — временный сбой, перепланировать по BackoffPolicy (экспоненциальной по умолчанию). failure() — неустранимый провал, повторять бессмысленно. Путаница опасна: retry() на постоянной ошибке даёт бесконечный цикл повторов и расход батареи; failure() на сетевом таймауте теряет задачу. Сетевые/IO-сбои → retry, битые данные / ошибка валидации → failure.

Можно ли WorkManager'ом сделать «каждые 5 минут» или «ровно в 9:00»? Middle+

Нет в обоих случаях. Минимальный период PeriodicWorkRequest — 15 минут (лимит JobScheduler), меньше фреймворк молча поднимет. И WorkManager не даёт гарантии момента — только факта выполнения, выбирая удобное системе время внутри окна. «Ровно в 9:00» — это AlarmManager (exact, под разрешением), а не Work.

Почему мой foreground service падает с ForegroundServiceStartNotAllowedException? Middle+

С Android 12 (API 31) нельзя стартовать FGS, когда приложение в фоне, кроме оговорённых исключений (недавнее взаимодействие, high-priority push, точный аларм, действие из нотификации). Типичная причина — пытаетесь поднять сервис из фонового broadcast'а или по таймеру. Решение: либо войти в исключение (например, разбудиться high-priority push'ем), либо вообще не использовать FGS, а отдать задачу WorkManager.

Что обязательно нужно для foreground service на Android 14? Senior

Объявить тип через android:foregroundServiceType в <service>, запросить базовое FOREGROUND_SERVICE и профильное FOREGROUND_SERVICE_* под этот тип, при необходимости — runtime-разрешение (например RECORD_AUDIO для microphone). В рантайме вызвать startForeground() с тем же типом и нотификацией в пределах дедлайна после startForegroundService(). Без объявленного типа — MissingForegroundServiceTypeException.

Когда брать data message, а когда notification message в FCM? Middle+

Notification message при фоновом приложении обрабатывает SDK — он сам кладёт уведомление в трей, а onMessageReceived не зовётся. Если нужна своя логика по приходу пуша (поставить Work, синхронизироваться, кастомное уведомление) — только data message: его onMessageReceived вызывается и в фоне, и при открытом приложении. Notification message — для простого «показать заголовок и текст».

Push может прийти дважды или не прийти. Как проектировать обработчик? Senior

FCM — best-effort: возможны дубли, задержки, схлопывание по collapse_key. Обработчик делают идемпотентным: повторный приход того же события не должен дублировать эффект (отсюда enqueueUniqueWork с ключом по сущности). И push трактуют как сигнал «сходи проверь», а не как источник данных: по приходу дёргают сервер за актуальным состоянием. Тогда схлопнутые промежуточные пуши не теряют данные — финальное состояние всё равно подтягивается.

Как WorkManager выполняет срочную работу и что такое OutOfQuotaPolicy? Senior

setExpedited() (с WorkManager 2.7) просит выполнить как можно скорее. У срочной работы есть системная квота времени; на её исчерпание обязательно указать OutOfQuotaPolicy: RUN_AS_NON_EXPEDITED_WORK_REQUEST — доделать как обычную работу. На старых API expedited бежит как foreground service, поэтому CoroutineWorker должен реализовать getForegroundInfo() с нотификацией.

Задачи

Задача 1. Напишите CoroutineWorker, который синхронизирует данные только по Wi-Fi и при зарядке, корректно различает временные и постоянные ошибки. Поставьте его уникальной задачей так, чтобы повторный вызов не плодил дублей.

Показать решение 
class SyncWorker(
    ctx: Context, params: WorkerParameters
) : CoroutineWorker(ctx, params) {
    override suspend fun doWork(): Result = try {
        repository.sync()
        Result.success()
    } catch (e: IOException) {
        Result.retry()                 // сеть отвалилась — повторим
    } catch (e: IllegalStateException) {
        Result.failure()               // неустранимо — не зацикливаемся
    }
}

fun scheduleSync(context: Context) {
    val constraints = Constraints.Builder()
        .setRequiredNetworkType(NetworkType.UNMETERED)
        .setRequiresCharging(true)
        .build()
    val request = OneTimeWorkRequestBuilder<SyncWorker>()
        .setConstraints(constraints)
        .setBackoffCriteria(BackoffPolicy.EXPONENTIAL, 30, TimeUnit.SECONDS)
        .build()
    WorkManager.getInstance(context)
        .enqueueUniqueWork("sync", ExistingWorkPolicy.KEEP, request)
}

Ключевые моменты: UNMETERED = безлимитная сеть (Wi-Fi), setRequiresCharging отложит до зарядки. IOException → retry, неустранимое → failure, иначе словим бесконечный backoff. ExistingWorkPolicy.KEEP гарантирует одну активную задачу — повторные вызовы не создадут дублей.

Задача 2. Для каждого кейса выберите инструмент и обоснуйте: (а) показывать прогресс скачивания фильма; (б) раз в сутки отправлять накопленную аналитику; (в) разбудить чат при новом сообщении; (г) напомнить о встрече ровно в её время.

Показать решение 
// (а) Foreground service (dataSync) — длительная видимая задача,
//     прогресс в нотификации, лимиты фона не действуют пока виден.
// (б) WorkManager (Periodic, ~раз в сутки) — гарантия без точности,
//     constraint на сеть, батчится, бережёт батарею.
// (в) FCM (data, high-priority) + WorkManager на доделку —
//     внешнее событие, мгновенность, идемпотентность через unique work.
// (г) AlarmManager (setExactAndAllowWhileIdle) — точное время суть кейса,
//     под разрешением SCHEDULE_EXACT_ALARM/USE_EXACT_ALARM.

Логика выбора: внешнее событие → FCM; видимая длительная работа → FGS; точное время → exact alarm; всё остальное гарантированное, но не точное → WorkManager. Ошибка-ловушка — тянуть exact alarm на (б) или опрос сервера по таймеру вместо push на (в).

Задача 3. Корректно запустите foreground service типа mediaPlayback на Android 14: что в манифесте, что в коде, чего нельзя забыть.

Показать решение 
// AndroidManifest.xml
// <uses-permission android:name="android.permission.FOREGROUND_SERVICE"/>
// <uses-permission android:name="android.permission.FOREGROUND_SERVICE_MEDIA_PLAYBACK"/>
// <service
//     android:name=".PlaybackService"
//     android:foregroundServiceType="mediaPlayback"
//     android:exported="false"/>

class PlaybackService : Service() {
    override fun onStartCommand(i: Intent?, f: Int, id: Int): Int {
        ServiceCompat.startForeground(
            this, NOTIF_ID, buildNotification(),
            ServiceInfo.FOREGROUND_SERVICE_TYPE_MEDIA_PLAYBACK
        )
        return START_STICKY
    }
    override fun onBind(i: Intent?): IBinder? = null
}

Нельзя забыть: тип в манифесте и в startForeground() должны совпадать, иначе MissingForegroundServiceTypeException. Оба разрешения — базовое и профильное. startForeground() — в пределах дедлайна после старта. И главное: с Android 12 поднять сервис из чистого фона нельзя — старт должен идти из контекста взаимодействия (нажал play) или окна-исключения.
