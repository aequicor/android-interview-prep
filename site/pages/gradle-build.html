Gradle и сборка

Как Gradle превращает исходники в APK или AAB: три фазы сборки и граф задач, build variants от AGP (Android Gradle Plugin), способы ускорить сборку, KSP против kapt, модуляризация, R8 и форматы доставки в Google Play. Тема, по которой видно, понимаете ли вы конвейер сборки целиком, а не только нажимаете «Run» в Android Studio.

Модель Gradle: фазы и граф задач

Каждая сборка Gradle проходит три фазы строго по порядку: initialization, configuration, execution. Понимание этого порядка — ключ к тому, почему одни правки в build.gradle ускоряют сборку, а другие незаметно тормозят вообще всё.

В initialization Gradle читает settings.gradle(.kts), чтобы узнать состав сборки: какие модули в неё входят (через include(":app", ":core")) и какие included builds подключены. На этой фазе создаётся объект Settings и по объекту Project на каждый модуль — но скрипты модулей ещё не исполнялись.

В configuration Gradle исполняет тело каждого build.gradle всех участвующих модулей, регистрирует задачи и строит из них граф — по входам и выходам каждой задачи. Здесь критичный момент: код на верхнем уровне build-скрипта (вне блоков tasks.register) исполняется на каждой сборке, даже если вы собираете одну-единственную задачу. Поэтому тяжёлая работа в конфигурации — чтение файлов, сетевые запросы, перебор зависимостей — замедляет любую команду Gradle, включая ./gradlew help.

В execution Gradle берёт построенный граф и выполняет только нужные задачи в правильном порядке, пропуская те, что уже актуальны.

состав модулей"] --> I["Initialization
создать Project-объекты"] I --> C["Configuration
выполнить build.gradle,

запустить нужные задачи"] C -. "тяжёлый код тут

Три фазы Gradle. Конфигурация исполняется на каждой сборке — поэтому её вес умножается на частоту запусков

Task: dependsOn, inputs/outputs, up-to-date

Задача (Task) — единица работы: компиляция, копирование, упаковка. Зависимости между задачами задаются явно через dependsOn либо неявно — через совпадение выхода одной задачи со входом другой. Граф эти связи собирает на фазе конфигурации.

Главный механизм скорости — up-to-date checks (инкрементальность). У задачи объявлены входы (@InputFiles, @Input) и выходы (@OutputFiles). Перед запуском Gradle хеширует входы и сверяет с прошлой сборкой. Если ни входы, ни выходы не менялись — задача помечается UP-TO-DATE и пропускается. Так чистая повторная сборка делает почти ничего.

build.gradle.kts
tasks.register("generateVersion") {
    val output = layout.buildDirectory.file("version.txt")
    // объявляем выход — без этого Gradle всегда считает задачу устаревшей
    outputs.file(output)
    inputs.property("version", project.version.toString())
    doLast {
        output.get().asFile.writeText("v=${project.version}")
    }
}
⚠️
Задача без объявленных inputs/outputs не кешируется. Gradle не знает, от чего она зависит, и гоняет её каждый раз. Кастомные задачи, которые «почему-то всегда выполняются», — почти всегда забытые inputs/outputs.
Почему добавление «безобидного» println с чтением файла в начало build.gradle может замедлить даже ./gradlew tasks? ★

Тело build-скрипта на верхнем уровне исполняется на фазе конфигурации, а она проходит при любой команде Gradle — даже когда вы не собираете ничего. Чтение файла на каждой конфигурации добавляет I/O ко всем сборкам сразу. Хуже того, без configuration cache конфигурация повторяется и тогда, когда сама задача давно UP-TO-DATE. Правильное место для такой логики — внутри doLast {} задачи (фаза execution) с объявленными входами, чтобы работа выполнялась лениво и кешировалась.

AGP: build types, flavors, variants

AGP добавляет поверх Gradle понятие build variant — конкретной сборки приложения. Вариант рождается из пересечения двух осей: build type (как собрано) и product flavor (что за версия).

Build type описывает технику сборки. Из коробки два: debug (отлаживаемый, без минификации, с дебажной подписью) и release (для публикации, с R8 и боевой подписью). Product flavor — это вариация продукта: free против paid, dev против prod, разные бренды. Флейворы группируются по измерениям (flavorDimensions), и AGP перемножает по одному флейвору из каждого измерения на каждый build type.

flowchart TB subgraph Flavors["productFlavors (dim: tier)"] F1["free"] F2["paid"] end subgraph Types["buildTypes"] T1["debug"] T2["release"] end F1 --> V1["freeDebug"] F1 --> V2["freeRelease"] F2 --> V3["paidDebug"] F2 --> V4["paidRelease"] T1 -.-> V1 T1 -.-> V3 T2 -.-> V2 T2 -.-> V4
2 флейвора × 2 build type = 4 варианта. Добавили измерение из 2 флейворов — вариантов уже 8

Под каждый вариант AGP подмешивает свои sourceSets: код и ресурсы из src/freeDebug/, src/free/, src/debug/ накладываются поверх src/main/. Так одна и та же фича получает разную реализацию без if в коде.

app/build.gradle.kts
android {
    buildTypes {
        getByName("release") {
            isMinifyEnabled = true
            signingConfig = signingConfigs.getByName("release")
            manifestPlaceholders["appLabel"] = "MyApp"
        }
        getByName("debug") {
            applicationIdSuffix = ".debug"   // ставится рядом с release
            manifestPlaceholders["appLabel"] = "MyApp DEBUG"
        }
    }
    flavorDimensions += "tier"
    productFlavors {
        create("free") {
            dimension = "tier"
            buildConfigField("boolean", "PREMIUM", "false")
        }
        create("paid") {
            dimension = "tier"
            applicationIdSuffix = ".paid"
            buildConfigField("boolean", "PREMIUM", "true")
        }
    }
}

buildConfigField прокидывает значение в сгенерированный класс BuildConfig — читаете его как BuildConfig.PREMIUM в рантайме. manifestPlaceholders подставляют значения в AndroidManifest.xml на сборке (имя приложения, ключи API под разные среды). signingConfigs хранят данные подписи; боевой keystore держат вне репозитория и подтягивают из переменных окружения или local.properties.

🛑
Никогда не коммитьте keystore и пароли в репозиторий. Подпись release — это и есть удостоверение приложения в Play. Утёкший ключ означает, что кто-то может выпустить «обновление» под вашим именем. Храните keystore вне git и читайте пароли из окружения CI.
Почему количество вариантов растёт мультипликативно, и чем это бьёт по сборке и тестам? ★

Каждое новое измерение флейворов умножает число вариантов на свой размер: при двух измерениях по 3 флейвора и двух build type получается 3 × 3 × 2 = 18 вариантов. AGP конфигурирует задачи на все варианты, даже если собираете один, — это раздувает фазу конфигурации и граф задач, замедляя любую команду. CI, прогоняющий тесты по матрице вариантов, растёт так же. Поэтому лишние измерения убирают, а для редких комбинаций используют variantFilter (или androidComponents { beforeVariants } в новых AGP), чтобы исключить ненужные варианты из конфигурации.

Ускорение сборки

Медленная сборка — налог на каждую итерацию разработчика. Хорошая новость: большую часть выигрыша дают несколько переключателей в gradle.properties и дисциплина в объявлении зависимостей.

База — Gradle daemon: фоновый JVM-процесс, который живёт между сборками и держит прогретый кеш и JIT. Он включён по умолчанию; выключать его (--no-daemon) есть смысл только на одноразовых CI-агентах. Parallel (org.gradle.parallel=true) собирает независимые модули одновременно — выигрыш тем больше, чем лучше разбит проект.

Configuration cache против build cache

Их постоянно путают, а спрашивают почти всегда. Это два разных кеша на разных фазах:

КритерийConfiguration cacheBuild cache
Какую фазу ускоряетConfiguration — пропускает повторное построение графа задачExecution — переиспользует выходы самих задач
Что кешируетСтруктуру графа и сконфигурированные задачиАртефакты: классы, dex, распакованные AAR
Где живётЛокально в проектеЛокально и/или удалённо (общий на команду/CI)
Когда заметнееБольшой проект с тяжёлой конфигурациейЧистые сборки и переключение веток
Флагorg.gradle.configuration-cache=trueorg.gradle.caching=true

Они не взаимоисключающие — включают оба. Configuration cache экономит на «сколько Gradle думает, прежде чем начать работу», build cache — на «сколько работы можно не делать заново». Удалённый build cache особенно окупается на CI, где гоняются сборки и тесты: один агент собрал — остальные переиспользуют артефакты.

Version catalogs и фиксированные версии

Version catalogs (gradle/libs.versions.toml) собирают координаты зависимостей и их версии в один типобезопасный файл. Это не про скорость напрямую, а про порядок: одна версия Kotlin на весь проект, автодополнение в IDE, нет рассинхрона между модулями.

gradle/libs.versions.toml
[versions]
kotlin = "2.0.21"
coroutines = "1.9.0"

[libraries]
coroutines-core = { module = "org.jetbrains.kotlinx:kotlinx-coroutines-core", version.ref = "coroutines" }

[plugins]
kotlin-android = { id = "org.jetbrains.kotlin.android", version.ref = "kotlin" }

И обратное правило: избегайте dynamic versions вроде 1.+ или latest.release. Они заставляют Gradle лезть в сеть за списком версий (медленно и недетерминированно), а сборка перестаёт быть воспроизводимой — вчера собралось одно, сегодня прилетела новая минорная версия и что-то сломалось.

💡
Модульность — это и про инкрементальность. Изменили один модуль — Gradle пересобирает его и зависящих от него, а не весь проект. В одном гигантском модуле любая правка задевает всё. Разбиение на модули — самый структурный способ ускорить инкрементальную сборку (детали — в параграфе про модуляризацию).
Почему включённый configuration cache может «упасть» там, где раньше всё работало, и при чём тут чтение System.getenv() в build-скрипте? ★

Configuration cache требует, чтобы конфигурация была детерминированной и сериализуемой: задачи не должны на этапе выполнения обращаться к «живым» объектам Project, Gradle, к переменным окружения или системным свойствам напрямую из doLast. Если задача читает System.getenv() во время выполнения, Gradle не может зафиксировать вход в кеш и помечает это нарушением. Лечится переносом чтения окружения в фазу конфигурации и передачей значения в задачу через Property/provider (например, providers.environmentVariable("CI")), которые корректно учитываются кешем. Поэтому миграция на configuration cache часто вскрывает скрытые нарушения изоляции фаз, которые годами «прощались».

KSP vs kapt

Аннотационные процессоры (Dagger/Hilt, Room, Moshi) генерируют код на сборке. Долгое время в Kotlin это делал kapt, и он был узким местом. KSP пришёл его заменить — и почти всегда быстрее.

Почему kapt медленный: процессоры написаны под Java и работают с Java-моделью кода. Чтобы их прогнать, kapt сначала компилирует Kotlin в Java-стабы (заглушки с сигнатурами, без тел методов), а для этого должен разрешить все символы программы. По данным команды Kotlin, генерация стабов стоит примерно треть полного анализа kotlinc. Затем эти стабы скармливаются javac с процессорами — отдельный проход компилятора.

KSP (Kotlin Symbol Processing) читает символы Kotlin напрямую через собственный API, без стабов и без javac. Он понимает Kotlin-специфику нативно и разрешает ссылки лениво, по запросу. На примере из документации Kotlin: при общем времени компиляции Kotlin около 21,5 с kapt тратил на генерацию ~8,7 с, а KSP — ~1,15 с.

flowchart LR K1["Kotlin-исходники"] --> S["kapt: генерация
Java-стабов"] S --> J["javac + процессоры"] J --> G1["сгенерированный код"] K2["Kotlin-исходники"] --> P["KSP: чтение
символов напрямую"] P --> G2["сгенерированный код"]
kapt делает лишний проход через стабы и javac; KSP работает по символам Kotlin без посредников

Миграция обычно сводится к замене плагина и конфигурации зависимости — при условии, что процессор поддерживает KSP. Room, Moshi, Hilt, Glide уже умеют.

build.gradle.kts
// было: kapt
// kapt("androidx.room:room-compiler:2.6.1")

// стало: ksp
plugins {
    id("com.google.devtools.ksp") version "2.0.21-1.0.25"
}
dependencies {
    implementation("androidx.room:room-runtime:2.6.1")
    ksp("androidx.room:room-compiler:2.6.1")   // тот же артефакт, другая конфигурация
}
⚠️
На kapt стоит оставаться, только если процессор не поддерживает KSP. Версия KSP привязана к версии Kotlin (суффикс -1.0.25 после версии Kotlin), поэтому при апгрейде Kotlin не забудьте поднять и KSP. Смешивать kapt и KSP в одном модуле можно, но тогда вы платите за оба конвейера.
Почему у KSP инкрементальная пересборка обычно затрагивает меньше файлов, чем у kapt, при той же правке? ★

kapt для генерации стабов разрешает каждую ссылку на тип во всём модуле — поэтому правка в одном файле легко «загрязняет» множество других, и инкрементальный механизм перестраивает широкий набор. KSP разрешает символы динамически, по мере обращения, и отслеживает реальные зависимости между символами. Его инкрементальная модель тоньше, чем простое деление на isolating/aggregating-процессоры: изменение в файле с меньшей вероятностью тянет за собой пересборку соседних, поэтому набор переобрабатываемых файлов меньше. Поэтому KSP быстрее на чистой сборке и обычно бережнее к соседним файлам при инкрементальной.

Модуляризация и влияние на сборку

Разбиение монолита на модули работает сразу в двух плоскостях: архитектурной и сборочной. Типовая раскладка по слоям: :app (точка сборки, DI-граф), :feature:* (экраны и их логика), :core:* (общие UI/утилиты), :data (репозитории, сеть, БД).

api против implementation

Это первое, что спрашивают про модули. Зависимость в api протекает в compile classpath потребителей: кто зависит от вашего модуля, видит и его api-зависимости транзитивно. implementation остаётся внутренней — она в classpath потребителя не попадает.

Скорость отсюда следует напрямую. Меньше classpath у потребителя — быстрее его компиляция. А главное: если вы меняете implementation-зависимость способом, не трогающим ваш публичный API (ABI), потребителей не нужно перекомпилировать — Gradle применяет compile-avoidance. С api такого щита нет: любое изменение протекает дальше по графу.

Критерийapiimplementation
Видна потребителю транзитивноДаНет
Размер compile classpath потребителяБольшеМеньше
Перекомпиляция потребителя при ABI-совместимой правкеПроисходитИзбегается

💡
Дефолт — implementation. api ставьте только когда тип зависимости реально торчит в публичных сигнатурах вашего модуля (параметр публичного метода, возвращаемый тип). Привычка лепить api «на всякий случай» превращает модульный граф в один большой classpath и убивает выигрыш от разбиения.

Параллельность и convention plugins

Независимые модули Gradle собирает параллельно (с org.gradle.parallel=true) — чем уже и глубже граф зависимостей, тем больше задач идут одновременно. Длинная цепочка :app → :feature → :core → :data сериализует сборку; широкий граф с несколькими независимыми ветками распараллеливается лучше.

Чтобы 20 модулей не повторяли один и тот же блок настроек, общую конфигурацию выносят в convention plugins. Раньше для этого использовали buildSrc, но любая правка в нём инвалидирует кеш всех сборок. Современный подход — отдельный included build build-logic с плагинами-соглашениями: модули применяют их одной строкой, а изменение одного плагина не сбрасывает кеш всего проекта целиком.

feature/profile/build.gradle.kts
plugins {
    // один плагин-соглашение вместо десятка повторяющихся настроек
    id("myapp.android.feature")
}
dependencies {
    implementation(project(":core:ui"))
    implementation(project(":data"))   // внутренняя зависимость — не протекает выше
}
Почему миграция с buildSrc на included build build-logic ускоряет сборки, хотя код плагинов тот же? ★

buildSrc Gradle трактует как неявную зависимость всех проектов сборки: любое изменение его исходников инвалидирует кеш конфигурации и заставляет переконфигурировать весь билд, даже если правка касалась одного плагина. Included build build-logic подключается явно через includeBuild и ведёт себя как обычная композитная сборка со своим кешем — изменение одного convention-плагина инвалидирует только то, что от него зависит, а не весь проект. Плюс included builds дают более чёткие границы и лучше дружат с configuration cache. Код плагинов при этом действительно тот же — выигрыш чисто в гранулярности инвалидации кеша.

R8: shrinking, obfuscation, optimization

В release-сборке за уменьшение и защиту кода отвечает R8 — он включается, когда вы ставите isMinifyEnabled = true. R8 делает три вещи за один проход.

Shrinking (tree shaking) — находит и выкидывает недостижимый код: неиспользуемые классы, поля, методы из вашего кода и зависимостей. Obfuscation — переименовывает классы и методы в короткие имена (com.example.MyActivity → a.b.a), уменьшая размер dex и заодно усложняя реверс. Optimization — переписывает код: инлайнит методы, убирает мёртвые ветки, упрощает выражения.

Точка входа в анализ достижимости — keep-правила. AGP и AAPT2 генерируют базовый набор автоматически (Activity, View, Service из манифеста). Остальное, что вызывается не из Kotlin-кода напрямую, R8 «не видит» и может удалить или переименовать.

🛑
Рефлексия, Gson и сериализация ломаются без keep-правил. Gson создаёт объект по имени поля через рефлексию. R8 переименует поля — и десериализация даст null или упадёт. То же с любой библиотекой, которая ищет классы/поля по строковым именам в рантайме. Симптом классический: в debug всё работает, в release-сборке падает.

Минимальное keep-правило для класса, который сериализует Gson — сохранить имена его членов:

proguard-rules.pro
# класс данных, сериализуемый Gson по именам полей
-keep class com.example.model.User { *; }

# либо аккуратнее: оставить имена членов всех @SerializedName-полей
-keepclassmembers class com.example.model.** {
    @com.google.gson.annotations.SerializedName <fields>;
}

mapping.txt и деобфускация

После обфускации стектрейсы из боевого приложения нечитаемы: вместо UserRepository.load() там a.b.c(). R8 пишет соответствие старых и новых имён в mapping.txt (обычно app/build/outputs/mapping/release/). Этот файл нужно сохранять на каждый релиз — без него краш-репорты с Play Console или Crashlytics не восстановить. Деобфускация делается утилитой retrace по соответствующему билду.

Рядом с minifyEnabled часто включают shrinkResources = true — выкидывает неиспользуемые ресурсы (картинки, layout). Работает только вместе с включённым R8: ресурсы считаются неиспользуемыми по ссылкам из кода, а код к тому моменту уже урезан. Эта же тема смыкается с профилированием: R8-оптимизации и удаление мёртвого кода влияют на размер и время старта приложения.

Почему -keep class Foo { *; } «чинит» краш, но это плохое решение, и чем оно отличается от -keepclassmembers? ★

-keep class Foo { *; } делает класс точкой входа: R8 не удалит его, не переименует и не будет оптимизировать ни класс, ни его члены — он выпадает из tree shaking и из оптимизаций целиком. Это глушит симптом, но раздувает dex и лишает класс инлайна и прочих улучшений. -keepclassmembers сохраняет имена только указанных членов если сам класс достижим — то есть не мешает R8 удалить класс, когда тот реально не нужен, и не выключает оптимизацию остального. Грамотные keep-правила точечны: сохраняют ровно то, к чему обращаются по имени в рантайме (поля для Gson, методы для JNI/рефлексии), а не «весь класс на всякий случай». Перебор с широкими -keep — частая причина того, почему R8 «почти ничего не уменьшил».

App Bundle, split APK, app size

Сегодня в Google Play приложения публикуют как AAB (Android App Bundle), а не APK. AAB — это формат публикации: его нельзя установить на устройство напрямую. Play принимает один bundle и сам генерирует из него оптимизированные APK под каждое устройство.

Механизм называется split APK. Из bundle Play собирает базовый APK плюс configuration APK под конкретные характеристики: плотность экрана (density), архитектуру процессора (ABI) и язык. Пользователь скачивает только то, что подходит его устройству — не все ресурсы всех плотностей и не библиотеки под все ABI сразу. Отсюда меньший размер загрузки «бесплатно», без вашего кода.

Что качает устройствоВсе плотности, все ABI, все языкиТолько свою density/ABI/язык

Play Feature Delivery

Поверх split APK работает Play Feature Delivery — доставка фич отдельными модулями. Режимы: install-time (модуль ставится вместе с приложением, просто отделён технически), on-demand (скачивается позже по запросу из кода — например, редко используемая фича не входит в первую загрузку) и conditional (по условиям устройства/страны).

build.gradle.kts
android {
    bundle {
        // включить/выключить дробление по осям (по умолчанию включено)
        language { enableSplit = true }
        density  { enableSplit = true }
        abi      { enableSplit = true }
    }
}

Измерять размер удобнее всего через APK Analyzer в Android Studio — он показывает вклад dex, ресурсов и нативных библиотек, а ещё умеет сравнивать два билда. Чтобы оценить реальные размеры загрузки под разные устройства из bundle, используют bundletool. Главные рычаги уменьшения: включённый R8 с shrinkResources, AAB вместо universal APK, вынос редких фич в on-demand модули и переход на векторные/WebP-ресурсы.

💡
«Размер APK на CI» обманчив. Локальный universal APK содержит всё под все устройства — он всегда крупнее, чем то, что реально скачает пользователь из Play. Сравнивайте размеры по APK Analyzer для конкретной конфигурации или через bundletool, а не по весу одного debug-APK.
Почему вынос фичи в on-demand модуль не всегда уменьшает то, что скачивает обычный пользователь, и где здесь подвох? ★

On-demand модуль убирает свой код и ресурсы из первой загрузки — но выигрыш есть только если фичей пользуется меньшинство. Если большинство пользователей всё равно открывает эту фичу, вы просто перенесли тот же объём на отложенную докачку (часто с задержкой и риском сбоя сети), а размер базового APK уменьшили номинально. К тому же on-demand добавляет сложность: обработку отсутствия модуля, загрузку через Play Core/SplitInstall API, тестирование сценариев без модуля. Поэтому on-demand оправдан для тяжёлых и редких фич (большая игра-мини, AR, экспорт-движок), а не как способ механически «уменьшить APK» для всего подряд — для повседневного веса гораздо надёжнее работают split по density/ABI/языку, которые Play делает сам и бесплатно.

Частые вопросы интервьюера

Чем отличаются три фазы сборки Gradle? Middle

Initialization читает settings.gradle и определяет состав модулей, создавая Project-объекты. Configuration исполняет тело каждого build.gradle и строит граф задач по входам и выходам. Execution выполняет нужные задачи из графа. Важно: конфигурация проходит на каждой команде Gradle, поэтому тяжёлый код в build-скриптах замедляет вообще все сборки.

В чём разница между configuration cache и build cache? Middle+

Это два кеша на разных фазах. Configuration cache кеширует граф задач и сконфигурированные задачи, ускоряя фазу конфигурации. Build cache кеширует выходы задач (классы, dex, распакованные AAR), ускоряя выполнение и особенно чистые сборки. Они независимы, включают оба: org.gradle.configuration-cache=true и org.gradle.caching=true.

Почему KSP быстрее kapt? Middle+

kapt генерирует Java-стабы, чтобы прогнать через них Java-процессоры с javac — для стабов нужно разрешить все символы программы, это отдельный дорогой проход. KSP читает символы Kotlin напрямую через свой API, без стабов и без javac, и разрешает ссылки лениво. По замерам Kotlin генерация кода у KSP кратно быстрее, плюс инкрементальная пересборка затрагивает меньше файлов.

Когда ставить api, а когда implementation? Middle+

По умолчанию — implementation: зависимость не протекает в compile classpath потребителей, classpath меньше, и при ABI-совместимой правке потребителей не надо перекомпилировать. api — только когда тип зависимости торчит в публичной сигнатуре вашего модуля (параметр или возвращаемый тип публичного метода), потому что тогда потребитель обязан его видеть.

Что именно делает R8 в release и почему может сломаться Gson? Senior

R8 за один проход выполняет shrinking (удаляет недостижимый код), obfuscation (переименовывает классы/члены в короткие имена) и optimization (инлайн, упрощения). Gson создаёт объекты по именам полей через рефлексию — R8 эти имена переименует, и десериализация сломается. Лечится keep-правилами на члены сериализуемых классов. Симптом: в debug работает, в release падает.

Зачем хранить mapping.txt? Middle+

R8 обфусцирует имена, и стектрейсы из боевых крашей нечитаемы (a.b.c() вместо реальных имён). mapping.txt хранит соответствие исходных и обфусцированных имён для конкретного билда. Без сохранённого на релиз файла краш-репорты из Play Console/Crashlytics не деобфусцировать утилитой retrace.

Чем AAB отличается от APK и зачем он нужен? Middle

APK — устанавливаемый формат, AAB — формат публикации, его нельзя поставить на устройство напрямую. Вы загружаете один bundle, а Play генерирует из него split APK под каждое устройство: пользователь качает только свою плотность экрана, ABI и язык. Итог — меньше размер загрузки без изменений в коде. С августа 2021 новые приложения в Play публикуются только как AAB.

Почему кастомная задача «всегда выполняется» и не кешируется? Senior

Скорее всего у неё не объявлены inputs/outputs. Up-to-date check сравнивает хеши входов и наличие выходов с прошлой сборкой; без объявленных входов и выходов Gradle не знает, от чего задача зависит, и считает её устаревшей каждый раз. Объявите inputs.files/property и outputs.files — тогда Gradle сможет помечать её UP-TO-DATE и кешировать.

Задачи

Задача 1. Класс data class User(val id: Long, val name: String) сериализуется в JSON и обратно через Gson. После включения R8 в release десериализация возвращает объект с пустыми полями. Напишите keep-правило и объясните, почему так происходит.

Показать решение 
# proguard-rules.pro
-keepclassmembers class com.example.model.User {
    <fields>;
}
# или точечно по @SerializedName, если поля размечены:
-keepclassmembers,allowobfuscation class com.example.model.** {
    @com.google.gson.annotations.SerializedName <fields>;
}

Gson сопоставляет ключи JSON с полями объекта по их именам через рефлексию. R8 в release переименовывает поля (name → a), и Gson больше не находит соответствия — поля остаются со значениями по умолчанию. -keepclassmembers ... <fields>; сохраняет исходные имена полей, не превращая весь класс в точку входа (в отличие от -keep class ... { *; }, который ещё и выключил бы оптимизацию). Альтернатива без правил — разметить поля @SerializedName("name"), тогда Gson использует строку из аннотации, а не имя поля.

Задача 2. Модуль использует Room через kapt и собирается медленно. Перепишите фрагмент build.gradle.kts на KSP и объясните, что меняется.

Показать решение 
// было:
// plugins { kotlin("kapt") }
// dependencies { kapt("androidx.room:room-compiler:2.6.1") }

// стало:
plugins {
    id("com.google.devtools.ksp") version "2.0.21-1.0.25"
}
dependencies {
    implementation("androidx.room:room-runtime:2.6.1")
    ksp("androidx.room:room-compiler:2.6.1")
}

Меняем плагин kapt на ksp и переносим зависимость-процессор из конфигурации kapt(...) в ksp(...) — артефакт компилятора тот же. После этого Room-процессор работает по символам Kotlin напрямую, без генерации Java-стабов и прохода javac, что ускоряет и чистую, и инкрементальную сборку. Версию KSP берут под версию Kotlin (суффикс после -): при апгрейде Kotlin её тоже поднимают.

Задача 3. Команда жалуется, что даже ./gradlew help идёт несколько секунд, а инкрементальные сборки медленные. В корневом build.gradle.kts на верхнем уровне есть чтение файла конфигурации и запрос версии зависимости как "com.lib:x:1.+", проект — один большой модуль. Объясните причины и что чинить.

Показать решение 
// Проблема 1: код на верхнем уровне build-скрипта
// исполняется на КАЖДОЙ команде (фаза configuration),
// поэтому чтение файла тормозит даже ./gradlew help.
// Решение: убрать из верхнего уровня; читать лениво
// внутри задачи или через providers.fileContents(...).

// Проблема 2: dynamic version "1.+" заставляет Gradle
// ходить в сеть за списком версий и ломает воспроизводимость.
// Решение: зафиксировать версию (через version catalog).

// Проблема 3: один большой модуль => любая правка
// пересобирает всё. Решение: разбить на модули
// (:app/:feature/:core/:data) + org.gradle.parallel=true
// + org.gradle.caching=true + configuration-cache.

Три независимых причины. Тяжёлая конфигурация (чтение файла на верхнем уровне) умножается на частоту любых команд Gradle. Dynamic version 1.+ добавляет сетевой резолв и делает сборку недетерминированной. Монолитный модуль убивает инкрементальность — менять надо всё сразу. Чинится переносом логики в задачи, фиксацией версий через version catalog и модуляризацией с включёнными кешами и параллельной сборкой.
