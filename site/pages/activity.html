Activity: lifecycle, состояние и back stack

Activity — точка входа в пользовательский сценарий: она держит окно, принимает Intent, участвует в task/back stack и переживает пересоздание при смене конфигурации. На собесе по ней быстро уходят от «назови callbacks» к состоянию, process death, isFinishing и ошибкам навигации.

Middle Senior Частый вопрос

Lifecycle: что означает каждый callback

Жизненный цикл Activity читается парами: onCreate/onDestroy — всё время жизни экземпляра, onStart/onStop — видимость, onResume/onPause — интерактивность и фокус. На интервью важно не просто перечислить методы, а объяснить, куда привязывать ресурсы: камеру и сенсоры обычно поднимают рядом с видимостью, а короткие UI-подписки — рядом с интерактивным состоянием.

Lifecycle Activity: onPause ещё не означает невидимость, а onStop уже означает, что пользователь экран не видит

Почему нельзя сохранять тяжёлое состояние в onPause(), если пользователь просто открыл permission dialog? ★

onPause() вызывается не только при уходе со страницы: системный диалог, split-screen, прозрачная Activity поверх текущей — и ваш экран уже не RESUMED, хотя всё ещё виден. Тяжёлая синхронная запись в onPause() блокирует переход и может дать jank. Долгую работу выносят из callback в репозиторий/WorkManager, а в onPause() оставляют только короткую фиксацию критичного состояния.

isFinishing, isChangingConfigurations и настоящее закрытие

onDestroy() сам по себе не отвечает на главный вопрос: Activity закрывается навсегда или пересоздаётся из-за конфигурации? Для этого нужны флаги. isFinishing становится true, когда Activity завершили через finish(), Back или системное завершение task. isChangingConfigurations показывает, что текущий экземпляр умирает из-за configuration change: поворот, смена языка, темы, размера окна.

CheckoutActivity.kt
class CheckoutActivity : AppCompatActivity() {

    override fun onStop() {
        super.onStop()
        // Видимый ресурс отпускаем всегда: пользователь экран уже не видит.
        cameraPreview.stop()
    }

    override fun onDestroy() {
        super.onDestroy()

        if (isFinishing) {
            // Пользователь реально вышел из флоу: черновик больше не нужен.
            viewModel.discardTemporaryDraft()
        }

        if (!isChangingConfigurations) {
            // Не поворот и не смена темы: текущий экземпляр не будет заменён новым.
            analyticsSession.close()
        }
    }
}
⚠️
Флаги — не замена сохранению состояния. При убийстве процесса Android не обязан вызвать onDestroy(). Всё, что должно пережить process death, кладём в SavedStateHandle/onSaveInstanceState или в постоянное хранилище. isFinishing нужен для развилки «пользователь ушёл навсегда или будет новый экземпляр», а не для гарантий финализации.
Почему проверка isFinishing в onPause() иногда даёт другой ответ, чем ожидали, и почему это не lifecycle-оракул? ★

onPause() вызывается раньше, чем Activity перестаёт быть видимой, и может приходить из-за временного перекрытия экраном поверх. В этот момент isFinishing отвечает только на вопрос «уже запрошено завершение этой Activity?», а не «экран точно исчезает навсегда». Для ресурсов видимости смотрите на onStart/onStop, для окончательного закрытия — на комбинацию isFinishing/isChangingConfigurations в callback уничтожения. И держите в голове process death: callback может вообще не случиться.

Configuration change, process death и сохранение состояния

Поворот экрана и смерть процесса часто смешивают, но это разные аварии. При configuration change процесс жив, Activity пересоздаётся, а ViewModel остаётся в ViewModelStore. При process death умирает весь процесс: ViewModel исчезает, и восстановить UI можно только из сериализованного состояния или постоянного хранилища.

SavedStateHandleпереживаетпереживаетМаленькие ключи: id, фильтр, tab, query
onSaveInstanceStateпереживаетпереживаетМинимальное UI-состояние, которое можно сериализовать в Bundle
Room / DataStoreпереживаетпереживаетДолгоживущие данные и большие объёмы
SearchViewModel.kt
class SearchViewModel(
    private val savedStateHandle: SavedStateHandle
) : ViewModel() {
    val query: StateFlow<String> =
        savedStateHandle.getStateFlow(KEY_QUERY, "")

    fun onQueryChanged(value: String) {
        savedStateHandle[KEY_QUERY] = value
    }

    companion object {
        private const val KEY_QUERY = "query"
    }
}
🛑
Bundle не безразмерный. Сохранённое состояние маршалится через Binder. Буфер транзакций ограничен, поэтому большой список, bitmap или тяжёлый Parcelable может закончиться TransactionTooLargeException. Правило простое: в saved state — ключи, в Room/сеть — сами данные.
Почему ViewModel переживает поворот, но не process death, хотя оба сценария приводят к новому onCreate()? ★

При повороте старый процесс жив. Framework переносит ViewModelStore через non-configuration state от старой Activity к новой, поэтому новый экземпляр Activity получает ту же ViewModel. При process death памяти процесса больше нет: вместе с ним исчезает и ViewModelStore. Новая Activity стартует в новом процессе и получает только то, что система смогла сохранить в Bundle, плюс данные из постоянного хранилища.

Activity Result API вместо onActivityResult

Старый startActivityForResult()/onActivityResult() завязан на requestCode, плохо масштабируется и легко ломается при пересоздании. Activity Result API регистрирует контракт заранее, возвращает типизированный результат и привязывает callback к lifecycle owner. Важная деталь: launcher регистрируют как поле или до старта lifecycle, а запускают позже, когда действие реально нужно.

ProfileActivity.kt
class ProfileActivity : AppCompatActivity() {

    private val pickAvatar = registerForActivityResult(
        ActivityResultContracts.GetContent()
    ) { uri: Uri? ->
        uri?.let(viewModel::onAvatarPicked)
    }

    private val requestCamera = registerForActivityResult(
        ActivityResultContracts.RequestPermission()
    ) { granted: Boolean ->
        if (granted) openCamera() else showCameraDenied()
    }

    fun onPickAvatarClick() {
        pickAvatar.launch("image/*")
    }

    fun onCameraClick() {
        requestCamera.launch(Manifest.permission.CAMERA)
    }
}

Контракт описывает обе стороны: какой input принимает launcher и какой output придёт в callback. Для кастомного сценария пишут свой ActivityResultContract<I, O>, но в большинстве случаев хватает готовых контрактов: запрос permission, выбор контента, создание документа, съёмка фото, запуск Intent sender.

Почему launcher нужно регистрировать стабильно и в одном порядке после пересоздания Activity? ★

Activity Result Registry должен сопоставить результат, который вернулся после пересоздания, с правильным callback. Если регистрировать launchers условно или менять порядок регистрации между запусками, результат может прийти не туда или потеряться. Поэтому launcher держат как поле, регистрируют при создании Activity/Fragment и не завязывают саму регистрацию на runtime-ветку. Условным остаётся только момент launch().

Task, back stack, launchMode и onNewIntent

Activity живёт не просто «в приложении», а в task — стеке экранов, который пользователь видит в Recents. Обычно каждый запуск создаёт новый экземпляр на вершине, но launch modes и Intent-флаги могут переиспользовать старый экземпляр, очистить часть стека или вынести Activity в отдельную задачу. Это сильная магия. Её лучше применять точечно.

ModeПоведениеТипичный кейс
standardВсегда новый экземплярОбычные экраны
singleTopЕсли экземпляр уже на вершине, приходит onNewIntent()Экран поиска, уведомление в уже открытый экран
singleTaskОдин корневой экземпляр в task; всё выше него очищаетсяГлавный экран, редкие deep link-флоу
singleInstanceЭкземпляр один и живёт в отдельной taskСистемные/launcher-подобные сценарии, почти не нужно в app-коде
singleInstancePerTaskОдин экземпляр на task, может быть несколько taskСпециальные multi-window/document-сценарии
SearchActivity.kt
class SearchActivity : AppCompatActivity() {

    override fun onCreate(savedInstanceState: Bundle?) {
        super.onCreate(savedInstanceState)
        handleIntent(intent)
    }

    override fun onNewIntent(intent: Intent) {
        super.onNewIntent(intent)
        setIntent(intent)      // иначе getIntent() останется старым
        handleIntent(intent)
    }

    private fun handleIntent(intent: Intent) {
        val query = intent.getStringExtra(EXTRA_QUERY).orEmpty()
        viewModel.search(query)
    }

    companion object {
        private const val EXTRA_QUERY = "query"
    }
}

Почему taskAffinity в связке с FLAG_ACTIVITY_NEW_TASK может сломать ожидаемую кнопку Back? ★

FLAG_ACTIVITY_NEW_TASK говорит системе искать task с подходящей affinity или создать новую. Если Activity имеет нестандартную taskAffinity, она может оказаться не в стеке вызывающего приложения, а в другой task. Пользователь нажимает Back и попадает не туда, куда вы мысленно положили экран. Поэтому custom affinity и launch modes меняют только под конкретный сценарий: document-mode, external entry point, launcher-подобная Activity. Для обычной внутренней навигации это почти всегда лишний риск.

Intent, deep links и экспортированная поверхность

Explicit Intent называет конкретный компонент; implicit Intent описывает действие, данные и категории, а систему просит найти получателя. Как только появляется intent-filter, Activity становится входной точкой. Внутренний экран с exported=true — это уже публичный API вашего приложения, даже если вы не планировали его документировать.

IntentExamples.kt
// Explicit: внутри своего приложения
startActivity(Intent(this, DetailActivity::class.java).apply {
    putExtra("id", itemId)
})

// Implicit: другой компонент выбирает система
val share = Intent(Intent.ACTION_SEND).apply {
    type = "text/plain"
    putExtra(Intent.EXTRA_TEXT, "Ссылка на заказ")
}
if (share.resolveActivity(packageManager) != null) {
    startActivity(Intent.createChooser(share, "Поделиться"))
}
AndroidManifest.xml
<activity
    android:name=".DeepLinkActivity"
    android:exported="true">
    <intent-filter android:autoVerify="true">
        <action android:name="android.intent.action.VIEW" />
        <category android:name="android.intent.category.DEFAULT" />
        <category android:name="android.intent.category.BROWSABLE" />
        <data android:scheme="https" android:host="example.com" />
    </intent-filter>
</activity>

С Android 12 для компонента с intent-filter нужно явно указать android:exported. Для публичной Activity проверяйте входящие extras и URI как недоверенные: другой процесс может передать неожиданный id, огромный Bundle или Intent redirection payload. Внутренние экраны держим exported=false.

Почему перед запуском implicit Intent иногда нужен resolveActivity(), но для chooser это не серебряная пуля безопасности? ★

resolveActivity() помогает не упасть с ActivityNotFoundException, если в системе нет обработчика. Но безопасность этим не заканчивается: chooser всё равно отдаёт данные другому приложению. Если вы передаёте файл, используйте FileProvider и временный FLAG_GRANT_READ_URI_PERMISSION, а не путь к файлу. Если принимаете implicit Intent сами, валидируйте data, размер extras и права вызывающего, потому что это вход снаружи процесса.

Типичные поломки Activity

Activity падает не потому, что callback «сложный», а потому что ресурс живёт дольше или короче, чем его владелец. На собесе хорошо работает разбор симптома: что видит пользователь, какой stack trace, какой lifetime нарушен.

СимптомПричинаФикс

Почему «не хранить Activity Context в singleton» остаётся актуальным даже в приложении с одной Activity? ★

Single-Activity не означает single-instance. При повороте, смене темы, multi-window и process recreation Activity всё равно пересоздаётся. Singleton живёт всё время процесса; если он держит старый Activity Context, GC не может собрать старое окно, View-дерево и всё, что оно удерживает. В singleton кладут applicationContext или вообще не кладут Context. Activity Context нужен там, где важны тема и окно: inflate UI, диалог, WindowInsets.

Частые вопросы интервьюера

Что вызывается при повороте экрана и где хранить состояние? Middle

Поворот — configuration change. Текущая Activity проходит остановку и уничтожение, затем создаётся новый экземпляр. ViewModel переживает этот сценарий, потому что ViewModelStore переносится в новый экземпляр Activity. onSaveInstanceState/SavedStateHandle тоже переживают поворот и ещё process death, но только для маленького сериализуемого состояния. Большие данные — Room/DataStore/сеть.

Чем isFinishing отличается от isChangingConfigurations? Middle+

isFinishing говорит, что Activity завершается навсегда: Back, finish(), завершение task. isChangingConfigurations говорит, что текущий экземпляр уничтожается ради нового экземпляра под другую конфигурацию. При повороте обычно isFinishing=false, isChangingConfigurations=true. При Back — наоборот. Оба флага не спасают от process death: если систему убила процесс, callback уничтожения может не прийти.

Почему onSaveInstanceState() не зовётся при обычном Back? Middle+

Back означает, что пользователь завершил экран. Системе не нужно сохранять состояние для восстановления этой Activity, поэтому callback сохранения может не вызываться. Он нужен для сценариев, где система может уничтожить Activity, но позже восстановить её: configuration change или убийство процесса в фоне. Если данные должны сохраниться после Back, это уже не saved instance state, а доменное сохранение в репозиторий/БД.

Когда использовать Activity Result API, а когда обычный Intent? Middle

Обычный startActivity() — когда результат не нужен. Activity Result API — когда нужен ответ: permission, выбранный файл, фото, документ, результат внешней Activity. Он типизирует контракт и переживает пересоздание лучше, чем старый onActivityResult(). Launcher регистрируют заранее и стабильно, а launch() вызывают по событию пользователя.

Почему singleTask редко нужен для обычной навигации? Senior

singleTask переиспользует существующий экземпляр и очищает всё, что лежит над ним в task. Это легко ломает ожидаемый Back и теряет промежуточные экраны. Для внутренней навигации проще и предсказуемее обычный back stack или Navigation Component. singleTask оставляют под корневые entry point-сценарии, где очистка стека осознанна.

Что происходит, если передать большой объект через Intent extras? Senior

Extras маршалятся в Parcel и идут через Binder. Буфер Binder-транзакций ограничен, поэтому большой список или bitmap может дать TransactionTooLargeException. Плюс вы тратите время на сериализацию на границе процесса/компонента. Правильнее передавать id, URI или ключ, а данные читать из общего источника: Room, файл, ContentProvider, репозиторий.

Задачи

Задача 1. Activity с launchMode="singleTop" открыта с query=A. Из push приходит query=B. Напишите обработку Intent так, чтобы второй query не потерялся.

Показать решение 
class SearchActivity : AppCompatActivity() {
    override fun onCreate(savedInstanceState: Bundle?) {
        super.onCreate(savedInstanceState)
        handleIntent(intent)
    }

    override fun onNewIntent(intent: Intent) {
        super.onNewIntent(intent)
        setIntent(intent)
        handleIntent(intent)
    }

    private fun handleIntent(intent: Intent) {
        val query = intent.getStringExtra("query").orEmpty()
        viewModel.search(query)
    }
}

При singleTop повторный запуск Activity на вершине стека не вызывает onCreate(). Новый Intent приходит в onNewIntent(); без setIntent(intent) последующий getIntent() вернёт старый Intent.

Задача 2. В onDestroy() нужно отличить Back от поворота. Покажите код и объясните, почему он не годится для process death.

Показать решение 
override fun onDestroy() {
    super.onDestroy()

    when {
        isFinishing -> viewModel.onUserClosedScreen()
        isChangingConfigurations -> viewModel.onScreenRecreated()
        else -> viewModel.onSystemDestroyedInstance()
    }
}

isFinishing покрывает явное завершение, isChangingConfigurations — пересоздание из-за конфигурации. Но при убийстве процесса Android не обязан вызвать onDestroy(), поэтому критичное состояние нельзя сохранять только здесь. Для process death нужны SavedStateHandle, onSaveInstanceState или постоянное хранилище.

Задача 3. Пользователь вводит текст в поиске, сворачивает приложение, процесс убивают, затем он возвращается. Какие данные хранить в ViewModel, какие — в SavedStateHandle, какие — в БД?

Показать решение

В ViewModel держим текущее состояние экрана в памяти: результаты поиска, loading/error, derived UI state. В SavedStateHandle — маленький ключ восстановления: строку query, выбранный tab, id выбранного фильтра. В Room/DataStore — долгоживущие данные и настройки. После process death ViewModel создаётся заново, берёт query из SavedStateHandle и по нему снова поднимает данные из репозитория.
