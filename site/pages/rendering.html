Конвейер кадра: Choreographer и VSYNC

Перед чтением к повторению сегодня

Кадр не успел подготовиться к следующему VSYNC, опоздав на 1 мс. Покажет ли Choreographer его «с небольшим опозданием»?

Показать разбор

Нет. Choreographer не выводит кадр позже дедлайна — он роняет его целиком, и на экране ещё раз держится предыдущий кадр. Глаз ловит это как рывок. Поэтому промах даже на миллисекунду стоит целого пропущенного кадра, и чем выше частота (8.3 мс на 120 Гц), тем дороже любая лишняя работа.

Дисплей обновляется не когда захочет приложение, а на своей частоте: 60 раз в секунду на обычном экране, 90 или 120 — на «плавных». Каждое такое обновление аппаратура анонсирует сигналом VSYNC (vertical synchronization). Между двумя соседними VSYNC у вас есть фиксированное окно, чтобы успеть подготовить новый кадр. Это и есть бюджет кадра: 1000 мс / 60 ≈ 16.6 мс при 60 Гц, 1000 / 90 ≈ 11 мс при 90 Гц и всего 1000 / 120 ≈ 8.3 мс при 120 Гц.

Связующее звено между этим аппаратным «тиком» и вашим UI — Choreographer. Он подписан на VSYNC и по каждому сигналу запускает один проход отрисовки: вызывает накопленные колбэки в строгом порядке — обработка ввода, аниматоры, затем обход дерева View или рекомпозиция Compose с measure/layout/draw. Когда вы пишете view.postOnAnimation { } или анимация Compose ждёт следующего кадра — заявка кладётся в очередь Choreographer и сработает на ближайшем VSYNC.

display list"] DRAW -->|sync| RT["RenderThread → GPU"] RT --> SF["SurfaceFlinger"] SF --> DISP["Дисплей"] DISP -.->|следующий VSYNC| VS
Один проход doFrame в Choreographer: всё слева от sync — на главном потоке, дальше — RenderThread, композиция и вывод

«Уложиться в кадр» значит: сумма ввода, аниматоров, measure/layout/draw на главном потоке плюс работа RenderThread должна влезть в окно до следующего VSYNC. Промахнулись хотя бы на миллисекунду — Choreographer не показывает кадр «с опозданием», а роняет его целиком: на экране ещё раз держится предыдущий кадр. Глаз ловит это как рывок. Чем выше частота дисплея, тем уже окно и тем дороже любая лишняя аллокация или тяжёлый onDraw.

⚠️
Бюджет — не весь кадр, а ваша доля. 16.6 мс делятся между вашим главным потоком, RenderThread и системой. На практике закладывайтесь на то, что у приложения есть условные ~12–14 мс, а не полные 16.6 — остальное съест композиция в SurfaceFlinger и накладные расходы. На 120 Гц с 8.3 мс запас тает совсем.
Почему частота дисплея может меняться на лету и при чём тут VSYNC? ★

С Android 11 (API 30) появился frame rate API и переменная частота обновления (VRR): дисплей умеет переключаться между 60/90/120 Гц прямо во время работы — экономя батарею на статичном контенте и поднимая частоту на скролле. Из-за этого нельзя зашивать «16.6 мс» в код как константу: бюджет кадра нужно брать из текущего периода VSYNC (Display.getRefreshRate() / данные кадра). Macrobenchmark и Perfetto именно поэтому считают промах не от 16.6 мс, а от фактического дедлайна кадра — отсюда метрика frameOverrunMs, на сколько кадр опоздал относительно своего реального окна.

UI Thread и RenderThread

Перед чтением к повторению сегодня

Вы синхронно распарсили большой JSON в обработчике клика, и кадр уронен. Виноват ли в этом RenderThread или GPU?

Показать разбор

Нет, они тут простаивают. UI Thread не закрашивает пиксели — он записывает команды в display list (RenderNode). Долгая работа на главном потоке мешает ему дописать display list к точке sync, и RenderThread остаётся без данных, которых ждёт. Поэтому «тяжёлое — в фон» — это правило про каждый кадр, а не только про защиту от ANR.

Распространённое заблуждение — что главный поток сам рисует пиксели. Со времён аппаратного ускорения (включено по умолчанию с Android 4.0, API 14) это не так. Главный поток (UI Thread) не закрашивает экран — он записывает, что нужно нарисовать. Каждый View.onDraw или draw-фаза Compose не льёт пиксели, а пишет команды рисования в display list — структуру внутри RenderNode. Это, по сути, запись на плёнку: «нарисуй прямоугольник тут, текст там, картинку сюда».

Готовый набор RenderNode синхронизируется на отдельный поток — RenderThread (появился в Android 5.0, API 21). Он берёт display list, транслирует команды в вызовы GPU (OpenGL ES / Vulkan) и отправляет на видеоядро. Синхронизация не «выстрелил и забыл»: на точке sync главный поток коротко ждёт RenderThread (передача состояния), после чего освобождается как можно раньше — он не блокируется до момента, пока кадр реально покажут.

sequenceDiagram participant CH as Choreographer participant UI as UI Thread participant RT as RenderThread participant GPU as GPU CH->>UI: VSYNC → doFrame UI->>UI: input, anim, measure, layout UI->>UI: draw → RenderNode (display list) UI->>RT: syncAndDrawFrame (короткий sync) RT-->>UI: разблокировка RT->>GPU: команды (GL/Vulkan) GPU-->>RT: кадр готов
Главный поток строит display list и отдаёт его RenderThread; пиксели считает GPU

Практический вывод прямой: любая долгая работа на главном потоке отъедает у measure/layout/draw то же самое окно кадра. Распарсили JSON в обработчике клика, синхронно дёрнули диск, прошли по списку из 10 000 элементов в onBindViewHolder — главный поток не успел дописать display list к точке sync, кадр уронен. RenderThread тут ни при чём: он простаивает, ожидая данные, которые UI Thread не успел подготовить. Поэтому правило «тяжёлое — в фон» — это не про «приложение не зависнет совсем» (для этого есть ANR), а буквально про каждый кадр.

ℹ️
Где это видно в трейсе. В Perfetto/systrace главный поток идёт дорожкой UI Thread, рядом — RenderThread. Если кадр красный, смотрят: проснулся ли UI Thread по VSYNC вовремя, какая фаза doFrame вылезла за бюджет, не блокировался ли RenderThread на sync или на постановке буфера в очередь.
Зачем нужен RenderNode и чем display list лучше, чем рисовать сразу? ★

Display list — это закэшированные команды рисования. Если содержимое View не менялось, его RenderNode переиспользуется как есть: главный поток не вызывает onDraw заново, а просто передаёт RenderThread тот же список. Отсюда дешёвые трансформации: сдвинуть, повернуть или поменять прозрачность RenderNode можно на RenderThread без захода на главный поток и без перезаписи команд — поэтому свойство-анимации (translation, alpha, rotation) почти бесплатны, а анимация, которая каждый кадр меняет, скажем, текст и зовёт invalidate(), заставляет перезаписывать display list и стоит ощутимо дороже. В Compose ту же роль играет слой графики и фаза draw поверх slot table: незатронутые узлы не перерисовываются.

Jank и пропущенные кадры

Перед чтением к повторению сегодня

Нужно понять, рвётся ли список в проде у реальных пользователей, и отдельно — стабильно воспроизвести просадку для поиска причины. Какие инструменты под какую задачу?

Показать разбор

JankStats — продакшен на реальных устройствах: даёт длительность каждого кадра с метками состояния, отвечает «рвётся ли и где». Для воспроизведения берут Macrobenchmark с FrameTimingMetric: прогоняет сценарий стабильно от раза к разу и выдаёт frameOverrunMs. Конкретную просадку разбирают в Perfetto FrameTimeline. Эмулятор для замеров кадров не годится — нет настоящего VSYNC-конвейера и GPU телефона.

Jank — это рывок, который видит пользователь, когда система пропустила кадр(ы) из-за того, что UI не успел отрисоваться в бюджет. Android vitals делит просадки на категории по времени отрисовки кадра: обычный кадр укладывается в окно (16 мс на 60 Гц), медленный кадр — от 16 мс примерно до 700 мс, frozen frame (зависший) — дольше 700 мс, а свыше 5 с — это уже ANR. Frozen frames выделяют отдельно: почти секунда без отклика на ввод воспринимается как «приложение повисло», особенно на старте и переходах.

КатегорияВремя кадраЧто чувствует пользователь
Нормальный кадр≤ 16 мс (при 60 Гц)Плавно
Slow / janky frame~16 мс – 700 мсРывок при скролле, дёрганая анимация
Frozen frame> 700 мс«Подвисло» на переходе или старте

Главные источники jank сводятся к перегрузу главного потока в окне кадра. Тяжёлый onDraw или тяжёлая рекомпозиция (Compose перевычисляет больше, чем нужно, потому что состояние нестабильно или ключи плохие). Аллокации прямо в кадре — каждый new в горячем пути приближает паузу GC, а пауза GC съедает миллисекунды бюджета. Лишние проходы layout — вложенные веса, двойной measure, цепочки requestLayout(). И блокировки главного потока: синхронный диск, чтение из БД, разбор больших структур.

Чем измерять

Замер jank — не «на глаз», а инструментами под разные стадии. В продакшене на реальных устройствах — JankStats: библиотека Jetpack поверх FrameMetrics, отдаёт длительность каждого кадра колбэком и позволяет навесить состояние (какой экран, что скроллим), чтобы понять, где рвётся. В лаборатории — Macrobenchmark с FrameTimingMetric: прогоняет сценарий (скролл, анимацию) и выдаёт frameOverrunMs — на сколько кадр промахнулся мимо дедлайна (положительное число = дропнутый кадр); метрика доступна с Android 12 (API 31). Для разбора конкретной просадки — Perfetto с дорожкой FrameTimeline (тоже с Android 12), которая показывает, какая именно работа уронила кадр на уровне системного компоновщика. Быстрый разовый замер из консоли — adb shell dumpsys gfxinfo <package> framestats.

JankStatsSetup.kt
// JankStats привязывается к окну; колбэк зовётся раз в кадр
val jankStats = JankStats.createAndTrack(window) { frameData: FrameData ->
    if (frameData.isJank) {
        // states — это метки, которые мы сами навесили (экран, действие)
        log("Jank ${frameData.frameDurationUiNanos / 1_000_000.0} ms, ${frameData.states}")
    }
}

// помечаем «что происходит», чтобы потом сгруппировать джанк по контексту
jankStats.state.putState("screen", "Feed")
recyclerView.addOnScrollListener(/* ... putState("scroll", "active") ... */)
💡
Связка с профилированием. JankStats отвечает на вопрос «рвётся ли и где», но не «почему». Когда нашли проблемный экран, дальше идёте в Профилирование: снимаете трейс в Perfetto/Android Studio Profiler и смотрите, какая именно фаза кадра вылезла за бюджет. Поле FrameData.states здесь — мостик: оно говорит, в каком сценарии искать.
JankStats показал джанк, но Profiler на эмуляторе всё «зелёное» — почему и где правда? ★

Эмулятор и флагман врут в разные стороны: на мощном железе вы не воспроизведёте джанк, который рвёт бюджет на бюджетном телефоне с 90 Гц. JankStats собирает данные с реальных устройств в проде, где частота дисплея и троттлинг другие, поэтому ему верят про «есть ли проблема и на каком парке». Для воспроизведения и поиска причины берут Macrobenchmark на реальном среднем устройстве с фиксированной конфигурацией (CompilationMode, прогретый кэш) — он стабилен от прогона к прогону, в отличие от ручного скролла. Эмулятор для замеров кадров не годится в принципе: у него нет настоящего VSYNC-конвейера и GPU телефона.

Overdraw

Overdraw — это когда один и тот же пиксель за один кадр закрашивается несколько раз. GPU честно рисует по алгоритму художника (back-to-front): сначала дальние слои, потом ближние поверх — это нужно для корректного смешивания полупрозрачных слоёв (теней, диммингов). Но если у вас фон Activity, поверх него непрозрачный фон контейнера, поверх — фон карточки, и всё это одного цвета, GPU трижды закрашивает те же пиксели. Видеоядро тратит время на пиксели, которых пользователь не увидит.

Смотрят overdraw через Debug GPU Overdraw в настройках разработчика. Он раскрашивает экран по числу перерисовок: синий — 1x (норма), зелёный — 2x, красный — 3x, тёмно-красный — 4x и больше (вот это лечат). Цель — чтобы экран был преимущественно синим, без больших тёмно-красных пятен.

ЦветПерерисовок пикселяВердикт
Без подсветки / синий1xНорма
Зелёный2xТерпимо
Красный3xПодозрительно
Тёмно-красный4x и большеЧинить

Лечение прозаичное. Убрать лишние фоны: контейнеры по умолчанию ничего не рисуют — фон ставьте один раз (например, на окне через android:windowBackground), а вложенным View не дублируйте тот же цвет. Сплющить иерархию — меньше слоёв непрозрачных View меньше перекрывают друг друга. Подрезать область рисования через canvas.clipRect(), если кастомный View рисует за пределами видимой зоны. И аккуратнее с прозрачностью: полупрозрачный пиксель требует сначала нарисовать то, что под ним, — серый текст напрямую дешевле, чем чёрный с альфой поверх фона.

⚠️
Не молитесь на overdraw. На современных GPU умеренный overdraw почти бесплатен, и оптимизировать его ради «чтобы было синё» — пустая трата времени. Сначала чинят главный поток (jank от рекомпозиций и layout), и только если в трейсе GPU реально упирается в fill-rate (заметный Process/swap-buffers) — лезут в overdraw. На слабых устройствах приоритет тоже у работы главного потока, а не у перекраски.
Почему в Compose overdraw устроен иначе, чем во View? ★

В Compose нет иерархии View с дефолтными фонами, но overdraw никуда не делся — он перешёл на уровень Modifier. Каждый Modifier.background(), Modifier.shadow() (тень — это отдельный слой с альфой) и graphicsLayer { alpha = } добавляет проход закраски. Особенно коварен shadow/elevation у многих вложенных Card: тени рисуются полупрозрачным слоем и множат overdraw. Плюс Modifier.graphicsLayer с alpha < 1f или compositingStrategy = Offscreen заставляет рисовать поддерево в отдельный буфер и потом смешивать — это и есть управляемый, но недешёвый overdraw. Debug GPU Overdraw работает с Compose так же, как с View, — смотрите по нему.

Производительность списков

Перед чтением к повторению сегодня

В LazyColumn вы поставили key = { it } по позиции элемента (его индексу). Что не так и что произойдёт при вставке элемента в начало списка?

Показать разбор

Ключ по индексу = поведение по умолчанию, толку ноль: личность элемента не стабильна. При вставке в начало индексы всех элементов ниже «съезжают», Compose считает их изменившимися — каскад лишних рекомпозиций и сломанная анимация перестановки. Нужен key по бизнес-id, тогда перерисуется только реально новое. Неуникальный ключ — другая крайность: рантайм бросит исключение.

Список — главный генератор jank в реальных приложениях: при скролле кадры идут подряд, и любая лишняя работа на элемент сразу превращается в рывки. Подходы во View и в Compose разные по механике, но цель одна — чтобы подготовка одного элемента стоила доли миллисекунды.

RecyclerView: переиспользование

RecyclerView держит небольшой пул ViewHolder и переиспользует их: ушедший за край экрана холдер не уничтожается, а заново привязывается к новым данным в onBindViewHolder. Отсюда правило: bind должен быть лёгким — никаких аллокаций, форматтеров «на лету», синхронного доступа к диску; документация целится в «много меньше 1 мс» на привязку. Обновления списка делают через ListAdapter + DiffUtil, который на фоне считает разницу старого и нового списка и шлёт точечные анимации вместо notifyDataSetChanged(). Чтобы DiffUtil различал «тот же элемент с новыми данными» и «другой элемент», нужны stable ids (через areItemsTheSame по бизнес-id). Префетч (LinearLayoutManager готовит холдеры впрок, пока главный поток простаивает между кадрами) включён по умолчанию и работает в связке с RenderThread.

UserAdapter.kt
class UserAdapter : ListAdapter<User, UserVH>(DIFF) {
    override fun onBindViewHolder(holder: UserVH, position: Int) {
        // лёгкий bind: только присвоить готовые поля, без форматирования и I/O
        holder.bind(getItem(position))
    }
    companion object {
        val DIFF = object : DiffUtil.ItemCallback<User>() {
            // тот же элемент? — сравниваем стабильный бизнес-id
            override fun areItemsTheSame(a: User, b: User) = a.id == b.id
            // содержимое поменялось? — data class даёт корректный equals
            override fun areContentsTheSame(a: User, b: User) = a == b
        }
    }
}

LazyColumn: ключи и contentType

В Compose LazyColumn устроен иначе: он не переиспользует View, а создаёт композаблы для видимых элементов — это дешевле, чем надувать View. Но у него свои рычаги. Параметр key задаёт стабильную личность элемента (как stable id): без него Compose опирается на индекс, и при вставке/удалении в начале списка «съезжают» все элементы ниже — лишние рекомпозиции и сломанная анимация перестановки. С ключом по бизнес-id перерисовывается ровно то, что изменилось. Параметр contentType — аналог view types из RecyclerView: он подсказывает рантайму, какие элементы однотипны, чтобы переиспользовать слоты композиции между ними (иначе разнородный список переиспользует слоты плохо).

FeedList.kt
LazyColumn {
    items(
        items = feed,
        key = { item -> item.id },              // стабильная личность элемента
        contentType = { item -> item.kind }     // группировка однотипных для reuse слотов
    ) { item ->
        // тело item должно быть «дешёвым» и стабильным:
        // тяжёлое — вынести, состояние читать узко, чтобы не рекомпозить лишнее
        FeedRow(item)
    }
}

🛑
Грабли ключа. key должен быть стабильным и уникальным. Поставите индекс (key = { it } по позиции) — толку ноль, это и есть поведение по умолчанию. Поставите неуникальный ключ (одинаковый id у разных элементов) — рантайм бросит исключение. И никогда не кладите в тело item { } создание ViewModel, запуск корутин без remember или чтение всего большого state-объекта целиком — это рекомпозит каждый видимый элемент на каждое изменение.
Когда LazyColumn проигрывает RecyclerView, несмотря на «создавать композаблы дешевле»? ★

На первом скролле «холодного» приложения. RecyclerView переиспользует уже надутые View, а LazyColumn при появлении нового contentType создаёт композицию с нуля — и если код ещё не AOT-скомпилирован (нет baseline profile), первые элементы строятся интерпретируемым байткодом и рвут кадры. Отсюда два следствия: для Compose-списков baseline profile критичнее, чем для View; и в режиме отладки (debug build) Compose заметно медленнее из-за выключенных оптимизаций компилятора — мерить производительность списка нужно только на release-сборке. На прогретом release-приложении с профилем разрыв исчезает, и Compose обычно не хуже.

Baseline Profiles и кадры

Перед чтением к повторению сегодня

Откуда вообще берётся jank на первом запуске после установки, если код вашего приложения не менялся, и за счёт чего baseline profile его убирает?

Показать разбор

ART сначала исполняет байткод интерпретатором и лишь постепенно JIT-компилирует горячие методы — первые секунды (старт, первый скролл) идут «холодными» и тормозят. Baseline profile — список горячих путей в APK/AAB; при установке ART делает по нему AOT-компиляцию заранее, и критичные пути с первого касания исполняются нативным кодом, без интерпретации, JIT-разогрева и джанка от фоновой dexopt. Порядок выигрыша — ~30% с первого запуска.

У первого запуска после установки есть встроенный jank, не связанный с вашим кодом напрямую. ART (Android Runtime) сначала исполняет ваш байткод интерпретатором, потом JIT-компилятор постепенно компилирует горячие методы в машинный код — но первые секунды работы идут «холодными» и тормозят: старт долгий, первый скролл рвётся. Baseline Profiles убирают именно это.

Baseline profile — это список горячих путей (методов и классов: старт, навигация, скролл, отрисовка кадра), который кладётся в APK/AAB файлом baseline-prof.txt. При установке ART делает по нему AOT-компиляцию (ahead-of-time) — компилирует эти методы в машинный код заранее, ещё до первого запуска. В итоге критичные пути исполняются скомпилированными с первого касания: нет ни интерпретации, ни JIT-разогрева, ни джанка от фоновой dexopt во время сессии. Документация называет порядок ~30% ускорения с первого запуска по сравнению с отсутствием профиля.

Профиль генерится бенчмарком, едет в сборке и применяется ART при установке — до первого запуска

Генерируют профиль не вручную, а тем же Macrobenchmark: BaselineProfileRule прогоняет реальный сценарий (старт, скролл списка, переходы), а инструмент записывает, какие методы оказались горячими. Этим же Macrobenchmark и проверяют эффект — гоняют один и тот же сценарий с разными CompilationMode (без профиля против с профилем) и сравнивают время старта и frameOverrunMs.

BaselineProfileGenerator.kt
@RunWith(AndroidJUnit4::class)
class BaselineProfileGenerator {
    @get:Rule val rule = BaselineProfileRule()

    @Test fun generate() = rule.collect(packageName = "com.example.app") {
        startActivityAndWait()                 // холодный старт — горячий путь
        // прокручиваем список — собираем горячие пути скролла
        device.findObject(By.res("feed")).fling(Direction.DOWN)
    }
}

Важно не путать это с самим холодным стартом. Cold start — это метрика: сколько времени от запуска процесса до первого полезного кадра (Time To Initial Display). Baseline profile — это инструмент, который этот cold start укорачивает (часть пути уже скомпилирована) и заодно убирает jank первого скролла. Профилирование cold start (где теряется время на старте) и применение baseline profile — соседние шаги одной работы.

ℹ️
Профиль и Cloud Profiles. Начиная с Android 9 (API 28) Google Play ещё агрегирует облачные профили из реального использования и досылает их, но это занимает время и нужен большой парк пользователей. Baseline profile работает с первой установки и на всех поддерживаемых версиях — поэтому его кладут руками, а облачный профиль идёт сверху.
Почему baseline profile почти не влияет на release-сборку, собранную с полной AOT-компиляцией? ★

Потому что выигрыш профиля — это разница между «интерпретация + JIT» и «нативный код». Если приложение целиком скомпилировано AOT (CompilationMode.Full(), или старое поведение Android 5–6, API 21–23, где при установке компилировалось всё), весь код и так нативный — профилю нечего ускорять. Но полная AOT-компиляция при установке давно не дефолт: она раздувает размер на диске и время установки, поэтому с Android 7 (API 24) перешли на частичную компиляцию по профилю. Поэтому корректный замер эффекта baseline profile — это CompilationMode.None() или Partial() против профиля, а не против Full(): на Full() разница исчезает, и это не баг, а ожидаемо.

Частые вопросы интервьюера

Что такое бюджет кадра и что происходит при его превышении? Middle

Бюджет — окно между двумя VSYNC: ~16.6 мс при 60 Гц, ~11 мс при 90 Гц, ~8.3 мс при 120 Гц. В него должны влезть ввод, анимации, measure/layout/draw на главном потоке и работа RenderThread. Если не влезли — Choreographer не показывает кадр позже, а роняет его целиком, на экране остаётся предыдущий кадр. Это и есть пропущенный кадр (jank).

Чем отличаются UI Thread и RenderThread? Middle+

UI Thread (главный поток) не рисует пиксели — он записывает команды рисования в display list внутри RenderNode. RenderThread (отдельный поток, с Android 5.0 / API 21) берёт этот display list, переводит в вызовы GPU и отправляет на видеоядро. На точке sync главный поток коротко ждёт передачи состояния и освобождается рано. Долгая работа на главном потоке мешает ему дописать display list к sync — кадр падает, хотя RenderThread простаивает.

Что такое frozen frame и чем он отличается от обычного jank? Middle+

Frozen frame — кадр, отрисовка которого заняла дольше 700 мс (по классификации Android vitals). Обычный медленный кадр — это рывок при скролле (16 мс – 700 мс), а frozen воспринимается как «приложение повисло» почти на секунду, типично на старте и переходах. Свыше 5 с — это уже ANR. Vitals считает их отдельной категорией, потому что причины и влияние на пользователя разные.

Чем мерить jank: JankStats, Macrobenchmark, Perfetto — что когда? Senior

JankStats — продакшен, реальные устройства: «рвётся ли и на каком экране» (поверх FrameMetrics, с метками состояния). Macrobenchmark с FrameTimingMetric — лаборатория: стабильный замер сценария, метрика frameOverrunMs (насколько кадр промахнулся мимо дедлайна), с Android 12 / API 31. Perfetto FrameTimeline — разбор конкретной просадки на уровне системы, тоже с Android 12. Разовый замер из консоли — dumpsys gfxinfo <pkg> framestats. Эмулятор для замеров кадров не годится.

Что такое overdraw и как его смотреть? Middle

Overdraw — многократная закраска одного пикселя за кадр (накладывающиеся непрозрачные фоны, тени, прозрачность). GPU рисует по алгоритму художника back-to-front, поэтому скрытые слои тоже закрашиваются. Смотрят через Debug GPU Overdraw в настройках разработчика: синий — 1x, зелёный — 2x, красный — 3x, тёмно-красный — 4x+. Лечат: убрать дублирующие фоны, сплющить иерархию, clipRect, меньше прозрачности. Но на современных GPU это вторично — сначала чинят главный поток.

Зачем в LazyColumn нужны key и contentType? Middle+

key задаёт стабильную личность элемента: без него Compose опирается на индекс, и вставка/удаление в начале списка вызывает каскад лишних рекомпозиций и ломает анимацию перестановки. С ключом по бизнес-id перерисовывается только изменившееся. contentType — аналог view types: группирует однотипные элементы, чтобы рантайм переиспользовал слоты композиции между ними. В разнородном списке без contentType переиспользование слотов работает плохо.

Почему bind в RecyclerView должен быть лёгким? Middle

onBindViewHolder вызывается на главном потоке при каждом появлении элемента — то есть в окне кадра при скролле. Если в bind есть форматирование дат, чтение с диска, аллокации или сложная логика, главный поток не успевает к точке sync и кадр падает. Документация целится в «много меньше 1 мс» на bind. Всё тяжёлое выносят: форматировать заранее в модель, грузить картинки асинхронно, никаких new в горячем пути.

Как baseline profiles убирают jank первого запуска? Senior

ART сначала интерпретирует байткод и лишь постепенно JIT-компилирует горячие методы, поэтому первые секунды (старт, первый скролл) тормозят. Baseline profile — список горячих путей в APK/AAB; при установке ART AOT-компилирует их заранее, и критичные пути с первого касания исполняются нативным кодом — без интерпретации, JIT-разогрева и джанка от фоновой dexopt. Порядок выигрыша — ~30% с первого запуска. Генерят и проверяют профиль через Macrobenchmark.

Почему аллокации в горячем пути отрисовки вредят кадрам? Senior

Каждый объект, созданный в onDraw, onBindViewHolder или теле рекомпозиции, приближает паузу сборщика мусора. Даже современный конкурентный GC даёт короткие stop-the-world-фазы, а они отъедают миллисекунды из бюджета кадра — на 120 Гц с окном 8.3 мс этого достаточно для дропа. Поэтому в горячих путях избегают аллокаций: не создают Paint/форматтеры/лямбды на кадр, переиспользуют объекты, выносят создание во remember (Compose) или в поля холдера (View).

Задачи

Задача 1. Пользователи жалуются: лента дёргается при скролле на средних телефонах, на вашем флагмане всё гладко. JankStats в проде подтверждает джанк на экране Feed. В коде элемента ленты — загрузка аватара, форматирование даты и подсчёт «сколько прошло времени». Назовите вероятную причину и предложите фикс по шагам.

Показать решение

Причина. Тяжёлый bind/рекомпозиция элемента на главном потоке в окне кадра: форматирование даты и «time ago» считаются синхронно на каждый видимый элемент при скролле, плюс возможна синхронная или плохо закэшированная загрузка аватара. На флагмане укладывается в бюджет, на среднем устройстве с 90 Гц (окно ~11 мс) — нет, кадры падают.

Фикс по шагам:

// 1) Форматирование — заранее, в модель (вне главного потока), а не в bind/item
data class FeedItemUi(
    val id: Long,
    val avatarUrl: String,
    val dateLabel: String,   // уже отформатировано в маппере на Default-диспетчере
    val timeAgo: String      // посчитано там же, не в кадре
)

// 2) Аватар — асинхронно, с кэшем (Coil/Glide), не синхронно в bind
AsyncImage(model = item.avatarUrl, contentDescription = null)

// 3) Для Compose-списка — стабильный ключ и contentType
items(feed, key = { it.id }, contentType = { it.kind }) { FeedRow(it) }

Как проверить: прогнать сценарий скролла в Macrobenchmark с FrameTimingMetric на том же среднем устройстве до и после — смотреть frameOverrunMs (должен уйти в ноль/отрицательные). JankStats в проде подтвердит снижение джанка на Feed. Подробнее про снятие трейса — в главе Профилирование.

Задача 2. Дан элемент LazyColumn, который рекомпозится у всех видимых строк на каждый тик таймера и создаёт ViewModel прямо в теле. Перепишите его так, чтобы рекомпозиция была точечной, а список переиспользовал слоты.

Показать решение 
// БЫЛО: ViewModel в теле item, чтение «жирного» state целиком,
// нет key/contentType — рекомпозит все строки на любой чих.
LazyColumn {
    items(state.items) { item ->
        val vm = MyItemViewModel()           // создаётся на каждую рекомпозицию!
        Row { Text(state.globalTitle); Text(item.name) } // читает весь state
    }
}

// СТАЛО:
LazyColumn {
    items(
        items = uiItems,
        key = { it.id },                     // стабильная личность
        contentType = { it.kind }            // reuse слотов между однотипными
    ) { item ->
        // тело читает только своё; ничего тяжёлого не создаём в кадре
        FeedRow(item)
    }
}

@Composable
private fun FeedRow(item: ItemUi) {
    // состояние/коллбэки — через remember, чтобы не пересоздавать на рекомпозицию
    val onClick = remember(item.id) { { /* … */ } }
    Row { Text(item.name) }                  // не читаем глобальный state здесь
}

Объяснение. Создание ViewModel в теле item — это аллокация на каждую рекомпозицию (та самая «работа в кадре»); владелец состояния поднимается выше или берётся через remember. Чтение state.globalTitle внутри строки делает все строки зависимыми от глобального состояния — любой тик таймера рекомпозит всё видимое; узкое чтение (только поля item) ограничивает рекомпозицию изменившейся строкой. key по бизнес-id и contentType дают рантайму переиспользовать слоты и не «съезжать» по индексу. Сложность: рекомпозиция падает с O(видимые строки) на тик до O(1) для изменившейся строки.
