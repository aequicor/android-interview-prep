Fragment: lifecycle, транзакции и Result API

Fragment — не «маленькая Activity», а объект с собственным lifecycle, отдельным lifecycle View и управлением через FragmentManager. На собесе по нему почти всегда копают в viewLifecycleOwner, back stack, commit()/commitNow(), state loss и передачу результата без прямых ссылок.

Middle Senior Частый вопрос

Зачем фрагменты ещё спрашивают

Даже если новый UI пишется на Compose, фрагменты остаются в legacy-коде, Navigation Component, BottomSheet/Dialog-сценариях, гибридных экранах и библиотеках. Интервьюеру важен не сам факт «умею создать Fragment», а понимание границ владения: кто держит View, кто держит back stack, кто отвечает за результат и почему прямые ссылки между фрагментами быстро превращаются в утечки.

ОсьActivityFragment
Владелец окнаДа, получает Window от системыНет, живёт внутри Activity/контейнера
Back stackTask/back stack ActivityBack stack внутри FragmentManager
Lifecycle ViewОбычно совпадает с экраномОтдельный lifecycle View, короче жизни фрагмента

Почему фрагмент нельзя воспринимать как «Activity без manifest»? ★

Activity — системный компонент с Window, task/back stack и Intent-входом. Fragment — объект, которым управляет FragmentManager внутри уже существующего окна. У него может быть уничтожена View, пока сам Fragment остаётся в back stack. Отсюда и главная разница: код, который трогает View, должен жить в lifecycle View, а не в lifecycle Fragment. В Activity такой двойственности обычно нет.

Два lifecycle: Fragment и его View

У фрагмента есть lifecycle самого объекта и lifecycle View. View создаётся в onCreateView()/onViewCreated() и уничтожается в onDestroyView(). Сам Fragment при этом может остаться живым: например, когда его положили в back stack через replace(). Именно на этом ломаются binding, Flow-подписки и LiveData-observers.

Lifecycle Fragment и View: после onDestroyView() Fragment либо остаётся в back stack без старой View, либо сразу идёт к onDestroy()
ProfileFragment.kt
class ProfileFragment : Fragment(R.layout.fragment_profile) {
    private var _binding: FragmentProfileBinding? = null
    private val binding: FragmentProfileBinding
        get() = checkNotNull(_binding)

    override fun onViewCreated(view: View, savedInstanceState: Bundle?) {
        super.onViewCreated(view, savedInstanceState)
        _binding = FragmentProfileBinding.bind(view)

        viewLifecycleOwner.lifecycleScope.launch {
            viewLifecycleOwner.repeatOnLifecycle(Lifecycle.State.STARTED) {
                viewModel.state.collect { state ->
                    binding.render(state)
                }
            }
        }
    }

    override fun onDestroyView() {
        _binding = null
        super.onDestroyView()
    }
}
🛑
this вместо viewLifecycleOwner — частая утечка. Подписка на lifecycle Fragment переживёт onDestroyView(). Новый emit после ухода в back stack попытается тронуть старый binding: получите NullPointerException, дубль подписки или удержание старой View.
Почему getViewLifecycleOwner() бросает исключение до onCreateView() и после onDestroyView()? ★

viewLifecycleOwner существует только пока у фрагмента есть View. До создания View и после её уничтожения owner отсутствует, поэтому обращение к нему бросает IllegalStateException. Это не каприз API, а защита от неявной подписки на мёртвую View. Подписки, которые трогают UI, ставим в onViewCreated() на текущий owner; после пересоздания View owner будет новым.

FragmentManager: чей back stack и чей scope

FragmentManager хранит активные фрагменты, выполняет транзакции и ведёт back stack. В Activity обычно берут supportFragmentManager; внутри Fragment — parentFragmentManager, если меняем соседей на уровне родителя, и childFragmentManager, если внутри текущего фрагмента есть вложенные фрагменты: tabs, pager, маленький internal-flow.

parentFragmentManagerВ FragmentНавигация между sibling-фрагментами
childFragmentManagerВ FragmentВложенный flow внутри текущего фрагмента

HostActivity.kt
class HostActivity : AppCompatActivity(R.layout.host_activity) {
    override fun onCreate(savedInstanceState: Bundle?) {
        super.onCreate(savedInstanceState)
        if (savedInstanceState == null) {
            supportFragmentManager.commit {
                setReorderingAllowed(true)
                replace<FeedFragment>(R.id.container)
            }
        }
    }
}

FragmentContainerView предпочтительнее старого FrameLayout под фрагменты: он лучше интегрирован с FragmentManager, корректно обрабатывает attach/detach View и не допускает случайных дочерних View, которые FragmentManager не контролирует.

Почему nested-фрагменты лучше класть в childFragmentManager, а не в общий manager Activity? ★

Если вложенный flow живёт внутри конкретного Fragment, его back stack должен умереть вместе с этим Fragment. childFragmentManager даёт именно такую область владения: удалили родителя — удалились и вложенные фрагменты. Если же класть внутренние tabs/pager в manager Activity, родитель умирает, а его «дети» остаются в общем back stack и начинают жить дольше своего UI. Это почти гарантированный путь к странной навигации и утечкам.

Транзакции: add, replace, commit, commitNow

Транзакция — пачка операций над FragmentManager: добавить, заменить, удалить, показать, скрыть, attach/detach. Она не обязана выполниться сразу: обычный commit() ставит работу в очередь главного потока. Это важно для рассуждения о порядке callback-ов, state loss и тестах, где хочется «прямо сейчас».

ОперацияЧто делаетПодводный камень
addДобавляет новый Fragment в контейнерСтарый остаётся; можно случайно наложить View
replaceУдаляет текущие Fragment в контейнере и добавляет новыйУ старого уничтожается View; сам Fragment может попасть в back stack
hide/showПрячет/показывает View без уничтоженияСостояние View остаётся, но память тоже занята
detach/attachУничтожает/пересоздаёт View, сохраняя FragmentВсе UI-ссылки нужно чистить в onDestroyView()
removeУдаляет Fragment из managerПосле pop back stack его уже не восстановить, если не было записи back stack
Navigation.kt
fun Fragment.openDetails(id: Long) {
    parentFragmentManager.commit {
        setReorderingAllowed(true)
        replace<DetailsFragment>(
            R.id.container,
            args = bundleOf("id" to id)
        )
        addToBackStack("details:$id")
    }
}

💡
setReorderingAllowed(true) — почти всегда да. Он разрешает FragmentManager оптимизировать порядок операций внутри транзакции и корректнее вести postponed transitions. Современные гайды рекомендуют включать его для транзакций, особенно если они идут в back stack.
Почему replace() не равен «поменять View на новую» и как это связано с back stack? ★

replace() — это сокращение для «remove всех Fragment из контейнера и add нового». Если транзакция добавлена в back stack, удалённый Fragment остаётся сохранённым как часть записи back stack: его View уничтожена, но сам объект может быть восстановлен при pop. Поэтому старый binding уже недействителен, а подписки на lifecycle Fragment всё ещё могут жить. Отсюда правило с viewLifecycleOwner и очисткой binding в onDestroyView().

Back stack, state saved и commitAllowingStateLoss

FragmentManager сохраняет своё состояние вместе с Activity. После onSaveInstanceState() система уже зафиксировала, какие фрагменты и back stack нужно восстановить. Если в этот момент попытаться сделать обычный commit(), FragmentManager бросит IllegalStateException: вы меняете состояние, которое уже сохранено.

SafeDialog.kt
fun FragmentActivity.showDialogSafely(dialog: DialogFragment, tag: String) {
    val manager = supportFragmentManager
    val canShow = lifecycle.currentState.isAtLeast(Lifecycle.State.STARTED) &&
        !manager.isStateSaved &&
        !isFinishing &&
        !isDestroyed

    if (canShow) {
        dialog.show(manager, tag)
    }
}

commitAllowingStateLoss() не делает транзакцию «безопасной». Он только говорит FragmentManager не падать, если состояние уже сохранено. Итог может потеряться после восстановления: пользователь думает, что диалог показан, а после process death или поворота его уже нет. Используйте его только там, где потеря UI-события приемлема.

APIПоведениеКогда уместен
commit()Асинхронно, падает после saved stateОбычная навигация
commitNow()Синхронно, без back stackИнициализация контейнера, тесты, когда нужен готовый Fragment сразу
commitAllowingStateLoss()Асинхронно, не падает после saved state, но результат может потерятьсяНекритичный UI, где потеря не ломает сценарий
executePendingTransactions()Принудительно выполняет pending commitsРедко; чаще сигнал, что архитектура слишком завязана на синхронность
Почему проверка isStateSaved лучше, чем ловить IllegalStateException вокруг commit()? ★

Исключение означает, что вы уже попытались изменить зафиксированное состояние FragmentManager. Это поздняя диагностика. isStateSaved проверяет условие до транзакции: если состояние сохранено, лучше отложить навигацию до возвращения в STARTED/RESUMED или сохранить событие как состояние ViewModel. Для диалога «показать ошибку» можно ничего не делать; для перехода на следующий шаг заказа потерять событие нельзя.

Передача данных: args, shared ViewModel и Fragment Result API

Прямая ссылка Fragment A на Fragment B — почти всегда плохой знак: нарушается lifetime, появляется связность с конкретным контейнером, сложнее тестировать. Есть три нормальных канала: аргументы при создании, shared ViewModel для общего состояния и Fragment Result API для одноразового результата между фрагментами.

ЗадачаИнструментПочему
Передать id при открытииArguments / Navigation argsДанные нужны новому Fragment с момента создания
Общий state двух экрановactivityViewModels() или navGraph ViewModelState живёт в общем owner, не в соседнем Fragment
Вернуть выбор из BottomSheet/DialogFragment Result APIОдноразовый результат без прямых ссылок
FragmentResult.kt
class FeedFragment : Fragment(R.layout.feed) {
    override fun onViewCreated(view: View, savedInstanceState: Bundle?) {
        parentFragmentManager.setFragmentResultListener(
            "filter",
            viewLifecycleOwner
        ) { _, bundle ->
            val sort = bundle.getString("sort").orEmpty()
            viewModel.applySort(sort)
        }
    }
}

class FilterBottomSheet : BottomSheetDialogFragment() {
    fun onSortSelected(sort: String) {
        parentFragmentManager.setFragmentResult(
            "filter",
            bundleOf("sort" to sort)
        )
        dismiss()
    }
}

Listener привязан к viewLifecycleOwner, поэтому результат доставится только когда View получателя жива. Если результат установлен раньше listener-а, FragmentManager хранит последний результат по ключу и отдаёт его, когда listener станет активным.

Почему Fragment Result API хранит только один результат на ключ, а не очередь событий? ★

API моделирует «последний результат для ключа», а не event queue. Это хорошо для выбора фильтра: если пользователь три раза поменял сортировку в BottomSheet, получателю нужен последний выбор. Для событий, где важна каждая запись, Fragment Result API не подходит: берите shared ViewModel с SharedFlow/Channel или доменное хранилище событий. И обязательно проектируйте ключи: строковый ключ — часть контракта между отправителем и получателем.

DialogFragment и BottomSheet: те же правила lifecycle

DialogFragment и BottomSheetDialogFragment удобны для выбора, подтверждения и короткого ввода, но правила те же: не держать ссылку на Activity/Fragment-получатель, не коммитить после saved state, возвращать результат через Fragment Result API или shared ViewModel. Показывать диалог из сетевого callback-а после onSaveInstanceState() — классический путь к Can not perform this action after onSaveInstanceState.

ConfirmDeleteDialog.kt
class ConfirmDeleteDialog : DialogFragment() {

    override fun onCreateDialog(savedInstanceState: Bundle?): Dialog {
        return AlertDialog.Builder(requireContext())
            .setTitle("Удалить элемент?")
            .setPositiveButton("Удалить") { _, _ ->
                parentFragmentManager.setFragmentResult(
                    REQUEST_KEY,
                    bundleOf("confirmed" to true)
                )
            }
            .setNegativeButton("Отмена", null)
            .create()
    }

    companion object {
        const val REQUEST_KEY = "confirm_delete"
    }
}
⚠️
Не передавайте callback через constructor. Fragment должен переживать пересоздание системой. Переданный в constructor listener потеряется или удержит старую Activity. Аргументы — через arguments, результат — через Fragment Result API или ViewModel.
Почему requireContext() в onCreateDialog() допустим, а хранить этот Context в поле — нет? ★

requireContext() в момент создания диалога берёт актуальный Context, привязанный к текущему attach. Хранить его в поле опасно: Fragment может detach/attach, Activity пересоздастся, а поле продолжит держать старый Context и всё окно. Если Context нужен кратко — берём локально в callback. Если нужен дольше UI — почти всегда нужен applicationContext или явная зависимость, не Activity.

Типичные поломки Fragment

Фрагменты ломаются на границах: не тот manager, не тот lifecycle owner, транзакция после saved state, передача результата через прямую ссылку. Хороший ответ на собесе называет не только «как правильно», но и симптом, по которому вы найдёте баг в проде.

СимптомПричинаФикс
NPE после возврата из back stackПодписка пережила onDestroyView() и трогает старый bindingviewLifecycleOwner, очистка binding
Диалог падает после сетевого ответаcommit() после onSaveInstanceState()Проверка isStateSaved, событие через state, показ после STARTED
Back закрывает не тот экранТранзакция ушла не в тот FragmentManagerРазвести parentFragmentManager и childFragmentManager
Результат не дошёлКлючи не совпали или listener привязан к мёртвому ownerКонстанта ключа, listener в onViewCreated()
Почему «вызову findFragmentByTag() и напрямую дёрну метод» плохо даже если работает в демо? ★

Такой код привязывает отправителя к конкретному контейнеру, tag и классу получателя. После пересоздания, другого layout-а, nested manager или Navigation Component получатель может быть не там, где вы его ищете. Плюс прямой вызов обходит lifecycle: можно вызвать метод, когда View получателя уже уничтожена. Контракт через arguments/result/shared ViewModel явно говорит, кому принадлежит состояние и когда его можно доставить.

Частые вопросы интервьюера

Зачем нужен viewLifecycleOwner? Middle

У Fragment есть lifecycle объекта и lifecycle его View. View уничтожается в onDestroyView(), а Fragment может остаться в back stack. Всё, что трогает View — binding, adapter, Flow/LiveData render, — привязываем к viewLifecycleOwner. Тогда подписка умирает вместе с View и создаётся заново в onViewCreated().

Чем commit() отличается от commitNow()? Middle+

commit() ставит транзакцию в очередь главного потока: callbacks нового Fragment не обязаны выполниться до следующей строки кода. commitNow() выполняет транзакцию синхронно, но не поддерживает addToBackStack(). Для обычной навигации берут commit(); commitNow() оставляют для инициализации или тестов, где нужен готовый Fragment сразу.

Что такое state loss и почему commitAllowingStateLoss() опасен? Senior

State loss — транзакция после того, как FragmentManager уже сохранил своё состояние для восстановления. Обычный commit() падает, потому что новое состояние не попадёт в сохранённый снимок. commitAllowingStateLoss() подавляет падение, но результат может исчезнуть после пересоздания. Это допустимо только для некритичного UI; навигацию и бизнес-события так терять нельзя.

Когда брать shared ViewModel, а когда Fragment Result API? Middle+

Shared ViewModel — для общего состояния, которое живёт в owner: Activity, nav graph или родительский Fragment. Fragment Result API — для одноразового результата: фильтр из BottomSheet, подтверждение из DialogFragment, выбранный id. Если нужно хранить stream событий или каждое событие важно, Fragment Result API уже тесен — берите ViewModel с Flow.

Почему replace() может привести к утечке binding? Middle+

replace() удаляет старый Fragment из контейнера; если транзакция в back stack, сам Fragment сохраняется, но его View уничтожается. Если binding хранится в поле и не очищается в onDestroyView(), Fragment удерживает старую View. При возврате создаётся новая View, а старая всё ещё висит в памяти. Поэтому pattern с nullable binding и очисткой обязателен для View Binding во Fragment.

Почему нельзя передавать аргументы Fragment через constructor? Middle

Система может пересоздать Fragment после process death или configuration change через пустой constructor и восстановленные arguments. Constructor-параметры при этом потеряются. Поэтому входные данные кладут в arguments Bundle или используют Navigation args; зависимости создаёт DI/FragmentFactory, но runtime-параметры экрана идут через arguments.

Задачи

Задача 1. Исправьте Fragment, который собирает Flow в lifecycleScope самого Fragment и падает после возврата из back stack.

Показать решение 
override fun onViewCreated(view: View, savedInstanceState: Bundle?) {
    super.onViewCreated(view, savedInstanceState)

    viewLifecycleOwner.lifecycleScope.launch {
        viewLifecycleOwner.repeatOnLifecycle(Lifecycle.State.STARTED) {
            viewModel.items.collect { items ->
                adapter.submitList(items)
            }
        }
    }
}

Подписка живёт ровно столько, сколько живёт View. При onDestroyView() owner переходит в DESTROYED, корутина отменяется, а при новой View код запускается заново в onViewCreated().

Задача 2. Покажите безопасную передачу результата из BottomSheet во Fragment списка без прямой ссылки на получателя.

Показать решение 
private const val FILTER_KEY = "filter"
private const val SORT_KEY = "sort"

class ListFragment : Fragment(R.layout.list) {
    override fun onViewCreated(view: View, savedInstanceState: Bundle?) {
        parentFragmentManager.setFragmentResultListener(
            FILTER_KEY,
            viewLifecycleOwner
        ) { _, bundle ->
            viewModel.applySort(bundle.getString(SORT_KEY).orEmpty())
        }
    }
}

class FilterSheet : BottomSheetDialogFragment() {
    fun select(sort: String) {
        parentFragmentManager.setFragmentResult(
            FILTER_KEY,
            bundleOf(SORT_KEY to sort)
        )

    }
}

Оба фрагмента договариваются только о строковом ключе и Bundle-контракте. Получатель не передаёт ссылку в sheet, а listener привязан к lifecycle его View.

Задача 3. Нужно показать DialogFragment из callback-а репозитория. Что проверить перед show() и почему commitAllowingStateLoss() не решает проблему?

Показать решение 
fun FragmentActivity.tryShowErrorDialog(message: String) {
    val manager = supportFragmentManager
    val active = lifecycle.currentState.isAtLeast(Lifecycle.State.STARTED)

    if (active && !manager.isStateSaved && !isFinishing && !isDestroyed) {
        ErrorDialog.newInstance(message).show(manager, "error")
    } else {
        viewModel.rememberPendingError(message)
    }
}

Если state уже сохранён, обычный commit упадёт, а commitAllowingStateLoss() просто разрешит потерять показ диалога после восстановления. Для важного события лучше записать его в state и показать, когда экран снова STARTED.
