Тема 02 · Язык и проектирование

Язык Kotlin

Глубокое знание Kotlin — базовая компетенция Android-разработчика. На собесе проверяют не синтаксис, а понимание того, во что компилируется код: почему !! опасен, чем lateinit отличается от by lazy, что генерирует data class, сколько стоит лямбда без inline и зачем нужен reified. Здесь — система типов, null-safety, generics, delegates и interoperability with Java на уровне «под капотом».

Middle Senior Частый вопрос

Null-safety: система типов против NPE

Kotlin делит типы на nullable (String?) и non-null (String) на уровне компилятора. Это не рантайм-проверка, а статическая гарантия: компилятор не даст присвоить null в non-null тип и не пустит «голый» вызов метода у nullable-ссылки.

NullSafety.kt
val a: String = "ok"
// val b: String = null            // ❌ не компилируется
val c: String? = null              // ✅ nullable

val len1 = c?.length               // safe call → Int? (null, если c == null)
val len2 = c?.length ?: 0          // Elvis → подставит 0
val len3 = c!!.length              // !! → бросит NPE, если c == null

c?.let { s -> println(s.length) }  // блок выполнится ТОЛЬКО если c != null

Платформенные типы из Java

Главная дыра в null-safety — интероп с Java. Тип без аннотаций (@Nullable/@NonNull) Kotlin видит как платформенный String!: компилятор не знает, nullable он или нет, и отключает проверки. Вы можете присвоить его в non-null тип — и получить NPE в неожиданном месте.

⚠️
Подводный камень. val name: String = javaObject.getName() скомпилируется, даже если Java вернёт null — NPE прилетит при присваивании, а стектрейс укажет на ваш Kotlin-код, а не на источник. На границе с Java явно указывайте тип как nullable и обрабатывайте null, либо доверяйте JSR-305 / @Nullable-аннотациям.
★ Null-safety под капотом: где именно появляется NPE Senior

Для чистого Kotlin null-safety в основном работает на этапе компиляции: nullable и non-null типы не дают написать опасный код. Runtime-проверки появляются на границах, где компилятор уже не может доказать безопасность.

СитуацияКогда упадётЧто происходит под капотом
x!!В момент выполнения !!, если x == null.Компилятор вставляет not-null assertion; дальше код работает как с non-null.
Java platform type → Kotlin non-nullСразу после Java-вызова и до сохранения в non-null переменную.Для val s: String = java.getName() компилятор вставляет assertion, чтобы null не распространился дальше.
Platform value в non-null параметрПеред входом в функцию или на границе вызова.В байткод добавляется проверка параметра/аргумента.
lateinit до присваиванияПри первом чтении свойства.Это не NPE, а UninitializedPropertyAccessException: поле ещё не инициализировано.

Оптимизация. Проверки не вставляются везде подряд: smart cast, явные null-check, safe call и Elvis помогают компилятору сузить тип. На JVM простые null-check часто дополнительно убираются JIT-компилятором как redundant checks. Но на границе Kotlin↔Java проверки остаются важными: Java может вернуть null даже там, где Kotlin-код дальше ожидает non-null.

lateinit vs by lazy

Оба решают «значение появится позже», но по-разному. lateinit var — обещание компилятору «инициализирую до первого чтения»; backing field — скрытое JVM-поле, где хранится реальное значение свойства. Доступ до инициализации бросает UninitializedPropertyAccessException. by lazy — делегат: значение вычисляется при первом обращении и кешируется.

ℹ️
Что такое backing field. В Kotlin свойство — это не просто поле, а пара accessor-ов (get/set) плюс хранилище, если значение нужно держать в памяти. Это хранилище и называется backing field. У var name = "" оно есть; у вычисляемого val fullName get() = "$first $last" его нет. В кастомном setter/getter к нему обращаются через ключевое слово field.
Критерийlateinit varval ... by lazy
МутабельностьТолько varТолько val
ТипыТолько не-примитивы (нет Int/Boolean)Любые, включая примитивы
Кто инициализируетВнешний код (DI, onCreate)Сам делегат — лямбда инициализации
Когда вычисляетсяКогда присвоятЛениво, при первом чтении
ПотокобезопасностьНетПо умолчанию SYNCHRONIZED
Можно проверить инициализацию::prop.isInitializedНет (но всегда инициализируется при чтении)
LateinitVsLazy.kt
class Screen {
    // значение придёт извне (например, из DI или onCreate)
    lateinit var presenter: Presenter

    // вычислится один раз при первом обращении, потом закешируется
    val formatter: DateFormatter by lazy { DateFormatter(Locale.getDefault()) }

    fun render() {
        if (::presenter.isInitialized) presenter.attach()
        println(formatter.now())   // здесь lazy-блок отработает первый раз
    }
}

lateinit в Activity и Fragment

В Activity lateinit var binding часто нормален: binding создаётся в onCreate() и живёт до onDestroy(), как сама Activity. У Fragment другая модель: Fragment-объект может жить на back stack, а его View уже уничтожена в onDestroyView(). Поэтому lateinit для view binding во Fragment опасен: его нельзя сбросить в null, и он будет держать старое дерево View.

ActivityVsFragmentBinding.kt
class ProfileActivity : AppCompatActivity() {
    private lateinit var binding: ActivityProfileBinding

    override fun onCreate(savedInstanceState: Bundle?) {
        super.onCreate(savedInstanceState)
        binding = ActivityProfileBinding.inflate(layoutInflater)
        setContentView(binding.root)
    }
}

class ProfileFragment : Fragment(R.layout.profile) {
    private var _binding: FragmentProfileBinding? = null
    private val binding get() = _binding!!

    override fun onViewCreated(view: View, savedInstanceState: Bundle?) {
        _binding = FragmentProfileBinding.bind(view)
    }

    override fun onDestroyView() {
        _binding = null
        super.onDestroyView()
    }
}
⚠️
Почему это не дискриминация, а lifecycle. У Activity один основной lifecycle UI. У Fragment два lifecycle: lifecycle самого Fragment и lifecycle его View. Binding ссылается именно на View, поэтому его срок жизни должен быть ограничен периодом между onCreateView/onViewCreated и onDestroyView. Для этого нужен nullable backing property или lifecycle-aware delegate.
💡
Режимы lazy. LazyThreadSafetyMode.SYNCHRONIZED (по умолчанию) — блокировка, гарантирует одно вычисление. PUBLICATION — лямбда может выполниться несколько раз, но запишется результат первого завершившегося. NONE — без синхронизации, быстрее всего, но только если доступ из одного потока (типично для UI).

Классы данных, sealed-иерархии, object

data class: что именно генерируется

Компилятор по свойствам из первичного конструктора генерирует: equals()/hashCode() (структурное равенство), toString(), copy() и componentN() (для деструктуризации). Свойства, объявленные в теле класса, в эти методы не попадают.

DataClass.kt
data class User(val id: Long, val name: String) {
    var note: String = ""   // ⚠️ НЕ учитывается в equals/hashCode/copy/toString
}

val u1 = User(1, "Ann")
val u2 = u1.copy(name = "Bob")        // copy: меняем часть полей
val (id, name) = u1                   // деструктуризация через component1/component2
println(u1 == User(1, "Ann"))         // true — структурное равенство

Почему data class нельзя наследовать другим классом и почему он не может быть open/abstract? Наследование сломало бы контракт сгенерированных equals/hashCode (симметричность, проблема «равенства подтипа и супертипа»), поэтому data class неявно final. Сам он может реализовывать интерфейсы, но не быть базовым data-классом.

sealed class / sealed interface и исчерпывающий when

Sealed-тип — это «закрытое» множество наследников, известных на этапе компиляции (все в одном модуле/пакете). Это даёт компилятору исчерпывающую проверку when: перечислите все ветки — и else не нужен; добавите нового наследника — код перестанет компилироваться, пока вы не обработаете новый случай. Идеальная модель UI-state и результатов.

UiState.kt
sealed interface UiState {
    data object Loading : UiState
    data class Success(val data: List<User>) : UiState
    data class Error(val message: String) : UiState
}

fun render(state: UiState) = when (state) {   // без else — компилятор знает все ветки
    UiState.Loading      -> showSpinner()
    is UiState.Success   -> showList(state.data)
    is UiState.Error     -> showError(state.message)
}
classDiagram class UiState <<sealed interface>> UiState class Loading <<data object>> Loading class Success { +List~User~ data } class Error { +String message } UiState <|.. Loading UiState <|.. Success UiState <|.. Error
Закрытая иерархия sealed interface: компилятор знает всех наследников → исчерпывающий when

enum vs sealed

enum — фиксированный набор экземпляров-констант одного типа (одинаковая структура, у каждого свои значения полей). sealed — набор разных типов, каждый со своей структурой и состоянием. Нужен общий контракт без данных — enum; ветки несут разные поля — sealed.

object, companion object, data object

object — синглтон (ленивый, потокобезопасный, гарантируется JVM-загрузчиком классов). companion object — «статика» класса: единственный объект-компаньон, через который вызывают фабрики/константы. data object (Kotlin 1.9+) добавляет осмысленный toString() вида Loading — удобно для логирования sealed-веток без состояния.

Objects.kt
object AppConfig { const val BASE_URL = "https://api.example.com" }  // синглтон

class User private constructor(val id: Long) {
    companion object Factory {        // вызывается как User.create(1)
        fun create(id: Long) = User(id)
    }
}

data object Loading                    // toString() == "Loading"

value / inline class

@JvmInline value class оборачивает одно значение, но в рантайме компилятор по возможности разворачивает его в само значение — нет аллокации обёртки. Даёт типобезопасность без накладных расходов: UserId и OrderId нельзя перепутать, хотя оба — Long.

ValueClass.kt
@JvmInline
value class UserId(val value: Long)

@JvmInline
value class OrderId(val value: Long)

fun load(id: UserId) { /* ... */ }
// load(OrderId(5))   // ❌ перепутать нельзя — разные типы, хотя оба Long
⚠️
Когда обёртка всё же создаётся (boxing). Если value class используется как nullable (UserId?), в дженериках, в коллекциях или передаётся в Java — компилятор вынужден «упаковать» его в реальный объект. Выигрыш в нуле аллокаций сохраняется только на «прямых» путях.

Extension-функции и inline

Extension'ы — это статика

Extension-функция не меняет класс и не даёт доступа к его private-членам. Компилятор превращает её в обычный статический метод, где приёмник (this) передаётся первым аргументом. Из этого следуют два важных свойства: extension'ы резолвятся статически (по объявленному типу, а не по рантайм-классу) и не участвуют в полиморфизме.

Extensions.kt
fun String.shout(): String = this.uppercase() + "!"
// ~ компилируется в: static String shout(String $receiver) { ... }

open class A
class B : A()
fun A.who() = "A"
fun B.who() = "B"

val ref: A = B()
println(ref.who())   // "A" — резолвится по СТАТИЧЕСКОМУ типу ref (A), не B!

inline, noinline, crossinline

Каждая лямбда без inline — это объект (анонимный класс / Function-инстанс): аллокация + вызов через invoke(). Для горячих higher-order функций (например, у коллекций) это заметно. Модификатор inline подставляет тело функции и тела лямбд прямо в место вызова — лямбда исчезает как объект.

МодификаторЧто делаетЗачем
inline (на функции)Встраивает код функции и лямбд в место вызоваУбрать аллокацию лямбд; разрешить reified и non-local return
noinline (на параметре)Конкретную лямбду НЕ встраиватьКогда лямбду надо сохранить/передать дальше как объект
crossinline (на параметре)Встроить, но запретить non-local returnЛямбда вызывается из другого контекста (например, из Runnable)

reified-дженерики

На JVM дженерики стираются (type erasure) — в рантайме T неизвестен. Но в inline-функции тип подставляется в место вызова, поэтому reified T «материализуется»: можно делать is T, T::class, as T. Без inline это невозможно — пришлось бы передавать Class<T> вручную.

Reified.kt
inline fun <reified T> Gson.fromJson(json: String): T =
    fromJson(json, T::class.java)        // T известен — type erasure обойдён

// вызов без передачи Class<T> вручную:
val user: User = gson.fromJson(jsonString)

inline fun <reified T> List<*>.filterByType(): List<T> =
    filter { it is T }.map { it as T }
💡
Не злоупотребляйте inline. Встраивание дублирует байткод в каждой точке вызова → раздувание методов и риск выйти за лимиты. Inline оправдан для маленьких higher-order функций и там, где нужны reified или non-local return. Для большой функции выигрыш от убранной лямбды не покроет рост кода.

Scope-функции: let / run / with / apply / also

Пять функций различаются по двум осям: как они ссылаются на объект (this vs аргумент it) и что возвращают (сам объект vs результат лямбды). Понимание этих осей убирает 90% путаницы.

ФункцияКонтекстВозвращаетКогда применять
letitРезультат лямбдыNull-safe цепочки (?.let), трансформация значения
runthisРезультат лямбдыКонфигурация + вычисление результата в одном блоке
withthisРезультат лямбдыНесколько вызовов одного объекта (не extension)
applythisСам объектНастройка объекта (builder-стиль), возврат его же
alsoitСам объектПобочные эффекты (лог, валидация) в цепочке
ScopeFunctions.kt
// apply — настроить и вернуть объект
val paint = Paint().apply {
    color = Color.RED
    isAntiAlias = true
}

// let — выполнить только при non-null + получить результат
val length: Int = nullableName?.let { it.trim().length } ?: 0

// also — вклиниться с побочным эффектом, не разрывая цепочку
val users = repository.load()
    .also { Log.d("TAG", "loaded ${it.size}") }
    .filter { it.isActive }

// run — конфигурация + результат
val message = StringBuilder().run {
    append("Hello, ")
    append(name)
    toString()
}

Дженерики и вариативность

По умолчанию дженерики в Kotlin инвариантны: Box<String> не является подтипом Box<Any>, даже если String : Any. Это безопасно, но иногда слишком строго — на помощь приходят модификаторы out и in.

МодификаторВариативностьРольПравило
out TКовариантностьProducer (только отдаёт T)Box<String> → подтип Box<Any>
in TКонтравариантностьConsumer (только принимает T)Box<Any> → подтип Box<String>
(без модификатора)ИнвариантностьИ читает, и пишетПодтипирования по T нет

Мнемоника PECS из Java работает и тут: Producer-Extends (out), Consumer-Super (in). out запрещает использовать T во входных позициях (нельзя «положить»), in — в выходных (нельзя «достать» как T, только как Any?).

Variance.kt
// out: только производит T → ковариантность
interface Source<out T> { fun next(): T }
val anySource: Source<Any> = object : Source<String> { override fun next() = "x" }  // ✅

// in: только потребляет T → контравариантность
interface Sink<in T> { fun put(item: T) }
val stringSink: Sink<String> = object : Sink<Any> { override fun put(item: Any) {} } // ✅

// star-projection: тип не важен, но безопасность сохраняется
fun printSize(box: Source<*>) { /* читаем как Any?, конкретный T неизвестен */ }
ℹ️
Star-projection <*> — «какой-то конкретный, но неизвестный тип». Не то же, что <Any?>: MutableList<*> можно читать (как Any?), но нельзя в него писать — компилятор не знает реальный T и не гарантирует безопасность записи. Это и есть встроенный в систему типов «безопасный wildcard».

Делегирование: by для классов и свойств

Делегирование класса (class delegation)

Конструкция class C : Iface by impl заставляет компилятор сгенерировать все методы интерфейса Iface как форвардинг к impl. Это композиция вместо наследования без boilerplate — реализуем интерфейс, «одолжив» готовый объект, и при желании переопределяем отдельные методы.

ClassDelegation.kt
interface Repository { fun load(): List<User> }

class CachingRepository(
    private val origin: Repository
) : Repository by origin {           // load() автоматически делегируется origin
    private var cache: List<User>? = null
    override fun load(): List<User> = cache ?: origin.load().also { cache = it }
}

Делегаты свойств (property delegates)

Свойство может делегировать чтение/запись объекту, у которого есть getValue/setValue. Стандартные делегаты:

ДелегатПоведение
by lazy { }Ленивое одноразовое вычисление val (см. выше)
Delegates.observable(init){ p,old,new -> }Колбэк после каждого изменения var
Delegates.vetoable(init){ p,old,new -> Bool }Колбэк до изменения; false отменяет присваивание
by mapСвойства читаются из Map по имени ключа
Delegates.notNull<T>()Non-null var с поздней инициализацией (аналог lateinit для примитивов)
PropertyDelegates.kt
import kotlin.properties.Delegates

class Form {
    // observable — реагируем после изменения
    var name: String by Delegates.observable("") { _, old, new ->
        println("name: $old -> $new")
    }
    // vetoable — можем отклонить недопустимое значение
    var age: Int by Delegates.vetoable(0) { _, _, new -> new in 0..150 }
}

// кастомный делегат — любой объект с getValue/setValue
class Prefs(private val sp: SharedPreferences) {
    operator fun getValue(ref: Any?, p: KProperty<*>): String =
        sp.getString(p.name, "") ?: ""
    operator fun setValue(ref: Any?, p: KProperty<*>, value: String) =
        sp.edit().putString(p.name, value).apply()
}

Равенство, иммутабельность, const vs val

== vs ===

== — структурное равенство, под капотом вызывает equals() (с null-safe обвязкой: a?.equals(b) ?: (b === null)). === — ссылочное равенство (тот же объект в памяти). Для data class == сравнивает поля, а === — идентичность ссылки.

Equality.kt
val a = User(1, "Ann")
val b = User(1, "Ann")
println(a == b)    // true  — структурно равны (data class equals)
println(a === b)   // false — разные объекты в памяти

// контракт: если a == b, то a.hashCode() == b.hashCode()
// нарушите — сломаются HashMap/HashSet (объект «потеряется» в корзине)
🛑
Контракт equals/hashCode. Равные объекты обязаны иметь равный hashCode. Если вручную переопределить только equals, объект «исчезнет» из HashMap/HashSet. И никогда не кладите в hash-коллекцию data class с var-полем: изменив поле после вставки, вы поменяете hashCode и потеряете элемент.

Иммутабельность: val ≠ immutable

val запрещает переприсваивание ссылки, но не делает объект неизменяемым. val list = mutableListOf(1) — список можно менять, нельзя лишь присвоить новую ссылку list. Настоящая иммутабельность — это неизменяемый тип (read-only интерфейс List, data class с одними val).

const val vs val

const val — compile-time константа: значение подставляется в байткод в местах использования, доступно только для примитивов и String, на верхнем уровне или в object/companion. Обычный val вычисляется в рантайме и хранится в поле с геттером.

Const.kt
object Api {
    const val VERSION = "v1"            // зашивается в байткод (инлайнится)
    val timestamp = System.currentTimeMillis()  // рантайм — const тут невозможен
}

Интероп с Java и коллекции

Аннотации для Java-клиентов

Kotlin компилируется в JVM-классы, но «удобный для Kotlin» байткод не всегда удобен для Java. Аннотации управляют генерацией:

АннотацияЧто делает для Java-вызова
@JvmStaticМетод companion/object становится настоящим static (вызов Foo.bar() вместо Foo.Companion.bar())

class Analytics {
    companion object {
        @JvmStatic fun init() { /* ... */ }   // Java: Analytics.init()
    }
    @JvmField val tag = "ANALYTICS"           // Java: analytics.tag

    @JvmOverloads
    fun track(event: String, value: Int = 0, force: Boolean = false) { /* ... */ }
    // Java получит 3 перегрузки: track(e), track(e,v), track(e,v,f)
}

List vs MutableList

В Kotlin List — это read-only интерфейс (только чтение), а MutableList добавляет add/remove. Это не гарантия иммутабельности: за List может скрываться изменяемый ArrayList, и кто-то по другой ссылке его изменит. Read-only означает «через эту ссылку менять нельзя», а не «объект неизменяем».

Sequence: ленивость и когда она быстрее

Операции над List (map, filter) энергичные: каждая создаёт новую промежуточную коллекцию и проходит её целиком. Sequence — ленивая: элементы протягиваются по цепочке по одному, промежуточных коллекций нет, и терминальная операция (toList, first) запускает конвейер.

flowchart LR A[Элемент 1] --> F1{filter} F1 -->|прошёл| M1[map] M1 --> T[toList] A2[Элемент 2] --> F2{filter} F2 -->|отброшен| X[стоп] A3[Элемент 3] --> F3{filter} F3 -->|прошёл| M3[map] M3 --> T
Sequence: каждый элемент проходит ВСЮ цепочку по одному, без промежуточных списков
Sequence.kt
// List: 2 промежуточных списка, оба полностью обрабатываются
val r1 = (1..1_000_000).toList()
    .map { it * 2 }      // создаёт список из 1 000 000
    .filter { it > 10 }  // создаёт ещё один список
    .take(3)             // и только теперь берём 3

// Sequence: ленивый конвейер — обработается ~немного элементов
val r2 = (1..1_000_000).asSequence()
    .map { it * 2 }
    .filter { it > 10 }
    .take(3)             // короткое замыкание: остановится после 3 подходящих
    .toList()
💡
Когда что. Sequence выигрывает на больших данных, длинных цепочках и при коротком замыкании (take/first/find) — экономит память и обходит лишние элементы. На маленьких коллекциях Sequence медленнее из-за оверхеда итераторов и лямбд на каждом шаге — обычный List предпочтительнее.

Частые вопросы интервьюера

Чем lateinit отличается от by lazy и когда что? Middle+

lateinit var — для var, не-примитивов, инициализируется извне (DI, onCreate), не потокобезопасен; до инициализации бросает UninitializedPropertyAccessException, проверка через ::prop.isInitialized. by lazy — для val, любого типа, инициализируется сам своей лямбдой лениво при первом чтении, кеширует результат, по умолчанию SYNCHRONIZED. Правило: значение приходит снаружи и может меняться → lateinit; вычисляется один раз дорого и лениво → by lazy. Для Fragment view binding обычно не используют lateinit: View живёт короче Fragment, binding нужно обнулять в onDestroyView().

Что такое backing field и чем он отличается от backing property? Middle

Backing field — скрытое поле, которое компилятор создаёт для свойства, если нужно хранить значение. В custom getter/setter доступно как field. Backing property — ручной паттерн с двумя свойствами: приватное mutable-хранилище и публичный read-only контракт, например private val _state = MutableStateFlow(...) и val state: StateFlow<UiState> = _state. В Fragment binding это private var _binding: Binding? = null + val binding get() = _binding!!.

Когда NPE появляется при Java→Kotlin interop? Senior

Если Java API без nullability-аннотаций возвращает platform type T!, Kotlin позволяет трактовать его и как T, и как T?. Когда ты выбираешь non-null T, компилятор вставляет runtime assertion. Поэтому val s: String = java.getName() падает сразу после Java-вызова, если тот вернул null. То же происходит при передаче platform value в non-null Kotlin-параметр. Лечение: явно принимать T?, обрабатывать null или добавлять корректные Java-аннотации @Nullable/@NonNull.

Что именно генерирует data class и почему его нельзя наследовать? Middle

По свойствам первичного конструктора генерируются equals, hashCode, toString, copy и componentN. Свойства в теле класса в них не входят. Data class неявно final: наследование сломало бы сгенерированный equals (симметричность, проблема равенства подтипа и базового типа), поэтому он не может быть open/abstract и быть базой для другого data class — но реализовывать интерфейсы может.

Во что компилируется extension-функция? Полиморфна ли она? Senior

В статический метод с приёмником первым параметром. Она не модифицирует класс и не видит его private-членов. Резолвится статически — по объявленному типу переменной, а не по рантайм-классу, поэтому не участвует в полиморфизме: если у A и B : A есть одноимённые extension'ы, для val x: A = B() вызовется версия для A.

Зачем inline и сколько стоит лямбда без него? Senior

Лямбда без inline — это объект (Function-инстанс): аллокация в куче + вызов через invoke(). На горячих путях (операции коллекций) это даёт давление на GC и оверхед вызова. inline подставляет тело функции и лямбд в место вызова — лямбда исчезает как объект, плюс открываются reified-дженерики и non-local return. Минус — дублирование байткода, поэтому inline уместен для небольших higher-order функций.

Что такое reified и почему он требует inline? Senior

На JVM дженерики стираются (type erasure) — в рантайме T неизвестен, нельзя is T или T::class. Но inline-функция встраивается в место вызова, где конкретный тип известен компилятору, поэтому reified T доступен как обычный тип: можно делать is T, T::class, создавать массивы. Цена — дублирование байткода в каждом месте вызова, поэтому reified допустим только в inline-функциях.

Когда Sequence быстрее List, а когда медленнее? Middle+

Sequence вычисляется лениво и поэлементно: цепочка map.filter.first() не создаёт промежуточных коллекций и может оборваться раньше (short-circuit). Это выигрывает на длинных цепочках и больших/бесконечных источниках. На коротких цепочках и маленьких коллекциях List быстрее — у Sequence есть оверхед на итераторы и боксинг, а операции над List жадные и кэш-дружелюбные.

В чём разница == и ===, и причём тут equals? Middle

== — структурное равенство, компилируется в a?.equals(b) ?: (b === null); === — ссылочное равенство (тот же объект в куче). Для data class equals сгенерирован по полям первичного конструктора, поэтому == сравнивает значения. Контракт: если a == b, то a.hashCode() == b.hashCode() — иначе сломается работа в HashMap/HashSet.
