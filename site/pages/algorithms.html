Алгоритмы и структуры данных

Big-O (асимптотическая оценка сложности), внутреннее устройство коллекций ArrayList (динамический массив), HashMap (хэш-таблица), TreeMap (отсортированная карта), потокобезопасные структуры и классические алгоритмы — бинарный поиск, two pointers (два указателя), DFS/BFS (поиск в глубину/поиск в ширину) и DP (dynamic programming, динамическое программирование). На собесе ждут не зубрёжку сложностей, а понимание trade-offs (компромиссов) и реализацию LRU-кэша (least recently used, вытеснение давно не использованных данных) «вживую».

Middle Senior Частый вопрос

Big-O: время, память, амортизация

Big-O описывает асимптотику худшего случая — как растёт число операций при увеличении входа n, отбрасывая константы и младшие члены. O(2n + 5) — это O(n). Кроме времени всегда оценивай память (auxiliary space): рекурсия с глубиной n — это O(n) на стеке, даже если «лишних» массивов нет.

Амортизированная сложность

Амортизация — это усреднённая стоимость операции по длинной серии вызовов. Классика — ArrayList.add(): обычно O(1), но изредка случается дорогое расширение массива O(n). Если ёмкость удваивается, суммарная стоимость n вставок — O(n), значит на одну вставку приходится амортизированные O(1). Важно не путать с «средним случаем»: амортизация — это гарантия по серии, а не вероятностное ожидание.

⚠️
Ловушка на собесе. list.contains(x) внутри цикла по той же коллекции — это скрытый O(n²). На 10 000 элементов это 100 млн операций на главном потоке → ANR. Решение: заменить List на HashSet для проверки вхождения и получить O(n).

ArrayList и LinkedList: что под капотом

ArrayList — динамический массив

ArrayList хранит элементы в обычном массиве Object[]. Доступ по индексу — O(1) (арифметика смещения, отличная локальность кэша). Главный механизм — рост ёмкости: когда массив заполнен, создаётся новый размером oldCapacity + (oldCapacity >> 1) — то есть в 1.5 раза (в Java; в Kotlin под капотом тот же java.util.ArrayList), и все элементы копируются через Arrays.copyOf. Это и даёт амортизированные O(1) на add() в конец, но O(n) на вставку/удаление в середину (нужен сдвиг хвоста).

CapacityDemo.kt
// Если знаешь итоговый размер — задай ёмкость сразу, избежишь N расширений
val list = ArrayList<Int>(10_000)        // не будет рекопирований
repeat(10_000) { list.add(it) }

// add в конец — амортизированный O(1)
list.add(42)
// вставка в середину — O(n): сдвигаются все элементы правее
list.add(0, -1)                          // дорого: сдвиг всего массива

LinkedList — двусвязный список

LinkedList — узлы Node, каждый хранит value, prev и next. Вставка/удаление по уже имеющемуся итератору — O(1) (перецепил две ссылки). Но доступ по индексу — O(n): нужно идти от головы или хвоста. На практике в Android LinkedList почти всегда проигрывает ArrayList: ссылки разбросаны по куче, кэш-промахи убивают производительность, а память больше (два указателя на узел). Для очереди/дека бери ArrayDeque, а не LinkedList.

LinkedList: каждый узел знает соседей слева и справа
💡
Ответ на «когда LinkedList?». Честный ответ Senior-уровня: «почти никогда». Единственный реальный кейс — частые вставки/удаления в начало или середину при последовательном обходе итератором. Во всех остальных случаях ArrayList быстрее за счёт локальности данных.

HashMap: бакеты, коллизии, treeify

Это «вопрос-маркер»: по тому, как кандидат рассказывает про HashMap, видно глубину. Внутри — массив бакетов Node[] table (по умолчанию 16). Индекс бакета вычисляется как (n - 1) & hash, где hash — это перемешанный хеш ключа: h = key.hashCode(); h ^ (h >>> 16) (старшие биты подмешиваются к младшим, чтобы плохие hashCode() не валились в один бакет).

Коллизии: chaining → treeify

Если два ключа попали в один бакет (коллизия), они складываются в связный список (chaining). Поиск в бакете идёт перебором с проверкой equals. С Java 8 добавлена оптимизация: если в одном бакете накопилось > 8 элементов (TREEIFY_THRESHOLD) и ёмкость таблицы ≥ 64 (MIN_TREEIFY_CAPACITY), список превращается в красно-чёрное дерево — поиск в бакете становится O(log n) вместо O(n). При уменьшении до < 6 (UNTREEIFY_THRESHOLD) дерево схлопывается обратно в список.

Load factor и рехеш

Load factor по умолчанию 0.75. Когда число элементов превышает capacity * loadFactor (для 16 это 12), запускается resize: ёмкость удваивается, создаётся новая таблица и все записи перераспределяются (rehash) — операция O(n). Поэтому при известном размере задавай начальную ёмкость: HashMap(expectedSize / 0.75 + 1), чтобы избежать серии resize.

Контракт hashCode / equals

🛑
Контракт, который ломают чаще всего. Если a.equals(b), то обязательно a.hashCode() == b.hashCode(). Нарушишь — объект «потеряется» в HashMap: положили по одному хешу, ищем по другому. Обратное не требуется (равные хеши у неравных объектов = допустимая коллизия). И ещё: ключ должен быть иммутабельным по полям, входящим в hashCode — иначе после мутации он окажется «не в том бакете».

LinkedHashMap и TreeMap

LinkedHashMap — хеш + порядок

LinkedHashMap наследует HashMap, но дополнительно связывает записи двусвязным списком, сохраняя порядок. По умолчанию — порядок вставки (insertion-order). Если создать с accessOrder = true, записи переупорядочиваются при каждом get/put — самый «свежий» доступ уходит в хвост. Это даёт встроенный механизм LRU: переопредели removeEldestEntry(), и самый давний по доступу элемент будет вытесняться.

LruByLinkedHashMap.kt
// LRU «из коробки»: accessOrder=true + removeEldestEntry
fun <K, V> lruCache(capacity: Int): LinkedHashMap<K, V> =
    object : LinkedHashMap<K, V>(capacity, 0.75f, /* accessOrder = */ true) {
        override fun removeEldestEntry(eldest: Map.Entry<K, V>): Boolean =
            size > capacity     // как только переполнились — выкидываем самый старый
    }

TreeMap — красно-чёрное дерево

TreeMap хранит ключи в самобалансирующемся красно-чёрном дереве, всегда отсортированными по Comparable/Comparator. Все основные операции — O(log n) (медленнее HashMap), но взамен ты получаешь упорядоченный обход и навигационные методы: firstKey(), ceilingKey(), floorKey(), headMap(), subMap(). Бери его, когда нужны диапазонные запросы или «ближайший ключ ≥ x».

Потокобезопасные структуры

Обычный HashMap и ArrayList не потокобезопасны: гонка при resize может зациклить или повредить структуру. Для конкурентного доступа есть специальные классы — и Senior должен понимать как они синхронизируются, а не только их имена.

ConcurrentHashMap — без глобального лока

До Java 8 использовалась сегментация (lock-striping): таблица делилась на сегменты, каждый со своим локом — параллельно работало столько потоков, сколько сегментов. С Java 8 сегменты убрали: теперь синхронизация идёт на уровне отдельного бакета. Вставка в пустой бакет делается одним CAS (compare-and-swap) без блокировки; при коллизии поток берёт synchronized на первом узле этого бакета — лок затрагивает один бакет, а не всю таблицу. Чтения почти всегда lock-free (идут по volatile-ссылкам). Глобального лока нет ни при чтении, ни при записи — отсюда высокая пропускная способность.

ConcurrentHashMap (Java 8+): CAS для пустого бакета, узловой lock при коллизии, чтение без локов

CopyOnWriteArrayList — снапшот при записи

CopyOnWriteArrayList при каждой модификации копирует весь внутренний массив целиком, а затем подменяет ссылку. Чтения идут по неизменяемому снапшоту вообще без синхронизации и без ConcurrentModificationException. Цена — запись O(n) и всплеск памяти. Уместен там, где читают часто, а пишут редко: списки слушателей/observer'ов, конфиги. Для частых записей — категорически нет.

Collections.synchronizedList и BlockingQueue

Collections.synchronizedList(list) оборачивает любой список и синхронизирует каждый метод на одном мониторе — то есть это один глобальный лок на всю коллекцию (проще, но не масштабируется; плюс итерацию надо вручную оборачивать в synchronized). BlockingQueue (например, ArrayBlockingQueue, LinkedBlockingQueue) — основа паттерна producer-consumer: put() блокируется, если очередь полна, take() — если пуста.

Типовые алгоритмы для интервью

Бинарный поиск — O(log n)

Работает только на отсортированном массиве: на каждом шаге отбрасываем половину диапазона. Классическая ловушка — переполнение при (lo + hi) / 2; пиши lo + (hi - lo) / 2.

BinarySearch.kt
fun binarySearch(a: IntArray, target: Int): Int {
    var lo = 0; var hi = a.size - 1
    while (lo <= hi) {
        val mid = lo + (hi - lo) / 2          // защита от переполнения
        when {
            a[mid] == target -> return mid
            a[mid] < target  -> lo = mid + 1
            else             -> hi = mid - 1
        }
    }
    return -1
}

Two pointers и sliding window

Two pointers — два указателя идут навстречу или в одну сторону, сводя O(n²) к O(n) (поиск пары с заданной суммой в сортированном массиве). Sliding window — частный случай: «окно» из двух указателей расширяется/сужается для задач на подотрезки (самая длинная подстрока без повторов, максимум суммы окна размера k).

SlidingWindow.kt
// Максимальная сумма подмассива длины k — окно сдвигается за O(n)
fun maxSum(a: IntArray, k: Int): Int {
    var windowSum = a.take(k).sum()
    var best = windowSum
    for (i in k until a.size) {
        windowSum += a[i] - a[i - k]          // добавили правый, убрали левый
        best = maxOf(best, windowSum)
    }
    return best
}

Стек, очередь, деревья, графы

Стек (LIFO) — проверка скобок, обход в глубину без рекурсии, undo. Очередь (FIFO) — обход в ширину, планировщики. Для обоих в Kotlin/JVM бери ArrayDeque. Обходы дерева: DFS (in/pre/post-order, через рекурсию или стек) и BFS (по уровням, через очередь).

TreeTraversal.kt
class TreeNode(val v: Int, val left: TreeNode? = null, val right: TreeNode? = null)

// BFS — обход по уровням через очередь, O(n)
fun bfs(root: TreeNode?): List<Int> {
    val res = mutableListOf<Int>()
    val queue = ArrayDeque<TreeNode>()
    if (root != null) queue.add(root)
    while (queue.isNotEmpty()) {
        val node = queue.removeFirst()
        res += node.v
        node.left?.let { queue.add(it) }
        node.right?.let { queue.add(it) }
    }
    return res
}

В графах те же DFS/BFS плюс топологическая сортировка (порядок вершин DAG, где зависимость идёт раньше зависимого — как порядок сборки модулей). BFS на невзвешенном графе даёт кратчайший путь по числу рёбер.

Рекурсия и динамическое программирование

DP применяют, когда у задачи есть оптимальная подструктура и перекрывающиеся подзадачи. Наивная рекурсия Фибоначчи — O(2ⁿ) из-за повторного пересчёта; мемоизация (кэш результатов, top-down) или табуляция (bottom-up) сводят её к O(n).

Fibonacci.kt
// O(2^n) — наивно, экспоненциальный взрыв вызовов
fun fibSlow(n: Int): Long = if (n < 2) n.toLong() else fibSlow(n - 1) + fibSlow(n - 2)

// O(n) время, O(1) память — табуляция bottom-up
fun fibFast(n: Int): Long {
    if (n < 2) return n.toLong()
    var a = 0L; var b = 1L
    repeat(n - 1) { val next = a + b; a = b; b = next }
    return b
}

Реализация LRU-кэша на Kotlin

«Напиши LRU-кэш» — один из самых частых live-coding вопросов. Ключевая идея: чтобы и get, и put были O(1), нужна связка HashMap (доступ по ключу за O(1)) + двусвязный список (порядок использования, перемещение узла в начало за O(1)). Самый недавно использованный — в голове, кандидат на вытеснение — в хвосте.

LruCache.kt
class LruCache<K, V>(private val capacity: Int) {

    private class Node<K, V>(val key: K, var value: V) {
        var prev: Node<K, V>? = null
        var next: Node<K, V>? = null
    }

    private val map = HashMap<K, Node<K, V>>()
    private val head = Node<K, V>(/* dummy */ null as K, null as V)  // фиктивные
    private val tail = Node<K, V>(null as K, null as V)             // границы

    init { head.next = tail; tail.prev = head }

    fun get(key: K): V? {
        val node = map[key] ?: return null
        moveToFront(node)                 // отметили как недавно использованный
        return node.value
    }

    fun put(key: K, value: V) {
        map[key]?.let { it.value = value; moveToFront(it); return }
        val node = Node(key, value)
        map[key] = node
        addFront(node)
        if (map.size > capacity) {        // переполнились — выкидываем LRU из хвоста
            val lru = tail.prev!!
            remove(lru); map.remove(lru.key)
        }
    }

    private fun addFront(n: Node<K, V>) {
        n.prev = head; n.next = head.next
        head.next!!.prev = n; head.next = n
    }
    private fun remove(n: Node<K, V>) {
        n.prev!!.next = n.next; n.next!!.prev = n.prev
    }
    private fun moveToFront(n: Node<K, V>) { remove(n); addFront(n) }
}
💡
Что сказать после реализации. Упомяни, что в Android уже есть готовый androidx.collection.LruCache (и DiskLruCache), а «ручной» вариант с LinkedHashMap(accessOrder = true) короче (см. раздел выше). Понимание ручной реализации показывает, что ты знаешь почему это O(1), а не просто дёргаешь библиотеку.

Частые вопросы интервьюера

Что такое амортизированная сложность? Пример. Middle

Это усреднённая стоимость операции по длинной серии вызовов, а не по одному вызову. Пример — ArrayList.add() в конец: обычно O(1), но при заполнении массива происходит расширение и копирование за O(n). Поскольку ёмкость растёт мультипликативно (в 1.5 раза), суммарная стоимость n вставок равна O(n), то есть на одну вставку — амортизированные O(1). В отличие от «среднего случая», это гарантия по серии, а не вероятностная оценка.

Как устроен HashMap и что изменилось в Java 8? Middle+

Массив бакетов Node[] (по умолчанию 16). Индекс = (n-1) & hash, где хеш дополнительно перемешивается (h ^ (h >>> 16)). Коллизии складываются в связный список (chaining). В Java 8 добавлен treeify: если в бакете больше 8 элементов и ёмкость ≥ 64, список превращается в красно-чёрное дерево, и поиск в бакете становится O(log n) вместо O(n) — защита от деградации при плохом hashCode или hash-flooding атаках. Load factor 0.75: при заполнении выше порога таблица удваивается с полным rehash.

Какой контракт hashCode и equals? Что будет при нарушении? Middle

Правила: (1) если объекты равны по equals, их hashCode обязаны совпадать; (2) hashCode стабилен, пока объект не меняется; (3) неравные объекты могут иметь равный хеш (коллизия — норма). Если нарушить первое правило, объект «теряется» в HashMap: положили в один бакет, ищем в другом. Отдельная ловушка — мутабельный ключ: после изменения поля, входящего в хеш, он оказывается в «неправильном» бакете. Поэтому ключи делают иммутабельными (часто data class).

Чем ConcurrentHashMap лучше synchronizedMap? Senior

synchronizedMap — это один глобальный лок на всю коллекцию: потоки сериализуются, throughput не масштабируется. ConcurrentHashMap (Java 8+) синхронизируется на уровне отдельного бакета: вставка в пустой бакет — атомарный CAS без блокировки, при коллизии — synchronized только на первом узле этого бакета. Чтения почти всегда lock-free (по volatile-ссылкам). Глобального лока нет, поэтому много потоков работают параллельно. Нюанс: атомарны лишь отдельные операции — для «check-then-act» нужны computeIfAbsent/putIfAbsent; null-ключи и null-значения запрещены.

Когда CopyOnWriteArrayList, а когда synchronizedList? Senior

CopyOnWriteArrayList копирует весь массив при каждой записи (O(n)), зато чтения и итерация — без локов и без ConcurrentModificationException. Идеален для сценария «много чтений, редкие записи»: списки слушателей, observer'ы, конфиги. synchronizedList — простая обёртка с одним локом на все операции; уместна при невысокой нагрузке, но итерацию надо вручную оборачивать в synchronized(list). Если записей много — оба плохи, лучше пересмотреть структуру (например, очередь).

ArrayList или LinkedList — что выбрать? Middle

В подавляющем большинстве случаев — ArrayList: доступ по индексу O(1) и отличная локальность кэша (данные лежат подряд). LinkedList теоретически даёт O(1) вставку/удаление, но только по уже имеющемуся итератору, а доступ по индексу — O(n); на практике кэш-промахи и накладные расходы на узлы делают его медленнее почти всегда. Для очереди или дека бери ArrayDeque, а не LinkedList.

Как реализовать LRU-кэш за O(1)? Middle+

Связка HashMap + двусвязный список. HashMap даёт доступ к узлу по ключу за O(1), двусвязный список хранит порядок использования: при get/put узел перемещается в голову за O(1) (перецепка ссылок). При переполнении удаляется узел из хвоста (наименее недавно использованный) и его ключ из мапы. Фиктивные head/tail-узлы упрощают перецепку без проверок на null. Короткая альтернатива — LinkedHashMap(accessOrder = true) с переопределённым removeEldestEntry; в Android есть готовый androidx.collection.LruCache.

Когда нужен TreeMap вместо HashMap? Middle+

Когда нужен порядок ключей или диапазонные запросы. TreeMap на красно-чёрном дереве держит ключи отсортированными и даёт навигацию: firstKey, ceilingKey/floorKey («ближайший ≥/≤ x»), subMap/headMap/tailMap, упорядоченный обход. Цена — все операции O(log n) вместо O(1). Если порядок не нужен — HashMap быстрее. Если нужен порядок вставки/доступа (не сортировка) — LinkedHashMap.

В чём разница между DFS и BFS, и где что применять? Middle

DFS (в глубину) идёт максимально далеко по ветке, потом откатывается; реализуется рекурсией или стеком, память O(h) (высота). Подходит для проверки связности, поиска циклов, топосорта, обходов pre/in/post-order. BFS (в ширину) обходит по уровням через очередь, память O(w) (ширина); на невзвешенном графе даёт кратчайший путь по числу рёбер, используется для поиска ближайших, обхода по слоям. Оба — O(V + E) по времени.

Задачи

Задача 1. Реализуй функцию, которая находит два числа в массиве nums, дающих в сумме target. Верни их индексы. Предполагай, что ровно одно решение существует, один элемент не использовать дважды. Сложность — лучше O(n²).

Показать решение 
fun twoSum(nums: IntArray, target: Int): IntArray {
    val seen = HashMap<Int, Int>()   // значение → индекс
    for ((i, num) in nums.withIndex()) {
        val complement = target - num
        seen[complement]?.let { j -> return intArrayOf(j, i) }
        seen[num] = i
    }
    error("No solution")
}

Объяснение: за один проход кладём в HashMap каждое число и его индекс. Для текущего элемента ищем дополнение target - num — если оно уже в мапе, нашли пару. Сложность: время O(n), память O(n). Наивный двойной цикл даёт O(n²) — на больших массивах неприемлемо.

Задача 2. Дан список целых чисел. Реализуй функцию maxSlidingWindow(a: IntArray, k: Int): IntArray, которая возвращает массив максимумов для каждого скользящего окна размера k. Вход: [1, 3, -1, -3, 5, 3, 6, 7], k = 3; ожидаемый выход: [3, 3, 5, 5, 6, 7]. Сложность — лучше O(n·k).

Показать решение 
fun maxSlidingWindow(a: IntArray, k: Int): IntArray {
    val result = IntArray(a.size - k + 1)
    val deque = ArrayDeque<Int>()   // хранит индексы, по убыванию значений

    for (i in a.indices) {
        // выкидываем индексы, вышедшие за левую границу окна
        if (deque.isNotEmpty() && deque.first() < i - k + 1) {
            deque.removeFirst()
        }
        // убираем с хвоста индексы, чьё значение меньше текущего
        while (deque.isNotEmpty() && a[deque.last()] < a[i]) {
            deque.removeLast()
        }
        deque.addLast(i)
        if (i >= k - 1) {
            result[i - k + 1] = a[deque.first()]
        }
    }
    return result
}

Объяснение: монотонная убывающая дека хранит индексы кандидатов на максимум. Голова деки — всегда максимум текущего окна. При сдвиге окна вправо выбрасываем устаревшие индексы из головы и меньшие элементы с хвоста. Сложность: O(n) время (каждый индекс добавляется и удаляется не более одного раза), O(k) память.

Задача 3. Реализуй потокобезопасный LRU-кэш с ёмкостью capacity. Методы get(key) и put(key, value) должны работать за O(1) и безопасно вызываться из нескольких потоков одновременно. Покажи минимальный рабочий вариант на Kotlin.

Показать решение 
import java.util.concurrent.locks.ReentrantReadWriteLock
import kotlin.concurrent.write
import kotlin.concurrent.read

class ThreadSafeLruCache<K, V>(capacity: Int) {

    private val lock = ReentrantReadWriteLock()
    private val cache: LinkedHashMap<K, V> =
        object : LinkedHashMap<K, V>(capacity, 0.75f, true) {
            override fun removeEldestEntry(eldest: Map.Entry<K, V>): Boolean =
                size > capacity
        }

    fun get(key: K): V? = lock.read { cache[key] }

    fun put(key: K, value: V): Unit = lock.write { cache[key] = value }
}

Объяснение: LinkedHashMap(accessOrder = true) с переопределённым removeEldestEntry даёт LRU-логику. Для потокобезопасности — ReentrantReadWriteLock: несколько потоков читают одновременно (read-lock), запись блокирует всех. Альтернатива — оборачивать каждый метод в synchronized, но тогда параллельные чтения тоже сериализуются. Сложность операций: амортизированный O(1) без учёта блокировки.
