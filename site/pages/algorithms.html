Алгоритмы и структуры данных

Big-O (асимптотическая оценка сложности), внутреннее устройство коллекций ArrayList (динамический массив), HashMap (хэш-таблица), TreeMap (отсортированная карта), потокобезопасные структуры и классические алгоритмы — бинарный поиск, two pointers (два указателя), DFS/BFS (поиск в глубину/поиск в ширину) и DP (dynamic programming, динамическое программирование). На собесе ждут не зубрёжку сложностей, а понимание trade-offs (компромиссов) и реализацию LRU-кэша (least recently used, вытеснение давно не использованных данных) «вживую».

Middle Senior Частый вопрос

Big-O: время, память, амортизация

Big-O описывает асимптотику худшего случая — как растёт число операций при увеличении входа n, отбрасывая константы и младшие члены. O(2n + 5) — это O(n). Кроме времени всегда оценивай память (auxiliary space): рекурсия с глубиной n — это O(n) на стеке, даже если «лишних» массивов нет.

Амортизированная сложность

Амортизация — это усреднённая стоимость операции по длинной серии вызовов. Классика — ArrayList.add(): обычно O(1), но изредка случается дорогое расширение массива O(n). Если ёмкость удваивается, суммарная стоимость n вставок — O(n), значит на одну вставку приходится амортизированные O(1). Важно не путать с «средним случаем»: амортизация — это гарантия по серии, а не вероятностное ожидание.

Ловушка на собесе. list.contains(x) внутри цикла по той же коллекции — это скрытый O(n²). На 10 000 элементов это 100 млн операций на главном потоке → ANR. Решение: заменить List на HashSet для проверки вхождения и получить O(n).
Почему амортизированный O(1) у ArrayList.add держится только при росте ёмкости в разы, а при росте на константу (+10) деградирует до O(n)? ★

Соберём суммарную работу копирования за n вставок и поделим на n — это и есть амортизация методом агрегации. При росте в k раз (k>1) расширения случаются всё реже, а их стоимость складывается в геометрическую прогрессию n + n/k + n/k² + …, которая сходится к константе на n. Итого суммарно O(n), на вставку — O(1).

А теперь растим на фиксированную добавку c. Расширение происходит каждые c вставок, и копировать приходится всё больше: c + 2c + 3c + … — арифметическая прогрессия, то есть ~O(n²/c) суммарно, или O(n) на одну вставку. Вот почему java.util.ArrayList растёт в 1.5 раза (oldCapacity + (oldCapacity >> 1)), а не на десяток ячеек: стоимость очередного копирования обязана быть пропорциональна уже накопленной работе, чтобы «размазаться» по серии, а не догонять её.

В таблице у HashMap стоит средний O(1) — но почему даже без коллизий это не настоящий O(1), и от чего реально зависит константа? ★

«Средний O(1)» опирается на гипотезу простого равномерного хеширования: ключи раскладываются по бакетам ровно, средняя длина цепочки n/capacity (это и есть load factor) держится около константы. Слово «средний» тут несёт всю нагрузку — это ожидание при допущениях, а не гарантия худшего случая.

Что ломает эти допущения на практике: плохой hashCode сваливает всё в один бакет — и ты получаешь O(n) (или O(log n) после treeify). Сам вызов hashCode и equals для длинного ключа уже O(длины ключа), а не O(1) — для строки на сотню символов это заметно. Плюс амортизированная цена rehash при росте. Поэтому строгая формулировка — amortized expected O(1) при равномерном хешировании. Подвох на собесе именно в этом «среднем»: злонамеренный или просто кривой hashCode превращает мапу в список.

ArrayList и LinkedList: что под капотом

ArrayList — динамический массив

ArrayList хранит элементы в обычном массиве Object[]. Доступ по индексу — O(1) (арифметика смещения, отличная локальность кэша). Главный механизм — рост ёмкости: когда массив заполнен, создаётся новый размером oldCapacity + (oldCapacity >> 1) — то есть в 1.5 раза (в Java; в Kotlin под капотом тот же java.util.ArrayList), и все элементы копируются через Arrays.copyOf. Это и даёт амортизированные O(1) на add() в конец, но O(n) на вставку/удаление в середину (нужен сдвиг хвоста).

CapacityDemo.kt
// Если знаешь итоговый размер — задай ёмкость сразу, избежишь N расширений
val list = ArrayList<Int>(10_000)        // не будет рекопирований
repeat(10_000) { list.add(it) }

// add в конец — амортизированный O(1)
list.add(42)
// вставка в середину — O(n): сдвигаются все элементы правее
list.add(0, -1)                          // дорого: сдвиг всего массива

LinkedList — двусвязный список

LinkedList — узлы Node, каждый хранит value, prev и next. Вставка/удаление по уже имеющемуся итератору — O(1) (перецепил две ссылки). Но доступ по индексу — O(n): нужно идти от головы или хвоста. На практике в Android LinkedList почти всегда проигрывает ArrayList: ссылки разбросаны по куче, кэш-промахи убивают производительность, а память больше (два указателя на узел). Для очереди/дека бери ArrayDeque, а не LinkedList.

Big-O полного обхода у обоих — O(n), но на практике ArrayList уходит вперёд: процессор тянет соседние элементы одной cache line, тогда как у LinkedList каждый next — прыжок в случайное место кучи.

Почему удаление по итератору в LinkedList — это O(1), а remove(index) и remove(value) — O(n), хотя структура одна и та же? ★

Сама перецепка ссылок (prev.next = next; next.prev = prev) и правда O(1) — но только когда ссылка на нужный узел уже у тебя в руках. У итератора она есть: он стоит на узле, который удаляет, поиск не нужен.

А remove(index) сначала тратит O(n) на проход до узла (LinkedList, правда, идёт с ближнего конца — смотрит, index < size/2 или нет, но это всё равно O(n)). remove(value) — O(n) на линейный поиск по equals. Тот же нюанс с add(index, e): по итератору O(1), по индексу O(n). Классический подвох: кандидат говорит «LinkedList — O(1) на удаление», а интервьюер уточняет «по индексу или по итератору?» — и без второго ответа фраза неверна.

HashMap: бакеты, коллизии, treeify

Это «вопрос-маркер»: по тому, как кандидат рассказывает про HashMap, видно глубину. Внутри — массив бакетов Node[] table (по умолчанию 16). Индекс бакета вычисляется как (n - 1) & hash, где hash — это перемешанный хеш ключа: h = key.hashCode(); h ^ (h >>> 16) (старшие биты подмешиваются к младшим, чтобы плохие hashCode() не валились в один бакет).

Коллизии: chaining → treeify

Если два ключа попали в один бакет (коллизия), они складываются в связный список (chaining). Поиск в бакете идёт перебором с проверкой equals. С Java 8 добавлена оптимизация: если в одном бакете накопилось > 8 элементов (TREEIFY_THRESHOLD) и ёмкость таблицы ≥ 64 (MIN_TREEIFY_CAPACITY), список превращается в красно-чёрное дерево — поиск в бакете становится O(log n) вместо O(n). При уменьшении до < 6 (UNTREEIFY_THRESHOLD) дерево схлопывается обратно в список.

Load factor и рехеш

Load factor по умолчанию 0.75. Когда число элементов превышает capacity * loadFactor (для 16 это 12), запускается resize: ёмкость удваивается, создаётся новая таблица и все записи перераспределяются (rehash) — операция O(n). Поэтому при известном размере задавай начальную ёмкость: HashMap(expectedSize / 0.75 + 1), чтобы избежать серии resize.

Контракт hashCode / equals

🛑
Контракт, который ломают чаще всего. Если a.equals(b), то обязательно a.hashCode() == b.hashCode(). Нарушишь — объект «потеряется» в HashMap: положили по одному хешу, ищем по другому. Обратное не требуется (равные хеши у неравных объектов = допустимая коллизия). И ещё: ключ должен быть иммутабельным по полям, входящим в hashCode — иначе после мутации он окажется «не в том бакете».

val map = HashMap<UserKey, String>()
map[UserKey(1, "RU")] = "Алиса"
println(map[UserKey(1, "RU")])   // "Алиса" — равные ключи, равные хеши

// ❌ антипаттерн: мутабельный ключ
class BadKey(var id: Int)        // нет equals/hashCode + поле меняется
// после badKey.id = 99 объект «теряется» в map
Зачем порог treeify (8) и порог untreeify (6) разные, а не оба, скажем, 7? И почему точка превращения в дерево — именно 8? ★

Зазор между TREEIFY_THRESHOLD = 8 и UNTREEIFY_THRESHOLD = 6 — это гистерезис. Будь они равны, цепочка, колеблющаяся около порога при чередующихся put/remove, дрожала бы между списком и деревом бесконечно, и каждое преобразование (а оно дорогое) било бы по производительности. Разница в два элемента гасит этот дребезг.

Само число 8 выбрано из распределения Пуассона: при хорошем хешировании и load factor 0.75 вероятность, что в один бакет упадёт восемь элементов, порядка 6·10⁻⁸ — практически невозможна без атаки или сломанного hashCode. То есть treeify задуман не как штатный режим, а как защита от деградации (hash-flooding), и порог поставлен так высоко, что в нормальной работе дерево почти не встречается.

LinkedHashMap и TreeMap

LinkedHashMap — хеш + порядок

LinkedHashMap наследует HashMap, но дополнительно связывает записи двусвязным списком, сохраняя порядок. По умолчанию — порядок вставки (insertion-order). Если создать с accessOrder = true, записи переупорядочиваются при каждом get/put — самый «свежий» доступ уходит в хвост. Это даёт встроенный механизм LRU: переопредели removeEldestEntry(), и самый давний по доступу элемент будет вытесняться.

LruByLinkedHashMap.kt
// LRU «из коробки»: accessOrder=true + removeEldestEntry
fun <K, V> lruCache(capacity: Int): LinkedHashMap<K, V> =
    object : LinkedHashMap<K, V>(capacity, 0.75f, /* accessOrder = */ true) {
        override fun removeEldestEntry(eldest: Map.Entry<K, V>): Boolean =
            size > capacity     // как только переполнились — выкидываем самый старый
    }

TreeMap — красно-чёрное дерево

TreeMap хранит ключи в самобалансирующемся красно-чёрном дереве, всегда отсортированными по Comparable/Comparator. Все основные операции — O(log n) (медленнее HashMap), но взамен ты получаешь упорядоченный обход и навигационные методы: firstKey(), ceilingKey(), floorKey(), headMap(), subMap(). Бери его, когда нужны диапазонные запросы или «ближайший ключ ≥ x».

TreeMap сравнивает ключи через compareTo/Comparator и игнорирует equals. К какому коварному багу это приводит, если compareTo несогласован с equals? ★

TreeMap считает два ключа одним и тем же, если compareTo вернул 0 — и плевать, что говорит equals. Равенство определяется деревом поиска, а не контрактом объекта.

Классический капкан — BigDecimal: BigDecimal("1.0").compareTo(BigDecimal("1.00")) == 0, но equals у них false. В обычном HashMap это два разных ключа, а в TreeMap — один: put перезапишет «другой по equals» ключ, а get/containsKey могут не найти то, что в HashMap прекрасно лежит. Документация прямо предупреждает: такой TreeMap нарушает общий контракт Map, который определён в терминах equals. Вывод: Comparator для ключей TreeMap обязан быть consistent with equals, иначе карта тихо схлопывает или теряет записи.

Потокобезопасные структуры

Обычный HashMap и ArrayList не потокобезопасны: гонка при resize может зациклить или повредить структуру. Для конкурентного доступа есть специальные классы — и Senior должен понимать как они синхронизируются, а не только их имена.

ConcurrentHashMap — без глобального лока

До Java 8 использовалась сегментация (lock-striping): таблица делилась на сегменты, каждый со своим локом — параллельно работало столько потоков, сколько сегментов. С Java 8 сегменты убрали: теперь синхронизация идёт на уровне отдельного бакета. Вставка в пустой бакет делается одним CAS (compare-and-swap) без блокировки; при коллизии поток берёт synchronized на первом узле этого бакета — лок затрагивает один бакет, а не всю таблицу. Чтения почти всегда lock-free (идут по volatile-ссылкам). Глобального лока нет ни при чтении, ни при записи — отсюда высокая пропускная способность.

CopyOnWriteArrayList — снапшот при записи

CopyOnWriteArrayList при каждой модификации копирует весь внутренний массив целиком, а затем подменяет ссылку. Чтения идут по неизменяемому снапшоту вообще без синхронизации и без ConcurrentModificationException. Цена — запись O(n) и всплеск памяти. Уместен там, где читают часто, а пишут редко: списки слушателей/observer'ов, конфиги. Для частых записей — категорически нет.

Collections.synchronizedList и BlockingQueue

Collections.synchronizedList(list) оборачивает любой список и синхронизирует каждый метод на одном мониторе — то есть это один глобальный лок на всю коллекцию (проще, но не масштабируется; плюс итерацию надо вручную оборачивать в synchronized). BlockingQueue (например, ArrayBlockingQueue, LinkedBlockingQueue) — основа паттерна producer-consumer: put() блокируется, если очередь полна, take() — если пуста.

Почему итератор ConcurrentHashMap не бросает ConcurrentModificationException и зовётся weakly consistent, а итератор обычного HashMap — fail-fast? Что это значит на практике? ★

Итераторы HashMap/ArrayList — fail-fast: они держат счётчик структурных изменений modCount и на каждом next() сверяют его с ожидаемым. Любое изменение коллекции мимо самого итератора — даже из того же потока — ломает сверку и кидает ConcurrentModificationException. Это не гарантия безопасности, а ранний детектор багов; документация так и пишет — best-effort.

Итератор ConcurrentHashMap — weakly consistent: он обходит состояние на момент создания и CME не бросает никогда, но изменения, внесённые после старта итерации, может увидеть, а может и нет — формальной гарантии нет. На практике: HashMap нельзя трогать в обычном for-each (только через iterator.remove(), см. отдельный параграф ниже), а ConcurrentHashMap можно итерировать параллельно с записями и не ловить исключение — ценой того, что итерация видит «примерно текущее» состояние.

Типовые алгоритмы для интервью

Бинарный поиск — O(log n)

Работает только на отсортированном массиве: на каждом шаге отбрасываем половину диапазона. Классическая ловушка — переполнение при (lo + hi) / 2; пиши lo + (hi - lo) / 2.

BinarySearch.kt
fun binarySearch(a: IntArray, target: Int): Int {
    var lo = 0; var hi = a.size - 1
    while (lo <= hi) {
        val mid = lo + (hi - lo) / 2          // защита от переполнения
        when {
            a[mid] == target -> return mid
            a[mid] < target  -> lo = mid + 1
            else             -> hi = mid - 1
        }
    }
    return -1
}

Two pointers и sliding window

Two pointers — два указателя идут навстречу или в одну сторону, сводя O(n²) к O(n) (поиск пары с заданной суммой в сортированном массиве). Sliding window — частный случай: «окно» из двух указателей расширяется/сужается для задач на подотрезки (самая длинная подстрока без повторов, максимум суммы окна размера k).

SlidingWindow.kt
// Максимальная сумма подмассива длины k — окно сдвигается за O(n)
fun maxSum(a: IntArray, k: Int): Int {
    var windowSum = a.take(k).sum()
    var best = windowSum
    for (i in k until a.size) {
        windowSum += a[i] - a[i - k]          // добавили правый, убрали левый
        best = maxOf(best, windowSum)
    }
    return best
}

Стек, очередь, деревья, графы

Стек (LIFO) — проверка скобок, обход в глубину без рекурсии, undo. Очередь (FIFO) — обход в ширину, планировщики. Для обоих в Kotlin/JVM бери ArrayDeque. Обходы дерева: DFS (in/pre/post-order, через рекурсию или стек) и BFS (по уровням, через очередь).

TreeTraversal.kt
class TreeNode(val v: Int, val left: TreeNode? = null, val right: TreeNode? = null)

// BFS — обход по уровням через очередь, O(n)
fun bfs(root: TreeNode?): List<Int> {
    val res = mutableListOf<Int>()
    val queue = ArrayDeque<TreeNode>()
    if (root != null) queue.add(root)
    while (queue.isNotEmpty()) {
        val node = queue.removeFirst()
        res += node.v
        node.left?.let { queue.add(it) }
        node.right?.let { queue.add(it) }
    }
    return res
}

В графах те же DFS/BFS плюс топологическая сортировка (порядок вершин DAG, где зависимость идёт раньше зависимого — как порядок сборки модулей). BFS на невзвешенном графе даёт кратчайший путь по числу рёбер.

Рекурсия и динамическое программирование

DP применяют, когда у задачи есть оптимальная подструктура и перекрывающиеся подзадачи. Наивная рекурсия Фибоначчи — O(2ⁿ) из-за повторного пересчёта; мемоизация (кэш результатов, top-down) или табуляция (bottom-up) сводят её к O(n).

Fibonacci.kt
// O(2^n) — наивно, экспоненциальный взрыв вызовов
fun fibSlow(n: Int): Long = if (n < 2) n.toLong() else fibSlow(n - 1) + fibSlow(n - 2)

// O(n) время, O(1) память — табуляция bottom-up
fun fibFast(n: Int): Long {
    if (n < 2) return n.toLong()
    var a = 0L; var b = 1L
    repeat(n - 1) { val next = a + b; a = b; b = next }
    return b
}
Почему запись (lo + hi) / 2 в бинарном поиске переполняется на больших индексах, а lo + (hi - lo) / 2 эту проблему лечит, давая тот же результат? ★

lo и hi — это int. На очень большом массиве, когда оба близки к Integer.MAX_VALUE (~2.1 млрд), сумма lo + hi вылезает за int и заворачивается в отрицательное число (знаковое переполнение в Java/Kotlin не бросает исключение, а считается по модулю 2³²). Деление даёт неверный отрицательный mid — и ArrayIndexOutOfBoundsException.

Запись lo + (hi - lo) / 2 алгебраически равна (lo + hi) / 2, но промежуточное (hi - lo) — это размер диапазона, он не больше длины массива и в int помещается всегда. Переполнения нет. Знаменитый баг: по признанию Джошуа Блоха, он годами жил в java.util.Arrays.binarySearch и в учебниках. На JVM редко стреляет из-за лимита на размер массива, но как индикатор внимательности спрашивают почти всегда.

Куча и приоритетная очередь

Бинарная куча (heap) — это полное бинарное дерево, упакованное в обычный массив, с одним инвариантом: родитель не больше детей (min-heap) или не меньше (max-heap). Раз дерево полное, его не нужно хранить ссылками — позиции считаются арифметикой по индексу.

Индексная арифметика и сложности

Для узла i: дети лежат в 2i+1 и 2i+2, родитель — в (i-1)/2. peek (минимум/максимум) — корень массива, O(1). offer (вставка) кладёт элемент в конец и просеивает вверх (sift-up), poll (извлечение корня) ставит на его место последний элемент и просеивает вниз (sift-down) — обе по высоте дерева, то есть O(log n). А вот построение кучи из готового массива — не n вставок по O(log n), а O(n) через heapify снизу вверх: большинство узлов сидят у листьев, где просеивать почти некуда, и сумма работы сходится к линейной.

Min-heap: корень — минимум; для узла i дети в 2i+1 и 2i+2, родитель в (i−1)/2

PriorityQueue и паттерн top-K

В Java/Kotlin куча — это java.util.PriorityQueue: по умолчанию min-heap по natural ordering, либо по переданному Comparator. Самое частое её применение на собесе — top-K: чтобы найти k наибольших элементов, держим min-heap размера k. Как только размер перевалил за k, выкидываем минимум — в куче остаются ровно k крупнейших. Время O(n log k), память O(k) — дешевле полной сортировки за O(n log n), если k мало.

KthLargest.kt
// k-й по величине элемент через min-heap размера k — O(n log k)
fun findKthLargest(nums: IntArray, k: Int): Int {
    val heap = java.util.PriorityQueue<Int>()   // min-heap
    for (x in nums) {
        heap.offer(x)
        if (heap.size > k) heap.poll()          // выбрасываем минимум
    }
    return heap.peek()                          // на вершине — k-й по величине
}

Та же куча решает слияние k отсортированных списков (Merge K Sorted Lists): кладём в кучу головы всех списков и на каждом шаге забираем минимум за O(log k) — итого O(N log k) вместо наивного O(k·N) (см. задачу в конце главы). На куче же стоят алгоритмы Дейкстры и Прима, планировщики задач и Huffman-кодирование.

Почему построение кучи из массива (heapify) — O(n), хотя это «n элементов по O(log n) на просеивание»? ★

Интуиция «n · log n» считает по верхней границе для каждого узла, но она грубая. heapify идёт снизу вверх и просеивает вниз, а стоимость sift-down ограничена высотой узла, а не высотой всего дерева. Ключевой факт: листьев — половина узлов, и им просеивать некуда (0 работы); на уровне выше — четверть узлов с одним шагом; и так далее.

Сумма получается Σ (n / 2^(h+1)) · h по высотам h, а этот ряд сходится к константе, умноженной на n — то есть O(n). Построение кучи целиком дешевле, чем n отдельных offer (которые как раз дали бы O(n log n)): просеивание вверх при вставке дороже, потому что большинство новых элементов оказывается у листьев и поднимается через всю высоту, а просеивание вниз при heapify для тех же листьев бесплатно.

Монотонный стек

Монотонный стек — это именованный паттерн, а не структура: обычный стек, в котором мы поддерживаем элементы по возрастанию или убыванию, выталкивая всё, что нарушает порядок. Он отвечает на вопросы вида «для каждого элемента — ближайший справа/слева, который больше (меньше) него» за один проход O(n) вместо наивного O(n²).

Каноничные задачи — Next Greater Element и Daily Temperatures («через сколько дней станет теплее»). Идея одна: храним в стеке индексы ещё не «закрытых» элементов; пришёл новый — пока он больше элемента на вершине, этот верхний элемент только что нашёл свой следующий больший, снимаем его и записываем ответ.

DailyTemperatures.kt
// Для каждого дня — сколько ждать до более тёплого. O(n) через монотонный стек
fun dailyTemperatures(t: IntArray): IntArray {
    val res = IntArray(t.size)
    val stack = ArrayDeque<Int>()           // индексы, температуры по убыванию
    for (i in t.indices) {
        while (stack.isNotEmpty() && t[i] > t[stack.last()]) {
            val j = stack.removeLast()      // день j наконец дождался тепла
            res[j] = i - j
        }
        stack.addLast(i)
    }
    return res                              // не дождавшиеся остаются 0
}

Каждый индекс кладётся и снимается со стека не более одного раза — отсюда O(n), несмотря на вложенный while. Это тот же приём, что в задаче «Sliding Window Maximum» из блока задач выше: там монотонная дека (двусторонний вариант) держит индексы кандидатов на максимум окна. Узнаёшь паттерн «ближайший больший/меньший» или «максимум на отрезке за один проход» — почти наверняка это монотонный стек или дека.

Почему монотонный стек даёт O(n), хотя внутри основного цикла стоит ещё один while, который вроде бы добавляет вложенность? ★

Считать сложность по «цикл в цикле = O(n²)» здесь нельзя — это ловушка поверхностного анализа. Нужен амортизированный учёт по элементам, а не по итерациям внутреннего цикла.

Каждый индекс попадает на стек ровно один раз (внешний цикл, один addLast на шаг) и снимается с него тоже не более одного раза (внутренний while, removeLast). Снятый элемент назад не возвращается. Значит суммарное число операций removeLast по всему алгоритму ограничено n — независимо от того, что на отдельном шаге внутренний while может крутануться несколько раз. Итого 2n операций со стеком, то есть O(n). Это та же логика, что у амортизированного O(1) у ArrayList.add: дорогие шаги редки и «оплачены» дешёвыми.

Коллекции Kotlin: read-only, Sequence, ArrayDeque

Глава пока шла по java.util, но на Android-собесе любят именно Kotlin-специфику поверх тех же JVM-классов. Здесь три самых частых вопроса.

Read-only ≠ immutable

List в Kotlin — это read-only интерфейс: у него нет add/remove. Но это не гарантия неизменности самого объекта. listOf(…) под капотом возвращает реальный java.util.ArrayList (или Arrays.asList), и если на тот же объект есть ссылка как на MutableList — хоть из Kotlin, хоть из Java-кода — его спокойно мутируют, а read-only ссылка этого не запретит.

ReadOnlyTrap.kt
val mutable = mutableListOf(1, 2, 3)
val readOnly: List<Int> = mutable   // та же ссылка, только read-only тип
mutable.add(4)                       // объект изменился…
println(readOnly)                    // [1, 2, 3, 4] — read-only «не уберёг»

Для настоящей неизменности нужна persistent-коллекция (kotlinx.collections.immutable) или защитная копия (readOnly.toList()). На собесе это любимая ловушка: «можно ли изменить List?» — через этот интерфейс нет, но базовый объект остаётся мутабельным.

Sequence против Iterable

Операции над List/Iterable — eager: каждый map/filter создаёт новый промежуточный список и проходит коллекцию целиком на каждом шаге. Sequence ленив: элементы текут по всей цепочке по одному, без промежуточных списков, а вычисление стартует только терминальной операцией (toList, first, sum). Выигрыш — на длинных цепочках над большими коллекциями и при коротком замыкании (find, first, take обрывают обход рано). На маленьких коллекциях Sequence проигрывает: ленивость добавляет накладные расходы, а экономия на промежуточных списках ничтожна. Правило: большой объём + много операций + ранний выход → Sequence; иначе обычная коллекция.

ArrayDeque и ArrayDeque

В Kotlin stdlib (с 1.4) есть свой kotlin.collections.ArrayDeque — кольцевой буфер на массиве, удобный и как стек (addLast/removeLast), и как очередь (addLast/removeFirst), с амортизированным O(1) с обоих концов. В отличие от java.util.ArrayDeque, он допускает null. Для LIFO/FIFO бери именно дек, а не устаревший Stack или LinkedList (почему — см. вопрос в FAQ).

Цепочка asSequence().map{}.filter{}.first() и та же цепочка без asSequence() на списке из миллиона элементов — в чём разница по числу проходов и аллокаций? ★

Без asSequence() всё eager: map пройдёт весь миллион и создаст новый список на миллион, filter пройдёт его снова и создаст ещё один список, и только потом first возьмёт нулевой элемент. Два полных прохода и две аллокации списков на миллион — притом что нужен был один-единственный элемент.

С asSequence() цепочка ленивая и работает «по вертикали»: первый элемент проходит map, затем filter; если предикат прошёл, first сразу возвращает результат — и обход обрывается. В лучшем случае это один элемент через всю цепочку, ноль промежуточных списков. Вот почему Sequence особенно выигрывает там, где терминальная операция способна замкнуться рано (first, find, take): eager-версия всё равно материализует промежуточные коллекции целиком, ленивая — нет.

Итераторы: fail-fast, ConcurrentModificationException

«Как удалить элемент из коллекции прямо в цикле?» — вопрос, который реально стреляет в проде падением с ConcurrentModificationException (CME). Механизм за ним — тот же modCount, что упоминался в параграфе про потокобезопасность, но это отдельная история, и спрашивают её часто.

Что такое fail-fast

Итераторы ArrayList/HashMap — fail-fast: при создании итератор запоминает текущий modCount коллекции, и на каждом next() сверяет его. Любое структурное изменение мимо самого итератора (даже из того же потока, даже однопоточно) ломает сверку — и летит CME. Это не защита от гонок, а best-effort детектор багов: документация прямо говорит не полагаться на него как на гарантию.

Как удалять при обходе правильно

Обычный for (x in list) { list.remove(x) } по ArrayList бросит CME на следующем next(). Корректные способы:

SafeRemoval.kt
val list = mutableListOf(1, 2, 3, 4, 5)

// 1) Явный итератор + iterator.remove() — единственный способ менять
//    коллекцию структурно во время её же обхода
val it = list.iterator()
while (it.hasNext()) { if (it.next() % 2 == 0) it.remove() }

// 2) removeAll / removeIf с предикатом — делают это корректно внутри
list.removeAll { it % 2 == 0 }

// 3) Идти по копии, удалять из оригинала
for (x in list.toList()) { if (x % 2 == 0) list.remove(x) }

Для конкурентного случая — ConcurrentHashMap и CopyOnWriteArrayList: их итераторы weakly consistent / снапшотные и CME не бросают (см. параграф про потокобезопасность). Но это другая гарантия: они дают безопасность при параллельных потоках, а не «можно как угодно менять в for-each».

Почему CME может прилететь даже в однопоточном коде без всякой конкуренции, и почему на него нельзя полагаться как на гарантию обнаружения багов? ★

Слово «concurrent» в имени вводит в заблуждение: CME — про структурную модификацию во время итерации, а не про потоки. Один поток, один for-each, внутри list.add(...) — и modCount разъехался с ожидаемым в итераторе, ловишь исключение на следующем next(). Никакой второй поток не нужен.

А «нельзя полагаться» — потому что проверка best-effort: modCount сверяется только на next()/remove(), не атомарно и не на каждое чтение. Есть сценарии, где модификация остаётся незамеченной — например, изменил и тут же вернул структуру к прежнему числу элементов, или последняя модификация пришлась так, что итератор до сверки не дошёл. Документация прямо просит не строить логику на факте броска CME: это диагностика, помогающая ловить баги в разработке, а не контрактная гарантия.

Реализация LRU-кэша на Kotlin

«Напиши LRU-кэш» — один из самых частых live-coding вопросов. Ключевая идея: чтобы и get, и put были O(1), нужна связка HashMap (доступ по ключу за O(1)) + двусвязный список (порядок использования, перемещение узла в начало за O(1)). Самый недавно использованный — в голове, кандидат на вытеснение — в хвосте.

LruCache.kt
class LruCache<K, V>(private val capacity: Int) {

    private class Node<K, V>(val key: K, var value: V) {
        var prev: Node<K, V>? = null
        var next: Node<K, V>? = null
    }

    private val map = HashMap<K, Node<K, V>>()
    private val head = Node<K, V>(/* dummy */ null as K, null as V)  // фиктивные
    private val tail = Node<K, V>(null as K, null as V)             // границы

    init { head.next = tail; tail.prev = head }

    fun get(key: K): V? {
        val node = map[key] ?: return null
        moveToFront(node)                 // отметили как недавно использованный
        return node.value
    }

    fun put(key: K, value: V) {
        map[key]?.let { it.value = value; moveToFront(it); return }
        val node = Node(key, value)
        map[key] = node
        addFront(node)
        if (map.size > capacity) {        // переполнились — выкидываем LRU из хвоста
            val lru = tail.prev!!
            remove(lru); map.remove(lru.key)
        }
    }

    private fun addFront(n: Node<K, V>) {
        n.prev = head; n.next = head.next
        head.next!!.prev = n; head.next = n
    }
    private fun remove(n: Node<K, V>) {
        n.prev!!.next = n.next; n.next!!.prev = n.prev
    }
    private fun moveToFront(n: Node<K, V>) { remove(n); addFront(n) }
}

Что сказать после реализации. Упомяни, что в Android уже есть готовый androidx.collection.LruCache (и DiskLruCache), а «ручной» вариант с LinkedHashMap(accessOrder = true) короче (см. раздел выше). Понимание ручной реализации показывает, что ты знаешь почему это O(1), а не просто дёргаешь библиотеку.
Почему в LRU на голом LinkedHashMap(accessOrder = true) даже один get из нескольких потоков без синхронизации опасен — ведь «мы просто читаем»? ★

При accessOrder = true get — это структурная модификация: он перемещает обращённую запись в хвост двусвязного списка порядка доступа, то есть меняет внутренние ссылки. Документация LinkedHashMap прямо это отмечает.

Значит конкурентные get из разных потоков — гонка записи, а не безобидное чтение: можно порвать связи списка или поймать ConcurrentModificationException, как при параллельной модификации обычного HashMap. Тонкость для задачи 3 этой главы: ReentrantReadWriteLock там недостаточно защищает get через read-лок — раз get мутирует, ему нужен write-лок, иначе несколько «читающих» потоков войдут под read-лок одновременно и подерутся за список. Либо честно бери готовый androidx.collection.LruCache, синхронизированный внутри.

Частые вопросы интервьюера

Что такое амортизированная сложность? Пример. Middle

Это усреднённая стоимость операции по длинной серии вызовов, а не по одному вызову. Пример — ArrayList.add() в конец: обычно O(1), но при заполнении массива происходит расширение и копирование за O(n). Поскольку ёмкость растёт мультипликативно (в 1.5 раза), суммарная стоимость n вставок равна O(n), то есть на одну вставку — амортизированные O(1). В отличие от «среднего случая», это гарантия по серии, а не вероятностная оценка.

Как устроен HashMap и что изменилось в Java 8? Middle+

Массив бакетов Node[] (по умолчанию 16). Индекс = (n-1) & hash, где хеш дополнительно перемешивается (h ^ (h >>> 16)). Коллизии складываются в связный список (chaining). В Java 8 добавлен treeify: если в бакете больше 8 элементов и ёмкость ≥ 64, список превращается в красно-чёрное дерево, и поиск в бакете становится O(log n) вместо O(n) — защита от деградации при плохом hashCode или hash-flooding атаках. Load factor 0.75: при заполнении выше порога таблица удваивается с полным rehash.

Какой контракт hashCode и equals? Что будет при нарушении? Middle

Правила: (1) если объекты равны по equals, их hashCode обязаны совпадать; (2) hashCode стабилен, пока объект не меняется; (3) неравные объекты могут иметь равный хеш (коллизия — норма). Если нарушить первое правило, объект «теряется» в HashMap: положили в один бакет, ищем в другом. Отдельная ловушка — мутабельный ключ: после изменения поля, входящего в хеш, он оказывается в «неправильном» бакете. Поэтому ключи делают иммутабельными (часто data class).

Чем ConcurrentHashMap лучше synchronizedMap? Senior

synchronizedMap — это один глобальный лок на всю коллекцию: потоки сериализуются, throughput не масштабируется. ConcurrentHashMap (Java 8+) синхронизируется на уровне отдельного бакета: вставка в пустой бакет — атомарный CAS без блокировки, при коллизии — synchronized только на первом узле этого бакета. Чтения почти всегда lock-free (по volatile-ссылкам). Глобального лока нет, поэтому много потоков работают параллельно. Нюанс: атомарны лишь отдельные операции — для «check-then-act» нужны computeIfAbsent/putIfAbsent; null-ключи и null-значения запрещены.

Когда CopyOnWriteArrayList, а когда synchronizedList? Senior

CopyOnWriteArrayList копирует весь массив при каждой записи (O(n)), зато чтения и итерация — без локов и без ConcurrentModificationException. Идеален для сценария «много чтений, редкие записи»: списки слушателей, observer'ы, конфиги. synchronizedList — простая обёртка с одним локом на все операции; уместна при невысокой нагрузке, но итерацию надо вручную оборачивать в synchronized(list). Если записей много — оба плохи, лучше пересмотреть структуру (например, очередь).

ArrayList или LinkedList — что выбрать? Middle

В подавляющем большинстве случаев — ArrayList: доступ по индексу O(1) и отличная локальность кэша (данные лежат подряд). LinkedList теоретически даёт O(1) вставку/удаление, но только по уже имеющемуся итератору, а доступ по индексу — O(n); на практике кэш-промахи и накладные расходы на узлы делают его медленнее почти всегда. Для очереди или дека бери ArrayDeque, а не LinkedList.

Как реализовать LRU-кэш за O(1)? Middle+

Связка HashMap + двусвязный список. HashMap даёт доступ к узлу по ключу за O(1), двусвязный список хранит порядок использования: при get/put узел перемещается в голову за O(1) (перецепка ссылок). При переполнении удаляется узел из хвоста (наименее недавно использованный) и его ключ из мапы. Фиктивные head/tail-узлы упрощают перецепку без проверок на null. Короткая альтернатива — LinkedHashMap(accessOrder = true) с переопределённым removeEldestEntry; в Android есть готовый androidx.collection.LruCache.

Когда нужен TreeMap вместо HashMap? Middle+

Когда нужен порядок ключей или диапазонные запросы. TreeMap на красно-чёрном дереве держит ключи отсортированными и даёт навигацию: firstKey, ceilingKey/floorKey («ближайший ≥/≤ x»), subMap/headMap/tailMap, упорядоченный обход. Цена — все операции O(log n) вместо O(1). Если порядок не нужен — HashMap быстрее. Если нужен порядок вставки/доступа (не сортировка) — LinkedHashMap.

В чём разница между DFS и BFS, и где что применять? Middle

DFS (в глубину) идёт максимально далеко по ветке, потом откатывается; реализуется рекурсией или стеком, память O(h) (высота). Подходит для проверки связности, поиска циклов, топосорта, обходов pre/in/post-order. BFS (в ширину) обходит по уровням через очередь, память O(w) (ширина); на невзвешенном графе даёт кратчайший путь по числу рёбер, используется для поиска ближайших, обхода по слоям. Оба — O(V + E) по времени.

Чем quicksort отличается от mergesort, и почему стандартная сортировка объектов в Java/Kotlin — это mergesort (TimSort), а не quicksort? Senior

Quicksort — in-place, в среднем O(n log n), но в худшем случае O(n²) (на уже отсортированном входе при плохом pivot) и нестабилен. Mergesort — всегда O(n log n), стабилен, но требует O(n) доп. памяти. В JDK Arrays.sort для примитивов использует Dual-Pivot Quicksort (память дешевле, стабильность для примитивов не нужна — одинаковые int неразличимы). А Arrays.sort/Collections.sort для объектов и Kotlin sortedBy используют TimSort — гибрид merge + insertion sort. Причина — именно стабильность: при сортировке по нескольким ключам (сначала по одному полю, потом по другому) порядок равных сохраняется. Плюс TimSort гарантирует O(n log n) в худшем случае, а на почти отсортированных данных доходит до O(n).

Чем in-place алгоритм отличается от не-in-place, и почему mergesort не in-place, а quicksort и heapsort — да? Middle+

In-place алгоритм работает в пределах исходной структуры, используя O(1) (или O(log n) на стек) дополнительной памяти, не зависящей от размера входа. Quicksort и heapsort переставляют элементы прямо в массиве (quicksort — O(log n) на стек рекурсии, heapsort — O(1)), поэтому считаются in-place. Mergesort на каждом слиянии заводит временный буфер размером с объединяемые части — суммарно O(n) доп. памяти, поэтому не in-place. Это и есть trade-off: mergesort платит памятью за стабильность и гарантированный O(n log n), а quicksort экономит память ценой нестабильности и риска O(n²).

Что такое куча и PriorityQueue, какие у них сложности и где это нужно? Middle+

Бинарная куча — полное бинарное дерево, упакованное в массив, где родитель не больше (min-heap) или не меньше (max-heap) детей. peek — O(1), offer/poll — O(log n) за счёт просеивания, построение из массива (heapify) — O(n). В Kotlin это java.util.PriorityQueue (по умолчанию min-heap по natural ordering или Comparator). Применяют для top-K (k-й по величине — через heap размера k за O(n log k)), слияния k отсортированных списков, алгоритмов Дейкстры и Прима, планировщиков и Huffman. Частая задача — «k самых частых элементов»: HashMap частот + min-heap размера k. Подробнее — в параграфе «Куча и приоритетная очередь».

Почему ArrayDeque предпочтительнее Stack и LinkedList и для стека, и для очереди? Middle+

java.util.Stack устарел: он наследует Vector, поэтому каждый метод synchronized (лишний лок без надобности) и допускает доступ по индексу/вставку в середину, нарушая LIFO-семантику. LinkedList как дек работает, но узлы разбросаны по куче — кэш-промахи и память на два указателя на элемент. ArrayDeque — кольцевой буфер на массиве: push/pop/peek с обоих концов амортизированный O(1), отличная локальность кэша, без синхронизации. Документация Java прямо рекомендует ArrayDeque и как замену Stack (LIFO), и как замену LinkedList (FIFO). Ограничение: java.util.ArrayDeque не допускает null (это маркер пустоты); у Kotlin-овского kotlin.collections.ArrayDeque такого запрета нет.

Задача 1. Реализуй функцию, которая находит два числа в массиве nums, дающих в сумме target. Верни их индексы. Предполагай, что ровно одно решение существует, один элемент не использовать дважды. Сложность — лучше O(n²).

Показать решение 
fun twoSum(nums: IntArray, target: Int): IntArray {
    val seen = HashMap<Int, Int>()   // значение → индекс
    for ((i, num) in nums.withIndex()) {
        val complement = target - num
        seen[complement]?.let { j -> return intArrayOf(j, i) }
        seen[num] = i
    }
    error("No solution")
}

Объяснение: за один проход кладём в HashMap каждое число и его индекс. Для текущего элемента ищем дополнение target - num — если оно уже в мапе, нашли пару. Сложность: время O(n), память O(n). Наивный двойной цикл даёт O(n²) — на больших массивах неприемлемо.

Задача 2. Дан список целых чисел. Реализуй функцию maxSlidingWindow(a: IntArray, k: Int): IntArray, которая возвращает массив максимумов для каждого скользящего окна размера k. Вход: [1, 3, -1, -3, 5, 3, 6, 7], k = 3; ожидаемый выход: [3, 3, 5, 5, 6, 7]. Сложность — лучше O(n·k).

Показать решение 
fun maxSlidingWindow(a: IntArray, k: Int): IntArray {
    val result = IntArray(a.size - k + 1)
    val deque = ArrayDeque<Int>()   // хранит индексы, по убыванию значений

    for (i in a.indices) {
        // выкидываем индексы, вышедшие за левую границу окна
        if (deque.isNotEmpty() && deque.first() < i - k + 1) {
            deque.removeFirst()
        }
        // убираем с хвоста индексы, чьё значение меньше текущего
        while (deque.isNotEmpty() && a[deque.last()] < a[i]) {
            deque.removeLast()
        }
        deque.addLast(i)
        if (i >= k - 1) {
            result[i - k + 1] = a[deque.first()]
        }
    }
    return result
}

Объяснение: монотонная убывающая дека хранит индексы кандидатов на максимум. Голова деки — всегда максимум текущего окна. При сдвиге окна вправо выбрасываем устаревшие индексы из головы и меньшие элементы с хвоста. Сложность: O(n) время (каждый индекс добавляется и удаляется не более одного раза), O(k) память.

Задача 3. Реализуй потокобезопасный LRU-кэш с ёмкостью capacity. Методы get(key) и put(key, value) должны работать за O(1) и безопасно вызываться из нескольких потоков одновременно. Покажи минимальный рабочий вариант на Kotlin.

Показать решение 
import java.util.concurrent.locks.ReentrantReadWriteLock
import kotlin.concurrent.write

class ThreadSafeLruCache<K, V>(capacity: Int) {

    private val lock = ReentrantReadWriteLock()
    private val cache: LinkedHashMap<K, V> =
        object : LinkedHashMap<K, V>(capacity, 0.75f, true) {
            override fun removeEldestEntry(eldest: Map.Entry<K, V>): Boolean =
                size > capacity
        }

    // ВАЖНО: get при accessOrder=true мутирует список доступа,
    // поэтому ему нужен write-лок, а не read-лок
    fun get(key: K): V? = lock.write { cache[key] }

    fun put(key: K, value: V): Unit = lock.write { cache[key] = value }
}

Объяснение: LinkedHashMap(accessOrder = true) с переопределённым removeEldestEntry даёт LRU-логику. Подвох в потокобезопасности: при accessOrder = true get перемещает запись в хвост, то есть структурно меняет карту — значит read-лок для него небезопасен, нужен write-лок (несколько параллельных «читателей» порвали бы список). Поэтому ReentrantReadWriteLock здесь фактически вырождается в обычный mutex; проще честно обернуть оба метода в synchronized или взять готовый androidx.collection.LruCache. Сложность операций: амортизированный O(1) без учёта блокировки.

Задача 4. Дан массив nums и число k. Верни k самых часто встречающихся элементов. Сложность — лучше O(n log n) полной сортировки. Вход: nums = [1,1,1,2,2,3], k = 2; выход: [1, 2] (в любом порядке).

Показать решение 
fun topKFrequent(nums: IntArray, k: Int): IntArray {
    val freq = HashMap<Int, Int>()                     // элемент → частота, O(n)
    for (x in nums) freq[x] = (freq[x] ?: 0) + 1
    // min-heap размера k по частоте
    val heap = java.util.PriorityQueue<Int>(compareBy { freq[it] })
    for (key in freq.keys) {
        heap.offer(key)
        if (heap.size > k) heap.poll()                 // выкидываем самый редкий
    }
    return IntArray(k) { heap.poll() }
}

Объяснение: частоты считаем в HashMap за O(n), затем держим min-heap размера k по частоте — после прохода в нём ровно k самых частых. Время O(n log k), память O(n). Классический паттерн top-K: HashMap + куча. Если нужно строго O(n) — bucket sort по частоте (массив списков, индекс = частота).

Задача 5. Реализуй mergeKLists(lists: List<ListNode?>): ListNode? — слияние k отсортированных связных списков в один отсортированный. Сложность — лучше наивного слияния по одному за O(k·N).

Показать решение 
class ListNode(val value: Int, var next: ListNode? = null)

fun mergeKLists(lists: List<ListNode?>): ListNode? {
    val heap = java.util.PriorityQueue<ListNode>(compareBy { it.value })
    for (node in lists) if (node != null) heap.offer(node)   // головы всех списков

    val dummy = ListNode(0)
    var tail = dummy
    while (heap.isNotEmpty()) {
        val min = heap.poll()           // минимум среди голов — O(log k)
        tail.next = min
        tail = min
        min.next?.let { heap.offer(it) } // продвигаем тот список вперёд
    }
    return dummy.next
}

Объяснение: в куче всегда лежат текущие головы k списков. На каждом шаге берём минимум за O(log k), цепляем к результату, добавляем его next. Всего N узлов → время O(N log k), память O(k) на кучу. Куда лучше наивного O(k·N) (повторное сравнение всех голов на каждом узле).

Задача 6. Реализуй quickSort(a: IntArray) на месте (in-place) и объясни, как выбор pivot влияет на сложность. Покажи вход, при котором наивный pivot = последний элемент даёт O(n²).

Показать решение 
// Схема разбиения Ломуто, pivot = последний элемент
fun quickSort(a: IntArray, lo: Int = 0, hi: Int = a.size - 1) {
    if (lo >= hi) return
    val pivot = a[hi]
    var i = lo                                   // граница «меньших pivot»
    for (j in lo until hi) {
        if (a[j] < pivot) { a[i] = a[j].also { a[j] = a[i] }; i++ }
    }
    a[i] = a[hi].also { a[hi] = a[i] }           // ставим pivot на место
    quickSort(a, lo, i - 1)
    quickSort(a, i + 1, hi)
}

Объяснение: разбиваем массив на < pivot и ≥ pivot перестановками in-place, рекурсивно сортируем половины. Память O(log n) на стек при сбалансированных разбиениях. На уже отсортированном (или обратно отсортированном) входе фиксированный pivot = последний элемент даёт максимально кривые разбиения 1 и n-1 → глубина рекурсии n → O(n²). Лечится рандомизацией pivot или median-of-three. Именно поэтому стандартный Arrays.sort для примитивов берёт Dual-Pivot Quicksort с защитами, а не наивную схему.

Задача 7. Дан набор номиналов монет coins и сумма amount. Верни минимальное число монет, которым можно набрать сумму, или -1, если нельзя. Монет каждого номинала — неограниченно. Вход: coins = [1,2,5], amount = 11; выход: 3 (5+5+1).

Показать решение 
// DP снизу-вверх: dp[s] — минимум монет на сумму s
fun coinChange(coins: IntArray, amount: Int): Int {
    val dp = IntArray(amount + 1) { amount + 1 }   // «бесконечность»
    dp[0] = 0
    for (s in 1..amount) {
        for (c in coins) {
            if (c <= s) dp[s] = minOf(dp[s], dp[s - c] + 1)
        }
    }
    return if (dp[amount] > amount) -1 else dp[amount]
}

Объяснение: классическая bottom-up DP. dp[s] — минимум монет на сумму s; набираем его, пробуя последней каждую монету c: dp[s] = min(dp[s], dp[s-c] + 1). Инициализация «бесконечностью» (amount+1 недостижимо) отличает «нельзя набрать» от валидного нуля. Время O(amount · coins), память O(amount). Жадность (брать самую крупную) тут не работает: на coins=[1,3,4], amount=6 жадный даёт 4+1+1=3, а оптимум 3+3=2.

Задача 8. Найди длину наибольшей общей подпоследовательности (Longest Common Subsequence) двух строк. Подпоследовательность — символы в исходном порядке, но не обязательно подряд. Вход: "abcde", "ace"; выход: 3 ("ace").

Показать решение 
// 2D-DP: dp[i][j] — LCS префиксов a[0..i) и b[0..j)
fun lcs(a: String, b: String): Int {
    val dp = Array(a.length + 1) { IntArray(b.length + 1) }
    for (i in 1..a.length) {
        for (j in 1..b.length) {
            dp[i][j] = if (a[i - 1] == b[j - 1])
                dp[i - 1][j - 1] + 1                       // символы совпали
            else
                maxOf(dp[i - 1][j], dp[i][j - 1])         // пропускаем один
        }
    }
    return dp[a.length][b.length]
}

Объяснение: таблица dp[i][j] — длина LCS префиксов. Совпали символы — берём диагональ +1; нет — лучшее из «выкинуть символ из a» и «из b». Время и память O(n·m); память сжимается до O(min(n,m)), если хранить два ряда. Основа diff-алгоритмов и сравнения версий.

Задача 9. Дана сетка grid из '1' (земля) и '0' (вода). Посчитай число островов — групп соседних по вертикали/горизонтали клеток земли. Соседство по диагонали не считается.

Показать решение 
// DFS «затопления»: нашли землю — гасим весь остров и считаем +1
fun numIslands(grid: Array<CharArray>): Int {
    if (grid.isEmpty()) return 0
    val rows = grid.size; val cols = grid[0].size
    var count = 0

    fun sink(r: Int, c: Int) {
        if (r < 0 || c < 0 || r >= rows || c >= cols || grid[r][c] != '1') return
        grid[r][c] = '0'                       // помечаем посещённую землю
        sink(r + 1, c); sink(r - 1, c); sink(r, c + 1); sink(r, c - 1)
    }

    for (r in 0 until rows) for (c in 0 until cols) {
        if (grid[r][c] == '1') { count++; sink(r, c) }
    }
    return count
}

Объяснение: обходим сетку; встретив непосещённую землю, запускаем DFS, который «затапливает» весь связный остров (метит клетки как '0'), и увеличиваем счётчик. Каждая клетка посещается раз → время O(rows·cols), память O(rows·cols) на стек рекурсии в худшем случае. На большой сетке DFS-рекурсия рискует переполнить стек — тогда берут BFS с очередью (ArrayDeque координат).

Задача 10. Коко ест бананы из piles кучек, за h часов. За час она съедает до speed бананов из одной кучки (если в кучке меньше — доедает и ждёт следующего часа). Найди минимальную целую speed, при которой она успеет за h часов. Вход: piles = [3,6,7,11], h = 8; выход: 4.

Показать решение 
// Бинарный поиск по ответу: скорость монотонна по «успеет ли»
fun minEatingSpeed(piles: IntArray, h: Int): Int {
    fun hoursAt(speed: Int): Long =
        piles.sumOf { ((it + speed - 1) / speed).toLong() }   // ceil(pile/speed)

    var lo = 1; var hi = piles.max()
    while (lo < hi) {
        val mid = lo + (hi - lo) / 2
        if (hoursAt(mid) <= h) hi = mid       // успевает — пробуем медленнее
        else lo = mid + 1                     // не успевает — нужно быстрее
    }
    return lo
}

Объяснение: «успеет ли за h часов при скорости speed» — монотонный предикат: чем выше скорость, тем меньше часов. Значит можно бинарно искать минимальную скорость в диапазоне [1, max(piles)] — это паттерн binary search on answer. Время O(n log(max)): log итераций поиска, на каждой O(n) проверка. ceil через (it + speed - 1) / speed избегает дробей; сумма часов в Long — от переполнения при больших кучках.
