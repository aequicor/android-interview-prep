UI / Compose

Jetpack Compose

Декларативный UI-тулкит (набор инструментов пользовательского интерфейса) Android: состояние и remember (запоминание между рекомпозициями), slot table (таблица слотов), рекомпозиция, stable/strong skipping (стабильность и усиленный пропуск лишних перерисовок), side effects (побочные эффекты), фазы measure/layout/draw (измерение/раскладка/отрисовка) и SubcomposeLayout (отложенная подкомпозиция). На Middle+/Senior (уверенный/старший уровень) ждут не «как написать кнопку», а «как это работает под капотом и почему тормозит».

Middle Senior Частый вопрос

Состояние: remember, mutableStateOf и друзья

mutableStateOf создаёт наблюдаемый контейнер MutableState<T>: чтение его .value внутри composable подписывает этот scope на изменения, а запись планирует рекомпозицию читателей. remember кэширует значение между рекомпозициями (по позиции в дереве). Без remember состояние создавалось бы заново при каждом проходе.

Counter.kt
@Composable
fun Counter() {
    // remember { } — переживает рекомпозицию, но НЕ пересоздание (поворот экрана)
    var count by remember { mutableStateOf(0) }

    Button(onClick = { count++ }) {        // запись в .value -> рекомпозиция читателей
        Text("Clicked $count times")       // чтение count -> подписка этого scope
    }
}

rememberSaveable: что переживает поворот и смерть процесса

remember теряется при уничтожении и пересоздании композиции (смена конфигурации, например поворот). rememberSaveable сохраняет значение в Bundle через механизм savedInstanceState — поэтому переживает и поворот, и смерть процесса (system-initiated process death), пока сохранён SavedStateRegistry. Для нестандартных типов нужен Saver.

МеханизмПоворот / смена конфигурацииСмерть процессаГде хранится
remember❌ теряется❌ теряетсяВ памяти, в slot table
rememberSaveable✅ переживает✅ переживаетВ Bundle (savedInstanceState)
ViewModel✅ переживает❌ теряется*В ViewModelStore
SavedStateHandle✅ переживает✅ переживаетBundle внутри ViewModel

* ViewModel переживает смену конфигурации, но не смерть процесса; для неё используйте SavedStateHandle.

SaverExample.kt
data class City(val name: String, val country: String)

// Saver описывает, КАК упаковать тип в сохраняемое представление и обратно
val CitySaver = listSaver<City, String>(
    save = { listOf(it.name, it.country) },
    restore = { City(it[0], it[1]) }
)

@Composable
fun CityPicker() {
    // переживёт и поворот, и смерть процесса
    var city by rememberSaveable(stateSaver = CitySaver) {
        mutableStateOf(City("Berlin", "DE"))
    }
}
⚠️
В Bundle кладут мало и сериализуемое (TransactionTooLargeException при больших данных). Списки на сотни элементов или объекты с логикой — это про ViewModel + источник данных, а не про rememberSaveable.

derivedStateOf и snapshotFlow

derivedStateOf — вычисляемое состояние: пересчитывается, только когда меняется результат, а не каждое исходное чтение. Нужен, когда из часто меняющегося state выводится редко меняющийся флаг (классика — «показать кнопку Scroll-to-top, если прокрутили > N»). snapshotFlow превращает чтение Compose-state в Flow, эмитя только при реальном изменении читаемых значений.

DerivedAndFlow.kt
@Composable
fun MessageList(listState: LazyListState) {
    // БЕЗ derivedStateOf: пересчёт на каждый пиксель скролла -> лишние рекомпозиции
    val showButton by remember {
        derivedStateOf { listState.firstVisibleItemIndex > 0 }
    }

    // snapshotFlow: реагируем на изменение state как на Flow (аналитика, дебаунс)
    LaunchedEffect(listState) {
        snapshotFlow { listState.firstVisibleItemIndex }
            .distinctUntilChanged()
            .collect { index -> analytics.log("scrolled_to", index) }
    }
}
Почему mutableStateOf по умолчанию сравнивает значения через structuralEqualityPolicy, и когда это приводит к багу? ★

mutableStateOf принимает параметр policy: SnapshotMutationPolicy, и по умолчанию это structuralEqualityPolicy(). При записи новое значение сравнивается со старым через equals(); если они равны, запись не считается изменением и подписчиков никто не уведомляет. Для data class и примитивов ровно то, что нужно — лишних рекомпозиций нет.

Ловушка вылезает, если положить в State мутабельный объект и менять его поля на месте, не пересоздавая инстанс. Тогда equals сравнивает объект сам с собой и возвращает true — рекомпозиция не происходит, UI замирает на старом виде. Зеркальная проблема у referentialEqualityPolicy() (сравнение по ===): два equals-равных, но разных инстанса дадут лишнюю рекомпозицию. Практика простая: храните в State неизменяемые значения и создавайте новый объект на каждое изменение — либо осознанно выбирайте policy под задачу.

Slot table и Composer: дерево под капотом

Composer — «мозг» рантайма. Он хранит композицию в slot table: плоской структуре групп (групп вызовов composable, состояний remember, узлов layout). Каждый @Composable при компиляции оборачивается в startRestartGroup()/endRestartGroup() — это и есть recomposition scope, минимальная единица, которую рантайм умеет переисполнить независимо.

flowchart TB Root["startRestartGroup #1: Screen"] Root --> S0["slot: remember state"] Root --> G1["startRestartGroup #2: Header"] Root --> G2["startRestartGroup #3: List"] G2 --> I0["group: item 0"] G2 --> I1["group: item 1"] G2 --> I2["group: item 2"] G1 --> T0["LayoutNode: Text"]
Дерево композиции в slot table: группы и слоты идут линейно, по позиции вызова

Позиционная мемоизация и gap buffer

Позиционная мемоизация: рантайм сопоставляет элементы текущего прохода с прошлым по позиции вызова в коде, а не по идентичности объекта. Если composable вызван в том же месте — Compose переиспользует его слот (его remember, его узел). Именно поэтому remember работает «магически»: ключ — позиция в дереве.

Под капотом slot table — это gap buffer: массив с «пустым окном» (gap), которое перемещается туда, где идёт следующая запись. Последовательные вставки в позицию gap дёшевы. Но при вставке/удалении/перестановке в другом месте gap приходится двигать, копируя элементы между текущей и целевой позицией — поэтому неструктурные изменения порядка дорогие.

Почему key() критичен в списках

Если порядок элементов меняется (вставка в начало, удаление, перестановка), позиционная мемоизация ломается: рантайм решит, что в позиции №0 «тот же» элемент, и переиспользует чужое состояние/анимацию. key() (и параметр key в items) даёт элементу стабильную идентичность поверх позиции — рантайм корректно переносит слоты при reorder.

KeyInList.kt
@Composable
fun TodoList(todos: List<Todo>) {
    LazyColumn {
        // БЕЗ key: при удалении/вставке состояние "съезжает" на соседей,
        // ломаются анимации и remember внутри строки
        items(todos, key = { it.id }) { todo ->
            TodoRow(todo)   // у каждого id — своя стабильная identity в slot table
        }
    }
}
💡
На собесе. «Зачем key в LazyColumn?» — не «для производительности» в общем, а конкретно: чтобы сохранить идентичность слота при изменении порядка, иначе ломается состояние/анимации внутри элементов и теряется корректность переиспользования.
Почему ветки if/else с одинаковым Text() получают разную идентичность в slot table, а элементы цикла без key — нет? ★

Компилятор присваивает каждой группе source-position key — ключ по позиции вызова в исходнике. Text("Hello") в ветке if и Text("World") в else стоят на разных позициях в коде, поэтому это две разные группы с разной идентичностью, и рантайм не перепутает их remember-слоты. Ветвление безопасно без ключа.

А элементы цикла items { } вызываются с одной и той же позиции вызова — у них совпадает source-position key, и различить их рантайм может только по семантическому ключу, который мы задаём через key(). Без него позиционная мемоизация привязывает слот к индексу: вставка в начало сдвигает всех, и состояние с анимацией «съезжают» на соседей. Отсюда правило — if/else обходятся без ключа, а переупорядочиваемые списки нет.

Рекомпозиция: что её триггерит и smart skipping

Рекомпозиция — переисполнение composable при изменении входов. Триггеры: запись в прочитанный State; изменение параметра, на котором scope подписан. Рантайм точечный: переисполняется не вся функция, а только scope, который читал изменившееся значение. Остальное пропускается (skipping).

Stable и skippable: когда scope можно пропустить

Compose-компилятор помечает composable как skippable, если может на входе сравнить параметры с прошлыми и пропустить тело при равенстве. Это возможно, только если все параметры стабильны. Стабильность — контракт: «equals корректен, и если значение изменится, Compose будет уведомлён».

АннотацияКонтрактГарантия
@StableСвойства могут меняться, но об изменении Compose узнает (через State); equals консистентен.Тип skippable; разрешена мутабельность через наблюдаемые поля.
@ImmutableПоля никогда не меняются после создания (все val, неизменяемые типы).Более строгий частный случай @Stable; рантайм может агрессивнее кэшировать.

Разница: @Immutable обещает полную неизменяемость значения; @Stable допускает изменения, но требует уведомлять рантайм. Оба делают тип стабильным и помогают пропускать рекомпозицию.

Stability.kt
// ✅ Immutable: всё val, типы неизменяемые -> стабильный, skippable
@Immutable
data class UiUser(val id: Long, val name: String)

// ✅ Stable: поле меняется, НО через наблюдаемый State -> рантайм уведомлён
@Stable
class FormState {
    var name by mutableStateOf("")
    val isValid: Boolean get() = name.isNotBlank()
}

@Composable
fun Profile(user: UiUser) {   // user стабилен -> Profile пропустится, если user не изменился
    Text(user.name)
}

Strong skipping mode

С Compose-компилятора 1.5.4+ можно включить strong skipping mode (с Kotlin 2.0.20 — по умолчанию). Он делает skippable даже composable с нестабильными параметрами: нестабильные сравниваются по === (instance equality), стабильные — по equals. Плюс он автоматически мемоизирует лямбды, в т.ч. с нестабильными захватами. Это снимает значительную часть боли с нестабильными типами, но не отменяет пользу делать модели стабильными.

🛑
Частая ошибка — нестабильные типы. List/Set/Map всегда нестабильны (рантайм не гарантирует, что под интерфейсом не ArrayList, который мутируют). Передача List<T> в параметр делает composable не-skippable (без strong skipping). Решения: ImmutableList из kotlinx.collections.immutable, обёртка-@Immutable data class, либо включить strong skipping.
UnstableList.kt
// ❌ List нестабилен -> Feed не skippable, рекомпозируется "за компанию"
@Composable fun Feed(items: List<Post>) { /* ... */ }

// ✅ ImmutableList стабилен -> Feed skippable
@Composable fun Feed(items: ImmutableList<Post>) { /* ... */ }

// ✅ или обернуть в @Immutable holder
@Immutable data class Posts(val list: List<Post>)
@Composable fun Feed(items: Posts) { /* ... */ }

Похожая ловушка — нестабильная лямбда: без strong skipping лямбда, захватывающая нестабильное значение, на каждой рекомпозиции — новый инстанс, что ломает skipping у дочернего composable, которому её передали. Выносите такие коллбэки через remember или метод-референс, либо полагайтесь на авто-мемоизацию strong skipping.

Рекомпозицию называют «оптимистичной» — что это значит и почему опасно делать сайд-эффект прямо в теле composable? ★

Оптимистичность значит, что рантайм запускает рекомпозицию, рассчитывая её завершить, но готов отменить и перезапустить, если входные значения изменятся до того, как результат применён к дереву. То есть тело composable может выполниться частично и быть выброшенным. Сверху к этому добавляются ещё два свойства модели: порядок выполнения соседних composable не гарантирован, а сама рекомпозиция может идти на другом потоке.

Поэтому любой побочный эффект прямо в теле — запись в переменную вне state, сетевой вызов, инкремент счётчика — выполнится непредсказуемое число раз или потеряется при отмене. Всё «грязное» выносят в effect-API (LaunchedEffect, SideEffect, DisposableEffect): рантайм вызывает их только на успешно применённой композиции.

Жизненный цикл композиции и порядок рекомпозиции

У composable свой жизненный цикл внутри композиции, и он не совпадает с lifecycle Activity/Fragment. Composable входит в композицию при первом вызове, рекомпозируется ноль или больше раз при изменении входов и покидает композицию, когда перестаёт вызываться. Ветка if стала false — composable вышел из композиции, хотя Activity жива; и наоборот, поворот экрана пересоздаёт композицию, не трогая процесс. Именно на вход и выход завязаны эффекты: LaunchedEffect/DisposableEffect стартуют на входе и очищаются на выходе, а remember держит значение, пока composable в композиции.

stateDiagram-v2 [*] --> Active: первый вызов (enter) Active --> Active: входы изменились (recompose 0..N) Active --> [*]: больше не вызывается (leave)
Жизненный цикл самого composable — это цикл вход → рекомпозиция → выход, отдельный от lifecycle Activity

Что Compose гарантирует о порядке и потоке

Модель исполнения рекомпозиции — то, что любят спрашивать на Senior. Держите в голове три свойства: рекомпозиция оптимистична (может быть отменена и перезапущена при изменении входов до применения результата); порядок выполнения соседних composable не гарантирован; рекомпозиция может идти не на main-потоке. Из этих трёх свойств выводится единственное практическое правило — тело composable обязано быть идемпотентным и без побочных эффектов.

🛑
Не полагайтесь на порядок и поток. Нельзя рассчитывать, что соседний composable уже выполнился, и нельзя писать в общую переменную из тела — на другом потоке или при отмене это даст гонку либо потерянную запись. Любой эффект — в effect-API.
Что такое donut-hole skipping и зачем нужен movableContentOf? ★

Donut-hole skipping — это когда меняется состояние, прочитанное только во вложенной лямбде или вложенном composable: рантайм рекомпозирует именно эту внутреннюю область («дырку от пончика»), а обёртку вокруг неё пропускает, если она сама skippable и её входы не изменились. Отсюда практический приём — выносить чтение state глубже и в лямбды, чтобы уменьшить объём рекомпозиции.

movableContentOf решает другую задачу: он позволяет переместить кусок UI в другое место дерева — например, переложить элемент между Row и Column при смене ориентации — сохранив его состояние и не пересоздавая. Без него рантайм счёл бы перемещённый блок новым по позиции и сбросил бы его слоты.

Side-effects: когда что использовать

Composable должны быть идемпотентны и без побочных эффектов в теле (их вызывают часто, в любом порядке, на любом потоке). Всё «грязное» — запуск корутин, подписки, логирование — выносят в effect-API, которые завязаны на жизненный цикл композиции.

APIКогда использоватьНюанс
LaunchedEffect(key)Запустить suspend-работу, привязанную к композиции; перезапуск при смене key.Корутина живёт, пока effect в композиции; cancel при смене key/уходе.
rememberCoroutineScope()Запускать корутины из коллбэков (onClick), а не при входе в композицию.Scope живёт, пока composable в дереве.
DisposableEffect(key)Эффект, который надо очистить (onDispose): слушатели, регистрации.Обязателен блок onDispose { }.
SideEffectОпубликовать Compose-state в не-Compose объект на каждой успешной композиции.Без ключа; выполняется после каждой рекомпозиции.
produceStateПревратить не-Compose источник (Flow/callback) в State.Внутри — корутина; удобно для загрузки.
rememberUpdatedStateЗахватить «свежее» значение внутри долгоживущего effect без его перезапуска.Спасает от устаревших захватов в LaunchedEffect(Unit).
Effects.kt
@Composable
fun Screen(userId: String, onTimeout: () -> Unit) {
    // перезапустится при смене userId
    LaunchedEffect(userId) {
        val user = repo.load(userId)   // suspend
        // ...
    }

    // DisposableEffect: подписка + очистка
    DisposableEffect(Unit) {
        val listener = NetworkListener { /* ... */ }
        connectivity.register(listener)
        onDispose { connectivity.unregister(listener) }
    }

    // rememberUpdatedState: effect НЕ перезапускаем, но onTimeout всегда свежий
    val currentOnTimeout by rememberUpdatedState(onTimeout)
    LaunchedEffect(Unit) {
        delay(5000)
        currentOnTimeout()             // вызовется актуальная лямбда, не устаревшая
    }
}
⚠️
LaunchedEffect(Unit) + захват изменяющегося параметра = устаревший захват: внутри останется первое значение. Либо добавь параметр в ключи (перезапуск), либо оберни в rememberUpdatedState (без перезапуска). Это любимый «caveat»-вопрос.
LaunchedEffect и rememberCoroutineScope оба дают корутины — на каком scope и dispatcher они стартуют и почему это важно? ★

Оба завязаны на CoroutineScope, привязанный к месту в композиции, и оба наследуют контекст с эффективным диспетчером AndroidUiDispatcher.Main — то есть стартуют на главном потоке. Значит, блокирующую или тяжёлую работу внутри надо явно увести через withContext(Dispatchers.IO/Default), иначе фризнете UI.

Разница — в жизненном цикле. Корутина LaunchedEffect отменяется, когда эффект покидает композицию или меняется ключ. Scope от rememberCoroutineScope живёт, пока сам composable в дереве, и его корутины не привязаны к ключам — поэтому он годится для запусков из коллбэков. Запустить suspend прямо в onClick нельзя: onClick не suspend-контекст, отсюда и нужен заранее запомненный scope.

ViewModel в Compose и UDF

Compose отлично ложится на однонаправленный поток данных (UDF): ViewModel держит состояние (обычно StateFlow<UiState>), UI его читает и отправляет события вверх. viewModel() даёт инстанс, привязанный к ViewModelStoreOwner (обычно к навигационному backstack-entry или Activity); hiltViewModel() — то же, но через Hilt.

flowchart LR UI["UI @Composable"] -->|"event: onClick"| VM["ViewModel"] VM -->|"reduce -> UiState"| ST["StateFlow"] ST -->|"collectAsStateWithLifecycle"| UI VM -->|"save"| SSH["SavedStateHandle"]
UDF: события идут вверх в ViewModel, состояние стекает вниз в UI
ProfileScreen.kt
@HiltViewModel
class ProfileViewModel @Inject constructor(
    private val repo: ProfileRepository,
    private val handle: SavedStateHandle      // переживает смерть процесса
) : ViewModel() {
    // восстановим черновик после process death
    private val draft = handle.getStateFlow("draft", "")
    private val _state = MutableStateFlow(ProfileUiState())
    val state: StateFlow<ProfileUiState> = _state.asStateFlow()

    fun onSave(text: String) { handle["draft"] = text }
}

@Composable
fun ProfileScreen(vm: ProfileViewModel = hiltViewModel()) {
    // collectAsStateWithLifecycle: собираем только в активном жизненном цикле
    val state by vm.state.collectAsStateWithLifecycle()
    ProfileContent(state, onSave = vm::onSave)   // событие -> вверх
}
💡
collectAsState vs collectAsStateWithLifecycle. Первый собирает Flow, пока composable в композиции (даже когда экран в фоне) — лишняя работа и риск утечек. Второй останавливает сбор ниже STARTED (из lifecycle-runtime-compose) — это рекомендованный дефолт для UI-state.
К какому ViewModelStoreOwner привязывается viewModel() внутри composable на экране Navigation Compose, и почему один экран может неожиданно получить общую или новую ViewModel? ★

viewModel() берёт ViewModelStoreOwner из LocalViewModelStoreOwner. В Navigation Compose внутри composable(route) этим owner'ом становится соответствующий NavBackStackEntry, поэтому ViewModel живёт ровно столько, сколько запись в backstack, и уничтожается на popBackStack.

Отсюда два класса сюрпризов. Если вызвать viewModel() выше графа навигации, owner'ом окажется Activity, и ViewModel станет общей для всех экранов — часто непреднамеренно. И наоборот: чтобы осознанно шарить одну ViewModel между экранами одного flow, явно берут owner родительского navigation-графа через getBackStackEntry(graphRoute). Область жизни ViewModel определяется не классом, а тем, чей ViewModelStore оказался в LocalViewModelStoreOwner в точке вызова.

Modifier, фазы и кастомный Layout

Порядок Modifier имеет значение

Modifier — упорядоченная цепочка: каждый элемент оборачивает следующий, и измерение идёт снаружи внутрь, а отрисовка — в порядке цепочки. Поэтому padding().background() и background().padding() дают разный результат.

ModifierOrder.kt
// фон НА ВСЮ область, отступ ВНУТРИ фона (padding "съедает" место под контент)
Modifier.background(Color.Blue).padding(16.dp)

// сначала отступ снаружи (прозрачный), потом фон -> фон МЕНЬШЕ, со смещением
Modifier.padding(16.dp).background(Color.Blue)

// clickable раньше padding -> кликабельна вся область включая отступ
Modifier.clickable { onClick() }.padding(16.dp)

Три фазы кадра

Compose рендерит кадр в три фазы. Ключ к производительности — откладывать чтение state на как можно более позднюю фазу: если значение читается в фазе draw (через лямбду), изменение НЕ вызовет рекомпозицию, только перерисовку.

flowchart LR C["Composition: какой UI"] --> L["Layout: measure + place"] L --> D["Drawing: отрисовка"] C -.->|"skip если state не менялся"| L L -.->|"offset { } читает в layout"| D
Фазы кадра: Composition -> Layout (measure/place) -> Drawing
DeferredRead.kt
val offsetX by animateDpAsState(if (pressed) 8.dp else 0.dp, label = "")

// ❌ чтение offsetX в composition -> рекомпозиция на каждый кадр анимации
Box(Modifier.padding(start = offsetX))

// ✅ чтение в layout-фазе через лямбду -> пропускаем composition, дешевле
Box(Modifier.offset { IntOffset(offsetX.roundToPx(), 0) })

Кастомный Layout, Constraints и кастомный Modifier

Composable Layout даёт список measurables и Constraints от родителя. Ты измеряешь детей (получая Placeable), задаёшь свой размер и размещаешь их. Constraints — это min/max ширины и высоты, спускаемые сверху.

CustomLayout.kt
@Composable
fun Column2(content: @Composable () -> Unit) {
    Layout(content) { measurables, constraints ->
        val placeables = measurables.map { it.measure(constraints) }   // измерили детей
        val width = placeables.maxOf { it.width }
        val height = placeables.sumOf { it.height }
        layout(width, height) {                                         // наш размер
            var y = 0
            placeables.forEach { p -> p.placeRelative(0, y); y += p.height }
        }
    }
}

// кастомный модификатор как обычная extension-функция -> переиспользуемая цепочка
fun Modifier.cardSurface() = this
    .background(MaterialTheme.colorScheme.surface)
    .padding(16.dp)

SubcomposeLayout: отложенная композиция

Обычно порядок строгий: composition → layout. Но иногда нужно знать размер до композиции детей. SubcomposeLayout позволяет компоновать контент во время measure-фазы, используя уже известные constraints/размеры. На нём построены LazyColumn/LazyRow (компонуют только видимые элементы) и BoxWithConstraints (даёт constraints в composition-scope детей).

ПримитивЗачемЦена
LayoutСвоя measure/place-политика для фиксированного набора детей.Дёшево, обычный путь.
SubcomposeLayoutКомпозиция зависит от размеров: lazy-списки, размер по доступному месту.Дороже: композиция в measure-фазе.
BoxWithConstraintsНужны Constraints родителя уже в composition (адаптивный UI).Внутри это SubcomposeLayout — не злоупотреблять.
Adaptive.kt
@Composable
fun Adaptive(content: @Composable (isWide: Boolean) -> Unit) {
    // constraints доступны прямо в composition детей
    BoxWithConstraints {
        val isWide = maxWidth > 600.dp     // решаем layout по доступной ширине
        content(isWide)
    }
}
⚠️
BoxWithConstraints не бесплатен. Он использует SubcomposeLayout и переносит композицию в measure-фазу. Для простого «если экран широкий» это нормально, но не оборачивай в него каждый элемент списка — на собесе ценят понимание этой цены.
Modifier.Node против старого Modifier.composed — почему фабрики на composed считаются дорогими и что изменил Modifier.Node? ★

Modifier.composed { } создаёт модификатор внутри composition-scope: его материализация — это вызов composable, со своим remember-слотом и участием в рекомпозиции, и он не мемоизируется между местами применения. Цепочка из нескольких composed-модификаторов на каждом элементе списка добавляет работу в фазу композиции и мешает skipping.

Modifier.Node (узлы поверх Modifier.Element) переносит логику модификатора из composition в подсистему модификаторов: узел создаётся один раз, переиспользуется и обновляется напрямую в Layout/Draw-фазах — без повторной композиции и без лишних аллокаций на рекомпозицию. Практический вывод: кастомные модификаторы с состоянием пишут на Modifier.Node, а не на composed.

CompositionLocal и производительность

CompositionLocal — неявная передача значения вниз по дереву без проброса через каждый параметр (так работают MaterialTheme, LocalContext, LocalDensity). Чтение LocalX.current подписывает scope: при смене значения рекомпозируются только читатели.

CompositionLocal.kt
val LocalAnalytics = compositionLocalOf<Analytics> { error("not provided") }

@Composable
fun App(analytics: Analytics) {
    CompositionLocalProvider(LocalAnalytics provides analytics) {
        Screen()   // глубоко внутри: LocalAnalytics.current — без проброса параметров
    }
}

Чек-лист производительности

  • Откладывай чтение state на фазу layout/draw (лямбда-модификаторы offset { }, drawBehind { }) вместо чтения в composition.
  • Делай типы стабильными (@Immutable, ImmutableList) — больше skippable composable, меньше лишних рекомпозиций.
  • key в lazy-списках по стабильному id; derivedStateOf для производных флагов.
  • Диагностика: Layout Inspector показывает счётчики recomposition/skip; Compose compiler metrics/reports покажут, какие composable не-restartable/не-skippable и какие классы нестабильны.
💡
Compose compiler metrics. Подключив отчёты компилятора, получаешь *-classes.txt и *-composables.txt: видно unstable class Foo и restartable skippable fun Bar. Это объективный способ найти источник лишних рекомпозиций — называй это на собесе, когда спрашивают «как искал перформанс-проблему».
Чем staticCompositionLocalOf отличается от compositionLocalOf по механике инвалидации и почему неверный выбор бьёт по производительности? ★

compositionLocalOf отслеживает чтения: значение хранится в snapshot-state, и при смене значения рекомпозируются только те scope, что реально читали .current. staticCompositionLocalOf чтения не отслеживает — за это он дешевле на чтение, но при смене значения рантайм рекомпозирует всё содержимое лямбды CompositionLocalProvider целиком, а не только читателей.

Правило: для значений, которые меняются в рантайме (тема при переключении день/ночь, конфигурация), берут compositionLocalOf ради точечной инвалидации; для практически неизменяемых зависимостей (ссылка на Analytics, синглтоны вроде LocalContext) — staticCompositionLocalOf ради экономии на чтении. Ошибка в выборе либо рекомпозирует пол-экрана на каждое изменение, либо тратится на трекинг там, где значение не меняется.

compositionLocalOf vs staticCompositionLocalOf

Оба способа объявить CompositionLocal передают значение неявно вниз по дереву — разница в том, как они реагируют на смену значения. compositionLocalOf трекает чтения: значение лежит в snapshot-state, и при provides нового значения рекомпозируются только те composable, которые читали .current. staticCompositionLocalOf чтения не трекает — читать его дешевле, но за это рантайм при смене значения рекомпозирует всё поддерево внутри CompositionLocalProvider, а не только читателей.

Когда CompositionLocal вообще оправдан

CompositionLocal — это неявная зависимость, и злоупотреблять им как «глобальной переменной» не стоит: явный параметр читается и тестируется проще, а неявное чтение прячет связь composable с контекстом. Хорошие кандидаты — сквозные значения, которые нужны на многих уровнях и редко передаются осознанно (тема, плотность, локаль, доступ к Context). Если значение специфично для конкретного экрана или его удобно увидеть в сигнатуре — это параметр, а не CompositionLocal.

Почему чтение compositionLocalOf в глубоком узле всё равно может рекомпозировать «много» при смене значения? ★

Точечность инвалидации работает на уровне recomposition scope, а не отдельной строки. Если .current прочитан высоко в дереве — например, в теле большого composable, который сам не skippable — то инвалидируется весь этот scope со всем его невынесенным содержимым, а не только место чтения. Поэтому «только читатели» означает «только те scope, что читали», и крупный нечитающий-но-обёрточный scope над чтением попадёт под рекомпозицию заодно.

Лечится тем же приёмом, что и обычная рекомпозиция: читать .current как можно глубже и в наименьшем skippable-composable, чтобы scope-читатель был маленьким. Тогда смена значения затронет минимум узлов.

Доступность (accessibility) в Compose

Compose строит для каждого composable дерево семантики (semantics tree) — именно его читают TalkBack и тесты, а не сам UI. Большинство стандартных компонентов (Button, Text, TextField) уже выставляют семантику. Доработка нужна там, где смысл несёт картинка, кастомный жест или нестандартный элемент.

contentDescription и декоративные элементы

У Image/Icon параметр contentDescription — это то, что произнесёт TalkBack. Для содержательной картинки дайте короткое описание смысла, а не имя файла. Для чисто декоративного элемента передайте null — тогда он не попадёт в обход скринридера и не создаст шум.

Semantics.kt
// содержательная иконка: TalkBack произнесёт описание
Icon(Icons.Default.Delete, contentDescription = "Удалить заметку")

// декоративная: null убирает из обхода скринридера
Icon(Icons.Default.Circle, contentDescription = null)

// кастомный кликабельный блок без встроенной семантики
Modifier.semantics {
    contentDescription = "Лайкнуть пост"
    role = Role.Button
}

mergeDescendants и группировка

По умолчанию каждый дочерний элемент — отдельный узел семантики, и TalkBack останавливается на каждом. Если строка списка состоит из аватара, имени и подписи, логично озвучить её одним фокусом: Modifier.semantics(mergeDescendants = true) { } склеивает потомков в один узел. clickable и toggleable делают это автоматически.

⚠️
Размер тач-таргета ≥ 48 dp. Material-компоненты сами расширяют интерактивную область до минимума 48×48 dp, даже если иконка визуально меньше. Для кастомных кликабельных элементов задавайте Modifier.size(48.dp) или minimumInteractiveComponentSize() — иначе по мелкой цели тяжело попасть, и это режут на ревью доступности.

Контраст и тестирование

Минимум по WCAG 2.1 уровня AA — контраст текста к фону 4.5:1 для обычного текста и 3:1 для крупного (от 18 pt, или 14 pt полужирного). Проверять руками — TalkBack включается в системных настройках, по экрану ходят свайпами влево/вправо. Семантику можно закрепить и в тесте: composeTestRule.onNodeWithContentDescription("Удалить заметку").assertHasClickAction() проверяет, что узел доступен и кликабелен.

Когда стоит занулять семантику через clearAndSetSemantics вместо mergeDescendants? ★

mergeDescendants склеивает потомков в один узел, но сохраняет их семантику — TalkBack произнесёт всё их содержимое подряд. Иногда этого мало: автоматически собранное описание получается длинным или неудобочитаемым (например, цена «1 299 ₽» произносится по цифрам, а график вообще не озвучить осмысленно).

clearAndSetSemantics { } отбрасывает семантику всех потомков и задаёт свою — это способ полностью переписать, что услышит пользователь. Применяют, когда нужно дать человекочитаемую формулировку вместо машинной склейки, либо скрыть внутреннюю структуру сложного компонента и озвучить его одной фразой. Цена — ответственность: после clearAndSetSemantics всё, что не выставили вручную, для скринридера исчезает.

Частые вопросы интервьюера

Что переживает поворот экрана, а что — смерть процесса? Middle+

remember — ничего (теряется при пересоздании композиции, т.е. при повороте). rememberSaveable — и поворот, и смерть процесса (хранится в Bundle/savedInstanceState), но только сериализуемое и небольшое. ViewModel — переживает поворот (живёт в ViewModelStore), но НЕ смерть процесса. Чтобы пережить смерть процесса в ViewModel — SavedStateHandle. Большие/несериализуемые данные восстанавливают из источника (БД/сеть), а не из Bundle.

Что такое slot table и позиционная мемоизация? Senior

Slot table — плоская структура (на базе gap buffer), где Composer хранит дерево композиции: группы вызовов, состояния remember, узлы layout. Позиционная мемоизация — рантайм сопоставляет элементы текущего прохода с прошлым по позиции вызова в коде, а не по идентичности объекта. Поэтому remember привязан к месту вызова. Gap buffer делает последовательные вставки дешёвыми, но перестановки/удаления в середине требуют двигать gap (копирование) — отсюда важность key() при reorder.

Чем @Stable отличается от @Immutable и зачем они? Senior

Оба делают тип стабильным, что позволяет компилятору сгенерировать skippable-composable (пропускать рекомпозицию при равных параметрах). @Immutable — строгий контракт: значения полей не меняются после создания (все val, неизменяемые типы), рантайм может агрессивнее кэшировать. @Stable — слабее: поля МОГУТ меняться, но об изменении Compose будет уведомлён (через State), а equals консистентен. Immutable — частный, более сильный случай Stable.

Почему передача List ломает skipping и как чинить? Middle+

List (как Set/Map) компилятор считает нестабильным: за интерфейсом может скрываться мутабельный ArrayList, и нет гарантии корректного equals относительно изменений. Параметр-List делает composable не-skippable, и он рекомпозируется «за компанию». Лечение: ImmutableList/PersistentList из kotlinx.collections.immutable; либо обёртка @Immutable data class; либо включить strong skipping mode, который делает skippable и нестабильные параметры (сравнивая их по ===).

Что такое strong skipping mode? Senior

Режим компилятора (1.5.4+, по умолчанию с Kotlin 2.0.20), который генерирует skipping-логику даже для composable с нестабильными параметрами: стабильные сравниваются по equals, нестабильные — по instance-equality (===). Дополнительно он автоматически мемоизирует лямбды, включая захватывающие нестабильные значения, что снимает проблему «новая лямбда на каждой рекомпозиции». Не отменяет пользу стабильных моделей, но сильно снижает порог входа в перформанс.

LaunchedEffect vs rememberCoroutineScope — когда что? Middle+

LaunchedEffect(key) запускает корутину при входе в композицию и перезапускает при смене ключа; нужен для работы, привязанной к появлению UI (загрузка по id, анимация). rememberCoroutineScope() даёт scope для запуска корутин из коллбэков (onClick, gesture), а не при композиции — потому что в обработчике события нельзя вызвать suspend напрямую. Грубо: эффект «когда показали» → LaunchedEffect; «когда пользователь нажал» → rememberCoroutineScope.

Зачем rememberUpdatedState? Senior

Чтобы внутри долгоживущего эффекта (например LaunchedEffect(Unit), который не должен перезапускаться) использовать всегда актуальное значение параметра/лямбды, а не захваченное при старте. Без него лямбда onTimeout внутри LaunchedEffect(Unit) останется «первой» версией — устаревший захват. rememberUpdatedState(onTimeout) оборачивает значение в State, который effect читает в момент использования, получая свежий коллбэк без перезапуска корутины.

Как объяснить порядок Modifier на собесе? Middle

Modifier — упорядоченная цепочка, где каждый элемент оборачивает следующий; measure идёт снаружи внутрь. Поэтому background().padding() рисует фон на всю область, а отступ внутри него; padding().background() — сначала прозрачный отступ снаружи, фон меньше и смещён. Аналогично clickable до/после padding меняет, попадает ли отступ в кликабельную зону. Вывод: порядок — это семантика, а не косметика.

Зачем нужен SubcomposeLayout? Senior

Стандартно порядок composition → layout: дети компонуются ДО измерения. SubcomposeLayout позволяет компоновать контент во время measure-фазы, когда уже известны constraints/размеры. Это нужно, когда композиция зависит от размера: LazyColumn/LazyRow компонуют только видимые элементы (отложенная композиция); BoxWithConstraints даёт Constraints родителя прямо в composition-scope детей. Цена — композиция переносится в measure-фазу, дороже обычного Layout, поэтому не злоупотребляют.

compositionLocalOf vs staticCompositionLocalOf — в чём разница и когда что брать? Senior

Оба передают значение неявно вниз по дереву. compositionLocalOf отслеживает чтения: при смене значения рекомпозируются только composable, которые читали .current — подходит для значений, меняющихся в рантайме (тема, локаль). staticCompositionLocalOf чтения не трекает, поэтому дешевле, но смена значения рекомпозирует всё содержимое CompositionLocalProvider целиком — берут его для практически неизменяемых зависимостей (синглтоны, Analytics, аналоги LocalContext).

Гарантирован ли порядок выполнения composable и на каком потоке идёт рекомпозиция? Senior

Нет. Рекомпозиция оптимистична (может быть отменена и перезапущена при изменении входов до применения результата), порядок вызова дочерних composable не гарантирован, и рекомпозиция может выполняться на потоке, отличном от main. Отсюда правила: не полагаться на порядок выполнения соседних composable, не делать побочных эффектов в теле (они выполнятся непредсказуемое число раз или потеряются при отмене) и выносить всё «грязное» в effect-API.

Жизненный цикл composable: enter / recompose / leave composition — как он соотносится с Activity lifecycle? Middle+

У composable свой жизненный цикл внутри композиции: он входит в композицию (первый запуск), рекомпозируется ноль или больше раз при изменении входов и покидает композицию, когда перестаёт вызываться. С lifecycle Activity/Fragment это напрямую не связано: composable может покинуть композицию, пока Activity жива (ветка if стала false), и наоборот. На вход/выход завязаны эффекты: LaunchedEffect/DisposableEffect стартуют при входе и очищаются при выходе. Поэтому состояние помнят через remember (живёт, пока composable в композиции), а не через поля.

Что такое donut-hole skipping и зачем нужен movableContentOf? Senior

Donut-hole skipping: когда меняется состояние, прочитанное только во вложенной лямбде/composable, рантайм рекомпозирует именно эту внутреннюю область («дырку от пончика»), а обёртку вокруг неё пропускает — если она сама skippable и её входы не изменились. Это объясняет, почему вынос чтения state глубже и в лямбды уменьшает объём рекомпозиции. movableContentOf — отдельный инструмент: он перемещает кусок UI в другое место дерева (например, между Row и Column при смене ориентации), сохраняя его состояние и не пересоздавая, вместо того чтобы рантайм счёл его новым по позиции и сбросил.

@Composable
fun ChatScreen(messages: List<Message>) {
    val listState = rememberLazyListState()

    // derivedStateOf: пересчёт только когда меняется РЕЗУЛЬТАТ (true/false),
    // а не на каждый пиксель скролла — без него рекомпозиция на каждый кадр
    val showScrollToTop by remember {
        derivedStateOf { listState.firstVisibleItemIndex > 0 }
    }

    Box {
        LazyColumn(state = listState) {
            items(messages, key = { it.id }) { msg ->
                MessageRow(msg)
            }
        }
        if (showScrollToTop) {
            FloatingActionButton(
                onClick = { /* scroll to top */ },
                modifier = Modifier.align(Alignment.BottomEnd)
            ) { Icon(Icons.Default.ArrowUpward, null) }
        }
    }
}

Объяснение: без derivedStateOf чтение listState.firstVisibleItemIndex напрямую подписывает весь ChatScreen на каждое изменение offset — рекомпозиция на каждый пиксель скролла. derivedStateOf { ... > 0 } создаёт производное состояние, которое эмитирует изменение только при смене булевого результата (false → true и обратно). Рекомпозиция ChatScreen происходит максимум дважды за весь скролл.

Задача 2. Реализуй composable TimerScreen, который при появлении на экране запускает таймер (каждую секунду увеличивает счётчик), а при исчезновении останавливает его. При смене параметра startFrom таймер должен сброситься и считать заново с нового значения. Использование побочных эффектов — обязательная часть решения.

Показать решение 
@Composable
fun TimerScreen(startFrom: Int) {
    var elapsed by remember(startFrom) { mutableStateOf(startFrom) }

    // LaunchedEffect(startFrom): перезапускается при смене startFrom,
    // отменяется при уходе composable из дерева
    LaunchedEffect(startFrom) {
        while (true) {
            delay(1000L)
            elapsed++
        }
    }

    Text(
        text = "Прошло: $elapsed сек",
        style = MaterialTheme.typography.headlineMedium
    )
}

Объяснение: LaunchedEffect(startFrom) запускает корутину при первой композиции и перезапускает её (отменяя предыдущую) при каждой смене startFrom. При удалении composable из дерева корутина отменяется автоматически — утечки нет. remember(startFrom) { mutableStateOf(startFrom) } сбрасывает счётчик синхронно при смене ключа, так что UI не покажет старое значение в момент перезапуска эффекта.

Задача 3. Команда жалуется, что экран со списком пользователей тормозит при прокрутке. Профайлер показывает постоянную рекомпозицию UserFeed. Найди все проблемы и исправь.

@Composable
fun UserFeed(

Показать решение 
// 1. @Immutable: все поля val с примитивными/String типами
//    — компилятор теперь считает User стабильным
@Immutable
data class User(val id: Long, val name: String, val bio: String)

// 2. ImmutableList вместо List: List нестабилен по умолчанию,
//    передача List<T> делает UserFeed не-skippable
@Composable
fun UserFeed(
    users: ImmutableList<User>,  // из kotlinx.collections.immutable
    onFollow: (Long) -> Unit
) {
    LazyColumn {
        // 3. key по стабильному id: иначе при удалении/вставке
        //    состояние "съезжает" на соседей
        items(users, key = { it.id }) { user ->
            // 4. Лямбда захватывает user, но со strong skipping (дефолт
            //    с Kotlin 2.0.20) авто-мемоизируется и не пересоздаётся
            UserCard(user, onFollow = { onFollow(user.id) })
        }
    }
}

@Composable
fun UserCard(user: User, onFollow: () -> Unit) {
    Row {
        Text(user.name)
        Button(onClick = onFollow) { Text("Follow") }
    }
}

Объяснение: три источника проблем. Первый — List<User> нестабилен, UserFeed не-skippable и рекомпозируется при любом изменении родителя. Второй — стабильность самого User: в пределах модуля компилятор Compose выводит её автоматически (все поля — Long/String, они стабильны), а явная аннотация нужна для типа из другого модуля или с невыводимым полем. Со strong skipping (дефолт с Kotlin 2.0.20) и нестабильные типы становятся skippable — рантайм сравнивает их по equals. Третий — отсутствие key в items ломает identity элементов при reorder. После исправлений UserCard пропускает рекомпозицию, если user и onFollow не изменились.

Задача 4. Тема приложения переключается день/ночь через CompositionLocal, объявленный как staticCompositionLocalOf. Пользователи жалуются, что при переключении темы заметно фризит весь экран. Объясни причину и исправь объявление.

Показать решение 
// ❌ static не трекает чтения -> смена значения рекомпозирует
//    ВСЁ содержимое CompositionLocalProvider целиком -> фриз
val LocalTheme = staticCompositionLocalOf { LightTheme }

// ✅ compositionLocalOf трекает чтения -> рекомпозируются
//    только реальные читатели LocalTheme.current
val LocalTheme = compositionLocalOf { LightTheme }

Объяснение: staticCompositionLocalOf не отслеживает чтения, поэтому при provides нового значения рантайм рекомпозирует всё поддерево внутри CompositionLocalProvider — отсюда фриз на каждое переключение. Для значения, меняющегося в рантайме, нужен compositionLocalOf: он хранит значение в snapshot-state и рекомпозирует только тех, кто читал .current. Обратное правило тоже важно: для практически неизменяемых зависимостей (синглтоны, Analytics) static, наоборот, дешевле — на нём нет трекинга чтений.

Задача 5. Composable показывает деталь товара. Внутри LaunchedEffect(Unit) запускается опрос наличия каждые 10 секунд и вызывается переданная лямбда onStockChanged(stock). При навигации между товарами без выхода с экрана (меняется только параметр productId) пользователь видит уведомления о наличии от ПРЕДЫДУЩЕГО товара. Найди баг и предложи два варианта починки.

Показать решение 
// ❌ баг: LaunchedEffect(Unit) не перезапускается при смене productId,
//    захваченные productId и onStockChanged остаются от первого товара
@Composable
fun ProductDetail(productId: String, onStockChanged: (Int) -> Unit) {
    LaunchedEffect(Unit) {
        while (true) {
            val stock = repo.checkStock(productId)   // устаревший productId
            onStockChanged(stock)                    // устаревшая лямбда
            delay(10_000)
        }
    }
}

// ✅ вариант 1: ключ productId -> опрос перезапускается на новый товар
@Composable
fun ProductDetail(productId: String, onStockChanged: (Int) -> Unit) {
    LaunchedEffect(productId) {
        while (true) {
            onStockChanged(repo.checkStock(productId))
            delay(10_000)
        }
    }
}

// ✅ вариант 2: опрос не перезапускаем, но читаем свежие значения
@Composable
fun ProductDetail(productId: String, onStockChanged: (Int) -> Unit) {
    val currentId by rememberUpdatedState(productId)
    val currentCb by rememberUpdatedState(onStockChanged)
    LaunchedEffect(Unit) {
        while (true) {
            currentCb(repo.checkStock(currentId))
            delay(10_000)
        }
    }
}

Объяснение: LaunchedEffect(Unit) запускается один раз и не перезапускается при смене productId, поэтому корутина держит первый productId и первую лямбду onStockChanged — классический устаревший захват. Вариант 1 добавляет productId в ключи: при навигации корутина отменяется и стартует заново для нового товара. Вариант 2 нужен, когда рестартовать сам опрос нежелательно: rememberUpdatedState оборачивает значения в State, и цикл на каждой итерации читает актуальные productId и коллбэк без перезапуска. Выбор зависит от того, надо ли обнулять интервал опроса при смене товара.
