Многопоточность

Корутины и Flow

Самый «глубокий» вопрос блока. От интервьюера ждут не фразу «launch — это запуск корутины», а понимание suspend как стейт-машины (CPS-преобразование), кооперативной отмены, structured concurrency и backpressure в Flow. Предполагается, что базовый слой — потоки и модель памяти — уже знаком (см. раздел «Потоки и память JVM»); здесь — абстракция исполнения поверх потоков.

Middle Senior Частый вопрос

Корутина: suspend как стейт-машина

Корутина — это «приостанавливаемое вычисление»: его можно поставить на паузу в точке suspend-вызова, освободить поток, а позже возобновить — возможно, на другом потоке. Ключ к пониманию — то, что компилятор делает с suspend-функцией.

Continuation и CPS-преобразование

Каждая suspend-функция компилируется в CPS (Continuation-Passing Style): компилятор добавляет скрытый последний параметр Continuation<T> — «что делать с результатом». Сама функция возвращает Any?: либо реальный результат, либо специальный маркер COROUTINE_SUSPENDED, означающий «я приостановилась, поток свободен».

// как пишем мы:
suspend fun loadUser(id: String): User { ... }

// во что превращает компилятор (упрощённо):
fun loadUser(id: String, cont: Continuation<User>): Any?
// вернёт либо User, либо COROUTINE_SUSPENDED

interface Continuation<in T> {
    val context: CoroutineContext
    fun resumeWith(result: Result<T>)   // возобновить (успех или ошибка)
}

Стейт-машина: каждая suspend-точка — это состояние

Тело функции с несколькими suspend-вызовами компилятор разбивает на части по точкам приостановки и сворачивает в стейт-машину: целочисленное поле label хранит текущее состояние, локальные переменные «спиллятся» в поля объекта-continuation, а when(label) прыгает в нужную ветку при возобновлении. По сути это «фальшивый стек вызовов» на куче — цепочка continuation’ов.

stateDiagram-v2 [*] --> L0 L0 --> L1: suspend fetchUser → приостановка L1 --> L2: suspend fetchPosts → приостановка L2 --> Done: возврат результата Done --> [*] note right of L1 : label=1, локали сохранены в Continuation note right of L2 : resumeWith возобновляет с нужного label
suspend-функция как стейт-машина: label кодирует, с какого места возобновлять исполнение
// Иллюстрация: что прячется за линейным suspend-кодом
suspend fun profile(id: String): Profile {
    val user = fetchUser(id)        // точка приостановки 1 (label 0 -> 1)
    val posts = fetchPosts(user)    // точка приостановки 2 (label 1 -> 2)
    return Profile(user, posts)     // финальное состояние
}
// компилятор: один класс-continuation с полем label (0,1,2),
// полями user/posts и when(label){ 0->...; 1->...; 2->... }
💡
Вывод для собеса. Корутина «не блокирует поток» именно потому, что приостановка — это возврат из функции с маркером, а не парковка потока. Освободившийся поток диспетчер отдаёт другим корутинам. Поэтому десятки тысяч корутин живут на пуле из нескольких потоков.
Почему компилятор делает один объект-continuation на всю suspend-функцию, а не по одному на каждую точку приостановки? ★

На всю suspend-функцию компилятор создаёт ровно один объект continuation (наследник ContinuationImpl), а не по одному на каждую suspend-точку. Состояние кодируется целочисленным label, а локальные переменные, которые переживают приостановку, спиллятся в поля этого же объекта. Когда функция вызывает другую suspend-функцию, она передаёт ей этот же continuation как completion — так складывается связанный список continuation’ов, и он играет роль «стека вызовов» на куче.

Смысл переиспользования — экономия аллокаций: глубокая последовательная цепочка suspend-вызовов не плодит объект на каждом шаге. При возобновлении resumeWith поднимает результат вверх по этой цепочке, и каждый кадр прыгает в свою ветку when(label).

Dispatchers и withContext

CoroutineDispatcher решает, на каком потоке или пуле исполнять корутину.

DispatcherПулДля чего
MainГлавный (UI) потокРабота с View, лёгкая оркестрация, обновление состояния.
IOБольшой эластичный пул (по умолчанию до 64 потоков)Блокирующий I/O: сеть, диск, БД, файлы.
DefaultПул размером ≈ числу ядер CPUCPU-bound: парсинг, сортировка, сжатие, вычисления.
UnconfinedНе привязан, стартует в вызывающем потокеСпец-случаи и тесты; в проде почти не нужен.

withContext(dispatcher) — приостанавливаемо переключает контекст на время блока и возвращает результат. Это идиоматичный способ «уйти в IO и вернуться».

suspend fun loadProfile(id: String): Profile = withContext(Dispatchers.IO) {
    val user = api.user(id)          // сеть — на IO
    val posts = api.posts(id)
    Profile(user, posts)             // результат вернётся в исходный контекст
}
// в ViewModel: viewModelScope.launch { val p = loadProfile(id); _state.value = p }
ℹ️
Dispatchers.Main на Android реализован поверх Handler к главному Looper (см. раздел «Looper и Handler»): возобновление корутины — это, по сути, post в очередь главного потока.
IO и Default используют один общий пул потоков. Что тогда значит лимит IO в 64 и зачем limitedParallelism? ★

IO и Default делят общий пул потоков CoroutineScheduler — они отличаются не отдельными пулами, а лимитом параллелизма. У Default лимит — число ядер CPU, у IO — max(64, число ядер). Поэтому переключение withContext(IO) с Default не обязательно меняет физический поток: оно меняет режим квоты на том же пуле, и такое переключение дёшево. Лимит IO в 64 страхует от исчерпания потоков при множестве блокирующих операций.

Dispatchers.IO.limitedParallelism(n) создаёт «вид» поверх того же пула со своим лимитом n — например, чтобы запись в БД не заняла все 64 слота. Тонкость: limitedParallelism на IO не урезает общий лимит пула, а добавляет локальное ограничение поверх него.

launch vs async

Оба — «билдеры» корутин, но с разной семантикой результата.

launchasync
ВозвращаетJob (без значения)Deferred<T> (есть await())
Назначение«Запустил и забыл» — побочный эффектПараллельно посчитать и получить результат
ИсключениеКидается сразу, уходит в CoroutineExceptionHandlerОткладывается до await()
 
// параллельная загрузка двух запросов и ожидание обоих
suspend fun dashboard(): Dashboard = coroutineScope {
    val user = async { api.user() }      // стартуют параллельно
    val feed = async { api.feed() }
    Dashboard(user.await(), feed.await())   // ждём оба результата
}
Корневой async бросает исключение сразу или только на await()? А дочерний внутри coroutineScope? ★

Зависит от того, корневая корутина или дочерняя. У корневого async (GlobalScope.async или async прямо на скоупе верхнего уровня) исключение инкапсулируется в Deferred и всплывает только при await() — до этого момента оно молчит и в CoroutineExceptionHandler не идёт.

У дочернего async внутри coroutineScope/supervisorScope исключение к тому же немедленно распространяется вверх по дереву Job и отменяет родителя и братьев — то есть проявится и без вызова await(). Поэтому ходовое «async откладывает ошибку до await» верно лишь для корневого случая; вложенный async с обычным Job рушит скоуп сразу. Любимый подвох: один и тот же код ведёт себя по-разному в зависимости от того, где запущен async.

Structured concurrency

Главная идея: у каждой корутины есть родитель, и она не может «пережить» свой скоуп. Это даёт автоматическое управление жизненным циклом — никаких «потерянных» корутин, утечек и забытых задач. Скоуп завершится, только когда завершатся все его дети; отмена скоупа отменяет всех детей.

CoroutineScope, Job и иерархия

CoroutineScope привязывает корутины к жизненному циклу (например, viewModelScope отменяется в onCleared()). Внутри контекста живёт Job — он образует дерево: Job запущенной корутины становится потомком Job скоупа.

flowchart TB Scope["CoroutineScope: parent Job"] --> J1["Job: launch A"] Scope --> J2["Job: launch B"] J1 --> J11["Job: вложенный launch"] J1 --> J12["Job: async"]
Иерархия Job: отмена/ошибка родителя каскадом доходит до всех потомков

coroutineScope vs supervisorScope, Job vs SupervisorJob

Разница — в распространении ошибок. С обычным Job (и coroutineScope) падение одного ребёнка отменяет родителя и всех братьев. С SupervisorJob (и supervisorScope) дети изолированы: падение одного не трогает остальных.

АспектcoroutineScope / JobsupervisorScope / SupervisorJob
Ошибка ребёнкаОтменяет родителя и всех братьевИзолирована, братья продолжают работу
Когда уместноСвязанные подзадачи: «всё или ничего»Независимые задачи (несколько виджетов, параллельные загрузки)
Где обрабатывать ошибкуНа уровне родительского скоупаВ каждом дочернем launch (свой handler/try)
 
// независимые секции: падение одной не рушит экран целиком
supervisorScope {
    launch { loadHeader() }     // если упадёт —
    launch { loadFeed() }       // эта продолжит работать
    launch { loadAds() }
}
Что будет, если внутри дочернего скоупа передать в билдер новый Job() — launch(Job()) { ... }? ★

Передача нового Job() в билдер разрывает связь родитель-потомок: переданный Job становится родителем новой корутины вместо Job скоупа, и корутина выпадает из дерева structured concurrency. Отсюда последствия — отмена исходного скоупа больше не отменит эту корутину (утечка), скоуп не дождётся её завершения, а исключения не пойдут по нормальному дереву к родителю.

Это классическая ошибка «я хотел изолировать ошибку, добавив Job() в launch». Для изоляции без разрыва иерархии нужен SupervisorJob у самого скоупа или supervisorScope, а не подмена Job в дочернем билдере. Правило: меняй в контексте дочерней корутины что угодно — диспетчер, имя, — но Job обычно оставляй наследоваться от родителя.

Кооперативная отмена

Отмена в корутинах — кооперативная: никто не «убивает» поток. cancel() лишь выставляет флаг и бросает CancellationException в ближайшей точке приостановки. Все библиотечные suspend-функции (delay, I/O, каналы) проверяют отмену сами. Но чисто вычислительный цикл без suspend нужно делать отменяемым вручную.

val job = scope.launch {
    var i = 0
    while (isActive) {                 // проверяем флаг отмены кооперативно
        i = heavyStep(i)               // CPU-работа без suspend-точек
        // или: ensureActive() — кинет CancellationException, если отменено
    }
}
job.cancel()                            // отмена сработает на следующей итерации
⚠️
Не глушите CancellationException. try/catch(e: Exception) вокруг suspend-кода случайно ловит и отмену — корутина «не отменится». Ловите конкретные типы либо пробрасывайте CancellationException дальше. Очистку ресурсов, которая должна выполниться даже при отмене, заворачивайте в withContext(NonCancellable) { ... }.

CoroutineExceptionHandler

Необработанные исключения из launch всплывают до корневого скоупа и попадают в CoroutineExceptionHandler (если он есть в контексте) — это «последний рубеж» для логирования/краш-репорта. Для async handler не вызывается: там ошибку получают через await().

val handler = CoroutineExceptionHandler { _, e ->
    Log.e("APP", "необработанная ошибка корутины", e)
}
val scope = CoroutineScope(SupervisorJob() + Dispatchers.Main + handler)
scope.launch { riskyWork() }            // если упадёт — попадёт в handler

Состояния Job

Отмена становится понятнее, если держать в голове машину состояний Job. Корутина рождается в New (или сразу Active), работает в Active, после тела ждёт детей в Completing и заканчивает в Completed. Ветка отмены ведёт из Active/Completing в Cancelling, а затем в Cancelled. Именно в Cancelling любой обычный suspend-вызов бросает CancellationException — поэтому очистку с приостановкой и заворачивают в NonCancellable.

stateDiagram-v2 [*] --> New New --> Active: start Active --> Completing: тело завершилось Completing --> Completed: дети завершились Active --> Cancelling: "cancel() / ошибка" Completing --> Cancelling: "cancel() / ошибка" Cancelling --> Cancelled: дети завершились Completed --> [*] Cancelled --> [*] note right of Cancelling : suspend-вызовы кидают CancellationException
Жизненный цикл Job: в Cancelling приостановка уже невозможна без NonCancellable
Почему отменённая корутина всё ещё «активна» внутри finally, и как закрыть ресурс через сеть при отмене? ★

После отмены Job переходит в Cancelling, и любой suspend-вызов в этом состоянии немедленно бросает CancellationException — обычный delay или сетевой вызов в finally сразу упадёт, и очистка не выполнится. Чтобы провести suspend-работу при отмене, блок оборачивают в withContext(NonCancellable): он даёт Job, который не в состоянии отмены, и suspend-вызовы внутри проходят.

Несуспендящая очистка — закрыть файл, освободить лок — в finally работает и без этого. И нюанс: NonCancellable — анти-паттерн для долгой работы; его место — короткая обязательная финализация, иначе отмена перестаёт быть отзывчивой.

Свой delay через suspendCancellableCoroutine

Любимое задание на «понимание корутин изнутри»: написать собственный delay, не блокируя поток. Решение — мост между миром коллбэков и корутинами: suspendCancellableCoroutine даёт Continuation, который мы возобновим из коллбэка таймера. Критично — корректно обработать отмену через invokeOnCancellation, иначе утечёт запланированная задача.

CustomDelay.kt
import kotlinx.coroutines.*
import java.util.concurrent.Executors
import java.util.concurrent.TimeUnit

// один общий планировщик на приложение
private val scheduler = Executors.newSingleThreadScheduledExecutor { r ->
    Thread(r, "custom-delay").apply { isDaemon = true }
}

suspend fun customDelay(timeMillis: Long): Unit =
    suspendCancellableCoroutine { cont ->                // даёт CancellableContinuation
        // планируем возобновление через timeMillis, НЕ блокируя поток
        val future = scheduler.schedule({
            cont.resume(Unit) {}                          // возобновляем корутину
        }, timeMillis, TimeUnit.MILLISECONDS)

        // если корутину отменят, пока мы «спим» — отменяем задачу таймера
        cont.invokeOnCancellation {
            future.cancel(false)                          // нет утечки
        }
    }

// проверка кооперативной отмены:
suspend fun demo() = coroutineScope {
    val job = launch { customDelay(5_000); println("проснулся") }
    delay(100)
    job.cancel()    // future.cancel сработает, "проснулся" не напечатается
}
💡
Что проверяет интервьюер. (1) Понимаешь, что приостановка — это сохранённый Continuation, который кто-то должен возобновить. (2) Не блокируешь поток (Thread.sleep — провал). (3) Чистишь ресурсы при отмене (invokeOnCancellation) — иначе таймер «выстрелит» в отменённую корутину или утечёт. Обратите внимание: планировщик здесь — это ScheduledExecutorService из раздела «Executor и пулы потоков».
Чем suspendCancellableCoroutine отличается от suspendCoroutine и что будет, если в обёртке коллбэка взять второй? ★

suspendCoroutine даёт обычный Continuation, который нельзя отменить: даже если окружающую корутину отменят, приостановленный suspendCoroutine продолжит ждать resume и не отреагирует на отмену — корутина становится неотменяемой в этой точке, а запланированная задача утекает.

suspendCancellableCoroutine даёт CancellableContinuation: он проверяет отмену при возобновлении, бросит CancellationException, если корутину отменили во время ожидания, и даёт invokeOnCancellation для снятия слушателя или таймера. Практическое правило: любой мост из коллбэков строят на suspendCancellableCoroutine, чтобы сохранить кооперативную отмену; suspendCoroutine уместен лишь когда операция гарантированно мгновенна и неотменяема.

Flow: cold и hot потоки данных

Flow — асинхронный поток значений поверх корутин. По умолчанию он холодный (cold): код-продюсер не выполняется, пока на поток не подписались через терминальный оператор (collect), и для каждого подписчика запускается заново. Горячие (hot) потоки (StateFlow, SharedFlow) эмитят независимо от подписчиков и шарят значения между ними.

// холодный Flow: тело запускается на каждый collect
fun ticks(): Flow<Int> = flow {
    var i = 0
    while (true) { emit(i++); delay(1000) }   // продюсер
}.flowOn(Dispatchers.Default)                  // upstream — на Default

// сборка (потребитель) на главном потоке:
scope.launch { ticks().collect { value -> render(value) } }

StateFlow / SharedFlow vs LiveData

StateFlow — горячий поток с одним актуальным значением (state holder), всегда имеет initial. SharedFlow — настраиваемая шина событий (replay/buffer). Сравнение с LiveData — частый вопрос в Android-контексте.

Сбор Flow на Android — через repeatOnLifecycle(STARTED) (или flowWithLifecycle), а не голый launch в onCreate. Иначе корутина продолжит собирать поток, когда экран в фоне — лишняя работа и риск падений при обновлении невидимого UI.

Операторы, flowOn и backpressure

Промежуточные операторы (map, filter, transform, debounce) ленивы и собираются в конвейер. flowOn(dispatcher) меняет контекст вышестоящей части потока (продюсера и операторов до него), не трогая collect — отсюда строгое разделение «producer на IO, collect на Main».

Backpressure — когда продюсер быстрее потребителя. У Flow базовая модель — приостановка (emit ждёт, пока соберут предыдущее). Управлять буферизацией можно операторами:

ОператорПоведениеКогда

conflate()Хранит только последнее значение, промежуточные дропаютсяВажно лишь актуальное состояние (прогресс, позиция)

callbackFlow — мост из коллбэков в Flow

Когда API отдаёт данные через слушатель (геолокация, BLE, Firebase), его оборачивают в callbackFlow: эмитим из коллбэка через trySend, а в awaitClose снимаем слушатель при отмене сбора.

fun locationUpdates(client: LocationClient): Flow<Location> = callbackFlow {
    val listener = object : LocationListener {
        override fun onLocation(loc: Location) { trySend(loc) }   // эмиссия из коллбэка
    }
    client.register(listener)
    awaitClose { client.unregister(listener) }   // обязательно снять при отмене
}
Почему flowOn меняет контекст только upstream, и что нарушает context preservation при withContext внутри flow { }? ★

Flow гарантирует context preservation: emit обязан вызываться из того же контекста корутины, в котором запущен сам блок flow { }. Поэтому менять контекст продюсера руками через withContext(...) { emit(...) } нельзя — на исполнении Flow кинет IllegalStateException про «Flow invariant is violated».

Легальный способ сменить контекст продюсера — оператор flowOn(dispatcher): он меняет контекст всей upstream-части (источник плюс операторы выше него), а downstream-операторы и collect остаются в контексте коллектора. Внутри flowOn вставляет канал-буфер между upstream и downstream и запускает upstream в отдельной корутине с заданным диспетчером. Отсюда идиома «продюсер на IO через flowOn, collect на Main».

Композиция потоков: flatMap, combine, zip

Один из самых ходовых блоков Flow-собеса. Как только источников становится несколько — type-ahead поиск, объединение нескольких StateFlow в один UI-state — встаёт вопрос: как развернуть Flow<Flow<T>> и как слить несколько потоков. Глава выше дала map/filter/debounce; здесь — операторы, которые любят спрашивать.

flatMap-семейство: три стратегии разворачивания

Когда на каждый элемент upstream нужно подписаться на новый внутренний Flow, получается Flow<Flow<T>>. Разворачивают его три оператора, и различаются они конкурентностью.

ОператорПорядокКонкурентностьОтмена предыдущегоТипичный кейс
flatMapConcatСохранёнОдин за другимНет, ждёт завершенияПоследовательные шаги
flatMapMergeНе гарантированДо concurrency (по умолчанию 16)Нет, сливает всеПараллельные запросы
flatMapLatest—Один активныйДа, отменяет старыйПоиск-по-вводу
⚠️
Дефолт flatMapMerge — 16. Параметр concurrency по умолчанию равен DEFAULT_CONCURRENCY = 16. На потоке с тысячами элементов это и узкое место, и риск исчерпания ресурсов — задавайте лимит осознанно.

combine vs zip vs merge

zip — попарное соединение: ждёт по одному значению от каждого из двух потоков, эмитит пару и завершается, когда завершился любой из потоков; темп задаёт более медленный. combine эмитит при изменении любого из потоков, комбинируя новое значение с последними известными значениями остальных — поэтому combine молчит, пока каждый вход не выдал хотя бы по одному значению (первая эмиссия ждёт всех). merge же просто сливает эмиссии нескольких потоков в один без комбинирования.

// несколько источников состояния → один UI-state
val uiState: Flow<UiState> = combine(user, cart, promo) { u, c, p ->
    UiState(u, c, p)               // эмитит на изменение любого; ждёт первую от всех трёх
}

// строгие пары «один к одному» из двух параллельных запросов
val pairs: Flow<Pair<A, B>> = flowA.zip(flowB) { a, b -> a to b }

Сквозной пример: type-ahead поиск

Собирается из стандартных операторов в нужном порядке: погасить дребезг ввода, отсечь повтор запроса, отменить предыдущий сетевой вызов на новом вводе.

fun search(queries: Flow<String>): Flow<List<Result>> = queries
    .filter { it.isNotBlank() }       // пустые игнорируем
    .debounce(300)                    // выждать паузу ввода
    .distinctUntilChanged()           // не повторять тот же запрос
    .flatMapLatest { q ->             // отменить старый запрос при новом
        flow { emit(api.search(q)) }
    }
💡
Почему именно flatMapLatest. Только он отменяет предыдущий внутренний поток (текущий запрос к API) при новом вводе. flatMapMerge оставил бы гонку устаревших ответов, flatMapConcat заставил бы ждать завершения каждого. Порядок операторов важен: filter/debounce/distinctUntilChanged до flatMapLatest.
В чём подвох первой эмиссии у combine по сравнению с zip? ★

combine комбинирует новое значение одного потока с последними известными значениями остальных. Пока хотя бы один из входов не выдал ни одного значения, «последнего известного» у него нет — и combine не эмитит вообще ничего. Если один из источников долго молчит (например, StateFlow без значения, собранный из cold-потока без initial), весь комбинированный поток будет молчать, даже когда остальные активно эмитят. Частая ловушка: UI «не обновляется», потому что один из входных потоков ещё не дал первое значение.

zip ведёт себя иначе — он буферизует по парам и ждёт ровно по одному новому значению от каждого, поэтому пропуск первой эмиссии у него невозможен по определению, но и реактивного «обновись на любой источник» он не даёт.

stateIn / shareIn: холодный Flow в горячий

Топовый практический вопрос Senior-собеса по Android: как отдать поток из репозитория в UI, не перезапуская дорогой upstream на каждый collect и на каждый поворот экрана. Ответ — превратить cold Flow в hot.

Cold Flow перезапускает продюсера на каждого подписчика. Если на один и тот же поток подпишутся два места UI, запрос уйдёт дважды. stateIn и shareIn запускают одну общую upstream-корутину в переданном скоупе и шарят её результат между всеми подписчиками. Разница: stateIn отдаёт StateFlow и требует initialValue, shareIn отдаёт SharedFlow и настраивается через replay.

Политика startedКогда стартует upstreamКогда останавливается
EagerlyСразу при вызовеТолько с концом скоупа
LazilyНа первом подписчикеТолько с концом скоупа
WhileSubscribed(t)На первом подписчикеЧерез t мс после ухода последнего
 
// стандартная отдача состояния из ViewModel
val state: StateFlow<UiState> = repository.dataFlow
    .map { it.toUiState() }
    .stateIn(
        scope = viewModelScope,
        started = SharingStarted.WhileSubscribed(5_000), // переживает поворот
        initialValue = UiState.Empty
    )

WhileSubscribed(5000) держит upstream активным ещё 5 секунд после ухода последнего подписчика. Это переживает смену конфигурации: при повороте экрана подписчик на миг отписывается и тут же переподписывается — за 5-секундное окно upstream не успевает остановиться, дорогой поток не перезапускается и кэш не теряется. При этом в фоне, когда подписчиков надолго нет, upstream всё же остановится и не будет жечь батарею.

stateDiagram-v2 [*] --> Stopped Stopped --> Active: первый подписчик Active --> GracePeriod: последний отписался GracePeriod --> Active: новый подписчик за 5с GracePeriod --> Stopped: таймаут 5000мс note right of GracePeriod : поворот экрана попадает в это окно
WhileSubscribed: 5-секундное окно гасит перезапуск upstream при повороте
ℹ️
WhileSubscribed работает только если сам сбор на UI тоже останавливается в фоне — через repeatOnLifecycle(STARTED). Иначе подписчик никогда не «уходит», окно не запускается, и поведение сводится к Lazily.
Почему именно 5000 мс, а не 0 или, скажем, 60_000? ★

Число выбирают как компромисс между двумя крайностями. Слишком маленький timeout (0) означает, что upstream остановится в момент отписки — а при повороте экрана отписка и переподписка идут с зазором, и поток перезапустится, потеряв смысл шаринга. Слишком большой (минута) держит дорогой upstream активным, когда экран давно ушёл в фон: лишний трафик и расход батареи, ровно то, от чего уходили.

5 секунд с запасом перекрывают типичную смену конфигурации (она занимает доли секунды) и при этом достаточно малы, чтобы реальный уход с экрана быстро останавливал поток. Это не магическая константа из SDK, а сложившаяся в гайдах Android практика — её можно подстроить под конкретный источник.

Тестирование корутин и Flow

Senior-тема, которую почти всегда трогают: как протестировать код с delay, не ожидая реального времени, и как подменить Dispatchers.Main в тестах ViewModel.

runTest из kotlinx-coroutines-test запускает тело теста в TestScope с виртуальным временем: delay не ждёт реально, а двигает виртуальные часы. Тест с delay(10_000) проходит мгновенно.

ДиспетчерЗапуск корутинКак продвинуть
StandardTestDispatcher (по умолчанию)Отложенный, в очередь планировщикаadvanceUntilIdle(), advanceTimeBy(), runCurrent()
UnconfinedTestDispatcherЖадный, до первой реальной приостановкиЧасто не нужно
 
@Test
fun loadsUser() = runTest {                       // виртуальное время
    val vm = UserViewModel(repo)
    vm.load()                                      // корутина в очереди (Standard)
    advanceUntilIdle()                             // выполнить всё до простоя
    assertEquals(UiState.Loaded(user), vm.state.value)
}

// подмена главного диспетчера в @Before/@After:
Dispatchers.setMain(StandardTestDispatcher())
// ...
Dispatchers.resetMain()
⚠️
Самый частый подвох. С StandardTestDispatcher запущенные корутины не стартуют сами — забыли advanceUntilIdle(), и проверка читает состояние до того, как корутина что-то сделала. С UnconfinedTestDispatcher корутины стартуют жадно, и продвигать время обычно не надо — но и точный порядок шагов вы уже не контролируете.

Для Flow результат собирают в список (flow.toList() на конечном потоке) или используют библиотеку Turbine (flow.test { ... awaitItem() }) — она удобна для горячих и бесконечных потоков, где toList() не завершится.

Почему runTest с StandardTestDispatcher может «зависнуть» или упасть по таймауту, хотя реального ожидания нет? ★

Виртуальное время продвигается, только пока есть запланированные на него задачи. Если корутина ждёт чего-то, что в виртуальном времени не запланировано — например, реальный Channel, внешний коллбэк или поток на другом, не-тестовом диспетчере, — планировщик доходит до простоя, а ожидаемое событие так и не наступает. runTest в этом случае срабатывает по собственному таймауту (по умолчанию около 60 секунд реального времени) и падает.

Лечится тем, чтобы весь тестируемый код шёл через тестовый TestDispatcher (включая Dispatchers.setMain и инъекцию диспетчеров в классы), а не уходил на реальные Default/IO. Тогда всё, что должно произойти, оказывается в одном виртуальном расписании, и advanceUntilIdle() его честно прогоняет.

Потоки vs корутины

Итоговое сравнение — частый «закрывающий» вопрос темы. Корутины не «быстрее» потоков сами по себе; они дешевле по памяти и удобнее по модели исполнения.

КритерийПотоки (Thread)Корутины
Стоимость созданияДорого: ~1 МБ стека, объект ядра ОСДёшево: объект на куче (сотни байт), тысячи на пуле
Переключение контекстаЧерез планировщик ОС (дорого, режим ядра)Приостановка = возврат из функции (в user-space, дёшево)
БлокировкаБлокирует поток целикомПриостановка освобождает поток для других корутин
МасштабСотни–тысячи (упор в память)Сотни тысяч на нескольких потоках
ЧитаемостьКоллбэки/Future → «callback hell»Линейный последовательный код с suspend
ОтменаСложно/небезопасно (interrupt)Кооперативная, структурированная, каскадная
Управление жизненным цикломВручнуюStructured concurrency (скоупы)
ℹ️
Точная формулировка для собеса. Корутина — не поток, а абстракция исполнения поверх потоков. Множество корутин мультиплексируются на пул потоков диспетчера; «лёгкость» — следствие того, что приостановка не держит поток. Поэтому IO-bound нагрузка (много ждущих запросов) выигрывает особенно сильно.
Если корутины мультиплексируются на пул потоков, почему блокирующий вызов на Dispatchers.Default опаснее, чем кажется? ★

Корутины кооперативно делят небольшой пул: у Default это всего ~число ядер потоков. Блокирующий вызов — Thread.sleep, синхронный сетевой или JDBC-вызов, тяжёлый цикл без suspend — занимает физический поток целиком и не возвращает его диспетчеру, потому что блокировка это не приостановка. Достаточно нескольких таких корутин, чтобы исчерпать все потоки Default и застопорить весь CPU-bound пул: остальные корутины просто не получат поток (thread starvation).

Блокирующий I/O уводят на Dispatchers.IO (лимит до 64 и эластичность под блокировки), а CPU-цикл делают кооперативно отменяемым и при нужде периодически зовут yield(). Корутина не превращает блокирующий код в неблокирующий — она лишь даёт дешёвую приостановку там, где код действительно suspend.

Частые вопросы интервьюера

Как suspend работает под капотом? Senior

Компилятор применяет CPS: добавляет скрытый параметр Continuation и заставляет функцию возвращать Any? — либо результат, либо маркер COROUTINE_SUSPENDED. Тело с несколькими suspend-точками превращается в стейт-машину: поле label кодирует состояние, локальные переменные спиллятся в поля continuation, а when(label) возобновляет с нужного места через resumeWith. Поэтому приостановка не блокирует поток — это возврат из функции, а не парковка потока.

Чем launch отличается от async? Middle

launch возвращает Job и предназначен для «запустил и забыл» — побочный эффект без результата; необработанное исключение сразу всплывает в CoroutineExceptionHandler. async возвращает Deferred<T> с await() — для параллельного вычисления значения; исключение откладывается до await(). Для параллельной загрузки нескольких значений берут несколько async внутри coroutineScope и ждут все через await().

Чем Job отличается от SupervisorJob? Middle+

Разница в распространении ошибок между детьми. С обычным Job падение любого ребёнка отменяет родителя и всех братьев («всё или ничего»). С SupervisorJob дети изолированы: падение одного не трогает остальных — удобно для независимых задач (несколько виджетов экрана, параллельные несвязанные загрузки). При этом обрабатывать ошибку нужно в каждом дочернем launch (или через CoroutineExceptionHandler), а не на уровне скоупа.

Почему корутина не отменяется и как это чинить? Middle+

Отмена кооперативная: чисто вычислительный цикл без suspend-точек её «не замечает». Нужно проверять isActive/вызывать ensureActive() или периодически yield(). Вторая частая причина — проглоченный CancellationException: широкий catch(e: Exception) вокруг suspend-кода ловит и отмену. Пробрасывайте CancellationException дальше; а финализацию, которая обязана выполниться при отмене, оборачивайте в withContext(NonCancellable).

StateFlow vs SharedFlow vs LiveData — когда что? Middle+

StateFlow — для состояния экрана: всегда есть актуальное значение, дедуплицирует одинаковые, начальное обязательно. SharedFlow — для одноразовых событий (навигация, snackbar): настраиваемый replay/buffer, без дедупликации. LiveData — lifecycle-aware из коробки, но привязан к главному потоку и беднее операторами; в новом коде его место занимают Flow + repeatOnLifecycle. Главный нюанс: Flow на Android собирают через repeatOnLifecycle(STARTED), иначе сбор продолжится в фоне.

Чем cold Flow отличается от hot? Middle

Cold (flow { }) ленив: продюсер не работает без подписчика и перезапускается для каждого collect — у каждого подписчика свой независимый поток данных. Hot (StateFlow/SharedFlow, каналы) эмитит независимо от наличия подписчиков и шарит значения между всеми. Cold подходит для «запросов по требованию» (вызов API), hot — для общего состояния/шины событий, живущей дольше отдельной подписки.

Как написать свой delay, не блокируя поток? Senior

Через suspendCancellableCoroutine: получаем CancellableContinuation, планируем его возобновление через ScheduledExecutorService (а не Thread.sleep — это бы заблокировало поток), и в invokeOnCancellation отменяем запланированную задачу таймера. Так корутина приостанавливается, поток свободен, а при отмене скоупа таймер корректно снимается без утечки. Это демонстрирует понимание, что приостановка — это сохранённый Continuation, который кто-то обязан возобновить.

Что такое stateIn/shareIn и зачем SharingStarted.WhileSubscribed(5000)? Senior

stateIn превращает холодный Flow в горячий StateFlow, shareIn — в SharedFlow; оба запускают одну общую upstream-корутину в переданном скоупе и шарят её результат между подписчиками, вместо перезапуска cold-потока на каждый collect. Параметр started задаёт политику запуска: Eagerly (стартует сразу и живёт весь скоуп), Lazily (стартует на первом подписчике и не останавливается), WhileSubscribed (живёт, пока есть подписчики). WhileSubscribed(5000) держит upstream активным ещё 5 секунд после ухода последнего подписчика — это переживает поворот экрана, когда подписчик на миг отписывается и тут же переподписывается, не перезапуская дорогой upstream и не теряя кэш. stateIn к тому же требует initialValue.

Чем различаются flatMapConcat, flatMapMerge и flatMapLatest? Middle+

Все три разворачивают Flow<Flow<T>>, но по-разному обращаются с конкурентностью. flatMapConcat последователен: ждёт полного завершения внутреннего потока, прежде чем взять следующий элемент upstream — порядок сохранён. flatMapMerge запускает внутренние потоки конкурентно и сливает их эмиссии, порядок не гарантирован; concurrency ограничивает число одновременных внутренних потоков, по умолчанию 16 (DEFAULT_CONCURRENCY). flatMapLatest при новом значении upstream отменяет предыдущий внутренний поток и стартует новый — основа поиска-по-вводу. Аналогично есть mapLatest/collectLatest.

Чем combine отличается от zip, и где у combine подвох с первой эмиссией? Middle+

zip — попарное соединение: ждёт по одному значению от каждого из двух потоков, эмитит пару и завершается, когда завершился любой из потоков; темп задаёт более медленный. combine эмитит при изменении любого из потоков, комбинируя его новое значение с последними известными значениями остальных — поэтому combine ничего не эмитит, пока каждый из входных потоков не выдал хотя бы по одному значению. zip — когда нужны строго пары «один к одному» (объединить два параллельных запроса), combine — когда нужно реактивное состояние из нескольких источников (несколько StateFlow в один UI-state).

Как тестировать корутины: runTest, виртуальное время, Standard vs Unconfined? Senior

runTest запускает тело теста в TestScope с виртуальным временем: delay не ждёт реально, а двигает виртуальные часы — тест с delay(10_000) проходит мгновенно. По умолчанию используется StandardTestDispatcher: запущенные корутины не стартуют сразу, а кладутся в очередь планировщика; их прогоняют advanceUntilIdle(), advanceTimeBy(ms) или runCurrent(). UnconfinedTestDispatcher, наоборот, запускает корутины жадно до первой реальной приостановки. Для подмены Dispatchers.Main в ViewModel-тестах ставят Dispatchers.setMain(testDispatcher).

Как настроить буфер SharedFlow: replay, extraBufferCapacity, onBufferOverflow? Senior

MutableSharedFlow настраивается тремя параметрами. replay — сколько последних значений переотдаётся новому подписчику (по умолчанию 0). extraBufferCapacity — дополнительный буфер сверх replay для развязки эмиттера и медленных подписчиков. onBufferOverflow — что делать, когда буфер (replay+extra) полон: SUSPEND (по умолчанию, emit приостанавливается — backpressure), DROP_OLDEST или DROP_LATEST. При SUSPEND и нулевых буферах emit приостанавливается, пока все текущие подписчики не примут значение, а tryEmit вернёт false. StateFlow — частный случай SharedFlow с replay=1, конфляцией по equals и стратегией, эквивалентной DROP_OLDEST.

Почему GlobalScope — антипаттерн и чем его заменить? Middle+

GlobalScope не привязан ни к какому жизненному циклу: запущенные в нём корутины живут, пока жив процесс, и никем не отменяются — прямой путь к утечкам (корутина переживает экран/ViewModel) и работе вне structured concurrency (ошибки и отмена не каскадируются). В Android вместо него берут lifecycle-aware скоупы: viewModelScope (отменяется в onCleared), lifecycleScope, а для слоя данных — собственный CoroutineScope с SupervisorJob, хранящийся в классе и отменяемый явно. GlobalScope оправдан лишь для редких задач уровня всего приложения, осознанно живущих весь процесс.

Почему StateFlow «теряет» промежуточные значения и пропускает повторы? Middle+

StateFlow хранит ровно одно последнее значение и конфлирует обновления: если значения меняются быстрее, чем подписчик их собирает, промежуточные теряются — подписчик увидит только актуальное (как conflate). Вдобавок StateFlow сравнивает новое значение со старым через Any.equals (как distinctUntilChanged) и не эмитит, если они равны — повтор того же значения «проглатывается». Следствие: StateFlow не годится для одноразовых событий (два одинаковых клика, повторная навигация потеряются) — для событий берут SharedFlow без конфляции. И из-за equals важно, чтобы UI-state был корректным data-классом, иначе обновления могут не доходить.

Задача 1. Реализуйте функцию parallelMap, которая принимает список элементов и suspend-преобразование, запускает все преобразования параллельно и возвращает список результатов в исходном порядке. При ошибке любого из элементов функция должна немедленно отменить остальные и пробросить исключение.

Показать решение 
suspend fun <T, R> List<T>.parallelMap(
    transform: suspend (T) -> R
): List<R> = coroutineScope {
    map { item -> async { transform(item) } }
        .map { it.await() }
}

Объяснение: coroutineScope даёт семантику «всё или ничего» — если любой async-блок упадёт с исключением, скоуп отменит все остальные дочерние корутины и пробросит ошибку наружу. Вызов async { ... } для каждого элемента запускает преобразования параллельно; второй map { it.await() } собирает результаты в том же порядке, что и исходный список — порядок гарантирован, потому что мы обходим список Deferred в том же порядке.

Задача 2. Найдите и исправьте баг в коде ниже — функция должна возвращать результат только самого быстрого из двух источников, отменяя медленный. Код компилируется, но работает неверно.

suspend fun fastest(
    sourceA: suspend () -> String,
    sourceB: suspend () -> String
): String = coroutineScope {
    val a = async { sourceA() }
    val b = async { sourceB() }
    val result = select<String> {
        a.onAwait { it }
        b.onAwait { it }
    }
    result
}
// Проблема: медленная корутина продолжит работать после возврата.
Показать решение 
suspend fun fastest(
    sourceA: suspend () -> String,
    sourceB: suspend () -> String
): String = coroutineScope {
    val a = async { sourceA() }
    val b = async { sourceB() }
    val result = select<String> {
        a.onAwait { it }
        b.onAwait { it }
    }
    // Явно отменяем обе — проигравшая ещё жива
    a.cancel()
    b.cancel()
    result
}

Объяснение: select снимает подписку со второго Deferred, но не отменяет корутину — та продолжает выполняться в фоне. После получения результата нужно явно вызвать cancel() на обоих. Вызов на уже завершённом Deferred — no-op, поэтому безопасно отменять оба без проверки. Альтернатива — raceN из библиотеки Arrow или использование channelFlow с first().

Задача 3. Реализуйте оператор throttleLatest(windowMs: Long) для Flow<T>: в течение окна windowMs оператор пропускает все значения, кроме последнего, и эмитит его по истечении окна. Если за всё окно пришло одно значение, эмитит его сразу по истечении. Используйте только стандартный API корутин без сторонних библиотек.

Показать решение 
fun <T> Flow<T>.throttleLatest(windowMs: Long): Flow<T> = channelFlow {
    var lastValue: T? = null
    var hasValue = false
    val mutex = kotlinx.coroutines.sync.Mutex()

    var timerJob: kotlinx.coroutines.Job? = null

    collect { value ->
        mutex.withLock {
            @Suppress("UNCHECKED_CAST")
            lastValue = value as T
            hasValue = true
        }
        if (timerJob?.isActive != true) {
            timerJob = launch {
                delay(windowMs)
                mutex.withLock {
                    if (hasValue) {
                        send(lastValue as T)
                        hasValue = false
                    }
                }
            }
        }
    }
    // эмитим оставшееся значение, если таймер ещё не сработал
    timerJob?.join()
}

Объяснение: channelFlow позволяет эмитить из дочерних корутин. При каждом входящем значении запоминаем «последнее» в переменной и запускаем таймер только если он ещё не активен — таймер по истечении windowMs эмитит накопленное. Mutex защищает от гонки между потоком-продюсером и таймером. В отличие от debounce, здесь таймер запускается по первому значению окна и не сбрасывается при новых — это throttle, а не debounce.

Задача 4. Во ViewModel написано val state = repository.dataFlow.stateIn(viewModelScope, SharingStarted.Eagerly, Data.EMPTY). При повороте экрана виден лишний сетевой запрос upstream, а в фоне поток жжёт батарею. Объясните причину и исправьте.

Показать решение 
val state = repository.dataFlow.stateIn(
    scope = viewModelScope,
    started = SharingStarted.WhileSubscribed(5_000),
    initialValue = Data.EMPTY
)
// плюс на UI собирать через repeatOnLifecycle(STARTED)

Объяснение: Eagerly запускает upstream сразу и держит его весь жизненный цикл скоупа, включая фон, когда подписчиков нет. WhileSubscribed(5000) держит upstream активным только при наличии подписчиков плюс 5-секундный grace-период — он переживает пересоздание подписчика при повороте, не перезапуская поток, но в фоне поток останавливается. Без сбора через repeatOnLifecycle(STARTED) подписчик не «уходит» в фоне, и WhileSubscribed не сработает.

Задача 5. Реализуйте retryWithBackoff для Flow<T>: повторять подписку до maxAttempts раз с экспоненциальной задержкой (initialDelayMs, фактор 2.0) и случайным jitter, но не повторять при CancellationException и при неисправимых ошибках (предикат retryOn, например HTTP 401). Почему .retry(3) — анти-паттерн?

Показать решение 
fun <T> Flow<T>.retryWithBackoff(
    maxAttempts: Long,
    initialDelayMs: Long,
    factor: Double = 2.0,
    maxDelayMs: Long = 30_000,
    retryOn: (Throwable) -> Boolean = { true },
): Flow<T> = retryWhen { cause, attempt ->
    if (cause is CancellationException) throw cause   // отмену не ретраим
    if (attempt >= maxAttempts || !retryOn(cause)) return@retryWhen false
    val base = (initialDelayMs * factor.pow(attempt.toInt())).toLong()
        .coerceAtMost(maxDelayMs)
    val jitter = Random.nextLong(base / 2 + 1)       // разводим всплески ретраев
    delay(base + jitter)
    true
}

Объяснение: retryWhen даёт и причину, и номер попытки, а delay в его предикате легален (это suspend). CancellationException ретраить нельзя — она часть механизма отмены, проброс обязателен. retryOn отсекает неисправимое (401, 404): повторять их бессмысленно. Jitter разносит ретраи множества клиентов во времени, иначе после сбоя сервера все ломанутся одновременно (thundering herd). Голый .retry(3) — анти-паттерн ровно потому, что повторяет всё подряд (включая отмену и неисправимые ошибки) без задержки, мгновенными залпами добивая упавший бэкенд.

Задача 6. Реализуйте debounce(timeoutMs) для Flow<T> с нуля (без стандартного оператора): эмитить значение, только если после него timeoutMs не пришло нового; на каждой новой эмиссии таймер сбрасывается. В разборе сравните debounce, throttleFirst и sample.

Показать решение 
fun <T> Flow<T>.debounce(timeoutMs: Long): Flow<T> = channelFlow {
    var timer: Job? = null
    collect { value ->
        timer?.cancel()                  // сброс таймера на новой эмиссии
        timer = launch {
            delay(timeoutMs)
            send(value)                  // прошло timeoutMs тишины — эмитим
        }
    }
    timer?.join()                        // не потерять последнее значение
}

Объяснение: ключ — отменять предыдущий таймер на каждой новой эмиссии, поэтому быстрая серия значений «съедается» и наружу выходит лишь последнее после паузы. Разница трёх операторов: debounce ждёт паузу после значения — для поля ввода (печатает пользователь — реагируем, когда остановился); throttleFirst пропускает первое значение окна, остальные в окне гасит — для тапа по кнопке (защита от дабл-клика, реагируем сразу на первый); sample с периодом T берёт самое свежее значение раз в T — для частого источника, который надо «прореживать» по таймеру (координаты, сенсор).

Задача 7. Напишите search(queries: Flow<String>): Flow<List<Result>> для type-ahead поиска: игнорировать пустые строки, выждать паузу ввода 300 мс, не повторять одинаковые запросы подряд и отменять предыдущий сетевой вызов при новом вводе. api.search — suspend-функция.

Показать решение 
fun search(queries: Flow<String>): Flow<List<Result>> = queries
    .filter { it.isNotBlank() }
    .debounce(300)
    .distinctUntilChanged()
    .flatMapLatest { q -> flow { emit(api.search(q)) } }

Объяснение: debounce(300) гасит дребезг ввода; distinctUntilChanged отсекает повтор того же запроса (например, после стирания и возврата буквы); flatMapLatest отменяет предыдущий внутренний поток (текущий запрос к api) при новом вводе — именно он, а не flatMapMerge/Concat, нужен для поиска. Подвох: порядок операторов важен — filter/debounce до flatMapLatest.

Задача 8. Напишите свой промежуточный оператор pairwise() для Flow<T> через flow { upstream.collect { ... } }: он эмитит пары соседних значений (предыдущее, текущее). Для потока 1,2,3 результат — (1,2),(2,3).

Показать решение 
fun <T> Flow<T>.pairwise(): Flow<Pair<T, T>> = flow {
    var prev: T? = null
    var has = false
    collect { value ->
        if (has) emit(prev as T to value)   // есть предыдущее — эмитим пару
        prev = value
        has = true
    }
}

Объяснение: канонический способ написать оператор — обернуть всё в flow { } и внутри собрать upstream через collect, эмитя в downstream нужное. Состояние (предыдущее значение) живёт в локальной переменной билдера, поэтому оно изолировано на каждого подписчика — оператор остаётся холодным и переиспользуемым. Первое значение пары не образует, флаг has отсекает фальстарт с null. Так же строятся scan, throttleLatest и прочие кастомные операторы — context preservation соблюдается автоматически, потому что emit идёт из того же контекста, что и тело flow.

Задача 9. Реализуйте token-bucket rate limiter: ёмкость capacity токенов, пополнение refillPerSec токенов в секунду. Метод suspend fun acquire() должен пропускать всплески до ёмкости, а при пустом ведре — приостанавливаться до появления токена (не блокируя поток).

Показать решение 
class TokenBucket(
    private val capacity: Double,
    private val refillPerSec: Double,
) {
    private val mutex = Mutex()
    private var tokens = capacity
    private var lastNanos = System.nanoTime()

    suspend fun acquire() {
        while (true) {
            val waitMs = mutex.withLock {
                val now = System.nanoTime()
                val elapsedSec = (now - lastNanos) / 1_000_000_000.0
                tokens = (tokens + elapsedSec * refillPerSec).coerceAtMost(capacity)
                lastNanos = now
                if (tokens >= 1.0) { tokens -= 1.0; return }    // токен есть — пропускаем
                ((1.0 - tokens) / refillPerSec * 1000).toLong()  // сколько ждать
            }
            delay(waitMs.coerceAtLeast(1))                       // suspend, не блокировка
        }
    }
}

Объяснение: ведро пополняется не таймером, а лениво — при каждом acquire мы доливаем токены пропорционально прошедшему времени (elapsed * refillPerSec), ограничив сверху ёмкостью. Накопленные до capacity токены и дают всплеск: если запросов давно не было, первые capacity вызовов проходят сразу. Пустое ведро — не reject, а расчёт времени до следующего токена и delay на него, поэтому поток свободен. Mutex сериализует доступ к состоянию без блокировки потока (в отличие от synchronized). Для варианта с отказом вместо ожидания достаточно вернуть false, когда токенов нет.
