# Выбор визуализации: тип и инструмент

Сначала решается не «как нарисовать», а «нужна ли схема и какого она типа». Неверно выбранный тип —
причина большинства плохих диаграмм: процесс рисуют деревом, сравнение втискивают в схему, цикл
заворачивают в кольцо. Этот файл — маршрутизация.

## Нужна ли вообще диаграмма

Диаграмма оправдана, когда смысл лежит в **структуре или потоке**, и текстом он передаётся хуже.
В остальных случаях есть инструменты точнее:

| Что у вас за материал | Чем подать |
| --- | --- |
| Последовательность шагов, ветвление, состояния, вложенность, связи компонентов | **Диаграмма** (Mermaid или SVG) |
| Ровное сравнение N вариантов по одинаковым критериям | **Таблица сравнения** (`.cmp` из `components.md`) |
| Количественные данные (доли, динамика, распределение) | **График** (Chart.js); на этом сайте — редко |
| Две-три связи, которые проще назвать словами | **Текст**, без схемы |

Не рисуйте диаграмму ради диаграммы. Если параграф объясняет лучше — оставьте параграф.

## Маршрутизация по глаголу, а не по теме

Тип определяется тем, **что спросили о предмете**, а не самим предметом. Один и тот же предмет даёт
разные схемы:

| Запрос (по сути) | Тип | Что рисуем |
| --- | --- | --- |
| «как идёт процесс», «какие шаги», «что происходит при …» | **Flowchart** | Блоки и стрелки в одном направлении; ромбы для ветвлений |
| «кто кому и в каком порядке шлёт» | **Sequence** | Участники сверху, сообщения-стрелки вниз по времени |
| «в каких состояниях бывает и как переходит» | **State** | Состояния-узлы, переходы-стрелки с условием |
| «что внутри чего», «где это живёт», «как устроено» | **Структурная** | Вложенные контейнеры (организм → клетка → органелла) |
| «как связаны сущности данных», «схема БД» | **ER / class** | Сущности с полями, связи с кардинальностью |
| «дай общую карту темы», «как разделы связаны» | **Mind-map / SVG** | Центр + ветви (пример — `buildMindmap()`) |
| «сравни A и B по критериям» | **Таблица**, не схема | Строки-критерии, столбцы-варианты |

Полезное различие из практики: пользователь хочет это **задокументировать** (нужна точная карта —
блоки, подписи, стрелки) или **понять интуитивно** (нужна правильная ментальная модель)? Для учебного
сайта почти всегда первое — опорные схемы. Иллюстративная метафора (нарисовать механизм: стек кадров
растёт и убывает, хэш раскидывает ключи по корзинам) уместна там, где озарение — в том, чтобы
**увидеть**, а не прочитать; но это более амбициозный и трудоёмкий формат, применяйте его осознанно.

## Mermaid или рукописный SVG

На сайте оба движка уже подключены и темизированы. Выбор между ними:

**Бери Mermaid**, когда раскладку лучше отдать движку и тип стандартный:

- `flowchart` (LR/TD) — процессы, конвейеры, дерево решений.
- `sequenceDiagram` — обмен сообщениями во времени (клиент ↔ сервер, IPC, рукопожатие).
- `stateDiagram-v2` — жизненный цикл, состояния и переходы.
- `classDiagram` — отношения классов/интерфейсов.
- `erDiagram` — сущности и связи данных (таблица из заголовка + строк полей — это задача раскладки
  текста, движок делает её корректно; руками в SVG такое стабильно не выходит).

Mermaid сам считает координаты, разводит стрелки и переносит текст — для перечисленного это
надёжнее ручного SVG и устойчиво к правкам. Контракт разметки (`<div class="diagram">` + `.mermaid` +
`.cap`, директива без отступа, кавычки у проблемных узлов) — в `interview-prep-author`.

**Бери рукописный инлайновый SVG**, когда:

- Нужна нестандартная композиция, которой нет среди типов Mermaid (карта-радиал, слоистый стек,
  «воронка», пользовательская структурная схема с точным расположением).
- Нужна иллюстративная схема-метафора (нарисовать механизм, а не блоки о механизме).
- Нужен полный контроль над цветом из переменных темы и над доступностью.

В репозитории уже есть образец рукописного SVG — `buildMindmap()` в `site/assets/app.js`: генерация
через шаблонные строки, `role="img"` + `aria-label`, цвета из конфигурации `GROUPS`. Им и
ориентируйтесь по стилю. Правила построения (математика «текст влезает», viewBox, обе темы) — в
`design-system.md`.

## Декомпозиция: когда тема не влезает в одну схему

Если в запросе 6+ компонентов или поток с множеством ветвлений — **не пытайтесь уместить всё в один
холст**. Получите налезающие блоки и стрелки сквозь текст. Считайте узлы заранее и разбивайте:

1. Снимите обзорную схему: только основные блоки и одна-две главные стрелки потока; без вееров
   «один-ко-многим» и сеток N×N.
2. Затем по одной схеме на каждый интересный подпоток («что происходит при размещении заказа», «как
   проходит авторизация»), 3–4 узла каждая, с воздухом.

Между схемами всегда идёт абзац-связка (его пишет `tech-author`): что показывает следующая схема и как
связана с предыдущей. Несколько ясных схем с текстом между ними сильнее одной перегруженной.

## Циклы не рисуются кольцом

Если последний шаг возвращается к первому (event loop, mark-and-sweep, повтор TCP), инстинкт —
расставить шаги по окружности. Не делайте этого: правила отступов и непересечения — декартовы, для
«узла на орбите» нет проверки коллизий, и спутники налезут на стадии, а подписи сядут на окружность.
Петлю передаёт **возвратная стрелка** в линейной раскладке или **пошаговый блок**, где «Далее» с
последнего шага ведёт на первый. Кольцо — это декор; смысл цикла несёт стрелка возврата.
