# Каталог ИИ-маркеров

То, что выдаёт текст, сгенерированный ИИ. Работает в обе стороны: при письме — список
«так не пиши»; при вычитке чужого/вставленного черновика — список «найди и перепиши».
В каждом пункте `✗` — маркерная версия, `✓` — конкретная замена (факт вместо штампа).

Принцип за всеми пунктами один: **ИИ заменяет конкретный факт оборотом о факте**
(важности, значимости, «рассмотрим»). Лечение всегда одно — выкинуть оборот и поставить
сам факт с механизмом или числом.

**Регистр.** Целевой голос методички — формальный академический (`voice.md`), не
разговорный. Часть «✓»-примеров ниже сформулирована разговорно ради краткости: сохраняй
их конкретику, но поднимай регистр по `voice.md` (без «ты»-обращения и сленга). Несколько
пунктов (связка «является», безличные «следует/необходимо», зачин «рассмотрим») помечены
для разговорного голоса — в академическом регистре они допустимы, когда несут конкретику;
маркером остаётся лишь пустое или механически повторяемое употребление.

## Русская лексика (слова и штампы)

- **«Стоит/Важно отметить, что…»** — анонс важности вместо самого факта.
  ✗ Стоит отметить, что StateFlow всегда хранит последнее значение и отдаёт его новым подписчикам.
  ✓ StateFlow хранит последнее значение и сразу отдаёт его новым подписчикам — поэтому ему нужен initial value, а SharedFlow можно создать без него.

- **«играет ключевую/важную роль»** — пустая метафора значимости вместо действия.
  ✗ Корутины играют ключевую роль в асинхронном программировании на Android.
  ✓ Корутины дают писать асинхронный код последовательно: suspend-функция приостанавливает корутину, не блокируя поток, поэтому Main не фризится.

- **«позволяет / обеспечивает / предоставляет возможность»** — безличная связка-затычка под любую фичу.
  ✗ ViewModel позволяет сохранять данные при пересоздании Activity и обеспечивает разделение логики и UI.
  ✓ ViewModel переживает смену конфигурации: Activity пересоздаётся, а ViewModelStore держит тот же инстанс, пока scope жив. Сетевой запрос не перезапускается на повороте.

- **Канцелярит «осуществляется / производится / выполняется»** — обезличенные отглагольные вместо живого глагола.
  ✗ Сборка мусора осуществляется средой выполнения ART, а выделение памяти производится в куче.
  ✓ Память чистит сборщик мусора ART. Объекты живут в куче; при нехватке памяти ART запускает GC, и если у объекта остались только слабые ссылки — он выгружается.

- **«глубокое понимание / глубокое погружение»** — оценочный штамп из лендингов.
  ✗ Для собеседования необходимо глубокое понимание жизненного цикла Activity.
  ✓ На собесе по жизненному циклу спросят: что вызовется при повороте и куда сохранять состояние — onSaveInstanceState или ViewModel, и чем они отличаются.

- **«и многое другое / и так далее»** — обрыв перечисления вместо конкретики.
  ✗ Compose решает проблемы старого View: бойлерплейт, ручной findViewById, кастомные вью и многое другое.
  ✓ Compose убирает три боли View: findViewById (UI описывается функцией), ручную синхронизацию состояния с вьюхами (рекомпозиция сама), наследование во вью-иерархии (вместо него — композиция функций).

- **«Безусловно / Несомненно / Очевидно»** — имитация авторитетности на ровном месте.
  ✗ Несомненно, Kotlin является предпочтительным языком для Android.
  ✓ Google с 2019 года объявил Kotlin приоритетным для Android: новые гайды, Jetpack и Compose идут Kotlin-first.

- **«В современном мире / На сегодняшний день»** — журнальный зачин «о вечном».
  ✗ В современном мире мобильной разработки многопоточность является неотъемлемой частью приложений.
  ✓ Любое приложение с сетью или базой упрётся в многопоточность: запрос нельзя делать на Main, иначе через 5 секунд прилетит ANR.

- **«является» связкой в КАЖДОМ предложении** — калька с «is»; маркер не в самом слове, а в его механическом повторе в каждой фразе. В академическом регистре умеренная связка «является», вводящая определение, допустима (см. `voice.md`); лечение — варьировать форму, а не запрещать слово.
  ✗ LiveData является observable-холдером данных, который является жизненно-цикло-зависимым.
  ✓ LiveData — observable-холдер, привязанный к жизненному циклу: шлёт обновления только активным наблюдателям и сам отписывается в onDestroy, поэтому не течёт.

- **«Давайте рассмотрим / Перейдём к рассмотрению»** — мостик из обучающего видео, шов между кусками.
  ✗ Давайте рассмотрим, как работает механизм рекомпозиции в Jetpack Compose.
  ✓ Рекомпозиция работает так: при изменении прочитанного state Compose помечает затронутые композаблы dirty и перевыполняет только их, а не всё дерево.

- **«Таким образом, можно сделать вывод, что…»** — риторическая концовка там, где вывод очевиден.
  ✗ Таким образом, можно сделать вывод, что использование корутин является более предпочтительным по сравнению с потоками.
  ✓ Итог: на 10 000 параллельных задач корутины дешевле потоков — поток это ~1 МБ стека и переключение ядром, корутина это объект на куче, мультиплексируемый на пул потоков.

- **«благодаря чему / тем самым / что в свою очередь позволяет»** — связки, раздувающие мысль в придаточную цепь.
  ✗ Hilt генерирует код во время компиляции, благодаря чему отсутствует рефлексия, что в свою очередь позволяет ускорить запуск.
  ✓ Hilt собирает граф зависимостей кодогенерацией на этапе компиляции — без рефлексии в рантайме. Ошибки в графе ловятся при сборке, а не падением на старте, как в Dagger без Hilt.

- **«широкий спектр / богатый набор / мощный инструмент»** — маркетинговые эпитеты из README.
  ✗ Корутины — мощный инструмент, предоставляющий широкий спектр возможностей для асинхронной работы.
  ✓ Корутины закрывают четыре задачи: фон (Dispatchers.IO), параллелизм (async/await), стримы (Flow), отмену по scope. Structured concurrency гарантирует отмену дочерних задач вместе с родителем.

- **«необходимо учитывать / следует помнить / важно понимать»** — преамбула, оттягивающая сам факт.
  ✗ При работе с корутинами важно помнить о необходимости правильной обработки контекста выполнения.
  ✓ Главная ловушка корутин: withContext(Dispatchers.IO) меняет поток только внутри блока. Дёрнешь тяжёлый CPU-расчёт в viewModelScope.launch без смены диспатчера — он уедет на Main и подвесит UI.

- **«не только…, но и…»** — парный усилитель, симметрично раздувающий фразу.
  ✗ Sealed-классы не только обеспечивают типобезопасность, но и позволяют исчерпывающе обрабатывать состояния.
  ✓ Sealed-классы дают исчерпывающий when без ветки else: добавишь состояние Error.Timeout — компилятор подсветит все when, где его забыли.

- **«в полной мере / в значительной степени / в большинстве случаев»** — хедж без числа за ним.
  ✗ Использование remember в значительной степени снижает количество ненужных вычислений при рекомпозиции.
  ✓ remember кэширует результат между рекомпозициями: тяжёлый парсинг посчитается один раз, а не на каждый кадр анимации. Без remember — каждый кадр заново.

- **«как мы видим / как уже было сказано / как известно»** — ложная отсылка к контексту, которого нет.
  ✗ Как известно, главный поток отвечает за отрисовку UI, поэтому, как мы видим, блокировать его нельзя.
  ✓ Main-поток рисует кадры каждые 16 мс (60 FPS). Заблокируешь дольше — пропущенные кадры (jank), свыше 5 секунд — диалог ANR.

## Русский синтаксис (конструкции)

- **«не просто X, а Y» / «не столько X, сколько Y»** — подставное противопоставление без реального контраста.
  ✗ Docker — не просто инструмент контейнеризации, а целая философия изоляции.
  ✓ Docker изолирует приложения в контейнерах: зависимости одного сервиса не конфликтуют с другим.

- **Триколон (перечисления ровно по три)** — третий пункт притянут ради ритма.
  ✗ Хороший API должен быть простым, понятным и предсказуемым.
  ✓ Хороший API предсказуем: одинаковые входы дают одинаковые ответы, ошибки возвращаются в едином формате.

- **«С одной стороны… с другой стороны»** — искусственное равновесие, где один вариант сильнее.
  ✗ С одной стороны, монолит проще, с другой — микросервисы лучше масштабируются.
  ✓ Для команды из трёх человек монолит проще: один деплой, одна кодовая база, никакой возни с сетью между сервисами.

- **Эмфатическое тире «— и это…»** — драматическая пауза перед довеском, где хватило бы точки.
  ✗ Кэш инвалидируется при каждом обновлении — и именно здесь кроется главная сложность всей архитектуры.
  ✓ Кэш инвалидируется при каждом обновлении. Сложно угадать момент, когда данные уже устарели, но клиент ещё их читает.

- **Цепочки причастных оборотов («позволяющий… обеспечивающий… упрощающий»)** — нанизанные причастия = канцелярит.
  ✗ Применяется паттерн, позволяющий абстрагировать слой доступа к данным, обеспечивающий гибкость и упрощающий тестирование.
  ✓ Паттерн прячет доступ к данным за интерфейсом. В тестах подставляем заглушку вместо реальной БД.

- **Безличное долженствование «следует / необходимо / важно понимать»** — менторский регистр без субъекта.
  ✗ Важно понимать, что при работе с транзакциями следует учитывать уровень изоляции.
  ✓ На READ COMMITTED два параллельных перевода прочитают один баланс и оба спишут деньги. Берите SERIALIZABLE.

- **Размытый квантор «тот или иной / те или иные»** — заполнитель вместо конкретики.
  ✗ В зависимости от тех или иных требований выбирают ту или иную стратегию репликации.
  ✓ Нужна строгая консистентность — синхронная репликация. Готовы терять секунды данных ради скорости — асинхронная.

- **Универсальный предикат «X позволяет + инфинитив»** — безагентная формула, размывающая кто и зачем.
  ✗ Использование Kafka позволяет организовать надёжную передачу событий между компонентами.
  ✓ Через Kafka сервис заказов шлёт событие «оплачено», а склад и доставка читают его независимо, каждый в своём темпе.

- **Парные синонимы через «и» («надёжность и стабильность», «гибкость и масштабируемость»)** — сдваивание ради веса.
  ✗ Такой подход обеспечивает надёжность и стабильность работы сервиса под высокой нагрузкой.
  ✓ Под нагрузкой 10к RPS сервис держит 99-й перцентиль под 50 мс и не падает — проверяли в нагрузочном тесте.

(Сюда же — «не только…, но и…», «играет ключевую роль», «Таким образом… является…»: те же приёмы из лексического списка, но в виде готовой синтаксической формы.)

## Структура и формат

- **Список «**Термин:** — объяснение» под копирку.** Сильнейший структурный тэлл: весь
  материал упакован в один шаблон. У живого автора форматы перемешаны.
  ✗ Десятки однотипных пунктов «- **StateFlow:** горячий… / - **SharedFlow:** горячий… / - **LiveData:** lifecycle-aware…» на весь документ.
  ✓ Чередуй подачу: связный абзац с trade-off, таблица сравнения, мини-пример кода, иногда простой список без жирных лейблов. Шаблон «термин — двоеточие» — для глоссария, не для объяснений.

- **Зеркальная структура всех разделов под одну гребёнку.** Идеальная симметрия рубрик — машинный признак.
  ✗ 15 разделов с буквально одинаковыми подзаголовками в одном порядке, включая темы, где рубрика пустует.
  ✓ Каркас — ориентир, не закон. Где рубрика не несёт нового — выкинь её для этой темы; где нужно иначе (сравнение, чек-лист, кейс) — меняй структуру. Лёгкая асимметрия читается как живой автор.

- **Одинаковый ритм: равная длина абзацев и параллельные формулировки.** Живой конспект рваный, формы смешаны.
  ✗ Каждый абзац ровно 3–4 предложения; список под кальку «глагол+объект»: «- Понимать жизненный цикл. / - Знать про утечки. / - Уметь сохранять состояние».
  ✓ Дай ритму дышать: на сложном/спорном — развёрнуто, на очевидном — одна строка или пропусти. Смешивай формы пунктов: где-то назывное, где-то вопрос с собеса, где-то пункт с мини-примером.

- **Резюме в конце КАЖДОГО раздела.** Человек резюмирует только там, где есть нетривиальный вывод.
  ✗ Каждая секция кончается близнецом «…Таким образом, корутины — мощный инструмент, который важно понимать».
  ✓ Заканчивай раздел там, где мысль закончилась. Итог — только если несёт новое: правило выбора, типичную ошибку, ссылку вперёд. Большинству разделов вывод не нужен.

- **Финальное «Подводя итог / В заключение» с мотивацией.** Фирменное завершение ответа чат-бота.
  ✗ «Подводя итог, мы рассмотрели все ключевые темы. Систематическая подготовка обязательно поможет вам пройти интервью. Удачи!»
  ✓ Заверши функционально: чек-лист готовности, маршрут по дням, источники. Без пересказа и мотивационного эпилога.

- **«Во-первых… Во-вторых… В заключение» (нарратив порядка).** Сигналинг порядка вместо реальной связи.
  ✗ «Отмена корутины. Во-первых, кооперативная. Во-вторых, проверяется через isActive. В-третьих, есть NonCancellable. В заключение, отмена важна».
  ✓ Связывай по смыслу: «Отмена кооперативная: корутина сама проверяет isActive и бросает CancellationException в точках suspend. Поэтому тяжёлый цикл без проверок не отменится — туда добавляют ensureActive()». Словесная нумерация — только для реальных шагов алгоритма.

- **«Давайте разберёмся / рассмотрим» как зачин.** Пустое обращение к воображаемому слушателю.
  ✗ «Давайте разберёмся, как работает HashMap».
  ✓ Начинай с сути: «HashMap хранит пары в бакетах; индекс — по hashCode, коллизии разрешаются цепочкой. С Java 8 длинная цепочка (от 8 узлов, при ёмкости таблицы ≥ 64) становится красно-чёрным деревом; если таблица меньше 64 — HashMap вместо этого делает resize».

- **Пустой вступительный абзац-вода о важности темы.** Разогрев модели, а не контент.
  ✗ «Многопоточность — одна из важнейших и сложных тем. Понимание её критично. Рассмотрим ключевые аспекты».
  ✓ Начни с информативной фразы: «UI-поток нельзя блокировать дольше ~16 мс на кадр; всё тяжёлое — в Dispatchers.Default/IO. Главные грабли — гонки за общим состоянием и утечки через незакрытые scope».

- **Однотипные мостики между разделами.** Единообразие выдаёт генерацию.
  ✗ «Теперь, когда мы разобрали Kotlin, перейдём к компонентам Android» — перед каждым разделом.
  ✓ Доверься заголовку и разделителю как переходу. Нужна связка по смыслу — делай её содержательной и разной.

- **Декоративные эмодзи в заголовках и буллетах.** Подпись чат-бота; в Senior-методичке чужеродны, не переживают копипаст в IDE/PDF.
  ✗ «## 🚀 Корутины», «## 🧠 Архитектура», в тексте «🔥 Важно:», «💡 Совет:».
  ✓ Заголовки словами: «## Корутины: отмена и structured concurrency». Светофор статуса (🟢🟡🔴) или ➕/➖ как осмысленная семантика — ок; декоративные ракеты/огонь — убрать.

- **Избыточная жирность.** Когда выделено всё — не выделено ничего.
  ✗ «**ViewModel** хранит **состояние** и переживает **смену конфигурации**, в отличие от **Activity**, через **SavedStateHandle**» — 5 болдов в фразе.
  ✓ Болд точечно: первое вхождение нового термина или ключевое предупреждение, 0–2 на абзац.

- **Парад «плюсы/минусы» в каждом подразделе.** Симметричный блок налеплен даже на тривиальное.
  ✗ У каждого инструмента ровный двойной список «**Плюсы:** … **Минусы:** …», включая вещи без реального компромисса.
  ✓ Сравнение — там, где есть настоящий выбор, и как решение: «Бери DataStore: асинхронный, транзакционный, на Flow; SharedPreferences — для легаси и простейших флагов». Где альтернативы нет — не выдумывай минусы.

## Английский (для англоязычных вставок и источников)

- **"delve into"** — signature LLM verb. ✗ we'll delve into how the coroutine state machine handles suspension → ✓ this section covers how the coroutine state machine handles suspension.
- **"leverage" (verb)** — filler for "use". ✗ leverage StateFlow to expose UI state → ✓ use StateFlow to expose UI state.
- **"robust / powerful"** — vague praise. ✗ Hilt provides a robust solution for DI → ✓ Hilt generates the Dagger wiring, so you write less DI boilerplate.
- **"seamless / seamlessly"** — marketing adjective rarely literally true. ✗ Navigation integrates seamlessly with Compose → ✓ Navigation 2.8 adds type-safe routes for Compose using @Serializable types.
- **"It's worth noting that"** — throat-clearing before a fact. ✗ It's worth noting that StateFlow emits the latest value → ✓ StateFlow replays its latest value to every new collector.
- **"In today's fast-paced world"** — canned opener, zero content. ✗ In today's fast-paced mobile world, state management matters more than ever → ✓ State management matters because a wrong update can leak memory or drop UI events.
- **"not only… but also"** — over-balanced correlative. ✗ Coroutines not only simplify async but also make cancellation easier → ✓ Coroutines simplify async and make cancellation easier.
